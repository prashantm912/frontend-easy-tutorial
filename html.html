HTML

HTML is the structural layer of the web: it describes what content means, not how it looks. This page moves concept by concept from the document skeleton and semantic elements through text, links, media, forms, and accessibility, then into the platform's more advanced building blocks — metadata, templates and slots, data attributes, custom elements, and the progressive-enhancement mindset that ties them together. Every runnable example edits live; change the code and press Run.

Structure & semantics

Every HTML document is a tree. The <!doctype html> declaration switches the browser into standards mode, <html> is the root, <head> holds metadata, and <body> holds rendered content. Inside the body, semantic elements name the role of each region: <header>, <nav>, <main>, <article>, <section>, <aside>, and <footer>.

Reach for the element whose meaning matches the content. <article> is a self-contained, independently distributable unit (a blog post, a comment, a product card). <section> is a thematic grouping that usually carries a heading. <aside> is tangential content, and <figure> with <figcaption> associates an illustration with its caption. A document should contain exactly one <main>, not nested inside <article>, <aside>, <header>, or <footer>.

Example: semantic blog-article layout

Note: Semantic elements generate ARIA landmark roles automatically — <nav> becomes a navigation landmark, <main> a main landmark — so screen-reader users can jump directly between regions. Search engines rely on the same structure to understand and rank content. A page built from <div>s alone conveys none of this.

Text elements

Headings <h1>–<h6> form the document outline; do not skip levels for visual sizing (style with CSS instead). Paragraphs use <p>. Inline semantics carry meaning: <strong> for importance, <em> for stress emphasis, <mark> for relevance, <small> for side comments, <abbr> for abbreviations, <time> for machine-readable dates, and <code> for code fragments.

Lists come in three flavours: <ul> (unordered), <ol> (ordered), and <dl> (description lists, pairing <dt> terms with <dd> descriptions). Use <blockquote> for extended quotations, with an optional cite attribute pointing at the source.

Note: <b> and <i> exist but carry no semantic weight — they simply mean "stylistically offset". Prefer <strong> and <em> when the emphasis is meaningful, and reserve <b>/<i> for cases like keywords or taxonomic names where no importance is implied.

Links & media

The anchor element <a> links to resources via href. Values can be relative (about.html), absolute (https://example.com), an in-page fragment (#section-id), or a scheme like mailto: or tel:. When opening a new tab with target="_blank", add rel="noopener" so the new page cannot reach window.opener (modern browsers imply this, but stating it is explicit and safe).

Images use <img> with a required alt attribute. loading="lazy" defers off-screen images. Responsive images combine srcset (candidate files with width descriptors) and sizes (how wide the image renders) so the browser downloads the smallest sufficient source. For art direction or format fallbacks, wrap sources in <picture> and pair the image with a <figure>/<figcaption>.

Example: anchors of every kind

Example: responsive picture with figure

Note: alt text should describe the image's purpose in context, not merely name it. Decorative images that add nothing should use empty alt="" so screen readers skip them. Never omit the attribute entirely — a missing alt makes assistive tech read the file name.

Note: loading="lazy" avoids fetching images the user may never scroll to, and srcset/sizes prevent shipping a 2000px image to a 320px phone. Always set explicit width and height (or an aspect ratio) so the browser reserves space and avoids layout shift.

Forms & validation

Forms collect input. Every control should have an associated <label> — either wrapping the control or referencing it via for matching the control's id. Input type selects the right keyboard, picker, and built-in validation: text, email, number, date, range, checkbox, radio, and file, among others.

Constraint-validation attributes let the browser check input before submit: required, pattern (a regular expression), min/max/step for numeric and date ranges, and minlength/maxlength. Group related fields with <fieldset> and name the group with <legend>. Use <select> with <optgroup> for grouped choices and <textarea> for multi-line text.

Example: settings form with select and radios

Note: Associate every control with a <label> so clicking the label focuses the field and screen readers announce it. When validation fails, expose the error message near the field and reference it with aria-describedby; do not rely on colour alone to signal an error.

Accessibility & ARIA

ARIA (Accessible Rich Internet Applications) fills gaps that native HTML cannot express. Landmark roles (banner, navigation, main, contentinfo) come free from semantic elements. When you build custom widgets, aria-label and aria-labelledby name an element, aria-describedby attaches supplementary description, and aria-live regions announce dynamic updates without moving focus.

Structured data is a related concern: microdata attributes (itemscope, itemtype, itemprop) annotate content with a vocabulary such as schema.org so search engines can extract entities like a Person or Product from the page.

Example: aria-live status region

Example: schema.org microdata (Person)

Note: Prefer native semantics over ARIA whenever possible — a real <button> is keyboard-operable and focusable for free, whereas <div role="button"> requires you to reimplement focus, Enter/Space handling, and disabled state. The first rule of ARIA is: don't use ARIA if a native element will do.

Document metadata

The <head> is where you configure how the document is interpreted, indexed, and shared. Essentials include <meta charset>, the responsive viewport, the <title>, and a meta[name=description] for search snippets. A <link rel="canonical"> declares the preferred URL for duplicated content, and meta[name=robots] controls indexing.

Social sharing relies on Open Graph (og:*) and Twitter Card (twitter:*) meta tags, which populate the rich preview when a link is pasted into a chat or social feed. A favicon is declared with <link rel="icon">. This markup is head content, not rendered, so the first example is a static code block; the second reads metadata back with JavaScript.

Example: a complete document head

<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Semantic HTML Guide — Acme</title>
  <meta name="description" content="A concise guide to writing meaningful HTML.">
  <link rel="canonical" href="https://acme.example/guides/html">
  <meta name="robots" content="index, follow">

  <!-- Open Graph -->
  <meta property="og:type" content="article">
  <meta property="og:title" content="Semantic HTML Guide">
  <meta property="og:description" content="Write meaningful HTML.">
  <meta property="og:image" content="https://acme.example/og.png">

  <!-- Twitter Card -->
  <meta name="twitter:card" content="summary_large_image">
  <meta name="twitter:title" content="Semantic HTML Guide">

  <link rel="icon" href="/favicon.ico" sizes="any">
</head>
Result: Browsers show "Semantic HTML Guide — Acme" in the tab; search engines use the description for snippets; and pasting the URL into a social app renders a large-image card titled "Semantic HTML Guide".

Example: read metadata with JavaScript

Note: The <title> and meta[name=description] are the two highest-leverage tags for search — the title becomes the clickable result heading and the description often becomes the snippet. Keep titles under about 60 characters and descriptions under about 155.

Templates & slots

The <template> element holds inert markup that the browser parses but does not render. Its contents live in a DocumentFragment exposed as .content; you clone it with cloneNode(true) and insert the clone where needed. This is the idiomatic way to stamp out repeated rows or cards without building strings by hand.

A <slot> is a placeholder inside a shadow tree that projects light-DOM children into the component's rendered output. Named slots (<slot name="title">) map to children carrying a matching slot="title" attribute, letting a component define structure while the consumer supplies content.

Data attributes

Custom data-* attributes attach arbitrary, valid metadata to any element without inventing non-standard attributes. They are readable in CSS via attribute selectors and in JavaScript via the dataset property, which camel-cases the suffix (data-user-id becomes el.dataset.userId).

Use them to drive styling states, stash identifiers for event handlers, or bridge server-rendered values into client scripts. They are not a substitute for a real data model, but they are ideal for small, element-scoped hints.

Note: dataset values are always strings. Convert explicitly with Number() or a === 'true' comparison, and remember that data-userId and data-user-id both map to dataset.userId — always author the hyphenated form.

Web components

Custom elements let you define new HTML tags backed by a class that extends HTMLElement. Register the tag with customElements.define('my-el', MyEl) — the name must contain a hyphen. Lifecycle callbacks (connectedCallback, disconnectedCallback, attributeChangedCallback) hook into insertion, removal, and attribute changes; the static observedAttributes array declares which attributes trigger the last one.

Encapsulation comes from an attached shadow root, which isolates the component's markup and styles. Combine a <template> for structure with a <slot> for consumer content to build reusable, framework-free UI.

Example: an accessible <toggle-tip>

Note: Custom elements, shadow DOM, and templates are supported in all current evergreen browsers. When a custom element's script fails to load or run, the browser renders it as an unknown inline element showing its light-DOM children — so put a sensible fallback (like the tooltip's plain text) inside the tag to degrade gracefully.

Progressive enhancement

Progressive enhancement means building a baseline that works with HTML alone, then layering on CSS and JavaScript as capabilities allow. The payoff is resilience: the core experience survives a failed script, a flaky network, or an unusual client. Native interactive elements like <details>/<summary> give you an accessible disclosure widget with zero JavaScript.

For forms, start with a real action and method so submission works server-side without scripting, then enhance with JavaScript to intercept, validate, and give instant feedback when it is available. Feature-detect before enhancing, and never make the baseline depend on the enhancement.

Note: Native elements like <details>, <button>, and validated inputs come with keyboard support, focus management, and screen-reader semantics already wired up. Reimplementing them in JavaScript almost always drops one of those behaviors — start native, enhance sparingly.

Feature detection in markup

An Employee Profile Dashboard has to render on whatever the browser can do today, not what you wish it could do. Feature detection asks the runtime whether a capability exists and adapts the markup accordingly, rather than sniffing user-agent strings. The classic markup-level technique is the <html class="no-js"> pattern: the page ships assuming JavaScript is off, and an inline script in the <head> — which runs before the body paints — swaps the class to js. CSS then keys off .no-js vs .js to show the right baseline.

This is the difference between graceful degradation (build the rich version, then fall back) and progressive enhancement (build the working baseline, then layer capability on). Markup gives you three fallback tools: <noscript> for a scripting-off alternative, fallback content between the tags of elements like <canvas>, and capability-gated regions that only appear once a feature is confirmed. This complements the programmatic detection taught on the JavaScript page (see javascript.html#feature-detection): markup sets the resilient default, and script upgrades it.

Example: capability-gated markup with native tests

Reference: the Modernizr approach

A library like Modernizr automates this. It runs on load, adds a class per detected feature to the <html> element, and exposes a boolean API. You cannot load it offline here, so the hand-rolled detector above is the runnable equivalent; the block below is how you would wire the real thing.

<!-- 1. Ship with the pessimistic class. -->
<html class="no-js">

<!-- 2. Modernizr swaps no-js -> js and adds
        one class per feature, e.g. class="js canvas
        localstorage csstransitions". -->
<script src="modernizr.js"></script>

<!-- 3a. Gate CSS on the generated classes. -->
<style>
  .no-canvas .chart { display: none; }
  .canvas    .chart-fallback { display: none; }
</style>

<!-- 3b. Gate JS on the boolean API. -->
<script>
  if (Modernizr.canvas) { /* draw the avatar */ }
  if (Modernizr.localstorage) { /* cache the profile */ }
  if (Modernizr.csstransitions) { /* animate panels */ }
</script>
Result: With Modernizr loaded, <html> reads something like class="js canvas localstorage no-touchevents", letting CSS and JS both branch on exactly the same detected-feature vocabulary.

Note: User-agent sniffing is fragile — strings are spoofed, frozen, and change constantly. Feature-detect the capability you actually need instead. The inline head script must be tiny and synchronous so the class flips before first paint, avoiding a flash of the wrong (no-js) layout.

The canvas element

The <canvas> element is a bitmap drawing surface — perfect for a dashboard's generated avatars, sparklines, or badges. It has two size systems that are easy to confuse. The width and height attributes set the intrinsic pixel buffer (the coordinate space you draw into). CSS width/height set the displayed box. If they disagree, the browser scales the bitmap, so a 300×150 buffer stretched to 600×300 by CSS looks blurry. Set the attributes to the real pixel dimensions you intend to draw.

Content placed between <canvas> and </canvas> is fallback content shown only when canvas is unavailable — use it for a text alternative. Reach for canvas when you need to draw many pixels imperatively (charts, image manipulation, games); prefer <svg> for resolution-independent, accessible, retained-mode vector shapes, and a plain <img> when you already have a raster asset. The 2D drawing API itself — fillRect, arc, fillText — is taught on the JavaScript page (see javascript.html#canvas-api).

Example: a circular avatar with initials

Example: fallback text plus DPR-aware sizing

Note: A canvas is an opaque bitmap to assistive tech. Always provide a text alternative — fallback content between the tags, plus role="img" and aria-label as in the avatar example — so a screen reader announces who the avatar represents rather than silence.

Note: Match the pixel buffer to the real display size. Over-sizing the buffer wastes memory and fill rate on every repaint; under-sizing it (then letting CSS upscale) produces blur. On HiDPI screens, multiply by devicePixelRatio and scale the context once, as shown, rather than redrawing at low resolution.

Contact links

A "Contact Employee" action is just an anchor with the right URL scheme — no JavaScript required. mailto: opens the user's mail client and accepts query parameters: subject, body, cc, and bcc. sms: composes a text message and accepts ?&body= for a prefilled message. tel: initiates a call. geo: hands latitude/longitude (and optional zoom) to a maps app. The browser or OS hands these off to the registered handler.

Every parameter value must be URL-encoded so that spaces, line breaks, and reserved characters (&, ?, #) survive intact — build them with encodeURIComponent rather than string concatenation. Treat the target address as data: never interpolate unescaped user input into a scheme URL, and remember these links reveal an address in the DOM, so use a role account when that matters.

Example: a contact card of scheme links

Example: build a mailto URL from fields

Note: Make the link text describe the action ("Email Dana", "Call the office"), not the raw address — screen-reader users hear the text, and "mailto:dana@example.com?subject=..." read aloud is noise. Reserve the visible address for where it genuinely helps sighted scanning.

Note: mailto: and tel: are broadly supported; sms: parameter syntax and geo: handling vary by platform (iOS, Android, and desktop differ, and some desktops have no handler at all). Provide a plain fallback — a visible phone number or address — so the contact still works when no handler is registered.

Structured data for an employee

Structured data annotates the profile so search engines, assistants, and rich-result cards can extract typed entities — a Person with a job title who works for an Organization — instead of guessing from prose. Microdata does this inline: itemscope opens an entity, itemtype names its vocabulary (a schema.org type URL), and itemprop tags each property. Nesting a second itemscope inside a property like worksFor expresses a relationship between two entities.

Microdata keeps the data attached to the visible markup, which stays in sync as the DOM changes. The modern alternative, JSON-LD, moves the same graph into a <script type="application/ld+json"> block in the head — decoupled from layout, easier to emit from a server, and the format Google recommends. The two are interchangeable in meaning; pick microdata when the data mirrors on-page content and JSON-LD when you want it centralized.

Example: a Person + Organization microdata card

Example: the equivalent JSON-LD

The same graph, expressed as JSON-LD, sits in the head decoupled from the layout. This block is head content, so it is shown as a static reference rather than run in the preview.

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Person",
  "name": "Dana Lee",
  "jobTitle": "Accessibility Engineer",
  "email": "mailto:dana@example.com",
  "telephone": "+1-555-123-4567",
  "image": "https://acme.example/people/dana.jpg",
  "worksFor": {
    "@type": "Organization",
    "name": "Acme Corp"
  }
}
</script>
Result: A search engine extracts one Person entity — name "Dana Lee", jobTitle "Accessibility Engineer", with email and telephone — linked to an Organization named "Acme Corp", and may surface it as a knowledge-panel or rich employee-profile result. Assistive tech and scrapers read the same typed graph without parsing the surrounding prose.

Note: Structured data must describe what is genuinely on the page — marking up content the user cannot see is both an SEO violation and an accessibility mismatch. With microdata the annotations ride on the real, visible DOM, so they stay honest; with JSON-LD, keep the block in sync with the rendered profile whenever it changes.

Data tables

A <table> is for two-dimensional data whose meaning depends on the intersection of a row and a column — never for page layout. Group rows into <thead> (column headers), <tbody> (the data), and <tfoot> (summaries such as totals). The grouping is semantic, not cosmetic: it lets user agents repeat the header when a long table breaks across printed pages and lets assistive tech distinguish headings from data.

Header cells use <th>; data cells use <td>. The scope attribute (col, row, colgroup, rowgroup) tells a screen reader which cells a header governs, so it can announce "Revenue, Q2: 41" when the user lands on that cell. For irregular tables, associate cells explicitly by giving each <th> an id and each <td> a matching headers list. colspan/rowspan merge cells; a <caption> supplies an accessible title; and <colgroup>/<col> style whole columns without touching every cell.

Example: quarterly metrics with caption, scope, and totals

Example: merged cells and reading a cell's header context

Example: a wide table that scrolls on a narrow viewport

Note: A caption gives the table an accessible name and stays visible on screen — prefer it to a preceding heading alone. When you wrap a table in a scroll container, add tabindex="0" and role="region" with an aria-label so keyboard users can focus and scroll it, and so its purpose is announced.

Note: Never use tables for layout. If a grid of cells is purely visual with no row/column relationship, use CSS grid or flexbox instead; as a last resort role="presentation" on a table strips its tabular semantics, but that is a smell, not a pattern to reach for.

Audio & video

The <video> and <audio> elements embed media natively. Add the controls attribute for the browser's built-in transport UI. Provide multiple <source> children in preference order — the browser picks the first one whose type it can play (for example WebM before MP4, or Ogg before MP3), so you cover every engine without scripting. Text between the tags is fallback content for browsers that support neither element.

Captions and subtitles come from <track> elements pointing at WebVTT files: kind="captions" for the deaf and hard of hearing, subtitles for translation, and chapters/descriptions for navigation and audio description. Tune loading with poster (a placeholder frame), preload (none/metadata/auto), and playback with autoplay, muted, loop, and playsinline. Browsers block autoplay with sound; a video must be muted to autoplay. The JavaScript media API (play(), pause(), currentTime, and events like timeupdate) drives custom players.

Example: a video with sources, a track, and fallback

<video controls width="480" height="270"
       poster="/media/cover.jpg" preload="metadata">
  <!-- The browser plays the first type it understands. -->
  <source src="/media/talk.webm" type="video/webm">
  <source src="/media/talk.mp4"  type="video/mp4">
  <!-- Captions for accessibility (WebVTT). -->
  <track kind="captions" srclang="en" label="English"
         src="/media/talk.en.vtt" default>
  <track kind="chapters" srclang="en"
         src="/media/talk.chapters.vtt">
  <!-- Shown only if <video> is unsupported. -->
  <p>Your browser cannot play video.
     <a href="/media/talk.mp4">Download the MP4</a>.</p>
</video>
Result: A Chromium or Firefox build with WebM support loads talk.webm; a browser without it falls through to talk.mp4. The poster image shows until playback, only metadata is fetched up front, and the English captions track is selected by default. A browser that supports neither codec shows the download link.

Example: a muted, autoplaying background video

<!-- muted is REQUIRED for autoplay to be allowed. -->
<video autoplay muted loop playsinline
       preload="auto" aria-hidden="true"
       class="bg-video">
  <source src="/media/loop.webm" type="video/webm">
  <source src="/media/loop.mp4"  type="video/mp4">
</video>

<style>
  /* Respect users who ask for less motion. */
  @media (prefers-reduced-motion: reduce) {
    .bg-video { display: none; }
  }

Result: The clip starts on its own, silent and looping, with playsinline keeping it inline on iOS rather than going fullscreen. Because it is decorative it carries aria-hidden="true", and the reduced-motion media query hides it entirely for users who have requested minimal animation.

Note: Captions are not optional for meaningful video — kind="captions" serves users who are deaf or hard of hearing, and kind="descriptions" narrates on-screen action for blind users. Never rely on autoplaying audio; it is disorienting and browsers block it by default.

Note: preload="metadata" fetches just enough to show duration and dimensions without downloading the whole file, and a poster gives an instant visual while bytes stream. Use preload="none" for below-the-fold media the user may never play.

Embedding external content

An <iframe> embeds another browsing context. Always give it a title — screen readers announce the frame by it — plus explicit width/height (or CSS sizing) to avoid layout shift. The sandbox attribute is the security lever: with an empty value the frame runs with every restriction on, and you re-grant capabilities token by token (allow-scripts, allow-forms, allow-popups, allow-same-origin). Crucially, omitting allow-same-origin forces the frame into a unique opaque origin, so it cannot read your cookies, storage, or DOM even when it comes from your own server.

Use srcdoc to supply an inline document instead of a URL, loading="lazy" to defer off-screen frames, and referrerpolicy to control the Referer header. The allow attribute applies a Permissions Policy (camera, microphone, fullscreen, geolocation). For documents like PDFs, <object> renders inline with fallback content between its tags; <embed> is the older, fallback-less equivalent. When in doubt, a plain link is often the most robust and accessible choice. On the receiving side, protect your own pages from being framed with X-Frame-Options or a CSP frame-ancestors directive to prevent clickjacking.

Example: a sandboxed iframe with srcdoc — scripts blocked until allowed

Example: cross-origin isolation from allow-same-origin

Example: an object embedding a PDF with a fallback link

<object data="/docs/report.pdf" type="application/pdf"
        width="100%" height="480">
  <!-- Rendered only if the browser can't display the PDF. -->
  <p>This browser can't show the PDF inline.
     <a href="/docs/report.pdf">Download report.pdf</a>.</p>
</object>
Result: Browsers with a built-in PDF viewer render the report inline in a 480px-tall box; those without one (many mobile browsers) fall back to the download link inside the element. Unlike <embed>, <object> supports this graceful fallback content.

Note: A frame with sandbox="" and no allow-same-origin runs in a unique opaque origin, so contentDocument access throws a SecurityError. Never combine allow-scripts and allow-same-origin for content you do not fully trust — together they let the frame remove its own sandbox.

Note: loading="lazy" on an iframe defers loading third-party embeds (maps, videos, widgets) until they are near the viewport, which can save hundreds of kilobytes and multiple connections on initial load.

Character references & escaping

Certain characters are structural in HTML, so writing them literally as data is ambiguous. The parser treats < as the start of a tag and & as the start of an entity; inside a double-quoted attribute, " ends the value. To show these as text you escape them with character references. There are five predefined named entities: &lt; (<), &gt; (>), &amp; (&), &quot; ("), and &apos; ('). In practice you always need & and <; the rest are situational.

References come in three forms: named (&copy;), decimal numeric (&#169;), and hexadecimal numeric (&#xA9;) — all three render ©. HTML collapses runs of whitespace to a single space; use &nbsp; to force a non-breaking space, &shy; for an invisible soft hyphen that only appears at a line break, &thinsp; for a thin space, and <pre> to preserve whitespace verbatim. This connects directly to security: when you inject user data into HTML you must escape it, or a value like <script> becomes executable markup. Setting textContent escapes automatically; concatenating into innerHTML does not.

Example: whitespace collapsing, <pre>, and spacing entities

Example: escaping user input — textContent vs innerHTML

Note: To display a code sample as text, escape every &, <, and >. To display the escaped source itself (teaching the reader what to type), double-escape: write &amp;lt; so the page renders the literal string &lt;. Note: Prefer real characters (é, —, ©) saved in a UTF-8 file over numeric references when you can type them — they are far more readable in source. Reserve references for the structural characters (&amp;, &lt;) and invisible ones (&nbsp;, &shy;) that you cannot see or type reliably.

Global attributes

Global attributes apply to every element. id (unique per document) and class (reusable) are the primary hooks for CSS and the DOM. title supplies an advisory tooltip, but it is unreliable for accessibility — it does not appear on touch, is hard to reach by keyboard, and is not consistently announced, so never put essential information only in a title. lang and dir drive internationalization: lang tells screen readers which pronunciation rules and hyphenation to use, and dir="rtl" flips text direction for scripts like Arabic and Hebrew.

Several globals control interactivity. hidden removes an element from rendering (like display:none); inert goes further, taking an entire subtree out of the tab order, pointer interaction, and the accessibility tree while leaving it visible — ideal for content behind a modal. tabindex="0" adds a non-interactive element to the natural tab order, tabindex="-1" makes it focusable only programmatically (for roving-tabindex widgets and focus management), and positive values are an anti-pattern that scrambles tab order. contenteditable makes a region editable, spellcheck toggles the checker, and hint attributes like enterkeyhint and inputmode tune the on-screen keyboard.

Note: inert is the correct tool for the background of a modal — unlike manually removing tabindex, it also hides the subtree from the accessibility tree so a screen reader cannot navigate into it. Avoid positive tabindex values: they override the natural DOM order and almost always create a confusing, unpredictable focus sequence.

Advanced form controls & JS validation

Beyond the basic inputs, HTML offers specialized controls. <datalist> bound via a control's list attribute supplies autocomplete suggestions while still allowing free text. <output> displays a calculated result, <progress> shows determinate or indeterminate task completion, and <meter> shows a scalar value within a known range (disk usage, a score) with optional low/high/optimum thresholds. A control can even sit outside its <form> and still belong to it via the form attribute.

The Constraint Validation API exposes validity in JavaScript. Every control has a validity object (with flags like valueMissing, typeMismatch, patternMismatch), a setCustomValidity(msg) method to force an error with your own message, and two check methods: checkValidity() returns a boolean silently, while reportValidity() also shows the browser's error bubble. The :valid/:invalid pseudo-classes style state in CSS. Pair this with autocomplete tokens (email, one-time-code) and inputmode for better mobile keyboards, and build an accessible error summary with aria-describedby.

Modal dialogs & popovers

The <dialog> element is a native modal or non-modal window. Call showModal() to open it as a true modal: the browser renders it in the top layer above all other content, paints a ::backdrop pseudo-element behind it, traps focus inside, and makes the rest of the page inert. show() opens it non-modally without a backdrop or focus trap. Close it with close(), the Escape key (modal only), or — most elegantly — a form with method="dialog", whose submit button's value becomes the dialog's returnValue.

The Popover API is lighter weight and needs no JavaScript. Any element with the popover attribute (auto or manual) starts hidden in the top layer; a button with popovertarget pointing at its id toggles it. An auto popover supports light dismiss — click outside or press Escape and it closes — and is non-modal with no backdrop, unlike a dialog. Reach for <dialog> when you need a modal that blocks the page, a popover for menus and transient overlays, and <details> for inline disclosure. For accessibility, give a dialog an accessible name with aria-labelledby, and note the browser restores focus to the opener on close.

Inline SVG

Inline <svg> is real DOM: each shape is an element you can style with CSS and script with JavaScript, unlike an opaque <canvas> bitmap or an external <img src=".svg">. The viewBox="minX minY width height" defines an internal coordinate system that scales to whatever size the SVG is rendered at, so a single drawing stays crisp at any resolution; preserveAspectRatio controls how it fits when the aspect ratios differ. Basic shapes — <rect>, <circle>, <line>, <path>, <text> — take fill and stroke.

For icon systems, define a shape once inside a <symbol> and stamp it anywhere with <use href="#id"> — the classic sprite pattern that avoids duplicating path data. Because inline SVG inherits CSS, using fill="currentColor" lets an icon take the surrounding text color automatically. Make meaningful SVG accessible with a <title> child plus role="img" and an aria-label; mark purely decorative SVG aria-hidden="true". Choose inline SVG for styleable, scriptable, scalable vector UI; <img> for a static asset you do not need to style; canvas for pixel-heavy imperative drawing; and a font icon only for legacy constraints. For equations, MathML (<math>) is the semantic vector option.

Loading scripts & resources

Where and how a <script> loads decides whether it blocks rendering. A classic script in the <head> with no attributes pauses the parser while it downloads and executes. Add defer and it downloads in parallel, then runs after the document is parsed, in document order — the right default for scripts that touch the DOM. Add async and it downloads in parallel and runs as soon as it arrives, in no guaranteed order — suited to independent third-party scripts. A type="module" script is deferred by default, runs in strict mode, and supports import; pair it with a nomodule fallback so legacy browsers load a transpiled bundle instead.

Resource hints in the <head> shape loading priority. rel="preload" with a matching as fetches a critical resource early; modulepreload does the same for ES modules; prefetch grabs something likely needed on the next navigation; preconnect and dns-prefetch warm up a connection to a third-party origin before you need it. loading="lazy" defers off-screen images and iframes, decoding="async" keeps image decoding off the main thread, and fetchpriority nudges the browser's priority. The <base> element sets a default URL and target for all relative links — powerful but easy to misuse, since it affects every relative reference on the page.

Example: a head reference block of resource hints

<head>
  <!-- Warm up the connection to a font host early. -->
  <link rel="preconnect" href="https://fonts.example"
        crossorigin>
  <link rel="dns-prefetch" href="https://cdn.example">

  <!-- Fetch the LCP hero image at high priority. -->
  <link rel="preload" as="image" href="/img/hero.avif"
        fetchpriority="high">

  <!-- Preload a font (as=font always needs crossorigin). -->
  <link rel="preload" as="font" type="font/woff2"
        href="/fonts/inter.woff2" crossorigin>

  <!-- Preload an ES module the entry script imports. -->
  <link rel="modulepreload" href="/js/app.js">

  <!-- Likely-next-page resource, low priority. -->
  <link rel="prefetch" href="/data/next-page.json">

  <script type="module" src="/js/app.js"></script>
</head>
Result: The browser opens the TLS connection to the font host before it parses any tag that needs it, fetches the hero image ahead of lower-priority requests, and downloads app.js and its module graph in parallel with parsing — so by the time the module executes, its imports are already in the cache. The prefetch sits at idle priority for a probable next navigation.

Example: the module / nomodule fallback pair

<!-- Modern browsers run this and IGNORE nomodule. -->
<script type="module" src="/js/app.modern.js"></script>

<!-- Legacy browsers run this and IGNORE type=module. -->
<script nomodule src="/js/app.legacy.js"></script>
Result: A browser that understands ES modules executes app.modern.js and skips any nomodule script; an older browser does the opposite. Each engine runs exactly one of the two, so you ship untranspiled modern code to capable browsers and a transpiled bundle only where it is actually needed.

Note: Prefer defer (or type="module", which defers by default) over placing scripts at the end of <body> — you get parallel download with in-order, after-parse execution. Reserve async for scripts with no DOM or ordering dependency, such as analytics beacons.

Note: A preload without a matching as is fetched at the wrong priority or downloaded twice, and an unused preload wastes bandwidth and triggers a console warning. Use <base> sparingly: because it rewrites every relative URL and default link target on the page, one <base> tag can silently break in-page fragment links and relative assets.

Native drag & drop

The HTML Drag and Drop API moves data with the pointer. Links and images are draggable by default; any other element becomes draggable with the draggable="true" attribute. A drag fires a sequence of events on the source and target: dragstart (set the payload), dragenter/dragover on a potential target, dragleave when leaving it, drop when released, and dragend to clean up. The counter-intuitive but essential rule is that an element is not a valid drop target until you call preventDefault() in its dragover handler — the default action is to reject the drop.

Data travels on the dataTransfer object: setData(type, value) in dragstart and getData(type) in drop, with effectAllowed and dropEffect hinting whether this is a copy or move. Files dragged from the desktop arrive in dataTransfer.files (a pointer into the File API). The major limitation is accessibility: native drag and drop is not keyboard-operable, and screen readers expose it poorly, so you must always provide an equivalent that works without a mouse — move buttons, a cut/paste command, or a form control.
