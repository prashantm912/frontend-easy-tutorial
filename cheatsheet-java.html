1. Collections & Maps

Full guide: Java Collections →

Collection hierarchy List/Set/Queue/Map

Collection is the root of List (ordered, indexed, duplicates), Set (unique), and Queue (FIFO ends). Map is a separate root: key -> value with unique keys.

List<String> list = new ArrayList<>();      // ordered, duplicates OK
Set<String> set = new HashSet<>();          // unique, no index
Queue<String> q = new LinkedList<>();        // FIFO: offer / poll
Map<String,Integer> map = new HashMap<>();   // key -> value, unique keys
list.add("a"); list.add("a");                // duplicate kept
set.add("a");  set.add("a");                 // duplicate ignored
System.out.println(list.size() + " " + set.size());
Output2 1

ArrayList ArrayList<E>

Resizable array backing a List: O(1) random access by index and amortized O(1) append; inserting/removing in the middle is O(n) (elements shift).

List<String> names = new ArrayList<>();
names.add("Ann");            // append
names.add("Bob");
names.add(1, "Cid");         // insert at index 1
System.out.println(names.get(0));   // O(1) random access
System.out.println(names);
names.remove("Cid");         // remove by value
System.out.println(names.size());
OutputAnn [Ann, Cid, Bob] 2

LinkedList as Deque addFirst/poll

Doubly-linked list implementing both List and Deque: O(1) add/remove at either end (use as queue, stack, or deque). Random access by index is O(n).

Deque<Integer> dq = new LinkedList<>();
dq.addFirst(1);              // [1]
dq.addLast(2);              // [1, 2]
dq.addFirst(0);              // [0, 1, 2]
System.out.println(dq.peekFirst() + " " + dq.peekLast());
System.out.println(dq.poll());   // removes head (0)
System.out.println(dq);
Output0 2 0 [1, 2]

HashMap put/get/merge

Hash-table Map with average O(1) put/get; allows one null key. Iteration order is UNSPECIFIED (do not rely on it). computeIfAbsent/merge update atomically per key.

Map<String,Integer> ages = new HashMap<>();
ages.put("Ann", 30);
ages.put("Bob", 25);
System.out.println(ages.get("Ann"));              // 30
System.out.println(ages.getOrDefault("Zoe", -1)); // key absent
ages.computeIfAbsent("Cid", k -> 0);              // insert if missing
ages.merge("Ann", 1, Integer::sum);               // 30 + 1
System.out.println(ages.get("Ann"));
Output30 -1 31

LinkedHashMap insertion order

A HashMap that keeps a linked list of entries, so iteration follows INSERTION order (predictable). An access-order variant enables simple LRU caches.

Map<String,Integer> m = new LinkedHashMap<>();
m.put("c", 3);
m.put("a", 1);
m.put("b", 2);
System.out.println(m);                 // insertion order kept
m.forEach((k, v) -> System.out.print(k));
Output{c=3, a=1, b=2} cab

TreeMap NavigableMap

Red-black tree Map sorted by key (natural order or a Comparator); O(log n) ops. As a NavigableMap it adds firstKey, floorKey, and range views like subMap.

NavigableMap<Integer,String> tm = new TreeMap<>();
tm.put(10, "ten"); tm.put(20, "twenty"); tm.put(30, "thirty");
System.out.println(tm.firstKey());     // smallest key
System.out.println(tm.floorKey(25));   // largest key <= 25
System.out.println(tm.subMap(10, 30)); // [10, 30)
Output10 20 {10=ten, 20=twenty}

HashSet / LinkedHashSet / TreeSet Set

All enforce uniqueness. HashSet has NO guaranteed order; LinkedHashSet keeps insertion order; TreeSet keeps elements sorted (O(log n), a NavigableSet).

Set<Integer> hs = new HashSet<>(List.of(3, 1, 2, 2));
Set<Integer> ls = new LinkedHashSet<>(List.of(3, 1, 2, 2));
Set<Integer> ts = new TreeSet<>(List.of(3, 1, 2, 2));
System.out.println(hs.size());   // duplicate 2 dropped
System.out.println(ls);          // insertion order
System.out.println(ts);          // sorted
Output3 [3, 1, 2] [1, 2, 3]

Iterating a collection Iterator/entrySet

Enhanced for for read-only passes; use Iterator.remove() to delete safely mid-iteration (avoids ConcurrentModificationException); iterate a map via entrySet().

List<Integer> nums = new ArrayList<>(List.of(1, 2, 3, 4));
Iterator<Integer> it = nums.iterator();
while (it.hasNext()) {
    if (it.next() % 2 == 0) it.remove();   // safe removal
}
System.out.println(nums);
Map<String,Integer> m = Map.of("a", 1);
for (Map.Entry<String,Integer> e : m.entrySet())
    System.out.println(e.getKey() + "=" + e.getValue());
Output[1, 3] a=1

Comparable vs Comparator compareTo/comparing

Comparable<T> defines ONE natural order via compareTo inside the class; Comparator<T> is an external, composable order built with comparing, thenComparing, and reversed.

record Person(String name, int age) {}
List<Person> people = new ArrayList<>(List.of(
    new Person("Ann", 30), new Person("Bob", 25), new Person("Ann", 20)));
people.sort(Comparator.comparing(Person::name)
                      .thenComparingInt(Person::age));  // .reversed() flips
people.forEach(p -> System.out.println(p.name() + " " + p.age()));
OutputAnn 20 Ann 30 Bob 25

Collections utility Collections.*

Static helpers in java.util.Collections: sort, reverse, max/min, frequency, plus read-only wrappers (unmodifiableList) and empties (emptyList).

List<Integer> xs = new ArrayList<>(List.of(3, 1, 2));
Collections.sort(xs);            // in place -> [1, 2, 3]
Collections.reverse(xs);         // [3, 2, 1]
System.out.println(Collections.max(xs));
System.out.println(xs);
List<Integer> ro = Collections.unmodifiableList(xs);
// ro.add(9) throws UnsupportedOperationException
System.out.println(Collections.emptyList());
Output3 [3, 2, 1] []

Arrays utility & array vs List Arrays.*

java.util.Arrays: sort, toString, stream, and asList. asList is a FIXED-SIZE view (add/remove throw); wrap in new ArrayList<>(...) for a growable copy.

int[] a = {3, 1, 2};
Arrays.sort(a);                        // sorts in place
System.out.println(Arrays.toString(a));
int sum = Arrays.stream(a).sum();
System.out.println(sum);
List<String> fixed = Arrays.asList("x", "y");
// fixed.add("z") throws UnsupportedOperationException (fixed size)
System.out.println(fixed.get(1));
Output[1, 2, 3] 6 y

equals() / hashCode() contract Object

Hash-based keys need both: equal objects MUST return equal hashCode. Override the two together, or HashMap/HashSet lookups silently miss. Records generate both from their components.

record Point(int x, int y) {}          // auto equals + hashCode
Set<Point> set = new HashSet<>();
set.add(new Point(1, 2));
System.out.println(set.contains(new Point(1, 2))); // equal + same hash
Map<Point,String> m = new HashMap<>();
m.put(new Point(0, 0), "origin");
System.out.println(m.get(new Point(0, 0)));
Outputtrue origin

Concurrent collections ConcurrentHashMap

Thread-safe without external locking: ConcurrentHashMap (lock-free reads, atomic merge), CopyOnWriteArrayList (read-heavy, snapshot iteration), BlockingQueue (producer/consumer). Prefer these over Collections.synchronizedList, which locks the whole list and still needs manual sync while iterating.

Map<String,Integer> counts = new ConcurrentHashMap<>();
counts.merge("hit", 1, Integer::sum);  // atomic, lock-free reads
counts.merge("hit", 1, Integer::sum);
System.out.println(counts.get("hit"));

List<String> log = new CopyOnWriteArrayList<>();
log.add("start");                      // copies array on write
System.out.println(log.get(0));

BlockingQueue<String> q = new ArrayBlockingQueue<>(4);
q.offer("job");                        // put(...) blocks if full
System.out.println(q.poll());          // take() blocks if empty
Output2 start job

2. Multithreading & Concurrency

Full guide: Java Multithreading →

Creating threads Runnable

Subclass Thread or (preferred) pass a Runnable/lambda. start() spawns a new thread; run() just calls the method on the current thread. join() waits for a thread to finish.

Runnable task = () -> System.out.println("run: " + Thread.currentThread().getName());
Thread t = new Thread(task, "worker");
t.start();      // spawns a NEW thread, runs task concurrently
t.join();       // main blocks here until 'worker' finishes
task.run();     // NO new thread -- runs on the caller (main)
System.out.println("done on " + Thread.currentThread().getName());
Outputrun: worker run: main done on main

synchronized synchronized

Acquires an object's intrinsic monitor lock so only one thread runs the guarded code at a time (mutual exclusion). A synchronized method locks this; a block locks any chosen object.

class Counter {
    private int n = 0;
    synchronized void inc() { n++; }                 // locks 'this'
    void incBlock() { synchronized (this) { n++; } } // block form
    synchronized int get() { return n; }
}
Counter c = new Counter();
Runnable job = () -> { for (int i = 0; i < 1000; i++) c.inc(); };
Thread a = new Thread(job), b = new Thread(job);
a.start(); b.start(); a.join(); b.join();
System.out.println(c.get());   // no lost updates
Output2000

volatile volatile

Guarantees a field's writes are visible to other threads immediately (and prevents reordering) -- but NOT atomicity. Fine for a stop-flag; use atomics/locks for compound updates like count++.

class Flag { volatile boolean running = true; }   // visible across threads
Flag f = new Flag();
Thread w = new Thread(() -> {
    while (f.running) { /* busy wait */ }
    System.out.println("stopped");
});
w.start();
Thread.sleep(50);
f.running = false;   // volatile write -> the loop sees it and exits
w.join();
System.out.println("main done");
Outputstopped main done

wait / notify wait()/notifyAll()

Thread coordination on an object's monitor: call inside synchronized; wait() releases the lock and sleeps until notify()/notifyAll(). Always re-check the condition in a while loop (guards spurious wakeups).

class Box {
    private String msg;
    synchronized void put(String m) { msg = m; notifyAll(); }
    synchronized String take() throws InterruptedException {
        while (msg == null) wait();   // loop, not if
        return msg;
    }
}
Box box = new Box();
Thread c = new Thread(() -> {
    try { System.out.println("got: " + box.take()); }
    catch (InterruptedException e) { }
});
c.start();
Thread.sleep(50);
box.put("hello");   // wakes the waiting consumer
c.join();
Outputgot: hello

ReentrantLock ReentrantLock

Explicit lock -- more flexible than synchronized: tryLock() (non-blocking/timed), interruptible acquisition, and optional fairness (FIFO). Always release in a finally.

ReentrantLock lock = new ReentrantLock(true);  // true = fair (FIFO)
int[] n = {0};
Runnable job = () -> {
    for (int i = 0; i < 500; i++) {
        lock.lock();                 // blocking, reentrant
        try { n[0]++; }              // critical section
        finally { lock.unlock(); }   // MUST unlock in finally
    }
};
Thread a = new Thread(job), b = new Thread(job);
a.start(); b.start(); a.join(); b.join();
System.out.println(n[0]);
Output1000

ReadWriteLock ReadWriteLock

Splits locking into a shared read lock (many readers at once) and an exclusive write lock (one writer, blocks all readers). Ideal for read-heavy shared state.

ReadWriteLock rw = new ReentrantReadWriteLock();
Map<String,Integer> map = new HashMap<>();
rw.writeLock().lock();               // exclusive: no other reader or writer
try { map.put("a", 1); }
finally { rw.writeLock().unlock(); }
rw.readLock().lock();                // shared: readers run concurrently
try { System.out.println(map.get("a")); }
finally { rw.readLock().unlock(); }
Output1

Semaphore Semaphore

Holds a set of permits; acquire() takes one (blocking if none), release() returns one. Bounds how many threads may enter a section at once (e.g. a connection pool).

Semaphore sem = new Semaphore(2);   // at most 2 threads hold a permit
sem.acquire();                      // permits: 2 -> 1
sem.acquire();                      // permits: 1 -> 0
System.out.println(sem.availablePermits());
boolean got = sem.tryAcquire();     // none left -> false, does not block
System.out.println(got);
sem.release();                      // permits: 0 -> 1
System.out.println(sem.availablePermits());
Output0 false 1

CountDownLatch CountDownLatch

A one-shot gate initialized to N: await() blocks until countDown() has been called N times. Used to wait for several tasks to finish (or for a start signal).

CountDownLatch latch = new CountDownLatch(3);
for (int i = 1; i <= 3; i++) {
    int id = i;
    new Thread(() -> {
        System.out.println("worker " + id + " done");
        latch.countDown();          // N -> N-1
    }).start();
}
latch.await();                      // blocks until count reaches 0
System.out.println("all workers finished");
Outputworker 1 done worker 3 done worker 2 done all workers finished // the 3 worker lines print in any order; final line is always last

ExecutorService newFixedThreadPool

A managed thread pool: submit() queues tasks onto reused threads (not a new Thread per task). shutdown() stops accepting work and drains the queue.

ExecutorService pool = Executors.newFixedThreadPool(2);
for (int i = 1; i <= 3; i++) {
    int id = i;
    pool.submit(() -> System.out.println(
        "task " + id + " on " + Thread.currentThread().getName()));
}
pool.shutdown();                              // no new tasks accepted
pool.awaitTermination(1, TimeUnit.SECONDS);   // wait for queued tasks
Outputtask 1 on pool-1-thread-1 task 2 on pool-1-thread-2 task 3 on pool-1-thread-1 // task order and which thread runs each: scheduler-dependent

Callable & Future Future<V>

A Callable returns a value (and may throw); submit() hands back a Future whose get() blocks until the result is ready. For non-blocking pipelines use CompletableFuture.

ExecutorService pool = Executors.newSingleThreadExecutor();
Future<Integer> f = pool.submit(() -> {   // Callable<Integer>
    int sum = 0;
    for (int i = 1; i <= 5; i++) sum += i;
    return sum;
});
System.out.println(f.get());   // blocks until the task returns
pool.shutdown();
// async, no blocking get():
CompletableFuture.supplyAsync(() -> 21).thenApply(x -> x * 2);  // -> 42
Output15

Atomics & CAS AtomicInteger

Lock-free thread-safe updates built on the CPU's compare-and-swap (CAS). incrementAndGet(), getAndAdd(), and compareAndSet(expected,new) avoid locks for single-variable state.

AtomicInteger counter = new AtomicInteger(0);
Runnable job = () -> { for (int i = 0; i < 1000; i++) counter.incrementAndGet(); };
Thread a = new Thread(job), b = new Thread(job);
a.start(); b.start(); a.join(); b.join();
System.out.println(counter.get());              // atomic, no lock
boolean ok = counter.compareAndSet(2000, 0);    // set 0 only if still 2000
System.out.println(ok + " " + counter.get());
Output2000 true 0

ConcurrentHashMap ConcurrentHashMap

A thread-safe map with high concurrency and no external locking. merge(), compute(), and computeIfAbsent() are atomic read-modify-write ops. Null keys/values are not allowed.

ConcurrentHashMap<String,Integer> counts = new ConcurrentHashMap<>();
Runnable job = () -> {
    for (int i = 0; i < 1000; i++)
        counts.merge("hits", 1, Integer::sum);   // atomic increment
};
Thread a = new Thread(job), b = new Thread(job);
a.start(); b.start(); a.join(); b.join();
System.out.println(counts.get("hits"));
Output2000

Deadlock lock ordering

Two threads each hold a lock the other needs and wait forever. Prevent it by acquiring locks in one consistent global order. Because scheduling is non-deterministic, a deadlock-prone program may run fine on some runs and hang on others.

// DEADLOCK: opposite lock orders can freeze both threads
// t1: synchronized (lockA) { synchronized (lockB) { ... } }
// t2: synchronized (lockB) { synchronized (lockA) { ... } }

// FIX -- global ordering: EVERY thread takes lockA before lockB
synchronized (lockA) {
    synchronized (lockB) {
        // ... work needing both locks
    }
}
OutputFour Coffman conditions (all required); break any one to avoid deadlock: mutual exclusion | hold-and-wait | no preemption | circular wait

Virtual threads Thread.ofVirtual()

Java 21 (JEP 444): lightweight threads scheduled by the JVM onto a few OS carrier threads. Blocking is cheap, so you can run millions -- ideal for blocking IO. Don't pool them; use one per task.

Thread vt = Thread.ofVirtual().start(() ->
    System.out.println("on " + Thread.currentThread()));
vt.join();

// one virtual thread per task:
try (var pool = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 3; i++) {
        int id = i;
        pool.submit(() -> System.out.println("task " + id));
    }
}   // close() waits for all tasks (ExecutorService is AutoCloseable)
Outputon VirtualThread[#21]/runnable@ForkJoinPool-1-worker-1 task 0 task 2 task 1 // task order scheduler-dependent; the VirtualThread id/worker vary

3. Functional Programming

Full guide: Java Functional Programming →

Lambda expression () -> ...

An anonymous function passed as a value (Java 8). It supplies the single abstract method of a functional interface, replacing a verbose anonymous class. Three forms: no params, one param, or a block body.

Runnable r = () -> System.out.println("Hi");      // no params
Function<Integer,Integer> sq = x -> x * x;         // one param, expr body
Comparator<String> byLen = (a, b) -> {             // block body + return
    return a.length() - b.length();
};
r.run();
System.out.println(sq.apply(5));
OutputHi 25

Functional interface @FunctionalInterface

An interface with exactly one abstract method (a SAM type) is the target of a lambda or method reference. The optional @FunctionalInterface makes the compiler enforce the one-abstract-method rule; default/static methods do not count.

@FunctionalInterface
interface Calculator {
    int operate(int a, int b);          // the single abstract method
    default int zero() { return 0; }    // allowed, does not count
}

Calculator add = (a, b) -> a + b;       // lambda supplies operate()
System.out.println(add.operate(3, 4));
Output7

Predicate<T> test()

From java.util.function: maps a value to a boolean via test(t). Compose with the default methods and, or, and negate.

Predicate<Integer> isEven = n -> n % 2 == 0;
Predicate<Integer> isPositive = n -> n > 0;

System.out.println(isEven.test(4));                  // true
System.out.println(isEven.and(isPositive).test(-4)); // even but not >0
System.out.println(isEven.negate().test(3));         // not even -> true
Outputtrue false true

Function<T,R> apply()

Transforms a T into an R via apply(t). f.andThen(g) runs f first, then g; f.compose(g) runs g first, then f.

Function<Integer,Integer> times2 = x -> x * 2;
Function<Integer,Integer> plus3  = x -> x + 3;

System.out.println(times2.andThen(plus3).apply(5)); // (5*2)+3
System.out.println(times2.compose(plus3).apply(5)); // (5+3)*2
Output13 16

Consumer<T> & Supplier<T> accept() / get()

Consumer<T> takes a value and returns nothing (a side effect) via accept. Supplier<T> takes no input and produces a value on demand via get.

Consumer<String> printer = s -> System.out.println("Hi " + s);
Supplier<Double> random  = () -> 0.42;   // supplies a value

printer.accept("Ranga");
System.out.println(random.get());
OutputHi Ranga 0.42

Method references Type::method

Shorthand for a lambda that just calls one method. Four kinds: static (Type::m), bound instance (obj::m), unbound instance (Type::instanceM), and constructor (Type::new).

Function<String,Integer> parse = Integer::parseInt;  // static
Supplier<String> greet = "hello"::toUpperCase;       // bound obj::m
Function<String,Integer> len = String::length;       // unbound
Supplier<ArrayList<String>> make = ArrayList::new;    // constructor

System.out.println(parse.apply("42"));
System.out.println(greet.get());
System.out.println(len.apply("abc"));
System.out.println(make.get());
Output42 HELLO 3 []

Creating a Stream .stream()

A Stream is a lazy, single-use pipeline over a source. Build one from a collection (.stream()), from values (Stream.of), or from an array (Arrays.stream, which yields a primitive IntStream for int[]).

List<Integer> list = List.of(1, 2, 3);
Stream<Integer> s1 = list.stream();
Stream<String>  s2 = Stream.of("a", "b", "c");
int[] arr = {10, 20, 30};
IntStream s3 = Arrays.stream(arr);   // primitive stream

System.out.println(s1.count());
System.out.println(s3.sum());
Output3 60

filter + map filter / map

Both are lazy intermediate ops: filter(predicate) keeps matching elements, map(fn) transforms each one. Nothing runs until a terminal op (here forEach) pulls the pipeline.

List<Integer> nums = List.of(1, 2, 3, 4, 5, 6);
nums.stream()
    .filter(n -> n % 2 == 0)   // keep evens: 2, 4, 6
    .map(n -> n * n)           // square each: 4, 16, 36
    .forEach(System.out::println);
Output4 16 36

sorted / distinct / limit / skip

Intermediate ops: distinct drops duplicates (by equals), sorted orders (natural or by a Comparator), skip(n) discards the first n, limit(n) truncates to n.

Stream.of(5, 3, 5, 1, 4, 2, 3)
      .distinct()          // 5, 3, 1, 4, 2
      .sorted()            // 1, 2, 3, 4, 5
      .skip(1)             // drop first -> 2, 3, 4, 5
      .limit(2)            // keep 2     -> 2, 3
      .forEach(System.out::println);
Output2 3

reduce reduce(identity, acc)

Terminal op that folds a stream into one value. The identity is the seed (and the result for an empty stream); the accumulator combines the running result with each element.

List<Integer> nums = List.of(12, 9, 13, 4, 6);
int sum = nums.stream()
              .reduce(0, (a, b) -> a + b);   // 0+12+9+13+4+6, or Integer::sum
System.out.println(sum);
Output44

collect + Collectors Collectors

The terminal collect gathers a stream into a container using a Collectors factory: toList, joining, groupingBy (key -> list), partitioningBy (true/false split), counting.

List<String> words = List.of("apple", "banana", "avocado", "cherry");

String joined = words.stream().collect(Collectors.joining(", "));
Map<Character,List<String>> byInitial =
    words.stream().collect(Collectors.groupingBy(w -> w.charAt(0)));
Map<Boolean,List<String>> parts =
    words.stream().collect(Collectors.partitioningBy(w -> w.length() > 5));

System.out.println(joined);
System.out.println(byInitial);
System.out.println(parts);
Outputapple, banana, avocado, cherry {a=[apple, avocado], b=[banana], c=[cherry]} {false=[apple], true=[banana, avocado, cherry]} // groupingBy returns a HashMap -- key order is not guaranteed

Optional<T> Optional<T>

A container that may hold a value or be empty (Java 8), used to avoid null. Create with of / ofNullable / empty; read with isPresent, orElse, orElseGet, map, ifPresent.

Optional<String> name = Optional.of("Ranga");
Optional<String> none = Optional.ofNullable(null);    // -> empty

System.out.println(name.isPresent());                // true
System.out.println(none.orElse("default"));          // eager fallback
System.out.println(name.map(String::length).get());  // 5
name.ifPresent(System.out::println);                 // runs only if present
System.out.println(none.orElseGet(() -> "lazy"));     // supplier fallback
Outputtrue default 5 Ranga lazy

Numeric streams IntStream

Primitive streams (IntStream/LongStream/DoubleStream) avoid boxing and add numeric ops. range/rangeClosed generate; mapToInt converts an object stream; sum/average/summaryStatistics aggregate (average returns an OptionalDouble).

int sum = IntStream.range(1, 5).sum();             // 1+2+3+4 (5 excluded)
double avg = IntStream.rangeClosed(1, 5)
                      .average().getAsDouble();    // (1..5)/5

List<String> words = List.of("a", "bb", "ccc");
IntSummaryStatistics st = words.stream()
    .mapToInt(String::length).summaryStatistics();

System.out.println(sum);
System.out.println(avg);
System.out.println(st.getMax() + " " + st.getSum());
Output10 3.0 3 6

Parallel streams .parallel()

Split work across the fork/join common pool via collection.parallelStream() or stream().parallel(). Only worth it for large, CPU-bound, stateless, associative work. A reduce/count result stays correct, but element/print order is not guaranteed.

long count = IntStream.rangeClosed(1, 1_000_000)
                      .parallel()
                      .filter(n -> n % 2 == 0)
                      .count();
System.out.println(count);              // deterministic

List.of(1, 2, 3, 4).parallelStream()
    .forEach(System.out::println);      // order NOT guaranteed
Output500000 // the forEach prints 1..4 in an unspecified, run-varying order

4. Modern Java (8 → 25)

Full guide: Modern Java →

Local-variable type inference var

Java 10: var infers a local's type from its initializer. Locals and for-loop variables only -- never fields, method params, or return types, and the initializer is required.

var names = new ArrayList<String>();  // inferred ArrayList<String>
names.add("Ann");
var count = names.size();             // inferred int
for (var n : names) {                 // n inferred String
    System.out.println(n + ":" + count);
}
OutputAnn:1

Records record Point(int x, int y){}

Java 16: an immutable data carrier. The compiler generates the canonical constructor, one accessor per component, plus equals, hashCode, and toString.

record Point(int x, int y) {}

Point p = new Point(3, 4);
System.out.println(p.x() + "," + p.y());       // accessors
System.out.println(p);                         // toString
System.out.println(p.equals(new Point(3, 4))); // value equality
Output3,4 Point[x=3, y=4] true

Sealed classes sealed ... permits

Java 17: restricts which types may extend/implement. Every permitted subtype must be declared final, sealed, or non-sealed (records are implicitly final).

sealed interface Shape permits Circle, Square {}

5. SOLID Principles

Full guide: SOLID & Design Patterns →

Single Responsibility (SRP) class

A class should have only one reason to change. Split unrelated responsibilities (data, printing, persistence) into separate classes.

// One class, one reason to change: separate data, output, storage.
class Invoice {                    // reason to change: tax / business rules
    final double amount;
    Invoice(double amount) { this.amount = amount; }
    double withTax() { return amount * 1.2; }
}
class InvoicePrinter {             // reason to change: output format
    void print(Invoice inv) { System.out.println("Total: " + inv.withTax()); }
}
class InvoiceRepository {          // reason to change: persistence
    void save(Invoice inv) { /* insert into DB */ }
}
BenefitThree independent reasons to change live in three classes -- a format tweak touches only InvoicePrinter, never Invoice.

Open/Closed (OCP) implements

Entities should be open for extension but closed for modification. Add behaviour via new types (polymorphism), not by editing existing code with type checks.

interface Shape { double area(); }
record Circle(double r) implements Shape { public double area(){ return Math.PI*r*r; } }
record Square(double s) implements Shape { public double area(){ return s*s; } }

// total() is closed for modification, open for extension:
static double total(List<Shape> shapes) {
    return shapes.stream().mapToDouble(Shape::area).sum();
}
// BAD: if (s instanceof Circle) ... else if (s instanceof Square) ...
//      -- editing total() every time a new shape type appears.
BenefitAdding a Triangle = one new record. total(), Circle and Square need zero edits.

Liskov Substitution (LSP) extends

Subtypes must be substitutable for their base type without breaking a caller's expectations -- each subtype honours the base contract.

interface Shape { int area(); }
record Rectangle(int w, int h) implements Shape { public int area(){ return w*h; } }
record Square(int side)        implements Shape { public int area(){ return side*side; } }

// Any Shape substitutes for another: area() means the same everywhere.
Shape s = new Square(4);
System.out.println(s.area());
// BAD: class Square extends Rectangle overriding setWidth to also set height
//      breaks callers that set width and height independently.
Output16

Interface Segregation (ISP) interface

Prefer many small, role-specific interfaces over one fat interface, so no client is forced to depend on methods it does not use.

interface Printer { void print(); }
interface Scanner { void scan();  }

// A print-only client depends on Printer alone.
class SimplePrinter implements Printer {
    public void print() { System.out.println("printing"); }
}
class AllInOne implements Printer, Scanner {
    public void print() { System.out.println("printing"); }
    public void scan()  { System.out.println("scanning"); }
}
// BAD: one fat interface Machine { print(); scan(); fax(); }
//      forces SimplePrinter to stub scan() and fax() it never calls.
BenefitSimplePrinter implements only print() -- no empty scan()/fax() stubs forced on it.

Dependency Inversion (DIP) constructor

High-level modules should depend on abstractions, not concrete classes; inject the concretion. Dependency Injection is one form of Inversion of Control (IoC).

interface MessageService { void send(String msg); }
class EmailService implements MessageService {
    public void send(String msg) { System.out.println("Email: " + msg); }
}
class Notifier {                             // high-level: depends on abstraction
    private final MessageService service;    // not `new EmailService()` inside
    Notifier(MessageService service) { this.service = service; }   // injected
    void notifyUser(String msg) { service.send(msg); }
}
// usage: new Notifier(new EmailService()).notifyUser("hi");
// swap in SmsService without touching Notifier.
OutputEmail: hi

6. Creational Patterns

Full guide: SOLID & Design Patterns →

Singleton enum INSTANCE

Creational: guarantee exactly ONE instance with a global access point. Participant is the class itself -- private constructor plus a static accessor. The one-line enum is the safest form (serialization- and reflection-proof).

enum Logger { INSTANCE;                     // enum = safest singleton
    void log(String m) { System.out.println(m); }
}
final class Config {
    private static final Config I = new Config();
    private Config() {}                     // private ctor blocks 'new'
    static Config get() { return I; }
}
// usage
System.out.println(Config.get() == Config.get());
Logger.INSTANCE.log("started");
Outputtrue started

Factory Method make()

Creational: a Creator declares a factory method that subclasses OVERRIDE to decide which single Product to instantiate; callers never call new on a concrete class. Participants: Product, ConcreteProducts, Creator, ConcreteCreators.

interface Shape { double area(); }
record Circle(double r) implements Shape { public double area(){ return Math.PI*r*r; } }
record Square(double s) implements Shape { public double area(){ return s*s; } }

abstract class Factory {                     // Creator
    abstract Shape create();                  // factory method: subclasses override
}
class CircleFactory extends Factory { Shape create(){ return new Circle(1); } }
class SquareFactory extends Factory { Shape create(){ return new Square(2); } }

for (Factory f : List.of(new CircleFactory(), new SquareFactory()))
    System.out.printf("%.2f%n", f.create().area());
Output3.14 4.00

Abstract Factory Theme

Creational: a factory of factories -- one object creates a whole FAMILY of related products (e.g. a matching Button + Checkbox) without naming their concrete classes. Participants: AbstractFactory, ConcreteFactory, AbstractProducts.

interface Button   { String paint(); }     // two related products
interface Checkbox { String paint(); }
interface Theme {                           // abstract factory (a family)
    Button   button();
    Checkbox checkbox();
}
class Dark implements Theme {               // one concrete family
    public Button   button()   { return () -> "[dark button]"; }
    public Checkbox checkbox() { return () -> "[dark check]"; }
}
Theme t = new Dark();
System.out.println(t.button().paint() + " " + t.checkbox().paint());
Output[dark button] [dark check]

Builder .build()

Creational: assemble a complex/immutable object step by step through a fluent builder, avoiding telescoping constructors. Participants: Product plus Builder (fluent setters that return this, ending in build()).

final class Pizza {
    final int size; final boolean cheese, bacon;
    private Pizza(Builder b){ size=b.size; cheese=b.cheese; bacon=b.bacon; }
    static class Builder {                            // no telescoping ctors
        int size = 12; boolean cheese, bacon;
        Builder size(int s){ this.size = s; return this; }  // fluent: return this
        Builder cheese(){ cheese = true; return this; }
        Builder bacon(){ bacon = true; return this; }
        Pizza build(){ return new Pizza(this); }
    }
}
Pizza p = new Pizza.Builder().size(16).cheese().build();
System.out.println(p.size + " cheese=" + p.cheese + " bacon=" + p.bacon);
Output16 cheese=true bacon=false

Prototype clone()

Creational: create new objects by CLONING a prototype instead of constructing from scratch. Caveat: Object.clone() is a SHALLOW copy -- nested references are shared, so copy mutable fields yourself for a deep copy. Participants: Prototype, ConcretePrototype.

class Sheet implements Cloneable {
    List<String> rows = new ArrayList<>(List.of("a", "b"));
    public Sheet clone() {                     // shallow: shares the SAME list
        try { return (Sheet) super.clone(); }
        catch (CloneNotSupportedException e) { throw new AssertionError(e); }
    }
    Sheet deepClone() {                        // deep: copy the mutable field too
        Sheet c = clone();
        c.rows = new ArrayList<>(this.rows);
        return c;
    }
}
Sheet a = new Sheet();
Sheet shallow = a.clone();
shallow.rows.add("c");                         // leaks: shared list mutates 'a'
Sheet deep = a.deepClone();
deep.rows.add("d");                            // isolated: own copy, 'a' untouched
System.out.println(a.rows + " | " + shallow.rows + " | " + deep.rows);
Output[a, b, c] | [a, b, c] | [a, b, c, d]

7. Behavioral Patterns

Full guide: SOLID & Design Patterns →

Strategy interface + delegate

Behavioral (GoF). Encapsulates a family of interchangeable algorithms behind one interface; the Context (Cart) holds a Strategy reference and delegates to it. A one-method strategy is a functional interface, so a lambda can replace a concrete class.

interface Discount { double apply(double price); }

class Cart {                       // Context
    private final Discount strategy;
    Cart(Discount s) { this.strategy = s; }
    double checkout(double price) { return strategy.apply(price); }
}

// lambda supplies the algorithm -- no concrete class needed
Cart c = new Cart(p -> p * 0.9);   // 10% off
System.out.println(c.checkout(100));
Output90.0

Observer register / notify

Behavioral (GoF). Defines a one-to-many dependency: the Subject keeps a list of Observers and notifies each when its state changes. Notification follows registration order (the list is iterated front to back).

interface Observer { void update(int temp); }

class Station {                       // Subject
    private final List<Observer> obs = new ArrayList<>();
    void register(Observer o) { obs.add(o); }
    void setTemp(int t) { for (Observer o : obs) o.update(t); }
}

Station s = new Station();
s.register(t -> System.out.println("Phone: " + t));
s.register(t -> System.out.println("TV: " + t));
s.setTemp(25);
OutputPhone: 25 TV: 25

Command execute()

Behavioral (GoF). Wraps a request as an object exposing execute(); the Invoker holds Commands, not Receivers, so requests can be queued, logged, or undone. Participants: Command, ConcreteCommand, Receiver, Invoker.

interface Command { void execute(); }

class Light { void on() { System.out.println("Light ON"); } }  // Receiver

class OnCommand implements Command {
    private final Light light;
    OnCommand(Light l) { this.light = l; }
    public void execute() { light.on(); }
}

Command c = new OnCommand(new Light());  // invoker sees only Command
c.execute();
OutputLight ON

Iterator hasNext() / next()

Behavioral (GoF). Gives sequential access to an aggregate's elements without exposing its internals. Java's java.util.Iterator is the built-in form; every Iterable returns one via iterator().

List<String> names = List.of("Ann", "Bob", "Cy");

Iterator<String> it = names.iterator();
while (it.hasNext()) {
    System.out.println(it.next());
}
OutputAnn Bob Cy

Template Method abstract steps

Behavioral (GoF). A base-class method fixes the algorithm's skeleton and calls abstract steps that subclasses fill in (inheritance-based). The template is usually final so the outline can't be overridden.

abstract class Game {
    final void play() {          // template method: fixed skeleton
        start();
        finish();
    }
    abstract void start();
    abstract void finish();
}

class Chess extends Game {
    void start()  { System.out.println("Chess start"); }
    void finish() { System.out.println("Chess over"); }
}

new Chess().play();
OutputChess start Chess over

Null Object neutral default

Behavioral pattern popularized after the GoF book (not one of the original 23). Supplies a do-nothing / neutral implementation of an interface instead of null, so callers can drop their null checks.

interface Customer { String name(); }

class Real implements Customer {
    public String name() { return "Ann"; }
}
class NullCustomer implements Customer {
    public String name() { return "Guest"; }   // neutral, never null
}

Customer find(boolean known) {
    return known ? new Real() : new NullCustomer();
}

System.out.println(find(false).name());         // no null check
OutputGuest

Visitor accept / visit

Behavioral (GoF). Moves an operation out of the object structure: each element's accept(v) calls back v.visit(this) (double dispatch). Adding a new operation is easy (a new Visitor); adding a new element type is hard (every Visitor must change).

interface Shape { double accept(AreaVisitor v); }
interface AreaVisitor { double visit(Circle c); double visit(Square s); }

record Circle(double r) implements Shape {
    public double accept(AreaVisitor v) { return v.visit(this); }
}
record Square(double side) implements Shape {
    public double accept(AreaVisitor v) { return v.visit(this); }
}

AreaVisitor area = new AreaVisitor() {
    public double visit(Circle c) { return Math.PI * c.r() * c.r(); }
    public double visit(Square s) { return s.side() * s.side(); }
};
System.out.println(new Square(2).accept(area));
Output4.0

Decorator wrap + delegate

Attach responsibilities to an object dynamically by wrapping it in another object of the same interface (Component) that holds a reference to the inner one and delegates — a flexible alternative to subclassing. JDK: the java.io streams, e.g. new BufferedReader(new FileReader(f)).

interface Coffee { double cost(); }
class Espresso implements Coffee {
    public double cost() { return 2.0; }
}
// decorator implements Coffee AND wraps a Coffee
class Milk implements Coffee {
    private final Coffee inner;
    Milk(Coffee inner) { this.inner = inner; }
    public double cost() { return inner.cost() + 0.5; }
}
Coffee c = new Milk(new Milk(new Espresso()));  // stack decorators
System.out.println(c.cost());
Output3.0

Facade one entry point

Provide one simplified interface over a complex subsystem: the Facade knows which subsystem classes to call and in what order, so clients depend only on it. Reduces coupling; the subsystem stays usable directly. JDK-style: DriverManager.getConnection(...), SLF4J's LoggerFactory.

class Amp       { void on()  { System.out.println("Amp on"); } }
class Projector { void on()  { System.out.println("Projector on"); } }
class Lights    { void dim() { System.out.println("Lights dim"); } }

class TheaterFacade {                 // hides the 3-step startup
    private final Amp amp = new Amp();
    private final Projector proj = new Projector();
    private final Lights lights = new Lights();
    void watchMovie() { lights.dim(); proj.on(); amp.on(); }
}
new TheaterFacade().watchMovie();
OutputLights dim Projector on Amp on

Flyweight share intrinsic state

Share many fine-grained objects to save memory. State is split: intrinsic (shared, immutable, stored in the flyweight) vs extrinsic (context-specific, passed in by the client). A factory caches and reuses instances. JDK: the String pool and Integer.valueOf's cache of -128..127.

Integer a = 127, b = 127;   // autoboxing -> Integer.valueOf, cached
Integer c = 128, d = 128;   // 128 is outside the cache -> new objects
System.out.println(a == b); // same cached instance
System.out.println(c == d); // distinct instances
System.out.println(c.equals(d)); // value equality still holds
Outputtrue false true

Adapter Target wraps Adaptee

Convert an existing class's interface (Adaptee) into the interface a client expects (Target). The Adapter implements Target and delegates to the wrapped Adaptee. Object adapter uses composition (shown); a class adapter would extend the Adaptee. JDK: InputStreamReader adapts an InputStream to a Reader.

interface Target { String request(); }          // what the client calls
class Adaptee { String specificRequest() { return "raw data"; } }

class Adapter implements Target {                // wraps the incompatible API
    private final Adaptee a;
    Adapter(Adaptee a) { this.a = a; }
    public String request() { return a.specificRequest().toUpperCase(); }
}
Target t = new Adapter(new Adaptee());
System.out.println(t.request());
OutputRAW DATA

Bridge abstraction has-a impl

Decouple an abstraction from its implementation so both vary independently — the Abstraction has-a Implementor (composition) rather than subclassing it, avoiding an N x M class explosion. Unlike Adapter (retrofits unrelated interfaces) it is designed up front. JDK: the JDBC API (abstraction) over vendor Drivers (implementation).

interface Renderer { String render(String shape); }   // implementor
class Vector implements Renderer {
    public String render(String s) { return "Vector:" + s; }
}
abstract class Shape {                                  // abstraction
    protected final Renderer r;
    Shape(Renderer r) { this.r = r; }
    abstract String draw();
}
class Circle extends Shape {
    Circle(Renderer r) { super(r); }
    String draw() { return r.render("circle"); }
}
System.out.println(new Circle(new Vector()).draw());
OutputVector:circle

Composite Leaf + Composite

Compose objects into part-whole trees and treat a single object (Leaf) and a group (Composite) uniformly through one Component interface — the Composite holds a list of Components and recurses. JDK: Swing's Container/Component, a file-system tree.

interface Node { int size(); }                 // component
class FileLeaf implements Node {               // leaf
    private final int bytes;
    FileLeaf(int bytes) { this.bytes = bytes; }
    public int size() { return bytes; }
}
class Dir implements Node {                     // composite holds children
    private final List<Node> kids = new ArrayList<>();
    Dir add(Node n) { kids.add(n); return this; }
    public int size() {                        // recurse over the tree
        int total = 0;
        for (Node n : kids) total += n.size();
        return total;
    }
}
Node root = new Dir()
    .add(new FileLeaf(10))
    .add(new Dir().add(new FileLeaf(5)));
System.out.println(root.size());   // 10 + 5
Output15

9. System Design

Full guide: SOLID & Design Patterns →

REST APIs GET /users/42

Model the domain as resources addressed by URIs; map CRUD to HTTP verbs (create=POST, read=GET, update=PUT, delete=DELETE). Each request is stateless and the payload is usually JSON.

GET /users/42 HTTP/1.1
Host: api.example.com
Accept: application/json

HTTP/1.1 200 OK
Content-Type: application/json

{ "id": 42, "name": "Ada", "role": "admin" }
Safe vs idempotentGET safe + idempotent (no change, repeatable) PUT idempotent (repeat -> same final state) DELETE idempotent (repeat -> still deleted) POST NEITHER (repeat -> duplicate resource)

SOAP vs REST vs RPC SOAP / REST / gRPC

Three API styles: SOAP wraps calls in an XML envelope with a WSDL contract (heavyweight); REST is resource-oriented over HTTP/JSON; RPC/gRPC calls a remote procedure like a local method, action-oriented, using protobuf over HTTP/2.

SOAP  POST /users   <soap:Envelope><soap:Body>
                      <getUser><id>42</id></getUser>
                    </soap:Body></soap:Envelope>   (XML + WSDL)

REST  GET /users/42                                (resource + JSON)

gRPC  UserService.GetUser(GetUserRequest{ id: 42 })  (protobuf / HTTP2)
Trade-offsSOAP: strict WSDL contract + WS-* (security, transactions); verbose XML REST: simple, cacheable, human-readable, ubiquitous over HTTP gRPC: compact binary + bi-di streaming, fast; needs .proto, not readable

Monolith vs Microservices 1 vs N deploys

A monolith ships as one deployable over one shared DB; microservices are many small, independently deployable services that each own their data. Microservices trade simplicity for autonomy -- they add, not remove, complexity.

Monolith:
  one codebase -> one deployable (WAR/JAR), one shared DB
  [ UI | Orders | Billing | Users ]  run in a single process

Microservices:
  [Orders]  [Billing]  [Users]   -- separate services
  each: own repo, own DB, deployed + scaled independently
  communicate over REST / gRPC / async message queue
Honest trade-offsMonolith: easy to build/test/deploy; scales as ONE unit, teams coupled Microservices: independent deploy + scale + team autonomy BUT network calls, partial failure, distributed txns, eventual consistency, more ops. NOT automatically better -- a real trade-off.

Layered (n-tier) & SOA n-tier / SOA

Layering splits an app into presentation, business, and persistence tiers where each depends only on the one below. SOA composes coarse-grained enterprise services via a central ESB; microservices are the fine-grained, decentralized alternative.

Layered (n-tier)             each layer calls only downward
  Presentation  (REST controllers, UI)
        v
  Business      (services, domain rules)
        v
  Persistence   (repositories / DAO)  ->  Database

SOA           coarse services glued by a central ESB (enterprise bus)
Microservices fine-grained services, own DB, dumb pipes (no ESB)
Layering ruleA lower layer must never call up into a higher one. SOA: centralizes integration + reuse in a shared ESB (enterprise). Micro: decentralizes it -- own DB per service, lightweight messaging.

SQL vs NoSQL ACID vs BASE

Relational SQL uses fixed schemas, joins, and ACID transactions. NoSQL trades that for flexible schemas, BASE semantics, and horizontal scale -- in four families: key-value, document, column-family, and graph.

-- SQL: fixed schema, joins, ACID transaction
BEGIN;
  UPDATE accounts SET bal = bal - 100 WHERE id = 1;
  UPDATE accounts SET bal = bal + 100 WHERE id = 2;
COMMIT;

// NoSQL (document): schema-flexible, denormalized, no join
{ "id": 42, "name": "Ada", "orders": [ { "total": 19.99 } ] }
When each fitsSQL -> strong consistency, joins, transactions: orders, payments NoSQL -> scale + flexible schema + high writes: feeds, sessions, catalog Families: key-value (Redis) doc (MongoDB) column (Cassandra) graph (Neo4j)

Scaling data replicate / shard

Replication copies the same data to multiple nodes for read scaling and availability/failover. Sharding (partitioning) splits data across nodes by a shard key to scale writes and storage.

Replication (every node holds ALL data):
  Primary --writes--> Replica A
                  \--> Replica B    reads spread across replicas

Sharding / partitioning (split by shard key = user_id % 3):
  Shard 0: ids 0,3,6...   Shard 1: 1,4,7...   Shard 2: 2,5,8...
  each shard holds only PART of the data
Trade-offsReplication: cheap read scale + failover; replicas lag (stale reads), does NOT raise write throughput. Sharding: scales writes + storage; cross-shard joins/txns are hard, a bad shard key -> hotspots. Often combined (shard + replicas).

CAP theorem C / A / P

In a distributed store, when a network Partition occurs you must sacrifice either Consistency or Availability. It is NOT "pick any 2 of 3 at all times" -- with no partition you can have both C and A.

Network partition happens -> you MUST pick one:
  CP  stay consistent, refuse/block some requests (lose availability)
      e.g. HBase, ZooKeeper, etcd
  AP  stay available, serve possibly-stale data (lose consistency)
      e.g. Cassandra, DynamoDB, Riak
  CA  only when there is NO partition (not a real distributed mode)
Key pointCAP is a PARTITION-TIME choice, not "pick 2 of 3 always". No partition -> a node can be both consistent AND available. PACELC extends it: Else (no partition) trade Latency vs Consistency.
