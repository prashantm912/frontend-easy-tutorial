Software Architectures — SOLID Principles & Design Patterns in Java

A practical, backend-oriented tour of how to structure object-oriented software well. It starts with the five SOLID principles — Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, and Dependency Inversion (with inversion of control) — the design rules that keep code flexible and testable. It then works through the classic Gang-of-Four design patterns, grouped as the book groups them: creational (Singleton, Factory, Abstract Factory, Builder, Prototype), behavioral (Strategy, Observer, Command, Iterator, Template Method, Null Object, Visitor), and structural (Decorator, Facade, Flyweight, Adapter, Bridge, Composite). It closes with system design: APIs and web services (SOAP, RPC, REST), software architectures (monolithic, layered, service-oriented, microservices), and data storage (relational vs. non-relational databases, scaling, and the CAP theorem). Each lecture pairs a focused, reimplemented Java example with the console output it produces, traced by hand (Java runs on the JVM, not in the browser).

How to read this The examples are ordinary Java (classes, interfaces, generics) — no framework and no build tool required; compile and run any of them with javac *.java && java App. Each pattern is presented the same way: the problem it solves, a UML-style diagram of the participants, a minimal implementation, and the trade-offs. Patterns are design vocabulary, not code to copy blindly — the "Pitfall" boxes flag when a pattern is overkill or commonly misapplied. Use the filter box in the sidebar to jump to any lecture. Prose is original; code examples are reimplemented for clarity.

1. Introduction & Foundations

This opening section sets up the whole guide: why deliberate design matters, the five SOLID principles that keep object-oriented code loosely coupled, and what the classic Gang of Four design patterns actually are. It is concept-only — a road map and shared vocabulary before we start writing patterns section by section.

Run locally Every example in this guide is plain Java — no framework, no build tool. 01. Course overview: architecture, SOLID & patterns

Objective: Understand what this guide covers, why deliberate design matters, and the road map from principles to patterns to system design.

Any competent programmer can write code that works. The harder, more valuable skill is writing code that keeps working as requirements change — code a teammate (or you, six months later) can read, extend, and test without fear. Software architecture is the set of decisions about how a system's parts are divided and how they talk to one another. Good decisions leave you flexible; bad ones calcify into a system that resists every change.

Three qualities recur throughout this guide. Maintainability: a change stays localized instead of rippling everywhere. Flexibility: new behavior arrives by writing new code, not editing old, tested code. Testability: a piece can be exercised in isolation because it depends on abstractions you can substitute with a stub. The common enemy of all three is tight coupling — when every class knows about every other, one edit breaks five things.

The guide is built in three layers, and that is the order we follow. First the SOLID principles, five guidelines for class-level design that reduce coupling and rigidity. Then the Gang of Four design patterns, named solutions grouped into creational (how objects are made), structural (how objects are composed), and behavioral (how objects collaborate). Finally system design — APIs, service architectures, data storage, and the trade-offs (like CAP) that shape systems at scale.

  • Working code is the baseline; maintainable, flexible, testable code is the goal.
  • Architecture = how a system is split into parts and how those parts communicate.
  • Tight coupling is the recurring problem; loose coupling through abstractions is the recurring cure.
  • Road map: SOLID principles → GoF patterns (creational / structural / behavioral) → system design.
  • Patterns are a shared vocabulary and a menu of trade-offs — not a checklist to maximize.
// Rigid: OrderService is welded to one concrete payment type
class OrderService {
    private final StripeGateway gateway = new StripeGateway();  // hard-wired detail
    void checkout(Order o) { gateway.charge(o.total()); }
}

// Flexible: depend on an abstraction; the concrete type is supplied from outside
interface PaymentGateway { void charge(long cents); }

class OrderService {
    private final PaymentGateway gateway;                       // any implementation
    OrderService(PaymentGateway gateway) { this.gateway = gateway; }
    void checkout(Order o) { gateway.charge(o.total()); }       // easy to swap or stub in a test
}

AdvancedA useful lens: the value of software lies both in what it does today and in how cheaply it can be changed tomorrow. Business needs change constantly, so a system that is rigid slowly loses value even if its behavior is correct. SOLID and the design patterns are simply tools for keeping the cost of change low — that is the yardstick to judge every design decision against.
Pitfall Don't confuse "uses lots of patterns" with "well-designed." Applying a pattern where a plain class would do adds indirection and cognitive load for no benefit. The goal is the least structure that keeps change cheap, not the most.

02. What are the SOLID principles?

Objective: Name the five SOLID principles and understand that they are object-oriented design guidelines aimed at reducing coupling and rigidity.

SOLID is a mnemonic for five object-oriented design principles. They were articulated and popularized by Robert C. Martin (“Uncle Bob”) around the turn of the 2000s; the memorable SOLID ordering itself was later arranged by Michael Feathers. Each is a guideline — a rule of thumb, not a law — for shaping classes and interfaces so a system stays soft and easy to change.

The five, in order:

  • S — Single Responsibility: a class should have one, and only one, reason to change — one job.
  • O — Open/Closed: entities should be open for extension but closed for modification — add behavior by adding code, not editing tested code.
  • L — Liskov Substitution: an object of a subtype must be usable anywhere its base type is expected, without breaking correctness.
  • I — Interface Segregation: prefer many small, focused interfaces over one fat one, so clients depend only on what they use.
  • D — Dependency Inversion: depend on abstractions, not concretions — high-level policy should not depend on low-level detail.

The through-line is decoupling. Following SOLID tends to produce small classes, narrow interfaces, and dependencies that point at abstractions — exactly the conditions under which the design patterns later in this guide become natural rather than forced. Each principle gets its own dedicated lecture.

// Violates SRP: Invoice both models the data AND knows how to store and send itself
class Invoice {
    BigDecimal total() { /* ... */ return null; }
    void saveToDatabase() { /* reason to change #2: storage   */ }
    void emailToCustomer() { /* reason to change #3: messaging */ }
}

// Honors SRP: one responsibility, one reason to change, per class
class Invoice           { BigDecimal total() { /* data + calculation */ return null; } }
class InvoiceRepository { void save(Invoice i) { /* storage   */ } }
class InvoiceMailer     { void send(Invoice i) { /* messaging */ } }

AdvancedSOLID is not the only body of guidance. Related maxims include DRY (Don't Repeat Yourself), KISS (Keep It Simple), and YAGNI (You Aren't Gonna Need It). SOLID targets class-level design specifically; higher-level cohesion and coupling principles govern modules and services, which is where the system-design sections later pick up.
Pitfall The principles can conflict, and they can be over-applied. Splitting every class until each holds a single method, or hiding everything behind an interface “just in case,” yields a maze of tiny types that is harder to follow. Apply a principle when it removes real pain, not reflexively.

03. What are design patterns?

Objective: Define what a design pattern is, learn its origin in the Gang of Four catalog, and see the three GoF categories with the patterns this guide works through.

A design pattern is a named, reusable solution to a problem that recurs in object-oriented design. It is not a finished class you copy and paste, but a template: a description of the participants (the roles that classes play), how they collaborate, and the trade-offs involved — which you adapt to your situation. Half its value is the solution and half is the shared vocabulary: saying “use a Strategy here” communicates an entire design in two words.

The catalog everyone means by “design patterns” comes from the 1994 book Design Patterns: Elements of Reusable Object-Oriented Software by Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides — collectively the Gang of Four (GoF). It catalogs 23 patterns and sorts them by intent into three families:

  • Creational — how objects are created, decoupling code from the concrete classes it instantiates (Singleton, Factory Method, Abstract Factory, Builder, Prototype).
  • Structural — how objects and classes are composed into larger structures (Adapter, Bridge, Composite, Decorator, Facade, Proxy, Flyweight).
  • Behavioral — how objects collaborate and distribute responsibility (Strategy, Observer, Command, Template Method, State, Iterator, Chain of Responsibility).

Patterns are trade-offs, not free wins. Each buys flexibility at the cost of some added indirection, and each fits certain forces and not others. Learn the intent and the problem a pattern solves, then reach for it when your problem genuinely matches — never because the pattern is clever. That is what “vocabulary and trade-offs, not code to paste blindly” means.

// "Use a Strategy" names a whole design in two words. Participants:
interface SortStrategy { void sort(int[] data); }         // Strategy  (the abstraction)
class QuickSort  implements SortStrategy { public void sort(int[] d) { /* ... */ } }  // Concrete Strategy
class BubbleSort implements SortStrategy { public void sort(int[] d) { /* ... */ } }  // Concrete Strategy

class Sorter {                                             // Context
    private SortStrategy strategy;
    void setStrategy(SortStrategy s) { this.strategy = s; }
    void run(int[] data) { strategy.sort(data); }          // delegates to whichever strategy is set
}

Source: reimplemented as an original illustration.

AdvancedPatterns are not universally loved. Singleton is now often treated as an anti-pattern because it smuggles in global state and complicates testing. Others became so useful they were absorbed into the language and standard library — java.util.Iterator is the Iterator pattern; java.util.function factories and Comparator chaining echo Factory and Strategy. Recognizing a pattern already living in the JDK is often easier than inventing one from scratch.
Pitfall The classic beginner mistake is pattern-driven design — picking a pattern first and forcing the problem to fit. Patterns are best discovered in a design under pressure to change, not scattered in up front. Start simple; refactor toward a pattern when real duplication or rigidity actually appears.

2. S — The Single Responsibility Principle

The first letter of SOLID. A class should have one, and only one, reason to change — a single responsibility owned by a single stakeholder. We contrast the "god class" that does everything with a design where parsing, validating, computing, and rendering each live in their own focused class.

01. The Single Responsibility Principle

Objective: State the principle precisely and recognise the "god class" anti-pattern that it fixes.

The Single Responsibility Principle (SRP), the S in SOLID, says a class should have one, and only one, reason to change. A responsibility here is not a method or a feature — it is an axis of change: a concern that some stakeholder can ask you to alter independently of the others.

A class that mixes several responsibilities is a god class: it computes business logic, talks to the database, and formats output all at once. Every unrelated requirement drags in an edit, so the class changes for many reasons and each change risks breaking the others. Splitting those concerns into separate classes raises cohesion (everything in a class belongs together), isolates change, and makes each piece independently testable and reusable.

  • One reason to change per class — measured by who asks for the change, not by line count.
  • SRP is really about cohesion: group what changes together, separate what changes apart.
  • Benefits: change isolation, easier unit testing, higher reuse, smaller merge conflicts.
  • It applies to methods and modules too, not just classes.

A god class collapses many concerns into one type; SRP pulls them apart:

// God class: FOUR reasons to change in one type
class InvoiceService {
    BigDecimal calculateTotal(Invoice inv) { /* pricing rules */ }
    void        saveToDatabase(Invoice inv) { /* JDBC / SQL     */ }
    String      renderHtml(Invoice inv)     { /* presentation   */ }
    void        emailCustomer(Invoice inv)  { /* SMTP transport */ }
}

// SRP: each concern is a class with ONE reason to change
class InvoiceCalculator { BigDecimal total(Invoice inv) { ... } }
class InvoiceRepository { void save(Invoice inv) { ... } }
class InvoiceRenderer   { String render(Invoice inv) { ... } }
class InvoiceMailer     { void send(Invoice inv) { ... } }

Source: reimplemented as an original illustration.

AdvancedRobert C. Martin later sharpened "reason to change" to actor: "gather together the things that change for the same reason, and separate those that change for different reasons." The pricing team, the DBA, and the marketing team are three different actors — so pricing, persistence, and presentation belong in three different classes. If two actors can force edits to the same class, they can break each other's code; that shared class is the real SRP violation.
Pitfall Over-correcting. Splitting every method into its own one-method class produces anemic classes and ceremony without cohesion. SRP means grouping by reason-to-change, not maximising the class count.

02. SRP example — separating concerns

Objective: Walk a small program whose workflow is split into four focused classes plus a thin orchestrator.

The program reads two numbers, checks they are valid integers, multiplies them, and prints the result. Rather than cramming all of that into main, each step is a class with a single job:

  • Pair — an immutable value holder for the two raw inputs.
  • InputProcessor — turns console input into a Pair (the only class that knows about the Scanner).
  • ViolationChecker — decides whether a Pair is a valid pair of integers.
  • Operation — performs the arithmetic (here, multiplication).
  • App — orchestrates: read → validate → compute → print. It owns the sequence, not the details.
// (package + java.util.Scanner imports omitted)

// Responsibility: hold the two raw values, immutably.
public final class Pair {
    private final String first, second;
    public Pair(String first, String second) {
        this.first = first;
        this.second = second;
    }
    public String getFirst()  { return first; }
    public String getSecond() { return second; }
}

// Responsibility: read console input into a Pair.
public final class InputProcessor {
    public static Pair process(Scanner scanner) {
        System.out.println("Enter the first number:");
        String first = scanner.nextLine();
        System.out.println("Enter the second number:");
        String second = scanner.nextLine();
        return new Pair(first, second);
    }
}

// Responsibility: is this a valid pair of integers?
public final class ViolationChecker {
    public static boolean isValid(Pair pair) {
        return isInteger(pair.getFirst()) && isInteger(pair.getSecond());
    }
    private static boolean isInteger(String s) {
        try { Integer.parseInt(s); return true; }
        catch (NumberFormatException e) { return false; }
    }
}

// Responsibility: perform the arithmetic.
public final class Operation {
    public static int execute(int a, int b) { return a * b; }
}

// Responsibility: orchestrate the workflow.
public class App {
    public static void main(String[] args) {
        System.out.println("Welcome to the Application!");
        Scanner scanner = new Scanner(System.in);
        Pair pair = InputProcessor.process(scanner);

        if (!ViolationChecker.isValid(pair)) {
            System.out.println("One of the inputs is invalid...");
            return;
        }
        int result = Operation.execute(
                Integer.parseInt(pair.getFirst()),
                Integer.parseInt(pair.getSecond()));
        System.out.println("The result is: " + result);
        System.out.println("End of the application!");
    }
}
Console outputWelcome to the Application! Enter the first number: 6 Enter the second number: 7 The result is: 42 End of the application!

Source: reimplemented from the course example. The lines 6 and 7 are the values the user types (echoed by the terminal); 6 × 7 = 42.

AdvancedSeparation buys testability. ViolationChecker.isValid and Operation.execute are pure, static functions with no I/O, so a unit test can call them directly with crafted inputs — no console, no mocking. Only InputProcessor touches Scanner, so it is the single place to change if input later comes from a file or an HTTP request; the validator and the operation are untouched. That is change isolation in practice.
Pitfall A leaked responsibility. Notice App still does the String→int parsing inline via Integer.parseInt. Conversion is arguably its own concern; leaving it in the orchestrator means App now has two reasons to change (workflow and parsing). A stricter design would give it a Converter (or let Pair expose typed values).

03. SRP exercise & solution

Objective: Refactor a class that reads, validates, sorts and prints all at once into single-responsibility collaborators.

The exercise's starting point is one App class that does everything: it loops on a Scanner reading and validating five integers in the range [0, 10], then sorts them, then prints them. Three concerns — acquiring valid input, ordering, and presentation — share a single class, so any of them forces an edit there.

The SRP refactor gives each concern its own class. RangeValidator answers "is this token a valid integer in range?"; NumberReader collects the required count of valid numbers from a token source; NumberSorter orders them; ResultPrinter renders them; App only wires the pieces together. Decoupling the reader from System.in (it takes any Iterator<String>) means the demo below can feed it a fixed list — which is exactly what makes it unit-testable.

  • Input acquisition, validation, sorting, and printing become four cohesive classes.
  • NumberReader depends on the RangeValidator abstraction, not on the range logic itself.
  • The token source is an Iterator<String>, so a test list replaces the console with no other change.
// imports: java.util.{List, ArrayList, Collections, Iterator}

// Responsibility: is `token` an integer within [min, max]?
final class RangeValidator {
    private final int min, max;
    RangeValidator(int min, int max) { this.min = min; this.max = max; }
    boolean isValid(String token) {
        try {
            int n = Integer.parseInt(token);
            return n >= min && n <= max;
        } catch (NumberFormatException e) {
            return false;
        }
    }
}

// Responsibility: collect `count` valid numbers from a token source.
final class NumberReader {
    private final RangeValidator validator;
    NumberReader(RangeValidator validator) { this.validator = validator; }
    List<Integer> read(int count, Iterator<String> source) {
        List<Integer> nums = new ArrayList<>();
        while (nums.size() < count && source.hasNext()) {
            String token = source.next();
            if (validator.isValid(token))
                nums.add(Integer.valueOf(token));
        }
        return nums;
    }
}

// Responsibility: sort WITHOUT mutating the caller's list.
final class NumberSorter {
    List<Integer> sorted(List<Integer> nums) {
        List<Integer> copy = new ArrayList<>(nums);
        Collections.sort(copy);
        return copy;
    }
}

// Responsibility: render the result.
final class ResultPrinter {
    void print(List<Integer> nums) {
        StringBuilder sb = new StringBuilder();
        for (int n : nums) sb.append(n).append(' ');
        System.out.println(sb.toString().trim());
    }
}

// Responsibility: wire the pieces together.
public class App {
    public static void main(String[] args) {
        // stand-in for typed input; "-4","11","abc" are invalid
        List<String> typed = List.of("8","3","-4","11","3","abc","10","0");

        RangeValidator validator = new RangeValidator(0, 10);
        NumberReader   reader     = new NumberReader(validator);
        NumberSorter   sorter     = new NumberSorter();
        ResultPrinter  printer    = new ResultPrinter();

        System.out.println("Reading up to 5 valid integers in [0, 10]...");
        List<Integer> nums = reader.read(5, typed.iterator());
        System.out.println("Collected: " + nums);

        List<Integer> ordered = sorter.sorted(nums);
        System.out.println("Sorted:    " + ordered);

        System.out.print("Result: ");
        printer.print(ordered);
    }
}
Console outputReading up to 5 valid integers in [0, 10]... Collected: [8, 3, 3, 10, 0] Sorted: [0, 3, 3, 8, 10] Result: 0 3 3 8 10

Source: reimplemented from the course example. -4 and 11 are out of range and abc is not a number, so all three are skipped; reading stops once five valid values are collected.

AdvancedSRP is what enables Dependency Inversion later: NumberReader holds a RangeValidator reference, so it depends on validation as a collaborator rather than owning the range rule. Extract a Validator interface and you can inject a different policy (say, [1, 100]) without touching the reader — the responsibility boundary drawn by SRP is exactly the seam where the dependency can be inverted.
Pitfall A "sorter" that mutates its argument. Had NumberSorter called Collections.sort(nums) on the caller's list, it would silently reorder data another class still relied on — a hidden side effect that violates the single, clear responsibility "produce a sorted view." Returning a sorted copy keeps the method honest.
Cheat-sheet
SRP: one class, one reason to change  (reason == actor/stakeholder)
  smell   -> a class that computes AND persists AND formats AND sends
  fix     -> one cohesive class per concern; a thin orchestrator wires them
  gains   -> change isolation, unit-testable units, reuse, fewer merge conflicts
  measure -> "who requests this change?"  two actors sharing a class = violation
  caution -> don't shatter into anemic one-method classes; group by reason
  synergy -> clean SRP seams are where OCP/DIP plug in later

3. Open/Closed Principle (O)

The second SOLID principle: software entities should be open for extension but closed for modification. You add new behaviour by adding new types behind a stable abstraction — never by re-opening and editing code that already works and is already tested.

01. The Open/Closed Principle

Objective: Understand what “open for extension, closed for modification” means in practice, and recognise the type-switch anti-pattern that OCP is designed to eliminate.

Bertrand Meyer’s Open/Closed Principle says a module should be open for extension — its behaviour can grow — yet closed for modification — its existing source stays untouched. The tension is resolved with abstraction and polymorphism: you fix a stable interface, then extend the system by writing new classes that implement it, rather than editing a central class every time a requirement appears.

The classic violation is a method that branches on a type code — a growing if / else if chain or a switch over an enum or string. Every new variant forces you back into that one method to add another branch, then recompile and re-test code that was already correct. That is exactly what “closed for modification” forbids: each change risks breaking working behaviour and creates a merge-conflict magnet.

  • Open for extension: new behaviour can be added — by adding a new type.
  • Closed for modification: existing, tested classes are not edited to accommodate it.
  • The seam is a stable abstraction (an interface or abstract class); variants are its implementations.
  • A switch or if/else that grows one branch per new type is the tell-tale OCP smell.
  • OCP reduces regression risk and merge conflicts: adding a feature touches only new files.

The anti-pattern — one method that must be reopened for every new category:

// ANTI-PATTERN: NOT closed for modification.
// Each new category reopens and edits this method.

    void process(String category) {
        if (category.equals("AI")) {
            System.out.println("AI related questions...");
        } else if (category.equals("ALGORITHMS")) {
            System.out.println("Algorithms related questions...");
        } else if (category.equals("CS")) {
            System.out.println("CS related questions...");
        }
        // Need "FINANCE"? Edit HERE, recompile, re-test this class again.
    }
}

AdvancedOCP is a target, not an absolute — you cannot make code closed against every conceivable change. The pragmatic move is to identify the axis of change that actually varies (here: the kind of question) and close the design against that axis with an abstraction, while accepting it stays open along others. This is the “strategic closure” Robert C. Martin describes: predict the likely direction of change, and put the seam there.
Pitfall Premature abstraction. Sprinkling interfaces and extension points everywhere “just in case” is speculative generality — it adds indirection with no payoff and can be harder to change than the concrete code. Introduce the OCP seam when a second variation actually appears (or is clearly imminent), not before. When in doubt, follow the rule of three.

4. The Liskov Substitution Principle (LSP)

The L in SOLID, coined by Barbara Liskov: if S is a subtype of T, then objects of type T may be replaced by objects of type S without altering the correctness of the program. LSP is about behavioural substitutability — not merely that the subclass compiles.

01. The Liskov Substitution Principle

Objective: Understand LSP as a contract between a base type and its subtypes, and recognise the classic violations that pass the compiler yet break at runtime.

Inheritance advertises an is-a relationship, but LSP demands more than that: a subtype must honour everything a caller may reasonably assume about the base type. Concretely, an overriding method must not strengthen preconditions (demand more of its inputs), must not weaken postconditions (deliver less than promised), must preserve the base type's invariants, and must not throw a new "not supported" exception where the base contract promised a result.

The textbook violations: a Square that extends Rectangle but breaks the independent-width/height invariant; a Penguin that extends Bird yet cannot honour fly(); a read-only collection that extends a mutable one and throws on add(). Each compiles cleanly — the damage only shows when client code, written against the base type, is handed the subtype.

  • LSP governs behaviour, not just method signatures — "compiles fine" is not "substitutable".
  • Rule of thumb: preconditions may only be weakened, postconditions only strengthened, invariants preserved.
  • Throwing UnsupportedOperationException from an override is a red flag that the is-a is a lie.
  • A LSP-clean hierarchy lets client code use a base reference and never care which concrete subtype it holds.
class Rectangle {
    protected int width, height;
    public void setWidth(int w)  { this.width  = w; }
    public void setHeight(int h) { this.height = h; }
    public int area() { return width * height; }
}

// "A square IS-A rectangle" -- true in geometry, false behaviourally.
class Square extends Rectangle {
    @Override public void setWidth(int w)  { this.width = w; this.height = w; }
    @Override public void setHeight(int h) { this.width = h; this.height = h; }
}

public class App {
    // Client written against Rectangle's contract: set 5x4, expect area 20.
    static void resizeAndCheck(Rectangle r) {
        r.setWidth(5);
        r.setHeight(4);
        System.out.println("Expected area 20, actual: " + r.area());
    }
    public static void main(String[] args) {
        resizeAndCheck(new Rectangle()); // honours the contract
        resizeAndCheck(new Square());    // silently breaks it
    }
}

Pitfall Reaching for inheritance purely to reuse code when the is-a contract does not truly hold. Square extends Rectangle shares fields but violates the caller's invariant. When only the implementation is shared, prefer composition (hold a field) over inheritance.

02. LSP example — vehicles & fuel

Objective: Trace a real LSP violation where Vehicle implements Fuel forces every subtype to expose a refuel behaviour, then fix the hierarchy so every subtype genuinely satisfies its base contract.

The course models an abstract Vehicle that implements Fuel, so a fuel() method is baked into the base type. PetrolCar implements it naturally. ElectricCar has no fuel tank — it is forced to fake the method or to reject it. If it rejects it with an UnsupportedOperationException, then any code holding a Vehicle (or Fuel) reference and calling fuel() breaks the moment an ElectricCar is substituted. That runtime failure through a base reference is the LSP violation.

The fix is to reshape the hierarchy so the base contract holds for every subtype. The base Vehicle keeps only what all vehicles honour — speedUp() and slowDown(). Energy replenishment becomes a capability interface: Refuelable for petrol/diesel, Rechargeable for electric. Now no subtype is forced to implement a behaviour it cannot fulfil, and a Vehicle reference is always safely substitutable.

  • Putting fuel() on a base that an ElectricCar cannot honour is the root cause — the subtype cannot keep the promise.
  • The tell-tale symptom is an override that throws "not supported" or silently no-ops.
  • Fix: model behaviours only every subtype shares in the base; push variant capabilities into narrow interfaces (this also satisfies the Interface Segregation Principle).
  • Client code depending on Vehicle then never risks a runtime surprise from substitution.

        v.slowDown();

    }
}

5. Interface Segregation Principle

The I in SOLID. A client should never be forced to depend on methods it does not use. Instead of one large "fat" interface, prefer several small, role-specific interfaces so that each implementer only pledges to do what it can actually do.

01. The Interface Segregation Principle

Objective: State the principle precisely and recognise the "fat interface" smell that signals a violation.

Formally: clients should not be forced to depend upon interfaces they do not use. When a single interface bundles many unrelated operations, every class that implements it must supply a body for all of them — even the ones that make no sense for that class. The class is then coupled to methods it never needs, and callers of the interface are misled into thinking every operation is supported.

The tell-tale symptom is an implementation that "fills in the blanks" with empty method bodies or with throw new UnsupportedOperationException(). That is the compiler forcing you to honour a contract you cannot really keep. The fix is to split the fat interface into focused, cohesive role interfaces, each describing one capability; a class then implements only the roles it truly plays.

  • Definition: no client depends on methods it does not use.
  • Smell: empty method bodies or UnsupportedOperationException to satisfy an interface.
  • Cure: many small, role-specific interfaces instead of one big one.
  • Design interfaces around how clients use a type, not around one all-in-one contract.
  • Interfaces can compose by extension: a richer role can extend a simpler one.

The snippet below shows the violation and the cure. An old, print-only device is forced to stub out scan and fax, which is exactly what ISP tells us to avoid.

// SMELL: one fat interface bundles unrelated operations
interface Machine {
    void print(Doc d);
    void scan(Doc d);
    void fax(Doc d);
}

// A print-only device is FORCED to implement scan() and fax()
class OldPrinter implements Machine {
    public void print(Doc d) { /* real work */ }
    public void scan(Doc d)  { throw new UnsupportedOperationException(); } // lie!
    public void fax(Doc d)   { throw new UnsupportedOperationException(); } // lie!
}

// CURE: split by role; implement only what you support
interface Printer { void print(Doc d); }
interface Scanner { void scan(Doc d); }
interface Fax     { void fax(Doc d); }

class SimplePrinter implements Printer {
    public void print(Doc d) { /* real work, nothing to fake */ }
}

AdvancedISP is the interface-level analogue of the Single Responsibility Principle: SRP keeps a class focused on one reason to change, ISP keeps an interface focused on one client role. It matches Martin Fowler's idea of a Role Interface (defined by how one particular client uses a type) as opposed to a bloated Header Interface that exposes everything an object can do to everyone.
Pitfall Do not over-correct into a swarm of single-method interfaces for every method you own. Segregation is driven by distinct client roles, not by counting methods — operations that are always used together should stay together in one cohesive interface.

02. ISP example — tree interfaces

Objective: Apply ISP to a family of tree data structures where only some trees support rotation.

Every tree supports insert, delete and traverse, so those belong in a small base interface Tree. Only self-balancing trees (AVL, red-black) perform leftRotation and rightRotation. Putting rotation on Tree would force a plain BinarySearchTree to implement operations it does not have. Instead, a second role interface RotationTree extends Tree adds the rotation methods, and only balancing trees implement it.

Because RotationTree extends Tree, a BalancedTree still gets the core operations "for free" while additionally promising to rotate — and a BinarySearchTree is never asked to. The client picks the narrowest interface that covers what it needs: a routine that only inserts and traverses can accept a Tree and work with either implementation.

  • Base role Tree: insert(), delete(), traverse() — shared by all trees.
  • Extended role RotationTree extends Tree: adds leftRotation(), rightRotation().
  • BinarySearchTree implements Tree only — it is never forced to rotate.
  • BalancedTree implements RotationTree — core operations plus rotation.
interface Tree {

    void delete();
    void traverse();

    void rightRotation();

6. D — Dependency Inversion Principle & Inversion of Control

The last SOLID letter: don't let policy code reach down and grab concrete tools. Make both the high-level policy and the low-level detail depend on an abstraction the policy owns, then hand the concretion in from outside. That single move loosens coupling, unlocks testing with stubs, and leads naturally into Inversion of Control and dependency-injection frameworks.

01. The Dependency Inversion Principle

Objective: State DIP precisely and understand what is actually being "inverted".

The Dependency Inversion Principle (the D in SOLID) has two clauses. First: high-level modules should not depend on low-level modules; both should depend on abstractions. Second: abstractions should not depend on details; details should depend on abstractions. In plain terms, a class that expresses business policy (a controller, a service) must talk to an interface, never to a concrete MySQLDatabase or PayPal.

The word inversion describes what changes versus the naive design. Normally the high-level module reaches downward and depends on the low-level one, so the direction of source-code dependency follows the direction of control flow. DIP inserts an abstraction between them and points both arrows at it: the low-level module now depends on (implements) an interface, and — crucially — that interface is defined by and for the high-level policy. Ownership of the contract moves up to the policy layer; the dependency arrow is flipped.

  • High-level and low-level modules both depend on an abstraction, not on each other.
  • The abstraction is owned by the high-level layer — it describes what the policy needs, not what a vendor happens to offer.
  • Concrete details (drivers, gateways, frameworks) depend on the abstraction, never the reverse.
  • Result: you can replace a detail without editing a single line of policy code.

The interface here is tiny — the exact contract the policy needs:

// The high-level policy declares WHAT it needs, as an interface it owns.
interface MessageStore {                 // abstraction
    void save(String message);
}

class NotificationService {              // high-level module
    private final MessageStore store;    // depends on the abstraction only
    NotificationService(MessageStore store) { this.store = store; }
    void notify(String msg) { store.save(msg); }   // never names a concrete DB
}

AdvancedDIP is about source-code dependency direction, not just "use interfaces". In a clean layered design the abstraction lives in (and is owned by) the high-level package, and the concrete adapter in the low-level package depends inward on it — this is the core of Hexagonal / Ports-and-Adapters and Clean Architecture, where the "port" is the interface and the "adapter" is the detail. Ownership matters: an interface that just mirrors a vendor SDK method-for-method has not inverted anything.
Pitfall Declaring a field or parameter with a concrete type (private OracleDatabase db;) silently violates DIP even if you were "planning" to swap it later — the compile-time dependency is already nailed to the detail. Type the field as the abstraction (private Database db;).

02. What is inversion of control?

Objective: Separate three often-confused ideas — the DIP principle, the IoC style, and dependency injection as one technique for achieving it.

Inversion of Control (IoC) is a design style in which a class gives up control over parts of its own execution — typically the creation of its collaborators and the sequencing of calls — to an outside party: a framework, container, or caller. The classic summary is the Hollywood Principle: "Don't call us, we'll call you." Your code stops calling out to fetch what it needs; instead the framework calls your code and supplies what it needs.

Dependency Injection (DI) is one form of IoC — the one that inverts control over dependency creation. Instead of a class writing new PayPal(), the concretion is handed to it through its constructor (constructor injection) or a setter (setter injection). DI is a strict subset of IoC: all DI is IoC, but IoC also covers template methods, callbacks, event subscriptions, and framework lifecycle hooks that never inject a thing.

Keep the three levels straight: DIP is a principle (depend on abstractions); IoC is a style/technique for who is in control; a DI container (Spring, CDI, Guice) is a tool that automates injection. You can fully satisfy DIP and DI with nothing but plain constructors and a main method — no framework required.

  • IoC = the framework/caller owns the flow and supplies collaborators (Hollywood Principle).
  • DI (constructor or setter injection) is a subset of IoC, not a synonym for it.
  • DIP is the principle; IoC is the technique; a DI container is just an optional automation tool.
  • Constructor injection is preferred: dependencies become final, required, and visible in the signature.
// Dependency injection = one form of IoC: the object never creates its collaborator.
class OrderService {
    private final PaymentMethod payment;    // needs a collaborator
    OrderService(PaymentMethod payment) {   // constructor injection
        this.payment = payment;             // handed in from outside
    }
}

// The caller (or a DI container) decides the concretion and injects it:
OrderService service = new OrderService(new PayPal());

Source: reimplemented as an original illustration.

AdvancedNot every IoC mechanism is injection. The Template Method pattern inverts control over an algorithm's steps; observer/event systems invert control over when your code runs; a Service Locator inverts creation but the object still pulls its dependencies from a global registry. A DI container is essentially an automated, reflection-driven assembler that reads your constructor signatures and wires the object graph for you — convenient at scale, but pure DIP is achievable with hand-written wiring in main.
Pitfall Assuming you "need Spring to do DI". A DI container is optional sugar; reaching for one on a small program adds magic and startup cost without changing the design. Equally, prefer constructor injection over a Service Locator — the locator hides a class's real dependencies behind a lookup call, making them invisible in the API and harder to test.

03. DIP example — the database controller

Objective: Apply DIP so a high-level controller can drive any database implementation injected at construction time.

The high-level module is DatabaseController. It holds a field typed as the Database abstraction and receives a concrete implementation through its constructor — this is constructor injection. The low-level modules OracleDatabase and MySQLDatabase each implements Database. Because the controller only ever calls connect() / disconnect() on the interface, swapping Oracle for MySQL touches only the wiring in main; the controller class is never edited.

  • DatabaseController depends on Database (abstraction), not on a vendor class.
  • The concretion is injected via the constructor — the controller does no new.
  • Adding a new database = write one class that implements Database; existing code is untouched (Open/Closed too).

Model the contract as a PaymentMethod interface with a single pay() operation. PayPal, Payoneer, and Skrill are the low-level modules that implement it. Onboarding a new provider is a new class plus one line of wiring — no change to WebStore.

  • WebStore (high-level) depends on the PaymentMethod abstraction; the provider is injected.
  • Each provider is a self-contained low-level module implementing a one-method contract.
  • A new gateway means a new implementing class — WebStore stays closed for modification.

7. Creational patterns I: Singleton & Factory

The creational patterns abstract the act of instantiation — they put a layer between "I need an object" and "here is the concrete class and constructor that build it". This section covers the two most common: Singleton (exactly one instance) and the Factory family (Factory Method and Abstract Factory), which hide new behind a stable interface.

01. Creational design patterns — overview

Objective: Understand what the creational category is for, and why decoupling object creation from object use makes a system easier to change and test.

The Gang-of-Four grouped their 23 patterns into three families. The creational family — Singleton, Factory Method, Abstract Factory, Builder, Prototype — is about how objects are made. Each one moves the new keyword out of client code and behind an abstraction, so the client depends on an interface rather than on a specific class and constructor.

Why bother? A bare new Dog() welds the caller to the concrete Dog class, its constructor signature, and its lifecycle. If tomorrow you need to pool instances, pick a subtype at runtime, wire dependencies, or swap in a test double, every new site is a place you must edit. Centralising creation gives you one place to change — and directly supports the Dependency Inversion Principle (depend on abstractions, not concretions).

  • Creational patterns answer "which object, built how?" without exposing concrete classes to the caller.
  • Singleton — one shared instance; Factory Method — a method chooses the concrete type; Abstract Factory — families of related objects; Builder — step-by-step assembly of complex objects; Prototype — clone an existing instance.
  • The payoff is flexibility and testability; the cost is indirection — do not add a factory where a plain constructor is perfectly clear.
Pitfall Reaching for a pattern reflexively. A single, stable, no-argument constructor needs no factory. Introduce creational patterns to solve a real problem (runtime type selection, one shared instance, complex wiring), not to look sophisticated.

02. What is the Singleton pattern?

Objective: Learn the Singleton's intent — one instance, one global access point — and when that is justified versus when it becomes hidden global state.

The Singleton (creational) guarantees a class has exactly one instance and provides a single, well-known access point to it. The mechanism is always the same two moves: make the constructor private so no one else can call new, and expose a static accessor that hands back the one instance.

It is justified when a single shared object genuinely models the domain: a read-only configuration registry, a connection pool, an in-memory cache, or a logging facility. Its bad reputation comes from misuse: a Singleton is global mutable state in disguise. It couples every caller to a concrete class, hides dependencies (a method that quietly calls Config.getInstance() lies about what it needs), and is awkward to isolate in unit tests.

  • Two ingredients: a private constructor + a static accessor returning the sole instance.
  • Good fits: configuration, connection pools, caches — things that are genuinely one-of-a-kind and shared.
  • Danger: it is global state. Prefer passing the single instance in via a constructor (dependency injection) so collaborators are explicit and testable.
public final class AppConfig {

    // the one and only instance, created up front (eager)
    private static final AppConfig INSTANCE = new AppConfig();

    private AppConfig() { }              // no one else can 'new' it

    public static AppConfig getInstance() {
        return INSTANCE;                 // single global access point
    }

    public String env() { return "production"; }
}

// caller
System.out.println(AppConfig.getInstance().env());
Console outputproduction

AdvancedThis is the eager Singleton: the instance is built when the class is loaded. It is inherently thread-safe (class initialisation is serialised by the JVM) and dead simple. The only downside is that the instance is created even if it is never used — irrelevant for a lightweight config object, but a reason to prefer lazy initialisation for something expensive to build.
Pitfall Treating "I only want one" as reason enough. If callers reach into getInstance() from deep in the code, you have created invisible coupling. Where you can, inject the single instance instead of statically fetching it — you keep "one instance" without the hidden-global cost.

03. Singleton example (lazy, thread-safe)

Objective: Build a lazily-initialised Singleton that is correct under concurrency, and see that every call returns the same object.

A lazy Singleton defers building the instance until the first getInstance() call — useful when the object is expensive and might never be needed. The naive lazy form (if (instance == null) instance = new ...;) is not thread-safe: two threads can both see null and both construct, so more than one instance escapes.

The cleanest correct fix is the initialisation-on-demand holder idiom: put the instance in a private nested class. The JVM loads that nested class — and runs its initialiser — only on first use, and class initialisation is guaranteed atomic and thread-safe by the language. You get laziness and safety with no explicit locking.

  • Lazy = build on first access; eager = build at class load. Lazy needs care under threads.
  • The holder idiom gives lazy + thread-safe initialisation for free, via the JVM's class-init lock.
  • Alternatives: a synchronized accessor (simple but locks every call) or double-checked locking with a volatile field (see Advanced).
  • Every call to the accessor returns the same reference — == and hashCode() agree across calls.
public final class DatabaseConnector {

    private DatabaseConnector() { }          // private constructor

    // nested holder: loaded (and INSTANCE built) only on first
    // getInstance() call -- lazy AND thread-safe by construction
    private static class Holder {
        static final DatabaseConnector INSTANCE = new DatabaseConnector();
    }

    public static DatabaseConnector getInstance() {
        return Holder.INSTANCE;
    }

    public void connect()    { System.out.println("Connecting to the database..."); }
    public void disconnect() { System.out.println("Disconnecting..."); }
}

// caller
DatabaseConnector db1 = DatabaseConnector.getInstance();
DatabaseConnector db2 = DatabaseConnector.getInstance();
db1.connect();
System.out.println("db1 == db2 ? "     + (db1 == db2));
System.out.println("same hashCode ? "  + (db1.hashCode() == db2.hashCode()));
Console outputConnecting to the database... db1 == db2 ? true same hashCode ? true

Source: reimplemented from the course example.

AdvancedThe course's original lazy version synchronises but does not re-check inside the lock, so two threads can still both build an instance. The classic lock-based fix is double-checked locking, and the volatile is not optional — without it a thread can observe a non-null but half-constructed object: 
private static volatile DatabaseConnector instance;

public static DatabaseConnector getInstance() {
    if (instance == null) {                       // 1st check, no lock
        synchronized (DatabaseConnector.class) {
            if (instance == null) {               // 2nd check, locked
                instance = new DatabaseConnector();
            }
        }
    }
    return instance;
}
Prefer the holder idiom for lazy singletons — it is shorter and has no locking on the hot path.
Pitfall "Optimising" the naive lazy check with only an outer if and a synchronized block that unconditionally assigns. Two threads that both pass the null check will each construct an instance — you have silently broken the "exactly one" guarantee. Always re-check inside the lock (and mark the field volatile), or avoid the whole problem with the holder idiom or an enum.

04. Singleton with enums

Objective: Use a single-element enum as the simplest bullet-proof Singleton, safe against serialization and reflection.

Joshua Bloch's Effective Java recommends the enum Singleton as the best approach. A single-constant enum is a Singleton by construction: the JVM guarantees the constant is instantiated exactly once, and acquiring it is inherently thread-safe. Crucially, it is also immune to the two attacks that break ordinary Singletons for free — reflection cannot invoke an enum constructor, and serialization is handled specially so deserializing never creates a second instance.

The trade-off is expressiveness: an enum cannot extend another class (it implicitly extends Enum), and lazy initialisation is coarser. For the common case of a stateless service holder, though, it is the least error-prone Singleton in Java.

  • A one-constant enum is the most robust Singleton: thread-safe, serialization-safe, reflection-safe — all by construction.
  • No boilerplate: no private constructor, no static accessor, no locking to write or get wrong.
  • Limitations: cannot subclass another class; the instance is created eagerly on enum load.

    public void disconnect() { System.out.println("Disconnecting the database..."); }

05. Factory pattern introduction

Objective: Understand the Factory idea — a method decides which concrete subtype to create and returns it as the abstraction, so callers never name a concrete class.

The Factory patterns (creational) put the decision "which concrete class?" behind a method that returns an interface. Callers ask the factory for a product and receive an abstraction; they never write new Cat() or new Dog(). This inverts the dependency: client code depends only on the Animal interface, and the concrete choice lives in one place.

Be precise about the vocabulary. A simple factory is just a static method with a switch that returns one of several subtypes — handy, but not one of the 23 GoF patterns. The GoF Factory Method pattern goes further: it declares the creation method in a base type and lets subclasses decide what to instantiate (creation via inheritance). Both share the goal of returning the abstraction and hiding the concretion.

  • Participants: a Product interface (e.g. Animal), several Concrete Products (Cat, Dog…), and a Factory whose method returns the Product type.
  • Callers depend on the interface, not on any concrete class or its constructor.
  • Simple factory (a static switch) is an idiom; Factory Method (subclasses override the creator) is the true GoF pattern.
Pitfall Calling every static creation method a "Factory Method pattern". A static switch is a simple factory. The GoF Factory Method specifically defers instantiation to subclasses via an overridable method. Precision matters in design reviews.

06. Factory pattern example — animal factory

Objective: Implement a simple factory that maps an AnimalType enum to the right Animal subclass and returns it as the interface.

The product is the Animal interface with a single eat() method; the concrete products are Cat, Dog, Lion, and Tiger. The AnimalFactory.getAnimal(AnimalType) method is the one place that knows how to turn a request into a concrete instance. The caller passes an enum constant and works only with the Animal abstraction.

Using an enum as the selector (rather than raw strings) makes the factory type-safe and lets an exhaustive switch cover every case — the compiler complains if a new constant is added and left unhandled, and no null/default fall-through is needed.

  • One factory method centralises the Cat/Dog/Lion/Tiger choice; callers see only Animal.
  • An enum selector + exhaustive switch is safer than string keys returning null.
  • Adding a new animal means editing exactly one method — the factory — not every call site.
public interface Animal { void eat(); }

public class Cat   implements Animal { public void eat() { System.out.println("Cat is eating...");   } }
public class Dog   implements Animal { public void eat() { System.out.println("Dog is eating...");   } }
public class Lion  implements Animal { public void eat() { System.out.println("Lion is eating...");  } }
public class Tiger implements Animal { public void eat() { System.out.println("Tiger is eating..."); } }

public enum AnimalType { DOG, CAT, TIGER, LION }

public class AnimalFactory {
    public static Animal getAnimal(AnimalType type) {
        return switch (type) {           // exhaustive over the enum
            case DOG   -> new Dog();
            case CAT   -> new Cat();
            case LION  -> new Lion();
            case TIGER -> new Tiger();
        };
    }
}

// caller -- never names a concrete class
Animal a = AnimalFactory.getAnimal(AnimalType.LION);
a.eat();
Animal b = AnimalFactory.getAnimal(AnimalType.CAT);
b.eat();

AdvancedThe original course factory used an if/else chain that returned null for an unknown type — a classic time-bomb, since the caller then dereferences null and gets a NullPointerException far from the real cause. An exhaustive switch over an enum removes the possibility entirely; if you must accept an open-ended key, throw IllegalArgumentException immediately rather than returning null.
Pitfall Returning null from a factory for an unrecognised argument. Callers rarely null-check factory results, so the failure surfaces later as an NullPointerException. Fail fast with an exception, or make the input type-safe so "unrecognised" cannot happen.

07. Abstract Factory pattern example — car factories

Objective: Use an Abstract Factory — a "factory of factories" — to produce whole families of related products without binding the client to a specific family.

The Abstract Factory (creational) is a layer above the plain factory. Instead of one factory choosing among products, an AbstractFactory interface has several concrete implementations, each producing a family of related objects. Here the families are drivetrains: an ElectricCarFactory and a PetrolCarFactory, each of which can build the same set of brands (Ford, Toyota). A FactoryProducer hands you the right factory; you then ask that factory for a specific product.

Participants: AbstractFactory (interface, getCar), Concrete Factories (ElectricCarFactory, PetrolCarFactory), the Product interface (Car), and Concrete Products in each family (ElectricFord, ElectricToyota, PetrolFord, PetrolToyota). The point is consistency: once you pick the electric family, every car you get from it is electric — the pattern makes mixing families across a request impossible.

  • Abstract Factory = factory of factories; it creates families of related products that are meant to be used together.
  • Client depends on two abstractions (AbstractFactory and Car) and picks a family once; product consistency is guaranteed thereafter.
  • Contrast with Factory Method: Factory Method varies one product; Abstract Factory varies a whole family of products.
public interface Car { void assemble(); }

public class ElectricFord   implements Car { public void assemble() { System.out.println("Assembling an electric Ford...");   } }
public class ElectricToyota implements Car { public void assemble() { System.out.println("Assembling an electric Toyota..."); } }
public class PetrolFord     implements Car { public void assemble() { System.out.println("Assembling a petrol Ford...");      } }
public class PetrolToyota   implements Car { public void assemble() { System.out.println("Assembling a petrol Toyota...");    } }

public interface AbstractFactory { Car getCar(String brand); }

public class ElectricCarFactory implements AbstractFactory {
    public Car getCar(String brand) {
        return brand.equals("FORD") ? new ElectricFord() : new ElectricToyota();
    }
}
public class PetrolCarFactory implements AbstractFactory {
    public Car getCar(String brand) {
        return brand.equals("FORD") ? new PetrolFord() : new PetrolToyota();
    }
}

public class FactoryProducer {                     // the factory of factories
    public static AbstractFactory getFactory(String family) {
        if (family.equals("ELECTRIC")) return new ElectricCarFactory();
        if (family.equals("PETROL"))   return new PetrolCarFactory();
        throw new IllegalArgumentException("Unknown family: " + family);
    }
}

// caller
AbstractFactory electric = FactoryProducer.getFactory("ELECTRIC");
electric.getCar("TOYOTA").assemble();

08. Factory pattern & IoC frameworks

Objective: See that a dependency-injection container is a generalised, configurable factory — and how that realises the Dependency Inversion Principle at application scale.

Once you accept that creation belongs behind a factory, an IoC (Inversion of Control) container is the logical endpoint. Spring's BeanFactory and its richer subtype ApplicationContext are, at heart, extremely sophisticated factories: you declare what objects (beans) exist and how they depend on each other, and the container instantiates them, wires the dependencies, manages their scope/lifecycle, and hands them out on request via getBean(...).

This is Inversion of Control: instead of your code calling new and pulling its collaborators, the framework pushes ready-wired collaborators into your objects (usually through constructors). Your classes depend only on interfaces — a direct application of the Dependency Inversion Principle — and the container is the single factory that knows the concrete implementations. Swapping an implementation (e.g. a real payment gateway for a stub in tests) becomes a configuration change, not a code change.

  • A DI container (ApplicationContext/BeanFactory) is a configurable factory: it builds, wires, scopes, and serves objects.
  • Inversion of Control: the framework creates and injects collaborators; your code stops calling new for its dependencies.
  • This operationalises the Dependency Inversion Principle — classes depend on abstractions; the container supplies the concretions.

8. Creational patterns II: Builder & Prototype

Two more creational patterns, each attacking a different construction problem. Builder assembles a complex object step by step through a fluent API, replacing telescoping constructors and long ambiguous parameter lists. Prototype creates new objects by cloning an existing one instead of building it from nothing — handy when construction is expensive or a fully configured template already exists.

01. Builder pattern introduction

Objective: Understand the problem Builder solves — separating the construction of a complex object from its representation so the same building steps can produce a fully-configured, often immutable, product.

When a class has many fields — some required, most optional — the naive solutions are both bad. Telescoping constructors pile up overloads (User(a), User(a,b), User(a,b,c)…) until callers can no longer tell which literal maps to which parameter, and new User("jo", "x@y", 29, true) compiles happily even when the arguments are in the wrong order. The JavaBeans alternative (a no-arg constructor plus setters) leaks a half-built, mutable object between calls and defeats immutability.

Builder (a creational pattern) introduces a separate builder object that accumulates configuration through named, chainable methods and hands back the finished product from a single build() call. The classic Gang-of-Four form names four roles — Director, Builder, ConcreteBuilder, and Product — but the idiom that dominates modern Java collapses the director into the client's fluent call chain and nests the builder inside the product.

  • Problem solved: telescoping constructors and long, order-sensitive parameter lists.
  • Each configuration method returns the builder (this) so calls chain fluently.
  • Named methods make each argument self-documenting at the call site.
  • The product can be immutable: all fields are set once, inside build().
  • GoF category: creational.
Pitfall Reaching for Builder on a two- or three-field value object is over-engineering — a plain constructor or a record is clearer. The pattern earns its keep only when the parameter list is long, mostly optional, or error-prone to order.

02. Builder example — avoiding telescoping constructors

Objective: Implement the canonical Java builder — an immutable User with a static nested Builder, required fields passed to the builder's constructor and optional ones set through fluent methods.

First, the anti-pattern the builder replaces. With overloaded constructors the call site is opaque, and a wrong argument order still compiles:

// Telescoping constructors -- which literal is which?
new User("jsmith", "j@mail.com");
new User("jsmith", "j@mail.com", 29);
new User("jsmith", "j@mail.com", 29, "555-0100");
new User("jsmith", "j@mail.com", 29, "555-0100", true);
// new User("jsmith", "j@mail.com", true, 29);  // silently wrong if signatures line up

The builder makes username and email mandatory (they are arguments to the builder itself) and leaves the rest optional with sensible defaults. Every configuration method returns this, and the User constructor is private so the builder is the only way in.

  • Required fields go through the Builder constructor; optional ones through fluent setters.
  • Each setter returns this to enable chaining.
  • User's fields are final and its constructor is private — the object is immutable.
  • Unset optional fields fall back to defaults declared on the builder.
public final class User {
    private final String username;   // required
    private final String email;      // required
    private final int age;           // optional
    private final String phone;      // optional
    private final boolean newsletter;// optional

    private User(Builder b) {
        this.username   = b.username;
        this.email      = b.email;
        this.age        = b.age;
        this.phone      = b.phone;
        this.newsletter = b.newsletter;
    }

    public static Builder builder(String username, String email) {
        return new Builder(username, email);
    }

    @Override public String toString() {
        return "User{username=" + username + ", email=" + email
             + ", age=" + age + ", phone=" + phone
             + ", newsletter=" + newsletter + "}";
    }

    public static class Builder {
        private final String username;      // required
        private final String email;         // required
        private int age = 0;                // defaults
        private String phone = "n/a";
        private boolean newsletter = false;

        Builder(String username, String email) {
            this.username = username;
            this.email = email;
        }
        public Builder age(int age)             { this.age = age; return this; }
        public Builder phone(String phone)      { this.phone = phone; return this; }
        public Builder newsletter(boolean on)   { this.newsletter = on; return this; }
        public User build()                     { return new User(this); }
    }
}

03. Builder exercise & solution

Objective: Extend the builder with validation — enforce required fields and invariants inside build() so an invalid object can never be constructed.

A builder is the natural place to validate, because build() is the single choke point where the object comes into existence. Here a Pizza requires a size and caps toppings at four; anything else throws IllegalStateException before a Pizza is ever returned. Because validation lives in build(), the constructed object is guaranteed valid for its whole lifetime.

  • Centralize invariants in build() — the one place every product passes through.
  • Fail fast: throw rather than return a half-valid object.
  • Accumulate repeatable values (toppings) with an addX() method backed by a list.
  • Copy mutable collections into the product so the builder can be reused safely.
import java.util.ArrayList;
import java.util.List;

public final class Pizza {
    public enum Size { SMALL, MEDIUM, LARGE }

    private final Size size;
    private final boolean cheese;
    private final List<String> toppings;

    private Pizza(Builder b) {
        this.size = b.size;
        this.cheese = b.cheese;
        this.toppings = List.copyOf(b.toppings);   // immutable snapshot
    }

    @Override public String toString() {
        return "Pizza{size=" + size + ", cheese=" + cheese
             + ", toppings=" + toppings + "}";
    }

    public static class Builder {
        private Size size;                              // required
        private boolean cheese = true;                  // default
        private final List<String> toppings = new ArrayList<>();

        public Builder size(Size size)      { this.size = size; return this; }
        public Builder noCheese()           { this.cheese = false; return this; }
        public Builder addTopping(String t) { toppings.add(t); return this; }

        public Pizza build() {
            if (size == null)
                throw new IllegalStateException("size is required");
            if (toppings.size() > 4)
                throw new IllegalStateException("at most 4 toppings");
            return new Pizza(this);
        }
    }
}

// --- client ---
Pizza p = new Pizza.Builder()
        .size(Pizza.Size.LARGE)
        .addTopping("mushroom")
        .addTopping("olive")
        .build();
System.out.println(p);

try {
    new Pizza.Builder().addTopping("ham").build();   // no size set
} catch (IllegalStateException e) {
    System.out.println("Rejected: " + e.getMessage());
}

AdvancedTo make required fields a compile-time guarantee rather than a runtime check, use a staged (step) builder: each step is a small interface that returns the next step's interface, and only the final step exposes build(). The type system then makes it impossible to call build() before setting the mandatory fields — at the cost of far more boilerplate.
Pitfall Validating in the Pizza constructor instead of build() still works, but validating in the fluent setters does not: a caller can set fields in any order, so a cross-field invariant (like "toppings require a size") can only be checked once, at the end.

04. Applying the Builder pattern in a Vaadin use case

Objective: Recognize the builder idiom in the wild — the JDK and UI frameworks such as Vaadin lean on fluent, chained construction constantly.

Fluent builders are everywhere in real Java. The JDK ships StringBuilder and StringJoiner for text, Stream.Builder for streams, Calendar.Builder and DateTimeFormatterBuilder for date/time, and HttpRequest.newBuilder() in the modern HTTP client. UI toolkits use the same shape: in Vaadin you assemble a component and chain configuration onto it — new Button("Save", e -> save()), then button.addClickShortcut(Key.ENTER) — so the layout code reads top-to-bottom like a description of the screen. The value is the same in every case: readable, order-tolerant construction of an otherwise fiddly object.

  • StringBuilder, StringJoiner, Stream.Builder, HttpRequest.Builder are all builders.
  • Fluent framework APIs (Vaadin components, query DSLs) read as a declarative description.
  • A chain of calls each returning this (or the next stage) is the tell-tale sign of a builder.
  • The payoff is call-site readability, not raw performance.

05. What is the Prototype pattern?

Objective: Understand Prototype — creating new objects by copying a fully-configured existing instance rather than constructing one from scratch with new.

Prototype is a creational pattern: instead of instantiating a class directly, the client asks an existing object (the prototype) to produce a copy of itself. The roles are a Prototype declaring a clone operation, one or more ConcretePrototypes that implement it, and a Client that clones without knowing the concrete class. It pays off when construction is expensive (heavy initialization, database or network setup) or when you already hold a template whose configuration you want to duplicate and tweak.

Java offers built-in support through Cloneable and Object.clone(), but the essence of the pattern is simply "each object knows how to copy itself" — you can equally expose your own copy() or cloneObject() method, which sidesteps Cloneable's well-known flaws (covered in lecture 07).

  • Create objects by cloning a prototype instead of calling a constructor.
  • Participants: Prototype (declares clone), ConcretePrototype, Client.
  • Useful when construction is costly or a configured template should be duplicated.
  • Java support: Cloneable + Object.clone() — or a hand-written copy method.
// The minimal idea: an object that copies itself
class Point implements Cloneable {
    int x, y;
    @Override public Point clone() {
        try { return (Point) super.clone(); }
        catch (CloneNotSupportedException e) { throw new AssertionError(e); }
    }
}
Pitfall Prototype is not "just clone()." The point is that the client duplicates an object without depending on its concrete class — if your client already knows it holds a Square and could as easily call the constructor, the pattern buys you nothing.

06. Prototype example — cloning shapes

Objective: Walk a concrete Prototype implementation — an abstract Shape with a cloneObject() operation that Rectangle and Square implement by returning fresh copies of themselves.

The abstract Shape holds the shared state (width, height) and declares two abstract operations: draw() and cloneObject(). Each concrete subclass implements cloneObject() to return a new instance of its own type carrying the same field values. The client calls cloneObject() against the Shape reference and gets back the right concrete type polymorphically — it never names Square or Rectangle at the clone site.

  • Shape is the Prototype; Rectangle/Square are ConcretePrototypes.
  • cloneObject() returns a Shape, so the client stays decoupled from concrete types.
  • Here each shape holds only primitives, so a plain field-by-field copy is a correct, complete copy.
  • A registry of pre-built prototypes (see Advanced) turns this into a lightweight object factory.
public abstract class Shape {
    protected int width;
    protected int height;

    public Shape(int width, int height) {
        this.width = width;
        this.height = height;
    }

    public abstract void draw();
    public abstract Shape cloneObject();   // the prototype operation
}

public class Rectangle extends Shape {
    public Rectangle(int width, int height) { super(width, height); }

    @Override public void draw() {
        System.out.println("Rectangle with width: " + width + " height: " + height);
    }
    @Override public Shape cloneObject() {
        return new Rectangle(width, height);
    }
}

public class Square extends Shape {
    public Square(int width, int height) { super(width, height); }

    @Override public void draw() {
        System.out.println("Square with width: " + width + " height: " + height);
    }
    @Override public Shape cloneObject() {
        return new Square(width, height);
    }
}

// --- client ---
public class App {
    public static void main(String[] args) {
        Shape square = new Square(10, 5);
        square.draw();

        Shape squareClone = square.cloneObject();   // a copy, not the same object
        squareClone.draw();

        Shape rect = new Rectangle(8, 3);
        rect.cloneObject().draw();
    }
}

Pitfall This example is safe only because the fields are primitives. The moment a shape holds a mutable reference (say a List<Point> of vertices), returning new Square(width, height) must also copy that list — otherwise clones share it. That is the shallow-vs-deep problem of the next lecture.

07. Prototype pattern & inheritance (deep vs shallow copy)

Objective: Distinguish shallow from deep copies, and see when cloning duplicates a nested object versus merely copying a reference to it.

A shallow copy duplicates the top-level object but copies each field value as-is — for a reference field, that value is the reference, so the original and the copy end up pointing at the same nested object. Object.clone() does exactly this. A deep copy also duplicates the nested objects, so the two graphs are fully independent. The distinction only matters for mutable nested state: sharing an immutable String is harmless, but sharing a mutable Address means a change through one owner is visible through the other (aliasing).

  • Shallow copy: nested references are shared — mutating through the clone affects the original.
  • Deep copy: nested objects are duplicated — the two are independent.
  • Only mutable nested state is at risk; immutable values (String, boxed primitives) are safe to share.
  • A copy constructor / copy method makes the shallow-vs-deep decision explicit, field by field.
class Address {
    String city;
    Address(String city) { this.city = city; }
}

class Person implements Cloneable {
    String name;        // immutable -- safe to share
    Address address;    // mutable -- the interesting field

    Person(String name, Address address) {
        this.name = name;
        this.address = address;
    }

    // Shallow: Object.clone() copies the address REFERENCE, not the Address
    public Person shallowCopy() {
        try { return (Person) super.clone(); }
        catch (CloneNotSupportedException e) { throw new AssertionError(e); }
    }

    // Deep: duplicate the nested Address too
    public Person deepCopy() {
        return new Person(this.name, new Address(this.address.city));
    }
}

// --- client ---
Person original = new Person("Ada", new Address("London"));
Person shallow  = original.shallowCopy();
shallow.address.city = "Paris";            // mutate the clone's address...
System.out.println(original.address.city); // ...original changed too (shared)

Person source = new Person("Grace", new Address("New York"));
Person deep   = source.deepCopy();
deep.address.city = "Boston";              // mutate the deep clone...
System.out.println(source.address.city);   // ...original untouched
Console outputParis New York

Source: reimplemented as an original illustration.

AdvancedDeep-copying a whole object graph by hand is tedious and error-prone once the graph is large or cyclic. Two escape hatches: serialize-and-deserialize (e.g. to JSON or via Serializable) to snapshot the entire graph, or a copy library. Both are slower than a targeted copy constructor and each has caveats (transient fields, non-serializable members, cycles), so reserve them for genuinely deep graphs.
Pitfall Cloneable's contract is broken: it is a marker interface with no clone() method, Object.clone() is protected, it bypasses constructors (so final fields cannot be set), and it silently produces shallow copies. Effective Java's advice: prefer a copy constructor or a static copy factory (Person(Person other) / Person.copyOf(other)) over implementing Cloneable.

BUILDER (creational) -- separate construction from representation

  * Required args via Builder constructor; optional args via withX()/setters
  * Validate invariants inside build() (the single choke point)
  * Kills telescoping constructors + long, order-sensitive parameter lists
  * JDK builders: StringBuilder, StringJoiner, Stream.Builder, HttpRequest.Builder
  * Overkill for 2-3 fields -- use a constructor or record

9. Behavioral patterns I: Strategy & Observer

Behavioral patterns describe how objects collaborate — how responsibilities are assigned and how algorithms vary at run time. We start with the two most useful ones: Strategy (swap an algorithm behind an interface) and Observer (a one-to-many change-notification link). Both trade inheritance for composition and lean hard on programming to an interface.

01. Behavioral design patterns — overview

Objective: Place the behavioral family within the GoF catalogue and preview the patterns covered next.

The Gang-of-Four catalogue splits its 23 patterns into three families: creational (how objects are made), structural (how objects are composed), and behavioral (how objects communicate and how responsibilities and algorithms are distributed at run time). Behavioral patterns are about interaction and the flow of control between collaborating objects.

The GoF behavioral patterns include Strategy, Observer, Command, Iterator, Template Method, and Visitor (Null Object is a widely used pattern but not one of the original 23). This section covers Strategy and Observer; later sections pick up the rest.

  • Behavioral = communication and responsibility; creational = construction; structural = composition.
  • They favour composition and delegation over deep inheritance hierarchies.
  • Most rest on one idea: depend on an interface, not a concrete type, so behaviour can vary independently.

02. What is the Strategy pattern?

Objective: Define Strategy — a family of interchangeable algorithms behind one interface, chosen at run time — and name its participants.

The Strategy pattern (behavioral) encapsulates a family of algorithms, puts each behind a common interface, and makes them interchangeable. The object that needs the behaviour — the Context — holds a reference to a Strategy and delegates to it, instead of hard-coding the algorithm or branching on a type flag.

Participants: the Strategy interface declares the operation; each ConcreteStrategy implements one algorithm; the Context is configured with a strategy (usually via a setter or constructor) and calls it without knowing which concrete class it holds. Because the field can be reassigned, the behaviour can change at run time.

  • Participants: Strategy (interface), ConcreteStrategy (implementations), Context (delegates to a strategy).
  • Uses composition (Context has-a Strategy), not inheritance — swap the object, swap the behaviour.
  • Choosing/replacing the strategy is a run-time decision, not a compile-time one.
// Strategy interface -- the abstraction the Context depends on
public interface Strategy {
    void execute(int num1, int num2);
}

// Context: holds a Strategy and delegates; knows nothing concrete
public class OperationManager {
    private Strategy strategy;                 // composition
    public void setStrategy(Strategy strategy) { this.strategy = strategy; }
    public void execute(int a, int b) { strategy.execute(a, b); }   // delegate
}

03. Strategy example — arithmetic operations

Objective: Walk a runnable Strategy: an OperationManager Context that executes any of four arithmetic algorithms selected at run time.

Each arithmetic operation is a ConcreteStrategy implementing Strategy.execute(int, int). The OperationManager Context stores one strategy and delegates to it; reassigning the field with setStrategy(...) switches the algorithm on the fly. Adding a new operation means adding a class — the Context never changes.

  • Addition, Subtraction, Multiplication, Division each implement one algorithm.
  • The Context is oblivious to which concrete strategy it holds — it just calls execute.
  • Integer division: 10 / 5 is 2 (truncating integer arithmetic).
public class Addition implements Strategy {

}
public class Multiplication implements Strategy {

}
public class Division implements Strategy {

}

04. Why use the Strategy pattern?

Objective: Contrast the strategy design with a type-switching anti-pattern and connect it to the Open/Closed Principle.

Without Strategy, the same requirement collapses into a Context that inspects the object's type and branches — an if/else (or switch) chain over instanceof. Every new operation forces you to edit and re-test that method, which violates the Open/Closed Principle: the class is not closed for modification. Worse, an unhandled type falls through silently and does nothing — a bug the compiler cannot catch.

The strategy version is open for extension, closed for modification: a new algorithm is a new class implementing Strategy; the Context is never touched. Polymorphic delegation replaces the conditional entirely.

  • Type-switch anti-pattern: one big method grows a branch per type — a modification magnet.
  • Missing branches fail silently; Strategy makes "does nothing" impossible.
  • Strategy upholds OCP; it also removes the Context's dependency on concrete classes (DIP).
// ANTI-PATTERN: the Context branches on the concrete type
public class OperationManagerBAD {
    public void execute(Object o, int a, int b) {
        if (o instanceof Addition)
            ((Addition) o).execute(a, b);
        else if (o instanceof Multiplication)
            ((Multiplication) o).execute(a, b);
        // Subtraction? Division? Every new op needs another branch HERE.
    }
}

public class App {
    public static void main(String[] args) {
        OperationManagerBAD manager = new OperationManagerBAD();
        manager.execute(new Multiplication(), 10, 5);  // matches -> prints 50
        manager.execute(new Subtraction(),    10, 5);  // no branch -> silent!
    }
}
Console output50 (the Subtraction call matches no branch, so it prints nothing at all)

Source: reimplemented from the course example.

AdvancedA switch on an enum with an exhaustiveness check catches the "missing branch" bug at compile time, but it still centralises behaviour and must be edited for each new case. Strategy distributes the behaviour into the types themselves, so extension needs no edit anywhere — the difference between OCP-violating and OCP-respecting design.
Pitfall Reaching for instanceof chains or downcasts is the tell-tale sign a Strategy (or polymorphism) is missing. Each added type multiplies the branches you must keep in sync across every such method.

06. Observer pattern introduction

Objective: Define Observer — a one-to-many notification link — and its participants, and distinguish push from pull.

The Observer pattern (behavioral) defines a one-to-many dependency: when one object (the Subject) changes state, all of its registered dependants (the Observers) are notified automatically. The Subject exposes attach/detach/notify operations and keeps a collection of observers; each Observer implements an update method the Subject calls. The Subject knows its observers only through that interface, so the coupling stays loose.

Two delivery models: in push, the Subject sends the changed data as arguments to update(...); in pull, it merely signals "something changed" and each Observer queries the Subject for what it needs. Push is simpler; pull decouples the Observer from the exact shape of the notification.

  • Participants: Subject (attach/detach/notify + observer list) and Observer (update).
  • One-to-many: a single state change fans out to every registered observer.
  • Loose coupling — the Subject depends only on the Observer interface, so observers can be added or removed independently.
  • Push passes data in update(...); pull passes a signal and lets observers fetch.
public interface Observer { void update(int pressure, int temperature, int humidity); }

public interface Subject {
    void addObserver(Observer o);
    void removeObserver(Observer o);
    void notifyAllObservers();          // fan-out to every registered observer
}

07. Observer example — weather station

Objective: Walk a runnable Observer: a WeatherStation Subject that pushes new readings to registered WeatherObservers.

The WeatherStation stores pressure, temperature, and humidity plus a List of observers. Every setter mutates one field and then calls notifyAllObservers(), which loops the list and calls update(...) on each — a push model. A WeatherObserver registers itself with the Subject in its constructor and prints the readings it receives.

Fields default to 0, so the first notification carries the untouched fields alongside the one just set. Because the Subject stores observers in a List, notifications follow registration order — the order observers were added. (The JDK does not otherwise promise any particular observer ordering; a List just happens to preserve insertion order.)

  • Each setter is a state change that triggers a fan-out notification.
  • Push model: the changed data travels as update(pressure, temperature, humidity) arguments.
  • Unset fields are 0, so setHumidity(100) notifies 0-0-100.
  • Notification order = registration (list) order; do not rely on any other ordering.
public class WeatherStation implements Subject {
    private int pressure, temperature, humidity;
    private final List<Observer> observers = new ArrayList<>();

    public void addObserver(Observer o)    { observers.add(o); }
    public void removeObserver(Observer o) { observers.remove(o); }
    public void notifyAllObservers() {
        for (Observer o : observers) o.update(pressure, temperature, humidity);
    }
    public void setPressure(int p)    { pressure = p;    notifyAllObservers(); }
    public void setTemperature(int t) { temperature = t; notifyAllObservers(); }
    public void setHumidity(int h)    { humidity = h;    notifyAllObservers(); }
}

public class WeatherObserver implements Observer {
    public WeatherObserver(Subject subject) { subject.addObserver(this); }
    public void update(int pressure, int temperature, int humidity) {
        System.out.println(pressure + "-" + temperature + "-" + humidity);
    }
}

public class App {
    public static void main(String[] args) {
        WeatherStation station = new WeatherStation();
        new WeatherObserver(station);      // registers itself
        station.setHumidity(100);
        station.setPressure(200);
        station.setTemperature(300);
    }
}

AdvancedThe course also sketches a DataPresenter interface to abstract how readings are displayed — a further decoupling so an observer can render to console, a chart, or a log without changing the notification wiring. 08. Observer with Swing UI components

Objective: Recognise that Swing's event-listener mechanism is the Observer pattern, and trace the click wiring.

Every Swing listener API is Observer under another name. This runs a GUI window rather than printing to stdout, so we describe the interaction instead of tracing console output.

In the sample, Toolbar owns the button and implements ActionListener. Rather than reach across to the text area directly, it forwards the event through its own small ClickListener interface.   • Swing mapping: JButton = Subject, ActionListener = Observer, addActionListener = attach.
  • The click fan-out (actionPerformed) is exactly notifyAllObservers for one event type.
  • A custom ClickListener interface decouples the toolbar from the panel — Observer layered on Observer.
  • GUI, not console: verify by clicking and watching the window, not by reading stdout.

09. Observer exercise & solution — algorithmic trading

Objective: Apply Observer to a market feed: a StockExchange Subject whose price changes drive buy/sell listeners.

StockExchange is the Subject; BuyStockListener and SellStockListener are Observers that each react only to the prices they care about — buy when the price is low, sell when it is high, ignore everything in between. Every price update fans out to both observers via notifyAllObserver(). The original course driver used a random walk in an endless loop, which is non-deterministic; below we feed a fixed price sequence so the output is reproducible and can be traced by hand.

  • Subject = StockExchange; observers = BuyStockListener (price < 95) and SellStockListener (price > 105).
  • Each observer filters: it acts only on prices in its range, so most updates produce no output.
  • Observers fire in registration order — Buy was added before Sell.
public class StockExchange implements Subject {

}

public class BuyStockListener implements Observer {

}
public class SellStockListener implements Observer {
    public void update(float price) {
        if (price > 105) System.out.println("Selling stock because the price is too high...");
    }
}

public class App {

10. Behavioral patterns II: Command, Iterator & Template Method

Three more behavioral Gang-of-Four patterns, all about how objects collaborate at run time. Command turns a request into a first-class object you can store, queue, log, and undo. Iterator gives sequential access to a collection without exposing how it is stored. Template Method fixes the skeleton of an algorithm in a base class and lets subclasses fill in the varying steps.

01. What is the Command pattern?

Objective: Understand how encapsulating a request as an object decouples the code that triggers an operation from the code that performs it.

The Command pattern (behavioral) wraps a request in an object that exposes a single execute() method. The object that fires the request — the invoker — holds only the Command interface and knows nothing about what the command actually does. That indirection is what makes requests storable: because a command is a value, you can put it in a list, hand it to another thread, log it, replay it, or reverse it.

The five participants are: Command (the interface with execute()), ConcreteCommand (binds a receiver to one action and remembers any arguments), Receiver (the object that does the real work), Invoker (triggers commands, optionally keeping a history), and Client (wires a concrete command to its receiver and hands it to the invoker).

  • Encapsulate a request as an object → decouples invoker from receiver.
  • Enables queuing, scheduling, logging, macro commands, and undo/redo.
  • The command stores its own parameters, so invocation can happen much later.
  • The invoker depends only on Command, never on a concrete command.
// Encapsulate a request as an object with one method.
public interface Command {
    void execute();
}

// The invoker holds Commands and triggers them -- it knows
// nothing about what each command actually does.
class Invoker {
    private final List<Command> queue = new ArrayList<>();
    void store(Command c) { queue.add(c); }
    void runAll()         { queue.forEach(Command::execute); }
}

Source: reimplemented as an original illustration.

AdvancedUndo/redo is the classic pay-off: add an undo() method whose implementation reverses execute(). Because reversing usually needs the pre-execution state, the command either stores that state itself or captures a Memento. Commands also compose: a macro command is just a command holding a List<Command> and calling each in turn (Composite + Command).
Pitfall Do not push the real logic into the command. A ConcreteCommand should be a thin adapter that captures arguments and delegates to a receiver; if it implements the behaviour itself you have merged Command with Receiver and lost the reuse and testability the pattern buys you.

02. Command example — light switch

Objective: Implement Command end to end — a Light receiver, two concrete commands, and a Switcher invoker that stores and replays them.

The Light is the receiver; TurnOnCommand and TurnOffCommand are ConcreteCommands that each hold a reference to a light and call one of its methods. Switcher is the invoker: it collects commands in a list and later fires them in order. Notice that main (the client) adds the off command before the on command, so that is the order the output appears in — the invoker just walks the list.

  • Each command captures its receiver in the constructor and forwards in execute().
  • The invoker (Switcher) is coded against Command, so new actions need no change to it.
  • Execution order follows the order commands were stored, not created.
// Receiver -- does the real work
class Light {
    void turnOn()  { System.out.println("Lights are on..."); }

}

}
class TurnOffCommand implements Command {

    TurnOffCommand(Light light) { this.light = light; }
    public void execute() { light.turnOff(); }
}

// Invoker -- stores commands, replays them later
class Switcher {
    private final List<Command> commands = new ArrayList<>();
    void addCommand(Command c) { commands.add(c); }
    void executeCommands()     { commands.forEach(Command::execute); }

03. Queuing model with the Command pattern

Objective: See why "a request is an object" matters — commands can be produced now and consumed later, even on another thread, through a queue.

Here a producer thread fills a thread-safe BlockingQueue<Command> with TaskSolver commands (each wrapping a Task receiver), while a consumer thread takes them one at a time and calls execute(). This is the design behind schedulers, thread pools, and job queues: the command packages a unit of work that some other component runs whenever it is ready. Because ArrayBlockingQueue is FIFO, the tasks run in the order 1…10; the consumer's one-second pause just paces the drain.

  • A command decouples when work is created from when it runs.
  • BlockingQueue gives safe hand-off between producer and consumer threads.
  • The JDK already uses this idea: Runnable/Callable submitted to an ExecutorService are commands.
// Receiver
class Task {
    private final int id;
    Task(int id) { this.id = id; }
    void solveProblem() { System.out.println("Solving problem " + id); }
}

// ConcreteCommand
class TaskSolver implements Command {
    private final Task task;
    TaskSolver(Task task) { this.task = task; }
    public void execute() { task.solveProblem(); }
}

public class App {
    public static void main(String[] args) {
        BlockingQueue<Command> queue = new ArrayBlockingQueue<>(10);

        // producer: create work now
        Thread producer = new Thread(() -> {
            for (int i = 1; i <= 10; i++) queue.add(new TaskSolver(new Task(i)));
        });

        // consumer: run it later, one per second
        Thread consumer = new Thread(() -> {
            try {
                for (int i = 0; i < 10; i++) {
                    Thread.sleep(1000);
                    queue.take().execute();
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        producer.start();
        consumer.start();
    }
}

04. Iterator pattern introduction

Objective: Understand how Iterator provides sequential access to an aggregate's elements without leaking how those elements are stored.

The Iterator pattern (behavioral) separates traversal from the collection. The Iterator role exposes just hasNext() and next(); the Aggregate (container) exposes a factory method that hands back an iterator. Client code walks the elements one by one and never sees whether the backing store is an array, a linked list, or a tree — you can change the internal representation without touching any caller.

Java builds this pattern into the platform: java.util.Iterator<T> is the Iterator role and java.lang.Iterable<T> is the Aggregate role. Implementing Iterable is what makes a type usable in the enhanced for loop.

  • Participants: Iterator (hasNext/next) and Aggregate (returns an iterator).
  • Hides the collection's internal representation from its clients.
  • Each iterator carries its own cursor, so multiple traversals can run independently.
  • The JDK equivalents are Iterator<T> and Iterable<T>.
// Provide sequential access without exposing the backing store.
public interface Iterator<T> {
    boolean hasNext();
    T next();
}

interface Aggregate<T> {
    Iterator<T> iterator();   // factory method for a fresh cursor
}

Source: reimplemented as an original illustration.

AdvancedThe JDK's Iterator adds an optional remove() and, since Java 8, a default forEachRemaining(). ListIterator extends it with backward traversal (hasPrevious/previous) and index-aware mutation. If you also implement Iterable<T>, your type drops straight into the enhanced for loop and into StreamSupport.stream(...).
Pitfall Do not add index/positioning methods to the aggregate itself (get(i), size()) and call it an iterator — that re-exposes the representation the pattern is meant to hide. Keep the cursor in a separate iterator object.

05. Iterator example — name repository

Objective: Build a custom iterator over an array-backed repository so callers traverse names without seeing the array.

NameRepository is the ConcreteAggregate: it holds a String[] and returns a fresh iterator from iterator(). NameIterator is the ConcreteIterator, implemented as an inner class so it can read the enclosing array directly while keeping its own index cursor. The client loops on hasNext()/next() and never touches the array — swapping the backing store for a List or a database cursor would leave every caller unchanged.

  • NameRepository.iterator() is the aggregate's factory method.
  • The inner-class iterator holds the index; each call to iterator() starts fresh at 0.
  • Clients depend on the Iterator<String> abstraction only.
// Aggregate -- exposes an iterator, hides its String[] store
class NameRepository {
    private final String[] names = { "Adam", "Joe", "John", "Sarah" };

    Iterator<String> iterator() {
        return new NameIterator();
    }

    private class NameIterator implements Iterator<String> {
        private int index;
        public boolean hasNext() { return index < names.length; }
        public String next()     { return names[index++]; }
    }
}

public class App {
    public static void main(String[] args) {
        NameRepository repo = new NameRepository();
        Iterator<String> it = repo.iterator();

            System.out.println(it.next());
        }
    }

06. External vs internal iteration

Objective: Contrast client-driven (external) iteration with collection-driven (internal) iteration and weigh the trade-offs.

With external iteration the client controls the loop: it calls hasNext()/next() (or uses the enhanced for), decides when to advance, and can break, continue, or mutate freely. With internal iteration the collection drives the traversal and you pass in the operation — forEach and the Stream API are the examples. Internal iteration is more concise and lets the library optimize (laziness, short-circuiting, parallelism), at the cost of finer step-by-step control.

  • External: maximum control (break/continue, index math, mutation); more boilerplate.
  • Internal: concise and declarative; the library owns the loop.
  • Streams are internal and lazy — nothing runs until a terminal op, and they can short-circuit (findFirst) or go parallel.
List<String> names = List.of("Adam", "Joe", "John", "Sarah");

// External iteration -- the client drives the loop, step by step.
for (String name : names) {
    System.out.println(name);
}

// Internal iteration -- the collection drives it; you pass the action.
names.forEach(System.out::println);

// Streams are internal AND lazy: this short-circuits at the first match.
String firstJ = names.stream()
                     .filter(n -> n.startsWith("J"))
                     .findFirst()
                     .orElse("none");
System.out.println("first J-name: " + firstJ);
Console outputAdam Joe John Sarah Adam Joe John Sarah first J-name: Joe

Source: reimplemented as an original illustration.

AdvancedLaziness is the real win of internal iteration: in the stream above, filter is applied element-by-element and findFirst stops after "Joe" — "John" and "Sarah" are never tested. External loops can do this too (with an explicit break), but the stream expresses it declaratively and can also be parallelized with parallelStream().
Pitfall You cannot break, return, or throw a checked exception cleanly out of a forEach lambda, and mutating the source collection inside it risks ConcurrentModificationException. When you need early exit or in-place structural changes, reach for external iteration (an explicit loop or an Iterator with remove()).

07. Template Method pattern introduction

Objective: Understand how a base class can fix the skeleton of an algorithm while deferring individual steps to subclasses.

The Template Method pattern (behavioral) puts the invariant structure of an algorithm into one method — the template method — in an abstract base class, and calls out to primitive operations that subclasses supply. The base class owns the order of the steps; subclasses only fill in the parts that vary. This is the Hollywood Principle: "don't call us, we'll call you" — the framework (base class) calls down into your code, not the other way round.

Participants: AbstractClass (defines the final template method plus abstract primitive operations and optional hooks) and ConcreteClass (overrides the primitives). Making the template method final is what guarantees every subclass runs the steps in the same order.

  • The template method defines the algorithm's skeleton and calls primitive operations.
  • Primitive operations are abstract (must override); hooks have a default and are optional to override.
  • Inverts control (Hollywood Principle): the base class drives, subclasses supply steps.
  • Relies on inheritance — one algorithm variant per subclass, fixed at compile time.
abstract class Game {
    // Template method -- the fixed skeleton. final so subclasses can't reorder it.
    public final void play() {
        initialize();
        startPlay();
        endPlay();
    }
    protected abstract void initialize(); // primitive operations,
    protected abstract void startPlay();  // supplied by each subclass
    protected abstract void endPlay();
}

AdvancedDistinguish primitive operations (abstract — the subclass must provide them) from hooks (concrete with a sensible default — the subclass may override to alter or veto an optional step). Hooks are how a template stays flexible without forcing every subclass to implement every step.
Pitfall Template Method's skeleton is only guaranteed if the template method is final. Leave it overridable and a subclass can reorder or skip steps, breaking the invariant the pattern exists to protect.

08. Template Method example — sorting algorithm skeleton

Objective: Implement a sorting template whose fixed sort() skeleton delegates the varying steps (initialize, sorting) to subclasses while sharing common helpers.

The abstract Algorithm defines the final template method sort() = initialize() → sorting() → showResult(). The steps that differ between algorithms — the announcement and the sort loop — are abstract primitives; the parts every sort shares — swap() and showResult() — live once in the base class. BubbleSort and SelectionSort override only the primitives, so both produce a correctly sorted array through the identical skeleton.

  • sort() is final: subclasses cannot change the order of the three steps.
  • Varying steps (initialize, sorting) are abstract; shared steps (swap, showResult) are concrete in the base.
  • Code reuse: the common logic is written once, not copied into each subclass.
abstract class Algorithm {
    protected final int[] nums;
    Algorithm(int[] nums) { this.nums = nums; }

    // Template method -- fixed order, final so it can't be overridden.
    public final void sort() {
        initialize();
        sorting();
        showResult();
    }

    protected abstract void initialize();  // primitive operations
    protected abstract void sorting();

    protected void swap(int i, int j) {    // shared helper
        int t = nums[i]; nums[i] = nums[j]; nums[j] = t;
    }
    protected void showResult() {          // shared step
        System.out.println(Arrays.toString(nums));
    }
}

class BubbleSort extends Algorithm {
    BubbleSort(int[] nums) { super(nums); }
    protected void initialize() {
        System.out.println("Initializing the bubble sort algorithm...");
    }
    protected void sorting() {
        for (int i = 0; i < nums.length - 1; i++)
            for (int j = 0; j < nums.length - i - 1; j++)
                if (nums[j] > nums[j + 1]) swap(j, j + 1);
    }
}

class SelectionSort extends Algorithm {
    SelectionSort(int[] nums) { super(nums); }
    protected void initialize() {
        System.out.println("Initializing the selection sort algorithm...");
    }
    protected void sorting() {
        for (int i = 0; i < nums.length - 1; i++) {
            int min = i;
            for (int j = i + 1; j < nums.length; j++)
                if (nums[j] < nums[min]) min = j;
            if (min != i) swap(i, min);
        }
    }
}

public class App {
    public static void main(String[] args) {
        int[] data = {1, 5, 3, -2, 10, 12, 8, 6, 5};
        new BubbleSort(data.clone()).sort();
        new SelectionSort(data.clone()).sort();
    }
}
Console outputInitializing the bubble sort algorithm... [-2, 1, 3, 5, 5, 6, 8, 10, 12] Initializing the selection sort algorithm... [-2, 1, 3, 5, 5, 6, 8, 10, 12]

Source: reimplemented from the course example.

AdvancedThe shared swap() and showResult() are the reuse pay-off of Template Method: write the boilerplate once, vary only the algorithm. Note data.clone() — each subclass sorts a copy, so the second sort sees the original unsorted input rather than the already-sorted array left by the first.
Pitfall Template Method binds you to inheritance: an object is a BubbleSort for life and you cannot switch its algorithm at run time. When you need to swap behaviour dynamically, prefer Strategy (composition) — see the next lecture's pitfall.

The exercise: model tea and coffee, which share a recipe (boil water → brew → pour → add condiments) but differ in the brew and condiment steps.   • Same skeleton (prepare()), two concrete beverages varying only the primitives.
  • The wantsCondiments() hook lets a subclass conditionally skip an optional step.
  • Because Coffee returns false, its output omits the condiment line.

11. Behavioral patterns III: Null Object & Visitor

Two more behavioral patterns that reshape how callers deal with structure. Null Object replaces a null return with a polymorphic do-nothing default so callers stop writing null checks; Visitor lifts an algorithm out of an object structure so you can add new operations without editing the element classes, using double dispatch. Visitor is a Gang-of-Four behavioral pattern; Null Object is a later, widely-adopted addition to the pattern canon and is likewise behavioral.

01. Null Object pattern introduction

Objective: Understand how a neutral, do-nothing object can stand in for an absent value so that callers never touch a null reference.

The Null Object pattern answers a mundane but pervasive problem: a lookup that finds nothing usually returns null, and every caller must then guard the result with an if (x != null) before using it. Forget one guard and you get a NullPointerException at runtime. The pattern removes the special case by making "nothing" a real object — one that implements the same interface as the real thing but whose methods do nothing (or return a harmless default).

The participants are an AbstractObject (an abstract class or interface declaring the operations), a RealObject that carries genuine state and behaviour, and a NullObject that implements the same contract with neutral behaviour. A factory or repository decides which of the two to hand back, so callers receive a usable object every time and can drop their null checks entirely.

  • Intent: substitute a benign default object for null, eliminating scattered null checks.
  • Roles: AbstractObject (shared contract), RealObject, NullObject (do-nothing).
  • Category: behavioral (not part of the original Gang-of-Four 23). It changes how objects collaborate, not how they are built.
  • Best when a missing value has a sensible, harmless default behaviour.
// Without the pattern: every caller must null-check.
AbstractCustomer c = factory.getCustomer(name);
if (c != null) {                 // forget this guard -> NullPointerException
    System.out.println(c.getName());
}

// With the pattern: the factory never returns null.
AbstractCustomer c = factory.getCustomer(name);   // RealCustomer OR NullCustomer
System.out.println(c.getName());  // NullCustomer.getName() returns a safe default

Source: reimplemented as an original illustration.

AdvancedThough it was catalogued after the Gang-of-Four book (by Bobby Woolf), Null Object is the object-oriented cousin of an identity element (0 for addition, an empty list for concatenation): a value that participates in operations without changing their outcome. It pairs naturally with a Factory — centralising the "real or null?" decision in one place — and with the Iterator/Composite patterns, where a NullNode can terminate a tree without per-branch checks.
Pitfall A Null Object must be genuinely neutral. If its methods throw, log noisily, or return values that later blow up (an empty string parsed as a number), you have merely moved the crash. Keep its behaviour boring and side-effect free.

02. Null Object example — customer lookup

Objective: Trace a concrete Null Object implementation where a customer factory resolves a name against a small database and always returns a usable object.

CustomerFactory.getCustomer(name) consults a Database of known names. If the name exists it returns a RealCustomer holding that name; otherwise it returns a NullCustomer whose getName() yields a fixed "Not Available in Customer Database" message. The calling loop is uniform — it prints getName() on whatever comes back — because both subclasses honour the same AbstractCustomer contract.

  • getCustomer is total: for any input it returns a non-null AbstractCustomer.
  • The absence case is expressed as data (a message), not as a missing reference.
  • The caller has zero null checks and cannot throw a NullPointerException here.
import java.util.List;

abstract class AbstractCustomer {
    protected String name;
    abstract boolean isNull();
    abstract String  getName();
}

class RealCustomer extends AbstractCustomer {
    RealCustomer(String name) { this.name = name; }
    boolean isNull() { return false; }
    String  getName() { return name; }
}

class NullCustomer extends AbstractCustomer {
    boolean isNull() { return true; }
    String  getName() { return "Not Available in Customer Database"; }
}

class Database {
    private final List<String> names =
        List.of("Daniel", "Adam", "Michael", "Joe");
    boolean exists(String name) { return names.contains(name); }
}

class CustomerFactory {
    private final Database db = new Database();
    AbstractCustomer getCustomer(String name) {
        return db.exists(name) ? new RealCustomer(name)
                               : new NullCustomer();
    }
}

public class App {
    public static void main(String[] args) {
        CustomerFactory factory = new CustomerFactory();
        for (String n : List.of("Joe", "Joel", "Daniela", "Adam"))
            System.out.println(factory.getCustomer(n).getName());
    }
}

Pitfall Null Object can hide a real error. If a customer id that must exist silently resolves to a placeholder, a bug becomes a shrug instead of a stack trace. Use it only where absence is expected and harmless; where a missing value signals a defect, fail loudly (throw) or return an Optional that forces the caller to decide.

03. Avoiding null with Optional

Objective: Contrast the Null Object pattern with java.util.Optional (Java 8) and know when each is the better tool for the "absence" problem.

Optional<T> attacks the same problem from the other direction. Instead of hiding absence behind a neutral object, it makes absence explicit in the type: a method that returns Optional<Customer> is honestly declaring "there may be no customer", and the caller must unwrap it — via orElse, map, ifPresent, and friends — before reaching the value. Optional.ofNullable lifts a possibly-null reference into that box, and orElse supplies the fallback.

Choose Null Object when the absent case has meaningful polymorphic default behaviour that should just run (a silent logger, an empty iterator, a guest user with no permissions). Choose Optional when the caller genuinely needs to notice absence and branch on it — the type makes the check impossible to forget. They are complementary, not competitors.

  • Optional encodes "maybe absent" in the return type; the compiler nudges you to handle it.
  • ofNullable(x).orElse(d) converts a nullable reference plus a default into a plain value.
  • Null Object = implicit neutral behaviour; Optional = explicit presence check.
import java.util.Optional;

public class App {

    static Optional<String> findEmail(String user) {
        if (user.equals("Joe")) return Optional.of("joe@shop.io");
        return Optional.empty();          // honest "no value"
    }

    public static void main(String[] args) {
        String a = findEmail("Joe").orElse("no email on file");
        String b = findEmail("Joel").orElse("no email on file");
        System.out.println(a);
        System.out.println(b);

        // ofNullable wraps a reference that might be null
        String raw = null;
        System.out.println(Optional.ofNullable(raw).orElse("default"));
    }
}
Console outputjoe@shop.io no email on file default

Source: reimplemented as an original illustration.

AdvancedOptional is intended for return types. Using it for fields or method parameters is discouraged (it adds a wrapper allocation and a second "null" state — the Optional reference itself can be null), and calling get() without a prior isPresent() throws NoSuchElementException, which is just a NullPointerException wearing a hat. Prefer orElse, orElseGet, map, or ifPresentOrElse.
Pitfall orElse(expensive()) always evaluates its argument, even when the value is present. When the default is costly to build, use orElseGet(() -> expensive()) so the fallback is computed only on absence.

04. Visitor pattern introduction

Objective: Learn how Visitor separates an algorithm from the object structure it walks, letting you add operations without modifying the element classes.

Visitor solves a stubborn problem: you have a fixed set of element types (shapes, AST nodes, file-system entries) and you keep needing new operations over them — compute area, render, serialise, type-check. Piling every operation into the element classes bloats them and violates the Single Responsibility and Open/Closed principles. Visitor moves each operation into its own Visitor object; the elements merely offer an accept(visitor) hook.

The participants are a Visitor interface with one visit overload per element type, one or more ConcreteVisitors (each a whole operation), an Element interface declaring accept(Visitor), and the ConcreteElements that implement accept as visitor.visit(this). That two-step handshake — element.accept(visitor) then visitor.visit(element) — is double dispatch: the concrete method is chosen by both the runtime element type and the visitor type. GoF category: behavioral.

  • Adds new operations cheaply: write a new ConcreteVisitor, touch no element class.
  • Relies on double dispatch (accept then visit) because Java resolves overloads by static type.
  • Trade-off: adding a new element type is expensive — every Visitor gains a new visit.
  • Best for a stable element hierarchy with a growing set of operations.
interface Visitor { void visit(Circle c); void visit(Square s); }

interface Element { void accept(Visitor v); }

class Circle implements Element {
    public void accept(Visitor v) { v.visit(this); }  // double dispatch:
}                                                      // 'this' is a Circle here

Source: reimplemented as an original illustration.

AdvancedWhy double dispatch is unavoidable in Java: overloaded methods are resolved at compile time by the static type of the argument. Given Shape s, a bare visitor.visit(s) would need a visit(Shape) overload — the compiler cannot see that s is really a Circle. The virtual call s.accept(visitor) dispatches on s's runtime type (dispatch #1) into Circle.accept, where this is statically Circle, so visitor.visit(this) resolves the correct overload (dispatch #2).
Pitfall Don't reach for Visitor when the element hierarchy is still churning. Each new element type forces a change to the Visitor interface and to every implementing visitor — the exact rigidity the pattern trades away to make operations cheap.

05. Visitor example — shapes & area calculator

Objective: Implement Visitor over a small Shape hierarchy and run two operations — area calculation and drawing — without touching the shape classes.

Circle and Rectangle implement Shape and expose only their own state plus accept. Two ConcreteVisitors — AreaCalculator and ShapeDrawer — each carry a complete operation across both shapes. The client walks the same List<Shape> twice, handing it a different visitor each pass. Adding a third operation later (say, a JSON exporter) is a new class and nothing else.

  • AreaCalculator and ShapeDrawer add operations without editing Circle/Rectangle — Open/Closed in action.
  • Each shape's accept is one line: visitor.visit(this).
  • The list of shapes is traversed uniformly; the visitor decides what happens per type.
import java.util.List;

interface Shape { void accept(ShapeVisitor v); }

class Circle implements Shape {
    private final double radius;
    Circle(double radius) { this.radius = radius; }
    double getRadius() { return radius; }
    public void accept(ShapeVisitor v) { v.visit(this); }
}

class Rectangle implements Shape {
    private final double width, height;
    Rectangle(double w, double h) { this.width = w; this.height = h; }
    double getWidth()  { return width; }
    double getHeight() { return height; }
    public void accept(ShapeVisitor v) { v.visit(this); }
}

interface ShapeVisitor {
    void visit(Circle circle);
    void visit(Rectangle rectangle);
}

class AreaCalculator implements ShapeVisitor {
    public void visit(Circle c) {
        System.out.println("Area of circle: "
            + Math.PI * c.getRadius() * c.getRadius());
    }
    public void visit(Rectangle r) {
        System.out.println("Area of rectangle: "
            + r.getWidth() * r.getHeight());
    }
}

class ShapeDrawer implements ShapeVisitor {
    public void visit(Circle c) {
        System.out.println("Drawing circle r=" + c.getRadius());
    }
    public void visit(Rectangle r) {
        System.out.println("Drawing rectangle "
            + r.getWidth() + "x" + r.getHeight());
    }
}

public class App {
    public static void main(String[] args) {
        List<Shape> shapes = List.of(new Circle(10), new Rectangle(5, 3));
        for (Shape s : shapes) s.accept(new AreaCalculator());
        for (Shape s : shapes) s.accept(new ShapeDrawer());
    }
}

AdvancedModern Java (21+) offers an alternative that inverts the trade-off: a sealed interface plus a pattern-matching switch. 
sealed interface Shape permits Circle, Rectangle {}
double area(Shape s) {
    return switch (s) {
        case Circle c    -> Math.PI * c.radius() * c.radius();
        case Rectangle r -> r.width() * r.height();
    };   // compiler enforces exhaustiveness over the sealed set

Pitfall Double dispatch is easy to short-circuit by accident: writing visitor.visit(shape) directly (bypassing accept) forces Java to resolve the overload on the static type Shape — which won't compile without a visit(Shape), or will call the wrong overload if one exists. Always go through shape.accept(visitor).

06. Visitor exercise & solution — export shapes

Objective: Apply Visitor to a new operation — exporting a mixed shape list to CSV — over a three-type hierarchy of Circle, Rectangle, and Triangle.

The exercise adds a Triangle and a SaveCSVFile visitor whose visit methods emit one CSV row per shape. 12. Structural patterns I: Decorator, Facade & Flyweight

Structural patterns describe how to compose classes and objects into larger structures while keeping those structures flexible and efficient. This section covers three of them: Decorator (wrap to add behaviour), Facade (simplify a subsystem) and Flyweight (share objects to save memory), plus the JVM string pool as a real-world flyweight.

01. Structural design patterns — overview

Objective: Understand what "structural" means in the Gang-of-Four classification and see the family of patterns this category contains.

The GoF catalogue splits its 23 patterns into three groups. Creational patterns are about how objects are created (Factory, Builder, Singleton). Behavioural patterns are about how objects communicate and share responsibility (Strategy, Observer, Command). Structural patterns — the focus here — are about how classes and objects are composed into larger structures without a rigid, hard-wired inheritance tree.

GoF defines seven structural patterns — Decorator, Facade, Flyweight, Adapter, Bridge, Composite and Proxy; this course covers all of them except Proxy. A useful mental split: some of them wrap one object inside another — Decorator and Proxy keep the same interface, while Adapter also wraps but converts it into a different one — one simplifies access to many objects (Facade), one shares objects to cut memory (Flyweight), and the rest reorganise a hierarchy (Bridge, Composite).

  • Structural patterns compose objects; they do not create them (creational) or route messages (behavioural).
  • Decorator and Proxy wrap an object and keep the same interface; Adapter also wraps, but converts the wrapped object's interface into a different one — what differs is their intent, not the wrapping mechanic.
  • Prefer composition (holding a reference) over deep inheritance — most structural patterns are composition in disguise.
Pitfall Because Decorator, Adapter and Proxy are all "an object that wraps another object of the same interface", people label wrappers by their code shape. Classify by intent instead: Decorator adds behaviour, Adapter changes the interface, Proxy controls access.

02. Decorator pattern introduction

Objective: Learn how Decorator attaches responsibilities to an object dynamically by wrapping it, as a flexible alternative to subclassing.

Decorator is a structural GoF pattern. Its intent is to attach additional responsibilities to an object dynamically, at runtime, without touching the wrapped object's class. Each decorator both is-a and has-a the component type: it implements the component interface and holds a reference to another component, delegating to it and then contributing its own extra behaviour.

The participants are the Component (the shared interface), the ConcreteComponent (the plain object being decorated), the Decorator (an abstract base that stores and forwards to a wrapped Component) and one or more ConcreteDecorator classes that override methods to add behaviour. Because decorators wrap objects rather than extend a class, you can combine them in any order and stack any number of them — a combinatorial reach that subclassing cannot match.

  • Category: structural. Intent: add responsibilities to a single object at runtime.
  • Participants: Component, ConcreteComponent, Decorator (abstract, wraps a Component), ConcreteDecorator.
  • It supports the Open/Closed Principle — extend behaviour by adding decorators, never by editing existing classes.
Beverage drink = new Milk(new PlainBeverage());  // wrap to add behaviour
drink.cost();        // Milk delegates to the wrapped Beverage, then adds its own cost
drink.description(); // ...and appends its own text to the wrapped description

Source: reimplemented as an original illustration.

Pitfall Decorators are order-sensitive. Wrapping is not always commutative — a decorator that trims whitespace then one that encrypts produces a different result than the reverse. Fix the intended order in one place (a builder or factory) rather than scattering new X(new Y(...)) calls across the codebase.

03. Decorator example — beverages

Objective: Build a coffee-shop pricing model where Milk and Sugar decorators accumulate cost and description around a PlainBeverage.

A PlainBeverage costs 5 and has the description "plain". The abstract BeverageDecorator implements Beverage and stores the wrapped base beverage. Each concrete decorator overrides cost() and description() to call base first and then add its own contribution: Milk adds 3 and " + milk", Sugar adds 1 and " + sugar". Because the wrappers all share the Beverage type, new Sugar(new Sugar(new Milk(new PlainBeverage()))) is itself just a Beverage.

Evaluating cost() on the outer object cascades inward: the outer Sugar asks its inner Sugar, which asks Milk, which asks PlainBeverage (5), then each layer adds on the way back out: 5 → 8 → 9 → 10. This is exactly how java.io streams work — new BufferedReader(new FileReader(f)) is the canonical JDK decorator chain.

  • The wrapped reference is held in the abstract BeverageDecorator; subclasses only add their delta.
  • Cost accumulates 5 + 3 + 1 + 1 = 10; description accumulates left-to-right as layers unwind.
  • The client never sees the concrete decorator types — it holds the Beverage interface.
interface Beverage {
    int cost();
    String description();
}

class PlainBeverage implements Beverage {          // ConcreteComponent
    public int cost()           { return 5; }
    public String description() { return "plain"; }
}

// Decorator: IS-A Beverage and HAS-A Beverage
abstract class BeverageDecorator implements Beverage {
    protected final Beverage base;
    protected BeverageDecorator(Beverage base) { this.base = base; }
}

class Milk extends BeverageDecorator {             // ConcreteDecorator
    Milk(Beverage b) { super(b); }
    public int cost()           { return base.cost() + 3; }
    public String description() { return base.description() + " + milk"; }
}

class Sugar extends BeverageDecorator {
    Sugar(Beverage b) { super(b); }
    public int cost()           { return base.cost() + 1; }
    public String description() { return base.description() + " + sugar"; }
}

public class App {
    public static void main(String[] args) {
        Beverage order = new Sugar(new Sugar(new Milk(new PlainBeverage())));
        System.out.println(order.description());
        System.out.println("cost = " + order.cost());
    }
}
Console outputplain + milk + sugar + sugar cost = 10

AdvancedThe JDK uses Decorator heavily. Beyond BufferedReader(new FileReader(...)), the Collections.unmodifiableList(...) and Collections.synchronizedMap(...) factory methods return decorators that wrap the original collection and add read-only or thread-safe behaviour without changing its interface. Swing's JScrollPane and the java.io/java.util.zip stream stacks are the classic examples.
Pitfall Decorator encourages an explosion of tiny wrapper classes, and a deep stack is hard to debug — a stack trace shows five cost() frames. Keep each decorator focused on one responsibility, and if you routinely need the same combination, expose it behind a factory method (e.g. latte()) instead of making callers reassemble it.

04. Facade pattern introduction

Objective: Learn how Facade provides a single, simplified entry point to a complex subsystem and thereby reduces coupling.

Facade is a structural GoF pattern. Its intent is to provide a unified, higher-level interface to a set of interfaces in a subsystem, making that subsystem easier to use. The facade knows which subsystem classes are responsible for a request and delegates to them, so clients depend on one small, stable surface instead of on many moving parts.

The participants are simply the Facade and the subsystem classes it fronts. A facade does not add behaviour and it does not change the subsystem's interfaces (that would be Adapter) — it just offers a convenient default path. Crucially, it should reduce coupling, not forbid direct access: advanced clients may still reach past the facade to a subsystem class when they need finer control.

  • Category: structural. Intent: one simple interface over a complex subsystem.
  • Participants: the Facade plus the subsystem classes it delegates to.
  • Reduces coupling between client and subsystem; contrast with Adapter (converts an interface) and Mediator (coordinates peers).
// Without a facade the client wires up every subsystem class itself:
new BubbleSort().sort();  new MergeSort().sort();  new HeapSort().sort();

// With a facade the client talks to ONE object:
SortingManager manager = new SortingManager();
manager.bubbleSort();

Source: reimplemented as an original illustration.

Pitfall A facade should not seal off the subsystem. If you make the subsystem classes package-private and force every call through the facade, the facade becomes the only path and grows into a bottleneck. Keep subsystem classes usable directly for the rare advanced case.

05. Facade example — sorting manager

Objective: Put a SortingManager facade in front of three sorting algorithms so the client triggers any of them through one object.

Three classes — BubbleSort, MergeSort, HeapSort — implement a common Algorithm interface with a single sort() method. The SortingManager facade instantiates all three once in its constructor and exposes three thin methods (bubbleSort(), mergeSort(), heapSort()) that delegate to the matching subsystem object. The client constructs one manager and calls those methods — it never touches the concrete algorithm classes.

  • The facade owns and wires up the subsystem objects; the client owns only the facade.
  • Each facade method is a one-line delegation — the facade adds convenience, not logic.
  • Swapping in a better MergeSort implementation changes one line in the facade, not every caller.
interface Algorithm { void sort(); }

class BubbleSort implements Algorithm {
    public void sort() { System.out.println("Bubble sort..."); }
}
class MergeSort implements Algorithm {
    public void sort() { System.out.println("Merge sort..."); }
}
class HeapSort implements Algorithm {
    public void sort() { System.out.println("Heap sort..."); }
}

// Facade: one simple entry point over three subsystem classes
class SortingManager {
    private final Algorithm bubble = new BubbleSort();
    private final Algorithm merge  = new MergeSort();
    private final Algorithm heap   = new HeapSort();

    public void bubbleSort() { bubble.sort(); }
    public void mergeSort()  { merge.sort();  }
    public void heapSort()   { heap.sort();   }
}

public class App {
    public static void main(String[] args) {
        SortingManager manager = new SortingManager();
        manager.bubbleSort();
        manager.mergeSort();
        manager.heapSort();
    }
}

AdvancedFacades appear everywhere in real systems: javax.faces's FacesContext, SLF4J's LoggerFactory, and most "service" classes in a layered app are facades over a data-access layer. A facade also aids testing — mock the facade to isolate the client from a heavyweight subsystem such as a database or an external HTTP API.
Pitfall A facade can silently become a god object: as the subsystem grows, teams keep adding methods until the facade knows about everything and depends on every subsystem class. Split it into several focused facades (one per use-case area) before it turns into a maintenance choke point.

06. Flyweight pattern introduction

Objective: Understand how Flyweight shares fine-grained objects to save memory by separating intrinsic from extrinsic state.

Flyweight is a structural GoF pattern used when a program creates a huge number of similar objects and memory becomes the bottleneck. The key move is to split an object's state into two parts. Intrinsic state is shared and context-independent — it is stored inside the flyweight and reused by everyone. Extrinsic state is context-dependent — it is not stored in the flyweight but passed in by the client on each call.

The participants are the Flyweight interface, the ConcreteFlyweight (holds the intrinsic state), the FlyweightFactory (a cache/pool that hands out shared instances, creating a new one only when none exists for a given key), and the Client (holds the extrinsic state and supplies it per operation). The factory guarantees that two requests for the same intrinsic state return the same object reference, not a copy.

  • Category: structural. Intent: share many small objects to reduce memory footprint.
  • Intrinsic state → shared, stored in the flyweight. Extrinsic state → passed in by the client.
  • Participants: Flyweight, ConcreteFlyweight, FlyweightFactory (pool), Client.
// The factory returns a SHARED instance for identical intrinsic state:
BulletType t1 = BulletFactory.of("bullet.png", 10, 5);
BulletType t2 = BulletFactory.of("bullet.png", 10, 5);
System.out.println(t1 == t2);   // true -- same object, not a copy

Source: reimplemented as an original illustration.

Pitfall Flyweight only pays off when there really are many objects with few distinct intrinsic states. For a handful of objects the factory, key computation and indirection cost more than they save — measure before reaching for it.

07. Flyweight example — bullets in a game

Objective: Let thousands of on-screen bullets share a few BulletType flyweights while each keeps its own position and direction.

A shooter game might have thousands of bullets on screen. The heavy, shared data — texture, speed, size — is intrinsic and lives in a BulletType flyweight. The per-bullet data — x, y, direction — is extrinsic and lives in a lightweight BulletInstance that merely points at a shared type. The BulletFactory keys its pool on the intrinsic values, so every request for the same texture/speed/size returns the identical BulletType reference.

In the trace below the factory prints only once for bullet.png105 even though it is requested twice — the second call is a pool hit, and a == b confirms the shared reference. Three BulletInstance objects then render through that single shared BulletType, each supplying its own coordinates as extrinsic state.

  • Intrinsic (texture, speed, size) lives in BulletType; extrinsic (x, y, dir) is passed into render(...).
  • The factory key texture + speed + size decides reuse — same key returns the same object.
  • Memory scales with the number of distinct types, not the number of bullets on screen.
import java.util.HashMap;
import java.util.Map;

interface Bullet {                                  // Flyweight
    void render(int x, int y, String dir);
}

// Concrete flyweight -- holds only INTRINSIC (shared) state
class BulletType implements Bullet {
    private final String texture;
    private final int speed, size;
    BulletType(String texture, int speed, int size) {
        this.texture = texture; this.speed = speed; this.size = size;
    }
    public void render(int x, int y, String dir) {
        System.out.println("draw '" + texture + "' at (" + x + "," + y
            + ") going " + dir + " speed=" + speed);
    }
}

// Factory / pool -- one shared flyweight per key
class BulletFactory {
    private static final Map<String, BulletType> pool = new HashMap<>();
    static BulletType of(String texture, int speed, int size) {
        String key = texture + speed + size;
        return pool.computeIfAbsent(key, k -> {
            System.out.println("new BulletType " + key);
            return new BulletType(texture, speed, size);
        });
    }
}

// Context -- holds EXTRINSIC state, points at a shared type
class BulletInstance {
    private final int x, y;
    private final String dir;
    private final BulletType type;
    BulletInstance(int x, int y, String dir, BulletType type) {
        this.x = x; this.y = y; this.dir = dir; this.type = type;
    }
    void draw() { type.render(x, y, dir); }
}

public class App {
    public static void main(String[] args) {
        BulletType a = BulletFactory.of("bullet.png", 10, 5);
        BulletType b = BulletFactory.of("bullet.png", 10, 5);  // same key
        System.out.println("shared? " + (a == b));

        BulletInstance[] shots = {
            new BulletInstance(100, 200, "UP", a),
            new BulletInstance(105, 210, "UP", a),
            new BulletInstance(110, 220, "UP", a)
        };
        for (BulletInstance s : shots) s.draw();
    }
}

Source: reimplemented from the course example.

AdvancedThe flyweight pool is shared mutable static state, so in a multithreaded game it must be thread-safe — use a ConcurrentHashMap and computeIfAbsent, which performs the check-and-insert atomically per key. Note that with a plain HashMap the factory's own print order is deterministic here (single-threaded, sequential calls), but HashMap makes no promise about iteration order if you ever loop over the pool.
Pitfall The flyweight breaks the instant you move extrinsic state into the shared object. If BulletType stored x and y, every bullet writing its position would corrupt all the others that share that instance. Keep the flyweight immutable and pass mutable per-instance state in as method arguments.

08. String interning — the string constant pool

Objective: See the JVM's string pool as a real flyweight and understand == vs .equals() for pooled versus new strings.

The JVM keeps a string constant pool — a table of unique String objects — and every string literal in your code is automatically interned: it points at the shared pooled instance. This is the Flyweight pattern built into the platform, with the immutable character data as intrinsic state. Two identical literals therefore refer to the same object, so == (reference identity) returns true.

A string built with new String("hello") is a distinct heap object even though its characters match a pooled literal, so == against the literal is false while .equals() is true. Calling intern() returns the canonical pooled reference for that content — after which == against the literal is true again. The rule stands: compare string content with .equals(), never with ==.

  • String literals are interned automatically — identical literals share one pooled object.
  • new String("x") forces a separate object; == to a literal is false, .equals() is true.
  • String.intern() returns the shared pooled reference for the given content.
public class App {
    public static void main(String[] args) {
        String a = "hello";               // literal -> string pool
        String b = "hello";               // same pooled object
        String c = new String("hello");   // brand-new heap object
        String d = c.intern();            // look up / return the pooled ref

        System.out.println(a == b);       // same pooled reference
        System.out.println(a == c);       // different objects
        System.out.println(a.equals(c));  // same characters
        System.out.println(a == d);       // intern() returns the pooled ref
    }
}
Console outputtrue false true true

AdvancedSince Java 7 the string pool lives on the normal heap (not PermGen), so it is garbage-collected and can grow large. intern() is not free — it may add to the pool and historically had a fixed-size hash table — so interning huge volumes of untrusted or unique strings can hurt more than it helps. The same flyweight idea underlies Integer.valueOf(), which caches boxed Integers in the range −128..127, so Integer.valueOf(100) == Integer.valueOf(100) is true but 200 is not.
Pitfall Relying on == for strings is a classic bug: it happens to work for literals (they are pooled) but silently fails for strings read from files, sockets or new String(...). Always use .equals() for content comparison and Objects.equals(a, b) when either side may be null.

13. Structural Patterns II: Adapter, Bridge & Composite

Three more structural GoF patterns, each about wiring objects together the right way. Adapter makes an existing, incompatible class fit an interface a client already expects; Bridge splits a design into two independent axes so they vary without a class explosion; Composite builds trees you can treat one node or a whole sub-tree the same way. We keep the course's shape/vehicle domain throughout.

01. Adapter pattern introduction

Objective: Recognise Adapter as the "wrapper" that converts one interface into another the client expects, letting two otherwise-incompatible types collaborate.

Adapter is a structural GoF pattern. You have a client that speaks one interface (the Target) and an existing, useful class that speaks a different one (the Adaptee). Rather than rewrite either, you introduce an Adapter that implements the Target interface and, inside each method, translates the call into whatever the Adaptee actually offers. The Client then programs against Target and never learns that an Adaptee is doing the work behind the wrapper.

There are two flavours. An object adapter (the common one in Java) holds a reference to the Adaptee and delegates — composition. A class adapter extends the Adaptee while implementing the Target — inheritance. Because Java allows only single class inheritance, the class-adapter form is restrictive (you can adapt only one concrete class and inherit its whole API), so most Java code — and this course — uses the object adapter.

  • Participants: Target (interface the client expects), Adaptee (existing, incompatible class), Adapter (implements Target, delegates to the Adaptee), Client.
  • Intent: reuse an existing class whose interface does not match — a mismatch you discover after the fact.
  • Object adapter = composition (preferred in Java); class adapter = inheritance (limited by single inheritance).
  • The Adapter changes the interface, not the behaviour — that is what separates it from Decorator.
// Target -- the interface the client already speaks.
interface Target { void request(); }

// Adaptee -- a useful class with a different, incompatible method name.
class Adaptee { void specificRequest() { /* ...real work... */ } }

// Object adapter -- implements Target, HOLDS an Adaptee, translates the call.
class Adapter implements Target {
    private final Adaptee adaptee;
    Adapter(Adaptee adaptee) { this.adaptee = adaptee; }
    @Override public void request() { adaptee.specificRequest(); }  // translate
}

AdvancedA two-way adapter implements both interfaces so the object is usable as either type — handy when two subsystems each expect the other's protocol. It only works as a class adapter (multiple interface implementation), and only when the two interfaces do not clash on method signatures.
Pitfall Confusing Adapter with Decorator or Bridge. Adapter changes an interface to fix a mismatch you found after the classes exist; Decorator keeps the same interface and adds behaviour; Bridge is planned up front to let two dimensions vary. Same "wrap an object" shape, three different intents.

02. Adapter example — vehicles / bicycle adapter

Objective: Trace the course example that adapts a Bicycle — whose method is go() — so it can be driven through the Vehicle interface's accelerate().

Vehicle is the Target: Car and Bus implement it directly. Bicycle is the Adaptee — a fine class, but its method is called go(), so it cannot be passed where a Vehicle is expected. BicycleAdapter is the Adapter: it implements Vehicle, wraps a Bicycle, and in accelerate() simply calls bicycle.go(). The client loop can now iterate a mixed list of buses, cars and (adapted) bicycles, calling accelerate() on each.

  • Vehicle = Target, Bicycle = Adaptee, BicycleAdapter = Adapter — classic object-adapter wiring.
  • The adapter holds the adaptee (private final Bicycle bicycle) and delegates — no inheritance from Bicycle.
  • Neither Bicycle nor the client changes; only the small adapter class is added.
interface Vehicle { void accelerate(); }

}
class Bus implements Vehicle {

}

}

  • The adapter lets an incompatible-by-format class (XML) satisfy a CSV-shaped contract without editing the third-party class.
  • Client code sees only CSVParser; the XML detail is hidden inside the adapter.
  • This is the everyday use of Adapter: bolting a foreign/legacy API onto your own interface.

04. The Adapter pattern in the JDK

Objective: Spot real adapters in the standard library — and avoid the trap of assuming every JDK class named *Adapter is this pattern.

Genuine Adapters in the JDK convert one interface into another. Arrays.asList(array) adapts a plain T[] to the List<T> interface (a fixed-size list backed by the array). InputStreamReader adapts a byte-oriented InputStream to a character-oriented Reader, decoding bytes to chars via a Charset; OutputStreamWriter is its write-side twin. These are textbook object adapters — they wrap the adaptee and expose a different interface.

Beware the naming collision: the old AWT/Swing classes WindowAdapter, MouseAdapter, KeyAdapter are not this pattern. They are empty default implementations of listener interfaces — convenience base classes so you override only the callbacks you need (closer to a Null-Object / default-methods idea). Same word, different concept. Since Java 8 default interface methods make most of them redundant.

  • Real adapters: Arrays.asList (array → List), InputStreamReader/OutputStreamWriter (bytes ↔ chars).
  • The java.io stream/reader hierarchy is full of adapters and decorators bridging byte and character worlds.
  • Swing's WindowAdapter/MouseAdapter = empty-default convenience classes, not GoF Adapter — the name misleads.

05. Bridge pattern introduction

Objective: Understand Bridge as decoupling an abstraction from its implementation so the two can vary independently — avoiding a combinatorial class explosion.

Bridge is a structural GoF pattern. Suppose you have two independent dimensions of variation — say shape type (Circle, Square, …) and render mode (Vector, Raster, …). Modelling every combination with inheritance gives N × M classes (VectorCircle, RasterCircle, VectorSquare…) — an explosion that grows every time you add either kind. Bridge instead lets the Abstraction hold a reference to an Implementor, so you need only N + M classes and can add a shape or a renderer without touching the other side.

Participants: Abstraction (defines the high-level operation and holds an Implementor), RefinedAbstraction (concrete subclasses of the abstraction), Implementor (the interface for the "how"), and ConcreteImplementor (its implementations). The has-a link from Abstraction to Implementor is literally "the bridge".

  • Intent: two orthogonal axes vary independently — what you do vs how it is done.
  • Replaces an N × M inheritance grid with an N + M composition.
  • Participants: Abstraction → Implementor (bridge), RefinedAbstraction, ConcreteImplementor.
  • Bridge is designed up front; that is what distinguishes it from Adapter (retrofitted).
// Implementor -- the "how".
interface Renderer { void render(String shape); }

// Abstraction -- the "what"; HOLDS an Implementor rather than subclassing per combo.
abstract class Shape {
    protected final Renderer renderer;      // <-- the bridge
    protected Shape(Renderer renderer) { this.renderer = renderer; }
    abstract void draw();
}
// N shapes + M renderers  =>  N + M classes, not N * M.

Pitfall Reaching for Bridge when you have only one axis of variation, or when the second axis will realistically never grow. The extra interface and indirection is pure overhead unless both dimensions genuinely vary. If you are adapting an existing mismatched class, you want Adapter, not Bridge.

06. Bridge implementation — shapes & renderers

Objective: Walk the course's Bridge: Shape (abstraction) holds a Renderer (implementor), with Circle/Square on one axis and Vector/Raster renderers on the other.

Shape is the Abstraction: it stores a Renderer and declares draw(). Circle and Square are RefinedAbstractions — each draw() simply hands its own name to renderer.render(...). Renderer is the Implementor; VectorRenderer and RasterRenderer are the ConcreteImplementors. The two dimensions are orthogonal: we separate what we draw (the shape) from how it is rendered.

The payoff is in maintenance. Add a Triangle and it works with every existing renderer for free. Add an SvgRenderer and every existing shape can use it — no shape class changes. Neither axis knows how many entries the other has.

  • Shape = Abstraction (holds Renderer); Circle/Square = RefinedAbstractions.
  • Renderer = Implementor; VectorRenderer/RasterRenderer = ConcreteImplementors.
  • The renderer is injected via the constructor — the same shape can be drawn in different modes at runtime.
  • Adding on one axis never edits the other — the defining benefit of Bridge.
// Implementor + concrete implementors -- the "how".
interface Renderer { void render(String shape); }

class VectorRenderer implements Renderer {
    @Override public void render(String shape) { System.out.println("Vector rendering: " + shape); }
}
class RasterRenderer implements Renderer {
    @Override public void render(String shape) { System.out.println("Raster rendering: " + shape); }
}

// Abstraction -- holds a Renderer (the bridge); no subclass per render mode.
abstract class Shape {
    protected final Renderer renderer;
    protected Shape(Renderer renderer) { this.renderer = renderer; }
    abstract void draw();
}
class Circle extends Shape {
    Circle(Renderer renderer) { super(renderer); }
    @Override void draw() { renderer.render("CIRCLE"); }
}
class Square extends Shape {
    Square(Renderer renderer) { super(renderer); }
    @Override void draw() { renderer.render("SQUARE"); }
}

        Shape s1 = new Circle(new VectorRenderer());
        Shape s2 = new Square(new VectorRenderer());
        Shape s3 = new Circle(new RasterRenderer());  // same shape, different renderer
        s1.draw();
        s2.draw();
        s3.draw();
    }
}

AdvancedBridge and the Strategy pattern share the same skeleton — an object holding a pluggable collaborator. The difference is intent: Strategy swaps an algorithm behind one operation; Bridge separates a whole abstraction hierarchy from an implementation hierarchy so both can grow. Structurally similar, semantically distinct.
Pitfall Confusing Bridge with Adapter because both delegate to a held object. 07. JDBC and the Bridge pattern

Objective: See JDBC as a real-world Bridge: the java.sql API is the abstraction, vendor drivers are the swappable implementation.

Your application code targets the java.sql abstraction — Connection, Statement, PreparedStatement, ResultSet. The actual work of talking to PostgreSQL, MySQL or Oracle is done by a vendor Driver, the implementation. DriverManager.getConnection(url, ...) selects the driver from the connection URL and hands you an implementation of the API. Swap the URL and the driver jar and not a line of your query code changes — two hierarchies (the API and the drivers) varying independently, which is exactly Bridge.

  • Abstraction = the java.sql interfaces you program against; Implementor = the vendor's Driver.
  • The database vendor varies without your code changing — the Bridge benefit.
  • The Connection the driver returns is a ConcreteImplementor plugged in behind the API.

08. Composite pattern introduction

Objective: Understand Composite as composing objects into tree structures and treating individual objects (leaves) and groups (composites) uniformly through one component interface.

Composite is a structural GoF pattern for part-whole hierarchies. You define a common Component interface; a Leaf is an end node with no children; a Composite is a node that holds child Components and, when asked to do work, delegates to each child. Because a Composite is itself a Component, groups can contain groups to any depth, and the client treats a single leaf and an entire sub-tree identically — it just calls the component operation and the call cascades.

  • Participants: Component (common interface), Leaf (no children), Composite (holds children, delegates to them).
  • Uniform treatment: the client calls the same method on a leaf or a whole tree.
  • Recursive by construction — a Composite's operation calls the same operation on each child.
  • Natural fit for menus, file systems, GUI widget trees, and org charts.
// Component -- common type for both leaves and composites.
interface Shape { void draw(); }

// Composite -- is-a Shape AND holds child Shapes.
class ShapeGroup implements Shape {
    private final List<Shape> children = new ArrayList<>();
    public void add(Shape s)    { children.add(s); }
    @Override public void draw() {                 // cascades to every child
        for (Shape child : children) child.draw();
    }
}

Source: reimplemented as an original illustration.

Pitfall The "transparency vs safety" dilemma. Putting add(child)/remove(child) on the Component interface makes leaves and composites look identical (transparent) but forces every Leaf to implement child operations it cannot support — a latent LSP problem. Keep child management on the Composite only (safe) and clients must type-check, or accept the transparent design and make Leaf's add throw/no-op. Choose deliberately.

09. Composite implementation — shapes & groups

Objective: Trace the course's Composite where a ShapeGroup holds child Shapes and draw() cascades recursively, including a group nested inside another group.

Shape is the Component; Circle and Square are Leaves. ShapeGroup is the Composite: it implements Shape, holds a List<Shape>, and its draw() prints a header then calls draw() on every child. Because a ShapeGroup is a Shape, you can add one group to another (bigGroup contains group), and the recursive draw() descends the whole tree — the client never distinguishes a leaf from a sub-tree.

  • Shape = Component, Circle/Square = Leaves, ShapeGroup = Composite.
  • ShapeGroup.draw() loops its children and calls their draw() — the recursion.
  • A group inside a group works automatically because a group is a Shape.
  • The same circle instance appears in two groups — the tree stores references, so shared children are fine here.
interface Shape { void draw(); }

class Circle implements Shape {
    @Override public void draw() { System.out.println("Drawing Circle"); }
}
class Square implements Shape {
    @Override public void draw() { System.out.println("Drawing Square"); }
}

// Composite -- is-a Shape and holds child Shapes; draw() cascades.
class ShapeGroup implements Shape {
    private final List<Shape> children = new ArrayList<>();
    public void add(Shape s)    { children.add(s); }
    public void remove(Shape s) { children.remove(s); }
    @Override public void draw() {
        System.out.println("Drawing a Group of Shapes:");
        for (Shape child : children) child.draw();
    }
}

public class App {
    public static void main(String[] args) {
        Shape circle = new Circle();
        Shape square = new Square();
        circle.draw();
        square.draw();

        System.out.println("---");
        ShapeGroup group = new ShapeGroup();
        group.add(circle);
        group.add(square);
        group.draw();

        System.out.println("---");
        ShapeGroup bigGroup = new ShapeGroup();   // a group that contains a group
        bigGroup.add(circle);
        bigGroup.add(group);
        bigGroup.draw();
    }
}

AdvancedThe recursive draw() is a depth-first pre-order traversal. Guarding against cycles matters in real trees: if you accidentally add a group to itself (or to one of its own descendants), the traversal recurses forever with a StackOverflowError. 10. Swing/AWT components & the Composite pattern

Objective: See the AWT/Swing Container/Component hierarchy as a live Composite: containers hold components, are themselves components, and rendering/layout cascade through the tree.

In AWT/Swing, Component is the Component role and a leaf widget such as JButton or JTextField is a Leaf. Container (and its subclass JPanel) is the Composite: it holds child Components via add(...) and — crucially — a Container is a Component, so panels nest inside panels to any depth. When the framework paints or lays out a container, the call recurses into every child, exactly like ShapeGroup.draw(). The GUI is a tree, and the toolkit walks it uniformly.

  • Component = the shared type; JButton/JLabel = leaves; Container/JPanel = composites.
  • Container.add(Component) is the child-management operation; nesting is unbounded because a container is a component.
  • paint() and layout cascade recursively down the widget tree — the Composite benefit.
// java.awt.Container IS-A Component and CONTAINS Components -- a composite.
JPanel panel = new JPanel();       // Container, itself a Component
panel.add(new JButton("OK"));      // leaf
panel.add(new JTextField(20));     // leaf

JPanel outer = new JPanel();
outer.add(panel);                  // a composite INSIDE a composite
outer.add(new JLabel("Title"));    // leaf alongside it

// paint()/doLayout() on 'outer' cascades recursively through the whole tree.
frame.add(outer);

Pitfall Claiming a leaf like JButton rejects children at runtime — it does not. Child management (add(Component)) is declared on java.awt.Container, not on the shared java.awt.Component type, so at the AWT level this is actually the safe variant. The wart is that every Swing widget extends JComponent → Container, so even a "leaf" like JButton inherits add() and will silently accept a nonsensical child — nothing throws; the child is simply added. 14. System design I: APIs, web services & software architectures

SOLID and the GoF patterns shape design in the small — classes and objects inside one process. System design zooms out to the whole system: how programs talk to each other over a network (APIs and web services), and how you slice a system into deployable units (monolith, layered, SOA, microservices). This is theory — the goal is a correct mental model and an honest grasp of the trade-offs.

01. System design — overview

Objective: Shift from designing individual classes to designing whole systems, and survey what "system design" actually covers.

Everything so far has been design in the small: SOLID and design patterns decide which class owns a method and how objects collaborate inside a single process. System design is design in the large: how independent programs, often on different machines, cooperate to serve many users reliably. The questions change scale — instead of "which class is responsible for this?" you ask "how do components communicate, how is the system structured, where does the data live, and how does it stay fast and available as load grows?"

A working vocabulary clusters into four recurring themes: communication — APIs and web services (SOAP, RPC, REST) that let programs call each other over a network; architecture style — monolith, layered, SOA, microservices, i.e. how you carve the system into deployable pieces; data storage — SQL vs NoSQL, replication, sharding; and scaling & availability — vertical vs horizontal scaling, load balancing, caching, and the CAP trade-offs. This section covers the first two; storage and scaling follow.

  • Class design is in-the-small (one process); system design is in-the-large (many cooperating programs).
  • Four themes: communication (APIs), architecture style, data storage, scaling/availability.
  • There is rarely one "right" architecture — every choice trades against team size, load, and complexity.
  • SOLID and patterns still apply inside each service; system design is a layer above them.
AdvancedConway's law — a system tends to mirror the communication structure of the organisation that builds it — is a key reason architecture and team topology are chosen together, not separately. A three-team org will find it hard to ship a system that isn't roughly three-part.
Pitfall Over-engineering: reaching for microservices, message brokers, and distributed infrastructure before the load or the team size justifies it. Most systems should start simple and distribute only when a concrete constraint forces it.

02. What are APIs and web services?

Objective: Distinguish an API (any contract for calling code) from a web service (an API exposed over a network), and frame the request/response model.

An API (Application Programming Interface) is any well-defined contract that lets one piece of software use another without knowing its internals — a Java interface, a library's public methods, and an operating-system syscall are all APIs. A web service is the special case where that contract is exposed over a network, typically over HTTP, so a client on one machine can invoke functionality on a server elsewhere. In short: every web service is an API; not every API is a web service.

The interaction is client–server and request/response: the client sends a request (an operation/verb, an address, headers, and often a body); the server does the work and returns a response (a status and usually a body). The two sides are decoupled — they can be written in different languages and deployed independently — as long as both honour the agreed contract and message format (XML, JSON, protobuf…). SOAP, RPC, and REST are three styles for defining that contract.

  • API = any contract to call code; web service = an API exposed over a network (usually HTTP).
  • All web services are APIs; not all APIs are web services (a local library API isn't).
  • Model: client–server, request/response; the two ends are language- and deployment-independent.
  • The contract fixes the address, the operation, and the message format both sides agree on.
AdvancedIn everyday speech "API" now often means "web API". Besides SOAP/RPC/REST there is GraphQL, where the client sends a query that shapes exactly which fields the response contains — solving REST's over-/under-fetching at the cost of a more complex server and caching story.
Pitfall Conflating the transport with the style. HTTP is the pipe; REST, SOAP, and RPC are different ways of using that pipe. "We use HTTP" says nothing about whether the API is RESTful.

03. SOAP

Objective: Understand SOAP as a strict, XML-based, contract-first web-service protocol, and weigh its heavyweight standards against its rigidity.

SOAP (Simple Object Access Protocol) is a protocol, not merely a style. Every message is an XML document called an Envelope, containing an optional Header (metadata — security, routing, transactions) and a mandatory Body (the payload, or a Fault element on error). The service contract is described formally in a WSDL (Web Services Description Language) document that lists operations, message types, and endpoints, so clients can generate strongly-typed stubs. SOAP is transport-agnostic (HTTP, SMTP, JMS…) though HTTP is by far the most common.

Its strength is a rich, standardized ecosystem: the WS-* stack adds WS-Security (message-level signing and encryption), WS-ReliableMessaging, and WS-AtomicTransaction. That is why SOAP still appears in banking, telecom, and enterprise integration, where formal contracts and built-in guarantees matter. The cost is weight — verbose XML, a strict contract that is awkward to evolve, and more tooling than a plain JSON API needs. Modern public and mobile APIs overwhelmingly prefer REST or gRPC; SOAP endures mostly in established enterprise systems.

  • Messages are XML Envelopes: optional Header + mandatory Body (a Fault on error).
  • The contract is a machine-readable WSDL, enabling generated client stubs and strong typing.
  • Transport-agnostic, but the WS-* standards (especially WS-Security) are its real differentiator.
  • Trade-off: strong guarantees and a formal contract vs verbosity, rigidity, and tooling overhead.

A minimal SOAP request envelope over HTTP:

<?xml version="1.0"?>
<soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
  <soap:Header>
    <!-- optional: WS-Security tokens, routing, transaction context -->
  </soap:Header>
  <soap:Body>
    <getPrice xmlns="http://example.com/stock">
      <symbol>ACME</symbol>
    </getPrice>
  </soap:Body>
</soap:Envelope>

AdvancedBecause the contract lives in WSDL + XSD, SOAP gives compile-time contract checking and cross-language stub generation. WS-Security operates at the message level, not just the transport level (TLS), so a message can stay signed/encrypted end-to-end across intermediaries — something plain HTTPS cannot offer.
Pitfall Choosing SOAP for a simple CRUD or public web/mobile API. The XML and WS-* overhead buys formal guarantees you probably don't need, and most client ecosystems (browsers, mobile) expect lightweight JSON, not envelopes.

04. RPC

Objective: Understand Remote Procedure Call — invoking a remote function as if it were local — and place gRPC in the modern landscape versus REST.

RPC (Remote Procedure Call) makes a call to another machine look like an ordinary local function call: the client invokes a stub, the RPC framework marshals (serializes) the arguments, ships them over the network, the server executes the real procedure, and the result is marshalled back. The programming model is action/verb-oriented — you call named procedures like getPrice(symbol) or createUser(data) — in contrast to REST, which is resource/noun-oriented around URIs and a fixed set of HTTP verbs.

RPC spans a long history: XML-RPC and later JSON-RPC send method calls as XML/JSON over HTTP; the modern standard-bearer is gRPC, which uses Protocol Buffers (a compact binary format) over HTTP/2, generates typed client and server code from a .proto contract, and supports streaming. gRPC excels for high-throughput, internal service-to-service calls; REST remains the default for public, browser-friendly APIs.

  • RPC: call a remote procedure as if local — the framework marshals args, transports, and returns the result.
  • Verb/action-oriented (getPrice) vs REST's resource/noun-oriented (GET /prices/ACME).
  • Evolution: XML-RPC → JSON-RPC → gRPC (protobuf over HTTP/2, code generation, streaming).
  • gRPC shines for internal, high-performance traffic; REST for public/browser APIs.

A gRPC contract and the client call it generates — the remote call reads like a local method:

// stock.proto -- the contract; codegen produces typed stubs in many languages
service StockService {
  rpc GetPrice(PriceRequest) returns (PriceReply);
}
message PriceRequest { string symbol = 1; }
message PriceReply   { double price  = 1; }

// Java client -- feels local, but the stub marshals it over HTTP/2:
PriceReply r = stub.getPrice(
        PriceRequest.newBuilder().setSymbol("ACME").build());
System.out.println(r.getPrice());

AdvancedgRPC's binary protobuf is not human-readable and needs a shim (grpc-web) to reach browsers. In return the .proto gives cross-language codegen and disciplined schema evolution: fields are identified by numbers, so you add new fields with new numbers and never reuse or renumber old ones.
Pitfall The "remote is just like local" illusion. A network call can be slow, time out, or fail partially in ways a local call never does — the classic fallacies of distributed computing (the network is not reliable, latency is not zero, bandwidth is not infinite). Treating RPC as free and reliable produces fragile systems.

05. REST

Objective: Understand REST as an architectural style built on resources, URIs, HTTP verbs, statelessness, and representations, and recall its six constraints.

REST (REpresentational State Transfer), defined by Roy Fielding, is an architectural style, not a protocol. Everything is a resource identified by a URI (e.g. /customers/42); clients act on resources with the standard HTTP verbs — GET (read), POST (create), PUT/PATCH (update), DELETE (remove) — and exchange representations of resource state, most commonly JSON. Interactions are stateless: each request carries everything the server needs, so the server keeps no client session between calls, which makes REST easy to scale horizontally.

HTTP status codes carry the outcome: 2xx success (200 OK, 201 Created, 204 No Content), 4xx client error (400, 401, 403, 404, 409), 5xx server error (500, 503). Fielding's six constraints define the style: client–server, stateless, cacheable, uniform interface, layered system, and (optional) code on demand. Honouring them — especially the uniform interface and statelessness — is what makes an API genuinely "RESTful" rather than merely "HTTP + JSON".

  • Resources + URIs; act with HTTP verbs; exchange representations (usually JSON).
  • Stateless: each request is self-contained → easy horizontal scaling and caching.
  • Status codes signal outcome: 2xx ok, 4xx client error, 5xx server error.
  • Six constraints: client–server, stateless, cacheable, uniform interface, layered system, code-on-demand (optional).

A REST request and its JSON response:

GET /api/v1/customers/42 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer <token>
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 42,
  "name": "Ada Lovelace",
  "email": "ada@example.com",
  "_links": { "self": "/api/v1/customers/42" }
}

Note the verb semantics: safe methods (GET, HEAD) must not change state; idempotent methods (GET, PUT, DELETE) can be retried without extra effect — POST is neither.
Pitfall The verb-in-URL anti-pattern: POST /createCustomer or GET /getCustomer?id=42. That is RPC wearing REST's clothes. Model nouns (/customers) and let the HTTP verb express the action.

06. API & web-service design principles

Objective: Collect the practical disciplines that make a web API usable, evolvable, and safe — independent of which style you picked.

Choosing a style (REST, gRPC, SOAP) is only the start. Good web APIs share a set of design disciplines that decide how easy the API is to consume, how safely it can change over time, and how it behaves under load and abuse. These are the rules reviewers look for.

  • Consistent naming: plural nouns for resources (/customers/42/orders), lowercase, hyphenated; verbs belong in the HTTP method, not the path.
  • Versioning: version the contract (/v1/… or a version header) so you can evolve without breaking existing clients.
  • Statelessness: keep no server-side session; pass auth/context on every request — this is what enables scaling and caching.
  • Correct status codes & error bodies: return the right 2xx/4xx/5xx and a machine-readable error payload — never "200 OK" with an error hidden inside.
  • Idempotency: GET/PUT/DELETE are idempotent; make unsafe operations retry-safe (e.g. an idempotency key on POST) so a retry doesn't double-charge.
  • Pagination, filtering, sorting: never return an unbounded list — page large collections (?page=/?limit= or cursors).
  • Security: HTTPS everywhere, real authentication/authorization (OAuth2 / JWT / API keys), least privilege, and validate all input.
  • Rate limiting & quotas: protect the service from overload and abuse; signal limits via headers and return 429 Too Many Requests.
  • Documentation: describe the contract with OpenAPI/Swagger so clients and code generators stay in sync.
  • Backwards compatibility: add fields, don't remove or rename them; deprecate before deleting.

Versioning, pagination, and an idempotency key in practice:

GET  /api/v1/customers?page=3&limit=50&sort=name   # version + pagination
POST /api/v1/payments
Idempotency-Key: 5f3b9c2a-...                       # safe to retry, no double charge

AdvancedAn OpenAPI document enables contract-first development, mock servers, and client-SDK generation. ETag + If-None-Match gives HTTP caching via conditional GET (a 304 Not Modified when unchanged); ETag + If-Match gives optimistic concurrency on updates — the server returns 412 Precondition Failed if the resource changed since the client's copy.
Pitfall Tunnelling everything through 200 OK and encoding success/failure in the body. It breaks clients, HTTP caches, and monitoring, all of which rely on the status code to tell success from failure.

07. The monolithic architecture

Objective: Define the monolith — one codebase, one deployable, one process — and recognise why it is the right default for many systems.

A monolithic application packages all functionality — UI, business logic, data access — into a single codebase that builds into one deployable artifact and runs in one process. Modules call each other with ordinary in-process method calls, share one database, and are released together.

Monoliths are simple where it counts early: one repository to clone, one thing to run locally, straightforward end-to-end debugging, no network hops or serialization between modules, and easy transactional consistency because everything shares one database. For a small team or a young product, a well-structured monolith ships faster and with far less operational burden than a distributed system. "Monolith" is not a slur — it is frequently the correct starting point.

  • One codebase → one deployable artifact → one process; modules call each other in-process.
  • Shared single database → simple transactions and strong consistency.
  • Easy to build, run locally, test end-to-end, and deploy early on.
  • Often the right default — distribute only when a real constraint forces it.
AdvancedA modular monolith enforces clear internal boundaries (well-separated packages/modules with explicit APIs between them), so the code stays maintainable and could later be split into services. You get much of the discipline of microservices without paying the distribution cost.
Pitfall Assuming "monolith" means "messy". Cleanliness is orthogonal to architecture: a monolith can be well-structured and modular, and a microservice system can be a distributed big ball of mud. Don't split to escape a mess you could untangle in place.

08. Problems with monolithic apps

Objective: Understand where monoliths strain as the codebase, the team, and the load grow.

The monolith's virtues erode with scale. Because everything deploys together, you must redeploy the whole application to change one line, and you can only scale by running more copies of the entire app — even when just one module (say, image processing) is the bottleneck. You cannot scale that hot module on its own.

Other pressures pile up: tight coupling makes changes ripple unpredictably; builds, test suites, and startup times grow until the feedback loop is slow; the whole system is locked to a single technology stack; a single bad deploy or a memory leak in one module can take everything down (a large blast radius); and many developers contending in one codebase and one release train step on each other. None of these is fatal, but past a certain size they push teams toward splitting the system.

  • Scaling is all-or-nothing: you replicate the whole app, not the one hot module.
  • Tight coupling → changes ripple; a large codebase → slow builds, tests, and startup.
  • Locked to a single technology stack across the whole application.
  • One fault (leak, crash) can bring down the entire process — large blast radius.
  • Many developers on one codebase and one release train slows large teams down.
AdvancedMost of these are scaling-of-organisation and scaling-of-load problems as much as code problems (Conway's law again). They appear only past a certain size — which is exactly why premature splitting is itself a mistake.
Pitfall Splitting into microservices to cure a coupling problem you could fix with better module boundaries. If the modules are still entangled, you get the same coupling — now stretched over the network, where it is slower and much harder to change.

09. Layered (n-tier) architecture

Objective: Understand the classic layered organisation — presentation, business, persistence, database — as separation of concerns, usually within a monolith.

The layered (n-tier) architecture organises code into horizontal layers, each with one responsibility: the presentation layer (UI / controllers) handles requests and formatting; the business layer holds domain logic and rules; the persistence layer (repositories / DAOs) maps objects to storage; and the database layer stores the data. The defining rule is that each layer talks only to the layer directly below it, so concerns stay separated and each layer can be tested or replaced in isolation.

Layering is an organisational pattern, not a deployment pattern — a layered application is still typically shipped as a single deployable (a monolith). Its clarity makes it the most common enterprise structure, but taken too literally it can degrade into an architecture sinkhole, where requests pass straight through layers that add nothing, or into a layered monolith that is hard to scale by feature.

  • Layers: presentation → business → persistence → database, each a single responsibility.
  • Closed layers: each depends only on the one below → strong separation of concerns.
  • An internal structure, not a deployment style — usually still one deployable (a monolith).
  • Easy to understand and test; watch for pass-through "sinkhole" layers and feature-scaling limits.
AdvancedContrast with hexagonal (ports & adapters) and clean architecture, which invert the dependencies: the domain sits at the centre and infrastructure depends inward on it, not the other way around. That is the Dependency Inversion Principle applied at architecture scale.
Pitfall Leaking responsibilities across layers — raw SQL in a controller, HTTP concerns in the domain. It defeats the entire point of layering; keep each layer's job pure so it can be tested and replaced independently.

10. Service-oriented architecture (SOA)

Objective: Understand SOA as enterprise-scale, coarse-grained services integrated through shared infrastructure, and contrast it with microservices.

SOA structures an enterprise as a set of coarse-grained, reusable services — an "Order service", a "Customer service" — that expose business capabilities to be shared across many applications. Integration is often mediated by an Enterprise Service Bus (ESB): a central backbone that handles routing, protocol translation, message transformation, and orchestration between services. SOA emphasises enterprise-wide reuse and governance, and historically leaned on SOAP / WS-* contracts.

Microservices are often called "SOA done right" or "fine-grained SOA", but the philosophies differ. SOA is enterprise-centric with heavy, shared middleware (the ESB is smart, the endpoints simpler) and frequently shared data. Microservices are application-centric, fine-grained, independently deployable, and favour "smart endpoints and dumb pipes" (logic in the services, lightweight transport like REST or messaging), giving each service its own data store. In one line: SOA centralises; microservices decentralise.

  • SOA: coarse-grained, reusable business services shared across the enterprise.
  • Often integrated through a central ESB (routing, transformation, orchestration) — "smart pipes".
  • Emphasis on enterprise reuse and governance; historically SOAP / WS-* based.
  • Vs microservices: SOA centralises (ESB, shared data, heavier); microservices decentralise (fine-grained, own data, dumb pipes).
AdvancedThe ESB is exactly what can turn into a central bottleneck and a single point of coupling: every integration flows through it, so it accumulates logic and becomes hard to change. That is a major reason microservices push logic out to the endpoints and deliberately keep the transport "dumb".
Pitfall Recreating an ESB-shaped "god service" or a shared database inside a microservices system. It quietly reintroduces the central coupling microservices were meant to remove — you get the costs of distribution without the independence.

11. Microservices architecture

Objective: Understand microservices — small, independently deployable services around business capabilities — with an honest account of both the benefits and the substantial costs.

A microservices architecture decomposes a system into many small services, each owning a single business capability (orders, payments, catalog), each independently developed, deployed, and scaled, and each with its own private data store. Services never reach into another's database; they communicate over the network — synchronously via REST/gRPC or asynchronously via a message broker.

The upside is real: teams deploy and scale their service independently (scale only the hot service), pick the technology that fits each service, and contain a failure so it does not take the whole system down (fault isolation). Small services map cleanly to small, autonomous teams.

But microservices trade in-process simplicity for distributed-systems complexity, and that bill is large. Network calls add latency and can fail partially; there is no single database, so consistency becomes eventual and cross-service transactions need patterns like sagas; observability requires distributed tracing and log aggregation; and operations demand CI/CD, containers/orchestration, service discovery, and versioning. Microservices solve the scaling problems of large systems and organisations — for a small team or product they usually add more cost than they remove.

  • Small services around business capabilities; each independently deployable and scalable.
  • Database-per-service: no shared DB; communicate over the network (REST/gRPC or messaging).
  • Benefits: independent deploy/scale, technology diversity, fault isolation, team autonomy.
  • Costs: network latency and partial failure, eventual consistency (sagas), observability, heavy operations overhead.
  • Right when scale and team size justify it; wrong as a default for small systems.
AdvancedManaging distributed data usually means embracing eventual consistency and the saga pattern (choreography or orchestration) instead of a single ACID transaction. Resilience patterns — timeouts, retries with backoff, circuit breakers, bulkheads — stop being optional and become mandatory once every call can fail independently.
Pitfall The distributed monolith: services so chatty and tightly coupled that they must be deployed together. You pay every network and operations cost of microservices while keeping all the coupling of a monolith. Split along genuine business boundaries — or not at all.

15. System design II: Data storage & the CAP theorem

Every backend eventually lives or dies by its data layer. This section surveys the two great families of stores — relational (SQL, ACID, schema-first) and non-relational (NoSQL, BASE, scale-first) — how each is tuned and scaled, and the fundamental limit every distributed store runs into: the CAP theorem. This is design theory, so there is no runnable Java here — just prose, diagrams, and small illustrative snippets.

01. Relational databases

Objective: Understand the relational model — tables, schema, relationships and foreign keys, SQL, and ACID transactions — and when it is the right default.

A relational database (PostgreSQL, MySQL, Oracle, SQL Server) stores data in tables: a table is a set of rows (records), each row a fixed set of typed columns. The column definitions form a schema fixed up front — every row of customer has exactly the same shape, and the database rejects rows that don't fit. You read and write with SQL (Structured Query Language), a declarative language where you describe what you want and the query planner decides how to fetch it.

Tables are related to one another through foreign keys: a column in one table that references the primary key of another (an order's customer_id points at a row in customer). This lets you model one-to-many and many-to-many relationships and reassemble them at query time with a JOIN, while the engine enforces referential integrity (you can't create an order for a customer who doesn't exist). Data is usually normalized — each fact stored exactly once, split across tables — so an update happens in one place and there is no chance of contradictory copies.

The relational model's headline guarantee is the ACID transaction: a group of statements that either all take effect or none do. Atomicity (all-or-nothing), Consistency (every commit leaves the database in a valid state, honouring constraints), Isolation (concurrent transactions don't see each other's half-finished work), and Durability (once committed, it survives a crash). This is what makes SQL databases the default for money, inventory, bookings — anywhere a partial or lost write is unacceptable.

  • Schema-first: structure is defined up front and enforced on every write; strong typing and constraints catch bad data early.
  • Relationships via foreign keys; reassemble normalized data with JOIN; the engine guarantees referential integrity.
  • ACID = Atomicity, Consistency, Isolation, Durability — the gold standard for correctness under concurrency and failure.
  • Normalization stores each fact once, eliminating update anomalies at the cost of more joins.
  • Use when data is structured and relational, queries are varied/ad-hoc, and transactional correctness matters more than raw horizontal scale.
Illustrative SQL
CREATE TABLE customer (
    id     BIGINT       PRIMARY KEY,
    name   VARCHAR(100) NOT NULL,
    email  VARCHAR(255) UNIQUE
);

CREATE TABLE orders (
    id           BIGINT PRIMARY KEY,
    customer_id  BIGINT NOT NULL REFERENCES customer(id),  -- foreign key
    total_cents  INT    NOT NULL,
    placed_at    TIMESTAMP DEFAULT now()
);

-- Reassemble the 1:N relationship with a JOIN.
SELECT c.name, o.id, o.total_cents
FROM   orders o
JOIN   customer c ON c.id = o.customer_id
WHERE  o.total_cents > 5000
ORDER  BY o.placed_at DESC;
AdvancedThe I in ACID is a spectrum, not a switch: SQL defines four isolation levels — read uncommitted, read committed, repeatable read, serializable — trading anomalies (dirty reads, non-repeatable reads, phantoms) for concurrency. Most engines default to read committed; only serializable makes concurrent transactions behave as if they ran one at a time, and it costs throughput. Knowing your engine's default level prevents subtle "it works until it's busy" bugs.
Pitfall Treating the schema as an afterthought. In a relational store the schema is the design — picking the wrong keys, missing a foreign key, or storing a comma-separated list where a child table belongs creates anomalies that no amount of application code fully fixes. Model the relationships first, then write the tables.

02. Relational database optimization & scaling

Objective: Learn how to make a single relational database fast (indexing, query tuning, caching, pooling) and how far it scales — and hit the wall that is scaling writes on one primary.

The first lever is the index. Without one, finding rows means a full table scan — O(n). A B-tree index keeps the indexed column sorted in a shallow, balanced tree so a lookup is O(log n), turning a scan of millions of rows into a handful of page reads. Indexes make reads dramatically faster but are not free: every INSERT, UPDATE and DELETE must also update every affected index, and each index consumes storage. You index the columns you filter and join on — not every column.

Beyond indexing come query optimization (read the query plan, avoid SELECT *, kill N+1 query patterns), denormalization (deliberately duplicating a column to skip an expensive join on a hot read path — trading write complexity and redundancy for read speed), caching (put a fast in-memory store like Redis in front of the database so repeated reads never touch it), and connection pooling (reuse a small fixed set of expensive database connections across many requests instead of opening one per request).

When one machine isn't enough, the first move is vertical scaling — a bigger box with more CPU, RAM and faster disks. It's simple but has a hard ceiling and gets expensive fast. The next move is read replicas: the primary streams its changes to follower copies, and read-only queries are spread across the followers. This scales reads beautifully. The catch: writes still all go to the single primary. You cannot scale writes by adding replicas — every write must funnel through one node, and that primary is the hard limit of the relational architecture. Pushing past it means either sharding (splitting the data, lecture 4) or a different data model entirely.

  • B-tree index: O(log n) reads instead of O(n) scans; costs write throughput and storage, so index selectively.
  • Denormalization + caching trade write complexity/redundancy for read speed on hot paths.
  • Connection pooling amortizes the cost of establishing database connections across requests.
  • Vertical scaling = bigger box (simple, capped); read replicas = replication that scales reads.
  • Writes don't scale horizontally on a single primary — that single write node is the architectural ceiling.
Illustrative SQL
-- Turn a full scan into a B-tree lookup on the hot filter column.
CREATE INDEX idx_orders_customer ON orders(customer_id);

-- Now this uses the index instead of scanning every order:
SELECT * FROM orders WHERE customer_id = 42;

-- Cost of the index: every write to orders must also maintain it.
--   read  path : O(log n)   (win)
--   write path : slower, + extra storage   (price)
AdvancedReplication is not automatically free of anomalies. Streaming changes to followers introduces replication lag: a read routed to a replica just after a write may return stale data (the "read your own writes" problem). Fixes include routing a user's reads to the primary briefly after they write, or using synchronous replication for critical reads — which trades latency for freshness. This foreshadows the consistency-vs-availability tension of the CAP theorem in lecture 5.
Pitfall Adding indexes reflexively "to make it faster." On a write-heavy table, over-indexing can make the system slower overall — each write pays to maintain ten indexes, most of which no query uses. Measure with the query planner (EXPLAIN), index the columns real queries filter and join on, and drop indexes nothing reads.

03. Non-relational (NoSQL) databases

Objective: Survey the four NoSQL families, understand schema flexibility and the BASE model, and know when NoSQL fits and when it doesn't.

NoSQL ("not only SQL") is an umbrella for stores that abandon the rigid table-and-join model to gain horizontal scalability and schema flexibility. They are typically denormalized — you store data pre-joined in the shape your queries need — and built from day one to spread across many commodity nodes. There are four main families:

  • Key-value (Redis, DynamoDB, Memcached): a giant distributed hash map — store and fetch an opaque value by key. Blistering for caches, sessions, counters; no querying inside the value.
  • Document (MongoDB, Couchbase): store self-contained JSON/BSON documents; fields can vary row to row, nested structures replace joins. Great for catalogs, user profiles, content.
  • Column-family / wide-column (Cassandra, HBase, Bigtable): rows keyed and partitioned, columns grouped into families; optimized for enormous write throughput and time-series/event data across many nodes.
  • Graph (Neo4j, Amazon Neptune): first-class nodes and edges; traversing relationships (friends-of-friends, fraud rings, recommendations) is the core operation, not an expensive join.

Because they distribute, most NoSQL stores relax ACID in favour of BASE: Basically Available (the system keeps serving even under partial failure), Soft state (state may be in flux without new input, as replicas sync), and Eventual consistency (given no new writes, all replicas eventually converge on the same value — but a read right after a write may see stale data). BASE is the pragmatic counterpart to ACID: you accept temporary inconsistency to gain availability and scale.

When NoSQL fits: massive scale beyond one machine, very high write throughput, flexible or varied data whose shape changes over time, and access patterns known in advance (you shape the data to the query). When it doesn't: complex ad-hoc queries you can't predict, rich multi-entity relationships needing joins, and workloads demanding strong multi-row transactions — there, a relational database is still the better tool. The honest answer is often both: a relational store of record plus a NoSQL cache or specialized store (polyglot persistence).

Illustrative document (MongoDB-style)
{
  "_id": "ord_10473",
  "customer": { "id": 42, "name": "Ada" },   // embedded, denormalized
  "items": [
    { "sku": "A1", "qty": 2 },
    { "sku": "B7", "qty": 1 }
  ],
  "total_cents": 5300,
  "status": "SHIPPED"
}
Advanced"NoSQL is schemaless" is a half-truth. There is no schema enforced by the database, but there is always a schema enforced by your application code — it just moved. Schema flexibility means new fields are cheap and mixed shapes coexist, which is powerful during rapid iteration; the trade is that the database no longer stops you from writing malformed data, so validation and versioning become the application's job (many document stores now offer optional schema validation to claw some of that back).
Pitfall Reaching for NoSQL because it sounds modern, then discovering your data is highly relational and every feature needs another join you now have to do by hand in application code. NoSQL removes joins — it does not remove the relationships. If your queries are varied and relationship-heavy, that work reappears in your code, slower and buggier.

04. Scaling NoSQL — replication & sharding

Objective: Distinguish the two orthogonal scaling techniques — replication (copies) and sharding (partitioning) — and the trade-offs each brings.

Replication keeps multiple copies of the same data on different nodes. It buys availability (if one node dies, another serves the data) and read scaling (spread reads across copies). Two broad styles: leader-follower (single-leader) — one node accepts writes and streams them to followers, like the relational replicas of lecture 2; and leaderless (Dynamo-style, as in Cassandra) — any replica accepts writes, and consistency is reconciled with quorum reads/writes and conflict resolution. Replication alone does not scale writes: single-leader still funnels writes through one node, and leaderless still stores every full copy on every replica.

Sharding (a.k.a. partitioning) splits the dataset across nodes so each holds only a slice. You pick a shard key, and a routing function (hash or range of that key) decides which shard owns each record. This is how you scale writes — different keys land on different shards, so write load spreads across the cluster instead of hammering one primary. In practice you combine both: shard for write capacity, then replicate each shard for availability.

Sharding's power comes with sharp trade-offs. A poorly chosen shard key creates a hot shard — one partition taking most of the traffic (e.g. sharding orders by country when 90% are one country). Rebalancing when you add nodes means moving data, which is disruptive (consistent hashing softens this). Cross-shard queries and joins are hard: any query that isn't scoped to a single shard key must fan out to every shard and merge results, and multi-shard transactions largely disappear. The whole game is choosing a shard key with high cardinality, even distribution, and alignment with your dominant query pattern.

  • Replication = copies: availability + read scale. Leader-follower vs leaderless (quorum). Does not scale writes.
  • Sharding = partition by shard key: the way to scale writes horizontally.
  • They are orthogonal — real clusters shard for capacity and replicate each shard for resilience.
  • Traps: hot shards, costly rebalancing, and cross-shard joins/transactions becoming impractical.
  • A good shard key has high cardinality, spreads load evenly, and matches your dominant query.
Routing by shard key (pseudocode)
// Hash the shard key to pick a partition; replicate each partition N times.
int shardId = Math.floorMod(hash(order.customerId), SHARD_COUNT);  // e.g. 0..3

// Good shard key  -> even spread, most queries hit ONE shard.
// Bad  shard key  -> "hot" shard, or every query fans out to ALL shards.
// Cross-shard query (no shard key in the filter) = scatter/gather + merge.
AdvancedRange vs hash partitioning is a classic trade. Range sharding (partition by key intervals) keeps ordered scans efficient but invites hot spots — sequential keys like timestamps all pile onto the newest shard. Hash sharding spreads load evenly but destroys locality, so range scans must hit every shard. Consistent hashing is the standard trick to add/remove nodes while relocating only a small fraction of keys, avoiding a full reshuffle.
Pitfall Choosing a low-cardinality or skewed shard key (status, boolean flag, country, "current month"). It looks fine in testing and creates a hot shard in production — one node melts while the rest idle. Resharding a live system to fix a bad key is one of the most painful operations in data infrastructure; choose the key deliberately, up front.

05. The CAP theorem

Objective: State the CAP theorem precisely, correct the "pick any 2" misconception, classify systems as CP or AP, and know the PACELC refinement.

The CAP theorem (Brewer) concerns any store spread across multiple nodes and names three properties: Consistency (every read sees the most recent write, or an error — all nodes agree), Availability (every request gets a non-error response, even if not the newest data), and Partition tolerance (the system keeps working when the network drops or delays messages between nodes). The theorem's real content: when a network partition occurs, you must choose between C and A — you cannot have both. If two halves of a partitioned cluster can't talk, either they keep accepting writes and risk diverging (choose A, sacrifice C), or they refuse writes to stay consistent (choose C, sacrifice A).

The common phrasing "pick any 2 of 3, always" is wrong, and worth correcting. In a distributed system, P is not optional — networks will partition, so a real system must tolerate it. That leaves the genuine choice as C vs A, and only during a partition. When the network is healthy (the normal case), a well-built system delivers both consistency and availability. CAP is a statement about behaviour under failure, not a permanent two-out-of-three tax.

So distributed stores are classified by which they sacrifice under partition. CP systems (consistency + partition tolerance) refuse or block requests rather than serve stale/conflicting data — e.g. HBase, ZooKeeper, MongoDB in its default configuration, and strongly/synchronously replicated relational setups. AP systems (availability + partition tolerance) keep answering during a partition and reconcile later via eventual consistency — e.g. Cassandra, DynamoDB/Dynamo-style stores, Riak, CouchDB. The choice follows the domain: a bank ledger leans CP (a wrong balance is unacceptable), a shopping cart or social feed leans AP (always-on matters more than a moment's staleness).

  • C = all nodes agree on the latest write; A = every request gets an answer; P = survives dropped messages between nodes.
  • The trade-off only bites during a partition: then you must pick C or A. Healthy network → you can have both.
  • P is mandatory in any distributed system — so the real design choice is CP vs AP under partition.
  • CP: HBase, ZooKeeper, many RDBMS setups. AP: Cassandra, DynamoDB, Riak.
  • PACELC extends CAP: if Partition then A or C, Else (no partition) trade Latency vs Consistency — even in normal operation, stronger consistency costs latency.
AdvancedPACELC (Abadi) exposes what CAP omits: the everyday cost when there is no partition. Read it as "if Partition then A vs C, Else Latency vs Consistency." Dynamo-style stores are typically PA/EL (favour availability under partition, low latency otherwise); a strongly-consistent store is PC/EC (consistent always, at latency's expense). It's a more honest lens because most of a system's life is spent without a partition — and consistency still isn't free then.
Pitfall Reading CAP as "consistency and availability are permanently mutually exclusive, so I must give one up forever." Not so — the sacrifice is conditional on a partition. Design for the healthy path to deliver both, and make a deliberate, documented choice about what happens during a partition. Vague hand-waving here is how split-brain data corruption ships to production.

Wrapping up

That is the whole arc — from principles to patterns to systems. The SOLID principles are the foundation: keep each class responsible for one thing, open it for extension but closed for modification, make subtypes honour their supertype's contract, split fat interfaces, and depend on abstractions rather than concretions. The design patterns are named, battle-tested answers to recurring design problems: creational patterns control how objects are made, behavioral patterns control how objects collaborate and vary, and structural patterns control how objects are composed. None of them is a goal in itself — a pattern earns its place only when it removes real duplication, coupling, or rigidity; reach for one because the design pressure is there, not to show that you know it. The system design chapters zoom out to the same idea at architectural scale: choose your API style, your architecture (monolith, layered, SOA, microservices), and your data store by the trade-offs they impose, not by fashion. The throughline from the first principle to the last theorem is the same — program to interfaces, isolate what changes, and make every dependency and trade-off explicit.
