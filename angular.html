Angular

Angular is a batteries-included, TypeScript-first framework for building single-page applications. It gives you components with a powerful template syntax, dependency injection, a router, forms, an HTTP client, and — since Angular 16–17 — a signals-based reactivity model and a new block control-flow syntax (@if, @for, @switch, @defer). This guide moves from first principles (what a component is) through the directives, DI, change detection, and lifecycle machinery, and on to the modern, signals-first Angular of versions 17–22: standalone components, deferrable views, the Resource API, and zoneless change detection. This docsite is fully offline and bundles no Angular compiler, so components cannot be built or executed live. For every concept the source is shown in a static code block paired, side by side, with an Output box that states the simulated rendered result, console output, or CLI diagnostic. Read the code on the left and the outcome on the right.

Note: version tags like (v17+) mark when an API became stable. The block control flow and deferrable views landed in v17; signal input()/output()/model() in v17.1–17.2; the Resource API and zoneless change detection matured across v18–20. Everything here targets the standalone, signals-first style Angular recommends going forward.

Introduction to Angular

Angular is a full-featured, TypeScript-first framework for building single-page applications (SPAs). Unlike a page that reloads on every navigation, an SPA loads once and then updates the DOM in place as the user interacts. Angular's job is to keep the visible page (the view) synchronized with your application's data (the model) so you rarely touch the DOM by hand. The core building block is the component: a reusable piece of UI that behaves, in practice, like a custom HTML element you define yourself.

A component is two things fused together: a TypeScript class and a template. The class holds state and behavior — its fields form the model, and its methods respond to events. The template is HTML enriched with Angular syntax that describes what to render. Each component declares a selector, a CSS-like tag name such as app-greeting; wherever that tag appears in another template, Angular instantiates the component and renders its template there. This is why people say a component is essentially a custom element backed by a class.

The bridge between model and view is data binding, and modern Angular gives you three everyday flavors. Interpolation {{ expression }} renders a class field or expression as text inside the template. Property binding [prop]="expr" sets a DOM property or an input on a child component from a class expression. Event binding (event)="handler()" runs a class method when a DOM event fires. Interpolation and property binding flow data class → view; event binding flows view → class. Together they form a complete loop with no manual DOM manipulation.

Example: a minimal standalone component

import { Component } from '@angular/core';

@Component({
  selector: 'app-greeting',
  template: `<h1>Hello, {{ name }}!</h1>`,
})
export class Greeting {
  name = 'Angular';
}
OutputRenders the element: <h1>Hello, Angular!</h1> i.e. the page shows the heading "Hello, Angular!"

Example: interpolation vs. property binding

<!-- text content via interpolation -->
<p>{{ title }}</p>

<!-- a DOM property via property binding -->
<img [src]="logoUrl" [alt]="title" />
OutputIf title = "DevDocs" and logoUrl = "/logo.png": <p>DevDocs</p> <img src="/logo.png" alt="DevDocs">

Example: event binding into the class

@Component({
  selector: 'app-counter',
  template: `
    <button (click)="increment()">Clicked {{ count }}x</button>
  `,
})
export class Counter {
  count = 0;
  increment() { this.count++; }
}
OutputInitial button label: "Clicked 0x" After three clicks: "Clicked 3x" (The class field is the model; the view updates automatically.)

Note: In modern Angular, expressions in {{ }} should stay simple and side-effect free — they can be re-evaluated often. Put real logic in class methods or, better, in computed signals, and only read the result in the template.

Why Angular: key features

Angular is an opinionated, batteries-included framework rather than a small view library. A router, a two-flavored forms system, an HttpClient, dependency injection (DI), internationalization, and a first-party CLI all ship as official, version-aligned packages under the @angular scope. Because these pieces are designed together, you spend less time choosing and gluing third-party libraries and more time building features — and upgrades move the whole stack forward in lockstep.

The framework is TypeScript-first: components, services, and templates are all type-checked, and Angular's template type-checking catches binding mistakes at build time. Its component + template model keeps UI modular and testable. Since Angular v16+, reactivity is built on signals — signal(), computed(), and effect() — giving fine-grained, predictable updates that are gradually replacing the older Zone.js change detection. Dependency injection makes code testable: any dependency can be swapped for a fake in a test, and the modern inject() function makes wiring concise.

Finally, Angular is known for its tooling and longevity. The CLI (ng) scaffolds projects, generates components, runs a dev server with hot reload, builds optimized bundles, and runs tests. Angular follows a predictable release cadence (roughly two major versions per year) with a documented support window and automated ng update migrations (schematics) that rewrite your code for you. For teams, that means a stable, well-signposted upgrade path over many years rather than churn.

Setup, CLI & versions

Angular runs on Node.js and uses npm (or an alternative like pnpm/yarn) to manage packages. Always install an LTS (Long-Term Support) release of Node — Angular pins supported versions per release, and recent Angular (v20+) requires Node 20 or 22. Using a non-LTS or too-old Node is the most common cause of confusing install errors, so check node --version first. A Node version manager (nvm on macOS/Linux, nvm-windows) makes it easy to keep the right version per project.

With Node in place, install the Angular CLI globally so the ng command is available everywhere: npm install -g @angular/cli. The CLI is your primary interface to Angular — it scaffolds, serves, generates, tests, and builds. Verify the install and see the aligned versions of the CLI, framework, and TypeScript with ng version. The important folders are src/app (your components and services), src/main.ts (the bootstrap entry point), and angular.json (build and serve configuration).

TypeScript jumpstart

Angular is written in and designed around TypeScript, a superset of JavaScript that adds a static type system. You get autocompletion, refactoring safety, and compile-time error checking — mistakes surface in your editor instead of at runtime. Angular leans on several TS features so heavily that understanding them is understanding Angular. The first is classes: every component and service is a class, with fields (state) and methods (behavior).

Next are decorators — the @Something(...) annotations attached to a class. Angular uses @Component to mark a class as a component (and supply its selector and template) and @Injectable to mark a class as a service that participates in dependency injection. Decorators are metadata: they tell Angular how to treat the class. You'll also use interfaces and type aliases to describe the shape of your data models, giving your app self-documenting, checkable contracts.

Rounding out the essentials: access modifiers (private, protected, public) and readonly control visibility and mutability; generics like Array<User> or Signal<number> let types be parameterized; strict null checks (on by default in ng new) force you to handle null/undefined explicitly; and union and literal types ('idle' | 'loading' | 'error') model a fixed set of allowed values precisely. Angular's template type-checking uses all of these to validate your bindings.

Example: an interface as a data model

export interface User {
  id: number;
  name: string;
  email: string;
  role: 'admin' | 'member' | 'guest'; // literal union
  avatarUrl?: string;                  // optional
}
OutputAssigning role: 'owner' fails to compile: Type '"owner"' is not assignable to type '"admin" | "member" | "guest"'.

Example: a class using modifiers & generics

export class UserStore {
  private readonly users: User[] = [];

  add(user: User): void {
    this.users.push(user);
  }

  find(id: number): User | undefined {
    return this.users.find(u => u.id === id);
  }
}
Outputfind() returns User | undefined — strict null checks force callers to handle the "not found" case before reading .name, preventing runtime "undefined" errors.

Example: a decorated, injectable service

import { Injectable, signal } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class Counter {
  private readonly value = signal<number>(0);
  readonly count = this.value.asReadonly();

  increment(): void {
    this.value.update(n => n + 1);
  }
}
Output@Injectable + providedIn:'root' registers one shared instance app-wide. Signal<number> gives a typed, reactive counter exposed read-only to consumers.

Note: Keep strict mode on (the ng new default). The extra up-front discipline around null, exact types, and access modifiers eliminates a large class of runtime bugs and makes Angular's template type-checking far more useful.

Your first component

A component is created by attaching the @Component decorator to a class. The decorator's configuration object drives everything Angular needs. selector is the custom tag name. The view can be defined inline with template or in a separate file with templateUrl; styles likewise use inline styles or an external styleUrl. In modern Angular, components are standalone by default, so they list their own template dependencies in an imports array instead of belonging to an NgModule.

The class beneath the decorator holds the model and behavior. To render the root component into the page, main.ts calls bootstrapApplication(App, appConfig), which mounts the root component's selector (found in index.html) and supplies app-wide providers. From there, you compose UI by nesting: import a child component into a parent's imports, then place the child's selector — as if it were a native element — inside the parent's template.

Inside templates you connect data with the bindings from the introduction: interpolation {{ }} for text, [prop] for properties and inputs, and (event) for events. A child exposes data entry points with the input() signal function (the modern replacement for the @Input() decorator), and the parent feeds them through property binding. This is how a component tree passes data downward while staying strongly typed end to end.

Example: inline template & bootstrap

// app.ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-root',
  template: `<h1>{{ title }}</h1>`,
  styles: [`h1 { color: rebeccapurple; }`],
})
export class App {
  title = 'DevDocs';
}

// main.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { App } from './app/app';

bootstrapApplication(App);
Outputindex.html contains <app-root></app-root>. Angular replaces it, rendering: <h1 style="color: rebeccapurple">DevDocs</h1>

Example: external template file

// user-card.ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-user-card',
  templateUrl: './user-card.html',
  styleUrl: './user-card.css',
})
export class UserCard {
  name = 'Ada Lovelace';
}
Output<!-- user-card.html --> <p>{{ name }}</p> renders: <p>Ada Lovelace</p>

Example: nesting with an input

import { Component, input } from '@angular/core';

  selector: 'app-badge',
  template: `<span class="badge">{{ label() }}</span>`,
})
export class Badge {
  label = input.required<string>();
}

@Component({
  selector: 'app-root',
  imports: [Badge],
  template: `<app-badge [label]="status" />`,
})
export class App {
  status = 'Online';
}
OutputThe parent binds status → the child's label input: <span class="badge">Online</span>

Note: Prefer the signal-based input() / input.required() over the legacy @Input() decorator in new code (Angular v17.1+). Inputs are read as functions in the template — write {{ label() }}, not {{ label }}.

Component @Input

Angular components are composed into trees, and data flows down from a parent to its children. An @Input() decorator marks a class property as a public entry point: the parent writes to it through template binding. Inside the child's class you simply declare @Input() name = '';, and the parent supplies a value with the square-bracket syntax [name]="expr". The brackets mean "property binding" — Angular evaluates expr as a component expression and assigns the result. Without brackets, as in name="Ada", the value is passed as a literal string, which is fine only when you really want a constant string.

Inputs can be aliased so the public template name differs from the internal field name: @Input('userName') name = ''; exposes the binding as [userName] while your code keeps using this.name. You can also intercept assignments with a setter/getter pair, which is the classic way to run logic every time a new value arrives — recomputing derived state, logging, or normalising input. A plain default value on the property acts as the fallback when the parent binds nothing.

Modern Angular hardens inputs with options. @Input({ required: true }) makes the compiler error at build time if a parent forgets to bind the input, catching wiring mistakes early. The transform option runs a pure function on the incoming value: the built-in booleanAttribute turns the presence of an attribute (or the strings "true"/"") into a real boolean, and numberAttribute coerces a string to a number. This mirrors how native HTML boolean attributes behave.

Example: basic input + property binding

import { Component, Input } from '@angular/core';

@Component({
  selector: 'app-greeting',
  standalone: true,
  template: `<p>Hello, {{ name }}!</p>`,
})
export class GreetingComponent {
  @Input() name = 'stranger';
}

// Parent template:
// <app-greeting [name]="userName" />
// <app-greeting name="Grace" />   // literal string
OutputWith userName = 'Ada': <p>Hello, Ada!</p> <p>Hello, Grace!</p>

Example: aliasing + setter that reacts to changes

@Component({
  selector: 'app-price',
  standalone: true,
  template: `<span>{{ formatted }}</span>`,
})
export class PriceComponent {
  formatted = '';

  // public binding name is [amount], field is _cents
  @Input('amount')
  set amount(value: number) {
    this._cents = value;
    this.formatted = '$' + (value / 100).toFixed(2);
  }
  get amount() { return this._cents; }
  private _cents = 0;
}

// Parent: <app-price [amount]="1999" />
Output<span>$19.99</span> Every time [amount] changes, the setter recomputes `formatted` automatically.

Example: required inputs + transforms

import { Component, Input, booleanAttribute, numberAttribute }
  from '@angular/core';

@Component({
  selector: 'app-badge',
  standalone: true,
  template: `<b>{{ label }} ({{ count }}) {{ active ? 'ON' : 'off' }}</b>`,
})
export class BadgeComponent {
  @Input({ required: true }) label!: string;
  @Input({ transform: numberAttribute }) count = 0;
  @Input({ transform: booleanAttribute }) active = false;
}

// <app-badge label="Inbox" count="42" active />
// <app-badge count="7" />   // compile error: label required
Output<b>Inbox (42) ON</b> count="42" (string) becomes number 42. Bare `active` attribute becomes true. Omitting label fails the build.

Note: @Input() is the decorator-based API and still fully supported. Angular v17.1+ also offers the signal-based input() function (name = input.required<string>()), which is reactive and read as name(). Prefer signal inputs in new code, but the concepts here transfer directly.

@Output & EventEmitter

If inputs push data down the tree, outputs send events up. A child cannot reach into its parent, but it can announce that something happened and let the parent decide how to respond. You declare an output as a property initialised to an EventEmitter: @Output() saved = new EventEmitter<string>();. The generic parameter is the payload type. When the child wants to notify the parent, it calls this.saved.emit(payload).

The parent subscribes using the parentheses (event) binding syntax: (saved)="onSaved($event)". Here $event is the exact value passed to emit() — not a DOM event. This is the same binding mechanism used for native events like (click), which keeps the mental model uniform: parentheses always mean "listen for something." Outputs can also be aliased, e.g. @Output('valueChange') change = new EventEmitter<number>(), so the template name and internal name can differ.

Together, @Input and @Output form the fundamental parent-child contract. A common pattern is a "dumb" presentational child that receives data via inputs and emits user intent via outputs, while a "smart" parent owns the state and reacts to those events. This keeps children reusable and side-effect free.

Example: child emits a value on click

import { Component, Output, EventEmitter } from '@angular/core';

@Component({
  selector: 'app-rater',
  standalone: true,
  template: `
    <button (click)="rate(1)">👍</button>
    <button (click)="rate(-1)">👎</button>
  `,
})
export class RaterComponent {
  @Output() rated = new EventEmitter<number>();
  rate(delta: number) { this.rated.emit(delta); }
}
OutputRenders two buttons. Clicking 👍 emits 1, 👎 emits -1 to whoever is listening on (rated).

Example: parent handles the event

@Component({
  selector: 'app-feed',
  standalone: true,
  imports: [RaterComponent],
  template: `
    <app-rater (rated)="onRated($event)" />
    <p>Score: {{ score }}</p>
  `,
})
export class FeedComponent {
  score = 0;
  onRated(delta: number) { this.score += delta; }
}
OutputStart: Score: 0 Click 👍 → Score: 1 Click 👍 → Score: 2 Click 👎 → Score: 1 $event is the emitted number.

Example: counter child with aliased output

@Component({
  selector: 'app-counter',
  standalone: true,
  template: `
    <button (click)="dec()">-</button>
    <span>{{ value }}</span>
    <button (click)="inc()">+</button>
  `,
})
export class CounterComponent {
  value = 0;
  // internal name `change`, public event `valueChange`
  @Output('valueChange') change = new EventEmitter<number>();
  inc() { this.value++; this.change.emit(this.value); }
  dec() { this.value--; this.change.emit(this.value); }
}

// Parent: <app-counter (valueChange)="log($event)" />
OutputClicking + three times emits: 1, then 2, then 3 Parent's log($event) receives each value.

Note: never emit inside a getter or a template expression — that fires on every change-detection cycle. Emit only from real user actions or lifecycle events. The signal-based output() function (v17.3+) is the modern alternative and does not need EventEmitter.

Two-way binding & ngModel

Two-way binding lets a value flow down into a child and changes flow back up, using a single compact syntax: [(x)]="value". The nickname is the "banana in a box" because the parentheses (the banana) sit inside the brackets (the box). It is pure syntactic sugar. Angular desugars [(x)]="value" into two separate bindings: a property binding [x]="value" plus an event binding (xChange)="value = $event". The convention is strict — for any input named x, the paired output must be named xChange for the shorthand to work.

The most familiar case is [(ngModel)], which binds a form control's value to a component property. It comes from FormsModule, so a standalone component must add it to its imports array. Typing in the input updates the property, and changing the property programmatically updates the input — both directions stay in sync automatically.

You can build your own two-way property by exposing an input and a matching Change output. Follow the naming rule and any consumer can use the banana-in-a-box shorthand on your component. This is exactly how ngModel and Angular Material components work under the hood.

Example: [(ngModel)] with FormsModule

import { Component } from '@angular/core';
import { FormsModule } from '@angular/forms';

@Component({
  selector: 'app-name-form',
  standalone: true,
  imports: [FormsModule],
  template: `
    <input [(ngModel)]="name" placeholder="Type..." />
    <p>Hello, {{ name || '???' }}</p>
  `,
})
export class NameFormComponent {
  name = '';
}
OutputType "Lin" in the box → <p>Hello, Lin</p> Clearing the box → <p>Hello, ???</p>

Example: the desugaring, written out

<!-- shorthand -->
<app-toggle [(on)]="isOn"></app-toggle>

<!-- is exactly equivalent to -->
<app-toggle
  [on]="isOn"
  (onChange)="isOn = $event">
</app-toggle>
OutputBoth forms behave identically. Input `on` receives isOn. Output `onChange` writes $event back into isOn — a full round trip.

Example: a custom two-way [(on)] property

import { Component, Input, Output, EventEmitter }
  from '@angular/core';

@Component({
  selector: 'app-toggle',
  standalone: true,
  template: `<button (click)="flip()">{{ on ? 'ON' : 'OFF' }}</button>`,
})
export class ToggleComponent {
  @Input() on = false;
  @Output() onChange = new EventEmitter<boolean>();
  flip() {
    this.on = !this.on;
    this.onChange.emit(this.on);   // completes the loop
  }
}
// Parent: <app-toggle [(on)]="isOn" />
OutputButton shows OFF (isOn = false). Click → emits true → parent's isOn = true. Button now shows ON, and any other binding to isOn updates too.

Note: since v17.2 the model() signal function replaces this boilerplate entirely — on = model(false) creates a writable signal that is automatically two-way bindable, no separate Change output required. It is covered later; the desugaring rule above still explains what happens under the hood.

Control flow: @if / @else

Angular v17 introduced built-in control flow blocks that live directly in the template syntax rather than as structural directives. The @if block conditionally renders a chunk of template when its expression is truthy. Unlike the old *ngIf, these blocks are part of the framework's syntax: there is nothing to import and no directive to add to imports. They are also faster and produce clearer type-narrowing for the compiler.

The full form chains with @else if (…) and a final @else, evaluated top to bottom until one condition matches — much like a JavaScript if/else ladder. The condition is any component expression: a property, a method call, or a signal read such as isLoggedIn(). Only the matching branch is instantiated in the DOM; the others are not created at all, which keeps the DOM minimal.

A powerful feature is aliasing the evaluated condition with as: @if (user(); as u) { … } stores the (truthy) result in the local variable u for the block's scope. This is ideal for unwrapping a nullable value or the emission of an async signal exactly once, avoiding repeated calls and giving you a non-null, well-typed reference inside the block.

Example: basic @if / @else

@if (isLoggedIn()) {
  <p>Welcome back!</p>
} @else {
  <a href="/login">Please sign in</a>
}
OutputisLoggedIn() === true → <p>Welcome back!</p> isLoggedIn() === false → <a href="/login">Please sign in</a>

Example: @else if ladder

@if (status() === 'loading') {
  <spinner />
} @else if (status() === 'error') {
  <p class="err">Something broke.</p>
} @else {
  <p>Loaded {{ count() }} items.</p>
}
Outputstatus()='loading' → <spinner> status()='error' → <p class="err">Something broke.</p> status()='ready' → <p>Loaded 12 items.</p> Only one branch ever exists in the DOM.

Example: aliasing the result with `as`

@if (user(); as u) {
  <p>{{ u.name }} — {{ u.email }}</p>
} @else {
  <p>No user loaded.</p>
}
<!-- `u` is the non-null result of user();
     called once, typed as User (not User|null) -->
Outputuser() = { name:'Rae', email:'r@x.io' } → <p>Rae — r@x.io</p> user() = null → <p>No user loaded.</p>

Note: @if fully replaces *ngIf. Run ng generate @angular/core:control-flow to auto-migrate an existing codebase. The as alias is scoped only to that block — you cannot use u in the @else.

Control flow: @for

The @for block iterates over a collection and renders its template once per item: @for (item of items; track item.id) { … }. The track expression is mandatory — the compiler rejects a @for without it. Its purpose is identity: track tells Angular how to recognise the "same" item across renders so it can reuse existing DOM nodes instead of destroying and recreating them. Tracking by a stable unique key like item.id yields the best performance and preserves focus, scroll, and animation state; use track $index only for primitive lists that never reorder.

Inside the block Angular exposes contextual variables. $index is the zero-based position, $count is the total length, $first and $last are booleans for the ends, and $even/$odd reflect the index parity — handy for zebra striping. You can alias them (let i = $index) if a nested loop needs to disambiguate.

An optional @empty block renders when the collection has zero items, giving you a clean fallback without a separate @if. @for blocks nest freely; each level has its own contextual variables. This block replaces *ngFor and, like @if, needs no import.

Example: basic @for with track

@for (todo of todos(); track todo.id) {
  <li>{{ todo.title }}</li>
}
Outputtodos() = [ {id:1,title:'Buy milk'}, {id:2,title:'Walk dog'} ] <li>Buy milk</li> <li>Walk dog</li>

Example: contextual variables + @empty

@for (u of users(); track u.id) {
  <div [class.stripe]="$even">
    {{ $index + 1 }}/{{ $count }}: {{ u.name }}
    @if ($first) { <b>(top)</b> }
    @if ($last)  { <b>(end)</b> }
  </div>
} @empty {
  <p>No users found.</p>
}
Outputusers = [Ana, Bo, Cy]: <div class="stripe">1/3: Ana <b>(top)</b></div> <div>2/3: Bo</div> <div class="stripe">3/3: Cy <b>(end)</b></div> users = []: <p>No users found.</p>

Example: nested loops with aliased index

@for (row of grid(); track row.id; let r = $index) {
  @for (cell of row.cells; track cell.id; let c = $index) {
    <span>[{{ r }},{{ c }}]={{ cell.value }}</span>
  }
}
Outputgrid = 2 rows x 2 cells: <span>[0,0]=A</span><span>[0,1]=B</span> <span>[1,0]=C</span><span>[1,1]=D</span> Aliasing keeps inner/outer $index distinct.

Note: choosing a bad track (e.g. tracking a whole object or an unstable value) forces Angular to recreate every row on each update, tanking performance and losing DOM state. Always track by a stable identifier when one exists.

Control flow: @switch

The @switch block is Angular's built-in control-flow primitive for choosing exactly one branch out of many based on a single value. It was introduced alongside @if and @for in v17 and is written directly in the template as @switch (expr) { @case (v) { ... } @default { ... } }. The runtime evaluates expr once, then renders the first @case whose value matches, or the @default block if none match. Unlike the older *ngSwitch, it requires no imports and produces no wrapper elements.

Matching uses strict equality — the exact same semantics as JavaScript's ===. That means @case (1) matches the number 1 but not the string '1', and object cases match by reference identity, not structural equality. There is intentionally no fall-through: once a case matches, only that block renders and evaluation stops, so you never need a break statement the way you would in a C-style switch. Cases are also not required to be exhaustive; if nothing matches and there is no @default, nothing is rendered.

Prefer @switch over a long chain of @if / @else if when every branch is testing the same expression against a set of discrete literal values — a status string, an enum, a numeric mode. It reads more clearly, evaluates the discriminant a single time, and pairs naturally with TypeScript discriminated unions. Reach for @if chains instead when each branch tests a different condition.

Example: status string

@switch (status()) {
  @case ('loading') { <p>Loading…</p> }
  @case ('success') { <p>Done!</p> }
  @case ('error')   { <p>Failed.</p> }
  @default          { <p>Idle</p> }
}
OutputWhen status() is 'success':
<p>Done!</p>

Example: numeric case, strict compare

@switch (code()) {
  @case (200) { <span>OK</span> }
  @case (404) { <span>Not found</span> }
  @default    { <span>Other</span> }
}
Outputcode() === 404 renders:
<span>Not found</span>
The string '404' would NOT match — falls to Other.

Example: no default, no match

@switch (role()) {
  @case ('admin')  { <button>Manage</button> }
  @case ('editor') { <button>Edit</button> }
}
OutputWhen role() is 'guest':
(nothing rendered — no matching case and no @default)

Note: @case compares with strict ===. A common bug is a numeric value coming in as a string (e.g. from a route param) and silently falling through to @default. Coerce the type before switching.

*ngIf / *ngFor / *ngSwitch

Before the built-in @-block syntax, Angular expressed conditional and repeated rendering with structural directives: *ngIf, *ngFor, and *ngSwitch. The asterisk is syntactic sugar — *ngIf desugars to an <ng-template> that the directive stamps in or out of the DOM. These directives are still fully supported and appear across the majority of existing codebases, so you must be able to read and maintain them.

They are not available by default, though. In a standalone component you must import them, either individually as NgIf, NgFor, NgSwitch from @angular/common, or in bulk via CommonModule. Forgetting the import produces a template error like "Can't bind to 'ngIf' since it isn't a known property". *ngIf supports an else branch that points at a named template: *ngIf="ok; else fallback" paired with <ng-template #fallback>. Inside expressions the safe-navigation ("Elvis") operator user?.name guards against null/undefined without throwing.

*ngFor exposes rich local variables — index, first, last, even, odd, count — assigned via let i = index, and a trackBy function so Angular can reuse DOM nodes across changes instead of destroying and rebuilding them. [ngSwitch] works with the *ngSwitchCase and *ngSwitchDefault companion directives on child elements.

Compared with the new control flow, the directives require imports, are slower (they run through the generic directive machinery rather than dedicated compiled code), and give weaker type narrowing. The @-blocks are built into the template compiler, need no imports, narrow types precisely, ship a faster diffing algorithm for @for, and make track mandatory. New code should prefer @if/@for/@switch; the directives remain for legacy and interop.

Example: *ngIf with else template

<p *ngIf="user; else anon">
  Hi {{ user?.name }}
</p>
<ng-template #anon>
  <p>Please sign in</p>
</ng-template>
OutputWhen user is null:
<p>Please sign in</p>
The Elvis operator keeps user?.name safe.

Example: *ngFor with trackBy and locals

<li *ngFor="let t of todos;
      let i = index; let last = last;
      trackBy: byId"
    [class.end]="last">
  {{ i + 1 }}. {{ t.text }}
</li>
Output<li>1. Buy milk</li>
<li class="end">2. Walk dog</li>
byId lets Angular reuse rows across updates.

Example: odd / even zebra rows

<tr *ngFor="let r of rows; let odd = odd"
    [class.shaded]="odd">
  <td>{{ r }}</td>
</tr>
Output<tr><td>A</td></tr>
<tr class="shaded"><td>B</td></tr>
<tr><td>C</td></tr>

Example: [ngSwitch]

<div [ngSwitch]="tab">
  <p *ngSwitchCase="'home'">Home</p>
  <p *ngSwitchCase="'help'">Help</p>
  <p *ngSwitchDefault>404</p>
</div>
OutputWhen tab === 'help':
<div><p>Help</p></div>

Note: In standalone components remember to add CommonModule (or the specific NgIf/NgFor/NgSwitch) to imports. The @-blocks need no import at all — one reason to prefer them in new code.

The @let block

Introduced in v18, @let declares a template-local variable: @let name = expression;. It lets you name the result of an expression once and reuse that name throughout the template instead of repeating the expression — the template equivalent of a local const. The declaration ends with a semicolon and the variable becomes available immediately after it, on the same line's following siblings and any nested content.

A key benefit is computing something a single time. Writing @let u = user(); reads the signal once and binds every subsequent {{ u.name }}, {{ u.email }} to that captured value, which is clearer and avoids scattering calls to user(). You can build derived values too, such as @let total = price() + tax();. The variable is reactive: whenever a dependency of its expression changes, Angular re-evaluates the @let and updates every place it is used.

Scope follows the view: a @let is visible in the view where it is declared and in any views nested below it (inside @if, @for, <ng-template>), but not in sibling or parent views. @let variables are read-only — you cannot assign to them from the template. They pair especially well with the async pipe: @let vm = data$ | async; subscribes once and gives every binding a stable snapshot, replacing the old *ngIf="data$ | async as vm" pattern.

Example: capture a signal once

@let u = user();
<h3>{{ u.name }}</h3>
<p>{{ u.email }}</p>
Output<h3>Ada Lovelace</h3>
<p>ada@calc.io</p>
user() is read a single time.

Example: derived value

@let total = price() + tax();
<span>Total: {{ total | currency }}</span>
Output<span>Total: $12.60</span>
Recomputes when price() or tax() changes.

Example: with the async pipe

@let vm = data$ | async;
@if (vm) {
  <ul>
    @for (x of vm.items; track x.id) {
      <li>{{ x.label }}</li>
    }
  </ul>
}
OutputOnce the observable emits:
<ul><li>First</li><li>Second</li></ul>
One subscription, shared snapshot.

Note: @let is read-only and scoped to its view and descendants. A variable declared inside an @if block is not visible after the block closes — declare it higher up if you need it outside.

ngClass & ngStyle

[ngClass] and [ngStyle] are directives from @angular/common that apply CSS classes and inline styles dynamically. [ngClass] accepts three input shapes: a string of space-separated class names ([ngClass]="'a b'"), an array of class names ([ngClass]="['a', 'b']"), or — most usefully — an object map where each key is a class name and each value is a boolean deciding whether that class is applied ([ngClass]="{ active: isOn, disabled: !enabled }").

[ngStyle] takes an object of CSS properties to values, e.g. [ngStyle]="{ color: c, 'font-weight': w }". Property names may carry a unit suffix so you can bind a plain number: 'font-size.px': 14 applies font-size: 14px, and 'width.%': 50 applies width: 50%. Both directives require CommonModule (or the standalone NgClass/NgStyle) in a standalone component's imports.

Angular also has native class and style bindings that need no directive and no import: [class.active]="isOn" toggles a single class, and [style.color]="c" or [style.width.px]="w" sets a single property. Prefer these native bindings when you are toggling one or two known classes/properties — they are lighter and read cleanly. Reach for [ngClass]/[ngStyle] when the set of classes or styles is dynamic, computed as an object, or too numerous to list individually.

Example: ngClass object map

<button
  [ngClass]="{
    primary: isPrimary,
    disabled: !enabled
  }">Save</button>
OutputisPrimary=true, enabled=false:
<button class="primary disabled">Save</button>

Example: ngStyle with units

<p [ngStyle]="{
     color: color,
     'font-size.px': size
   }">Hi</p>
Outputcolor='red', size=20:
<p style="color: red; font-size: 20px;">Hi</p>

Example: native [class.x] binding

<li [class.active]="isSelected">
  Item
</li>
OutputisSelected=true:
<li class="active">Item</li>
No import needed for [class.x].

Example: native [style.x.unit] binding

<div [style.width.%]="pct"
     [style.color]="'navy'">
  bar
</div>
Outputpct=75:
<div style="width: 75%; color: navy;">bar</div>

Note: For a single known class or style, prefer native [class.x] / [style.x] bindings — they skip the directive overhead and object diffing that [ngClass]/[ngStyle] incur.

ng-container

<ng-container> is a logical grouping element that renders no DOM of its own. Angular processes its bindings and children but emits nothing in place of the container tag itself, so it is the tool for grouping or applying directives without polluting the markup with an extra <div>. This matters when a wrapper element would break CSS layout (flex/grid children, table structure) or semantic HTML.

Its classic use is hosting a structural directive when you have nothing to attach it to. Instead of adding a throwaway <div *ngIf>, you write <ng-container *ngIf="ok">, and only the inner content reaches the DOM. It also solves the "two structural directives on one element" restriction: since Angular forbids *ngIf and *ngFor on the same tag, you nest them — one directive on an <ng-container> and the other on the child.

<ng-container> works with the new control-flow blocks too — you can put @if or @for content inside one to group siblings under a shared condition without a wrapper. It is also the standard host for [ngTemplateOutlet], which renders a referenced <ng-template> in place; combined with an outlet context you can stamp reusable template fragments with no surrounding element.

Example: *ngIf with no wrapper div

<ng-container *ngIf="user">
  <span>{{ user.name }}</span>
  <span>{{ user.role }}</span>
</ng-container>
Output<span>Ada</span><span>Admin</span>
No wrapping element around the two spans.

Example: two structural directives

<ng-container *ngFor="let u of users">
  <li *ngIf="u.active">{{ u.name }}</li>
</ng-container>
Output<li>Ada</li>
<li>Grace</li>
Inactive users emit nothing — no stray container tags.

Example: grouping with @if

<ng-container>
  @if (loaded()) {
    <h2>Title</h2>
    <p>Body</p>
  }
</ng-container>
Output<h2>Title</h2><p>Body</p>
The container leaves no trace in the DOM.

Example: ngTemplateOutlet

<ng-template #greet let-n>
  <p>Hello {{ n }}</p>
</ng-template>

<ng-container
  [ngTemplateOutlet]="greet"
  [ngTemplateOutletContext]="{ $implicit: 'Sam' }">
</ng-container>
Output<p>Hello Sam</p>
Rendered in place with no host element.

Note: <ng-container> accepts structural directives, template bindings, and outlets but cannot take a class, id, or style — it never becomes a real element. Use a normal tag if you actually need a DOM node.

Attribute directives

An attribute directive attaches behavior to an existing host element without changing the DOM structure. You declare it with @Directive and give it an attribute-style selector such as selector: '[appHighlight]'. When Angular finds an element carrying that attribute, it instantiates the directive class and links it to that host element. Modern directives should be standalone: true (the default from v19) so they can be imported directly into a component's imports array.

To read or mutate the host element you inject() its ElementRef (a wrapper around the native node) and, for safe DOM manipulation, Renderer2. Prefer Renderer2 over touching nativeElement directly: it keeps your code compatible with server-side rendering and other non-browser platforms where nativeElement may not exist. Even better, prefer @HostBinding, which lets Angular manage the binding declaratively and participate in change detection.

@HostBinding binds a class property to something on the host: a class (@HostBinding('class.active')), a style (@HostBinding('style.color')), an attribute (@HostBinding('attr.role')), or a DOM property (@HostBinding('disabled')). Note the DOM property vs HTML attribute distinction: attr.* writes the static HTML attribute (useful for ARIA and things with no property), while a bare name binds the live JavaScript property on the element. @HostListener subscribes to host events, and @Input() on the directive lets a consumer configure it. Finally, exportAs names the directive instance so a template reference variable can grab it, e.g. #ref="appThing".

Example: highlight-on-hover directive

import { Directive, ElementRef, HostBinding,
  HostListener, Input, inject } from '@angular/core';

@Directive({ selector: '[appHighlight]', standalone: true })
export class HighlightDirective {
  private el = inject(ElementRef<HTMLElement>);

  @Input('appHighlight') color = 'yellow';
  @HostBinding('style.backgroundColor') bg = '';
  @HostBinding('style.cursor') cursor = 'pointer';

  @HostListener('mouseenter') onEnter() { this.bg = this.color; }
  @HostListener('mouseleave') onLeave() { this.bg = ''; }
}
OutputTemplate: <p [appHighlight]="'#cff'">Hover me</p>. On mouseenter the paragraph's background turns light-blue; on mouseleave it clears. The cursor shows a pointer at all times.

Example: Renderer2 vs direct DOM

import { Directive, ElementRef, Renderer2,
  HostListener, inject } from '@angular/core';

@Directive({ selector: '[appPulse]', standalone: true })
export class PulseDirective {
  private el = inject(ElementRef);
  private r = inject(Renderer2);

  @HostListener('click') pulse() {
    // Safe, SSR-friendly DOM writes:
    this.r.addClass(this.el.nativeElement, 'pulsing');
    this.r.setAttribute(this.el.nativeElement, 'aria-busy', 'true');
  }
}
OutputClicking the host adds the pulsing CSS class and sets aria-busy="true" on it. Using Renderer2 instead of nativeElement.classList keeps the directive usable under server-side rendering.

Example: exportAs + template reference variable

@Directive({
  selector: '[appToggle]',
  standalone: true,
  exportAs: 'appToggle',
})
export class ToggleDirective {
  open = false;
  toggle() { this.open = !this.open; }
}
OutputTemplate:
<div appToggle #t="appToggle">
  <button (click)="t.toggle()">Toggle</button>
  <p *ngIf="t.open">Now open</p>
</div>
The #t variable is the directive instance, so the template can call t.toggle() and read t.open directly.

Note: @HostBinding/@HostListener still work, but modern Angular also supports the host object in the decorator metadata (host: { '[class.active]': 'isActive', '(click)': 'onClick()' }). Both are equivalent; pick one style per project for consistency.

Custom structural directives

Structural directives add and remove elements from the DOM. The leading asterisk in *ngIf or *appUnless is pure syntactic sugar: Angular desugars it by wrapping the host element in an <ng-template> and moving the directive (plus its microsyntax expression) onto that template. So <div *appUnless="cond">…</div> becomes <ng-template [appUnless]="cond"><div>…</div></ng-template>. The template's content is not rendered until the directive explicitly instantiates it.

A structural directive inject()s two collaborators: TemplateRef, a handle to the wrapped template, and ViewContainerRef, a slot in the DOM where views can be created and destroyed. To render, call viewContainer.createEmbeddedView(templateRef); to remove, call viewContainer.clear(). The directive decides when and how many times to create views based on its inputs.

Directives can pass data to the template through a context object. The special $implicit property fills a bare let x in the template, and named properties fill let name = .... The microsyntax (the string after *) is a compact grammar that Angular parses into inputs and let- bindings — for example *appRepeat="5 as count" binds the value and exposes a variable.

Example: *appUnless (renders when false)

import { Directive, Input, TemplateRef,
  ViewContainerRef, inject } from '@angular/core';

@Directive({ selector: '[appUnless]', standalone: true })
export class UnlessDirective {
  private tpl = inject(TemplateRef<unknown>);
  private vcr = inject(ViewContainerRef);
  private shown = false;

  @Input() set appUnless(cond: boolean) {
    if (!cond && !this.shown) {
      this.vcr.createEmbeddedView(this.tpl);
      this.shown = true;
    } else if (cond && this.shown) {
      this.vcr.clear();
      this.shown = false;
    }
  }
}
OutputTemplate: <p *appUnless="isLoggedIn">Please sign in</p>. When isLoggedIn is false the paragraph renders; when it becomes true the view is cleared and the paragraph disappears.

Example: desugared form of *appUnless

<!-- What you write -->
<p *appUnless="isLoggedIn">Please sign in</p>

<!-- What Angular compiles it to -->
<ng-template [appUnless]="isLoggedIn">
  <p>Please sign in</p>
</ng-template>
OutputBoth forms behave identically. The * version is just a shorthand; the compiler produces the explicit <ng-template> with the directive applied as an ordinary input binding.

Example: *appRepeat with context ($implicit)

@Directive({ selector: '[appRepeat]', standalone: true })
export class RepeatDirective {
  private tpl = inject(TemplateRef<{ $implicit: number; index: number }>);
  private vcr = inject(ViewContainerRef);

  @Input() set appRepeat(count: number) {
    this.vcr.clear();
    for (let i = 0; i < count; i++) {
      this.vcr.createEmbeddedView(this.tpl, { $implicit: i + 1, index: i });
    }
  }
}
OutputTemplate:
<span *appRepeat="3; let n; let i = index">#{{n}}({{i}}) </span>
Renders: #1(0) #2(1) #3(2). let n receives $implicit; let i = index receives the named context property.

Note: In new code the built-in control flow (@if, @for, @switch) replaces most uses of *ngIf/*ngFor and is faster. Custom structural directives remain the right tool when you need reusable, parameterized rendering logic that the built-in blocks do not cover.

ng-template & ngTemplateOutlet

<ng-template> declares a chunk of markup that Angular parses but does not render immediately. It is inert until something instantiates it — either a structural directive or an explicit outlet. This makes it the foundation of deferred and reusable rendering: define the shape once, decide later where and how many times it appears.

You obtain a TemplateRef to a template in several ways. Placing a template reference variable on it (<ng-template #tpl>) lets you query it with @ViewChild('tpl') or reference it directly in the template. To render one, apply [ngTemplateOutlet]="tpl" to any element (commonly <ng-container>) and Angular stamps the template's content there. Pass data with [ngTemplateOutletContext]="{ $implicit: x, name: y }"; inside the template, let-value binds $implicit and let-n="name" binds a named key.

A powerful pattern is passing a TemplateRef into a component as an @Input(). The component owns the loop or layout logic but delegates the per-item markup to the caller's template. This inverts control cleanly: a generic list component renders each row using a template supplied from outside, so the consumer controls appearance without the component knowing anything about the data shape.

Example: ngTemplateOutlet with context

<ng-template #greeting let-name let-role="role">
  <p>Hello {{ name }} — {{ role }}</p>
</ng-template>

<ng-container
  [ngTemplateOutlet]="greeting"
  [ngTemplateOutletContext]="{ $implicit: 'Ada', role: 'admin' }">
</ng-container>
OutputRenders: Hello Ada — admin. let-name picks up $implicit ('Ada'); let-role="role" picks up the named context key ('admin').

Example: reusing one template twice

<ng-template #badge let-text>
  <span class="badge">{{ text }}</span>
</ng-template>

<ng-container *ngTemplateOutlet="badge; context: { $implicit: 'New' }"></ng-container>
<ng-container *ngTemplateOutlet="badge; context: { $implicit: 'Sale' }"></ng-container>
OutputRenders two badges: New and Sale. The same TemplateRef is instantiated twice with different context objects. The *ngTemplateOutlet microsyntax is an alternate shorthand for the property bindings.

Example: passing a row template into a list component

@Component({
  selector: 'app-list',
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <div *ngFor="let item of items">
      <ng-container
        [ngTemplateOutlet]="rowTpl"
        [ngTemplateOutletContext]="{ $implicit: item }">
      </ng-container>
    </div>`,
})
export class ListComponent {
  @Input() items: unknown[] = [];
  @Input() rowTpl!: TemplateRef<{ $implicit: unknown }>;
}
OutputConsumer:
<app-list [items]="users" [rowTpl]="row"></app-list>
<ng-template #row let-u><b>{{u.name}}</b></ng-template>
The list owns iteration; the caller controls how each row looks. Each user renders bold via the supplied template.

Note: A template is inert until instantiated, which makes <ng-template> ideal for lazy/optional content — nothing in it runs or renders until an outlet or structural directive creates its view.

ViewChild & content queries

A component can query elements, directives, and child components inside its own template. @ViewChild(ChildComponent) returns the first matching instance; @ViewChildren(ChildComponent) returns a live QueryList that updates as the view changes and can be iterated or subscribed to via its changes observable. You can query by component/directive type, by a template reference variable name (@ViewChild('ref')), or by an injection token.

Timing matters for the classic decorators. A view query with { static: true } resolves before ngOnInit, but only works for children that are unconditionally present (not inside *ngIf/@if). The default is dynamic (static: false), and those results are guaranteed available in ngAfterViewInit, after the child views have been created. Reading a value in the constructor is always too early.

By default a query returns the component/directive instance. The { read: ... } option changes what you get: @ViewChild(ChildComponent, { read: ElementRef }) returns the host element wrapper instead of the instance. Since v17.2 Angular offers signal queries — viewChild(), viewChildren(), and contentChild() — that return signals. They need no lifecycle hook: you read them like any signal (this.child()) and they are reactive, so effects and computeds can depend on them. Signal queries are the recommended modern approach.

Example: classic @ViewChild in ngAfterViewInit

@Component({ selector: 'app-parent', standalone: true,
  imports: [ChildComponent],
  template: `<app-child></app-child>` })
export class ParentComponent implements AfterViewInit {
  @ViewChild(ChildComponent) child!: ChildComponent;

  ngAfterViewInit() {
    console.log(this.child.title); // safe here
  }
}
OutputConsole: the child's title value, e.g. Child ready. Accessing this.child in the constructor or ngOnInit (with the default dynamic query) would be undefined.

Example: @ViewChildren QueryList + read: ElementRef

@Component({ /* template has three <app-item> rows */ })
export class ListComponent implements AfterViewInit {
  @ViewChildren(ItemComponent) items!: QueryList<ItemComponent>;
  @ViewChild('title', { read: ElementRef }) titleEl!: ElementRef;

  ngAfterViewInit() {
    console.log(this.items.length);           // 3
    console.log(this.titleEl.nativeElement.tagName); // 'H1'
    this.items.changes.subscribe(list => { /* reacts to add/remove */ });
  }
}
OutputConsole: 3 then H1. QueryList exposes .length, iteration, and a changes stream; { read: ElementRef } yields the DOM wrapper rather than a component/directive instance.

Example: modern signal queries (v17.2+)

import { Component, viewChild, viewChildren, effect } from '@angular/core';

@Component({ selector: 'app-host', standalone: true,
  imports: [ChildComponent],
  template: `<app-child /><app-child />` })
export class HostComponent {
  first = viewChild(ChildComponent);            // Signal<Child | undefined>
  all = viewChildren(ChildComponent);           // Signal<readonly Child[]>
  required = viewChild.required('title');       // Signal<ElementRef>

  constructor() {
    effect(() => console.log('count', this.all().length));
  }
}
OutputConsole (via effect): count 2. Signal queries are read as functions (this.all()), need no ngAfterViewInit, and integrate with effect()/computed(). viewChild.required guarantees a non-null result or throws.

Note: Prefer signal queries in new code — they eliminate lifecycle-timing bugs and are reactive by design. Use viewChild.required when the child must exist so you avoid undefined checks.

Content projection

Content projection lets a component render markup that the consumer places between its opening and closing tags. Inside the component's template, <ng-content> marks the slot where that projected content appears. This is Angular's version of transclusion / "slots" and is the basis for reusable wrappers like cards, dialogs, and tabs whose inner content is controlled by the caller.

A single bare <ng-content> projects everything. For multiple slots, add a CSS-selector to each: <ng-content select=".header"> captures children matching that selector, and a trailing selector-less <ng-content> catches the rest. The selector can match by class, attribute, element name, or component selector. Projected nodes keep their original component context — they are not re-parented into the host component's data.

To query projected content from the host component, use @ContentChild/@ContentChildren (or the signal form contentChild()). Crucially, this is where the view vs content distinction lives: view children are declared in the component's own template and resolve in ngAfterViewInit; content children are projected in from outside and resolve in ngAfterContentInit, which runs earlier. Mixing the two hooks up incorrectly is a common source of undefined query results.

View encapsulation

Angular scopes a component's styles so that CSS written in one component cannot accidentally bleed into another. This behaviour is controlled by the encapsulation field of the @Component decorator, whose value comes from the ViewEncapsulation enum. The default is ViewEncapsulation.Emulated, which emulates the isolation of the Shadow DOM without actually using it. At build time Angular rewrites your selectors and stamps unique attributes onto elements so the rules only ever match this component's markup.

Under Emulated, every host element receives a generated _nghost-xyz attribute and every element inside the template receives a matching _ngcontent-xyz attribute. A rule like p { color: red } is compiled to p[_ngcontent-xyz] { color: red }, so it can never touch a paragraph belonging to a different component. The :host pseudo-class targets the component's own host element, and :host-context(.theme) matches the host only when some ancestor carries the .theme class — perfect for theming.

The two other modes trade isolation for reach. ViewEncapsulation.None disables scoping entirely: your styles are injected as-is into the document <head> and become global, affecting the whole application. ViewEncapsulation.ShadowDom uses the browser's native Shadow DOM — the component gets a real shadow root, styles are truly encapsulated by the platform, and outside CSS cannot reach in at all. The legacy ::ng-deep combinator forces a rule to pierce through child encapsulation, but it is deprecated; prefer CSS custom properties for crossing component boundaries.

Example: emulated scoping (default)

@Component({

  standalone: true,
  template: `<p class="label">Hi</p>`,
  styles: [`
    :host { display: block; padding: 1rem; }
    .label { color: crimson; }
  `],
  // encapsulation: ViewEncapsulation.Emulated (default)
})
export class CardComponent {}
OutputRendered DOM (attributes generated): <app-card _nghost-a1 class="..."> <p _ngcontent-a1 class="label">Hi</p> </app-card> Compiled CSS: [_nghost-a1] { display: block; padding: 1rem; } .label[_ngcontent-a1] { color: crimson; }

SCSS architecture & theming

Angular scopes a component's styles to that component by default (see View encapsulation). You author those styles in whatever CSS dialect the project is configured for — plain CSS, SCSS, Sass, or Less. SCSS is the most common choice because it adds variables, @mixin/@include, nesting, and file @use composition on top of CSS while still compiling down to it. You select the dialect when scaffolding (ng new app --style=scss) or later in angular.json, and the CLI wires the SCSS compiler into the build for you.

A maintainable stylesheet architecture usually splits into three layers: global styles (src/styles.scss — resets, typography, utility classes, and CSS custom properties that define your theme), shared SCSS partials (_variables.scss, _mixins.scss) consumed with @use, and component styles (each component's own .scss, kept small and scoped). Because component styles are encapsulated, class names inside them can be short and local without collision risk.

Theming — including light/dark mode — is best done with CSS custom properties rather than SCSS variables. SCSS variables are resolved at build time and cannot change at runtime; CSS variables cascade and can be swapped live. Define your palette as CSS variables on :root, override them under a theme selector such as [data-theme="dark"], and toggle a single attribute at the document root to flip the whole app. Components read the variables (var(--surface)) and automatically restyle. The :host-context() selector lets an encapsulated component react to an ancestor theme class.

Pipes (built-in & custom)

A pipe transforms a value for display without mutating the underlying data. In a template you apply one with the | operator inside an interpolation or binding: {{ value | pipeName }}. Pipes are pure, declarative formatters — the component holds raw data (a Date, a number, a string) and the pipe renders a human-friendly version, keeping formatting logic out of your class.

Angular ships a rich catalog in @angular/common: date, currency, percent, number (alias decimal), json, uppercase, lowercase, titlecase, slice, keyvalue, and the special async pipe that unwraps Observables and Promises. With standalone components you import the ones you use (e.g. DatePipe, CurrencyPipe) directly into the component's imports array. Pipes accept parameters after a colon and can be chained left-to-right: {{ now | date:'short' | uppercase }}.

You write your own with the @Pipe decorator on a class implementing PipeTransform's transform(value, ...args) method. By default a pipe is pure: Angular memoizes it and only re-invokes transform when the input reference (or a primitive value) changes, which is highly efficient. Setting pure: false makes it impure, running on every change-detection cycle — necessary for pipes that depend on mutable data like arrays being pushed to, but a real performance cost. The async pipe is itself impure by necessity.

Example: built-ins with parameters

<p>{{ today | date:'fullDate' }}</p>
<p>{{ 1299.5 | currency:'USD' }}</p>
<p>{{ 0.734 | percent:'1.0-1' }}</p>
<p>{{ 'hello world' | titlecase }}</p>
<p>{{ [1,2,3,4] | slice:1:3 | json }}</p>
OutputThursday, July 5, 2026 $1,299.50 73% Hello World [2,3]

Example: chaining pipes

<!-- imports: [DatePipe, UpperCasePipe] -->
<span>{{ order.createdAt | date:'short' | uppercase }}</span>
Output7/5/26, 2:30 PM → after uppercase: 7/5/26, 2:30 PM (digits unchanged, AM/PM upper)

Example: a pure custom pipe

import { Pipe, PipeTransform } from '@angular/core';

@Pipe({ name: 'truncate', standalone: true })
export class TruncatePipe implements PipeTransform {
  transform(value: string, limit = 20, ellipsis = '…'): string {
    if (!value || value.length <= limit) return value;
    return value.slice(0, limit) + ellipsis;
  }
}
// template: {{ post.body | truncate:10 }}
OutputInput: "Angular pipes are great" Output: "Angular pi…" Re-runs only when post.body changes.

Example: impure pipe (runs every CD)

@Pipe({ name: 'filterDone', standalone: true, pure: false })
export class FilterDonePipe implements PipeTransform {
  transform(todos: Todo[]): Todo[] {
    return todos.filter(t => !t.done);
  }
}
// Needed because todos is mutated in place (push/splice).
OutputList updates even when the SAME array reference is mutated — but transform() runs on EVERY change-detection pass. Prefer immutable updates + a pure pipe.

Note: Default to pure pipes and update data immutably (return a new array) so the reference changes and the pure pipe re-runs. Reach for pure:false only when unavoidable, since impure pipes execute on every tick and can dominate render cost.

Services & @Injectable

A service is a plain TypeScript class that holds reusable logic or state — data fetching, business rules, caching, shared signals — that does not belong to any single component. Marking it with @Injectable() tells Angular's dependency-injection system it can be constructed and injected. The idiomatic form is @Injectable({ providedIn: 'root' }), which registers the service with the root injector as an application-wide singleton and, crucially, makes it tree-shakeable: if nothing injects it, the bundler drops it.

Services keep components thin. Instead of a component holding an HTTP call, an array of items, and mutation methods, you move that into a service; the component just injects it and renders. Because the root-provided service is a singleton, every component that injects it shares the same instance and therefore the same state — a natural way to share data across an app without a global store.

There are two ways to obtain a dependency: the classic constructor parameter with an access modifier, or the modern inject() function called in the field initializer or constructor context. inject() reads more cleanly, avoids long constructor signatures, and composes well with base classes and functional guards/resolvers. Both resolve against the same injector tree.

Example: a root singleton service

import { Injectable, signal } from '@angular/core';

  private items = signal<string[]>([]);
  readonly count = () => this.items().length;

  add(name: string) {
    this.items.update(list => [...list, name]);
  }
}
OutputOne shared instance across the whole app. add('Book'); add('Pen') → count() === 2 Tree-shaken away if never injected.

HttpClient & async pipe

Angular talks to REST APIs through HttpClient from @angular/common/http. In a modern standalone app you enable it once by adding provideHttpClient() to the providers of bootstrapApplication — no HttpClientModule needed. After that any service or component can inject(HttpClient).

Its methods — get<T>(), post<T>(), put<T>(), delete<T>() — return cold RxJS Observables: nothing happens until something subscribes, and each subscription fires its own request. The generic type parameter types the parsed JSON body, so get<User[]>(url) yields an Observable<User[]>. You shape the request with HttpParams for query strings and HttpHeaders (or a plain object) for headers, passed via the options argument.

You can subscribe manually in the class, but then you must remember to unsubscribe to avoid leaks and manually trigger change detection of the assigned value. The cleaner path is to keep the Observable as a field and bind it in the template with the async pipe: it subscribes for you, renders each emission, and unsubscribes automatically when the component is destroyed. Combined with @if you get a tidy loading/loaded pattern.

Example: enabling HttpClient at bootstrap

import { bootstrapApplication } from '@angular/platform-browser';

import { AppComponent } from './app.component';

bootstrapApplication(AppComponent, {
  providers: [provideHttpClient()],
});
OutputHttpClient is now injectable anywhere in the application. No NgModule required.

HTTP interceptors & global errors

An HTTP interceptor sits in the pipeline between HttpClient and the network. Every outgoing request passes through the interceptor chain on the way out, and every response (or error) passes back through it on the way in. This is the right place for cross-cutting HTTP concerns that should not be repeated in every service: attaching an auth token, logging, adding correlation ids, retrying, showing a global spinner, and centralized error handling.

Modern Angular (v15+) uses functional interceptors — a plain function of type HttpInterceptorFn that receives the request and a next handler and returns an Observable of the event stream. You register them with provideHttpClient(withInterceptors([...])), and they run in array order. Requests are immutable: you never mutate req directly — you call req.clone({ setHeaders: ... }) to produce a modified copy and pass it to next().

For error handling there are two complementary tools. Inside an interceptor, catchError lets you inspect failed responses (by HttpErrorResponse.status), react (redirect on 401, toast on 500), and either recover or rethrow. Separately, Angular's ErrorHandler provider is the last-resort hook for uncaught errors anywhere in the app (not just HTTP) — override it to log or report to a monitoring service.

Example: auth interceptor (add Authorization header)

// auth.interceptor.ts
import { HttpInterceptorFn } from '@angular/common/http';
import { inject } from '@angular/core';
import { AuthService } from './auth.service';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(AuthService).token();      // a signal/getter
  if (!token) return next(req);                    // pass through

  // Requests are immutable — clone with the extra header.
  const authReq = req.clone({
    setHeaders: { Authorization: `Bearer ${token}` },
  });
  return next(authReq);
};
OutputEvery request made through HttpClient now carries Authorization: Bearer <token> — services stay unaware of auth. Requests without a token pass through unchanged.

Example: global error interceptor

// error.interceptor.ts
import { HttpInterceptorFn, HttpErrorResponse } from '@angular/common/http';
import { inject } from '@angular/core';
import { catchError, throwError } from 'rxjs';
import { Router } from '@angular/router';
import { NotificationService } from './notification.service';

export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  const router = inject(Router);
  const notify = inject(NotificationService);
  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 401) router.navigate(['/login']);
      else notify.error(`Request failed (${err.status})`);
      return throwError(() => err);   // rethrow for callers
    }),
  );
};
OutputA 401 redirects to /login; any other failure pushes a message into the global NotificationService (a Subject-backed toast stream). Callers still receive the rethrown error if they need it.

Example: register interceptors at bootstrap

// main.ts

OutputInterceptors run in array order on the way out (auth then error) and in reverse on the way back. Order matters: put auth before logging/error so logged requests already carry the header.

Note: interceptors run in an injection context, so inject() works directly inside them — that is how a functional interceptor reaches services like AuthService or Router without a constructor.

Note: use interceptors for retry too — next(req).pipe(retry({ count: 2, delay: 1000 })) transparently retries transient failures for every request, no per-call code.

Dependency injection

Dependency injection (DI) is a design pattern where a class declares what it needs and a framework supplies it, rather than the class constructing its own collaborators. In Angular this decouples consumers from concrete implementations: a component asking for a Logger does not care which logger it gets, which makes code easier to test (swap in a fake), reconfigure, and reuse. Angular's injector reads a token, looks up a matching provider, and returns an instance.

A provider maps a token to a recipe. There are four recipes: useClass instantiates a class; useValue supplies a ready-made object or primitive; useFactory runs a function (optionally with its own injected deps) to compute the value at resolution time; and useExisting aliases one token to another so both resolve to the same instance. The shorthand { provide: Logger } is just useClass: Logger.

Not every dependency is a class — config objects, base URLs, and feature flags have no constructor to use as a token. For these you create an InjectionToken<T>, a unique, typed key you can provide and inject. Angular's injectors form a hierarchy: a root injector plus a child injector for every component that declares its own providers. Resolution starts at the component and walks up the tree until a provider is found, so a component-level provider shadows the root one for that subtree. Services declared with providedIn: 'root' live at the top, are singletons, and remain tree-shakeable.

Example: the four provider recipes

providers: [
  { provide: Logger, useClass: ConsoleLogger },
  { provide: API_URL, useValue: 'https://api.dev' },
  { provide: Clock, useFactory: () => new Clock(Date.now()) },
  { provide: OldLogger, useExisting: Logger }, // alias
]
OutputLogger → new ConsoleLogger() API_URL → 'https://api.dev' Clock → factory-built instance OldLogger → same object as Logger

{
  provide: HttpCache,
  useFactory: (url: string) => new HttpCache(url),
  deps: [API_URL],
}
OutputAngular resolves API_URL first, then calls factory('https://api.dev') → HttpCache pointed at that base URL.

DI decorators & inject()

Angular's dependency injection resolves a token by walking up a hierarchy of injectors. By default, when a class asks for a dependency, Angular starts at the current element injector and climbs toward the root, throwing a runtime error if nothing provides the token. A family of resolution-modifier decorators lets you change that search. @Optional() makes a missing dependency non-fatal — Angular injects null instead of throwing. @Self() restricts the search to the current injector only. @SkipSelf() does the opposite, ignoring the current injector and starting at the parent. @Host() stops the search at the host component's boundary, which is useful for directives that should bind only to their host.

Two more decorators cover token-based needs. @Inject(TOKEN) tells Angular exactly which token to look up — essential when the dependency is an InjectionToken or an interface (interfaces vanish at runtime, so they cannot be used as tokens on their own). @Attribute('name') is a special one: it reads a static host attribute string once at construction time, never reacting to later changes, which makes it cheaper than an @Input() for values that never move.

Modern Angular prefers the inject() function over constructor-parameter decorators. inject() must be called in an injection context — inside a constructor, a field initializer, a factory, or a functional guard/resolver. It returns the same instance a constructor parameter would, but reads more cleanly and, crucially, works inside reusable helper functions that are not classes. The modifier decorators have inject() option equivalents: inject(TOKEN, { optional: true, self: true, skipSelf: true, host: true }).

Example: constructor decorators vs inject()

// Classic constructor injection with modifiers
@Component({ selector: 'app-old', standalone: true, template: '' })
export class OldCmp {
  constructor(
    @Optional() private logger: Logger | null,
    @SkipSelf() private parentSvc: PanelService,
    @Attribute('data-role') public role: string,
  ) {}
}

// Same thing with inject()
@Component({ selector: 'app-new', standalone: true, template: '' })
export class NewCmp {
  private logger = inject(Logger, { optional: true });
  private parentSvc = inject(PanelService, { skipSelf: true });
  // @Attribute has no inject() form; use HostAttributeToken (v18+)
  role = inject(new HostAttributeToken('data-role'), { optional: true });
}
OutputBoth components construct identically: logger is null when no Logger is provided, parentSvc resolves from an ancestor injector, and role holds the literal data-role attribute string.

Example: token-based injection with @Inject / inject()

export const API_URL = new InjectionToken<string>('API_URL');

// provided at bootstrap:
// { provide: API_URL, useValue: 'https://api.example.com' }

@Injectable({ providedIn: 'root' })

  constructor(@Inject(API_URL) legacy: string) {} // decorator form

OutputBoth this.url and the legacy parameter receive 'https://api.example.com'. An interface could never be the token here — only the runtime InjectionToken value can.

Example: inject() in a functional guard

export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);   // injection context: the guard call
  const router = inject(Router);
  return auth.isLoggedIn() ? true : router.parseUrl('/login');
};

Note: inject() throws NG0203 if called outside an injection context (for example inside a setTimeout or an event handler). Capture the dependency in a field or constructor first, then use it later.

Change detection

Change detection (CD) is how Angular keeps the DOM in sync with your component state. Traditionally, Angular ships with zone.js, a library that monkey-patches asynchronous browser APIs — setTimeout, addEventListener, Promise, XHR/fetch. When any patched task finishes, zone.js notifies Angular, which then runs a top-down check of the entire component tree, re-evaluating every template binding and updating the DOM where a value changed. This is reliable but can be wasteful in large trees.

ChangeDetectionStrategy.Default checks a component every CD cycle. ChangeDetectionStrategy.OnPush tells Angular to skip a component unless one of a few things happens: an @Input() receives a new reference, an event fires from within the component's template, the async pipe emits a new value, or (in modern Angular) a signal read in the template changes. Because OnPush relies on reference identity for inputs, immutability matters: mutating an object in place (obj.name = 'x') keeps the same reference and the view will not update; you must assign a new object instead.

When you need to override the defaults, inject ChangeDetectorRef. markForCheck() flags this component and its ancestors to be checked on the next cycle — the go-to for OnPush components whose state changed outside the four triggers. detectChanges() runs CD synchronously for this component now. detach() removes the component from the CD tree entirely (you then drive updates manually), and reattach() puts it back. The idiomatic modern pattern is Observables + the async pipe + OnPush: the pipe subscribes, calls markForCheck() on every emission, and unsubscribes automatically on destroy.

Example: OnPush and immutable updates

@Component({
  selector: 'app-user',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `{{ user.name }}`,
})
export class UserCmp {
  @Input() user!: { name: string };
}

// Parent:
mutate()  { this.user.name = 'Ada'; }        // same reference
replace() { this.user = { name: 'Ada' }; }   // new reference
OutputCalling mutate() leaves the displayed name unchanged (OnPush ignores same-reference inputs). Calling replace() updates the view to Ada.

Example: async pipe + OnPush

@Component({
  selector: 'app-clock',
  standalone: true,
  imports: [AsyncPipe],
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `{{ time$ | async }}`,
})
export class ClockCmp {
  time$ = interval(1000).pipe(map(() => new Date().toLocaleTimeString()));
}
OutputThe time updates every second even under OnPush — the async pipe calls markForCheck() on each emission, and it cleans up the subscription automatically on destroy.

Example: markForCheck after an out-of-band update

@Component({ changeDetection: ChangeDetectionStrategy.OnPush, /* ... */ })
export class LiveCmp {
  private cdr = inject(ChangeDetectorRef);
  count = 0;

  constructor() {
    // event outside Angular's four OnPush triggers
    window.addEventListener('storage', () => {
      this.count++;
      this.cdr.markForCheck(); // without this, the view is stale
    });
  }
}
OutputEach cross-tab storage event increments and re-renders count. Remove markForCheck() and the number changes internally but the DOM never refreshes.

Note: reach for OnPush plus signals or the async pipe by default in new code. It prunes whole subtrees from every CD pass and, combined with Angular's zoneless mode (v18+ experimental, stabilizing in later releases), can remove zone.js entirely.

Lifecycle hooks

A component moves through a well-defined sequence of phases, and Angular calls named methods on your class at each phase. The constructor runs first, but its only job should be dependency injection and cheap field setup — the component's inputs and view do not exist yet. Real initialization belongs in ngOnInit, which fires once after Angular has set the first input values. ngOnChanges(changes: SimpleChanges) runs before ngOnInit and again every time a bound @Input() changes, handing you a map of previous and current values.

ngDoCheck runs on every change-detection cycle for custom dirty-checking (use it sparingly — it is hot). The content hooks ngAfterContentInit and ngAfterContentChecked fire when projected <ng-content> has been initialized/checked. The view hooks ngAfterViewInit and ngAfterViewChecked fire once the component's own template and its children are ready — this is where @ViewChild references become available. Beware the ExpressionChangedAfterItHasBeenChecked error: mutating a bound value inside ngAfterViewInit changes the view after Angular already checked it, so defer such changes (a microtask, or move the logic earlier).

ngOnDestroy runs just before Angular tears the component down — the place to unsubscribe, clear timers, and detach listeners. The modern alternative is DestroyRef: inject it and register cleanup with destroyRef.onDestroy(fn), or pipe an Observable through takeUntilDestroyed() so it auto-completes when the injection context is destroyed, no manual ngOnDestroy needed.

Example: constructor vs ngOnInit

@Component({ selector: 'app-profile', standalone: true, template: '' })
export class ProfileCmp implements OnInit {
  @Input() userId!: string;
  private http = inject(HttpClient);

  constructor() {
    console.log('ctor, userId =', this.userId); // undefined!
  }
  ngOnInit() {
    console.log('init, userId =', this.userId); // set
    this.http.get(`/api/users/${this.userId}`).subscribe();
  }
}
Output
ctor, userId = undefined
init, userId = 42
Inputs are unavailable in the constructor; the HTTP call correctly waits for ngOnInit.

Example: takeUntilDestroyed vs ngOnDestroy

@Component({ selector: 'app-ticker', standalone: true, template: '' })
export class TickerCmp {
  private destroyRef = inject(DestroyRef);

  constructor() {

NgModules & feature modules

Before standalone components, every Angular building block lived inside an @NgModule. A module is a class decorated with metadata that groups related code: declarations lists the components, directives, and pipes the module owns; imports pulls in other modules whose exported declarations you want to use; exports re-publishes declarations so importing modules can use them; providers registers services for DI; and bootstrap (root module only) names the top-level component to render.

The root module is conventionally AppModule. In main.ts, platformBrowserDynamic().bootstrapModule(AppModule) compiles it and mounts the component named in bootstrap. As an app grows, you split it into feature modules — one per domain area (e.g. OrdersModule, AdminModule) — which keeps declarations scoped and enables lazy loading. Cross-cutting UI (shared components, common pipes, and the Angular modules many features need such as CommonModule and FormsModule) is collected in a SharedModule that each feature imports.

A declaration can belong to exactly one module, and a module must import whatever it uses — forget to import FormsModule and [(ngModel)] silently fails to compile. Standalone components (next section) now replace most of this ceremony, but you will still meet NgModules in older codebases and some third-party libraries expose an NgModule alongside (or instead of) standalone exports.

Example: a root AppModule

@NgModule({
  declarations: [AppComponent, HeaderComponent],
  imports: [BrowserModule, FormsModule],
  providers: [{ provide: API_URL, useValue: '/api' }],
  bootstrap: [AppComponent],
})
export class AppModule {}

// main.ts
platformBrowserDynamic()
  .bootstrapModule(AppModule)
  .catch(err => console.error(err));
OutputAngular compiles AppModule, instantiates AppComponent, and renders it into <app-root>. HeaderComponent is usable in templates because it is declared here.

Example: a SharedModule

@NgModule({
  declarations: [CardComponent, TruncatePipe],
  imports: [CommonModule],
  exports: [CardComponent, TruncatePipe, CommonModule],
})
export class SharedModule {}

// A feature module reuses everything with one import:
@NgModule({
  declarations: [OrderListComponent],
  imports: [SharedModule],
})
export class OrdersModule {}
OutputOrderListComponent's template can use <app-card>, the truncate pipe, and *ngIf/*ngFor — all re-exported by SharedModule from a single import.

Note: CommonModule (not BrowserModule) is what feature and shared modules import for *ngIf, *ngFor, and the async pipe. BrowserModule belongs only in the root module and must be imported exactly once.

Standalone components

A standalone component declares its own dependencies and no longer needs an @NgModule. Instead of being listed in some module's declarations, it sets standalone: true (the default from v17, so you can omit it) and lists the components, directives, pipes, and Angular modules it uses directly in its own imports: [] array. This makes each component self-contained: what it depends on is visible in one place, and there is no risk of a missing module import elsewhere.

Bootstrapping changes too. In main.ts you call bootstrapApplication(App, { providers: [...] }) instead of bootstrapModule. Application-wide providers are supplied through configuration functions: provideRouter(routes), provideHttpClient(), provideAnimations(), and so on. These replace the old pattern of importing RouterModule.forRoot() or HttpClientModule into a root module.

Standalone is the default when you scaffold new components from v17 onward, and by v19 it is fully the norm across the framework and CLI. For existing NgModule apps, Angular ships an automated migration: ng generate @angular/core:standalone runs in stages — converting declarations to standalone, removing now-empty NgModules, and switching main.ts to bootstrapApplication.

Example: a standalone component

@Component({
  selector: 'app-root',
  standalone: true, // default in v17+, optional
  imports: [RouterOutlet, CardComponent, AsyncPipe],
  template: `
    <app-card>{{ title }}</app-card>
    <router-outlet />
  `,
})
export class App {
  title = 'DevDocs';
}
OutputRenders the card containing DevDocs plus the routed view. No module declares App; its imports array makes <app-card> and <router-outlet> available.

Example: bootstrapApplication in main.ts

// main.ts
bootstrapApplication(App, {
  providers: [
    provideRouter(routes),
    provideHttpClient(),
  ],
}).catch(err => console.error(err));
OutputThe app boots with routing and HttpClient configured application-wide — no AppModule, no RouterModule.forRoot(), no HttpClientModule.

Example: before (NgModule) vs after (standalone)

// BEFORE
@NgModule({
  declarations: [MenuComponent],
  imports: [CommonModule],
  exports: [MenuComponent],
})

// AFTER

  selector: 'app-menu',
  standalone: true,
  imports: [NgFor, NgIf], // only what it needs
  template: `...`,
})

OutputThe wrapping module disappears; MenuComponent is imported directly wherever it is used. Running ng generate @angular/core:standalone performs this transformation across the project automatically.

Note: standalone components interoperate with NgModules in both directions — you can list a standalone component in an NgModule's imports, and import an NgModule into a standalone component's imports. This lets you migrate incrementally rather than all at once.

Reusable & shared UI

A well-structured app separates presentational ("dumb") components from container ("smart") components. A presentational component takes data through inputs and emits events through outputs; it holds no services and no business logic, which makes it trivially reusable and easy to test. A container component injects services, owns the data flow, and composes presentational children. In the Employee portal, EmployeeListComponent is a container that fetches data and hands each record to a reusable, presentational EmployeeCardComponent.

With standalone components, "sharing" UI no longer needs a SharedModule. You simply import the reusable standalone components (and directives/pipes) into whichever component's imports: [] array needs them. A common convention is a shared/ folder of standalone building blocks (buttons, cards, dialogs, a spinner) plus a barrel index.ts that re-exports them, so consumers import from one path. Legacy codebases still use an NgModule that declares and exports the shared pieces — you will meet both.

Two composition tools make components genuinely reusable: inputs/outputs for data and events, and content projection (ng-content) for letting the consumer supply arbitrary markup into a slot. Mark presentational components OnPush so they only re-render when their inputs change — see Change detection.

Example: a reusable presentational button

// shared/ui-button.component.ts
import { Component, ChangeDetectionStrategy } from '@angular/core';
import { input, output } from '@angular/core';

@Component({
  selector: 'app-ui-button',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <button [class]="'btn btn--' + variant()"
            [disabled]="disabled()"
            (click)="pressed.emit()">
      <ng-content />
    </button>`,
  styleUrl: './ui-button.component.scss',
})
export class UiButtonComponent {
  variant = input<'primary' | 'ghost'>('primary');
  disabled = input(false);
  pressed = output<void>();
}
Usage & Output<app-ui-button variant="ghost" (pressed)="save()"> Save </app-ui-button> Renders a themed <button class="btn btn--ghost">Save</button>. The label is projected via ng-content; the click surfaces as the pressed output. No services — reusable anywhere.

Example: container passes data to presentational children

// employee-list.component.ts (container)
@Component({
  selector: 'app-employee-list',
  standalone: true,
  imports: [EmployeeCardComponent, UiButtonComponent],

Routing & guards

Angular's router maps URL paths to components. In a modern standalone app you enable it during bootstrap with provideRouter(routes) instead of the legacy RouterModule.forRoot(). The routes value is a Routes array where each entry describes a path (without a leading slash) and what to render: a component, a redirectTo (paired with pathMatch: 'full' for exact matches), or child routes. A wildcard entry { path: '**', component: NotFound } placed last catches anything unmatched and typically renders a 404 page.

Wherever you place <router-outlet> in a template, the router renders the matched component. Navigation in templates uses the routerLink directive rather than a raw href, so navigation happens client-side without a full page reload; routerLinkActive adds a CSS class when its link matches the current URL. To read the current URL's parameters you can inject ActivatedRoute and subscribe to paramMap, or — far simpler — enable withComponentInputBinding() so route params, query params, and resolved data are bound straight into matching component @Input() (or input()) properties.

Lazy loading keeps the initial bundle small. loadComponent: () => import('./x').then(m => m.X) defers a single standalone component; loadChildren defers a whole set of routes. Route protection is now expressed as functional guards: a CanActivateFn is just a function that can call inject() to reach services, and returns a boolean, a UrlTree (to redirect), or a Promise/Observable of those.

Example: bootstrap + routes + outlet

// main.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { provideRouter, withComponentInputBinding } from '@angular/router';
import { Routes } from '@angular/router';
import { App } from './app';
import { Home } from './home';

export const routes: Routes = [
  { path: '', component: Home },
  { path: 'users/:id', loadComponent: () =>
      import('./user').then(m => m.User) },
  { path: 'old', redirectTo: '', pathMatch: 'full' },
  { path: '**', loadComponent: () =>
      import('./not-found').then(m => m.NotFound) },
];

bootstrapApplication(App, {
  providers: [provideRouter(routes, withComponentInputBinding())],
});
OutputVisiting / renders Home. /users/42 lazily fetches the User chunk. /old 301-style redirects to /. Any other URL renders the NotFound chunk.

Example: routerLink and active styling

<nav>
  <a routerLink="/" routerLinkActive="active"

  <a routerLink="/users/42" routerLinkActive="active">User 42</a>
</nav>
<router-outlet />
OutputOn /users/42 the second link gains the active class; the Home link stays inactive because of exact: true. Clicking navigates without a page reload.

Forms (template & reactive)

Angular ships two form systems. Template-driven forms keep the source of truth in the template: you import FormsModule, bind an input with [(ngModel)], and Angular builds the form model implicitly. A template reference variable like #form="ngForm" exposes the aggregate form state, while each control exposes validity flags such as valid, invalid, touched, dirty, and pristine. This style is quick for small forms but harder to unit test.

Reactive forms make the model explicit in TypeScript. You import ReactiveFormsModule and build a tree of FormControl and FormGroup instances — usually via the injected FormBuilder for brevity. Validators like Validators.required and Validators.email attach synchronous validation. Each control exposes a valueChanges Observable so you can react to edits, and the whole group is testable without touching the DOM.

Since v14 reactive forms are typed: FormControl<string> and the shape inferred by FormBuilder give you compile-time safety on value and setValue. Note that a control created with an initial value is nullable by default (it becomes null when reset); use nonNullable: true to keep the type non-null. Display errors by reading a control's errors map only after it is touched, so messages don't flash before the user interacts.

Example: template-driven form

import { Component } from '@angular/core';
import { FormsModule } from '@angular/forms';

@Component({
  selector: 'app-tpl',
  imports: [FormsModule],
  template: `
    <form #f="ngForm" (ngSubmit)="save(f.value)">
      <input name="email" [(ngModel)]="email"
             required email #e="ngModel">
      <p *ngIf="e.invalid && e.touched">Bad email</p>
      <button [disabled]="f.invalid">Save</button>
    </form>`,
})
export class Tpl {
  email = '';
  save(v: unknown) { console.log(v); }
}
OutputTyping "abc" then blurring shows "Bad email" and keeps Save disabled. Entering "a@b.co" hides the message and enables Save; submitting logs { email: 'a@b.co' }.

RxJS in Angular

An Observable is a lazy, push-based stream that can emit zero, one, or many values over time and then complete or error. "Lazy" means nothing runs until you subscribe; "push" means the producer decides when values arrive. Angular's HttpClient, router events, and reactive-form valueChanges are all Observables. Because a manual subscribe keeps running until it completes, long-lived streams must be unsubscribed to avoid memory leaks.

Operators transform streams functionally inside .pipe(). map reshapes each value, filter drops values, debounceTime waits for a quiet gap (ideal for search boxes), and switchMap flattens an inner Observable while cancelling the previous inner subscription — perfect for type-ahead HTTP where stale requests should be abandoned. combineLatest merges the newest value from several sources. For shared, imperative state, a Subject is a multicast Observable you can push into with next(), and BehaviorSubject additionally holds a current value that new subscribers receive immediately.

In templates the async pipe subscribes for you and unsubscribes automatically when the view is destroyed, eliminating most manual cleanup. In component code, takeUntilDestroyed() ties a subscription's lifetime to the component. Finally, the interop helpers bridge both worlds: toSignal(obs$) converts an Observable into a signal, and toObservable(sig) converts a signal back into a stream.

Example: async pipe (no manual unsubscribe)

import { Component, inject } from '@angular/core';

@Component({
  selector: 'app-clock',
  imports: [AsyncPipe],
  template: `<p>{{ user$ | async | json }}</p>`,

export class Clock {
  private http = inject(HttpClient);
  user$ = this.http.get('/api/me');
}
OutputWhen the request resolves, the paragraph renders { "name": "Ada" }. The pipe unsubscribes automatically when the component is destroyed.

import { interval } from 'rxjs';

// stream -> signal usable directly in templates
seconds = toSignal(interval(1000), { initialValue: 0 });
// template: {{ seconds() }}
OutputThe template shows 0, then 1, 2, … each second. toSignal manages the subscription and cleanup for you.

Note: Prefer the async pipe or toSignal() over manual subscribe(). If you must subscribe in code, pair it with takeUntilDestroyed() so the stream is torn down with the component.

Deferrable views @defer

Introduced in v17, the @defer block is a template-level way to lazy-load a section of UI along with its component dependencies. Everything inside @defer { … } becomes a separate JavaScript chunk that Angular fetches only when a trigger fires. This is more granular than route-level lazy loading: you can defer a heavy chart or comment thread that sits below the fold without splitting your routes.

A defer block has companion blocks for the states around loading. @placeholder shows content before the trigger fires (and can declare a minimum display time). @loading shows while the chunk downloads and supports minimum and after to avoid flicker on fast connections. @error renders if the import fails. Only @defer is required; the rest are optional.

Triggers decide when to load: on idle (the default, when the browser is idle), immediate, on timer(2s), on viewport (when it scrolls into view), on interaction, and on hover. The viewport/interaction/hover triggers can reference an element via a template variable, e.g. on viewport(ref). You can also prefetch independently — @defer (on viewport; prefetch on idle) downloads early but renders late. Finally, when takes a boolean expression, and multiple triggers can be combined.

Example: viewport trigger with placeholder

@defer (on viewport) {
  <app-heavy-chart />
} @placeholder (minimum 500ms) {
  <p>Scroll down to load the chart…</p>
} @loading (after 100ms; minimum 1s) {
  <p>Loading chart…</p>
} @error {
  <p>Could not load chart.</p>
}
OutputInitially shows "Scroll down to load the chart…". When it scrolls into view, "Loading chart…" appears (only if load takes >100ms), then the real <app-heavy-chart> renders. A failed import shows the error text.

Example: interaction trigger referencing #ref

<button #loadBtn>Show comments</button>

@defer (on interaction(loadBtn); prefetch on idle) {
  <app-comments />
} @placeholder {
  <p>Comments hidden</p>
}
OutputThe chunk prefetches during idle time. Clicking the button swaps "Comments hidden" for the live <app-comments> instantly, since the code was already fetched.

Example: when condition + timer

@defer (when isReady(); on timer(2s)) {
  <app-dashboard />
} @loading {
  <p>Preparing dashboard…</p>
}
OutputThe dashboard loads as soon as the isReady() signal returns true, or after 2 seconds at the latest — whichever fires first. "Preparing dashboard…" shows meanwhile.

Note: Reach for @defer on below-the-fold or rarely-used UI. Combine a late render trigger (on viewport) with an early prefetch on idle so the content appears instantly the moment it's needed.

Signals

A signal is a reactive wrapper around a value. signal(initial) returns a getter function you call to read the current value — count(), not count. You change it with .set(v) to replace the value outright, or .update(fn) to derive the next value from the current one, e.g. count.update(n => n + 1). Expose a read-only view to consumers with .asReadonly() so they can read but not write.

Signal equality is by reference for objects. When a signal holds an object or array, you must set a new reference to trigger updates — mutating the existing object in place will not notify anything. So use items.update(list => [...list, next]) rather than list.push(next). computed(() => …) creates a derived signal that is both lazy (it only recomputes when read) and memoized (it caches until a dependency changes). Angular discovers dependencies automatically: whichever signals you read during the computation are tracked, so you never declare a dependency list.

effect(() => …) runs a side effect whenever any signal it reads changes — logging, syncing to localStorage, imperative DOM work. Effects must be created in an injection context (a constructor or with an explicit Injector) and are cleaned up when that context is destroyed; register per-run teardown via the onCleanup callback. Signals integrate tightly with change detection: reading a signal in a template registers that view as a consumer, so a signal change marks exactly the affected views dirty — the basis of Angular's zoneless future.

Example: signal, set, update

import { signal } from '@angular/core';

const count = signal(0);
count();            // read -> 0
count.set(5);       // replace
count();            // -> 5
count.update(n => n + 1);
count();            // -> 6

const ro = count.asReadonly();
// ro.set(9)  // compile error: no set on readonly
OutputReads yield 0, then 5, then 6. The read-only view exposes only the getter — attempting ro.set fails at compile time.

Example: computed derived value

import { signal, computed } from '@angular/core';

const price = signal(100);
const qty = signal(2);
const total = computed(() => price() * qty());

total();          // -> 200 (computed & cached)
qty.set(3);
total();          // -> 300 (recomputed on read)
Outputtotal() is 200, then 300 after qty changes. It only recomputes when a tracked dependency (price or qty) changes and is then read.

Example: immutable array update

const items = signal<string[]>(['a']);

// WRONG: mutation, no notification
// items().push('b');

// RIGHT: new reference
items.update(list => [...list, 'b']);
items();   // -> ['a', 'b']
OutputThe mutating push would leave dependents stale. The update with a fresh array yields ['a', 'b'] and notifies all consumers.

Example: effect with cleanup

import { Component, signal, effect } from '@angular/core';

@Component({
  selector: 'app-counter',
  template: `<button (click)="inc()">{{ count() }}</button>`,
})
export class Counter {
  count = signal(0);
  inc() { this.count.update(n => n + 1); }

  constructor() {
    effect((onCleanup) => {
      const id = setTimeout(() => console.log(this.count()));
      onCleanup(() => clearTimeout(id));
    });
  }
}
OutputEach click re-renders the button label and re-runs the effect; the previous timeout is cleared via onCleanup before the next run. The template updates without zone.js ceremony.

Note: effect() must be created in an injection context (typically a constructor or field initializer). For pure derivations prefer computed() over an effect that writes another signal — it's lazier, memoized, and avoids feedback loops.

Signal inputs, outputs & model

Modern Angular replaces the decorator-based @Input() / @Output() API with a set of functions that produce signals. The input() function declares an optional input and returns a read-only InputSignal<T>; you read the current value by calling it like any signal (this.name()). Because it is a signal, it participates in the reactive graph, so you no longer implement ngOnChanges to react to changes — you derive with computed() or run side effects with effect(). For a mandatory input use input.required<T>(), which has no default and makes Angular enforce that a value is bound.

Both forms accept an options object with alias (expose a different binding name to templates) and transform (coerce the incoming value, e.g. booleanAttribute or numberAttribute). The transform runs on every write, so the signal always holds the normalized value. This is fully typed: the accepted binding type and the stored type can differ (a transform from string | number to number, for example).

Outputs are declared with output(), which returns an OutputEmitterRef<T> — call emit(value) to fire. This replaces @Output() plus new EventEmitter() and is not an RxJS Subject, though outputFromObservable() can bridge one. For two-way binding, model() (and model.required()) creates a writable signal that Angular pairs with an implicit xChange output, so a parent can use the banana-in-a-box syntax [(value)]="...". Finally, the signal-based queries viewChild() and contentChild() (plus their *Children plurals) return signals instead of requiring @ViewChild and AfterViewInit timing.

Example: input() vs @Input()

// Old: decorator + lifecycle hook
@Input() count = 0;
ngOnChanges() { this.doubled = this.count * 2; }

// New: signal input + computed
count = input(0);                 // InputSignal<number>
label = input.required<string>(); // must be bound
doubled = computed(() => this.count() * 2);
OutputNo ngOnChanges needed. When the parent binds [count]="5", count() returns 5 and doubled() recomputes to 10 automatically.

Example: alias & transform

import { input, booleanAttribute } from '@angular/core';

// Template binds [isDisabled], component reads disabled()
disabled = input(false, {
  alias: 'isDisabled',
  transform: booleanAttribute,
});
// <my-btn isDisabled>  → disabled() === true
OutputBinding the bare attribute isDisabled passes the string ""; booleanAttribute coerces it, so disabled() returns true.

Example: output() & model()

// Child
saved = output<string>();          // replaces @Output EventEmitter
value = model('');                 // two-way signal
save() { this.saved.emit(this.value()); }

// Parent template
<editor [(value)]="text" (saved)="onSaved($event)" />
Output[(value)] keeps text and the child's value() in sync both ways; calling save() emits the string to onSaved.

Example: signal queries

import { viewChild, contentChild, ElementRef } from '@angular/core';

box = viewChild<ElementRef>('box');         // Signal<ElementRef | undefined>
header = contentChild.required(HeaderCmp);  // guaranteed present

focus() { this.box()?.nativeElement.focus(); }
Outputbox() is undefined until the view renders, then resolves to the element ref — no AfterViewInit juggling required.

Note: signal inputs are read-only from inside the component — you cannot assign to them. Use model() when the component itself must change the value, and reserve effect() for genuine side effects (never to sync one signal into another; prefer computed() or linkedSignal()).

Resource API & httpResource

The Resource API models asynchronous data as signals. resource({ request, loader }) takes a request function that reads signals and returns a plain request object, and a loader that receives that request and returns a Promise. Whenever any signal read inside request changes, Angular re-invokes the loader (aborting the previous call via an AbortSignal). The returned resource exposes reactive members: value() (the loaded data), status() (an enum like idle, loading, resolved, error), error(), isLoading(), and an imperative reload(). It shipped in v19 and is still marked experimental.

When your loader is more naturally an Observable — for example an existing HttpClient stream or something with cancellation semantics — use rxResource(). It has the same shape but the loader returns an Observable, and Angular takes the first emission and manages the subscription for you. This lets you keep RxJS operators while still exposing the data as signals to the template.

Going one level higher, httpResource() (stable in v20) is a declarative HTTP-as-a-signal helper built on HttpClient. You give it a function returning a URL (or a full request descriptor) that reads signals; it performs the GET and re-fetches automatically when those signals change. The result is a resource whose value() is the typed response body, plus headers(), statusCode(), and the usual isLoading()/error(). It requires provideHttpClient() to be configured.

Example: resource() with a request signal

userId = signal(1);

user = resource({
  request: () => ({ id: this.userId() }),
  loader: async ({ request, abortSignal }) => {
    const r = await fetch(`/api/users/${request.id}`, { signal: abortSignal });
    return r.json();
  },
});
// user.isLoading(), user.value(), user.error(), user.reload()
OutputInitially isLoading() is true, value() is undefined. After the fetch resolves, value() holds the user and status() becomes resolved. Setting userId.set(2) aborts the old request and reloads.

Example: rxResource() with an Observable loader

private http = inject(HttpClient);
query = signal('angular');

results = rxResource({
  request: () => this.query(),
  loader: ({ request }) =>
    this.http.get<Repo[]>(`/search?q=${request}`),
});
OutputTyping into a box that calls query.set(...) re-runs the HTTP GET; results.value() updates to the newest response and prior in-flight requests are unsubscribed.

Example: httpResource() in a template

id = signal(42);
product = httpResource<Product>(() => `/api/products/${this.id()}`);
OutputTemplate can branch on state:
@if (product.isLoading()) { Loading… }
@else if (product.error()) { Failed }
@else { {{ product.value()?.name }} }

Note: keep the request function pure and cheap — it re-runs on every dependency change and its return value is compared to decide whether to reload. Do not call the loader manually; drive it by updating the signals it depends on, and use reload() only for explicit refreshes.

Zoneless & modern reactivity

Historically Angular relied on zone.js to monkey-patch async APIs (setTimeout, DOM events, XHR) and trigger change detection globally after anything ran. With signals, Angular knows precisely which state changed, so the zone becomes unnecessary. Calling provideZonelessChangeDetection() in your application providers drops zone.js entirely; change detection is then scheduled by signal writes, template event bindings, and explicit markForCheck calls. This API is the stabilized successor to the earlier provideExperimentalZonelessChangeDetection() and became available around v20. Remember to remove zone.js from the polyfills array in your build config.

The payoff is concrete: a smaller bundle (no zone.js), lower runtime overhead (no global patching or full-tree dirty checking), and precise, targeted updates that only refresh the components whose signals actually changed. It also removes a class of confusing timing bugs and makes interop with third-party libraries that fought the zone much simpler.

Two v19 helpers round out the reactivity story. linkedSignal() creates a writable signal that is derived from a source signal: it recomputes when the source changes (like computed()) but can also be set imperatively (like a plain signal()), which is ideal for state that should reset when inputs change yet still be user-editable. For DOM work, afterNextRender() runs a callback once after the next render (a safe place to read layout), and afterRenderEffect() runs reactively after every render that touched its dependencies — both only execute in the browser, never during SSR.

Example: going zoneless

bootstrapApplication(App, {
  providers: [
    provideZonelessChangeDetection(),
    provideHttpClient(),
  ],
});
// also delete 'zone.js' from angular.json polyfills
OutputThe app boots without zone.js. A signal write like count.update(c => c + 1) schedules change detection for just the affected view.

Example: linkedSignal()

options = input<string[]>([]);

// resets to first option when options change,
// but the user can still override it
selected = linkedSignal(() => this.options()[0]);

choose(o: string) { this.selected.set(o); }
Outputselected() tracks the first option; picking another sets it manually. When options changes, selected() resets to the new first item.

Example: afterNextRender()

chart = viewChild<ElementRef>('chart');

constructor() {
  afterNextRender(() => {
    const w = this.chart()!.nativeElement.offsetWidth;
    this.width.set(w);           // safe DOM read
  });
}
OutputRuns once in the browser after the view is painted; offsetWidth is measured reliably and never executes on the server.

Note: in zoneless apps, mutate state through signals or immutable updates. A raw array push (this.items.push(x)) with an OnPush/zoneless tree will not be noticed — call this.items.update(...) so the signal actually changes.

SSR, hydration, i18n & Elements

Angular's server-side rendering lives in the @angular/ssr package. Running ng add @angular/ssr wires up a Node/Express server entry, a server build target, and the app-shell plumbing so pages are rendered to HTML on the server and streamed to the browser for fast first paint and SEO. On the client you enable hydration with provideClientHydration(), which reuses the server-rendered DOM instead of destroying and re-creating it. Adding withIncrementalHydration() (v19) goes further: components stay dormant and hydrate lazily on interaction or viewport visibility (driven by @defer triggers), shipping and executing less JavaScript up front.

Internationalization uses the compile-time i18n system. Mark translatable text with the i18n attribute; give a stable custom id with i18n="@@myId" so translations survive text edits. Complex grammar is expressed with ICU expressions — {count, plural, ...} for counting and {gender, select, ...} for alternatives. You extract all marked messages into an XLIFF/XMB file with ng extract-i18n, translate per locale, then produce a separate optimized build per locale (each with its own baked-in translations and LOCALE_ID).

Finally, Angular Elements lets you ship a component as a standard Web Component. createCustomElement(MyComponent, { injector }) returns a class you register with customElements.define('my-widget', ...). The result is a framework-agnostic custom element with inputs surfaced as attributes/properties and outputs surfaced as DOM CustomEvents, usable in plain HTML, React, Vue, or anywhere the browser runs.

Example: SSR + hydration

ng add @angular/ssr
OutputScaffolds server.ts, a server build target, and adds provideClientHydration() to the app config. Pages now render to HTML on the server.

Example: incremental hydration

bootstrapApplication(App, {
  providers: [
    provideClientHydration(withIncrementalHydration()),
  ],
});
OutputServer DOM is reused (no flicker). A block marked @defer (hydrate on interaction) stays inert until the user interacts, deferring its JS.

Example: i18n with ICU plural

<h1 i18n="@@homeTitle">Welcome</h1>

<p i18n>{count, plural,
  =0 {No items}
  =1 {One item}
  other {{{count}} items}
}</p>
Outputng extract-i18n collects homeTitle and the plural message. With count = 3, the rendered text reads: 3 items.

Example: Angular Elements

const el = createCustomElement(RatingComponent, {
  injector: this.injector,
});
customElements.define('star-rating', el);

// Now usable anywhere:
// <star-rating value="4"></star-rating>
OutputThe browser registers <star-rating>. Setting the value attribute feeds the input; an output() named rated dispatches a rated CustomEvent.

Note: hydration requires the server and client to render the same DOM. Manual DOM manipulation outside Angular, or invalid HTML nesting, causes hydration mismatch errors — keep DOM changes inside templates and directives so the reused markup matches.

Testing & Angular 18–22 highlights

Angular's unit-testing foundation is TestBed. You configure a testing environment with TestBed.configureTestingModule({ imports: [MyComponent], providers: [...] }) — with standalone components you import the component directly rather than declaring it in an NgModule. Create an instance with TestBed.createComponent(MyComponent), which returns a ComponentFixture. Call fixture.detectChanges() to run change detection and render the template, then assert against the DOM through fixture.debugElement (Angular's wrapper, queryable with By.css(...)) or the raw fixture.nativeElement.

For richer, reuse-friendly interactions, component test harnesses from @angular/cdk/testing provide a stable API over a component's DOM: you obtain a HarnessLoader and load typed harnesses (e.g. a Material button harness) so tests describe behavior, not brittle selectors. Meanwhile the ecosystem is shifting away from the long-default Karma + Jasmine runner toward faster, modern alternatives — Angular now offers first-class support for Vitest and web-test-runner, which run in-process or in real browsers with quicker feedback and native ESM.

Example: TestBed component test

TestBed.configureTestingModule({ imports: [CounterComponent] });

const fixture = TestBed.createComponent(CounterComponent);
fixture.detectChanges();

const btn = fixture.debugElement.query(By.css('button'));
btn.nativeElement.click();
fixture.detectChanges();

expect(fixture.nativeElement.textContent).toContain('Count: 1');
OutputThe button click increments the signal; after detectChanges() the DOM shows Count: 1 and the assertion passes.

Example: a component harness

const loader = TestbedHarnessEnvironment.loader(fixture);
const button = await loader.getHarness(MatButtonHarness);

await button.click();
expect(await button.getText()).toBe('Saved');
OutputThe harness clicks the Material button and reads its text without any CSS selectors, so the test survives internal markup changes.

Key takeaways: modern Angular is standalone-first and signal-centric. Declare state with signal(), derive with computed()/linkedSignal(), pass data with signal input()/output()/model(), fetch with the Resource API, and let signals — not zone.js — drive precise change detection.
