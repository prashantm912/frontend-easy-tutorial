Orientation

How to use this guide

01

Follow the phases in order

Each phase assumes the one before it. Breadth first (know every topic exists), depth on the load-bearing few. Skip nothing in Phases 1–3.

02

Prepare every topic in 3 layers

What it is (one sentence) → why & when (the trade-off — this is what separates candidates) → a failure story (what breaks without it in production).

03

Build after every phase

Reading stores a concept temporarily; building makes it permanent. Every author repeats it: tutorial-watchers stay stuck, builders get hired.

04

Rehearse out loud

Interviewers rarely ask "define X." They ask "your API is timing out — what do you do?" You need the concepts wired together, spoken, not memorized as a flat list.

The one meta-skillTrade-offs are the whole game. Almost every real interview question is secretly "tell me the trade-off, and when you'd choose the other way." A junior asks "what's the best database?"; a senior asks "what does this workload need, and what am I willing to give up to get it?"

Prep strategy — how to run the whole thing

2–3 days · mindset & setup

Before any code, set the operating system for your prep. The single most common failure isn't lack of intelligence — it's scattered effort, tutorial hell, and quitting a path out of FOMO.

  • Pick one path (backend / full-stack / AI) and a small resource set. Excessive resources cause tutorial hell.
  • Practice > consume. Watching stores a concept temporarily; building makes it permanent.
  • Use Git from day 1. Every practice project on GitHub — your progress log and proof of work.
  • Use AI within limits. Generate code, then debug, test and rewrite it yourself. Interviews expose people who can't work without autocomplete.
  • Learn the whole craft, not just coding: unit testing, debugging, documenting, logging, monitoring, deploying.
  • Ship a portfolio + tight resume and build a network — skills without connections reach no one.
MindsetCollege tier does not decide your ceiling. Time spent on regret is time not spent learning. Every author in this guide independently made the same point.

2

Backend engineering fundamentals

4–6 weeks · the systematic core

The 8-concept spine: REST API design → Auth (JWT/OAuth) → DB indexing → Caching (Redis) → Rate limiting → Logging & monitoring → Docker → System-design fundamentals. Once you grasp these, backend stops being mysterious and becomes systematic.

2a · HTTP & API design — the 20 concepts

1 Endpoint — a unique URL naming a resource. Nouns not verbs: /users/42/orders.

2 HTTP methods — GET/POST/PUT/PATCH/DELETE. PUT = full replace, PATCH = partial.

3 Request–response — headers, body, status, the round-trip shape.

4 Status codes — know each: 200/201/204/301/304/400/401/403/404/409/429/500/503. 401 = who are you? 403 = I know you, you still can't.

5 Authentication — verify identity.

6 Authorization — verify permissions (RBAC vs ABAC).

7 Access tokens — short-lived credentials; pair with refresh-token rotation.

8 OAuth 2.0 — delegated access without sharing passwords; know the auth-code flow.

9 Rate limiting — cap requests/client/window. Token bucket, leaky bucket, sliding window.

10 Throttling — slow requests down instead of rejecting.

11 Pagination — offset vs cursor. Cursor survives inserts/deletes; offset drifts.

12 Caching — response caching, ETags, Cache-Control.

13 Idempotency — same request repeated → same result, no double side-effects.

14 Webhooks — server pushes event notifications; the inverse of polling.

15 API versioning — URL / header / content-negotiation; never break existing clients.

16 OpenAPI / Swagger — machine-readable API contracts.

17 REST vs GraphQL — over/under-fetching trade-off; GraphQL moves complexity to resolvers & caching.

18 API gateway — one entry point: routing, auth, rate limiting, TLS termination.

19 Microservices — small independent services talking over APIs.

20 Error handling — consistent error envelope; never leak stack traces.

The #1 nugget interviewers probeIdempotency keys. GET/PUT/DELETE are idempotent by spec; POST is not. To make "create charge" retry-safe, the client sends a unique Idempotency-Key header; the server stores the result against that key and replays it instead of executing twice. Without it, one timeout + one client retry = a double charge. This concept appears in four of the source threads — know it cold.

2b · Security & identity

  • JWT vs sessions vs cookies — JWT = stateless, signed, hard to revoke; sessions = server state, easy to revoke. Know where each wins.
  • Password storage — bcrypt / Argon2 (slow, salted, adaptive), never MD5/SHA alone. Salt = per-user random; pepper = server-wide secret.
  • 2FA, SSO, OAuth 2.0 flows.
  • Web attacks — CORS (a browser policy, not a security control), CSRF (SameSite cookies + tokens), XSS (escape output), SQL injection (parameterized queries), SSRF (server tricked into internal calls). For each: what it is, an attack example, the standard defense.

2c · Databases

  • Design & normalization; when to denormalize (read-heavy paths).
  • SQL vs NoSQL — relational integrity + joins vs flexible schema + horizontal scale. Trade-off, not tribalism.
  • Indexing — B-Tree structure, why writes slow as indexes grow, composite-index column order, covering indexes. The concept that most changes how engineers think.
  • Query optimization — EXPLAIN plans, the N+1 query (1 query becomes 101), connection pooling.
  • ACID + isolation levels & their anomalies (dirty / non-repeatable / phantom reads).
  • CAP — under a partition, pick consistency or availability. Know what Postgres, Cassandra, DynamoDB choose.
  • Optimistic vs pessimistic locking — version-column retry vs lock-up-front, and when each fits.

2d · Caching & performance

  • Strategies: cache-aside (lazy), read-through, write-through, write-behind.
  • Eviction: LRU / LFU / TTL. Invalidation — one of the two hard problems; have a concrete strategy (TTL + event-driven purge).
  • Cache stampede — hot key expires, thousands hit the DB at once. Fix with single-flight lock, staggered TTLs, early refresh.
  • Redis vs Memcached; Redis pipelining for batching; CDN & edge caching.

2e · Concurrency & background work

  • Middleware, centralized error handling, structured logging, APM.
  • Async/await, event loop, thread pools, process vs thread, GC basics, memory leaks.
  • Race conditions & deadlocks — define, detect, prevent (lock ordering, timeouts).
  • Background jobs, cron, workers — move slow work off the request path via queues.
Build target for this phaseA REST API with JWT auth, rate limiting, cursor pagination, an idempotency-key on writes, a Postgres schema with a deliberately-indexed hot query (measure it before/after with EXPLAIN), and a Redis cache in front. That single project answers half the Phase-2 questions on its own.

System design & scalability

4–6 weeks · the differentiator

This is where senior signal lives. The anchor is a 25-day scaling series written by a Principal Engineer at Atlassian — the single richest resource in this guide. Its Day 1 (load balancing) is below; all 25 days are in the deep-dive section.

3a · Load balancing — interview-ready depth

Core insightA load balancer distributes requests — that is not the same as balancing load. It doesn't know one request takes 2ms and the next takes 12 seconds. Real case: 4 servers, Server 3 at 95% CPU while Server 1 sits at 12% — and the balancer reports "fine."

Algorithms & where each fails

AlgorithmHowFailure mode
Round-robinRotate in orderBlind to server state; only works if all requests & servers are identical
Least connectionsFewest active winsConnection count ≠ load (2 report exports vs 15 sub-ms health checks)
Weighted RRManual weights for big boxesWeights are static; traffic isn't; nobody re-tunes them
IP hashSticky sessions, no storeOne corporate NAT (10k users, one IP) = instant hotspot
Least response timeRoutes on real latencyClosest to true awareness — what production should graduate to

Layer 4 vs Layer 7

L4 (transport) — sees IPs/ports only; fast, cheap, content-blind. Use for raw throughput & TCP passthrough. L7 (application) — reads URL/headers/cookies/body; enables path routing, cookie-based tiering, canary (5% to a new version), TLS termination (NGINX, HAProxy, ALB, Envoy). Production pattern: L4 in front for throughput, L7 behind for intelligent routing.

Health checks

Active polls "are you alive?" (a 10s interval = up to 10s of traffic to a dead box). Passive watches real responses (catches "alive but broken," but needs traffic). Production uses both, failure threshold ~3 (not 1), plus the part everyone forgets — slow-start recovery: ramp a recovered server up gradually or its backlog crushes it and it dies again.

3c · Deploy & operate (DevOps literacy)

  • Docker → Kubernetes (pods, services, liveness/readiness probes) → CI/CD.
  • Deployment strategies: blue-green, canary, rolling; rollbacks; feature flags.
  • Observability trio: logs, metrics, traces; alerting; SLO / SLI / error budgets.

The framing that landsWriting code is rarely the bottleneck — scaling it, observing it, and keeping it alive when real users depend on it is where engineering begins. That ~100-item production checklist is exactly what separates "works on my machine" from "survives its first million users."

Language depth

2–3 weeks · systems, not syntax

The bar moved. "HashMap vs Hashtable" and "what is the JVM" are over. Modern interviews (shown here for Java — substitute your stack) test whether you understand systems, not syntax.

Real microservices design

Timeouts, retries, circuit breakers, idempotency — implemented, not defined.

Distributed thinking

Consistency models, partial failures, message reprocessing (consumer crashes mid-message — then what?).

Framework internals

Spring Boot auto-config, AOP proxies (why @Transactional on a self-invoked method silently does nothing), startup bottlenecks.

DB & caching strategy

Query tuning, index design, connection-pool exhaustion, cache stampedes, eviction.

Messaging

Kafka partitions & ordering, consumer lag, backpressure, exactly-once semantics (and why it's hard).

Concurrency & runtime depth

Virtual vs platform threads, atomics/VarHandles, false sharing, lock elimination, escape analysis, how GC pauses hit tail latency.

For your languageNode → event-loop internals, libuv, backpressure in streams. Python → the GIL, asyncio, C-extension boundaries. Go → the scheduler, goroutine leaks, channel semantics. Same principle: know the runtime, not just the syntax.

Deep dive · added

SQL essentials

SQL rounds are common and separate the "I use an ORM" crowd from engineers who understand the database. Master these.

Joins

INNER (match both) · LEFT (all left + matches) · RIGHT · FULL OUTER · CROSS · SELF-join. Know exactly which rows each returns and when NULLs appear.

Aggregation

GROUP BY + HAVING (filter groups, vs WHERE filtering rows). COUNT/SUM/AVG/MIN/MAX. The classic bug: filtering aggregates in WHERE.

Window functions

ROW_NUMBER(), RANK(), DENSE_RANK(), LAG/LEAD, running totals with SUM() OVER (...). The tool for "top N per group" and "compare to previous row."

Subqueries & CTEs

Correlated vs non-correlated subqueries; WITH CTEs for readability; recursive CTEs for hierarchies/graphs.

Indexing in practice

B-Tree indexes, composite-index column order (equality first, range last), covering indexes, why LIKE '%x' and functions on columns kill index use.

Transactions & isolation

ACID, isolation levels (Read Committed → Serializable) and the anomalies each prevents; deadlocks and lock ordering.

Classic interview asks"Second-highest salary" (window function or correlated subquery) · "Nth highest per department" (ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC)) · "find duplicates" (GROUP BY + HAVING COUNT > 1) · "running total" (SUM window). Practice each until it's reflexive.
Deep dive · added

How to answer any system-design question

A repeatable 6-step frame. Interviewers grade your questions and trade-offs as much as the boxes you draw. Never jump to "let's use Kafka and Cassandra" before the numbers justify it.

  1. Clarify requirements & scope — functional (what it must do) and non-functional (scale, latency, consistency, availability). Nail the read/write ratio.
  2. Estimate (back-of-envelope) — DAU × actions/day ÷ 86,400 → QPS; peak ≈ 2–3× average; storage = rows × size × retention. Do the math out loud — designing for the wrong scale is the #1 failure at Google/Amazon/Meta.
  3. Define the API — a few endpoints/signatures. Grounds the abstract in something concrete.
  4. High-level design — client → LB → services → cache → DB → queue. Draw the boxes and data flow.
  5. Deep-dive 1–2 components — the interesting ones: the datastore choice, the hot path, the consistency boundary. Show the trade-off you're making.
  6. Address bottlenecks & failures — apply the patterns: caching, sharding, replication, queues, circuit breakers, idempotency. What happens when a node dies?
Consistency is per-feature, not per-systemThe answer interviewers love: "It depends on the feature." An e-commerce app is AP for its product catalogue and reviews, but CP for last-item inventory and payments — same system, different guarantees. Saying that shows you understand CAP instead of reciting it.
Deep dive · added

Mini case studies

Four canonical designs, each with the numbers and the decisions that fall out of them. Run each through the 6-step template above.

URL shortener (Bitly)

100M URLs/month → ~40 writes/s (one DB, no write sharding). Read-heavy ~100:1 → ~4,000 reads/s → Redis in front + read replicas. ~3 TB over 5 years fits one disk + replication. No queue, no sharding. Base62 short codes or a hash+counter. The estimation is what saves you from over-engineering.

Rate limiter

Token bucket in Redis (atomic INCR + TTL). Distributed → counters must be shared, not per-instance. Return 429 with Retry-After; clients back off with jitter. Layer it: gateway global + per-user + per-endpoint. Whitelist health checks.

Chat / messaging

WebSockets for realtime delivery; a message queue for fan-out & offline delivery; shard by conversation_id so a chat's messages co-locate. At-least-once delivery + idempotent client dedup. Read receipts & presence via pub/sub.

News feed (Twitter timeline)

Fan-out on write (push to followers' feeds) for most users vs fan-out on read (pull) for celebrities with millions of followers — hybrid. Cache timelines in Redis; precompute; paginate with cursors. The celebrity problem is the whole interview.

The deep-dive · all 25 days

The 25-day scaling series

By a Principal Engineer at Atlassian. Every concept answers one of three questions — handle more load, store more data than one machine holds, stay correct when things fail — and every one is a trade-off between speed, cost, and correctness. Expand any day.

Day 1Load Balancingthe balancer is lying to you›
Covered in full in Phase 3a above: algorithms & failure modes, L4 vs L7, active/passive health checks, and slow-start recovery. Core lesson — distributing requests isn't balancing load.
Day 2CDNprotection, not just speed›
A CDN's real value is protection: 10k req/s at 92% edge hit ratio means the origin sees ~800. Pull-based (95% of teams; first user pays a 200–300ms penalty) vs push-based (pre-warm for launches/live events). Cache-hit ratio is the one metric: <80% = expensive proxy, 95%+ = origin basically retired. Cache the things that look dynamic but aren't (same-for-everyone API responses, HTML fragments, per-cohort via Vary). Invalidation: TTL vs purge (bulk purge = thundering herd) vs stale-while-revalidate (the go-to — serve stale instantly, refresh in background). Modern CDNs run code at the edge (Workers, Lambda@Edge). If read:write ≈ 100:1, the CDN is your most important layer.
Day 3Caching — the 5 layersRedis is one of five›
1 Browser (0ms, per-user; version filenames or a deploy serves stale JS). 2 CDN edge (per-region; one fill serves thousands). 3 App cache Redis/Memcached (cache-aside is the 90% pattern; Redis = data structures, Memcached = simple KV). 4 DB query cache (MySQL removed it in 8.0; Postgres uses prepared statements + shared buffers — why pooling keeps caches warm). 5 OS page cache (the OS caches disk reads in RAM automatically — why "fits in memory" is fast and why more RAM often beats more CPU). Method: start at the top; a Cache-Control header may do what Redis would, with zero infra.
Day 4Cache Invalidationone of two hard problems›
Three write strategies: write-through (always consistent, double writes), write-behind (fast, risks data loss if cache dies before flush — never for money), write-around + cache-aside (write DB, refill on miss — the 80% choice). The cheat code: on write delete the key, don't update it — updates can race and leave the older value. Invalidation: start with TTL; add event-driven purge only where staleness costs money (inventory, pricing, permissions).
Day 5Rate Limitingenforce your known capacity›
Not mainly about attackers — about enforcing capacity before things break (their own batch job sent 400k requests in 90s and folded the API). Algorithms: token bucket (allows bursts — the default, AWS/Stripe), sliding window (strict — auth/payments), leaky bucket (perfectly smooth output). Enforce at ≥2 layers (gateway, per-service, per-user, per-endpoint). Return 429 with Retry-After; beware retry storms → exponential backoff + jitter. Whitelist health checks. Counters in Redis (local counters break behind an LB).
Day 6API Gatewaythe single front door›
Six jobs: authn/authz at the door, rate limiting, routing, protocol translation (REST outside, gRPC inside), response caching, centralized logging. Routing at scale: header-based (version/device), weighted/canary (5%→25%→100%, instant rollback — "deploying without a canary is just hoping"), BFF request aggregation (1 mobile round trip instead of 3). Gateway = north-south (external in); service mesh = east-west (service↔service, mTLS/retries via sidecars, overkill below ~20 services). Buy don't build; keep it thin; monitor it (it's a single entry point).
Day 7CAP Theorem"pick 2 of 3" is wrong›
Correct framing: when a partition occurs, choose consistency or availability — you don't choose P, partitions will happen. CP: refuse to answer rather than serve stale (banks, Spanner, ZooKeeper/etcd). AP: always answer, maybe stale (Cassandra, DynamoDB, DNS). The interview-winning line: CAP is per-feature, not per-system. PACELC completes it: during Partition A-vs-C; Else Latency-vs-Consistency.
Day 8Shardingthe decision you can't undo›
Sharding splits data across independent servers (cross-shard queries, broken joins, painful rebalancing) — partition before you shard. Strategies: range (hotspots), hash (kills range queries), directory (SPOF). Most teams: hash + consistent hashing. Shard-key rules: shard on what you query by, high cardinality, even distribution, co-locate joined data. Try first, in order: optimize queries → pooling → read replicas → partition → vertical scale → then shard. Heuristic: <500GB and <10k writes/s → you don't need it.
Day 9Replicationwhere your data lies to you›
Leader-follower via WAL. Sync (no loss, slower) vs async (fast, loses recent writes on leader death); production uses semi-sync. Replication lag causes read-your-own-write bugs — fix with read-from-leader-after-write, causal tokens, or leader-reads for flows that matter. Failover: election → promote most-current follower; watch for split-brain (fencing/STONITH), data-loss windows, connection storms (put PgBouncer between). Test failover quarterly — untested failover is a theory. Topologies: single-leader (default), multi-leader (multi-region writes + conflicts), leaderless (Dynamo/Cassandra).
Day 10Partitioningthe step before sharding›
Split one table within one DB. Range (dates), list (country/tenant — watch skew), hash (even spread, kills range queries). Partition pruning is the whole point — verify with EXPLAIN; it breaks if the WHERE omits the partition key, on type mismatch, or on functions applied to the key. Killer feature: DROP PARTITION for retention (ms, no locks) vs an hours-long DELETE. Sweet spot 1M–100M rows/partition; pre-create future partitions.
Day 11Queuesmakes failure graceful›
A queue doesn't make you faster — it turns an uncontrollable spike into a controllable stream. Producer → queue → consumer(ack). Delivery: at-most-once, at-least-once (SQS/RabbitMQ default — "sometimes twice"), exactly-once (hard). Production norm: at-least-once + idempotent consumers. RabbitMQ = task distributor; Kafka = replayable event log, multi-consumer, millions/s; SQS = managed. Production trio: backpressure (monitor depth, autoscale, shed at source), dead-letter queues (poison messages after 3–5 fails), idempotency. Pattern: sync fast path + async everything else.
Day 12Microservicesan org problem, not a tech one›
"Microservices solve an organisational problem, not a technical one." Real value: independent deploy by independent teams (one team, one service). Costs: function calls become network calls (retries/circuit breakers), ACID becomes sagas, debugging becomes distributed tracing, ops multiplies. Need them when teams block each other's deploys, scaling needs diverge wildly, or fault isolation is critical. Don't at <10 engineers or early-stage (Segment's 140 services became a cautionary tale). Prereqs before extracting: centralized logging + correlation IDs, health checks, gateway/mesh, circuit breakers, tracing.
Day 13Monolith vs Microservicesthe modular monolith›
The receipts: Amazon Prime Video merged back (-90% cost), Segment rebuilt their monolith, Istio re-merged. Start monolith (5–30 engineers): you don't know your boundaries until 6–12 months of real users; the bottleneck is shipping speed; operational tax doesn't pay off below ~50–80 engineers. The missing third option: the modular monolith (Shopify's "majestic monolith") — one deployable, service-grade internal boundaries; 80% of the benefit, none of the distributed pain. Extract one service only on real pain; most teams stop at 3–5.
Day 14Fault Tolerancefour patterns›
1 Timeouts — many libraries wait forever by default; without one a slow dependency fills your pool and the failure walks upstream. Set connect 1–3s, read = p99 + buffer, total < caller's timeout. 2 Circuit breakers — closed → open (fail fast, let it recover) → half-open (probe); trip at 50% over 10s with ≥20 requests; use a library. 3 Bulkheads — per-dependency thread pools so one hang can't starve all; size = throughput × p99 + headroom. 4 Retries + backoff + jitter — the most dangerous pattern (retry storms helped keep S3 down in 2017); cap at 3; only retry idempotent ops. Then chaos-test in staging.
Day 15Database Scalingthe 5-step ladder›
Each step ~10× cheaper than the next; walk top to bottom; most products never reach the bottom. 1 Indexes (a CREATE INDEX turns 4s into 4ms; foreign-key indexes and composite indexes are the two everyone misses). 2 Connection pooling (PgBouncer transaction mode: 5,000 app connections → 50 real ones). 3 Read replicas (N× read throughput; mind lag). 4 Partitioning. 5 Sharding (last resort). Start with the symptom, not the solution — the slow-query log tells you which step. Most problems are solved at steps 1–2.
Day 16Service Discoverywhere is service X right now?›
Hardcoding works until instances go ephemeral / deploys go frequent / you cross regions. DNS-based (universal, but TTL caching serves dead IPs). Registry-based (Consul/Eureka/etcd — self-register + heartbeat + metadata routing; the registry becomes critical infra). Platform-native (Kubernetes) — Service objects, zero app code — use it if you're on K8s. Service mesh sits above discovery. "Discovery should be boring. The day you notice it is the day it broke."
Day 17Consistency Modelsfive things, not one›
Strongest → weakest: Linearizability (single real-time order; needs a primary or consensus; costs latency + availability) → Sequential (per-client order; mostly academic) → Causal (happened-before respected — comments after posts; vector clocks) → Eventual (stop writing, replicas converge; nearly free; breaks read-your-own-write → sticky sessions). CP examples: Spanner, etcd, ZooKeeper. AP: DynamoDB, Cassandra, S3. Pick the weakest model that doesn't break your product, per operation; test with Jepsen.
Day 18Eventual Consistency in ProductionKnight Capital, $440M›
Four failures: read-your-own-write, monotonic-reads breaking (5→4→5 orders), lost updates, causality breaks. Four fixes: read-your-writes (sticky sessions / read-from-primary / causal tokens), monotonic reads (pin user to a replica), session consistency (combine both per session — what Facebook/Twitter/Instagram do), conflict resolution (LWW = silent data loss; vector clocks; CRDTs — how Google Docs/Figma merge; app-defined merge). Traps: trusting the DB's default LWW; read-modify-write on eventual data (use atomic ops); hide latency with optimistic UI.
Day 19Distributed Transactions & SagasBritish Airways, £80M›
No shared transaction across services = no shared ROLLBACK; "undo a credit-card charge" ≠ ROLLBACK. 2PC: prepare/commit — killed by being slow, coordinator SPOF (the blocking problem), and resource locking; almost never in modern microservices. Sagas: a sequence of local transactions + compensating actions; choreography (event-driven, loosely coupled, hard to debug) vs orchestration (central coordinator — use for 4+ steps / money). Hard parts: compensations that can fail, mandatory idempotency, dirty reads (mark "reserved"), stuck-saga timeouts, concurrency on the last item. Best move: restructure so you don't need the distributed transaction at all.
Day 20Leader ElectionGitHub, 24h outage›
Raft: leader sends heartbeats; on timeout a follower becomes candidate, increments the term, and needs a majority — so two leaders in one term is impossible; split votes resolve via randomized timeouts. Split-brain (the GitHub failure — two leaders both accepting writes) is prevented by fencing tokens, STONITH, or leases. Tools: etcd (modern default, K8s' brain), ZooKeeper, Consul; Redis+Redlock only for non-critical. Don't need it for stateless servers or idempotent work.
Day 21Horizontal vs Vertical ScalingTwitter's Fail Whale›
Vertical — boring, works longer than you think (one AWS box = 128 cores/4TB; Stack Overflow ran billions of pageviews on a handful of machines); ceilings are the physical max and a non-linear cost curve, plus it's a single point of failure. Horizontal — near-unlimited + fault tolerance for free, but demands statelessness (the make-or-break: sessions → Redis, files → S3, so any server handles any request). Path: scale up first, make it stateless early (highest-leverage, cheap now), add a read replica, then scale out for a real reason. Best architecture for 1,000 users is usually one good machine.
Day 22Back-of-the-Envelope Estimationthe #1 interview failure›
Candidates fail by designing for the wrong scale. Numbers: 1 day ≈ 86,400s (round to 100k); one server ≈ 1k–10k req/s; one DB ≈ 1000s writes/s; a text record ≈ 1KB. Method: users → DAU → req/s (÷100k) → peak (×2–3) → storage (×retention) → bandwidth → read the numbers, decide the architecture. Worked URL-shortener: 100M URLs/mo → 40 writes/s, 4k reads/s → one DB + Redis + replicas, no sharding. Estimation drives design, not the reverse.
Day 23Idempotencythe ₹9,998 double charge›
Duplicate delivery is the default behaviour of every distributed system — retries, LB re-routes, at-least-once queues, double-clicks. "Set balance to X" is naturally idempotent; "add X" is a landmine. Idempotency keys: client generates one key per operation; server processes-and-stores on first sight, replays the stored result on repeat (Stripe's contract). Sharp edges: the store must be shared (Redis / DB unique constraint) and fast; store the result not just the key; the in-flight collision is caught by the unique constraint; keys expire in 24–48h.
Day 24Data Localitycloseness is speed›
Almost every performance problem = how far the data travels to the compute. The latency ladder spans a million-fold (CPU cache 1ns → cross-region 150ms). Three levers: co-locate data used together (the deep reason shard keys matter), move compute to data (push filtering into the DB; don't ship a billion rows to keep fifty), serve from the edge (CDNs + regional replicas). Caching, sharding, CDNs, replication are one idea at different layers — shorten the distance between question and answer.
Day 25The One Idea Behind All 25it depends — on what?›
The 25 concepts answer three questions: handle more load, store more than one machine holds, stay correct when things fail. The single sentence: every architecture decision trades off speed, cost, and correctness — you can't maximize all three. Strong consistency costs latency; low latency costs consistency; fault tolerance costs money. "It depends" is the correct answer — the skill is knowing what it depends on. The patterns are vocabulary; judgment, built by shipping and breaking things, is the craft.

Interview-day tactics

Coding round

Restate the problem & confirm examples → clarify constraints & edge cases → state brute force → optimize out loud → narrate as you code → test with a small example → state complexity. Silence is the enemy; a wrong turn you explain beats a right answer in silence.

System design round

Drive it: clarify → estimate → API → boxes → deep-dive → bottlenecks. Ask questions constantly — interviewers grade the questions. State trade-offs explicitly. It's a conversation, not a monologue.

When you're stuck

Think out loud, don't freeze. Solve a simpler version first. Ask a clarifying question. Reason from a known pattern ("this smells like sliding window"). Interviewers can only credit reasoning they can hear.

Communication

Structure every answer: headline → detail → trade-off. Manage time (don't over-invest in step 1). Be coachable — take hints gracefully. Stay calm; the loop is a collaboration, not an interrogation.

Scenario questions

Q1A payment POST times out and the client retries. How do you prevent a double charge?›
AnswerIdempotency-Key header. Client sends a unique key per operation; the server stores the result against it on first processing and replays that stored result on the retry — the card is touched once. The key store must be shared (Redis / DB unique constraint) and fast; store the result, not just the key.
Q2One server in your pool is at 95% CPU while others idle. Why, and what do you change?›
AnswerRound-robin blindness or an IP-hash hotspot — the balancer distributes requests without knowing their cost. Move to least-response-time routing; check the health-check config and enable slow-start so recovered nodes ramp up.
Q3A hot cache key expires and your DB falls over. What happened, how do you prevent it?›
AnswerCache stampede — the hot key expired and thousands of requests hit the DB simultaneously. Fix with a single-flight lock (one request refills, others wait), staggered TTLs, or early/background refresh (stale-while-revalidate).
Q4Your ORM page issues 101 queries for one screen. Diagnose and fix.›
AnswerN+1 query — one query for the list, then one per row. Fix with eager loading / a JOIN / batching (an IN (...) query). Confirm with the query log and EXPLAIN.
Q5A downstream service is failing, and now your service is failing too. What pattern is missing?›
AnswerTimeouts (a slow dependency is filling your connection pool) + a circuit breaker (stop hammering the dead dependency, fail fast) + a fallback / graceful degradation. Bulkheads would isolate the blast radius.
Q6All users behind one corporate IP land on the same server. Which LB algorithm caused it?›
AnswerIP hash — it routes by client IP for sticky sessions, so one NAT with thousands of users hashes to a single server: an instant hotspot. Switch to a stateful session store + a load-aware algorithm.
Q7A recovered server dies again seconds after rejoining the pool. Why?›
AnswerNo slow-start. It rejoined at full weight and the backlog (plus cold caches) crushed it. Enable slow-start recovery so traffic ramps up gradually.
Q8Kafka consumer lag is growing. What do you look at?›
AnswerPartition count vs number of consumers (parallelism ceiling), per-message processing time, backpressure downstream, and poison messages blocking progress → route them to a dead-letter queue. Scale consumers or speed up processing.
Q9Why can't you just hash(key) % N across cache servers?›
AnswerAdding or removing a server changes N, so nearly every key remaps and the cache is effectively flushed. Consistent hashing (with virtual nodes) remaps only ~1/N of keys when the ring changes.
Q10Design a URL shortener / rate limiter / chat / news feed.›
AnswerRun the 6-step template: clarify → estimate → API → boxes → deep-dive → bottlenecks. See the four worked case studies above for the numbers and key decisions.
