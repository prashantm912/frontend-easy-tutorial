Modern Java — Mastering Features from Java 8 to Java 25

A guided tour of everything that turned Java from a verbose, imperative language into a concise, expressive, modern one. It opens with the Java 8 revolution — lambda expressions, functional interfaces, method and constructor references, the Streams API, Optional, default and static interface methods, and the java.time date/time library — then follows the language forward through the features that arrived in Java 9 and beyond: var, text blocks, enhanced switch, records, sealed types, pattern matching, unnamed variables, stream gatherers, the simple web server, the modern HttpClient, and the module system. Every feature is tied to the JDK release that introduced it, and each lecture pairs a focused, reimplemented example with the console output it produces, traced by hand (Java runs on the JVM, not in the browser).

Prerequisites & how to read this You should know core Java — classes, interfaces, generics, collections. This guide spans Java 8 (2014) through Java 25 (2025 LTS); each feature is labelled with the release it became final in, so you can tell what your JDK supports. To run a snippet, use a JDK at or above the noted version: javac File.java && java File, or for a single file java File.java. A few late features (records/sealed/pattern-matching in switch, stream gatherers, compact source files) need a recent JDK — the pitfalls call this out. Use the filter box in the sidebar to jump to any lecture. Prose is original; code examples are reimplemented for clarity.

1. Why Modern Java? (Java 8 → 25)

Modern Java is the story of one big idea — treating behaviour as data — growing across a decade of releases. This section maps the timeline from Java 8 to Java 25, explains why Java 8 was the turning point, and contrasts the two mindsets you will use throughout the course: imperative (describe every step) versus declarative (describe the result).

Run locally Save a snippet as e.g. Demo.java and run javac Demo.java && java Demo, or run a single file directly with java Demo.java (JDK 11+). Each feature notes the minimum JDK it needs.

01. The modern-Java big picture — Java 8 to Java 25

Objective: Get a mental map of which JDK release added which headline feature, and understand the six-month release cadence and the LTS milestones (8, 11, 17, 21, 25).

Java 8 (2014) is where the language turned functional: lambdas, the Streams API, Optional, and the java.time date/time library all landed together. Everything you learn in this course builds on that foundation.

Starting with Java 9 (2017) the JDK moved to a time-based, six-month release cadence, so version numbers now climb quickly. Most releases are short-lived "interim" builds; every few years one is designated Long-Term-Support (LTS). The LTS line so far is 8, 11, 17, 21, and 25 — those are the versions production teams usually target.

The headline features by era: var local type inference (Java 10); a standard HttpClient (Java 11); switch expressions (Java 14), text blocks (Java 15), records & instanceof pattern matching (Java 16), and sealed types (Java 17); pattern matching for switch, record patterns, and virtual threads (Java 21); stream gatherers (Java 24); and compact source files with an instance main (Java 25).

  • Java 8 (2014) introduced functional programming: lambdas, streams, Optional, java.time.
  • From Java 9 (2017) the JDK ships every six months; LTS releases are 8, 11, 17, 21, 25.
  • var (10), HttpClient (11), switch expressions (14), text blocks (15), records + instanceof patterns (16), sealed types (17).
  • Java 21 (2023 LTS): pattern matching for switch, record patterns, virtual threads, sequenced collections.
  • Java 24 added stream gatherers; Java 25 finalized compact source files + instance main (JEP 512).

A one-line taste of the functional style you will meet everywhere — a stream pipeline that filters, transforms, and joins:

List<String> names = List.of("Ann", "Bob", "Cara");   // List.of: Java 9
String shortNames = names.stream()                     // Streams: Java 8
        .filter(n -> n.length() == 3)
        .map(String::toUpperCase)
        .collect(Collectors.joining(", "));
System.out.println(shortNames);   // ANN, BOB

02. Why Java 8 changed everything

Objective: See concretely how Java 8 replaced verbose anonymous classes with lambdas, letting you pass behaviour as data instead of ceremony.

Before Java 8, "passing behaviour" meant writing an anonymous inner class — several lines of boilerplate to wrap a single expression. Java 8 introduced lambda expressions, a compact literal for an instance of a functional interface (an interface with exactly one abstract method, such as Comparator, Runnable, or Function).

But conciseness is only the surface reason Java 8 mattered. The deeper driver was hardware: modern machines — even the phone in your pocket — have many cores, and before Java 8 the only way to use them was to write and coordinate threads by hand, which is verbose and error-prone. Expressing logic as behaviour you can pass lets a library run a pipeline across cores for you (e.g. parallelStream()) without you managing threads. The functional style also favours immutable data, and since shared mutable state is the root of a whole class of concurrency bugs, leaning on immutability removes many of them by construction.

The three sorts below are behaviourally identical, but the code shrinks from an eight-line anonymous class to a single lambda — and then to a ready-made comparator via a method reference. Less boilerplate, and the intent (sort in natural order) is now front and centre.

  • A lambda is a concise instance of a functional interface — one abstract method.
  • Behaviour becomes a value you can store, pass, and return (Java 8+).
  • The deeper driver was multi-core hardware: passing behaviour as data lets libraries parallelise the work for you, and the functional preference for immutable data avoids a major class of concurrency bugs.
  • Method references (Comparator.naturalOrder(), String::toUpperCase) are the shortest form when a method already does the work.
  • The compiler infers parameter types, so (a, b) -> a.compareTo(b) needs no type annotations.
import java.util.*;

public class Demo {
    public static void main(String[] args) {
        List<String> names =
            new ArrayList<>(Arrays.asList("Charlie", "Alice", "Bob"));

        // Pre-Java 8: pass behaviour as a verbose anonymous class
        Collections.sort(names, new Comparator<String>() {
            @Override public int compare(String a, String b) {
                return a.compareTo(b);
            }
        });
        System.out.println(names);

        // Java 8: the SAME behaviour, expressed as a lambda
        names.sort((a, b) -> a.compareTo(b));
        System.out.println(names);

        // Idiomatic Java 8: a ready-made comparator (method reference)
        names.sort(Comparator.naturalOrder());
        System.out.println(names);
    }
}
Console output[Alice, Bob, Charlie] [Alice, Bob, Charlie] [Alice, Bob, Charlie]

Source: reimplemented as an original illustration.

AdvancedA lambda is not just syntactic sugar for an anonymous class. An anonymous class always creates a new object and its own this; a lambda captures the enclosing this and is compiled to an invokedynamic call site (via LambdaMetafactory), which the JVM can link and reuse cheaply. Captured local variables must be effectively final in both forms.
Pitfall A lambda only targets a functional interface — exactly one abstract method. Trying to write a lambda for an interface with two abstract methods will not compile; use an anonymous class (or refactor the interface) instead.

03. Imperative vs declarative programming

Objective: Distinguish the imperative style (spell out every step and mutate state) from the declarative style (describe the desired result and let the library choose the steps), using two side-by-side examples.

Imperative code answers "how": initialise an accumulator, loop, mutate, repeat. Declarative code answers "what": you name the outcome — the sum of a range, the distinct elements — and a stream pipeline handles the mechanics. Declarative code tends to be shorter, has less mutable state to get wrong, and reads closer to the problem statement.

Example (a) — sum 0 to 100. The imperative version keeps a mutable sum and adds inside a for loop; the declarative version simply asks an IntStream for its sum. Both yield 5050.

import java.util.stream.IntStream;

public class SumDemo {
    public static void main(String[] args) {
        // Imperative: describe HOW, mutating an accumulator
        int sum = 0;
        for (int i = 0; i <= 100; i++) {
            sum += i;
        }
        System.out.println(sum);

        // Declarative: describe WHAT -- "the sum of 0..100"
        int total = IntStream.rangeClosed(0, 100).sum();   // Streams: Java 8
        System.out.println(total);
    }
}
Console output5050 5050

Example (b) — remove duplicates. Imperatively you juggle a Set to remember what you have seen plus a result List; declaratively you call distinct(). Both keep encounter order and produce [1, 2, 3, 4].

import java.util.*;

public class DistinctDemo {
    public static void main(String[] args) {
        List<Integer> numbers = List.of(1, 2, 2, 3, 3, 3, 4);  // List.of: Java 9

        // Imperative: track seen values in a Set by hand
        Set<Integer> seen = new HashSet<>();
        List<Integer> unique = new ArrayList<>();
        for (int n : numbers) {
            if (seen.add(n)) {          // add() returns false if already present
                unique.add(n);
            }
        }
        System.out.println(unique);

        // Declarative: describe the outcome
        List<Integer> distinct = numbers.stream().distinct().toList();  // toList(): Java 16
        System.out.println(distinct);
    }
}
Console output[1, 2, 3, 4] [1, 2, 3, 4]

Source: reimplemented as an original illustration.

AdvancedDeclarative pipelines are easy to parallelise — numbers.parallelStream().distinct() or .stream().parallel() splits the work across a fork/join pool without you touching the logic. That freedom is a payoff of describing what rather than how. (For a tiny sum, the plain for loop still wins on raw speed; reach for parallel streams only with large, CPU-bound data.)
Pitfall Stream.toList() needs Java 16+ and returns an unmodifiable list; on Java 8–15 use .collect(Collectors.toList()) instead. Likewise List.of(...) (Java 9) is immutable — calling add() on it throws UnsupportedOperationException.

2. Lambda Expressions

Lambda expressions (final in Java 8, 2014) are the single feature that opened the door to functional-style Java. A lambda is a compact, anonymous implementation of a functional interface's one abstract method — the syntactic building block that later powers method references, streams, and Optional. This section defines what a lambda is, then puts it to work replacing two verbose anonymous classes: Runnable and Comparator.

01. What is a lambda expression?

Objective: Define a lambda as an anonymous function that implements a functional interface's single abstract method, and read every form of its (parameters) -> body syntax.

A lambda expression (Java 8) is an anonymous function — a block of behaviour with parameters and a body, but no name, no modifiers, and no explicit return type. It exists only to supply the one abstract method of a functional interface (an interface with exactly one abstract method). Because the interface fixes the method signature, the compiler infers the parameter types for you, so the lambda can stay terse.

The syntax is (parameters) -> body. The body comes in two shapes: an expression body whose value is returned implicitly (x -> x * 2), or a block body in braces with an explicit return (x -> { return x * 2; }). A lambda has no name of its own, but it can be assigned to a variable whose type is the functional interface, passed as an argument, or returned from a method.

A lambda also captures its enclosing scope: it can read local variables from the surrounding method, provided those variables are effectively final (assigned once). This is what makes lambdas closures rather than plain functions.

  • A lambda implements the single abstract method of a functional interface — nothing else.
  • Expression body x -> x * 2 returns implicitly; block body x -> { return x * 2; } needs an explicit return.
  • Parameter types are inferred; a single parameter may drop the parentheses (x -> ...); zero parameters is () -> ....
  • A lambda has no name but is assignable to a functional-interface-typed variable.
  • It captures effectively-final local variables from the enclosing scope (a closure).
import java.util.function.*;

public class LambdaBasics {
    public static void main(String[] args) {
        // Expression body — type of x inferred as Integer
        Function<Integer, Integer> doubler   = x -> x * 2;
        // Block body — explicit return, same result
        Function<Integer, Integer> doublerBlk = x -> { return x * 2; };
        // Zero parameters
        Supplier<String> greeter = () -> "Hello";
        // Two parameters
        BinaryOperator<Integer> add = (a, b) -> a + b;

        System.out.println(doubler.apply(5));
        System.out.println(doublerBlk.apply(5));
        System.out.println(greeter.get());
        System.out.println(add.apply(3, 4));
    }
}
Console output10 10 Hello 7

Source: reimplemented as an original illustration.

AdvancedA lambda is not just sugar for an anonymous class. It captures this from the enclosing instance (an anonymous class's this refers to the anonymous class itself), it does not introduce a new scope for names, and the compiler emits it via the invokedynamic bytecode instruction (backed by LambdaMetafactory) rather than generating a separate Outer$1.class file. This makes lambdas lighter to allocate and often reusable.
Pitfall A lambda only works where the target type is a functional interface. Writing Object o = () -> "hi"; fails to compile — Object is not a functional interface, so the compiler cannot infer which method the lambda implements. Give it a concrete functional-interface type such as Supplier<String>.

02. Implementing Runnable with a lambda

Objective: Replace a verbose anonymous Runnable with a one-line lambda and appreciate the readability win.

Runnable is a textbook functional interface: it declares exactly one abstract method, void run(), taking no arguments and returning nothing. Before Java 8 you implemented it with an anonymous class — a five-line block of ceremony (new Runnable() { public void run() { ... } }) wrapped around a single statement.

Because run() is the lone abstract method, a lambda can supply its body directly. With no parameters and a single statement, the lambda collapses to () -> System.out.println("Running"). The two forms are interchangeable: you can call r.run() directly, or hand the Runnable to a thread with new Thread(r).start().

  • Runnable has one abstract method — void run() — so it is a functional interface.
  • Anonymous class: new Runnable() { public void run() {...} }; lambda: () -> ....
  • The lambda takes zero parameters, so the empty parentheses () are required.
  • Invoke with r.run() (same thread) or run it concurrently with new Thread(r).start().
public class RunnableDemo {
    public static void main(String[] args) {
        // Legacy: anonymous class implementing Runnable
        Runnable legacy = new Runnable() {
            @Override public void run() {
                System.out.println("Running (anonymous class)");
            }
        };

        // Modern: a lambda supplies run() directly
        Runnable modern = () -> System.out.println("Running (lambda)");

        legacy.run();
        modern.run();
    }
}
Console outputRunning (anonymous class) Running (lambda)

Source: reimplemented as an original illustration.

AdvancedCalling legacy.run() / modern.run() executes on the current thread, so the output order above is deterministic. If you instead write new Thread(modern).start(), the body runs on a separate thread and its output can interleave with or precede/follow other threads' output — the ordering is no longer guaranteed. Use run() when you just want the behaviour; use a thread (or an ExecutorService) when you actually want concurrency.
Pitfall A lambda's braces mean a block body, not a call. Runnable r = () -> { doWork(); } is a statement block (correct). But do not confuse run() with start(): calling r.run() just invokes the method inline; only new Thread(r).start() launches a new thread.

03. Implementing Comparator with a lambda

Objective: Sort a list with a Comparator written as a lambda, using Integer.compare for safe integer comparison, and preview the cleaner Comparator.comparing form.

Comparator<T> is a functional interface whose single abstract method is int compare(T a, T b). It must return a negative number when a should sort before b, zero when they are equal, and a positive number when a should sort after b. A lambda expresses this rule in one line — for example, comparing strings by length.

Write the body as Integer.compare(a.length(), b.length()) rather than the a.length() - b.length() shortcut. As a habit, prefer Integer.compare(x, y) over x - y: the subtraction trick can overflow the int range and flip the sign when the operands may be large and of opposite sign — for example arbitrary or negative int fields like a.getBalance() - b.getBalance(). For string lengths specifically the subtraction is actually safe (a length is non-negative and at most Integer.MAX_VALUE, so the difference always fits in an int), but using Integer.compare everywhere means you never have to stop and reason about the bounds.

Passing the comparator to List.sort reorders the list in place. And because sorting by a key is so common, Java 8 also offers Comparator.comparing(String::length) — a key-extractor form that reads even more cleanly (its default methods, such as thenComparing and reversed, are covered in Section 11).

  • Comparator<T>'s single method is int compare(T a, T b) — negative / zero / positive.
  • Prefer Integer.compare(x, y) over x - y as a habit: subtraction can overflow and flip the sign for large, opposite-sign operands (string lengths happen to be safe, but the habit spares you from checking).
  • names.sort(byLength) sorts the list in place (needs a mutable list).
  • Comparator.comparing(String::length) is the cleaner key-extractor form (foreshadows Section 11).
import java.util.*;

public class ComparatorDemo {
    public static void main(String[] args) {
        // Comparator as a lambda — compare strings by length
        Comparator<String> byLength =
            (a, b) -> Integer.compare(a.length(), b.length());

        // List.of(...) is immutable, so wrap it for in-place sort
        List<String> names = new ArrayList<>(List.of("Charlie", "Al", "Bob"));
        names.sort(byLength);
        System.out.println(names);

        // Even cleaner: key extractor (Java 8) — see Section 11
        names.sort(Comparator.comparing(String::length));
        System.out.println(names);
    }
}
Console output[Al, Bob, Charlie] [Al, Bob, Charlie]

Source: reimplemented as an original illustration.

AdvancedLengths here are Al=2, Bob=3, Charlie=7, so ascending order is [Al, Bob, Charlie]. When two elements compare equal, List.sort keeps their original relative order — it is a stable sort (backed by TimSort). That stability is why chaining byLength.thenComparing(Comparator.naturalOrder()) can break ties predictably without disturbing already-ordered runs.
Pitfall List.of("Charlie", "Al", "Bob") returns an immutable list; calling .sort(...) on it throws UnsupportedOperationException. Copy it into a mutable list first — new ArrayList<>(List.of(...)) — as shown above.

3. Functional Interfaces

Lambdas need a target type, and that type is always a functional interface: an interface with exactly one abstract method. Java 8 shipped a ready-made toolkit of them in java.util.function — Consumer, Predicate, Function, Supplier and their two-argument and operator variants — so you rarely have to declare your own.

Version note Every interface in this section — the whole java.util.function package plus the default combinator methods (andThen, compose, and, or, negate) — is Java 8 (2014). The one later addition to the package is the static Predicate.not(...) factory in Java 11.

01. Functional interfaces & @FunctionalInterface

Objective: Understand what makes an interface "functional" (a single abstract method), how a lambda supplies that method, and what the @FunctionalInterface annotation buys you.

A functional interface has exactly one abstract method — the SAM (Single Abstract Method). Because there is only one method to implement, the compiler can treat a lambda or method reference as an instance of that interface: the lambda body is the implementation of that one method.

Default methods, static methods, and re-declared Object methods (like equals) don't count toward the "one abstract method" rule, so an interface can have many of those and still be functional. The optional @FunctionalInterface annotation asks the compiler to enforce the SAM rule: add a second abstract method and the code won't compile.

The java.util.function package groups its interfaces into four families by shape: Consumer<T> takes a value and returns nothing, Supplier<T> takes nothing and returns a value, Predicate<T> takes a value and returns a boolean, and Function<T, R> takes a value and returns another. Consumer, Predicate and Function each also have a two-argument Bi- cousin (BiConsumer, BiPredicate, BiFunction), and all four families have primitive-specialized cousins (IntConsumer, IntSupplier, IntPredicate, …) — but there is no BiSupplier.

  • Functional interface = exactly one abstract method (SAM).
  • A lambda / method reference provides that single method's body.
  • @FunctionalInterface is optional but makes the compiler enforce the SAM rule.
  • Default and static methods do not count as abstract methods.
  • Familiar SAM types predate the package: Runnable, Comparator<T>, Callable<V>.
@FunctionalInterface
interface Greeter {
    String greet(String name);          // the one abstract method (SAM)

    // default methods are allowed and don't break the SAM rule
    default String greetLoudly(String name) {
        return greet(name).toUpperCase();
    }
}

public class Demo {
    public static void main(String[] args) {
        // the lambda IS the body of greet(...)
        Greeter g = name -> "Hello, " + name + "!";
        System.out.println(g.greet("Ada"));
        System.out.println(g.greetLoudly("Ada"));
    }
}

02. Consumer & BiConsumer

Objective: Use Consumer<T> for side-effecting operations that return nothing, chain them with andThen, and reach for BiConsumer<T, U> when you have two inputs.

Consumer<T> represents an operation that accepts one input and returns no result — its abstract method is void accept(T t). It exists purely for the side effect: printing, logging, mutating, adding to a collection. Its default method andThen(Consumer) returns a new consumer that runs this consumer first, then the argument, both on the same input.

BiConsumer<T, U> is the two-argument version — void accept(T t, U u) — the exact shape that Map.forEach hands you, one call per key/value pair.

  • Consumer<T>: void accept(T) — side effect, no return value.
  • andThen chains two consumers over the same input, in order.
  • BiConsumer<T, U>: void accept(T, U) — pairs well with Map.forEach.
  • List.forEach(Consumer) and Map.forEach(BiConsumer) take these directly.
import java.util.function.Consumer;
import java.util.function.BiConsumer;
import java.util.Map;

public class Demo {
    public static void main(String[] args) {
        Consumer<String> print = s -> System.out.println("value: " + s);
        Consumer<String> shout = s -> System.out.println(s.toUpperCase());

        // runs print first, then shout, on the same input
        print.andThen(shout).accept("hello");

        BiConsumer<String, Integer> entry =
            (k, v) -> System.out.println(k + " = " + v);
        Map.of("age", 30).forEach(entry);
    }
}

AdvancedPrimitive specializations avoid boxing: IntConsumer, LongConsumer, DoubleConsumer accept an int/long/double directly. There are also mixed BiConsumer-style variants like ObjIntConsumer<T> — accept(T, int) — used by some collector implementations.
Pitfall andThen returns a new Consumer — it does not mutate the original. print.andThen(shout); on its own line does nothing; you must capture the result or call .accept(...) on the returned consumer.

03. Predicate, BiPredicate & combinators

Objective: Write boolean tests as Predicate<T> and compose them with and, or, and negate instead of hand-writing nested boolean logic.

Predicate<T> represents a boolean-valued test — boolean test(T t). Its power comes from the default combinators: and, or, and negate build a new predicate from existing ones, so small named tests compose into complex conditions that read like English. This is what Stream.filter consumes.

BiPredicate<T, U> is the two-argument form — boolean test(T, U) — for tests that need two inputs, such as "is this string longer than N characters".

  • Predicate<T>: boolean test(T).
  • and / or combine two predicates; negate flips one.
  • Combinators short-circuit just like && / ||.
  • BiPredicate<T, U>: boolean test(T, U) for two inputs.
  • Static Predicate.not(p) (Java 11) reads better than p.negate() at call sites.
import java.util.function.Predicate;
import java.util.function.BiPredicate;

public class Demo {
    public static void main(String[] args) {
        Predicate<Integer> isEven     = n -> n % 2 == 0;
        Predicate<Integer> isPositive = n -> n > 0;

        Predicate<Integer> evenAndPositive = isEven.and(isPositive);
        Predicate<Integer> evenOrPositive  = isEven.or(isPositive);
        Predicate<Integer> notEven         = isEven.negate();

        System.out.println(evenAndPositive.test(4));    // even AND positive
        System.out.println(evenAndPositive.test(-4));   // even but not positive
        System.out.println(evenOrPositive.test(3));     // odd but positive
        System.out.println(notEven.test(3));            // negate of isEven

        BiPredicate<String, Integer> longerThan = (s, len) -> s.length() > len;
        System.out.println(longerThan.test("hello", 3));
    }
}

AdvancedCombinator order can matter for side effects and cost: a.and(b) evaluates a first and skips b when a is false. Put the cheapest or most-selective predicate first. Note that p.and(q).negate() is !(p && q), whereas p.negate().and(q) is !p && q — the parentheses are implicit in the call chain.
Pitfall and/or/negate return new predicates; the originals are unchanged. Also, don't confuse Predicate.isEqual(target) (a static factory that tests Objects.equals) with the instance combinators.

04. Function, BiFunction, andThen & compose

Objective: Transform values with Function<T, R>, and master the difference between andThen (this-then-next) and compose (other-first-then-this).

Function<T, R> maps an input to an output — R apply(T t) — the workhorse behind Stream.map. Two default methods chain functions, and their order is the classic gotcha: f.andThen(g) applies f first then feeds the result to g; f.compose(g) applies g first then f. So f.andThen(g) equals g.compose(f).

The trace below makes the difference concrete with f = x*2 and g = x+3 on the input 5: andThen gives (5*2)+3 = 13, compose gives (5+3)*2 = 16. BiFunction<T, U, R> is the two-argument version — R apply(T, U).

  • Function<T, R>: R apply(T).
  • f.andThen(g): apply f, then g — left to right.
  • f.compose(g): apply g, then f — right to left.
  • Function.identity() returns its input unchanged (handy in collectors).
  • BiFunction<T, U, R>: R apply(T, U).
import java.util.function.Function;
import java.util.function.BiFunction;

public class Demo {
    public static void main(String[] args) {
        Function<Integer, Integer> f = x -> x * 2;
        Function<Integer, Integer> g = x -> x + 3;

        // andThen: f FIRST, then g  ->  (5*2)+3
        System.out.println(f.andThen(g).apply(5));
        // compose: g FIRST, then f  ->  (5+3)*2
        System.out.println(f.compose(g).apply(5));

        Function<Integer, String> toLabel = n -> "n=" + n;
        System.out.println(f.andThen(toLabel).apply(4));   // 4*2 -> "n=8"

        BiFunction<Integer, Integer, Integer> add = (a, b) -> a + b;
        System.out.println(add.apply(7, 8));
    }
}

For multi-argument-to-single-result adapters there are also ToIntFunction<T>, IntFunction<R>, and ToIntBiFunction<T, U> etc. to avoid boxing.
Pitfall Reaching for compose when you meant andThen silently reverses the order and gives a wrong-but-plausible number (16 instead of 13 here). When in doubt, prefer andThen and read left to right.

05. UnaryOperator & BinaryOperator

Objective: Use the operator interfaces for the common case where the output type equals the input type, and know the static minBy/maxBy factories on BinaryOperator.

When a function's output type is the same as its input type, the operator interfaces read more clearly and constrain the API. UnaryOperator<T> extends Function<T, T> — one T in, one T out — and is what List.replaceAll expects. BinaryOperator<T> extends BiFunction<T, T, T> — two Ts in, one T out — the shape Stream.reduce uses to fold a stream.

BinaryOperator also adds two static factories, minBy(comparator) and maxBy(comparator), that return the smaller or larger of two arguments according to the given comparator.

  • UnaryOperator<T> extends Function<T, T> — same in/out type.
  • BinaryOperator<T> extends BiFunction<T, T, T>.
  • Static BinaryOperator.minBy/maxBy(comparator) pick the extreme of two values.
  • List.replaceAll takes a UnaryOperator; Stream.reduce takes a BinaryOperator.
import java.util.function.UnaryOperator;
import java.util.function.BinaryOperator;
import java.util.Comparator;

public class Demo {
    public static void main(String[] args) {
        UnaryOperator<String> shout = s -> s.toUpperCase();
        System.out.println(shout.apply("hello"));

        BinaryOperator<Integer> sum = Integer::sum;   // (a, b) -> a + b
        System.out.println(sum.apply(20, 22));

        // pick the shorter of two strings
        BinaryOperator<String> shorter =
            BinaryOperator.minBy(Comparator.comparingInt(String::length));
        System.out.println(shorter.apply("apple", "kiwi"));
    }
}

Pitfall These are convenience subtypes, not new capabilities: a method that declares Function<T, T> won't auto-accept a variable typed UnaryOperator<T> in every generic context, and vice-versa. Prefer the operator type in your own APIs when in/out are the same, but don't expect the compiler to interchange them freely under wildcards.

06. Supplier

Objective: Use Supplier<T> to defer producing a value until it is actually needed, and understand the eager-vs-lazy distinction that makes it useful.

Supplier<T> takes no input and produces a value — T get(). Its whole reason to exist is laziness: wrapping an expression in a supplier means the expression's body doesn't run until someone calls get(), and it runs again on every call. This is why APIs like Optional.orElseGet(Supplier), Objects.requireNonNullElseGet, and lazy logging take a Supplier rather than a ready-made value.

In the trace below, "supplier ready" prints before "...loading" — proof that assigning the method reference did not run the method. Contrast this with passing an already-computed value (eager), which would have run the work up front whether or not it was used. A constructor reference like ArrayList::new is a common supplier that manufactures a fresh object on demand.

  • Supplier<T>: T get() — no input, produces a value.
  • Lazy: the body runs only when get() is called, every time.
  • Powers Optional.orElseGet, requireNonNullElseGet, deferred logging.
  • ArrayList::new is a Supplier<List<String>> that makes a new list each call.
  • Primitive forms avoid boxing: IntSupplier, BooleanSupplier, etc.
import java.util.function.Supplier;
import java.util.ArrayList;
import java.util.List;

public class Demo {
    static String load() {
        System.out.println("...loading");
        return "config";
    }
    public static void main(String[] args) {
        Supplier<String> lazy = Demo::load;   // load() does NOT run yet
        System.out.println("supplier ready");
        System.out.println(lazy.get());       // now the body runs

        Supplier<List<String>> newList = ArrayList::new;
        System.out.println(newList.get());    // a fresh empty list
    }
}

4. Method References & Effectively-Final

A method reference is a lambda whose body is a single method call, written more compactly. This section covers the four kinds of method reference, when to prefer (and when to avoid) them, constructor references, and the rule that lambdas may only capture local variables that are final or effectively final.

JDK context Everything here shipped in Java 8 (2014) alongside lambdas: the :: method-reference syntax, constructor references, and the relaxed capture rule. Java 8 replaced the older requirement that a captured local be declared final with the softer notion of effectively final.

01. Method references — the four kinds

Objective: Recognise the four kinds of method reference and write the equivalent lambda for each.

When a lambda does nothing but forward its arguments to one existing method, the ClassOrObject::method shorthand says the same thing with less noise. The compiler still infers a functional-interface instance; the reference is purely a nicer way to spell the same lambda.

There are exactly four kinds. The subtle one is the unbound instance reference: String::toUpperCase has no receiver baked in, so the first lambda argument becomes the receiver — s -> s.toUpperCase(). Contrast that with a bound reference like System.out::println, where the receiver (System.out) is fixed when the reference is created.

  • Static: Integer::parseInt ⇒ s -> Integer.parseInt(s).
  • Bound instance (receiver fixed): System.out::println ⇒ x -> System.out.println(x).
  • Unbound instance (first arg is the receiver): String::toUpperCase ⇒ s -> s.toUpperCase().
  • Constructor: ArrayList::new ⇒ () -> new ArrayList<>().
import java.util.*;
import java.util.function.*;

public class MethodRefKinds {
    public static void main(String[] args) {
        // 1. STATIC: Integer::parseInt  ==  s -> Integer.parseInt(s)
        Function<String, Integer> parse = Integer::parseInt;
        System.out.println(parse.apply("42") + 1);

        // 2. BOUND instance: receiver System.out is fixed
        Consumer<String> printer = System.out::println;
        printer.accept("bound to System.out");

        // 3. UNBOUND instance: first arg becomes the receiver
        Function<String, String> upper = String::toUpperCase;
        System.out.println(upper.apply("hello"));

        // 4. CONSTRUCTOR: ArrayList::new  ==  () -> new ArrayList<>()
        Supplier<List<String>> newList = ArrayList::new;
        List<String> list = newList.get();
        list.add("made by ArrayList::new");
        System.out.println(list);
    }
}

02. Refactoring lambdas to method references

Objective: Convert single-call lambdas into method references inside a stream pipeline, and recognise when a lambda must stay a lambda.

The refactor is mechanical: if a lambda is exactly one call that passes its parameters straight through, replace it with a reference. s -> s.toUpperCase() becomes String::toUpperCase; n -> Integer.parseInt(n) becomes Integer::parseInt; x -> System.out.println(x) becomes System.out::println.

The moment the lambda does anything extra — a second call, an operator, a reordered or literal argument — a method reference can no longer express it. s -> s.trim().toUpperCase() is two calls, so it stays a lambda.

But "stays a lambda" only means you cannot reference a pre-existing method that already bundles those calls. If the logic is your own, pull it into a named method and reference that: s -> s.getRankLevel() >= 3 has no built-in method to point at, yet becomes MyClass::seniorEnough once you add static boolean seniorEnough(Student s){ return s.getRankLevel() >= 3; }.

  • Refactor only when the lambda is a single call whose arguments are passed through unchanged.
  • Chained calls (s.trim().toUpperCase()), arithmetic, or fixed literals block the conversion.
  • Method references read best in pipeline stages: .map(String::toUpperCase), .forEach(System.out::println).
  • Custom logic isn't a dead end: extract the body into a named method, then reference it — a lambda that can't map to an existing method often can map to one you write.
import java.util.*;
import java.util.stream.*;

public class RefactorToMethodRef {
    public static void main(String[] args) {
        List<String> names = Arrays.asList("ada", "grace", "linus");

        // lambda  s -> s.toUpperCase()  ==>  String::toUpperCase
        List<String> shouting = names.stream()
                .map(String::toUpperCase)
                .collect(Collectors.toList());
        System.out.println(shouting);

        // lambda  n -> Integer.parseInt(n)  ==>  Integer::parseInt
        int total = Stream.of("10", "20", "30")
                .mapToInt(Integer::parseInt)
                .sum();
        System.out.println("total = " + total);

        // KEEP the lambda: two calls, no single method to reference
        List<String> messy = Arrays.asList(" ada ", " grace ");
        List<String> cleaned = messy.stream()
                .map(s -> s.trim().toUpperCase())
                .collect(Collectors.toList());
        System.out.println(cleaned);

        // bound reference for a side effect: x -> System.out.println(x)
        shouting.forEach(System.out::println);
    }
}
Console output[ADA, GRACE, LINUS] total = 60 [ADA, GRACE] ADA GRACE LINUS

03. Constructor references

Objective: Use Type::new to supply objects to functional interfaces and stream pipelines, including array constructors.

A constructor reference is Type::new. Which constructor it targets is decided by the functional interface it is assigned to: a no-arg constructor fits a Supplier, a one-arg constructor fits a Function, and so on. Arrays get their own form — int[]::new is n -> new int[n].

Constructor references shine in map steps: mapping a stream of names through Person::new produces a stream of Person objects with no boilerplate lambda.

  • Supplier<List<String>> s = ArrayList::new; — no-arg constructor via s.get().
  • Function<String, StringBuilder> f = StringBuilder::new; — one-arg constructor via f.apply(x).
  • IntFunction<int[]> a = int[]::new; — array constructor via a.apply(len).
  • The assigned functional-interface type selects the matching constructor arity.
import java.util.*;
import java.util.function.*;
import java.util.stream.*;

public class ConstructorRefs {
    // records are final in Java 16; the ctor reference itself is Java 8
    record Person(String name) {}

    public static void main(String[] args) {
        // no-arg constructor -> Supplier
        Supplier<List<String>> listMaker = ArrayList::new;
        List<String> list = listMaker.get();
        list.add("a");
        System.out.println(list);

        // one-arg constructor -> Function
        Function<String, StringBuilder> toBuilder = StringBuilder::new;
        System.out.println(toBuilder.apply("hi").reverse());

        // array constructor -> IntFunction
        IntFunction<int[]> arrayMaker = int[]::new;
        int[] arr = arrayMaker.apply(3);
        System.out.println(arr.length);

        // map names to a record via a constructor reference
        List<Person> people = Stream.of("Ada", "Grace")
                .map(Person::new)          // name -> new Person(name)
                .collect(Collectors.toList());
        System.out.println(people);
    }
}

04. Lambdas & effectively-final variables

Objective: Explain why lambdas may capture only final or effectively-final local variables, and why fields are exempt.

A lambda (like an anonymous class) may capture a local variable only if that variable is final or effectively final — assigned exactly once and never reassigned. A lambda can outlive the method call that created it: it may be stored, returned, or handed to another thread. Java therefore captures the value of the local, not the variable slot. If you could reassign the local afterwards, the captured copy and the live variable would silently disagree, so the language forbids it outright.

Instance and static fields are not restricted. A lambda that reads a field captures this (or the class) and reads the field live, so the field may be freely reassigned before or after the lambda is created.

  • Captured locals must be assigned once — effectively final (you needn't write the final keyword since Java 8).
  • Reassigning a captured local ⇒ compile error: local variables referenced from a lambda expression must be final or effectively final.
  • Fields escape the rule: they are read live through this/the class and may change.
  • The identical rule applies to anonymous inner classes.
  • Distinct rule: a lambda parameter — or a local declared inside the lambda body — may not reuse the name of a local variable of the enclosing method. int i = 0; Consumer<Integer> c = i -> System.out.println(i); fails to compile with variable i is already defined in method main. (A lambda adds no new naming scope; an anonymous class does, so it could shadow.)
import java.util.function.Function;

public class EffectivelyFinal {
    static int counter = 0;               // a FIELD, not a local

    public static void main(String[] args) {
        int factor = 10;                  // assigned once -> effectively final
        Function<Integer, Integer> scale = n -> n * factor;   // OK
        System.out.println(scale.apply(5));

        counter = 1;                      // reassigning a FIELD is allowed...
        Runnable r = () -> System.out.println("counter = " + counter);
        counter = 2;                      // ...even after the lambda exists
        r.run();                          // reads the field LIVE
    }
}

5. Streams — Fundamentals

The Streams API (Java 8) lets you express computations over a sequence of elements as a declarative pipeline instead of a hand-written loop. This section covers what a stream really is, how a pipeline executes lazily, how streams differ from collections, and how to watch elements flow with peek.

01. Introduction to the Streams API

Objective: Understand what a Stream<T> is — and, just as importantly, what it is not — and learn the three parts every pipeline is built from.

A Stream<T> (introduced in Java 8, package java.util.stream) is a sequence of elements that supports aggregate operations such as filtering, mapping and reducing. Crucially, a stream is not a data structure: it stores no elements of its own. Instead it describes a computation over an external source — a collection, an array, a file, or a generator function.

Every stream program has the same shape: a source produces the elements, zero or more intermediate operations transform them, and exactly one terminal operation consumes them to produce a result or a side effect.

  • Stream<T> lives in java.util.stream and arrived in Java 8, alongside lambdas.
  • A stream holds no data — it only records what to compute over a source.
  • Common sources: collection.stream(), Arrays.stream(array), Stream.of(...), generators like Stream.iterate/generate.
  • Pipeline anatomy: source → intermediate ops → terminal op (exactly one terminal).
Version The whole Streams API — Stream, Collectors, and the primitive variants IntStream/LongStream/DoubleStream — is part of Java 8 (2014).
import java.util.List;

public class StreamIntro {
    public static void main(String[] args) {
        List<String> names =
            List.of("Ann", "Bob", "Charlie", "Dan", "Elena");

        long longNames = names.stream()   // source
            .filter(n -> n.length() > 3)  // intermediate op
            .count();                     // terminal op

        System.out.println("names longer than 3: " + longNames);
    }
}

AdvancedA stream can be sequential (the default from .stream()) or parallel (.parallelStream() or .parallel()), and the same pipeline code works either way. Streams also come in primitive-specialised flavours — IntStream, LongStream, DoubleStream — that avoid boxing for numeric work.
Pitfall A java.util.stream.Stream has nothing to do with the I/O InputStream/OutputStream classes — same word, unrelated concept. And because a stream is not a collection, it has no size() and no random access; you cannot index into it.

02. How a stream pipeline works

Objective: Learn why intermediate operations are lazy, why the terminal operation is what actually runs the pipeline, and what "internal iteration" means.

Intermediate operations (map, filter, sorted, distinct, peek, …) are lazy: each returns a new Stream and does no work — it merely records the step to perform later. The terminal operation (forEach, collect, reduce, count, toList [Java 16+], …) is what triggers execution and yields a result or a side effect. With no terminal operation, nothing is processed at all.

Execution uses internal iteration: you describe what should happen and the library controls how it loops. The steps are fused into a single pass, so each element is typically pulled through the entire chain before the next element starts — not stage-by-stage across the whole collection.

  • Intermediate ops are lazy and return a new Stream; they do nothing on their own.
  • Exactly one terminal op ends the pipeline and drives the computation.
  • No terminal op ⇒ no elements processed.
  • Internal iteration: the Stream owns the loop, not your code.
  • Elements flow one at a time through the fused chain.
import java.util.stream.Stream;

Stream<String> pipeline = Stream.of("a", "b", "c")
    .filter(s -> {
        System.out.println("filter sees " + s);
        return true;
    });

System.out.println("pipeline built - nothing has run yet");
pipeline.forEach(s -> System.out.println("forEach got " + s));
Console outputpipeline built - nothing has run yet filter sees a forEach got a filter sees b forEach got b filter sees c forEach got c

Source: reimplemented as an original illustration.

AdvancedBecause the steps fuse into one pass, short-circuiting terminal ops such as findFirst, anyMatch and the intermediate limit can stop early without touching every element. That is exactly why laziness matters: it lets an infinite source such as Stream.iterate(1, n -> n + 1).limit(5) terminate.
Pitfall Forgetting the terminal operation. list.stream().map(String::toUpperCase); compiles and runs but does nothing — the mapping never happens. Add a terminal op (.forEach(...), .collect(...), or .toList() [Java 16+]) to actually consume the pipeline.

03. Streams vs Collections

Objective: Contrast collections (which store data) with streams (which compute over data), and understand why a stream can be used only once.

A Collection is a stored, owned set of elements sitting in memory — you can add to it, remove from it, and iterate it as many times as you like. A Stream owns nothing: it is a one-shot recipe for computing a result. It is evaluated lazily, iterated internally, and — the key gotcha — consumed exactly once.

Aspect Collection Stream
Role Stores & owns its elements Computes over elements — stores nothing
Mutability Mutable — add / remove / set elements No add / remove / set — elements cannot be changed once the stream is created
Evaluation Eager — all elements already in memory Lazy — elements produced on demand
Iteration External — you write the loop Internal — the library loops for you

  • Collections are about storage; streams are about computation.
  • Collections use external iteration and are reusable; streams use internal iteration and are single-use.
  • Reusing a stream after a terminal (or intermediate) op throws IllegalStateException.
  • Need to run the same pipeline twice? Re-create the stream from the source each time.
import java.util.stream.Stream;

AdvancedInternal iteration is what makes parallelism nearly free: switching .stream() to .parallelStream() lets the library split the work across a fork/join pool without you touching a single thread. With external for loops, you would have to partition and coordinate the threads yourself.
Pitfall Caching a stream in a variable and "reusing" it. Storing Stream<T> s = list.stream(); and calling two terminal ops on s throws "stream has already been operated upon or closed". Streams are single-use — build a fresh one from the collection whenever you need to run the pipeline again.

04. Debugging stream pipelines with peek

Objective: Use peek to observe elements as they flow through a pipeline, and understand why its output interleaves — one element traverses the whole chain before the next.

peek(Consumer) (Java 8) is an intermediate operation that lets you look at each element as it passes without changing it — it returns the same stream unaltered. Its intended use is debugging: drop a peek between two stages to log what is flowing through. Because the pipeline processes elements one at a time, the log lines from different stages interleave per element rather than appearing stage-by-stage.

  • peek(Consumer<T>) observes each element and passes it on unchanged.
  • Great for inspecting values between stages while debugging.
  • Output is interleaved: each element runs through the entire pipeline before the next begins.
  • peek is lazy — with no terminal op it never executes.
  • Don't use it for real side effects; the JVM may skip it.
import java.util.stream.Stream;

Stream.of("a", "bb", "ccc")
    .filter(s -> s.length() > 1)
    .peek(s -> System.out.println("after filter: " + s))
    .map(String::toUpperCase)
    .forEach(s -> System.out.println("result: " + s));
Console outputafter filter: bb result: BB after filter: ccc result: CCC

Source: reimplemented as an original illustration.

Trace it element by element: "a" has length 1, fails the filter, and is dropped before peek ever sees it. "bb" passes the filter → peek prints after filter: bb → map makes "BB" → forEach prints result: BB. Only then does "ccc" begin the same journey. That is why the lines interleave vertically instead of printing all the "after filter" lines first.

AdvancedSince Java 9, if a terminal op can produce its answer without traversing elements, peek may not run at all. For example Stream.of("a","bb","ccc").peek(System.out::println).count() can return 3 and print nothing, because the count is known from the source's size and no size-changing op (like filter) is present. This is the JLS/API license that makes peek safe only as a debugging aid.
Pitfall Using peek to do real work, e.g. list.stream().peek(results::add).count(); — the added elements may never appear because count() can short-circuit the traversal. And a peek with no terminal op runs nothing. Use map/forEach/collect for side effects and results.

6. Stream Operations

The Streams API ships a small, composable vocabulary of operations: intermediate ones that reshape the pipeline (map, filter, sorted, limit…) and terminal ones that produce a result (reduce, count, max, anyMatch…). This section walks each operator with a fixed pair of sample lists so the examples read together.

Context Everything here is part of the Streams API, standard since Java 8 (2014). Examples assume import java.util.*; and import java.util.stream.*;, and reuse two lists throughout: numbers = List.of(3, 1, 4, 1, 5, 9, 2, 6) and words = List.of("apple", "banana", "cherry", "avocado").

01. map & flatMap

Objective: Distinguish the one-to-one transform (map) from the one-to-many-then-flatten transform (flatMap).

map applies a Function to every element and emits exactly one output per input — the element type may change (a String becomes its Integer length). The stream length is unchanged.

flatMap maps each element to a stream, then concatenates those sub-streams into a single flat stream. Use it to collapse nested structures (a List of Lists, or each word split into its letters) into one flat stream. Reach for flatMap whenever your mapping function would otherwise produce a stream-of-streams.

  • map: N in → N out, one-to-one; may change element type.
  • flatMap: each element → a stream; results are flattened into one stream.
  • If your map lambda returns a collection or array, you probably want flatMap.
List<String> words = List.of("apple", "banana", "cherry", "avocado");

// map: 1-to-1 transform (element type may change, String to Integer)
List<Integer> lengths = words.stream()
        .map(String::length)
        .collect(Collectors.toList());
System.out.println(lengths);

// flatMap: each element becomes a stream; the streams are flattened
List<List<Integer>> nested = List.of(
        List.of(1, 2),
        List.of(3, 4),
        List.of(5));
List<Integer> flat = nested.stream()
        .flatMap(List::stream)
        .collect(Collectors.toList());
System.out.println(flat);
Console output[5, 6, 6, 7] [1, 2, 3, 4, 5]

Source: reimplemented as an original illustration.

AdvancedTo turn each word into a stream of its letters, use words.stream().map(w -> w.split("")).flatMap(Arrays::stream) — map yields a Stream<String[]>, and flatMap(Arrays::stream) flattens every array into one continuous stream of single-character strings. For primitives there are specialised variants (flatMapToInt, etc.).
Pitfall Using map where you need flatMap leaves you with a Stream<List<X>> (a stream of collections) instead of a Stream<X>. The compiler accepts it, but the downstream operations then run on whole lists, not elements.

02. filter, distinct, count & sorted

Objective: Select, de-duplicate, count and order elements with the four workhorse operations.

filter(Predicate) keeps only the elements for which the predicate is true. distinct() drops duplicates using equals() (and hashCode()) — our list contains the value 1 twice, so one copy is removed. count() is a terminal operation returning a long. sorted() orders elements by their natural ordering (Comparable).

  • filter and sorted are intermediate (lazy); count is terminal (eager).
  • distinct() compares with equals(); it preserves first-encounter order.
  • count() returns long, not int — streams can exceed 2 billion elements.
List<Integer> numbers = List.of(3, 1, 4, 1, 5, 9, 2, 6);

System.out.println(numbers.stream()
        .filter(n -> n % 2 == 0)          // keep evens
        .collect(Collectors.toList()));

System.out.println(numbers.stream()
        .distinct()                       // drop the duplicate 1
        .collect(Collectors.toList()));

long total = numbers.stream().count();    // terminal, returns long
System.out.println("count = " + total);

System.out.println(numbers.stream()
        .sorted()                         // natural ascending order
        .collect(Collectors.toList()));
Console output[4, 2, 6] [3, 1, 4, 5, 9, 2, 6] count = 8 [1, 1, 2, 3, 4, 5, 6, 9]

Source: reimplemented as an original illustration.

AdvancedFor your own classes, distinct() only works as expected if you override equals/hashCode (records do this for you). To de-duplicate by a derived key instead, there is no built-in operator — collect into a Map keyed by that field, or filter with a seen-set.
Pitfall count() consumes the stream, so you cannot reuse the same stream afterwards — a stream is single-use. Calling another terminal on it throws IllegalStateException: stream has already been operated upon or closed.

03. Custom sorting with Comparator

Objective: Order a stream by a derived key, break ties with a secondary key, and reverse the whole comparator.

sorted(Comparator) takes an explicit ordering. Build one fluently with Comparator.comparing(keyExtractor), chain a tie-breaker with thenComparing(…), and flip the direction with reversed(). Below we sort words by length, breaking length ties alphabetically. Because banana and cherry are both length 6, the alphabetical tie-breaker keeps banana first.

  • Comparator.comparing(String::length) orders by the extracted key.
  • thenComparing(Comparator.naturalOrder()) breaks ties on equal keys.
  • reversed() inverts the entire composed comparator (both keys).
List<String> words = List.of("apple", "banana", "cherry", "avocado");

// sort by length; ties broken alphabetically
Comparator<String> byLenThenAlpha =
        Comparator.comparing(String::length)
                  .thenComparing(Comparator.naturalOrder());

System.out.println(words.stream()
        .sorted(byLenThenAlpha)
        .collect(Collectors.toList()));

System.out.println(words.stream()
        .sorted(byLenThenAlpha.reversed())   // longest first, then reverse-alpha
        .collect(Collectors.toList()));
Console output[apple, banana, cherry, avocado] [avocado, cherry, banana, apple]

Source: reimplemented as an original illustration.

AdvancedPrefer Comparator.comparingInt(String::length) for primitive keys — it avoids boxing each length into an Integer. Note that reversed() reverses the composed comparator as a whole, so ties that were alphabetical ascending become alphabetical descending; to reverse only the primary key, negate that key's comparator before chaining.
Pitfall Comparator.comparing(String::length).reversed().thenComparing(...) does not do what most expect: reversed() flips whatever precedes it, so put your tie-breakers before reversed() or reverse individual keys instead.

04. reduce & the map-filter-reduce pattern

Objective: Fold a stream to a single value with reduce, and assemble the canonical map–filter–reduce pipeline.

reduce(identity, accumulator) starts from an identity value and repeatedly combines it with each element using an associative BinaryOperator. Summing is reduce(0, Integer::sum). The classic functional pipeline — reshape (map), select (filter), then fold (reduce) — expresses aggregations declaratively. Here we sum the squares of the even numbers: the evens 4, 2, 6 become 16, 4, 36, which fold to 56.

  • reduce(identity, acc) returns a plain value (never empty — identity is the seed).
  • The accumulator must be associative so parallel reduction gives the same result.
  • filter–map–reduce is the declarative counterpart of a hand-written accumulation loop.
List<Integer> numbers = List.of(3, 1, 4, 1, 5, 9, 2, 6);

int sum = numbers.stream()
        .reduce(0, Integer::sum);            // 0+3+1+4+1+5+9+2+6
System.out.println("sum = " + sum);

// the canonical filter -> map -> reduce pipeline (sum of squares of evens)
int sumSqEven = numbers.stream()
        .filter(n -> n % 2 == 0)             // 4, 2, 6
        .map(n -> n * n)                     // 16, 4, 36
        .reduce(0, Integer::sum);            // 16 + 4 + 36
System.out.println("sum of squares of evens = " + sumSqEven);
Console outputsum = 31 sum of squares of evens = 56

Source: reimplemented as an original illustration.

AdvancedThere is a three-argument reduce(identity, accumulator, combiner) for when the result type differs from the element type (e.g. reducing Stream<String> into an int total length). The combiner merges partial results and is only actually invoked for parallel streams — but it must still be correct and associative.
Pitfall Passing a non-associative or non-identity-respecting function (e.g. subtraction, or identity 1 with addition) yields correct-looking sequential results but wrong answers when the stream is parallelised. Keep the identity truly neutral and the operator associative.

05. max & min with reduce

Objective: Find extremes both via a bare reduce and via the idiomatic max/min, understanding why the result is an Optional.

The single-argument reduce(accumulator) has no identity, so it returns an Optional<T> — an empty stream has no maximum. reduce((a, b) -> a > b ? a : b) keeps the larger of each pair. The idiomatic form is max(Comparator) / min(Comparator), which also return Optional. For our list the maximum is 9 and the minimum is 1.

  • Identity-less reduce, max, and min all return Optional<T> (empty-safe).
  • max/min take a Comparator; use Comparator.naturalOrder() for Comparable elements.
  • Printing an Optional shows Optional[value] (or Optional.empty).
List<Integer> numbers = List.of(3, 1, 4, 1, 5, 9, 2, 6);

        .reduce((a, b) -> a > b ? a : b);

        .reduce((a, b) -> a < b ? a : b);
System.out.println("min (reduce) = " + minR);

06. limit & skip

Objective: Truncate a stream with limit, drop a prefix with skip, and combine them for pagination.

limit(n) keeps only the first n elements in encounter order and is short-circuiting — it stops pulling from the source once it has n. skip(n) discards the first n and passes the rest through. Chaining skip(2).limit(3) yields a “page” of three elements starting at index 2.

  • limit(n): first n elements; short-circuits the pipeline.
  • skip(n): drop the first n, keep the remainder.
  • skip(page*size).limit(size) is the standard stream-pagination idiom.
List<Integer> numbers = List.of(3, 1, 4, 1, 5, 9, 2, 6);

System.out.println(numbers.stream()
        .limit(3)                 // first 3 elements
        .collect(Collectors.toList()));

System.out.println(numbers.stream()
        .skip(3)                  // drop the first 3
        .collect(Collectors.toList()));

// pagination: skip past earlier elements, then take a page
System.out.println(numbers.stream()
        .skip(2).limit(3)         // indexes 2, 3, 4
        .collect(Collectors.toList()));
Console output[3, 1, 4] [1, 5, 9, 2, 6] [4, 1, 5]

Source: reimplemented as an original illustration.

Advancedlimit is the trick that makes infinite streams usable: Stream.iterate(1, n -> n + 1).limit(5) produces [1, 2, 3, 4, 5]. Java 9 added the more expressive takeWhile(predicate) and dropWhile(predicate), which cut based on a condition rather than a fixed count.
Pitfall On an ordered parallel stream, limit/skip must preserve encounter order, which forces coordination and can be surprisingly expensive. If order does not matter, call unordered() first to let them run cheaply.

07. Matching & finding

Objective: Answer boolean questions with the match operations and grab an element with the find operations.

allMatch, anyMatch and noneMatch take a Predicate, return a boolean, and short-circuit as soon as the answer is decided. findFirst() returns an Optional holding the first element in encounter order; findAny() may return any element (useful for parallel streams). For our list every element is positive, one element exceeds 8, and none exceeds 10; the first element greater than 4 is 5.

  • allMatch / anyMatch / noneMatch return boolean and short-circuit.
  • findFirst() respects encounter order; findAny() does not (parallel-friendly).
  • Both find* operations return an Optional — the stream may match nothing.
List<Integer> numbers = List.of(3, 1, 4, 1, 5, 9, 2, 6);

System.out.println(numbers.stream().allMatch(n -> n > 0));    // all positive?
System.out.println(numbers.stream().anyMatch(n -> n > 8));    // any above 8?
System.out.println(numbers.stream().noneMatch(n -> n > 10));  // none above 10?

Optional<Integer> first = numbers.stream()
        .filter(n -> n > 4)
        .findFirst();                                          // first match, in order
System.out.println("first > 4 = " + first);
Console outputtrue true true first > 4 = Optional[5]

AdvancedOn an empty stream the vacuous-truth rules bite: allMatch and noneMatch both return true, while anyMatch returns false. So “all elements satisfy P” is trivially true when there are no elements — guard for emptiness if that matters.
Pitfall findAny() is not guaranteed to return the first element even on a sequential stream (it merely tends to). If you depend on encounter order, use findFirst().

08. Short-circuiting, stateless vs stateful

Objective: Classify operations by whether they can stop early and by how much of the stream they must observe — and see why sorted cannot run on an infinite stream while filter can.

Short-circuiting operations (limit, anyMatch/allMatch/noneMatch, findFirst/findAny) can produce their result without consuming the whole stream. Stateless operations (map, filter, peek) process each element independently, so they stream one element at a time. Stateful operations may need to see other elements: distinct remembers everything seen so far, and sorted must buffer the entire stream before it can emit even its first (smallest) element. That is exactly why sorted never terminates on an infinite stream, whereas a stateless filter followed by a short-circuiting limit works fine.

The peek trace below shows short-circuiting concretely: anyMatch pulls elements lazily and stops the instant one passes, so only three of the eight numbers are ever touched.

  • Stateless: map, filter, peek — element handled in isolation.
  • Stateful: distinct, sorted, limit, skip — may buffer or track prior elements.
  • Short-circuiting: limit, anyMatch/allMatch/noneMatch, findFirst/findAny.
  • sorted buffers everything → it breaks on infinite streams; filter does not.
List<Integer> numbers = List.of(3, 1, 4, 1, 5, 9, 2, 6);

// anyMatch short-circuits: it stops at the first element that passes
boolean any = numbers.stream()
        .peek(n -> System.out.println("testing " + n))
        .anyMatch(n -> n > 3);
System.out.println("any > 3 = " + any);
Console outputtesting 3 testing 1 testing 4 any > 3 = true

Source: reimplemented as an original illustration.

AdvancedA stateless + short-circuiting pipeline tames an infinite source: Stream.iterate(1, n -> n + 1).filter(n -> n % 2 == 0).limit(3).collect(Collectors.toList()) yields [2, 4, 6] and terminates. Insert a .sorted() before limit and it hangs forever — sorted is stateful and demands the whole (never-ending) stream before emitting anything.
Pitfall Because peek is lazy and only fires for elements the terminal actually pulls, using it as a “log every element” tool under a short-circuiting terminal (or with no terminal at all) will silently log fewer elements than you expect — it is a debugging aid, not a guaranteed side-effect for-each.

7. Creating & Numeric Streams

So far streams have come from collections, but the Streams API (Java 8) can also manufacture them from scratch — from literal values, from a growth rule, or from a supplier. This section covers those factory methods, the primitive IntStream/LongStream/DoubleStream variants that avoid boxing, and how to move data between the object and primitive worlds.

01. Factory methods — of, iterate, generate

Objective: Build streams without a source collection using Stream.of, Stream.iterate, and Stream.generate, and learn to bound the infinite ones.

Stream.of(...) takes varargs and produces a small, finite stream — handy for literals and tests. Stream.iterate(seed, next) applies a unary function repeatedly (seed, f(seed), f(f(seed)), …); it is an infinite stream, so it must be capped with limit(n) or it will run forever. Stream.generate(supplier) calls a Supplier on demand and is likewise infinite.

Java 9 added a third, bounded form: Stream.iterate(seed, hasNext, next). The middle predicate acts like the condition of a for loop, so the stream stops on its own — no limit needed. Think of it as a functional for (int n = 1; n <= 16; n *= 2).

  • Stream.of(a, b, c) — finite varargs stream (Java 8).
  • Stream.iterate(seed, next) and Stream.generate(supplier) are infinite — always follow with limit.
  • The 3-arg Stream.iterate(seed, hasNext, next) is self-bounding (Java 9+).
  • These streams are lazy: nothing is computed until a terminal operation pulls elements.
import java.util.stream.Stream;

public class Factories {
    public static void main(String[] args) {
        // 1. Stream.of — finite varargs (Java 8)
        Stream.of("a", "b", "c")
              .forEach(System.out::println);

        // 2. Stream.iterate(seed, next) — INFINITE -> limit!
        Stream.iterate(1, n -> n * 2)
              .limit(5)
              .forEach(System.out::println);

        // 3. Stream.iterate(seed, hasNext, next) — bounded (Java 9)
        Stream.iterate(1, n -> n <= 16, n -> n * 2)
              .forEach(System.out::println);

        // 4. Stream.generate(Supplier) — INFINITE -> limit!
        Stream.generate(() -> "x")
              .limit(3)
              .forEach(System.out::println);
    }
}
Console outputa b c 1 2 4 8 16 1 2 4 8 16 x x x

Source: reimplemented as an original illustration.

AdvancedStream.iterate works with any type, not just numbers — Stream.iterate(LocalDate.now(), d -> d.plusDays(1)).limit(7) yields the next seven dates. Because generation is lazy, an infinite iterate/generate is fine as long as a short-circuiting operation (limit, findFirst, anyMatch) eventually stops the pull.
Pitfall Calling a terminal operation on Stream.iterate(1, n -> n * 2) or Stream.generate(...) without a limit (or other short-circuit) hangs the program — it tries to produce endlessly. Always bound infinite streams. The 3-arg iterate (Java 9+) is the exception: its predicate provides the stop condition.

02. Numeric streams — range & rangeClosed

Objective: Generate streams of consecutive integers with IntStream.range and rangeClosed, and understand the exclusive-vs-inclusive upper bound.

The Streams API (Java 8) ships three primitive specializations — IntStream, LongStream, and DoubleStream — that hold int/long/double directly instead of boxed Integer objects. For counting loops they expose two factories: IntStream.range(start, end) is end-exclusive (stops before end), while IntStream.rangeClosed(start, end) is inclusive (includes end).

The mnemonic: range mirrors the classic for (int i = start; i < end; i++), whereas rangeClosed mirrors i <= end. So range(1, 5) gives four elements and rangeClosed(1, 5) gives five.

  • IntStream.range(1, 5) → 1 2 3 4 (upper bound excluded).
  • IntStream.rangeClosed(1, 5) → 1 2 3 4 5 (upper bound included).
  • Primitive streams avoid autoboxing — cheaper for large numeric ranges.
  • LongStream/DoubleStream have the same shape; DoubleStream has no range factories — build one with IntStream.rangeClosed(1, 5).asDoubleStream() (or .asLongStream() for a LongStream; both are Java 8 primitive-stream conversions).
import java.util.stream.IntStream;

public class Ranges {
    public static void main(String[] args) {
        System.out.print("range(1, 5):       ");
        IntStream.range(1, 5)
                 .forEach(i -> System.out.print(i + " "));
        System.out.println();

        System.out.print("rangeClosed(1, 5): ");
        IntStream.rangeClosed(1, 5)
                 .forEach(i -> System.out.print(i + " "));
        System.out.println();

        System.out.println("range count:       " + IntStream.range(1, 5).count());
        System.out.println("rangeClosed count: " + IntStream.rangeClosed(1, 5).count());
    }
}

AdvancedNeither factory throws when start > end — both just return an empty stream. When start == end, range is empty while rangeClosed has one element. To iterate a List by index the idiomatic form is IntStream.range(0, list.size()), which pairs each index with list.get(i) — a common bridge between index-based and stream-based code.

03. Numeric aggregates — sum, max, min, average

Objective: Use the reduction shortcuts that exist only on primitive streams — sum, max, min, average, and summaryStatistics.

Because IntStream/LongStream/DoubleStream know their element is numeric, they offer aggregate operations the object Stream does not. sum() returns a plain primitive (int, long, or double) — an empty stream simply sums to 0. But max(), min(), and average() return an OptionalInt/OptionalLong/OptionalDouble, because an empty stream has no maximum or minimum to report.

When you need several of these at once, summaryStatistics() computes count, sum, min, average, and max in a single pass and returns an IntSummaryStatistics (or the long/double sibling) — cheaper than re-streaming for each figure.

  • sum() → primitive; empty stream sums to 0.
  • max()/min()/average() → OptionalInt/OptionalLong/OptionalDouble (empty stream has none).
  • average() always returns an OptionalDouble, even for an IntStream.
  • summaryStatistics() gathers count+sum+min+average+max in one traversal.
import java.util.stream.IntStream;

public class Aggregates {
    public static void main(String[] args) {
        System.out.println("sum     = " + IntStream.rangeClosed(1, 5).sum());
        System.out.println("max     = " + IntStream.rangeClosed(1, 5).max());
        System.out.println("min     = " + IntStream.rangeClosed(1, 5).min());
        System.out.println("average = " + IntStream.rangeClosed(1, 5).average());

        var stats = IntStream.rangeClosed(1, 5).summaryStatistics();
        System.out.println(stats);
    }
}
Console outputsum = 15 max = OptionalInt[5] min = OptionalInt[1] average = OptionalDouble[3.0] IntSummaryStatistics{count=5, sum=15, min=1, average=3.000000, max=5}

AdvancedTo unwrap the optionals, use their primitive accessors: getAsInt(), getAsDouble(), or orElse(fallback). For example IntStream.rangeClosed(1, 5).average().orElse(0.0) yields 3.0. On an object stream you would instead reach for collect(Collectors.summarizingInt(...)) to get the same statistics object.
Pitfall Calling .getAsInt() on the result of max() for a stream that turned out empty throws NoSuchElementException. Guard with isPresent() or supply a default via orElse(...).

04. Boxing & mapping — mapToInt, mapToObj…

Objective: Move between object streams and primitive streams with mapToInt, boxed, and mapToObj to unlock numeric aggregates and avoid boxing overhead.

An object Stream<T> and a primitive IntStream are different types, so you convert explicitly. mapToInt(ToIntFunction) turns a Stream<String> into an IntStream (e.g. mapToInt(String::length)), which then exposes sum, average, and friends. Going the other way, boxed() lifts an IntStream back to a Stream<Integer> so you can collect into a List<Integer>, and mapToObj(IntFunction) maps each primitive to any object (mapToObj(i -> "#" + i) → Stream<String>).

There are parallel mapToLong and mapToDouble for the other primitive streams. The payoff is twofold: you gain the numeric aggregate operations, and you avoid the per-element autoboxing cost of carrying Integer objects through the pipeline.

  • mapToInt/mapToLong/mapToDouble: object stream → primitive stream.
  • boxed(): IntStream → Stream<Integer> (needed to collect into a List).
  • mapToObj(fn): primitive stream → Stream<R> of any object type.
  • Motivation: unlock numeric aggregates and skip autoboxing overhead.
import java.util.List;
import java.util.stream.Collectors;
import java.util.stream.IntStream;

public class BoxingMapping {
    public static void main(String[] args) {
        List<String> words = List.of("Java", "is", "modern");

        // Stream<String> -> IntStream : unlocks sum(), no boxing
        int totalChars = words.stream()
                              .mapToInt(String::length)
                              .sum();
        System.out.println("total chars = " + totalChars);

        // IntStream -> Stream<Integer> via boxed(), then collect
        List<Integer> lengths = words.stream()
                                     .mapToInt(String::length)
                                     .boxed()
                                     .collect(Collectors.toList());
        System.out.println("lengths     = " + lengths);

        // IntStream -> Stream<String> via mapToObj
        String labels = IntStream.rangeClosed(1, 3)
                                 .mapToObj(i -> "#" + i)
                                 .collect(Collectors.joining(", "));
        System.out.println("labels      = " + labels);
    }
}
Console outputtotal chars = 12 lengths = [4, 2, 6] labels = #1, #2, #3

Note the asymmetry: an object stream has mapToInt but an IntStream uses boxed() (not mapToObj(Integer::valueOf), though that works too) to return to the object world.

8. Collectors & Terminal Operations

A stream pipeline ends in a terminal operation. The most versatile one is collect(Collector) — a mutable reduction that accumulates elements into a container. The java.util.stream.Collectors factory (all Java 8) supplies ready-made recipes for lists, sets, strings, counts, sums, and — most powerfully — groupingBy and partitioningBy.

Version note Everything in this section — the Streams API, Optional, and every Collectors method shown — is Java 8 (2014). The sample data uses a record (final in Java 16) purely for brevity; a plain class with the same fields would behave identically. A couple of finishing collectors noted as "Advanced" arrived later (toUnmodifiableList in Java 10, teeing in Java 12).

01. collect & Collectors — joining, counting

Objective: Use collect(Collectors...), the general-purpose terminal reduction, to gather a stream into a List/Set, join elements into one String, and count them.

Where reduce folds elements into a single immutable value, collect performs a mutable reduction: it fills a result container (a list, a set, a StringBuilder) as elements flow past. You almost never build a Collector by hand — you pass a pre-built one from the Collectors factory.

toList() and toSet() gather elements; joining(delimiter, prefix, suffix) concatenates CharSequence elements; counting() returns the element count as a Long.

  • collect is the most general terminal op — most other terminals are special cases of it.
  • Collectors.toList() currently returns a mutable ArrayList in practice, but its type/mutability are not guaranteed — treat it as read-mostly (see Pitfall below); toSet() a HashSet (no defined order).
  • joining has three forms: no-arg, delimiter only, and delimiter + prefix + suffix.
  • counting() is handy as a downstream collector (see groupingBy).
import java.util.*;
import java.util.stream.*;

record Person(String name, String city, int age) {}   // record: final in Java 16

List<Person> people = List.of(
    new Person("Alice", "NYC", 30),
    new Person("Bob",   "LA",  25),
    new Person("Carol", "NYC", 35),
    new Person("Dave",  "LA",  28),
    new Person("Eve",   "NYC", 22));

// collect() = the general mutable-reduction terminal op
List<String> names  = people.stream().map(Person::name)
                             .collect(Collectors.toList());
Set<String>  cities = people.stream().map(Person::city)
                             .collect(Collectors.toSet());
String joined = people.stream().map(Person::name)
                      .collect(Collectors.joining(", ", "[", "]"));
long count = people.stream().collect(Collectors.counting());

System.out.println(names);
System.out.println(cities);
System.out.println(joined);
System.out.println(count);
Console output[Alice, Bob, Carol, Dave, Eve] [NYC, LA] (HashSet: order may vary) [Alice, Bob, Carol, Dave, Eve] 5

AdvancedNeed a specific container or immutability? Use toCollection(TreeSet::new) for a sorted set, toUnmodifiableList() / toUnmodifiableSet() (Java 10) for read-only results, or toMap(keyFn, valueFn) to build a Map. Java 12 added teeing(down1, down2, merger), which feeds each element to two collectors and merges their results in a single pass.
Pitfall Collectors.toList() makes no promise about the list's mutability or type — treat it as read-mostly. 02. mapping, minBy, maxBy, summing & averaging

Objective: Reduce a stream to a single derived value with collectors that transform, pick extremes, and aggregate numbers — the same collectors you will later reuse as downstream collectors inside groupingBy.

mapping(mapper, downstream) applies a function to each element and then feeds the results to another collector — here, extract names and gather them into a list. minBy(cmp) and maxBy(cmp) pick the extreme element and return an Optional<Person> (empty if the stream is empty). summingInt and averagingInt aggregate a numeric property in one pass, returning an Integer and a Double respectively.

  • mapping adapts a downstream collector to a transformed element type.
  • minBy / maxBy return Optional<T> — safe on empty streams.
  • summingInt/summingLong/summingDouble and their averaging* siblings aggregate numerics.
  • averagingInt always returns a double — the average of int ages here is 28.0.
// people: Alice/NYC/30, Bob/LA/25, Carol/NYC/35, Dave/LA/28, Eve/NYC/22

// mapping: transform each element, then collect the results
List<String> nycNames = people.stream()
    .filter(p -> p.city().equals("NYC"))
    .collect(Collectors.mapping(Person::name, Collectors.toList()));

// minBy / maxBy -> Optional<Person>
Optional<Person> youngest = people.stream()
    .collect(Collectors.minBy(Comparator.comparingInt(Person::age)));
Optional<Person> oldest = people.stream()
    .collect(Collectors.maxBy(Comparator.comparingInt(Person::age)));

// numeric aggregation in one pass
int    totalAge = people.stream().collect(Collectors.summingInt(Person::age));
double avgAge   = people.stream().collect(Collectors.averagingInt(Person::age));

System.out.println(nycNames);
System.out.println(youngest);
System.out.println(oldest);
System.out.println(totalAge);   // 30 + 25 + 35 + 28 + 22
System.out.println(avgAge);     // 140 / 5
Console output[Alice, Carol, Eve] Optional[Person[name=Eve, city=NYC, age=22]] Optional[Person[name=Carol, city=NYC, age=35]] 140 28.0

AdvancedNeed count, sum, min, max and average together? summarizingInt(Person::age) computes all of them in a single pass and returns an IntSummaryStatistics (getCount, getSum, getMin, getMax, getAverage). The fully general form is reducing(identity, mapper, op), of which summingInt and maxBy are specializations.
Pitfall minBy/maxBy hand you an Optional because a stream may be empty — do not blindly call .get(). Note the asymmetry: averagingInt on an empty stream returns 0.0 (not an Optional), while maxBy returns Optional.empty(). All of these are Java 8.

03. groupingBy — basics

Objective: Use Collectors.groupingBy to split a stream into buckets keyed by a classifier function, and swap the default downstream collector to summarize each bucket.

groupingBy(classifier) is the SQL GROUP BY of streams: it applies the classifier to every element and files elements into a Map keyed by the classifier's result. With one argument each bucket's value is a List of the matching elements — the return type is Map<String, List<Person>>.

The two-argument form groupingBy(classifier, downstream) replaces that implicit toList() with any collector — e.g. counting() to get a Map<String, Long> of bucket sizes.

  • Keys are exactly the distinct classifier values that occur — absent categories get no entry.
  • One arg → Map<K, List<T>>; add a downstream collector to reshape each bucket.
  • The result is a plain HashMap, so key order is not guaranteed.
  • Elements within a bucket keep their encounter order.
// people: Alice/NYC/30, Bob/LA/25, Carol/NYC/35, Dave/LA/28, Eve/NYC/22

// one bucket (List<Person>) per distinct city
Map<String, List<Person>> byCity = people.stream()
    .collect(Collectors.groupingBy(Person::city));

// swap the default toList() downstream for counting()
Map<String, Long> countByCity = people.stream()
    .collect(Collectors.groupingBy(Person::city, Collectors.counting()));

System.out.println(byCity);
System.out.println(countByCity);

04. groupingBy — multi-level & collectingAndThen

Objective: Nest collectors to build multi-level groupings, reshape each bucket with mapping, and post-process a bucket's result with collectingAndThen.

Because the downstream argument is itself any collector, you can nest a second groupingBy inside the first to get a Map<K1, Map<K2, ...>> — a two-level grouping. Combine it with mapping to store just the names in the innermost buckets.

collectingAndThen(downstream, finisher) runs a collector and then applies a finishing function to its result. The classic use: turn maxBy(...), which yields an Optional<Person>, into a plain Person with Optional::get — safe here because groupingBy never produces an empty bucket.

  • Nesting groupingBy inside groupingBy yields multi-level maps.
  • mapping(Person::name, toList()) as the downstream keeps only names per group.
  • collectingAndThen(down, finisher) = collect, then transform the collected value.
  • maxBy+Optional::get is the idiomatic "one representative per group" pattern.
// people: Alice/NYC/30, Bob/LA/25, Carol/NYC/35, Dave/LA/28, Eve/NYC/22
Function<Person, String> band = p -> p.age() >= 30 ? "30+" : "under 30";

// (a) mapping downstream: keep only the names in each city bucket
Map<String, List<String>> namesByCity = people.stream()
    .collect(Collectors.groupingBy(Person::city,
             Collectors.mapping(Person::name, Collectors.toList())));

// (b) collectingAndThen: collect, THEN post-process (unwrap the Optional)
Map<String, Person> oldestByCity = people.stream()
    .collect(Collectors.groupingBy(Person::city,
             Collectors.collectingAndThen(
                 Collectors.maxBy(Comparator.comparingInt(Person::age)),
                 Optional::get)));

// (c) multi-level: city -> age band -> names
Map<String, Map<String, List<String>>> nested = people.stream()
    .collect(Collectors.groupingBy(Person::city,
             Collectors.groupingBy(band,
             Collectors.mapping(Person::name, Collectors.toList()))));

System.out.println(namesByCity);
System.out.println(oldestByCity);
System.out.println(nested);

AdvancedcollectingAndThen is also the standard way to make a grouping's buckets immutable: groupingBy(Person::city, collectingAndThen(toList(), List::copyOf)) yields Map<String, List<Person>> whose value lists cannot be modified. Any finisher works — you could map each bucket to its size, sort it, or wrap it in a domain object.
Pitfall Optional::get as a finisher is safe inside groupingBy (buckets are never empty), but the same trick on a top-level maxBy over a possibly-empty stream throws NoSuchElementException. Reach for orElse/orElseThrow there instead.

05. partitioningBy

Objective: Split a stream into exactly two buckets with a boolean predicate using partitioningBy, and understand how it differs from groupingBy.

partitioningBy(predicate) is a specialized groupingBy whose classifier is a Predicate returning a boolean. The result is always a Map<Boolean, List<Person>> with both keys true and false present — even when one side matches nothing, that bucket is an empty list rather than a missing key. This differs from groupingBy, which only creates keys for values that actually occur.

As with groupingBy, a second argument supplies a downstream collector: partitioningBy(pred, counting()) yields a Map<Boolean, Long>.

  • Classifier is a Predicate → the key type is always Boolean.
  • Both false and true keys are always present (empty list, never absent).
  • The result map iterates false then true deterministically — unlike groupingBy's HashMap.
  • For more than two groups, use groupingBy with a non-boolean classifier.
// people: Alice/NYC/30, Bob/LA/25, Carol/NYC/35, Dave/LA/28, Eve/NYC/22

// partition by a boolean predicate: age >= 30
Map<Boolean, List<Person>> byAge = people.stream()
    .collect(Collectors.partitioningBy(p -> p.age() >= 30));

System.out.println(byAge.get(true));    // matched the predicate
System.out.println(byAge.get(false));   // the rest

// downstream collector: count each side
Map<Boolean, Long> counts = people.stream()
    .collect(Collectors.partitioningBy(p -> p.age() >= 30, Collectors.counting()));
System.out.println(counts);

// BOTH keys are ALWAYS present, even when a side is empty:
Map<Boolean, List<Person>> over100 = people.stream()
    .collect(Collectors.partitioningBy(p -> p.age() > 100));
System.out.println(over100.keySet());   // both false and true exist
System.out.println(over100.get(true));  // present, but empty
Console output[Person[name=Alice, city=NYC, age=30], Person[name=Carol, city=NYC, age=35]] [Person[name=Bob, city=LA, age=25], Person[name=Dave, city=LA, age=28], Person[name=Eve, city=NYC, age=22]] {false=3, true=2} [false, true] []

AdvancedThe downstream collector can be anything, so you can build a two-sided summary in one pass — e.g. partitioningBy(p -> p.age() >= 30, Collectors.mapping(Person::name, Collectors.toList())) gives {false=[Bob, Dave, Eve], true=[Alice, Carol]}. The returned map is a tiny fixed-size two-entry structure, cheaper than a general HashMap.

9. Parallel Streams

Parallel streams (part of the Streams API, Java 8, 2014) let one pipeline spread its work across CPU cores with a single method call. The result is identical to the sequential version only when the operation is associative and side-effect-free — otherwise parallelism is either slower or plain wrong. This section shows how to turn a stream parallel, how the fork/join machinery underneath splits and recombines the work, and the crucial cases where you should not reach for it.

01. Introduction to parallel streams

Objective: Turn a sequential stream into a parallel one with parallelStream() or stream().parallel(), and confirm that an associative, side-effect-free reduction yields the same total.

Any stream can run in parallel. From a collection, call collection.parallelStream(); from an existing stream, call .parallel() at any point in the pipeline. Under the hood the elements are divided into chunks that are processed on multiple threads of the shared common ForkJoinPool, and the partial results are merged back together.

The payoff is potential speed on multicore hardware for large, CPU-bound work. The contract is strict: the pipeline must be associative (grouping of the combine step must not matter — (a+b)+c == a+(b+c)) and side-effect-free (no writing to shared mutable state). Sum, min, max, count, and reduce with an associative combiner all qualify, so 1 + 2 + … + 1,000,000 produces the same total whether computed on one thread or eight.

Any timing you observe is illustrative — the actual speed-up depends on core count, data size, JIT warm-up, and memory bandwidth, and for small inputs the parallel version is usually slower because splitting and merging cost more than the work saved.

  • collection.parallelStream() or stream().parallel() switches a pipeline to parallel.
  • Work is split across the common ForkJoinPool and the partials are recombined.
  • Correctness requires an associative, side-effect-free operation — then the result equals the sequential one.
  • LongStream.rangeClosed(1, 1_000_000).parallel().sum() gives 500000500000, same as sequential.
  • Speed-up is hardware-dependent and only likely for large, CPU-bound datasets.
import java.util.*;
import java.util.stream.*;

public class ParallelIntro {

        List<Integer> nums = List.of(1, 2, 3, 4, 5);

02. How parallel streams work — fork/join

Objective: Understand the fork/join mechanics — a Spliterator recursively splits the source, worker threads process the chunks, and partial results are combined — and know the common pool's default size.

A parallel stream is built on the fork/join framework. The stream's Spliterator (“splittable iterator”) repeatedly divides the source in two via trySplit(), forking the halves until each chunk is small enough to process on its own. Each leaf chunk runs the pipeline on a worker thread, and the results are joined back up the tree by the combiner (for a sum, the partial sums are added). The fork/join framework itself predates streams — ForkJoinPool, RecursiveTask/RecursiveAction shipped in Java 7 (2011), and Java 8's parallel streams simply drive it for you; the common pool they submit to (ForkJoinPool.commonPool()) was the Java 8 addition layered on top.

These worker threads live in the shared common ForkJoinPool (ForkJoinPool.commonPool()). Its default parallelism is one fewer than the number of available processors (availableProcessors() - 1, minimum 1); the thread that submitted the pipeline also pitches in, so the effective concurrency is roughly the core count. Because the pool is shared process-wide, one long blocking task can starve every other parallel stream in the JVM.

The illustrative run below prints which thread handled each element. The set of values is fixed (1–8), but the thread names and the interleaving order change from run to run and machine to machine — never rely on either.

  • A Spliterator recursively splits the source via trySplit() (fork); chunks run on worker threads.
  • Partial results are merged by the combiner (join) — e.g. partial sums are added together.
  • Threads come from the shared common ForkJoinPool; default parallelism is availableProcessors() - 1 (min 1).
  • The fork/join framework (ForkJoinPool, RecursiveTask) is itself a Java 7 feature; Java 8 streams reuse it via the Java 8 common pool.
  • The submitting thread also does work, so effective concurrency ≈ core count.
  • The pool is process-wide — a blocking task inside it can stall every other parallel stream.

03. When not to use parallel streams

Objective: Recognize the cases where parallel streams are slower or outright incorrect — especially shared mutable state — and apply a checklist for when parallelism is actually worth it.

Parallel is the exception, not the default. It is frequently slower for small datasets (split/merge overhead dominates), for I/O-bound or blocking work (threads just wait), and it is wrong for order-sensitive pipelines and non-associative reductions. In an ordered stream, forEach visits elements in an unspecified order across threads; use forEachOrdered if encounter order matters (at the cost of much of the parallelism).

The most dangerous mistake is shared mutable state. Handing many worker threads a single non-thread-safe collection — list.parallelStream().forEach(sharedArrayList::add) — is a data race: ArrayList.add is not synchronized, so two threads can write the same backing-array slot, overwrite each other, or trigger a resize mid-write. The result is lost elements, or a NullPointerException/ArrayIndexOutOfBoundsException from a torn resize. The output below is illustrative: the broken size varies from run to run and the program may instead throw.

The fix is to stop mutating shared state: let the stream do the collecting with collect(Collectors.toList()). The collector accumulates into per-thread containers and combines them safely, so the count is always correct. (A concurrent collector such as groupingByConcurrent is the analogous fix for maps.)

  • Small N, I/O-bound/blocking work, order-sensitive pipelines, and non-associative reductions all favor sequential.
  • Boxed sources cost extra: a Stream<Integer> / List<Integer> numeric reduction unboxes and re-boxes every element on the hot path — prefer a primitive stream (IntStream/LongStream/DoubleStream) for parallel numeric work.
  • forEach is unordered across threads; use forEachOrdered when encounter order matters.
  • Shared mutable state is a bug: parallelStream().forEach(list::add) races and loses/corrupts elements.
  • Fix by collecting: collect(Collectors.toList()) combines per-thread partials safely.
  • Use parallel only when: large N • CPU-bound • stateless/side-effect-free • associative • a cheaply splittable source (arrays, ArrayList, IntStream.range).

Pitfall Assuming parallelStream().forEach(list::add) — or putting into a shared HashMap inside forEach — is “probably fine because it usually works.” It is a data race whether or not it crashes today — sometimes it drops elements silently, sometimes it throws. 10. Optional

Optional<T> (final in Java 8, 2014) is a container that either holds one non-null value or holds nothing. Its job is to make "a value might be absent" explicit in the type system, so a method's signature warns the caller instead of silently returning null. This section covers what Optional is and when to use it, the three factory methods that create one, the reading methods that get the value back out (including the crucial orElse-vs-orElseGet difference), and the map/flatMap/filter transformers that let you work through an Optional without ever unwrapping it by hand.

01. Introduction to Optional

Objective: Understand Optional<T> as a typed container for a maybe-absent value, why it beats returning null, and where it belongs (as a return type — not as a field or parameter).

An Optional<T> (Java 8) is a small immutable wrapper with exactly two shapes: present, holding one non-null value, or empty, holding nothing. Crucially, an Optional reference is itself never null in correct code — the emptiness lives inside the container, not in the reference to it.

The point is to encode "this might not be there" in the type. A method declared as Optional<String> findNameById(int id) forces the caller to acknowledge the miss case; a method declared as String findNameById(int id) that can return null does not, and the miss surfaces later as a NullPointerException — Tony Hoare's self-described "billion-dollar mistake". Optional turns a runtime surprise into a compile-time conversation.

Use Optional as a return type for lookups that may find nothing (a map miss, an empty query, a parse that failed). Do not use it as a field type or a method parameter: it adds a wrapper allocation and a second "empty vs null-Optional" failure mode for no real gain, and it is not Serializable. For absent collections, prefer an empty List/Set over Optional<List<...>>.

  • An Optional<T> is present (one non-null value) or empty — nothing else.
  • The container itself should never be null; return Optional.empty(), never a null Optional.
  • It makes absence explicit in the type, replacing null and its NullPointerException surprises.
  • Best used as a return type for maybe-missing lookups — not as a field or a parameter.
  • toString() shows the state: Optional[Ada] when present, Optional.empty when empty.
import java.util.*;

02. Creating Optionals — of, ofNullable, empty

Objective: Choose correctly among the three Java 8 factory methods — Optional.of, Optional.ofNullable, and Optional.empty — and know exactly what each does with a null input.

Optional.of(value) wraps a value you know is non-null. If you hand it null, it throws NullPointerException immediately — a deliberate fail-fast that flags a bug at the source. Use it only for values a constant or an invariant guarantees are present.

Optional.ofNullable(value) is the safe, general factory: it returns an empty Optional when the value is null and a present one otherwise. This is the method you use to bridge from legacy null-returning APIs (like Map.get) into the Optional world. Optional.empty() is the explicit "no value" constant.

  • Optional.of(x) — present if x is non-null; throws NullPointerException if x is null.
  • Optional.ofNullable(x) — empty if x is null, present otherwise; the safe bridge from null-returning code.
  • Optional.empty() — the canonical empty instance (a shared singleton).
  • Rule of thumb: reach for ofNullable unless you can prove the value is non-null, then use of.

03. Reading values — orElse, orElseGet, orElseThrow

Objective: Get the value back out safely and pick the right reader — supplying a default (orElse vs the lazy orElseGet), throwing (orElseThrow), or running side effects (ifPresent, ifPresentOrElse).

The safe readers all answer "give me the value, and here's what to do if it's empty". orElse(default) returns the value or a fixed default. orElseGet(supplier) returns the value or the result of calling a Supplier. orElseThrow(() -> new ...) (Java 8) throws a supplied exception when empty; the no-arg orElseThrow() (added in Java 10) throws NoSuchElementException. There is also raw get() (Java 8), which throws NoSuchElementException when empty — avoid it in favour of the safer readers.

The subtle but important distinction is orElse vs orElseGet. orElse takes an already-computed value, so its argument is always evaluated — even when the Optional is present and the default is thrown away. orElseGet takes a Supplier and is lazy: the supplier runs only when the Optional is empty. When the default is expensive or has side effects (a DB call, a log line), that difference is a real bug waiting to happen — reach for orElseGet.

For side effects without extracting a value, use ifPresent(consumer) (Java 8) — run this only when present — or ifPresentOrElse(consumer, emptyAction) (Java 9), which also gives you an empty branch. The state predicates are isPresent() (Java 8) and its inverse isEmpty() (Java 11).

  • orElse(v) — return the value or v; v is always evaluated, even when present.
  • orElseGet(supplier) — lazy: the supplier runs only when empty. Prefer it for costly defaults.
  • orElseThrow(() -> ex) (Java 8) throws your exception when empty; no-arg orElseThrow() (Java 10) throws NoSuchElementException.
  • ifPresent(c) (Java 8) and ifPresentOrElse(c, r) (Java 9) run side effects; skip the imperative if (isPresent()) get().
  • isPresent() (Java 8) / isEmpty() (Java 11) test the state.
import java.util.*;

public class ReadingOptionals {

04. Transforming Optionals — map, flatMap, filter

Objective: Transform and screen a maybe-absent value without unwrapping it, using map, flatMap, and filter, and know when flatMap is required instead of map.

These three methods (all Java 8) let you build a pipeline that short-circuits cleanly on empty. map(fn) applies fn to the value if present and re-wraps the result as Optional<R>; on an empty Optional it does nothing and stays empty (the function never runs). filter(predicate) keeps the value only if it matches, otherwise turns the Optional empty.

flatMap(fn) is for when the mapping function itself returns an Optional. If you used map there you would get a nested Optional<Optional<R>>; flatMap flattens that one level so you get a plain Optional<R>. This is exactly the tool for navigating a chain of maybe-absent links (a user that may have an address that may have a zip code).

Chained, they read as a clean sequence: Optional.of(" hello ").map(String::trim).filter(s -> !s.isEmpty()).map(String::toUpperCase) yields Optional[HELLO], and any empty or failed-filter step propagates emptiness to the end untouched.

  • map(fn) — transform a present value; empty stays empty and fn is skipped.
  • flatMap(fn) — use when fn returns an Optional; it avoids Optional<Optional<T>>.
  • filter(pred) — keep the value if it matches, otherwise become empty.
  • Steps short-circuit: once empty, the rest of the chain is skipped.
  • Finish with a reader (orElse, orElseGet, ifPresent) to leave the Optional world.
import java.util.*;

public class TransformingOptionals {
    static final class User {
        private final String name;
        private final String email;                 // may be null
        User(String name, String email) { this.name = name; this.email = email; }
        String name() { return name; }
        Optional<String> email() { return Optional.ofNullable(email); }
    }

    public static void main(String[] args) {
        // Chain: trim -> keep non-empty -> upper-case
        Optional<String> upper = Optional.of("  hello  ")
            .map(String::trim)             // Optional["hello"]
            .filter(s -> !s.isEmpty())     // stays (non-empty)
            .map(String::toUpperCase);     // Optional["HELLO"]
        System.out.println(upper);

        // map on empty: mapper never runs, stays empty
        Optional<String> fromEmpty = Optional.<String>empty()
            .map(String::toUpperCase);
        System.out.println(fromEmpty);

        // filter that fails -> empty
        System.out.println(Optional.of(3).filter(n -> n > 10));

        // flatMap: User::email already returns Optional<String>
        User u = new User("Ada", "ada@x.io");
        System.out.println(Optional.of(u).flatMap(User::email));

        // map here would double-wrap:
        System.out.println(Optional.of(u).map(User::email));
    }
}

11. Default & Static Interface Methods

Before Java 8 (2014) an interface could hold only abstract methods (plus constants). Java 8 gave interfaces default methods (a real body, so an API can grow without breaking existing implementers) and static methods (utility factories that live on the interface itself); Java 9 (2017) added private interface methods to share code between defaults. This section shows those method kinds, how Comparator is built almost entirely from them, how interfaces now compare to abstract classes, and how to resolve the diamond conflict when two interfaces supply the same default.

01. Default & static methods in interfaces

Objective: Add a concrete default method and a static factory to your own interface, and understand why Java 8 introduced them.

A default method (Java 8) is an interface method with a body, marked default. Implementing classes inherit it automatically and may override it. Its purpose is interface evolution: the JDK could add stream() to Collection, sort() to List, and forEach() to Iterable without breaking the thousands of classes that already implemented those interfaces — every implementer inherited the new behaviour for free.

A static interface method (also Java 8) belongs to the interface type, not to instances, and is called as InterfaceName.method(...). It is the natural home for factory and helper methods related to the type — the JDK uses this for Comparator.comparing(...), Stream.of(...), and List.of(...). Static interface methods are not inherited by implementing classes.

Java 9 later added private (and private static) interface methods so several default methods can share helper logic without exposing it as public API. Private methods have a body but are invisible to implementers and callers.

  • default methods (Java 8) carry a body and are inherited by implementers; they exist to grow an API without breaking it.
  • static interface methods (Java 8) are called on the interface (Greeter.blank()), are not inherited, and suit factories/helpers.
  • private interface methods (Java 9) share code between defaults but are hidden from callers.
  • An implementer may override a default method; if it does not, it silently inherits the default body.
interface Greeter {
    String name();                       // abstract — implementer must supply

    // default (Java 8): inherited body, overridable
    default String greet() {
        return "Hello, " + name() + punctuation();
    }

    // private (Java 9): shared helper, hidden from implementers
    private String punctuation() {
        return "!";
    }

    // static (Java 8): factory on the interface itself
    static Greeter of(String who) {
        return () -> who;                // lambda supplies name()
    }
}

public class GreeterDemo {
    public static void main(String[] args) {
        Greeter ada = () -> "Ada";       // inherits greet()
        System.out.println(ada.greet());

        Greeter grace = Greeter.of("Grace");   // static factory
        System.out.println(grace.greet());

        // override the default for one instance
        Greeter shout = new Greeter() {
            public String name() { return "Linus"; }
            public String greet() { return name().toUpperCase() + "!!!"; }
        };
        System.out.println(shout.greet());
    }
}
Console outputHello, Ada! Hello, Grace! LINUS!!!

AdvancedA default method cannot access instance fields (interfaces have no state) — it can only call other methods on this, so greet() reaches the data through the abstract name(). This "template method via defaults" pattern lets an interface define behaviour in terms of a few abstract hooks. Also note the static of(...) returning a lambda: because Greeter has one abstract method it is a functional interface, so () -> who is a valid Greeter.
Pitfall A static interface method is not inherited: you must call it as Greeter.of(...), never through an instance or an implementing class name. And private interface methods need JDK 9+ — on Java 8 the compiler rejects private in an interface.

02. Comparator's default methods

Objective: Build a multi-key comparator from Comparator's static factories and default combinators, then sort a list of records by department, then salary descending.

Comparator is the showcase for this feature: almost its entire fluent API is made of static factories and default combinators added in Java 8. The static entry points include Comparator.comparing(keyExtractor), naturalOrder(), reverseOrder(), and the null-safe wrappers nullsFirst(...) / nullsLast(...). The default (instance) combinators include thenComparing(...) for tie-breakers and reversed() to flip an ordering.

Chaining them reads like a sentence. To sort employees by department ascending, then by salary descending, start from comparing(Employee::dept) and append thenComparing(comparing(Employee::salary).reversed()). Using a record Employee (records are final in Java 16) gives us tidy accessor method references Employee::dept and Employee::salary.

  • Static factories: comparing, naturalOrder, reverseOrder, nullsFirst/nullsLast (all Java 8).
  • Default combinators: thenComparing (secondary key) and reversed (flip order).
  • Apply .reversed() to the sub-comparator so it only flips that one key, not the whole chain.
  • For an int key, comparingInt(Employee::salary) avoids boxing (same idea, primitive-specialised).
import java.util.*;
import static java.util.Comparator.comparing;

public class ComparatorChain {
    record Employee(String name, String dept, int salary) {}

    public static void main(String[] args) {
        List<Employee> staff = new ArrayList<>(List.of(
            new Employee("Ada",   "ENG", 120),
            new Employee("Bob",   "OPS",  90),
            new Employee("Cleo",  "ENG", 150),
            new Employee("Dan",   "OPS",  90),
            new Employee("Eve",   "ENG", 120)
        ));

        // dept ASC, then salary DESC
        Comparator<Employee> byDeptThenPay =
            comparing(Employee::dept)
                .thenComparing(comparing(Employee::salary).reversed());

        staff.sort(byDeptThenPay);
        staff.forEach(e ->
            System.out.println(e.dept() + " " + e.salary() + " " + e.name()));
    }
}
Console outputENG 150 Cleo ENG 120 Ada ENG 120 Eve OPS 90 Bob OPS 90 Dan

AdvancedAda and Eve tie on both keys (ENG, 120). List.sort is a stable sort (TimSort), so it preserves their original relative order — Ada was listed before Eve, so Ada stays first. The same stability keeps Bob before Dan. If you wanted a fully deterministic total order you would add a final tie-breaker such as .thenComparing(Employee::name).
Context This is what the nullsFirst/nullsLast wrappers are for. Sorting a list that contains a null element with an ordinary comparator throws NullPointerException — staff.sort(comparing(Employee::name)) blows up because the key extractor dereferences the null element. Wrap the comparator to survive it: staff.sort(Comparator.nullsFirst(comparing(Employee::name))) sorts null elements to the front, and Comparator.nullsLast(...) sorts them to the end.
Pitfall Call .reversed() in the wrong place and you flip too much: comparing(Employee::dept).thenComparing(Employee::salary).reversed() reverses the entire composite (dept DESC and salary DESC). To reverse only salary, reverse the sub-comparator: thenComparing(comparing(Employee::salary).reversed()).

03. Abstract classes vs interfaces

Objective: Contrast interfaces (now with default/static/private methods) against abstract classes, and choose the right tool for a design.

Since Java 8 blurred the line by giving interfaces method bodies, the remaining differences are about state and inheritance shape. An interface models a capability or role: a class can implement many interfaces (multiple inheritance of type), its methods are implicitly public, and it may carry default/static/private methods — but it has no instance fields, no constructors, and no mutable state (only public static final constants).

An abstract class models a partial implementation of an is-a hierarchy: it can declare instance fields and mutable state, constructors to initialise them, and protected/package-private members — but a class can extend only one. Rule of thumb: reach for an interface to express a role many unrelated types can play; reach for an abstract class when subclasses genuinely share state and a common construction path.

  • Multiplicity: implement many interfaces; extend exactly one (abstract) class.
  • State: interfaces hold no instance fields; abstract classes can hold fields and mutable state.
  • Constructors: interfaces have none; abstract classes can define constructors for subclasses to call via super(...).
  • Access: interface members are effectively public (plus private helpers); abstract classes allow protected/package-private.
  • Choose: interface = a role/capability; abstract class = shared state + template for a family of subclasses.
// Abstract class: shared STATE + constructor + a template hook
abstract class Shape {
    private final String name;           // instance field — interfaces can't have this
    protected Shape(String name) {       // constructor — interfaces can't have this
        this.name = name;
    }
    abstract double area();              // subclass hook
    String describe() { return name + " area=" + area(); }
}

// Interface: a pure capability, implementable by ANY type
interface Drawable {
    void draw();
    default void render() { System.out.println("rendering..."); draw(); }
}

class Circle extends Shape implements Drawable {   // one class + many interfaces
    private final double r;
    Circle(double r) { super("circle"); this.r = r; }
    double area() { return Math.PI * r * r; }
    public void draw() { System.out.println(describe()); }
}

public class ShapeDemo {
    public static void main(String[] args) {
        new Circle(2).render();
    }
}

AdvancedCircle uses both tools at once: it extends the abstract Shape to inherit the name field and the describe() template, and it implements Drawable to advertise the drawing capability. This is the idiomatic split — abstract classes carry shared state up an is-a chain, interfaces bolt on orthogonal roles. Java's own library does exactly this: AbstractList (abstract class, shared plumbing) implements List (interface, the contract).
Context The old "abstract class = code, interface = contract" distinction is obsolete since Java 8. Today an interface can supply plenty of code via default methods; the durable differences are (1) an abstract class can hold mutable instance state and constructors, and (2) you can only extend one class but implement many interfaces.

04. Multiple inheritance & resolving conflicts

Objective: Understand the diamond problem for default methods and resolve it with an explicit override plus Interface.super.method().

A class may implement many interfaces, which reopens a classic question: what if two of them provide the same default method? Java's answer is strict and safe — if a class inherits two conflicting default methods with the same signature, it does not compile until the class resolves the clash by overriding the method itself. There is no silent "left wins" rule.

Inside its override, the class can still reuse a specific parent's implementation with the syntax InterfaceName.super.method(...). So a class C that implements both A and B can override greet() and delegate to A's version via A.super.greet(), optionally combining both. (Note the general resolution rule: a concrete method inherited from a superclass beats any interface default — "class wins" — but two competing interface defaults must be resolved by hand.)

  • Two interfaces supplying the same default method = a compile error until the class overrides it.
  • Java has no implicit precedence between interface defaults — you must choose explicitly.
  • Delegate to a chosen parent with A.super.greet() (qualified super).
  • A concrete method inherited from a superclass automatically wins over an interface default ("class wins").
interface A {
    default String greet() { return "Hello from A"; }
}
interface B {
    default String greet() { return "Hello from B"; }
}

// C inherits TWO greet() defaults, so it MUST override to compile.
class C implements A, B {
    @Override
    public String greet() {
        // pick A's version explicitly, then add our own touch
        return A.super.greet() + " (chosen by C)";
    }
}

public class DiamondDemo {
    public static void main(String[] args) {
        System.out.println(new C().greet());
    }
}
Console outputHello from A (chosen by C)

AdvancedThe qualified-super form is InterfaceName.super.method() — the interface must be one that the current class directly implements, and it targets that interface's default. You are free to call both (A.super.greet() + " / " + B.super.greet()) to merge them. This resolution rule also fires when one interface extends another: if B extends A and both define the default, the more specific type (B) wins automatically and no manual override is required.
Pitfall You cannot skip a level: A.super.greet() only works when A is a directly implemented interface of the enclosing class. Also, plain super.greet() will not compile here: in a class, super refers to the superclass (Object), which declares no greet() — so you get a "cannot find symbol" error. Unqualified super never reaches an interface default; you must write the qualified form A.super.greet() (or B.super.greet()).

12. The Date/Time API (java.time)

Java 8 (2014) shipped a brand-new date and time library under java.time — JSR-310, designed by the author of Joda-Time. Unlike the old java.util.Date/Calendar, every type here is immutable and thread-safe, uses a fluent API, follows the ISO-8601 standard, and cleanly separates a date from a time from a date-time from an absolute machine instant. This section walks the core types you reach for daily: LocalDate, LocalTime, LocalDateTime, Period, Duration, Instant, ZonedDateTime, and DateTimeFormatter.

01. Why a new Date/Time API?

Objective: Understand the design flaws of the legacy Date/Calendar classes and why Java 8 introduced java.time, and learn the map of its core types.

The original java.util.Date (Java 1.0) and java.util.Calendar (Java 1.1) were notoriously hard to use. They are mutable — any method holding a Date can change it under you, which also makes them not thread-safe. Months are 0-based (January is 0), Date's year is offset from 1900, and a single Date conflates a date, a time, and an implicit time zone. The companion formatter, SimpleDateFormat, is likewise mutable and not thread-safe — a classic source of production bugs.

java.time (JSR-310, Java 8) replaces all of that with a set of immutable, thread-safe value types. Every "modify" method returns a new object rather than mutating the receiver, the API reads fluently (date.plusDays(1).withYear(2025)), months are 1-based, and text I/O defaults to the unambiguous ISO-8601 format. Crucially, distinct concepts get distinct types instead of one overloaded Date.

  • Legacy Date/Calendar: mutable, not thread-safe, 0-based months, confusing 1900-based year.
  • java.time (Java 8, JSR-310): immutable, thread-safe, fluent, ISO-8601, 1-based months.
  • Human types: LocalDate (date), LocalTime (time), LocalDateTime (both, no zone).
  • Machine/zoned types: Instant (UTC timestamp), ZonedDateTime (date-time + ZoneId).
  • Amounts: Period (date-based: years/months/days), Duration (time-based: hours/seconds/nanos). Text: DateTimeFormatter.
import java.time.LocalDate;

        LocalDate next = date.plusDays(1);

        System.out.println(date);              // original, untouched
        System.out.println(next);              // a new immutable object
        System.out.println(date.getMonthValue()); // 1, not 0
    }
}

02. LocalDate — create, read & modify

Objective: Create a LocalDate three ways, read its fields, and produce modified copies while proving the original stays unchanged.

LocalDate (Java 8) is a date with no time-of-day and no zone — a year, month, and day, such as a birthday or an invoice date. You create one with LocalDate.now() (today, from the system clock), LocalDate.of(2024, 1, 15) (explicit fields, month 1-based), or LocalDate.parse("2024-01-15") (from an ISO-8601 string).

Reading is a set of get… accessors: getYear(), getMonth() (returns the Month enum, e.g. JANUARY), getMonthValue() (the number 1–12), getDayOfMonth(), getDayOfWeek() (the DayOfWeek enum), and helpers like isLeapYear(). Because the type is immutable, the "modify" methods — plusDays, minusMonths, withYear, and friends — each return a new LocalDate; the receiver is never touched.

  • Create: LocalDate.now(), LocalDate.of(year, month, day) (month 1-based), LocalDate.parse("2024-01-15").
  • Read: getYear(), getMonth() (a Month enum), getMonthValue(), getDayOfMonth(), getDayOfWeek(), isLeapYear().
  • Modify (returns a new copy): plusDays, minusMonths, withYear, plusWeeks, minusDays…
  • The original is immutable — capture the return value or the change is lost.
import java.time.LocalDate;

public class LocalDateDemo {

03. LocalTime & LocalDateTime

Objective: Use LocalTime for a wall-clock time and LocalDateTime for a combined date-and-time, and convert freely between a date, a time, and a date-time.

LocalTime (Java 8) is a time-of-day with no date and no zone — hours, minutes, seconds, nanoseconds. Build one with LocalTime.of(14, 30) or LocalTime.now(); shift it with plusHours, plusMinutes, and so on. LocalDateTime pairs a LocalDate with a LocalTime (still no zone): create it directly with LocalDateTime.of(2024, 1, 15, 14, 30).

The three human types compose cleanly. From a LocalDate, date.atTime(14, 30) attaches a time and date.atStartOfDay() attaches midnight — both yield a LocalDateTime. Going the other way, a LocalDateTime splits back apart with toLocalDate() and toLocalTime(). Note the ISO text form: a LocalDateTime prints with a T between the date and time (2024-01-15T14:30), and zero seconds are simply omitted.

  • LocalTime.of(14, 30) — time only; plusHours(2) returns a new time.
  • LocalDateTime.of(y, mo, d, h, mi) — date + time, still no zone.
  • Compose: date.atTime(14, 30), date.atStartOfDay() → LocalDateTime.
  • Decompose: dateTime.toLocalDate(), dateTime.toLocalTime().
  • ISO text uses a T separator: 2024-01-15T14:30.

04. Period & Duration

Objective: Measure the gap between two points in time — Period for a date-based amount (years/months/days), Duration for a time-based amount (hours/minutes/seconds).

Period (Java 8) models an amount of time in date units: years, months, and days. Period.between(start, end) takes two LocalDates and breaks the gap into calendar components — for example, 2024-01-01 to 2024-03-15 is 0 years, 2 months, 14 days. You read the parts with getYears(), getMonths(), getDays(), or build one directly with Period.ofDays(10). Its ISO text form is P… (period), e.g. P2M14D.

Duration (Java 8) models an amount in time units: hours, minutes, seconds, and nanoseconds. Duration.between(t1, t2) works on time-bearing values like LocalTime or Instant; factories include Duration.ofHours(2). Convert to a single unit with toMinutes(), toHours(), getSeconds(). Its text form starts with PT (period of time), e.g. PT2H30M. Rule of thumb: dates → Period, clock times → Duration.

  • Period = date-based (years/months/days); Period.between(d1, d2), Period.ofDays(10).
  • Read a period with getYears()/getMonths()/getDays(); text form P2M14D.
  • Duration = time-based (hours/seconds/nanos); Duration.between(t1, t2), Duration.ofHours(2).
  • Convert a duration with toMinutes()/toHours()/getSeconds(); text form PT2H30M.
  • Pick by what you are measuring: calendar span → Period, elapsed clock time → Duration.
import java.time.LocalDate;
import java.time.LocalTime;
import java.time.Period;
import java.time.Duration;

public class PeriodDurationDemo {
    public static void main(String[] args) {
        // Period — date-based amount (years / months / days)
        Period p = Period.between(
                LocalDate.of(2024, 1, 1),

05. Instant & time zones

Objective: Represent an absolute moment with Instant, and project it into a human calendar in a specific zone with ZoneId and ZonedDateTime.

An Instant (Java 8) is a single point on the timeline, counted as seconds and nanoseconds from the Unix epoch (1970-01-01T00:00:00Z). It is absolute and always in UTC — the ideal type for a machine timestamp, a "created at" field, or measuring elapsed real time. Create it with Instant.now() or Instant.parse("2024-01-15T12:00:00Z"); its text form ends in Z (Zulu = UTC).

A time zone is named by a ZoneId, e.g. ZoneId.of("America/New_York") — region IDs that carry the full daylight-saving rules. A ZonedDateTime is a LocalDateTime plus a ZoneId, so it pins wall-clock time to the actual timeline. You cross between the two worlds with instant.atZone(zoneId) (an absolute instant → local wall-clock reading) and localDateTime.atZone(zoneId) / zonedDateTime.toInstant() (a local reading → an absolute instant). The same Instant shows different wall-clock times in different zones, but it is one moment.

  • Instant = absolute UTC timestamp (seconds from the 1970 epoch); text ends in Z.
  • ZoneId.of("America/New_York") — a region ID carrying DST rules (not a fixed offset).
  • ZonedDateTime = LocalDateTime + ZoneId — a wall-clock time pinned to the timeline.
  • Convert: instant.atZone(zone), localDateTime.atZone(zone), zonedDateTime.toInstant().
  • An Instant is one moment; a LocalDateTime is a reading with no zone.
import java.time.Instant;
import java.time.LocalDateTime;
import java.time.ZoneId;
import java.time.ZonedDateTime;

public class InstantZoneDemo {
    public static void main(String[] args) {
        // Instant — an absolute point on the UTC timeline (Z = UTC)
        Instant instant = Instant.parse("2024-01-15T12:00:00Z");
        System.out.println(instant);            // 2024-01-15T12:00:00Z

        // Same moment, read on a New York clock. In January NY is EST (UTC-5)
        ZoneId ny = ZoneId.of("America/New_York");
        ZonedDateTime nyTime = instant.atZone(ny);
        System.out.println(nyTime);             // 2024-01-15T07:00-05:00[America/New_York]

        // A LocalDateTime has NO zone; attach one to place it on the timeline
        LocalDateTime local = LocalDateTime.of(2024, 1, 15, 12, 0);
        ZonedDateTime zoned = local.atZone(ny);
        System.out.println(zoned);              // 2024-01-15T12:00-05:00[America/New_York]
        System.out.println(zoned.toInstant());  // 2024-01-15T17:00:00Z
    }
}

06. Formatting & parsing with DateTimeFormatter

Objective: Convert dates to text and back using a custom pattern and a predefined formatter, and understand why DateTimeFormatter is safe to share.

By default, java.time types print in ISO-8601 (2024-01-15). To control the layout you build a DateTimeFormatter from a pattern: DateTimeFormatter.ofPattern("dd/MM/yyyy"). Formatting is date.format(formatter) → a String; parsing is LocalDate.parse(text, formatter) → a LocalDate. The parse must match the same pattern the text was written with, or it throws DateTimeParseException.

For common shapes there are ready-made constants such as DateTimeFormatter.ISO_LOCAL_DATE (yields 2024-01-15). Pattern letters are case-sensitive and meaningful: dd day, MM month, yyyy year, HH 24-hour, mm minute. Unlike the legacy SimpleDateFormat, a DateTimeFormatter is immutable and thread-safe, so one instance can be a shared static final constant across threads.

  • Build: DateTimeFormatter.ofPattern("dd/MM/yyyy").
  • Format: date.format(fmt) → text; Parse: LocalDate.parse(text, fmt) → date.
  • Predefined constants, e.g. DateTimeFormatter.ISO_LOCAL_DATE → 2024-01-15.
  • Pattern letters are case-sensitive: MM = month, mm = minute, HH = 24-hour, yyyy = year.
  • DateTimeFormatter is immutable & thread-safe (unlike SimpleDateFormat) — share it as a constant.
import java.time.LocalDate;
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;

public class FormatterDemo {
    public static void main(String[] args) {
        LocalDate date = LocalDate.of(2024, 1, 15);

        DateTimeFormatter fmt = DateTimeFormatter.ofPattern("dd/MM/yyyy");
        String text = date.format(fmt);
        System.out.println(text);                    // 15/01/2024

        // Parse back using the SAME pattern
        LocalDate parsed = LocalDate.parse("15/01/2024", fmt);
        System.out.println(parsed);                  // 2024-01-15

        System.out.println(date.format(DateTimeFormatter.ISO_LOCAL_DATE)); // 2024-01-15

        // A date-time pattern
        LocalDateTime dt = LocalDateTime.of(2024, 1, 15, 14, 30);
        DateTimeFormatter dtFmt = DateTimeFormatter.ofPattern("dd/MM/yyyy HH:mm");
        System.out.println(dt.format(dtFmt));        // 15/01/2024 14:30
    }
}

07. Converting legacy Date to java.time

Objective: Bridge existing java.util.Date / java.sql.Date code into java.time and back, so you can migrate incrementally without rewriting everything at once.

Real code rarely starts on a blank page: you inherit APIs that still hand you a java.util.Date. Java 8 added the conversion bridge directly onto the legacy classes so you never have to stay stuck. A java.util.Date is an absolute instant, so it maps to Instant via toInstant(); from an Instant you pick a zone to land on a LocalDate. Going the other way, java.util.Date.from(instant) rebuilds a legacy Date. The database type java.sql.Date is date-only and adds its own valueOf(LocalDate) / toLocalDate() pair.

  • java.util.Date ↔ Instant: date.toInstant() and java.util.Date.from(instant) (both Java 8).
  • Instant → LocalDate: pick a zone — instant.atZone(zone).toLocalDate() (use ZoneId.systemDefault() in real code).
  • java.sql.Date ↔ LocalDate: java.sql.Date.valueOf(localDate) and sqlDate.toLocalDate().
  • All of these conversion methods are Java 8 additions layered onto the old classes.
import java.time.*;

public class LegacyDateBridge {

        System.out.println(sqlDate);

01. Compact source files & instance main methods

Objective: Write a runnable program with no class declaration, no static, and no String[] args — the Java 25 "first program" experience — and know it is final only in Java 25 (JEP 512).

  • The compiler supplies an implicit, unnamed enclosing class.
  • IO.println / IO.readln live in java.lang.IO (Java 25).

02. Local variable type inference — var

Objective: Use var to let the compiler infer a local variable's type from its initializer, and memorize the places where var is not allowed. Final in Java 10.

var (final in Java 10) is pure syntax sugar for local variables: the compiler reads the initializer's static type and uses it. The variable is still statically, strongly typed — var is not Object and Java does not become dynamic. var list = new ArrayList<String>(); is exactly ArrayList<String>, and calling a non-String method on its elements is still a compile error.

Because inference needs an initializer with a definite type, several forms are rejected: no fields, no method parameters or return types, no bare var x; without an initializer, and no var x = null; (nothing to infer). You also cannot use var for a lambda or method-reference target, since those have no type of their own.

  • Java 10. Local variables with an initializer only.
  • Still statically typed — the type is inferred, not dynamic.
  • Great for verbose generics: var m = new HashMap<String, List<Order>>();.
  • Not allowed: fields, params, return types, var x;, var x = null;, lambda targets.
  • Java 11 additionally allows var in lambda parameter lists.

// var x;              // no initializer
// var y = null;       // nothing to infer

03. Text blocks

Objective: Write multi-line string literals with """, understand how incidental leading whitespace is stripped, and control line breaks with \ and \s. Final in Java 15.

A text block (final in Java 15) is a string literal delimited by """ followed by a line terminator. It spans multiple lines, and embedded double quotes need no escaping — perfect for JSON, HTML, or SQL. The compiler removes incidental leading whitespace: it finds the least-indented non-blank line (the closing """ counts as a line) and strips that common indent from every line, so you can indent the block to match your code without polluting the value.

Two escapes fine-tune the result: a trailing \ suppresses the line break (joining a line with the next), and \s emits a single space and protects trailing whitespace from being stripped. Trailing spaces on a line are otherwise removed.

  • Java 15. Opens with """ + newline; embedded " needs no escape.
  • Incidental indentation is stripped relative to the least-indented line / the closing delimiter.
  • Relative indentation is preserved (nested tags stay nested).
  • \ = no line break here; \s = keep one trailing space.
  • Combine with String::formatted for simple templating.
String html = """

System.out.println("---");

04. Enhanced switch — expressions & yield

Objective: Use switch as an expression that produces a value, with arrow labels, multi-label cases, and yield for block bodies. Final in Java 14.

Switch expressions became final in Java 14. Arrow labels (case X ->) run exactly one branch — no fall-through, no break — and the whole switch can be used as an expression whose value you assign or return. Group several constants with commas: case MON, TUE ->. When a branch needs statements, use a block and hand back its value with yield.

A switch expression must be exhaustive: it has to cover every possible input or provide a default. Switching over an enum where every constant is listed is exhaustive on its own, so no default is required.

  • Java 14. case X -> arrows: one branch, no fall-through, no break.
  • Used as an expression — it yields a value you can assign or return.
  • Multi-label: case MON, TUE, WED -> ....
  • Block body returns its value with yield.
  • Must be exhaustive: cover all cases or add default.
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

static int hoursOfSleep(Day d) {
    return switch (d) {                 // switch as an EXPRESSION
        case MON, TUE, WED, THU -> 7;   // multi-label arrow
        case FRI                -> 6;
        case SAT, SUN -> {
            int base = 9;
            yield base + 1;             // block body -> yield
        }
    };                                  // all enum constants covered: exhaustive
}

public static void main(String[] args) {
    System.out.println(hoursOfSleep(Day.MON));
    System.out.println(hoursOfSleep(Day.FRI));
    System.out.println(hoursOfSleep(Day.SUN));
}

05. Records

Objective: Declare an immutable data carrier with record and get a canonical constructor, accessors, and value-based equals/hashCode/toString for free. Final in Java 16.

A record (final in Java 16) is a transparent carrier for a fixed set of values. record Point(int x, int y) {} auto-generates a canonical constructor, the accessors x() and y() (no get prefix), and value-based equals, hashCode, and toString. Two records with equal components are .equals-equal.

For validation or normalization, write a compact canonical constructor — just the name and braces, no parameter list — which runs before the implicit field assignment. Records are implicitly final, cannot extend a class (they already extend java.lang.Record), and hold no instance state beyond their components, but they can add static and instance methods and implement interfaces.

  • Java 16. Canonical constructor, accessors, equals/hashCode/toString generated.
  • Accessors are component-named: x(), not getX().
  • Compact constructor Point { ... } validates/normalizes before assignment.
  • Implicitly final; cannot extend a class; can implement interfaces.
  • Value equality: same components → equals true.
record Point(int x, int y) {
    Point {                                   // compact canonical constructor
        if (x < 0 || y < 0)
            throw new IllegalArgumentException("negative coordinate");
    }
    double distanceFromOrigin() {             // records may add methods
        return Math.sqrt(x * x + y * y);
    }
}

    Point a = new Point(3, 4);
    Point b = new Point(3, 4);
    System.out.println(a);                    // generated toString
    System.out.println(a.x() + ", " + a.y()); // component accessors
    System.out.println(a.equals(b));          // value equality
    System.out.println(a.distanceFromOrigin());

06. Sealed classes & interfaces

Objective: Restrict which types may implement or extend a type with sealed ... permits ..., and see how a closed hierarchy enables exhaustive switches. Final in Java 17.

Sealed types (final in Java 17) let a class or interface declare exactly which subtypes are allowed: sealed interface Shape permits Circle, Square {}. Every permitted subtype must itself be final, sealed, or non-sealed, so the hierarchy stays closed and cannot be secretly extended elsewhere. Records are implicitly final, so a record implementing a sealed interface satisfies the rule with no extra keyword.

The payoff is exhaustiveness: because the compiler knows the complete set of subtypes, a pattern switch over a sealed type needs no default once every permitted type is handled — you get a compile error the day someone adds a new subtype and forgets a case.

  • Java 17. sealed ... permits A, B closes the hierarchy.
  • Each permitted subtype is final, sealed, or non-sealed.
  • Records satisfy the rule automatically (implicitly final).
  • Enables exhaustive pattern switches without a default.
  • permits may be omitted if all subtypes are in the same source file.
sealed interface Shape permits Circle, Square {}

record Circle(double radius) implements Shape {}   // records are implicitly final
record Square(double side)   implements Shape {}

public static void main(String[] args) {
    Shape s1 = new Circle(2.0);
    Shape s2 = new Square(3.0);
    System.out.println(s1);
    System.out.println(s2);
}

AdvancedA non-sealed subtype deliberately re-opens that branch of the hierarchy for arbitrary extension, letting you seal most of a design while leaving one extension point. Permitted subtypes must live in the same module (or, for the unnamed module, the same package) as the sealed parent.
Pitfall Forgetting a modifier on a permitted subtype is a compile error: a plain class Circle implements Shape without final/sealed/non-sealed won't build. Records sidestep this because they are already final.

07. Pattern matching — instanceof, switch & records

Objective: Replace cast-heavy type checks with pattern matching: instanceof patterns (Java 16), then type, record, and guarded patterns in switch (Java 21).

The instanceof pattern (final in Java 16) binds a variable in the same test: if (o instanceof String s) checks the type and, on success, makes s available already cast. In Java 21, pattern matching graduated to switch: type patterns (case Integer i ->), record patterns that deconstruct components (case Circle(double r) ->), and guarded patterns that add a boolean with when (case Integer i when i > 10 ->).

Combined with a sealed type, a pattern switch is exhaustive without a default. A case null label lets the switch handle null explicitly instead of throwing.

  • instanceof patterns: Java 16 — test and bind in one step.
  • Type patterns in switch: Java 21 — case Integer i ->.
  • Record (deconstruction) patterns: Java 21 — case Circle(double r) ->.
  • Guarded patterns: Java 21 — case Integer i when i > 10 ->.
  • Sealed type → exhaustive switch, no default needed.
// instanceof pattern — Java 16
static String describe(Object o) {
    if (o instanceof String s)  return "String[" + s.length() + "]";
    if (o instanceof Integer i) return "Integer:" + i;
    return "other";
}

// type + guarded patterns in switch — Java 21
static String classify(Object o) {
    return switch (o) {
        case Integer i when i > 10 -> "big int";   // guarded
        case Integer i             -> "small int";
        case String s              -> "text:" + s;
        case null                  -> "null!";     // explicit null label
        default                    -> "other";
    };
}

// record patterns + exhaustive sealed switch — Java 21
sealed interface Shape permits Circle, Square {}
record Circle(double radius) implements Shape {}
record Square(double side)   implements Shape {}

static double area(Shape s) {
    return switch (s) {
        case Circle(double r)    -> Math.PI * r * r;   // deconstruction
        case Square(double side) -> side * side;
    };                                                 // sealed: no default
}

public static void main(String[] args) {
    System.out.println(describe("hi"));
    System.out.println(classify(42));
    System.out.println(classify(3));
    System.out.println(classify("x"));
    System.out.printf("%.2f%n", area(new Circle(2)));
    System.out.println(area(new Square(3)));
}
Console outputString[2] big int small int text:x 12.57 9.0

Source: reimplemented as an original illustration.

AdvancedRecord patterns nest: case Line(Point(var x1, var y1), Point(var x2, var y2)) -> ... destructures a whole tree in one label, and var is allowed for each component's inferred type. The compiler enforces case ordering by dominance: a broader pattern may not precede a narrower one it would shadow.
Pitfall A guarded case must come before the unguarded case of the same type (case Integer i when i > 10 before case Integer i); the reverse is "dominated" and won't compile. And switch/record patterns need JDK 21 — only the instanceof form works on 16–20.

Case study — a sealed PaymentService. Here is why sealed types plus an exhaustive switch matter in real code. Model each payment method as a subclass of a sealed PaymentGateway, and let a PaymentFactory pick the implementation with an exhaustive switch over a CardType enum — with no default branch.

// Applied case study: model payment types as a sealed hierarchy and
// dispatch with an exhaustive switch. Combines sealed classes (Java 17),
// switch expressions (Java 14) and the case null label (Java 21).

enum CardType { CREDIT, DEBIT, REWARDS }
enum PaymentResponse { SUCCESS, FAILURE }

sealed abstract class PaymentGateway
        permits CreditCardPayment, DebitCardPayment, RewardsCardPayment {
    abstract PaymentResponse makePayment(int amount);
}

final class CreditCardPayment extends PaymentGateway {
    PaymentResponse makePayment(int amount) {
        System.out.println("Acquire credit card payment for the amount " + amount);
        return PaymentResponse.SUCCESS;
    }
}
final class DebitCardPayment extends PaymentGateway {
    PaymentResponse makePayment(int amount) {
        System.out.println("Acquire debit card payment for the amount " + amount);
        return PaymentResponse.SUCCESS;
    }
}
final class RewardsCardPayment extends PaymentGateway {
    PaymentResponse makePayment(int amount) {
        System.out.println("Acquire rewards card payment for the amount " + amount);
        return PaymentResponse.SUCCESS;
    }
}

class PaymentFactory {
    // exhaustive over the enum -- NO default. Add a new CardType and this
    // switch stops compiling until you handle it: a compile-time safety net.
    static PaymentGateway paymentGateway(CardType type) {
        return switch (type) {
            case CREDIT  -> new CreditCardPayment();
            case DEBIT   -> new DebitCardPayment();
            case REWARDS -> new RewardsCardPayment();
            case null    -> throw new IllegalArgumentException(
                                "card type null not supported");
        };
    }
}

public class CheckoutService {
    public static void main(String[] args) {
        for (CardType type : CardType.values()) {
            PaymentFactory.paymentGateway(type).makePayment(99);
        }
    }
}

Why this design pays off The sealed hierarchy plus the default-free switch is the production payoff: because PaymentGateway is sealed and the switch is exhaustive over the enum, adding a new CardType (or a new PaymentGateway subtype) forces a compile-time gap you cannot forget to fill — the compiler, not a runtime NullPointerException, catches the missing case. 08. Unnamed variables & patterns

Objective: Use _ for a binding you are required to declare but never read, making "intentionally ignored" explicit to both readers and the compiler. Final in Java 22.

The unnamed variable/pattern _ (final in Java 22) marks a declaration you must write but never use: an exception you don't inspect (catch (NumberFormatException _)), a loop variable you only count (for (var _ : items)), an unused lambda parameter, or an ignored record component (case Point(int x, _)). It documents intent and lets the compiler confirm the binding is deliberately discarded.

Unlike a normal name, _ introduces no readable variable — you cannot reference it — and, uniquely, you may use _ more than once in the same scope without a "duplicate variable" error.

  • Java 22. _ = "declared but intentionally unused".
  • Works in catch, enhanced-for, lambda params, and record patterns.
  • Cannot be read — there is no variable to reference.
  • May appear multiple times in one scope (normal names cannot).
  • Signals intent to readers and silences unused-variable warnings.
import java.util.List;
import java.util.Map;

// unused catch binding — Java 22
static int parseOrZero(String s) {
    try {
        return Integer.parseInt(s);
    } catch (NumberFormatException _) {   // we don't inspect the exception
        return 0;
    }
}

public static void main(String[] args) {
    System.out.println(parseOrZero("42"));
    System.out.println(parseOrZero("oops"));

    // unused loop variable: we only count elements
    var names = List.of("Ada", "Linus", "Grace");
    int count = 0;
    for (var _ : names) count++;
    System.out.println("count = " + count);

    // unused lambda parameter: ignore the key, use the value
    Map.of("Ada", 36).forEach((_, age) -> System.out.println("age = " + age));

    // record pattern with an ignored component
    record Point(int x, int y) {}
    Object p = new Point(7, 9);
    if (p instanceof Point(int x, _))     // ignore y
        System.out.println("x = " + x);
}

01. Stream Gatherers — windowFixed & windowSliding

Objective: Understand that a gatherer is a customizable intermediate stream operation, and use the two built-in windowing gatherers to batch a stream into non-overlapping and overlapping groups.

Stream Gatherers were finalized in Java 24 (JEP 485; they previewed in 22 and 23). Just as Collector lets you customise the terminal step of a pipeline, a Gatherer lets you customise an intermediate step. You apply one with the new Stream.gather(...) method, and the ready-made ones live in java.util.stream.Gatherers.

Gatherers.windowFixed(n) groups consecutive elements into non-overlapping Lists of size n (the final window may be shorter). Gatherers.windowSliding(n) instead produces overlapping windows that advance one element at a time.

  • A Gatherer is a customizable intermediate operation; apply it via stream.gather(g) (Java 24).
  • windowFixed(n) → consecutive, non-overlapping Lists; a shorter trailing window is still emitted.
  • windowSliding(n) → overlapping windows, sliding by one element each step.
  • Each window is an unmodifiable List<T>; the result type is Stream<List<T>>.
import java.util.List;
import java.util.stream.Gatherers;
import java.util.stream.Stream;

public class WindowDemo {
    public static void main(String[] args) {
        // windowFixed(2): consecutive, NON-overlapping chunks
        List<List<Integer>> fixed = Stream.of(1, 2, 3, 4, 5)
                .gather(Gatherers.windowFixed(2))
                .toList();
        System.out.println(fixed);

        // windowSliding(2): OVERLAPPING windows, step of 1
        List<List<Integer>> sliding = Stream.of(1, 2, 3, 4)
                .gather(Gatherers.windowSliding(2))
                .toList();
        System.out.println(sliding);
    }
}
Console output[[1, 2], [3, 4], [5]] [[1, 2], [2, 3], [3, 4]]

AdvancedGatherers compose: you can chain gather(a).gather(b), and a gatherer can be stateful and even short-circuit the upstream (like limit). Windowing gatherers are inherently sequential over their input order, but the rest of the pipeline around them can still run in parallel.
Pitfall Stream.gather / Gatherers need JDK 24+. On JDK 22–23 they existed only as a preview (compile and run with --enable-preview --release 23); before 22 they do not exist at all.

02. Stream Gatherers — fold, scan, mapConcurrent & custom

Objective: Use the reduction and running-accumulation gatherers, map with bounded concurrency, and write a custom gatherer with Gatherer.ofSequential.

Beyond windowing, java.util.stream.Gatherers (Java 24) ships several more built-ins. fold(init, fn) is a reduction gatherer that consumes the whole stream and emits a single final value. scan(init, fn) is a prefix/running accumulation that emits each intermediate result — a running total, for example. mapConcurrent(maxConcurrency, fn) maps elements using virtual threads with a bounded number in flight, while preserving encounter order in the output.

When none of the built-ins fit, build your own with Gatherer.of(...) (parallel-capable) or Gatherer.ofSequential(...). You supply an initializer for private state and an integrator that inspects each element and decides what (if anything) to push downstream.

  • fold(() -> init, fn) → one final value (a reduction into a single-element stream).
  • scan(() -> init, fn) → each running result; over sums of 1,2,3,4 that is 1,3,6,10.
  • mapConcurrent(k, fn) → at most k concurrent mappings (virtual threads), order preserved.
  • Custom: Gatherer.ofSequential(initializer, integrator); the integrator returns true to keep going.
import java.util.List;
import java.util.stream.Gatherer;
import java.util.stream.Gatherers;
import java.util.stream.Stream;

public class ScanFoldDemo {
    public static void main(String[] args) {
        // scan: running prefix sums -- emits EACH intermediate result
        List<Integer> running = Stream.of(1, 2, 3, 4)
                .gather(Gatherers.scan(() -> 0, (acc, x) -> acc + x))
                .toList();
        System.out.println(running);

        // fold: a reduction -- emits ONE final value (single-element stream)
        List<Integer> folded = Stream.of(1, 2, 3, 4)
                .gather(Gatherers.fold(() -> 0, (acc, x) -> acc + x))
                .toList();
        System.out.println(folded);

        // custom gatherer: emit an element only when it beats the running max
        Gatherer<Integer, ?, Integer> risingMax = Gatherer.ofSequential(
                () -> new int[]{Integer.MIN_VALUE},         // private state
                (state, elem, downstream) -> {
                    if (elem > state[0]) {
                        state[0] = elem;
                        return downstream.push(elem);        // forward new maxima
                    }
                    return true;                             // skip, keep going
                });
        List<Integer> peaks = Stream.of(3, 1, 4, 1, 5, 9, 2, 6)
                .gather(risingMax)
                .toList();
        System.out.println(peaks);
    }
}
Console output[1, 3, 6, 10] [10] [3, 4, 5, 9]

Source: reimplemented as an original illustration.

AdvancedmapConcurrent is ideal for I/O-bound mapping — e.g. urls.stream().gather(Gatherers.mapConcurrent(4, this::fetch)).toList() keeps at most four requests in flight on virtual threads yet still returns results in input order. Note that scan emits N results for N inputs and does not emit the bare seed, whereas fold emits exactly one.
Pitfall Don't expect fold to give you the running values — it collapses to a single element. If you want every intermediate (a prefix sum), reach for scan instead. Both require JDK 24+.

03. The Simple Web Server

Objective: Serve a directory of static files for prototyping using the JDK's built-in web server, both from the command line and programmatically — and know it is a dev tool, not a production server.

The Simple Web Server arrived in Java 18 (JEP 408). Its command-line front end, jwebserver, serves static files from a directory over HTTP with zero configuration — handy for quickly viewing generated docs, a static site, or test fixtures. For control from code, SimpleFileServer.createFileServer(addr, root, level) returns a com.sun.net.httpserver.HttpServer that you .start().

It only serves static GET/HEAD requests, binds to the loopback address by default, and has no TLS, auth, or dynamic handlers. It is explicitly a development and testing aid, never a production server.

  • CLI: jwebserver (Java 18) serves the current directory; -p port, -d directory, -b bind address.
  • Programmatic: SimpleFileServer.createFileServer(...) returns an HttpServer; call .start().

  • Static GET/HEAD only, loopback by default — a prototyping tool, not for production.

04. The modern HttpClient — synchronous

Objective: Perform a blocking HTTP GET with the standard java.net.http.HttpClient, and read the status code and body from the response.

The modern HttpClient was standardized in Java 11 (it incubated in Java 9 as module jdk.incubator.httpclient / package jdk.incubator.http, then moved to java.net.http when standardized in Java 11). It replaces the old HttpURLConnection with a fluent, immutable builder API and supports HTTP/2. The three core types are HttpClient (create once, reuse), HttpRequest (built via HttpRequest.newBuilder()), and HttpResponse<T>.

The synchronous call is client.send(request, bodyHandler): it blocks the calling thread until the full response is available. A BodyHandler decides how the body is materialised — BodyHandlers.ofString() collects it into a String.

  • java.net.http.HttpClient became standard in Java 11 (incubated in Java 9).
  • Create one HttpClient and reuse it; requests are built with an immutable builder.
  • send(...) is blocking; it throws checked IOException and InterruptedException.
  • Read results with response.statusCode() and response.body().
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;

public class SyncGet {
    public static void main(String[] args) throws Exception {
        HttpClient client = HttpClient.newHttpClient();      // reuse this

        HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create("https://api.example.com/movies/1"))
                .GET()                                        // GET is the default
                .build();

        // send() BLOCKS this thread until the response arrives
        HttpResponse<String> response =
                client.send(request, HttpResponse.BodyHandlers.ofString());

        System.out.println("Status: " + response.statusCode());
        System.out.println("Body:   " + response.body());
    }
}

05. The HttpClient — asynchronous

Objective: Issue a non-blocking request with sendAsync, process the eventual response through a CompletableFuture pipeline, and understand why it does not block the caller.

The same HttpClient (Java 11) offers sendAsync(request, bodyHandler), which returns immediately with a CompletableFuture<HttpResponse<String>>. The request runs on the client's executor, and you attach continuations that fire when it completes: .thenApply(HttpResponse::body) transforms the response into its body String, and .thenAccept(System.out::println) consumes it.

Because the calling thread is not blocked, it can start other work while the network round-trip is in flight — the key advantage over the synchronous send. In a short demo you still call .join() at the end so the JVM waits for the callback instead of exiting first.

  • sendAsync(...) (Java 11) returns a CompletableFuture<HttpResponse<String>> right away — non-blocking.
  • Chain .thenApply(...) / .thenAccept(...) to process the response when it arrives.
  • The caller thread is free to do other work meanwhile; nothing blocks until you ask it to.
  • .join() (or .get()) awaits completion — needed in a demo so main doesn't return early.
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.util.concurrent.CompletableFuture;

public class AsyncGet {
    public static void main(String[] args) {
        HttpClient client = HttpClient.newHttpClient();
        HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create("https://api.example.com/movies/1"))
                .build();

        // sendAsync returns immediately -- this thread is NOT blocked
        CompletableFuture<Void> future =
                client.sendAsync(request, HttpResponse.BodyHandlers.ofString())
                      .thenApply(HttpResponse::body)     // HttpResponse<String> -> String
                      .thenAccept(System.out::println);  // print once it arrives

        System.out.println("Request sent; doing other work...");
        future.join();   // block HERE only so the demo waits for the callback
    }
}
Console output (illustrative)Request sent; doing other work... {"id":1,"title":"The Matrix","year":1999}

Source: reimplemented as an original illustration.

AdvancedBecause it returns a CompletableFuture, an async request composes with the whole java.util.concurrent toolkit: .thenCompose(...) chains a dependent second request, .exceptionally(...) handles failures, and CompletableFuture.allOf(f1, f2, f3) fans out many requests and waits for all of them without a thread per call.
Pitfall The "Request sent..." line prints first because sendAsync returns before the response exists; the body prints later, from the callback. Omit .join() in a main that has nothing else to do and the JVM may exit before the callback runs — so nothing prints at all.

06. The Java Platform Module System (JPMS)

Objective: Declare a module with module-info.java, and understand the two guarantees it adds — strong encapsulation and reliable configuration — plus the module path vs the classpath.

The Java Platform Module System shipped in Java 9 (JEP 261). A module is a named, self-describing set of packages declared by a module-info.java file at the source root. Two directives do most of the work: requires names a module this one depends on, and exports names a package this module makes visible to others.

This buys you strong encapsulation — only exports-ed packages are accessible; everything else is truly hidden even to reflection — and reliable configuration: the full module graph is validated when the JVM starts, so a missing or duplicate dependency fails fast rather than as a NoClassDefFoundError deep into a run. Modules are resolved on the module path (--module-path), the modular counterpart to the classpath.

  • JPMS is Java 9; a module is defined by module-info.java at the source root.
  • requires X; declares a dependency; exports pkg; makes exactly that package public.
  • Strong encapsulation: non-exported packages are inaccessible, even via reflection.
  • Reliable configuration: the module graph is checked at startup — missing modules fail fast.
// File: src/com.example.app/module-info.java
module com.example.app {
    requires com.example.util;      // I depend on this module
    exports com.example.app.api;    // others may use ONLY this package
    // (packages not listed here stay private to the module)
}

AdvancedBeyond requires/exports, a module can declare opens pkg; for deep reflective access (used by frameworks like Jackson or Hibernate), and provides ... with ... / uses ... to publish and consume services via ServiceLoader. Plain JARs on the classpath live in the unnamed module, which reads every module — that's how pre-module libraries keep working.
Pitfall Two modules must not export the same package (a split package) — the module system rejects it at startup. And code in another module cannot touch a non-exported package: you get an IllegalAccessError at runtime (or a compile error), not silent access. JPMS needs Java 9+.

07. Multiple modules & transitive dependencies

Objective: Use requires transitive to give consumers implied readability, apply a qualified exports ... to, and know how plain JARs become automatic modules.

When a library's public API exposes types from another module, plain requires is not enough — every consumer would have to re-declare that dependency. requires transitive B solves this: any module that reads A now also reads B automatically. This is implied readability, and it lets a module re-export the API it depends on.

Two more tools round out real multi-module builds. A qualified export, exports pkg to some.module;, reveals a package to only the named modules (useful for internal APIs shared between your own modules). And a plain, non-modular JAR placed on the module path becomes an automatic module whose name is derived from the JAR filename (or its Automatic-Module-Name manifest entry) — the bridge that lets modular code depend on not-yet-modularized libraries.

  • requires transitive B → consumers of this module also read B (implied readability), so you can re-export a dependency's API.
  • exports pkg to m1, m2; → a qualified export visible only to the listed modules.
  • A plain JAR on the module path becomes an automatic module named after the file; on the classpath it joins the unnamed module.
  • Plain requires is not transitive — your consumers must declare their own dependency unless you mark it transitive.

A three-module chain: app requires only core, yet can use util types because core re-exports util transitively.

// ---- module 1 -------------------------------------------------
module com.example.util {
    exports com.example.util;
}

    exports com.example.core.internal to com.example.app;  // qualified export

}

Wrapping up

That is modern Java end to end. The Java 8 features — lambdas, functional interfaces, streams, Optional, default methods, and java.time — reshaped how everyday Java is written: declarative pipelines instead of hand-rolled loops, behaviour passed as values, absence modelled explicitly instead of with null. Everything after Java 8 sharpened the language itself: var and text blocks cut boilerplate; enhanced switch, records, sealed types, and pattern matching turn data modelling and branching into concise, exhaustive, compiler-checked code; stream gatherers extend the pipeline with custom intermediate steps; and the module system, simple web server, and modern HttpClient round out the platform. The throughline: prefer expressions to statements, immutability to mutation, and let the compiler and the standard library carry the weight. Keep an eye on your target JDK — reach for a feature only once the release you deploy on supports it — and the same code will read as a clear description of intent rather than a script of mechanics.
