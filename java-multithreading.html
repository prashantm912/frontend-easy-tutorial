Java Multithreading, Concurrency & Performance Optimization

A practical study guide to writing correct, fast concurrent programs in core Java. It builds up from what a thread is and how the operating system schedules it, through creating and coordinating threads, optimizing for latency and throughput, the stack/heap memory model, and the classic hazards of shared state — race conditions, data races, and deadlocks — then on to the full synchronization toolkit: synchronized and volatile, ReentrantLock and read-write locks, semaphores and condition variables, lock-free CAS algorithms, the blocking vs non-blocking IO threading models, and finally virtual threads. Each lecture pairs a focused Java example with the console output it produces, traced by hand (Java runs on the JVM, not in the browser).

Prerequisites & how to read this You should be comfortable with core Java syntax (classes, methods, generics, lambdas). Examples target JDK 21+ (virtual threads need 21); compile and run any snippet with javac File.java && java File. Concurrency output is inherently non-deterministic — where a trace depends on thread interleaving, it is labelled illustrative and will differ run to run. 1. Introduction

Before writing a single line of concurrent code, it pays to understand why we reach for multiple threads and what a thread actually is. This section motivates multithreading through two goals — responsiveness and performance — and lays the operating-system groundwork: processes, threads, and how the OS scheduler shares CPU time between them.

Run locally Every example in this guide is plain backend Java for JDK 21+ (virtual threads need 21). Save a snippet to a file named after its public class, e.g. Main.java, then javac Main.java && java Main. Virtual-thread demos: java --enable-preview is not needed on JDK 21+.

01. Motivation & Operating Systems Fundamentals — Part 1

Objective: Understand the two reasons we want multithreading — responsiveness and performance — and define what a process is at the operating-system level.

There are only two fundamental drivers behind multithreading. The first is responsiveness: a UI must keep reacting to clicks while it renders, and a server must keep accepting new clients while it is busy handling earlier ones. If a movie player froze every time it decoded a frame, or a web shop ignored a second customer until the first customer's huge order finished writing to the database, users would abandon the product. By handling each task — or each request — on its own thread, the application stays reactive.

The second driver is performance / throughput: a modern CPU has several cores, and a single-threaded program uses only one of them. Splitting a heavy job across threads lets it run in parallel on multiple cores and finish in a fraction of the time. Run a large service and this also means fewer machines and a smaller infrastructure bill.

Two words describe these goals precisely. Latency is how long one operation takes (a single click feeling instant, one request completing quickly). Throughput is how much work completes per unit of time (requests served per second, records processed per minute). Responsiveness is mostly about latency; the performance driver is mostly about throughput. Note that even a single core can improve responsiveness — the OS rapidly interleaves threads to create the illusion of simultaneous progress (concurrency), whereas true parallelism needs multiple cores.

To ground all this, recall what happens when you launch a program. Your applications live on disk as files. When you start one, the OS loads it into memory and creates a running instance called a process (sometimes called the application's context). A process is an isolated container: it owns its own memory and cannot casually touch another process's memory.

  • Two drivers for threads: responsiveness (stay reactive) and performance (use every core).
  • Latency = time for one operation; throughput = work completed per unit time.
  • Concurrency (interleaving) creates the illusion of simultaneity even on one core; parallelism genuinely runs tasks at once on many cores.
  • A process = a program in execution: it owns a heap, the program code, a PID, and open file handles — fully isolated from other processes.
  • Switching the CPU between two processes is comparatively expensive.

AdvancedConcurrency and parallelism are independent axes. Pitfall Adding threads does not automatically make a program faster. If the work is a single sequential chain of dependent steps, extra threads only add scheduling overhead. Threads help when work is genuinely divisible (throughput) or when it spends time waiting on I/O (responsiveness).

02. Operating Systems Fundamentals — Part 2

Objective: Define a thread and what it owns versus shares, understand context switching and OS scheduling, and meet the six Thread.State values.

A thread is the smallest unit the operating system can schedule — the actual stream of instructions running on a CPU. Every process starts with at least one thread (the main thread) and may create more. All the threads of one process share the big, expensive resources: the heap, the loaded code, static data, and open file handles. What each thread keeps to itself is small: its own stack (local variables and the call chain), its own instruction pointer (the address of the next instruction), and its own CPU registers. This makes sense — at any instant each thread is executing a different instruction in a different method, so each needs its own bookmark and scratch space.

There are always more threads than cores, so the OS must run one thread, pause it, run another, pause it, and so on. Saving one thread's registers and kernel state and restoring another's is a context switch. It is not free — it is the price of concurrency. Because threads of the same process share so much, switching between two threads of one process is much cheaper than switching between threads of different processes. Create far too many threads, though, and the OS spends more time switching than doing useful work — a condition called thrashing.

The scheduler decides who runs when. Simple policies have obvious failures: first-come-first-served lets one long task starve everyone; always-shortest-first starves the long tasks. Real schedulers slice time into epochs and give each thread a time slice based on a dynamic priority (a developer-set static priority nudged up or down by the OS), favouring interactive threads that need quick attention while still making progress on compute-heavy ones. Each vendor's exact algorithm differs, so never rely on a specific ordering.

Java models a thread's life as the enum java.lang.Thread.State with six values: NEW (created, not started), RUNNABLE (running or ready to run), BLOCKED (waiting to enter a synchronized monitor), WAITING (waiting indefinitely, e.g. join() / wait()), TIMED_WAITING (waiting with a timeout, e.g. sleep(ms)), and TERMINATED (finished).

  • A thread = the smallest schedulable unit of execution inside a process.
  • Shared per process: heap, code, static data, file handles. Private per thread: stack, instruction pointer, registers.
  • A context switch saves/restores register + kernel state; cheaper between threads than between processes, but never free.
  • Too many threads → thrashing (managing threads instead of running work).
  • Six states in Thread.State: NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, TERMINATED.

    public static void main(String[] args) {
        Thread t = Thread.currentThread();   // the thread running main()

        System.out.println("Name     : " + t.getName());
        System.out.println("Priority : " + t.getPriority());
        System.out.println("State    : " + t.getState());   // it is running, so RUNNABLE
        System.out.println("Alive    : " + t.isAlive());

        System.out.print("All states:");
        for (Thread.State s : Thread.State.values()) {
            System.out.print(" " + s);
        }

    }
}

2. Threading Fundamentals — Thread Creation

Everything about a thread in Java is wrapped by the java.lang.Thread class. In this section you meet the two canonical ways to run code on a new thread — hand a Runnable to a Thread, or extend Thread and override run() — plus the handy per-thread knobs (setName, setPriority, an uncaught-exception handler) and how to inspect live threads in a debugger.

01. Tips about Coding Lectures & Debugging

Objective: Set up an effective way to follow along with the coding lectures and to reason about concurrent code as you run it.

The most valuable habit for this course is to type the examples into a real IDE (IntelliJ IDEA, Eclipse, or VS Code) and run them yourself. Concurrency is hard to picture from static text — watching threads actually interleave is what makes it click.

Two IDE features matter most. First, the debugger: put a breakpoint next to a line number and launch in debug mode. Second, remember that concurrent output is non-deterministic — the OS decides when each thread runs, so line order changes from run to run. Never assume the ordering you happened to see is guaranteed.

  • A cheap tracing trick: print Thread.currentThread().getName() at points of interest.

02. Threads Creation — Part 1: Thread Capabilities & Debugging

Objective: Create a thread by passing a Runnable to new Thread(...), configure it, start it, and give it an uncaught-exception handler.

The first way to run code on another thread: wrap the work in a Runnable and pass it to a Thread constructor. Whatever you put in run() executes on the new thread once the OS schedules it. Configure the thread before starting — setName(...) gives it a meaningful name for debugging, and setPriority(...) takes a value from 1 to 10 (or the constants MIN_PRIORITY, NORM_PRIORITY, MAX_PRIORITY). Calling start() asks the JVM to hand a fresh thread to the OS and returns immediately, so main keeps going on its own.

public class ThreadBasicsDemo {
    public static void main(String[] args) {
        // 1) Wrap the work in a Runnable and hand it to a Thread.
        Thread worker = new Thread(new Runnable() {
            @Override
            public void run() {
                System.out.println("Inside " + Thread.currentThread().getName()
                        + " (priority " + Thread.currentThread().getPriority() + ")");
            }
        });

        // 2) Configure the thread BEFORE starting it.
        worker.setName("Worker");
        worker.setPriority(Thread.MAX_PRIORITY);   // 10

        System.out.println(Thread.currentThread().getName() + ": before start()");
        worker.start();   // hands a new thread to the OS, then returns at once
        System.out.println(Thread.currentThread().getName() + ": after start()");
    }
}
Console output (worker line may land anywhere — order varies)main: before start() main: after start() Inside Worker (priority 10)

Notice both main lines can print before the worker's line: start() is asynchronous, so the worker often hasn't been scheduled yet when main races ahead. That interleaving is illustrative — on another run the worker might print between the two main lines.

Second capability: by default an unchecked exception that escapes run() just kills that one thread. With setUncaughtExceptionHandler(...) you register a callback that fires when an exception is thrown inside the thread and never caught — the place to log context or clean up resources.

public class UncaughtHandlerDemo {
    public static void main(String[] args) {
        Thread worker = new Thread(() -> {
            throw new RuntimeException("Intentional failure");
        });
        worker.setName("Misbehaving");

        // Runs when an exception escapes run() and is caught nowhere.
        worker.setUncaughtExceptionHandler((t, e) ->
                System.out.println("Caught from " + t.getName() + ": " + e.getMessage()));

        worker.start();
    }
}
Console outputCaught from Misbehaving: Intentional failure
  • new Thread(runnable) then start() — run() holds the code to execute.
  • Set name/priority before start(); Thread.currentThread() reads the running thread.
  • start() returns immediately — the caller does not wait for the worker.
  • An uncaught exception terminates only its own thread; a handler lets you react.

AdvancedPriority is only a hint to the OS scheduler and its effect is platform-dependent — some OSes largely ignore Java thread priorities. Never build correctness on priority ordering; use it at most as a soft nudge for responsiveness. You can also install a JVM-wide fallback with the static Thread.setDefaultUncaughtExceptionHandler(...).
Pitfall Calling worker.run() instead of worker.start(). run() is an ordinary method call — it executes the body on the current thread and no new thread is ever created. Always call start().

03. Threads Creation — Part 2: Thread Inheritance

Objective: Create threads the second way — subclass Thread and override run() — and use inheritance to share behaviour across a family of threads.

Because Thread itself implements Runnable, you can skip the separate Runnable object and instead extend Thread, overriding run() directly. A bonus of this style: inside the class, this is the thread, so you call getName(), getPriority(), etc. without going through Thread.currentThread().

The demo: a Vault holds a random password, and two hacker threads race to guess it — one counting up, one counting down — while a PoliceThread counts down ten seconds. Both hackers share common setup (a readable name + max priority) via an abstract HackerThread base class, which also overrides start() to log before delegating to super.start().

public class VaultDemo {
    public static final int MAX_PASSWORD = 9999;

    public static void main(String[] args) {
        Random random = new Random();
        Vault vault = new Vault(random.nextInt(MAX_PASSWORD));

        List<Thread> racers = List.of(
                new AscendingHackerThread(vault),
                new DescendingHackerThread(vault),
                new PoliceThread());

        for (Thread t : racers) {
            t.start();
        }
    }

    static class Vault {
        private final int password;
        Vault(int password) { this.password = password; }

        boolean isCorrectPassword(int guess) {
            try { Thread.sleep(5); } catch (InterruptedException ignored) {}
            return password == guess;
        }
    }

    // Shared behaviour for every hacker: a readable name + top priority.
    abstract static class HackerThread extends Thread {
        protected final Vault vault;

        HackerThread(Vault vault) {
            this.vault = vault;
            setName(getClass().getSimpleName());
            setPriority(Thread.MAX_PRIORITY);
        }

        @Override
        public void start() {
            System.out.println("Starting " + getName());
            super.start();   // MUST delegate, or the thread never runs
        }
    }

    static class AscendingHackerThread extends HackerThread {
        AscendingHackerThread(Vault vault) { super(vault); }

        @Override
        public void run() {
            for (int guess = 0; guess < MAX_PASSWORD; guess++) {
                if (vault.isCorrectPassword(guess)) {
                    System.out.println(getName() + " cracked it: " + guess);
                    System.exit(0);
                }
            }
        }
    }

    static class DescendingHackerThread extends HackerThread {
        DescendingHackerThread(Vault vault) { super(vault); }

        @Override
        public void run() {
            for (int guess = MAX_PASSWORD; guess >= 0; guess--) {
                if (vault.isCorrectPassword(guess)) {
                    System.out.println(getName() + " cracked it: " + guess);
                    System.exit(0);
                }
            }
        }
    }

    static class PoliceThread extends Thread {
        @Override
        public void run() {
            for (int i = 10; i > 0; i--) {
                try { Thread.sleep(1000); } catch (InterruptedException ignored) {}
                System.out.println(i);
            }
            System.out.println("Game over, hackers!");
            System.exit(0);
        }
    }
}
Console output (illustrative — winner & countdown depend on the random password)Starting AscendingHackerThread Starting DescendingHackerThread 10 9 8 7 AscendingHackerThread cracked it: 843
  • Extend Thread, override run(), then new Subclass().start().
  • Inside a Thread subclass, this is the thread — call getName() directly.
  • An abstract base thread lets a family of threads share name/priority setup.
  • Only the two hackers print "Starting…" because only HackerThread overrides start(); the police thread does not.
  • Which hacker wins (and how far the police counts) changes every run — never guaranteed.

Source: reimplemented (and trimmed) from the course "Threads Creation — Part 2" Vault / HackerThread example.

AdvancedPrefer implementing Runnable (composition) over extending Thread (inheritance) in most code: extending Thread spends your single inheritance slot and couples your logic to the thread mechanism, whereas a Runnable can be reused, tested without a thread, and later handed to an ExecutorService. Extend Thread only when you genuinely need to specialise thread behaviour — as with the shared start()/naming here.
Pitfall Overriding start() but forgetting super.start(). Without it, no OS thread is created and run() is never scheduled — the thread silently does nothing.

04. Thread Creation — MultiExecutor — Solution

Objective: Build a small MultiExecutor that holds a list of tasks and runs every one of them concurrently, each on its own thread.

This coding exercise ties the section together. MultiExecutor stores a List<Runnable> and exposes executeAll(), which walks the list, wraps each task in a fresh new Thread(task), and starts it. Because each task gets its own thread, they all run in parallel and their output interleaves unpredictably.

public class MultiExecutor {
    private final List<Runnable> tasks;

    public MultiExecutor(List<Runnable> tasks) {
        this.tasks = tasks;
    }

    /** Runs every task concurrently — one fresh thread per task. */
    public void executeAll() {
        for (Runnable task : tasks) {
            new Thread(task).start();
        }
    }
}

// --- Usage ---
class Demo {
    public static void main(String[] args) {
        List<Runnable> tasks = List.of(
                () -> System.out.println("A on " + Thread.currentThread().getName()),
                () -> System.out.println("B on " + Thread.currentThread().getName()),
                () -> System.out.println("C on " + Thread.currentThread().getName()));

        new MultiExecutor(tasks).executeAll();
    }
}

  • executeAll() = one new Thread(task).start() per task — fully concurrent.
  • Unnamed threads get JVM defaults Thread-0, Thread-1, … in creation order.

AdvancedThis is essentially a hand-rolled executor. 3. Threading Fundamentals — Thread Coordination

Starting a thread is easy; controlling one is the real skill. This section covers how one thread can ask another to stop (interrupt()), how background work is kept from blocking shutdown (daemon threads), and how a thread waits for another to finish and safely collects its result (join()).

01. Thread Termination & Daemon Threads

Objective: Learn why and how to stop a thread from another thread — using interrupt() for cooperative cancellation and setDaemon(true) for background threads that must not block application exit.

A thread costs resources even while idle: memory, some kernel state, a slot in the CPU cache, and CPU time when it runs. There are three common reasons to stop one: to free the resources of a thread that has finished its useful work, to abort a thread that is misbehaving (an unresponsive network call or a runaway calculation), and to let the whole application exit. That last point matters because, by default, the JVM keeps running as long as at least one non-daemon thread is alive — even after main has returned.

Java does not offer a safe "kill" button. Instead, thread.interrupt() sends a request. There are two ways a target thread reacts to it:

  • Blocked threads: if the thread is parked in sleep(), wait(), or join(), the interrupt makes that call throw InterruptedException, which you catch and use to bail out.
  • CPU-bound loops: a thread busy in a compute loop never throws anything, so it must poll Thread.currentThread().isInterrupted() itself and stop when the flag is set.
  • Daemon threads: call setDaemon(true) before start(). The JVM exits as soon as only daemon threads remain, so they never hold up shutdown.
  • Interrupt is cooperative — it only works if the code you run actually responds to the signal.

Variant A — interrupting a blocked thread. The task sleeps for a long time; the interrupt turns that sleep() into a thrown InterruptedException.

public class InterruptBlockingDemo {
    public static void main(String[] args) {
        Thread worker = new Thread(new BlockingTask());
        worker.start();
        worker.interrupt();          // ask the worker to stop

    }

    static class BlockingTask implements Runnable {
        @Override public void run() {
            try {
                Thread.sleep(500_000);   // blocked for ~8 minutes
            } catch (InterruptedException e) {
                System.out.println("Exiting blocking thread");
            }
        }
    }

02. Joining Threads

Objective: Use thread.join() to make one thread block until another completes, so results are read only after they are ready — replacing an inefficient busy-wait and closing a race condition.

Once started, threads run independently and in an order you cannot predict. That is fine until one thread depends on another's output. Suppose the main thread needs the values produced by several worker threads: if it reads them right after starting the workers, it races them and sees partial or empty results. A naive fix is a busy-wait — spinning in a loop checking "are you done yet?" — but that burns CPU cycles and actually steals time from the very worker you are waiting on.

thread.join() is the clean tool: the calling thread goes to sleep and is woken only when the target thread has finished. In this example each FactorialThread computes a factorial as a BigInteger (a long would overflow almost immediately) and records an isFinished flag. The main thread starts all of them, joins each one, then prints results in input order.

  • join() blocks until the target thread terminates; joining every worker guarantees all results are ready.
  • Always defend against a runaway worker: join(2000) waits at most 2000 ms, then returns whether or not the thread finished.
  • After a timed join, check isFinished() before reading the result — the thread may still be running.
  • Reading results in a separate ordered loop keeps output aligned with the input list, regardless of finish order.
import java.math.BigInteger;
import java.util.ArrayList;
import java.util.List;

public class Main {
    public static void main(String[] args) throws InterruptedException {
        List<Long> inputNumbers =
                List.of(100000000L, 3435L, 35435L, 2324L, 4656L, 23L, 5556L);

        List<FactorialThread> threads = new ArrayList<>();
        for (long inputNumber : inputNumbers) {
            threads.add(new FactorialThread(inputNumber));
        }

        for (Thread thread : threads) {
            thread.setDaemon(true);      // let the app exit past a stuck worker
            thread.start();
        }

        for (Thread thread : threads) {
            thread.join(2000);           // wait at most 2 s per worker
        }

        for (int i = 0; i < inputNumbers.size(); i++) {
            FactorialThread ft = threads.get(i);
            if (ft.isFinished()) {
                System.out.println("Factorial of " + inputNumbers.get(i)
                        + " is " + ft.getResult());
            } else {
                System.out.println("The calculation for " + inputNumbers.get(i)
                        + " is still in progress");
            }
        }
    }

    static class FactorialThread extends Thread {
        private final long inputNumber;
        private BigInteger result = BigInteger.ZERO;
        private boolean isFinished = false;

        FactorialThread(long inputNumber) {
            this.inputNumber = inputNumber;
        }

        @Override public void run() {
            this.result = factorial(inputNumber);
            this.isFinished = true;
        }

        private BigInteger factorial(long n) {
            BigInteger temp = BigInteger.ONE;
            for (long i = n; i > 0; i--) {
                temp = temp.multiply(BigInteger.valueOf(i));
            }
            return temp;
        }

        public BigInteger getResult() { return result; }
        public boolean isFinished()   { return isFinished; }
    }

03. Multithreaded Calculation — Solution

Objective: Given a List<Long>, compute each factorial on its own thread, join them all with a timeout, and return the results in input order — the standard fork-then-join pattern in one clean method.

This is the exercise solution, reimplemented from scratch. The key idea is to separate the three phases cleanly: create and start one thread per input, join every thread (with a timeout as insurance against a runaway input), and finally collect results into a list positioned by input index. Collecting in a dedicated ordered loop is what keeps the output aligned with the input, no matter which thread happens to finish first.

  • One thread per input value; all started before any is joined, so they run in parallel.
  • join(2000) bounds the wait; unfinished tasks contribute a sentinel instead of blocking forever.
  • Results are gathered into a List<BigInteger> indexed by input position, preserving order.
  • Daemon workers ensure a stuck computation never keeps the application alive.
import java.math.BigInteger;
import java.util.ArrayList;
import java.util.List;

public class ParallelFactorials {
    public static void main(String[] args) throws InterruptedException {
        List<Long> inputs = List.of(0L, 5L, 10L, 15L, 20L);

        List<BigInteger> results = computeFactorials(inputs);

        for (int i = 0; i < inputs.size(); i++) {
            System.out.println(inputs.get(i) + "! = " + results.get(i));
        }
    }

    static List<BigInteger> computeFactorials(List<Long> inputs)
            throws InterruptedException {
        List<FactorialTask> tasks = new ArrayList<>();
        for (long n : inputs) {
            tasks.add(new FactorialTask(n));
        }

        for (FactorialTask t : tasks) {   // start every task first...
            t.setDaemon(true);
            t.start();
        }
        for (FactorialTask t : tasks) {   // ...then wait for each (bounded)
            t.join(2000);
        }

        List<BigInteger> results = new ArrayList<>();
        for (FactorialTask t : tasks) {   // collect in input order
            results.add(t.isFinished() ? t.getResult() : BigInteger.valueOf(-1));
        }
        return results;
    }

    static class FactorialTask extends Thread {
        private final long n;
        private BigInteger result = BigInteger.ZERO;
        private boolean finished = false;

        FactorialTask(long n) { this.n = n; }

        @Override public void run() {
            result = factorial(n);
            finished = true;
        }

        private BigInteger factorial(long n) {
            BigInteger r = BigInteger.ONE;
            for (long i = 2; i <= n; i++) {
                r = r.multiply(BigInteger.valueOf(i));
            }
            return r;
        }

        BigInteger getResult() { return result; }
        boolean isFinished()   { return finished; }
    }
}

4. Performance Optimization

Multithreading is a means to an end, and the end is performance. This section defines the two metrics that matter most — latency (how fast one task finishes) and throughput (how many tasks finish per second) — and shows how each drives a different design: split one task across cores to cut latency, or run many tasks on a thread pool to raise throughput. Two hands-on examples ground the theory: a parallel image recolorer and a concurrent HTTP search server.

01. Introduction to Performance & Optimizing for Latency — Part 1

Objective: Distinguish the two headline performance metrics, and understand how — and how far — you can lower latency by splitting one task across multiple threads.

"Performance" is context-dependent: a trading system cares about latency, a video player about a steady frame rate, a nightly analytics job about how much data it can chew through. Across all of them, two metrics recur. Latency is the time to finish a single task, measured in units of time. Throughput is the number of tasks finished per unit time (tasks/second). They are independent — improving one may leave the other unchanged, or even hurt it.

To reduce the latency of one task with threads, you split it into N independent subtasks and run them in parallel on N threads. In an ideal world a task that took time T now takes T / N — an N× speedup. Reality falls short for three reasons: (1) part of the work is inherently sequential and cannot be parallelized (Amdahl's law caps the speedup); (2) splitting the input, spawning threads, and aggregating the partial results all cost time; and (3) more threads than cores just thrash the CPU with context switches.

For CPU-bound work (pure computation, no blocking I/O), the optimal thread count is roughly the number of cores: only distinct cores run truly in parallel, and an extra thread merely evicts a running one, adding context-switch and cache-miss overhead. Because tiny tasks are dominated by the fixed splitting cost, only tasks large enough to amortize that overhead are worth parallelizing.

  • Latency = time per task; throughput = tasks per unit time. They are independent metrics.
  • Cut latency by splitting one task into N subtasks run on N threads — ideal speedup N×.
  • Amdahl's law: if fraction p of the work is parallelizable, max speedup is 1 / ((1 - p) + p/N), which is capped by the serial fraction as N grows.
  • For CPU-bound tasks, optimal thread count ≈ number of cores; more threads add overhead, not speed.
  • Splitting + aggregation has a fixed cost, so trivial tasks are not worth parallelizing.
// Amdahl's law: the ceiling on speedup when only a fraction of the
// work can run in parallel. Even infinite cores cannot beat 1/(1 - p).
public class Amdahl {
    static double maxSpeedup(double parallelFraction, int cores) {
        double serial = 1.0 - parallelFraction;
        return 1.0 / (serial + parallelFraction / cores);
    }

    public static void main(String[] args) {
        // 95% of the work parallelizes, 5% is stubbornly sequential.
        System.out.printf("%.2f%n", maxSpeedup(0.95, 8));      // 8 cores
        System.out.printf("%.2f%n", maxSpeedup(0.95, 1000));   // 1000 cores
    }
}

02. Optimizing for Latency — Part 2: Image Processing

Objective: Parallelize a real image-recoloring algorithm by slicing the image into horizontal bands, one thread per band, and measure the drop in wall-clock time.

The task: recolor a JPEG so that every near-white ("shade of gray") flower turns purple, leaving the rest untouched. Each pixel is independent, which makes the work embarrassingly parallel. The strategy is to split the image into N horizontal bands and give each band to its own thread. With height H and N threads, each band is bandHeight = H / N rows tall, and band i starts at row yOrigin = bandHeight * i. Every thread runs the exact same per-pixel routine over its band; the main thread join()s them all before writing the result.

The per-pixel rule: decode the pixel into red/green/blue; if it is a shade of gray (all three channels within 30 of each other), nudge it toward purple — bump red, drop green hard, drop blue a little (clamped to the 0..255 range) — otherwise copy the original color. As the thread count climbs toward the number of physical cores, the bands run genuinely in parallel and wall-clock time falls; past the core count the gains flatten and eventually reverse.

  • Band math: bandHeight = height / numberOfThreads, and band i starts at yOrigin = bandHeight * i.
  • Each thread recolors only its band; independent pixels mean no locking is needed.
  • Fan-out with start(), then fan-in with join() so the file is written only after every band is done.
  • Wall-clock time drops as threads → physical cores, then levels off (hyper-threaded logical cores share hardware units).
  • A single thread through the multithreaded path is slower than the plain sequential version — you pay thread-creation overhead for no parallelism.
import java.awt.image.BufferedImage;
import java.util.ArrayList;
import java.util.List;

public class ParallelRecolor {

    // Split the image into N horizontal bands; one thread per band.
    static void recolorInParallel(BufferedImage src, BufferedImage dst, int threads)
            throws InterruptedException {
        int width      = src.getWidth();
        int bandHeight = src.getHeight() / threads;   // rows per band

        List<Thread> workers = new ArrayList<>();
        for (int i = 0; i < threads; i++) {
            int yOrigin = bandHeight * i;             // top row of this band
            workers.add(new Thread(() ->
                recolorRegion(src, dst, 0, yOrigin, width, bandHeight)));
        }
        for (Thread t : workers) t.start();           // fan out
        for (Thread t : workers) t.join();            // wait for every band
    }

    static void recolorRegion(BufferedImage src, BufferedImage dst,
                              int x0, int y0, int w, int h) {
        for (int x = x0; x < x0 + w && x < src.getWidth();  x++)
            for (int y = y0; y < y0 + h && y < src.getHeight(); y++)
                recolorPixel(src, dst, x, y);
    }

    static void recolorPixel(BufferedImage src, BufferedImage dst, int x, int y) {
        int rgb   = src.getRGB(x, y);
        int red   = (rgb & 0x00FF0000) >> 16;
        int green = (rgb & 0x0000FF00) >> 8;
        int blue  =  rgb & 0x000000FF;

        int newRed = red, newGreen = green, newBlue = blue;
        if (isShadeOfGray(red, green, blue)) {        // a near-white flower
            newRed   = Math.min(255, red + 10);
            newGreen = Math.max(0,   green - 80);
            newBlue  = Math.max(0,   blue - 20);
        }
        int newRgb = 0xFF000000 | (newRed << 16) | (newGreen << 8) | newBlue;
        dst.setRGB(x, y, newRgb);
    }

    static boolean isShadeOfGray(int r, int g, int b) {
        return Math.abs(r - g) < 30 && Math.abs(r - b) < 30 && Math.abs(g - b) < 30;
    }
}

03. Additional Resource — Image Processing, Color Spaces, Extraction & Manipulation

Objective: Understand how a color pixel is packed into a single 32-bit integer, and use bit masks and shifts to extract and re-pack the alpha, red, green, and blue channels.

A pixel's color is stored as one int holding four bytes: Alpha (opacity), Red, Green, Blue, laid out from the most significant byte to the least as 0xAARRGGBB. To read a single channel you mask off the others with a bitwise AND, then shift the surviving byte down to the 0..255 range: red is bits 16–23, green bits 8–15, blue bits 0–7. To build a pixel you reverse it — shift each channel up to its slot and OR them together, forcing the alpha byte to 0xFF so the pixel is fully opaque.

  • One pixel = one 32-bit int, byte layout 0xAARRGGBB.
  • Extract: red = (rgb & 0x00FF0000) >> 16, green = (rgb & 0x0000FF00) >> 8, blue = rgb & 0xFF.
  • Read alpha with an unsigned shift: (rgb >>> 24) & 0xFF — a signed >> would sign-extend for pixels whose top bit is set.
  • Re-pack: 0xFF000000 | (red << 16) | (green << 8) | blue.
  • Each channel is one byte, so it must stay within 0..255 before packing — clamp with Math.min/Math.max.
public class PixelBits {
    public static void main(String[] args) {
        int rgb = 0xFFB428C8;                      // one ARGB pixel (a purple)

        int alpha = (rgb >>> 24) & 0xFF;            // unsigned shift for the top byte
        int red   = (rgb & 0x00FF0000) >> 16;
        int green = (rgb & 0x0000FF00) >> 8;
        int blue  =  rgb & 0x000000FF;
        System.out.printf("A=%d R=%d G=%d B=%d%n", alpha, red, green, blue);

        // Re-pack the same channels back into one int, forcing opaque alpha.
        int packed = 0xFF000000 | (red << 16) | (green << 8) | blue;
        System.out.println(Integer.toHexString(packed));
    }
}

04. Optimizing for Throughput — Part 1

Objective: Learn the throughput strategy — a fixed thread pool draining a task queue — and see why it differs from the latency approach of splitting one task.

When the goal is throughput — completing as many tasks as possible per second — the tasks are usually already independent (separate HTTP requests, separate jobs), so there is nothing to split and nothing to aggregate. Running N tasks in parallel on N threads multiplies throughput up to N×. The clean way to do this is a thread pool: create a fixed set of worker threads once, place incoming tasks on a shared queue, and let each idle worker pull the next task. Threads are reused instead of recreated per task, which removes the per-task creation cost.

This is the mirror image of the latency approach. For latency you took one task and divided it across threads, paying a split-and-merge cost. For throughput you take many tasks and hand each to a pooled thread, with no split and no merge — so throughput scales more cleanly with core count. In Java the ready-made tool is Executors.newFixedThreadPool(n), which bundles n worker threads and an internal unbounded queue; you submit work with execute or submit. Pool size is tuned to the workload — near the core count for CPU-bound work, higher when threads spend time blocked on I/O.

  • Throughput strategy: a fixed pool of worker threads processes independent tasks concurrently from a shared queue.
  • Running N independent tasks in parallel multiplies throughput up to N× — no split/merge overhead as in the latency approach.
  • Executors.newFixedThreadPool(n) = n reusable threads + a queue; add work with submit/execute.
  • If all workers are busy, extra tasks wait in the queue until a thread frees up.
  • Tune the pool size to the workload; reusing threads eliminates per-task creation cost.
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class PoolThroughput {
    public static void main(String[] args) throws InterruptedException {
        // 4 reusable workers drain a queue of 8 independent tasks.
        ExecutorService pool = Executors.newFixedThreadPool(4);

        for (int i = 0; i < 8; i++) {
            int taskId = i;
            pool.submit(() -> {
                String worker = Thread.currentThread().getName();
                System.out.println(worker + " handled task " + taskId);
            });
        }
        pool.shutdown();   // stop accepting; let queued tasks finish
    }
}

05. Optimizing for Throughput — Part 2: HTTP server + JMeter

Objective: Build a concurrent HTTP search server backed by a fixed thread pool, then load-test it with Apache JMeter to see throughput rise as the pool grows.

The application is a tiny search engine. At startup it reads a very large book — Tolstoy's War & Peace, several megabytes — into a string.             long count = 0;

            return count;

5. Data Sharing Between Threads

Threads within one process do not share everything equally. This section maps what lives on each thread's private stack versus the process-wide heap, then shows how sharing a single heap object between two threads produces silently corrupted results the instant an update is not atomic.

01. Stack & Heap Memory Regions

Objective: Understand the two memory regions the JVM uses at runtime and, crucially, which data is private to a thread and which can be reached by many threads at once.

The JVM divides runtime memory into two regions with very different rules. The stack is where methods actually execute. Each thread gets its own stack, allocated with a fixed size when the thread is created. Every method call pushes a stack frame holding that call's arguments and local variables; the frame is popped (invalidated) the moment the method returns — last in, first out, exactly like any stack. Because each thread has a private stack, its local variables are invisible to every other thread.

The heap is a single region shared by the whole process. Everything created with new lives here: your own objects, JDK objects like String, array contents, and collections. Class-level static fields live here too, because a static field is really a member of the class object that itself sits on the heap. The heap is managed by the garbage collector, which reclaims an object once no reference to it remains.

The subtle part is the difference between a reference and the object it points to — they are separate entities. A local reference variable sits on the stack, but the object it addresses is always on the heap. So two threads can each hold a local reference (private, on their own stacks) that points to one and the same shared heap object. Reaching an object through a shared field or a static is exactly where data sharing — and the trouble that follows — begins.

  • Stack: one per thread; holds call frames with local primitives and object references; LIFO; fixed size; freed on method return; not shared.
  • Heap: one per process; holds all new objects, array contents, and static fields; GC-managed; shared by every thread.
  • A primitive passed to a method is copied into the callee's frame; mutating the copy never affects the caller.
  • An object reference can be copied, but all copies still point to one heap object — mutating that object is visible everywhere it is reachable.
  • Deeply nested or unbounded recursion exhausts the fixed stack and throws StackOverflowError.
public class MemoryDemo {

    // static field -> lives on the HEAP (member of the class object)
    static int shipmentsToday = 0;

    public static void main(String[] args) {
        int localCount = 5;                       // primitive local -> main's STACK frame
        int[] inventory = new int[]{ 1, 2, 3 };   // the ARRAY object -> HEAP
                                                  // 'inventory' reference itself -> STACK

        System.out.println("inventory[0] after call: " + inventory[0]);

02. Resource Sharing & Introduction to Critical Sections

Objective: See a minimal shared-resource race first-hand and understand precisely why an innocent-looking items++ corrupts state when two threads run it concurrently.

A resource is anything that holds data or state — a counter, a collection, a file handle, a DB connection, a queue. Any resource on the heap (object fields, static fields) can be shared between threads, and sharing is often the whole point: a save-thread and an edit-thread share one document, a dispatcher hands work to pooled workers through a shared queue, request threads share one database connection pool. Sharing buys us performance and coordination.

But sharing without discipline breaks. Consider a warehouse's InventoryCounter holding an int items. One thread calls increment() 10 000 times (new shipments) and another calls decrement() 10 000 times (sales), both against the same counter object. Intuitively the two exactly cancel and the final count should be 0. Run it unsynchronized and you instead get a different, unpredictable value near zero on every run.

The reason: items++ is not atomic. At the machine level it is a read‑modify‑write triple — (1) load the current value, (2) add one, (3) store the result. When two threads interleave these steps against one variable, they can both read the same stale value, both compute from it, and the last store silently overwrites the other's work — a lost update. The region items++ / items-- that must run start-to-finish without interference is a critical section. This lecture states the problem; the fix (synchronized) arrives in Section 6.

  • Passing one InventoryCounter to both threads makes its items field a shared, mutable resource.
  • items++ compiles to load → add → store — three steps, interruptible between any two.
  • Concurrent interleaving of these steps loses updates, so the total drifts away from the expected 0.
  • An atomic operation appears to the rest of the system as a single all-or-nothing step with no observable intermediate state.
  • The unguarded read-modify-write span is the critical section — it must be made mutually exclusive.
class InventoryCounter {
    private int items = 0;                 // shared heap field

    void increment() { items++; }          // read-modify-write -> NOT atomic
    void decrement() { items--; }          // read-modify-write -> NOT atomic
    int  getItems()  { return items; }
}

public class RaceDemo {
    public static void main(String[] args) throws InterruptedException {
        InventoryCounter counter = new InventoryCounter();   // ONE shared object

        Thread up = new Thread(() -> {
            for (int i = 0; i < 10_000; i++) counter.increment();

        down.start();          // both run at once -> critical section is unguarded

    }

6. The Concurrency Challenges & Solutions

Sharing mutable state between threads is where multithreading gets hard. This section names the problems formally — critical sections, race conditions, data races, and deadlocks — and applies the JVM's two foundational tools to fix them: the synchronized keyword (intrinsic locks) and the volatile keyword (visibility and ordering).

01. Critical Section & Synchronization

Objective: Fix the lost-update bug from the inventory counter by declaring the read-modify-write code a critical section and protecting it with an intrinsic lock, using both the synchronized method form and the synchronized(lock) block form.

A critical section is a run of instructions that must execute as one indivisible unit — no other thread may be partway through the same section at the same time. In the previous section the counter's items++ and items-- were not atomic (read, add, write back), so two threads interleaved and updates were lost. The cure is to make each of those operations mutually exclusive.

Java gives every object a hidden intrinsic lock, also called its monitor. Only one thread can hold a given monitor at a time; any other thread that reaches a synchronized region guarded by that same monitor is suspended until the owner exits. There are two ways to use it. A synchronized method implicitly locks on this. A synchronized (lockObject) block locks on an object you name — logically identical to synchronized (this) { ... } when the object is this, but far more flexible: you can protect just a few lines instead of a whole method, and use separate lock objects for unrelated fields to allow more concurrency.

  • A critical section must run atomically; wrap it so only one thread is inside at a time.
  • A synchronized method locks on this; a synchronized (lock) block locks on the object you pass.
  • The lock is the object's monitor (intrinsic lock) — one owner at a time; others block at the entry point and resume at the exit point.
  • Prefer a small synchronized block over a whole synchronized method: less serialized code means more parallelism.
  • With both increment and decrement mutually exclusive, 10,000 ups and 10,000 downs now cancel to exactly 0.
public class Main {
    public static void main(String[] args) throws InterruptedException {
        InventoryCounter counter = new InventoryCounter();

        Thread up   = new Thread(() -> { for (int i = 0; i < 10_000; i++) counter.increment(); });
        Thread down = new Thread(() -> { for (int i = 0; i < 10_000; i++) counter.decrement(); });

        up.start();  down.start();
        up.join();   down.join();

        System.out.println("We currently have " + counter.getItems() + " items");
    }

    // Form 1 -- synchronized METHOD: every call locks the counter instance ('this').
    static class InventoryCounter {
        private int items = 0;

        public synchronized void increment() { items++; }   // critical section
        public synchronized void decrement() { items--; }   // critical section
        public synchronized int  getItems()  { return items; }
    }

    private final Object lock = new Object();          // a dedicated monitor

    public void increment() { synchronized (lock) { items++; } }
    public void decrement() { synchronized (lock) { items--; } }
    public int  getItems()  { synchronized (lock) { return items; } }

AdvancedIntrinsic locks are reentrant: if a thread already owns a monitor and then enters another synchronized region guarded by the same monitor (for example a synchronized method that calls another synchronized method on the same object), it is let straight through instead of deadlocking against itself. The JVM tracks a per-owner hold count, incrementing it on each re-entry and releasing the lock only when the count returns to zero. Without reentrancy, ordinary recursion or one synchronized method calling another would freeze instantly.
Pitfall Locking on the wrong object gives no protection at all. A synchronized method on instance a and one on instance b use different monitors, so they never exclude each other. Likewise, never lock on a boxed primitive or interned String (e.g. Integer caches, string literals) — a value another part of the program might also lock on. Use a private final Object lock = new Object(); that nothing else can reach.

02. Atomic Operations, Volatile & Metrics

Objective: Learn which operations are already atomic, what volatile guarantees, and combine both in a realistic metrics aggregator where a background thread reads a running average produced by worker threads.

Not everything needs a lock. In Java, reference assignments are atomic, and reads and writes of most primitives (int, short, byte, char, float, boolean) are atomic too. The exceptions are long and double: a 64-bit write may be split into two 32-bit stores, so another thread can observe a torn half-written value. And compound operations are never atomic — x++, or updating two fields together, is read-modify-write no matter the type.

The volatile keyword fixes two separate things. It guarantees visibility: a write by one thread is immediately seen by every other thread (no caching in a register), and it establishes happens-before ordering (a memory barrier) so reads and writes are not reordered around it. As a bonus, marking a long or double field volatile also makes its individual reads and writes atomic — no more tearing.

The metrics class below uses each tool for its purpose. addSample is synchronized because it is a compound update of both count and average that must not interleave. average is volatile so the separate printer thread reads a fresh, non-torn value without taking the lock — the readout never slows the workers down.

  • Atomic already: reference assignment, and reads/writes of primitives except long/double.
  • Never atomic: compound ops like x++ or updating two fields as a unit — these still need synchronized.
  • volatile = visibility + ordering (happens-before), and makes long/double reads/writes atomic.
  • volatile does not make compound operations atomic — it is not a substitute for a lock on read-modify-write.
  • Pattern: guard the compound writer with synchronized; expose the result through a volatile field readable lock-free.
public class Main {
    public static void main(String[] args) {
        Metrics metrics = new Metrics();

        // Deterministic trace of the running average, one sample at a time.
        for (long sample : new long[] { 4, 6, 2, 8 }) {
            metrics.addSample(sample);
            System.out.println("after " + sample + " -> average = " + metrics.getAverage());
        }
    }

    static class Metrics {
        private long count = 0;
        private volatile double average = 0.0;   // read lock-free by the printer thread

        // Compound update of count + average -> must be one critical section.
        public synchronized void addSample(long sample) {
            double currentSum = average * count;  // reconstruct the total so far
            count++;
            average = (currentSum + sample) / count;
        }

        // A single volatile read: safe and fresh without locking.
        public double getAverage() {
            return average;
        }
    }
}

03. Min-Max Metrics — Solution

Objective: Build a thread-safe MinMaxMetrics that tracks the smallest and largest sample seen, correctly handling the empty (no-sample) starting state.

This is the section exercise: extend the metrics idea to record the running minimum and maximum. The trap is the initial state. If you start both fields at 0, the first sample of 5 would wrongly leave min = 0. The fix is to seed min with Long.MAX_VALUE and max with Long.MIN_VALUE, so the very first sample always replaces both.

Every field touched by multiple threads is guarded. addSample is synchronized because it reads and updates min/max together, and the getters are synchronized too so a reader never sees a half-updated pair. (Here long fields are also why we do not rely on plain reads — a lock covers both atomicity and visibility.)

  • Seed min = Long.MAX_VALUE, max = Long.MIN_VALUE so the first real sample overrides the sentinels.
  • addSample is synchronized: it is a compound update of two fields.
  • Getters are synchronized for a consistent, visible read of the shared state.
  • Use Math.min/Math.max to keep the update branch-free and clear.
public class Main {
    public static void main(String[] args) {

        // Empty state: sentinels, before any sample arrives.
        System.out.println("empty: min=" + metrics.getMin() + " max=" + metrics.getMax());

        for (long sample : new long[] { 5, 2, 9, 7, 1 }) {
            metrics.addSample(sample);
        }

        System.out.println("min=" + metrics.getMin());
        System.out.println("max=" + metrics.getMax());

        // Start "inverted" so the first sample replaces both sentinels.

        private long max = Long.MIN_VALUE;

        public synchronized void addSample(long newSample) {
            min = Math.min(min, newSample);
            max = Math.max(max, newSample);
        }

        public synchronized long getMin() { return min; }
        public synchronized long getMax() { return max; }
    }

04. Race Conditions & Data Races

Objective: Distinguish a race condition from a data race, and see how instruction reordering lets a thread observe y > x even though the code always writes x first.

A race condition is a correctness bug where the outcome depends on the timing of threads through a critical section — like the inventory counter, where a decrement scheduled between an increment's read and write loses an update. A data race is different and more subtle: it is concurrent access to a non-volatile, unsynchronized field where at least one thread writes, and the compiler or CPU is free to reorder instructions.

In increment() the source writes x++ then y++. Within a single thread those are independent, so the JIT compiler or the processor may reorder them for performance — the single-threaded result is identical either way. But a second thread running checkForDataRace() can then read the updated y before the updated x and observe y > x — an "impossible" state, purely because the writes were reordered or not made visible in program order.

The fix is to forbid that reordering. Declaring x and y as volatile establishes happens-before ordering (a memory barrier), so the write to x is guaranteed to be visible before the write to y. Alternatively, synchronized on both methods removes concurrency entirely. Since there is no compound update here, volatile is the lighter fix.

  • Race condition: outcome depends on the timing of threads through a critical section (an atomicity problem).
  • Data race: unsynchronized, non-volatile shared access where reordering/visibility breaks program-order assumptions.
  • Reordering is legal because it preserves single-threaded semantics; it only surprises other threads.
  • Fix with volatile (ordering + visibility) or synchronized (mutual exclusion). Here volatile on both fields suffices.
public class Main {
    public static void main(String[] args) {
        SharedClass shared = new SharedClass();

        Thread writer  = new Thread(() -> { while (true) shared.increment(); });
        Thread checker = new Thread(() -> { while (true) shared.checkForDataRace(); });

AdvancedWhy is reordering allowed at all? Compilers and CPUs reorder independent instructions to hide latency — branch prediction, vectorization, prefetching, out-of-order execution when a hardware unit is momentarily busy. 05. Locking Strategies & Deadlocks

Objective: Understand how fine-grained locking introduces deadlock, identify the four Coffman conditions, and prevent deadlock by imposing a global lock-acquisition order.

With several shared resources you choose between coarse-grained locking (one lock for everything — simple, but serializes unrelated work) and fine-grained locking (a lock per resource — more parallelism, but the door to deadlock). A deadlock is a circular wait in which each thread holds a resource the other needs and neither can proceed. In the railway intersection below, takeRoadA locks roadA then roadB, while takeRoadB locks them in the opposite order. Once Train A holds A and Train B holds B, each waits forever for the lock the other owns.

Deadlock requires all four Coffman conditions to hold at once. Break any one and deadlock becomes impossible:

  • Mutual exclusion: a resource is held exclusively (only one train per road).
  • Hold and wait: a thread holds one lock while waiting for another.
  • No preemption: a lock cannot be forcibly taken; the owner must release it voluntarily.
  • Circular wait: a cycle of threads each waiting on the next (A waits on B, B waits on A).

The simplest, most reliable fix targets circular wait: define a single global ordering for the locks and always acquire them in that order everywhere. If every thread takes roadA before roadB, no cycle can form. Note the release order does not matter — only the acquisition order.

public class Main {
    public static void main(String[] args) {
        Intersection intersection = new Intersection();
        new Thread(new Train(intersection, true),  "Train-A").start();  // A-first
        new Thread(new Train(intersection, false), "Train-B").start();  // B-first
    }

    static class Train implements Runnable {
        private final Intersection intersection;
        private final boolean roadAFirst;
        Train(Intersection intersection, boolean roadAFirst) {
            this.intersection = intersection;
            this.roadAFirst = roadAFirst;
        }
        public void run() {
            while (true) {
                if (roadAFirst) intersection.takeRoadA();
                else            intersection.takeRoadB();
            }
        }
    }

    static class Intersection {
        private final Object roadA = new Object();
        private final Object roadB = new Object();

        public void takeRoadA() {
            synchronized (roadA) {                 // order: A then B
                System.out.println("road A locked by " + Thread.currentThread().getName());
                synchronized (roadB) {
                    System.out.println("Train passing on road A");
                }
            }
        }

        public void takeRoadB() {
            synchronized (roadB) {                 // order: B then A (opposite!) -- deadlock risk
                System.out.println("road B locked by " + Thread.currentThread().getName());
                synchronized (roadA) {
                    System.out.println("Train passing on road B");
                }
            }
        }
    }
}

Pitfall Nesting synchronized blocks in whatever order feels natural in each method. The moment two code paths lock the same pair of objects in opposite orders, you have a latent deadlock. Pick a canonical order (e.g. by a stable id, or by System.identityHashCode) and honor it everywhere you nest locks.

7. Advanced Locking

Advanced locking moves past the synchronized keyword to the explicit lock types in java.util.concurrent.locks. ReentrantLock keeps all the mutual-exclusion behaviour of synchronized but adds non-blocking (tryLock) and interruptible acquisition, timeouts, an optional fairness policy, and diagnostic queries — in exchange for remembering to release the lock yourself in a finally block. ReentrantReadWriteLock then splits access into a shared read lock and an exclusive write lock, which greatly speeds up read-heavy shared data.

01. ReentrantLock — Part 1: tryLock & interruptible Lock

Objective: Understand how an explicit ReentrantLock differs from a synchronized block and learn the extra powers it unlocks — tryLock, timed and interruptible acquisition, fairness, and diagnostics.

A ReentrantLock behaves like locking on an object with synchronized, but instead of the JVM acquiring and releasing the monitor for you at the braces of a block, you call lock() and unlock() explicitly. That flexibility comes with a responsibility: if a method returns early or throws inside the critical section, you could skip the unlock() and leave the lock held forever — a classic deadlock source. The mandatory pattern is therefore to acquire the lock, wrap the whole critical section in a try, and put unlock() in a finally so it always runs.

In return you gain operations that synchronized simply cannot express. tryLock() attempts to grab the lock and returns true if it succeeds or false immediately if another thread already holds it — it never blocks. tryLock(timeout, unit) waits only up to a bound. lockInterruptibly() lets a blocked waiter be woken by Thread.interrupt() (a plain lock() ignores interrupts). Passing true to the constructor, new ReentrantLock(true), requests a fair, roughly first-come ordering. Diagnostic methods such as isLocked(), isHeldByCurrentThread(), getHoldCount(), and getQueueLength() help when testing concurrent code (note getQueuedThreads() exists too but is protected, so only a subclass can call it). And like synchronized, the lock is reentrant: the owning thread may lock it again, incrementing a hold count that it must decrement with a matching number of unlock() calls.

  • ReentrantLock requires explicit lock() / unlock() — always release in a finally.
  • tryLock() returns false instead of blocking; tryLock(timeout, unit) waits at most the given time.
  • lockInterruptibly() lets a waiting thread be interrupted out of the wait; a plain lock() cannot.
  • new ReentrantLock(true) enables a fairness policy (at some throughput cost).
  • Reentrant: the holder can re-lock, tracked by a hold count that needs an equal number of unlocks.
  • Diagnostics like isLocked(), getHoldCount(), and getQueueLength() aid testing.
import java.util.concurrent.locks.ReentrantLock;

// tryLock() lets a thread step aside instead of blocking on a busy lock.
public class TryLockDemo {
    private static final ReentrantLock lock = new ReentrantLock();

    public static void main(String[] args) throws InterruptedException {
        // A worker grabs the lock and keeps it busy for 300 ms.
        Thread holder = new Thread(() -> {
            lock.lock();
            try {
                System.out.println("holder: acquired the lock, working...");
                sleep(300);
            } finally {
                lock.unlock();
                System.out.println("holder: released the lock");
            }
        }, "holder");

        holder.start();
        Thread.sleep(50);                 // let the holder win the lock first

        // The main thread refuses to block: tryLock() returns right away.
        if (lock.tryLock()) {
            try {
                System.out.println("main: got the lock");
            } finally {
                lock.unlock();
            }
        } else {
            System.out.println("main: lock busy - skipping, doing other work");
        }

        holder.join();

        // Holder is gone now, so a second tryLock() succeeds.
        if (lock.tryLock()) {
            try {
                System.out.println("main: lock free now, acquired it");
            } finally {
                lock.unlock();
            }
        }
    }

    private static void sleep(long ms) {
        try { Thread.sleep(ms); }
        catch (InterruptedException e) { Thread.currentThread().interrupt(); }
    }
}

AdvancedBecause the lock is reentrant, getHoldCount() reflects how many times the current thread has locked it. If a thread calls lock() three times it must call unlock() three times before any other thread can acquire it. This is exactly what makes a recursive method or a method that calls another synchronized/locked method on the same lock safe from self-deadlock. lockInterruptibly() is the tool for a watchdog: a supervisor thread can interrupt() a worker that is stuck waiting on a lock so it wakes up in the catch (InterruptedException) branch, runs cleanup, and exits gracefully — enabling deadlock recovery or a clean shutdown that a plain blocking lock() would prevent.
Pitfall Unconditionally calling unlock() after a tryLock(). If tryLock() returned false you do not hold the lock, and unlocking it throws IllegalMonitorStateException. Only unlock inside the if (lock.tryLock()) { ... } branch. Likewise, never put lock() itself inside the try whose finally unlocks — acquire first, then open the try, so a failed acquisition never leads to unlocking a lock you do not own.

02. ReentrantLock — Part 2: User Interface Application example

Objective: See why a real-time thread — here a JavaFX render loop — should read shared state with tryLock() rather than a blocking lock(), so the UI never freezes.

Consider a dashboard showing live cryptocurrency prices. Two threads share a PricesContainer: a background PriceUpdater fetches fresh prices and writes all five at once, and the JavaFX UI thread redraws the labels many times per second. The five prices must update as a single atomic snapshot — a user must never see a new Bitcoin price next to a stale Ether price — so both sides guard the container with one ReentrantLock.

The write side uses a plain lock(): the updater is a background thread, so blocking briefly is fine. The read side is different. The UI thread runs a tight render loop and must never block, or the whole interface freezes (the dreaded spinning-wheel). So in the per-frame callback it calls tryLock(): if the updater happens to hold the lock this instant, tryLock() returns false and the frame simply re-draws the previous prices and moves on. The UI stays smooth; at worst a single frame shows values that are a few milliseconds old. Swapping that tryLock() for a blocking lock() is what makes the app hang.

  • Writer thread: blocking lock() is fine — it writes all five prices atomically, then unlocks.
  • UI / render thread: tryLock() so a busy lock never blocks the frame; skip and redraw old data instead.
  • One lock guards the whole snapshot, preventing a torn read of mixed old and new prices.
  • General rule: any real-time thread (UI, audio, video, low-latency trading) should prefer tryLock over blocking.

03. Reentrant Read Write Lock & Database Implementation

Objective: Use a ReentrantReadWriteLock to let many reader threads work concurrently on read-heavy shared data, and measure the speed-up over a single exclusive lock.

Every lock so far grants access to just one thread at a time, whether it reads or writes. But readers do not conflict with one another — several threads can safely read the same data at once, as long as nobody is mutating it. For a read-heavy workload (a cache, an in-memory index) forcing readers to take turns wastes a lot of parallelism. ReentrantReadWriteLock solves this by exposing two locks: a readLock() that any number of readers may hold simultaneously, and a writeLock() that is fully exclusive — while a writer holds it, no reader and no other writer may enter, and a writer must wait until the last reader releases.

The example is an InventoryDatabase backed by a TreeMap<Integer, Integer> mapping a price to the number of items at that price. A range query walks the sub-map between two prices and sums the counts — a read, guarded by the read lock, so all seven reader threads run in parallel. addItem and removeItem mutate the tree and take the write lock. With one plain ReentrantLock guarding all three methods, the readers serialize and the benchmark takes roughly 3000 ms; switching to a read-write lock lets the readers overlap and cuts that to around 1000 ms — about three times faster.

  • ReentrantReadWriteLock gives a shared readLock() (many holders) and an exclusive writeLock() (one holder).
  • Readers and writers are mutually exclusive; multiple readers are not.
  • Ideal for read-heavy shared data — reads overlap instead of serializing.
  • The read-write lock object itself provides no locking; you lock/unlock its two inner locks.
  • On this benchmark, RW lock (~1000 ms) beats a single exclusive lock (~3000 ms) roughly 3x.
import java.util.*;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ReadWriteLockDemo {
    static final int HIGHEST_PRICE = 1000;

    // price -> number of items at that price. Read-heavy shared state.
    static class InventoryDatabase {
        private final TreeMap<Integer, Integer> priceToCount = new TreeMap<>();
        private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
        private final Lock readLock  = rwLock.readLock();   // shared: many readers at once
        private final Lock writeLock = rwLock.writeLock();  // exclusive: one writer, no readers

        // READ side: guarded by the read lock, so all 7 readers overlap.
        public int itemsInPriceRange(int lowerBound, int upperBound) {
            readLock.lock();
            try {
                Integer fromKey = priceToCount.ceilingKey(lowerBound);
                Integer toKey   = priceToCount.floorKey(upperBound);
                if (fromKey == null || toKey == null) return 0;
                NavigableMap<Integer, Integer> range =
                        priceToCount.subMap(fromKey, true, toKey, true);
                int sum = 0;
                for (int count : range.values()) sum += count;
                return sum;
            } finally {
                readLock.unlock();
            }
        }

        // WRITE side: guarded by the write lock, exclusive of all readers and writers.
        public void addItem(int price) {
            writeLock.lock();
            try {
                priceToCount.merge(price, 1, Integer::sum);
            } finally {
                writeLock.unlock();
            }
        }

        public void removeItem(int price) {
            writeLock.lock();
            try {
                Integer count = priceToCount.get(price);
                if (count == null) return;
                if (count == 1) priceToCount.remove(price);
                else            priceToCount.put(price, count - 1);
            } finally {
                writeLock.unlock();
            }
        }
    }

    public static void main(String[] args) throws InterruptedException {
        InventoryDatabase db = new InventoryDatabase();
        Random random = new Random();
        for (int i = 0; i < 100_000; i++) db.addItem(random.nextInt(HIGHEST_PRICE));

        // One writer keeps mutating the tree in the background.
        Thread writer = new Thread(() -> {
            while (true) {
                db.addItem(random.nextInt(HIGHEST_PRICE));
                db.removeItem(random.nextInt(HIGHEST_PRICE));
                try { Thread.sleep(10); } catch (InterruptedException e) { return; }
            }
        });
        writer.setDaemon(true);
        writer.start();

        // Seven readers hammer the range query as fast as they can.
        List<Thread> readers = new ArrayList<>();
        for (int r = 0; r < 7; r++) {
            Thread reader = new Thread(() -> {
                for (int i = 0; i < 100_000; i++) {
                    int upper = random.nextInt(HIGHEST_PRICE);
                    int lower = upper > 0 ? random.nextInt(upper) : 0;
                    db.itemsInPriceRange(lower, upper);
                }
            });
            reader.setDaemon(true);
            readers.add(reader);
        }

        long start = System.currentTimeMillis();
        for (Thread reader : readers) reader.start();
        for (Thread reader : readers) reader.join();
        System.out.println("Reading took " + (System.currentTimeMillis() - start) + " ms");
    }
}

AdvancedA ReentrantReadWriteLock supports lock downgrading: a thread holding the write lock may acquire the read lock and then release the write lock, moving to a shared read without ever letting go. The reverse — upgrading from read to write — is not allowed and will deadlock, because the writer would have to wait for all readers (including itself) to release the read lock. Construct the lock with new ReentrantReadWriteLock(true) for a fair ordering that reduces the risk of writer starvation when readers arrive continuously.
Pitfall Assuming a read-write lock is always faster. Its bookkeeping is heavier than a plain lock, so for write-heavy or very short critical sections it can be slower than a single ReentrantLock. It only pays off when reads genuinely dominate and are not trivially cheap — measure your workload rather than adopting it reflexively.

04. Product Reviews Service — Solution

Objective: Apply the read-write lock to a small, realistic service — a product-reviews store where posting is rare and reading the average rating is frequent.

Product reviews are a textbook read-heavy workload: many shoppers read a product's average rating for every one who posts a new review. A ProductReviewsService keeps a Map from product id to a list of ratings and guards it with a ReentrantReadWriteLock. addReview(productId, rating) mutates the map, so it takes the exclusive write lock. getAverageRating(productId) only inspects the data, so it takes the shared read lock, allowing any number of concurrent readers while still being fully excluded whenever a review is being written.

  • Write operation (addReview) takes the write lock; read operation (getAverageRating) takes the read lock.
  • Concurrent average-rating reads overlap; a review post briefly excludes all of them.
  • computeIfAbsent creates the per-product list on first review under the write lock.
  • Never leak the internal list outside the lock, or a reader could see it change mid-iteration.
import java.util.*;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ProductReviewsService {
    // productId -> star ratings. Read-heavy: many reads, few writes.
    private final Map<String, List<Integer>> reviews = new HashMap<>();
    private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock  = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();

    // Posting a review mutates the map -> exclusive write lock.
    public void addReview(String productId, int rating) {
        writeLock.lock();
        try {
            reviews.computeIfAbsent(productId, id -> new ArrayList<>()).add(rating);
        } finally {
            writeLock.unlock();
        }
    }

    // Computing the average only reads -> shared read lock (many at once).
    public double getAverageRating(String productId) {
        readLock.lock();
        try {
            List<Integer> ratings = reviews.get(productId);
            if (ratings == null || ratings.isEmpty()) return 0.0;
            long sum = 0;
            for (int r : ratings) sum += r;
            return (double) sum / ratings.size();
        } finally {
            readLock.unlock();
        }
    }

    public static void main(String[] args) {
        ProductReviewsService service = new ProductReviewsService();
        service.addReview("book-42", 5);
        service.addReview("book-42", 4);
        service.addReview("book-42", 3);
        service.addReview("phone-7", 2);

        System.out.printf("book-42 average: %.2f%n", service.getAverageRating("book-42"));
        System.out.printf("phone-7 average: %.2f%n", service.getAverageRating("phone-7"));
        System.out.printf("tv-9 average:    %.2f%n", service.getAverageRating("tv-9"));
    }
}

8. Inter-Thread Communication

Locks stop threads from stepping on each other; communication primitives let one thread wait until another has done something. This section covers four signalling tools — counting Semaphores, Lock-based condition variables, every object's intrinsic wait()/notify() monitor, and the one-shot CountDownLatch — and applies them to the classic bounded producer-consumer problem.

01. Semaphore — Scalable Producer-Consumer

Objective: Use a counting semaphore as a permit-issuing gate, and build a bounded producer-consumer with two semaphores so producers can never outrun consumers.

A java.util.concurrent.Semaphore is a counter guarding a bounded resource. It is created with a number of permits; acquire() takes one permit (blocking while none are free), and release() returns one. Think of a car park with eight bays: eight permits are issued, the ninth car waits, and any leaving car frees a bay.

Unlike a lock, a semaphore has no owner: permits can start above 1, and a permit released by one thread can be acquired by a different thread. That "release from anywhere" property is exactly what makes it a communication tool. The bounded queue below uses two semaphores — full counts items available (starts at 0) and empty counts free slots (starts at the capacity). The producer acquires an empty slot before adding and releases a full item after; the consumer does the mirror image.

  • acquire() blocks until a permit is free; release() hands one back and can wake a blocked thread.
  • Two counting semaphores model the two scarce quantities: items to consume (full) and free space to produce into (empty).
  • A short lock still guards the queue's internal structure — the semaphores handle flow control, not mutual exclusion of the data structure itself.
  • If the consumer is slower, the producer blocks on empty.acquire(): automatic back-pressure, no busy-waiting, no unbounded memory growth.
import java.util.concurrent.Semaphore;
import java.util.LinkedList;
import java.util.Queue;

public class SemaphoreProducerConsumer {
    static final int CAPACITY = 3;
    static final Queue<Integer> queue = new LinkedList<>();
    static final Semaphore full  = new Semaphore(0);        // items available
    static final Semaphore empty = new Semaphore(CAPACITY); // free slots
    static final Object lock = new Object();                // guards the queue

    public static void main(String[] args) throws InterruptedException {
        Thread producer = new Thread(() -> {
            for (int i = 1; i <= 5; i++) {
                try {
                    empty.acquire();                        // wait for a free slot
                    synchronized (lock) { queue.add(i); }
                    System.out.println("produced " + i);
                    full.release();                         // one more item ready
                } catch (InterruptedException e) { return; }
            }
        });

        Thread consumer = new Thread(() -> {
            for (int i = 1; i <= 5; i++) {
                try {
                    full.acquire();                         // wait for an item
                    int value;
                    synchronized (lock) { value = queue.remove(); }
                    empty.release();                        // one more free slot
                    System.out.println("consumed " + value);
                } catch (InterruptedException e) { return; }
            }
        });

        producer.start();
        consumer.start();
        producer.join();
        consumer.join();
    }
}

02. Condition Variables — All-purpose Inter-Thread Communication

Objective: Coordinate threads with Lock.newCondition() — the explicit-lock equivalent of wait()/notify() — and learn why the predicate is always re-checked in a loop.

A condition variable is always tied to a lock. From a ReentrantLock you create one or more Condition objects with lock.newCondition(). While holding the lock, a thread calls await(): this atomically releases the lock and parks the thread, letting another thread take the lock, change state, and call signal() (wake one waiter) or signalAll() (wake all). The woken thread must re-acquire the lock before it returns from await().

Because a thread can wake spuriously, and because signalAll() may wake several waiters that then race for the lock, you must always re-test the predicate in a while loop, never a bare if. Separate conditions for separate predicates (here notFull and notEmpty) let you signal exactly the threads that could now make progress.

  • await() releases the lock and sleeps; on wake it silently re-acquires the lock before continuing.
  • Guard every await() with while (!predicate) — spurious wakeups and multi-waiter races make if unsafe.
  • One lock can own several conditions; notFull.signal() targets producers, notEmpty.signal() targets consumers.
  • Always await()/signal() while holding the owning lock, and unlock in a finally block.
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;
import java.util.LinkedList;
import java.util.Queue;

public class BoundedBuffer {
    private final Queue<Integer> queue = new LinkedList<>();
    private final int capacity = 3;
    private final Lock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    public void put(int value) throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == capacity) {  // re-check in a loop
                notFull.await();                // release lock, park
            }
            queue.add(value);
            notEmpty.signal();                  // wake one waiting consumer
        } finally {
            lock.unlock();
        }
    }

    public int take() throws InterruptedException {
        lock.lock();
        try {
            while (queue.isEmpty()) {           // re-check in a loop
                notEmpty.await();
            }
            int value = queue.remove();
            notFull.signal();                   // wake one waiting producer
            return value;
        } finally {
            lock.unlock();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        BoundedBuffer buffer = new BoundedBuffer();
        Thread producer = new Thread(() -> {
            for (int i = 1; i <= 4; i++) {
                try { buffer.put(i); System.out.println("put  " + i); }
                catch (InterruptedException e) { return; }
            }
        });
        Thread consumer = new Thread(() -> {
            for (int i = 1; i <= 4; i++) {
                try { System.out.println("took " + buffer.take()); Thread.sleep(30); }
                catch (InterruptedException e) { return; }
            }
        });
        producer.start();
        consumer.start();
        producer.join();
        consumer.join();
    }
}

03. Objects as Condition Variables — wait(), notify() and notifyAll()

Objective: Use any object's intrinsic monitor to build a bounded, thread-safe queue with wait()/notify(), the foundation of the course's matrix producer-consumer.

Every Java object inherits wait(), notify() and notifyAll() from Object, and every object has an intrinsic monitor usable as a lock via synchronized. Inside a synchronized block, wait() releases that monitor and parks the thread; another thread entering the same monitor can change state and call notify() (wake one arbitrary waiter) or notifyAll() (wake them all). It is Condition without a separate Lock object — the object is both lock and condition.

The course example wires a matrix producer (reads pairs of 10×10 matrices from a file) to a consumer (multiplies each pair and writes the result) through a shared bounded queue. Trimmed to its essentials, the queue below caps at capacity 5: add() waits while full, remove() waits while empty, and both re-test their predicate in a while loop. A terminate() flag lets the producer tell consumers "no more input" so they can drain and exit.

  • Call wait()/notify()/notifyAll() only while holding that object's monitor (inside its synchronized block), or you get IllegalMonitorStateException.
  • Always loop on the condition — while (queue.size() == CAPACITY) wait(); — never a single if.
  • Prefer notifyAll() when distinct kinds of waiters (producers and consumers) share the same monitor; plain notify() might wake the wrong one and stall.
  • The queue bounds itself, giving the fast producer back-pressure so memory can't blow up.
import java.util.LinkedList;
import java.util.Queue;

// Bounded, thread-safe hand-off queue between a producer and consumers.
class ThreadSafeQueue {
    private final Queue<MatrixPair> queue = new LinkedList<>();
    private static final int CAPACITY = 5;
    private boolean terminated = false;

    public synchronized void add(MatrixPair pair) {
        while (queue.size() == CAPACITY) {          // full: producer waits
            try { wait(); } catch (InterruptedException e) { return; }
        }
        queue.add(pair);
        System.out.println("queue size " + queue.size());
        notifyAll();                                // wake a waiting consumer
    }

    public synchronized MatrixPair remove() {
        while (queue.isEmpty() && !terminated) {     // empty: consumer waits
            try { wait(); } catch (InterruptedException e) { return null; }
        }
        if (queue.isEmpty() && terminated) {
            return null;                            // drained and no more input
        }
        MatrixPair pair = queue.remove();
        notifyAll();                                // wake a waiting producer
        return pair;
    }

    public synchronized void terminate() {
        terminated = true;
        notifyAll();                                // wake every consumer to exit
    }
}

04. Simple CountDownLatch — Solution

Objective: Use CountDownLatch as a one-shot "wait for N tasks" gate — a coordinator blocks until every worker reports done.

A java.util.concurrent.CountDownLatch(n) starts at count n. Worker threads call countDown() as they finish, decrementing the count; a coordinator calls await(), which blocks until the count reaches zero, then proceeds. It is a single-use gate: once the count hits zero it stays there and every subsequent await() returns immediately.

This cleanly separates "start N pieces of work" from "continue only after all of them complete", without the coordinator polling flags or joining each thread individually.

  • Construct with the number of events to wait for; each countDown() subtracts one.
  • await() blocks until the count is 0; if it is already 0 it returns at once.
  • One-shot — the count cannot be reset. Need a reusable barrier? Use CyclicBarrier.
  • Put countDown() in a finally block so a failing worker still releases the latch instead of hanging the coordinator forever.
import java.util.concurrent.CountDownLatch;

public class CountDownLatchDemo {
    public static void main(String[] args) throws InterruptedException {
        int workers = 3;
        CountDownLatch latch = new CountDownLatch(workers);

        for (int i = 1; i <= workers; i++) {
            final int id = i;
            new Thread(() -> {
                System.out.println("worker " + id + " started");
                try {
                    Thread.sleep(50L * id);                 // simulate work
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    System.out.println("worker " + id + " done");
                    latch.countDown();                      // signal completion
                }
            }).start();
        }

        System.out.println("coordinator: waiting for workers");
        latch.await();                                      // block until count == 0
        System.out.println("coordinator: all workers finished");
    }
}

9. Lock-Free Algorithms, Data Structures & Techniques

Locks solve almost every concurrency problem, but they carry real costs: deadlock, convoy effects, priority inversion, and the pause that hits every waiter when a lock holder is descheduled. This section explores the alternative — lock-free code built on atomic hardware instructions, chiefly Compare-And-Swap (CAS). We use the java.util.concurrent.atomic classes to fix a race without a lock, then build a genuinely lock-free stack that scales far better than a synchronized one under contention.

01. Introduction to Non-blocking, Lock-Free operations

Objective: Understand why lock-based (blocking) code can misbehave, and how atomic hardware instructions — above all Compare-And-Swap — let threads make progress without mutual-exclusion locks.

Everything learned about locks still holds: most concurrent code is written with them, and they are the simplest, safest tool for the majority of problems. But locks are not free. A blocking critical section — one guarded by synchronized or a Lock — exposes you to several hazards. Deadlock can halt the whole application and is unrecoverable unless you built detection logic. A single long critical section drags every thread down to the speed of the slowest, because only one may enter at a time. Priority inversion strikes when a low-priority thread holds a lock the OS-favored high-priority thread needs, and the low-priority thread is not scheduled soon enough to release it. Worst of all, a thread that dies, is interrupted, or simply forgets to unlock while holding a lock leaves every other thread blocked forever. And even in the happy path there is overhead: a thread that fails to acquire a lock is context-switched out and later switched back in — negligible for most apps, but ruinous for sub-millisecond systems such as high-frequency trading.

The root cause of the race that made us reach for a lock is that a single Java statement compiles to several hardware instructions. count++ is really read the value, compute value + 1, then store it back — and another thread can slip in between those steps. Lock-free programming sidesteps this by using operations that the CPU guarantees to execute as one indivisible instruction, and a single hardware instruction is atomic — hence thread-safe — by definition. The pivotal one is Compare-And-Swap: "if this memory still equals the value I previously read, atomically replace it with my new value; otherwise report failure so I can retry." Because a failed CAS just loops and tries again rather than parking the thread, lock-free algorithms guarantee system-wide progress — some thread always advances — with no mutual exclusion. Java surfaces CAS through java.util.concurrent.atomic (AtomicInteger, AtomicLong, AtomicReference, and friends) and the compareAndSet method.

  • Lock hazards: deadlock, threads slowed to the slowest holder, priority inversion, and a dead/forgetful holder leaving everyone blocked forever.
  • Races arise because one Java op (count++) is many hardware ops; another thread interleaves between read, compute, and store.
  • CAS = "compare the current value to what I expected; if equal, set the new value atomically; if not, fail and let me retry."
  • Lock-free code guarantees system-wide progress without mutual exclusion — a failed CAS retries instead of parking the thread.
  • Java exposes CAS via java.util.concurrent.atomic classes and their compareAndSet method, compiled to a single atomic CPU instruction.
import java.util.concurrent.atomic.AtomicInteger;

// A hand-written lock-free increment, spelled out as a CAS retry loop.
// This is essentially what AtomicInteger.incrementAndGet() does internally.
public class CasLoopDemo {
    public static void main(String[] args) {
        AtomicInteger value = new AtomicInteger(0);

        int result = incrementLockFree(value);
        System.out.println("After one lock-free increment: " + result);
        System.out.println("Stored value is now:           " + value.get());
    }

    // "Read, compute off to the side, then publish only if nobody moved it."
    static int incrementLockFree(AtomicInteger value) {
        int current, next;
        do {
            current = value.get();   // 1. read the value we assume is stable
            next    = current + 1;   // 2. compute the candidate new value
        } while (!value.compareAndSet(current, next)); // 3. CAS; loop if it moved
        return next;
    }
}

Advanced"Lock-free" and "wait-free" are distinct guarantees. Lock-free means the system as a whole always makes progress: at every moment some thread completes an operation, so no thread can block all others — but an unlucky, heavily-contended thread might retry its CAS loop many times. Wait-free is stronger: every thread finishes in a bounded number of steps regardless of contention, with no unbounded retry loops. Most java.util.concurrent.atomic algorithms are lock-free; a few operations (such as getAndIncrement on modern CPUs with a native fetch-and-add instruction) can be effectively wait-free.
Pitfall Assuming lock-free means "faster, always." Under low or no contention a plain variable, or a lock, can beat an atomic, because CAS involves memory-ordering fences and cache-line traffic. Lock-free shines under contention and where you must avoid blocking (e.g. cannot tolerate a descheduled holder). Reach for it deliberately, not reflexively.

02. Atomic Integers & Lock-Free E-Commerce

Objective: Replace a synchronized counter with an AtomicInteger and see the same inventory race resolved with lock-free atomic updates.

AtomicInteger wraps an int and offers a family of atomic operations, each compiled down to a CAS-based hardware update with no lock involved. Construct it with an initial value, then use incrementAndGet() / decrementAndGet() (which return the new value) or getAndIncrement() / getAndDecrement() (which return the previous value). addAndGet(delta) and getAndAdd(delta) add any amount — the delta may be negative — and get() reads the current value. Every one of these is a single atomic step, so no synchronized keyword is needed.

Recall the e-commerce inventory counter from the race-condition lectures: one thread increments the stock 10,000 times while another decrements it 10,000 times. With a plain int and items++ / items--, the read-modify-write interleaves and the final total is unpredictable — so the original fix wrapped both methods in synchronized. Swapping the field to an AtomicInteger and calling incrementAndGet() / decrementAndGet() makes each update atomic on its own, the lock disappears, and because the two threads perform exactly equal and opposite work the final count is deterministically 0.

  • incrementAndGet()/decrementAndGet() return the new value; getAndIncrement()/getAndDecrement() return the old one — all atomic, no lock.
  • addAndGet(delta)/getAndAdd(delta) add any (possibly negative) amount atomically; get() reads the current value.
  • Equal increments and decrements by two threads leave the counter at exactly 0 — the race is gone with no synchronized.
  • Atomics are ideal for counting and metric aggregation across threads; AtomicLong and AtomicBoolean offer the same API for other types.
  • If a value is only ever touched by one thread, a plain int is preferable — atomicity has a cost and disables some compiler/CPU optimizations.

03. Atomic References, Compare-And-Set, Lock-Free High-Performance Data Structure

Objective: Use AtomicReference<T> and compareAndSet to build a lock-free stack, and see it outperform an equivalent synchronized stack under contention.

AtomicReference<T> holds a reference to an object and lets you operate on it atomically: get() reads it, set(x) replaces it, and — the workhorse of every lock-free structure — compareAndSet(expected, newValue) swaps in newValue only if the current reference is still expected, returning true on success and false (changing nothing) otherwise. It compiles to a single atomic CPU instruction; most other atomic methods are built on top of it.

A stack is a linked list whose head is the top. push wraps the value in a new node, points that node's next at the current head, and makes it the new head; pop returns the head's value and moves the head to next. With a plain reference this has a race: between reading the head and writing it back, another thread can change it. The classic fix makes both methods synchronized — the blocking StandardStack. The lock-free version instead stores the head in an AtomicReference<StackNode<T>> and wraps each update in a CAS retry loop: read the current head, build the candidate, then head.compareAndSet(current, candidate); if it fails, some other thread moved the head first, so back off with LockSupport.parkNanos(1), re-read, and try again. No lock is ever taken, and under heavy contention from many pushers and poppers this scales dramatically better.

  • AtomicReference<T>: get() / set() / compareAndSet(expected, newValue) — CAS is the building block of lock-free structures.
  • Lock-free push: build a new node whose next is the current head, then CAS the head from current to new; retry on failure.
  • Lock-free pop: read the head, then CAS the head to head.next; retry on failure; return the old head's value.
  • On a failed CAS the thread does not block — it briefly parks (parkNanos(1)), re-reads, and loops.
  • Under contention the lock-free stack performs several times more operations than the synchronized one, because no thread is ever blocked waiting for a lock.
import java.util.ArrayList;
import java.util.List;
import java.util.Random;
import java.util.concurrent.atomic.AtomicInteger;
import java.util.concurrent.atomic.AtomicReference;
import java.util.concurrent.locks.LockSupport;

public class LockFreeStackBench {

    // ---- The lock-free stack: no synchronized, only CAS retry loops ----
    public static class LockFreeStack<T> {
        private final AtomicReference<StackNode<T>> head = new AtomicReference<>();
        private final AtomicInteger counter = new AtomicInteger(0);

        public void push(T value) {
            StackNode<T> newHead = new StackNode<>(value);
            while (true) {
                StackNode<T> currentHead = head.get();  // read the current top
                newHead.next = currentHead;             // link new node to it
                // Publish only if the top is still what we read a moment ago.
                if (head.compareAndSet(currentHead, newHead)) {
                    break;                              // won the race: done
                }
                LockSupport.parkNanos(1);               // lost: back off, retry
            }
            counter.incrementAndGet();
        }

        public T pop() {
            StackNode<T> currentHead = head.get();
            while (currentHead != null) {
                StackNode<T> nextHead = currentHead.next;
                if (head.compareAndSet(currentHead, nextHead)) {
                    break;                              // detached the old top
                }
                LockSupport.parkNanos(1);
                currentHead = head.get();               // re-read and try again
            }
            counter.incrementAndGet();
            return currentHead != null ? currentHead.value : null;
        }

        public int getCounter() { return counter.get(); }
    }

    private static class StackNode<T> {
        final T value;
        StackNode<T> next;
        StackNode(T value) { this.value = value; }   // next defaults to null
    }

    // ---- Benchmark: 2 pushers + 2 poppers hammer the stack for 10 seconds ----
    public static void main(String[] args) throws InterruptedException {
        LockFreeStack<Integer> stack = new LockFreeStack<>();
        Random random = new Random();

        for (int i = 0; i < 100_000; i++) {      // pre-fill so poppers find items
            stack.push(random.nextInt());
        }

        List<Thread> threads = new ArrayList<>();
        for (int i = 0; i < 2; i++) {            // 2 pushing threads
            Thread t = new Thread(() -> { while (true) stack.push(random.nextInt()); });
            t.setDaemon(true);
            threads.add(t);
        }
        for (int i = 0; i < 2; i++) {            // 2 popping threads
            Thread t = new Thread(() -> { while (true) stack.pop(); });
            t.setDaemon(true);
            threads.add(t);
        }

        threads.forEach(Thread::start);
        Thread.sleep(10_000);                    // let them race for 10 seconds

        System.out.printf("%,d operations in 10 seconds%n", stack.getCounter());
    }

AdvancedThis stack is vulnerable to the ABA problem. A popping thread reads head A and plans to CAS it to A.next. Before it does, other threads pop A, pop the node below it, then push a recycled node that happens to reuse address A. The head is A again, so the CAS succeeds — but A.next now points into a stale, wrong part of the list, corrupting the structure. CAS only compares the reference, and A == A even though the world changed underneath. The fix is to version the reference: AtomicStampedReference<T> pairs the reference with an int stamp and CASes both together, so a stamp bumped on every push/pop makes the "same address, different history" case fail the CAS. (Java's own ConcurrentLinkedQueue dodges ABA differently, and garbage collection makes plain-reference ABA rarer in Java than in C/C++, but it is not impossible.)
Pitfall Forgetting to re-read the head after a failed CAS. If you loop but keep using the stale currentHead, the CAS can never succeed and you spin forever. Every iteration of a CAS loop must re-fetch the current value and recompute the candidate from it before trying again.

10. Threading Models for High-Performance IO

Most servers spend their lives waiting — for a socket, a database, a disk — not computing. This section looks at how the shape of the workload (CPU-bound vs IO-bound) decides how many threads you should run, walks through the classic thread-per-request model and where it breaks, and contrasts it with the non-blocking, thread-per-core event-loop model that trades readability for scale.

01. Introduction to Blocking IO

Objective: Separate CPU-bound work from IO-bound work, and understand why a blocking IO call leaves a thread doing nothing while still consuming real resources.

Work comes in two flavours. CPU-bound work keeps the processor busy — hashing, compression, number-crunching. IO-bound work spends most of its time waiting for an external device: a socket read, a disk read, a database query. The distinction matters because it changes the right number of threads to run.

A blocking IO call hands control to the operating system, which parks the calling thread until the device delivers its data. While parked, the thread runs no instructions at all — but it is not free: it still holds its call stack (on the order of ~1 MB per platform thread) and a slot in the OS scheduler. The CPU itself is idle for that thread and is free to run other threads (or nothing, if there are none ready).

The consequence: for a CPU-bound program the sweet spot is threads ≈ cores — more threads than cores just adds context-switch overhead with no extra parallelism. But for an IO-bound server, threads ≈ cores wastes the machine: with one thread per core, a single blocking call idles a whole core. To keep throughput up you need many more threads than cores, so that while some are parked on IO, others can make progress.

  • CPU-bound = busy computing; IO-bound = mostly waiting on a device.
  • A blocking call parks the thread; the OS wakes it only when the data is ready (often via an interrupt / DMA completion).
  • A parked thread does zero work yet still costs a stack (~1 MB) and scheduler bookkeeping.
  • Sizing rule of thumb: CPU-bound → threads ≈ cores; IO-bound with blocking calls → threads ≫ cores.
  • Even a few blocking calls can starve an entire fixed pool, hurting requests that need no IO at all.

02. Thread-Per-Task / Thread-Per-Request Model

Objective: Implement the classic one-platform-thread-per-task model, see how it lets 1000 blocking tasks finish in about a second, and understand the memory and scheduling ceilings that eventually break it.

The oldest answer to "my threads keep blocking on IO" is simply: add more threads — one per incoming task or request. If each task blocks for a second, running a thousand of them on a thousand threads lets all the waits overlap, so the whole batch still finishes in roughly a second. This is the thread-per-task model (also called thread-per-request), and for decades it was the default for web servers.

Executors.newCachedThreadPool() fits this model perfectly: it creates a fresh thread whenever a task is submitted and no idle thread is available, and keeps idle threads around (60 s by default) for reuse. Submit 1000 blocking tasks and the pool grows to about 1000 threads that all wait in parallel.

But platform threads are heavyweight: each carries a stack (commonly ~1 MB) and a native OS thread. Push toward tens of thousands and you hit a wall — the JVM runs out of memory or the OS refuses to create more threads, and the application crashes. The safe fix is to bound the pool with newFixedThreadPool(N), but then only N tasks run at once and the rest sit in a queue: throughput is capped and latency climbs. Neither choice is ideal, which is exactly the motivation for virtual threads (next section).

  • newCachedThreadPool() = one thread per task on demand, reused when idle; ideal for bursts of blocking IO.
  • 1000 one-second blocking tasks finish in ~1 s because the waits overlap, not ~1000 s.
  • Each platform thread costs ~1 MB of stack + a native thread, so the model tops out at a few thousand threads.
  • An unbounded cached pool under load can crash with OutOfMemoryError: unable to create native thread.
  • newFixedThreadPool(N) avoids the crash but caps throughput at N concurrent tasks and queues the rest.
import java.util.concurrent.ExecutorService;

03. Asynchronous, Non-Blocking IO with Thread-Per-Core Model

Objective: Understand the non-blocking alternative — a tiny pool of threads driven by an OS readiness selector — and weigh its huge scalability against its readability and stability costs.

Instead of parking a thread on every IO call, non-blocking (asynchronous) IO returns immediately and lets you register a callback (or complete a Future) that fires later, when the result is ready. That inverts the sizing problem: a single thread can start thousands of operations and then loop over whichever ones have become ready. This is the thread-per-core model — a pool of roughly cores threads, each running an event loop.

The engine underneath is an OS readiness selector — epoll on Linux, kqueue on macOS/BSD, exposed in Java as NIO's Selector. You register many non-blocking channels with one Selector; a call to select() parks the single thread until any registered channel is ready, then hands back exactly the ready ones to process. One thread thus multiplexes thousands of connections with almost no memory overhead and no per-connection context switching.

The upside is dramatic: minimal threads, minimal memory, and enormous scalability — the model behind Netty, Node.js, and most high-concurrency servers. The downside is human: chaining dependent async calls produces deeply nested callbacks ("callback hell") that are hard to read, test, and debug, and the raw JDK APIs are low-level enough that most teams pull in a framework. Worst of all, one accidental blocking call stalls its entire core — every connection that thread was juggling freezes until the call returns.

  • Non-blocking IO returns at once; results arrive later via callbacks / CompletableFuture.
  • An OS selector (epoll/kqueue, Java NIO Selector) reports which channels are ready, so one thread serves thousands of connections.
  • Thread-per-core: pool size ≈ cores; add threads only to use more cores, not to cover blocking.
  • Pros: tiny memory footprint, no context-switch storms, top-tier scalability (Netty / Node.js style).
  • Cons: callback hell, harder testing/debugging, framework dependence, and one blocking call freezes a whole core.

                    handleReadableBytes((SocketChannel) key.channel());  // process ready bytes, no blocking
                }
            }
        }

11. Virtual Threads and High-Performance IO

Virtual threads (JDK 21, Project Loom) are lightweight threads managed entirely by the JVM rather than mapped one-to-one onto scarce OS threads. Many of them are multiplexed onto a small pool of carrier platform threads, so you can write plain blocking code and still scale to hundreds of thousands or millions of concurrent tasks. This section introduces how they mount and unmount, why they excel at IO-bound work, and the practices that keep them fast.

01. Introduction to Virtual Threads

Objective: Understand what a virtual thread is, how it differs from a platform thread, and how the JVM schedules many virtual threads onto a handful of carrier threads.

Every Thread we have used so far is a platform thread: a thin wrapper around a dedicated OS thread. Each one is expensive and heavy because it maps 1:1 to a limited OS resource and reserves a large fixed-size stack (roughly 1 MB). A virtual thread is different: it is just an ordinary Java object on the heap, owned and scheduled by the JVM. The OS is not even aware of it, and it is reclaimed by garbage collection when no longer needed. Because there is no OS thread and no pre-reserved stack, creating one is cheap enough to spin up millions.

So how does a heap object run on a CPU? The moment the first virtual thread is created, the JVM builds a small internal pool of platform threads (a ForkJoinPool). To run virtual thread A, the JVM mounts it onto one of those platform threads, which is then called its carrier thread. When A finishes, it is unmounted and the carrier is freed for another virtual thread. Crucially, if A hits a blocking call and can make no progress, the JVM saves its stack, unmounts it, and runs a different virtual thread on that carrier instead. We have almost no control over this scheduling; the JVM manages it for us, just as the OS manages platform-thread scheduling.

  • Three ways to create one: Thread.startVirtualThread(task) and Thread.ofVirtual().start(task) start immediately; Thread.ofVirtual().unstarted(task) creates it for a later start().
  • Virtual threads are multiplexed onto a small carrier pool sized by default to the number of CPU cores (8 carriers on an 8-core machine), no matter how many virtual threads exist.
  • Printing the current thread shows a VirtualThread mounted on a ForkJoinPool worker, e.g. VirtualThread[#21]/runnable@ForkJoinPool-1-worker-1.
  • A blocking call (like Thread.sleep) unmounts the virtual thread so its carrier can run someone else.

02. High-Performance IO with Virtual Threads

Objective: See why virtual threads make IO-bound workloads scale, and rewrite the section 10 thread-per-task application so it no longer hits the OS thread ceiling.

Virtual threads give no speed-up for pure CPU work — there they are just an extra layer over platform threads. Their payoff is throughput on blocking IO. When a virtual thread makes a blocking call — a socket read, a DB query or HTTP request over the network, Thread.sleep, or a java.util.concurrent lock/queue wait — the JDK has re-implemented that operation to park (unmount) the virtual thread and let the OS's non-blocking facilities do the waiting, freeing the carrier for other work. From your side the code still looks like ordinary, easy-to-debug blocking API. There is no OS-level context switch, and creating each new virtual thread is cheap.

A couple of blocking operations are exceptions in JDK 21 and still hold the carrier rather than unmounting: synchronized blocks and native calls (pinning, covered in the next lecture) and ordinary file I/O (FileInputStream and friends). For sockets, sleeps, and java.util.concurrent waits — the bulk of server-side blocking — unmounting works as described.

In section 10 the platform thread-per-task model hit a wall: a cached pool tried to allocate one OS thread per task and the application crashed once the OS refused more threads; a fixed pool of 1000 platform threads survived but thrashed on context switches. The fix is Executors.newVirtualThreadPerTaskExecutor() — it creates a brand-new virtual thread for every submitted task, with no pool to size. The exact same simple blocking code now scales to millions of tasks.

  • Executors.newVirtualThreadPerTaskExecutor() spawns one virtual thread per task — no pooling, no tuning.
  • 10,000 tasks each doing 100 blocking IO ops complete fast because blocked virtual threads unmount and release their carriers.
  • Mount/unmount is far cheaper than an OS context switch, but it is not free — many tiny blocking calls cost more than a few long ones.
  • The carrier pool still stays near the core count (about 8) no matter how many virtual threads block.
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class IoBoundApplicationV2 {
    private static final int NUMBER_OF_TASKS = 10_000;

    public static void main(String[] args) {
        System.out.printf("Running %d tasks%n", NUMBER_OF_TASKS);

        long start = System.currentTimeMillis();
        performTasks();
        System.out.printf("Tasks took %dms%n", System.currentTimeMillis() - start);
    }

    private static void performTasks() {
        // A NEW virtual thread per task -- nothing to size.
        try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < NUMBER_OF_TASKS; i++) {
                executor.submit(() -> {
                    for (int j = 0; j < 100; j++) {
                        blockingIoOperation();
                    }
                });
            }
        } // close() blocks until every task finishes (JDK 19+)
    }

    private static void blockingIoOperation() {
        try {
            Thread.sleep(10);   // stand-in for a DB / HTTP / socket call
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
    }
}

03. Virtual Threads Best Practices

Objective: Learn the handful of rules that keep virtual threads fast and safe: one thread per task, plain blocking code, no pooling, and — above all — avoid pinning.

Everything you already know about thread safety (race conditions, deadlocks, data races, safe publication) applies unchanged to virtual threads; they are simply another abstraction over platform threads. The mindset shift is about how you use them. DO give each task its own virtual thread and write straightforward, synchronous, blocking code — that is the entire point. DON'T pool virtual threads: they are disposable and cheap to create, and the JVM already manages the real (platform) pool for you. Reach for Executors.newVirtualThreadPerTaskExecutor() rather than a fixed-size pool of virtual threads.

The one performance trap is pinning. A virtual thread that blocks inside a synchronized block or inside a native call cannot unmount — it pins its carrier, tying up a scarce platform thread for the whole blocking duration. If enough carriers get pinned, throughput collapses. The fix is to guard blocking IO with a java.util.concurrent.locks.ReentrantLock instead of synchronized, since a lock lets the virtual thread unmount while it waits. To cap how many operations run at once (for example, protecting a database), use a Semaphore rather than a bounded thread pool — the pool would defeat the point of unlimited cheap threads.

  • DO: one virtual thread per task; plain blocking, synchronous code.
  • DON'T: pool virtual threads — they are disposable; recreate them freely.
  • Avoid pinning: no blocking IO inside synchronized or native calls; prefer ReentrantLock.
  • Limit concurrency with a Semaphore, not a bounded thread pool.
  • Be careful with thread-locals: millions of threads holding per-thread state can blow up memory.
  • Virtual threads are always daemon threads and their priority is fixed at normal and ignored.
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Semaphore;

public class ConcurrencyLimitDemo {
    // Cap concurrency with a Semaphore -- NOT a bounded thread pool.
    private static final Semaphore DB_PERMITS = new Semaphore(100);

    public static void main(String[] args) {
        try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < 10_000; i++) {
                executor.submit(ConcurrencyLimitDemo::queryDatabase);
            }
        }
        System.out.println("All queries done; never more than 100 ran at once");
    }

    private static void queryDatabase() {
        try {
            DB_PERMITS.acquire();      // blocks + UNMOUNTS if 100 in flight
            try {
                Thread.sleep(50);      // stand-in for the DB round-trip
            } finally {
                DB_PERMITS.release();
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}

12. Beyond Multithreading

Multithreading extracts parallelism from the cores of a single machine, but one machine has a ceiling. This closing section looks outward — how the same ideas of dividing work, coordinating, and sharing state safely reappear at cluster scale in distributed systems and big data — and inward, pointing you to the concurrency toolkit worth learning next.

01. Distributed Systems, Big Data & Performance

Objective: Understand why and how work scales beyond a single JVM, and see that the concepts you already learned — partitioning work, coordinating actors, and guarding shared state — carry directly over to systems spanning many machines.

Multithreading is scaling up (vertical): you add more threads to exploit more cores inside one box, all sharing a single heap. That works beautifully until the data volume or request rate outgrows what the biggest affordable machine can handle. At that point you scale out (horizontal): run many instances across a cluster of machines behind a load balancer. This is, in essence, multithreading at a far larger scale — but the machines no longer share memory, so they coordinate by talking over the network via RPC, HTTP, or raw TCP.

Because there is no shared heap, common state lives in external distributed stores, and the coordination problems you met earlier return in harder forms: a non-atomic update across nodes may need a distributed lock; two nodes racing to update the same record is a distributed race condition. On top of that, distribution introduces failure modes a single process never sees — partial failure (some nodes down while others run), network partitions, and message loss. Distributed algorithms exist to cope: leader election to agree on a coordinator, failure detection to declare an unreachable host dead, replication so data survives a machine loss, and recovery protocols for nodes that rejoin after a partition heals.

Big data is a subset of this world dealing with datasets too large to move: instead of pulling data into compute nodes, you push the computation to where the data already lives, often on a distributed file system such as HDFS, using frameworks like Hadoop or Apache Spark. The dominant factor is the sheer amount of work to run in parallel — map-reduce style parallelism takes the "divide the work" instinct to a new level and is what lets machine-learning pipelines chew through enormous inputs. Message queues / brokers (Kafka, RabbitMQ) decouple producers from consumers across nodes, the same producer–consumer pattern you saw with a BlockingQueue, now stretched across the network.

Squeezing real performance out of any of this remains a discipline of its own. To master it you reach below your code: CPU caches and branch prediction, how the OS schedules and mediates hardware, how the JVM and JIT compile bytecode and manage garbage collection, and — crucially — the ability to measure rather than guess. A distributed system that must feel instant behind a phone app is only as good as the efficiency of each node inside it.

  • Scale up = more threads/cores in one machine (shared heap); scale out = more machines, no shared memory, coordinated over the network.
  • Data partitioning / sharding splits a dataset across nodes; replication copies it so the system survives a node failure and stays available.
  • Map-reduce moves computation to the data (HDFS, Hadoop, Spark) — big-data parallelism is "divide the work" at cluster scale.
  • Message brokers (Kafka, RabbitMQ) decouple producers and consumers across nodes — producer/consumer over the network.
  • Distribution adds new failure modes: partial failure, network partitions, and the resulting need for eventual consistency and the CAP trade-off (under a partition, choose consistency or availability).
  • The concurrency instincts you built — divide work, coordinate, guard shared state — are exactly what distributed design needs, just harder.

AdvancedThe CAP theorem states that during a network partition (P) a distributed store can guarantee at most one of Consistency (every read sees the latest write) or Availability (every request gets a non-error response). Note that CAP only forces a choice during a partition; when the network is healthy a well-designed system can offer both. In practice teams tune this with quorum reads/writes (e.g. requiring R + W > N replicas to overlap) to trade latency for stronger consistency, and lean on idempotent operations so a retried-after-timeout message does no harm.
Pitfall Assuming the network is reliable, has zero latency, and never partitions — the classic "fallacies of distributed computing." Local code that shares a heap can treat a method call as instant and guaranteed; a remote call can be slow, duplicated, reordered, or silently lost. Design every cross-node interaction to time out, retry, and be idempotent, and never port a single-JVM lock straight onto a cluster without a real distributed-coordination mechanism.

import java.util.concurrent.*;

public class NextSteps {
    // A tiny teaser: submit async work to a pool and compose the results
    // without ever blocking a thread by hand.
    public static void main(String[] args) throws Exception {
        ExecutorService pool = Executors.newFixedThreadPool(2);

        CompletableFuture<Integer> a =
            CompletableFuture.supplyAsync(() -> slowSquare(3), pool);
        CompletableFuture<Integer> b =
            CompletableFuture.supplyAsync(() -> slowSquare(4), pool);

        // Combine two async results, then map the total — all declarative.
        CompletableFuture<String> result = a
            .thenCombine(b, (x, y) -> x + y)
            .thenApply(sum -> "3^2 + 4^2 = " + sum);

        System.out.println(result.get()); // blocks only here, at the edge
        pool.shutdown();
    }

    static int slowSquare(int n) {
        try { Thread.sleep(100); } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        return n * n;
    }
}
