Web Design: HTML, CSS & JavaScript

A complete, original study guide to front-end web design, rewritten for reading and fast reference. It walks from the very first HTML tag through CSS styling and responsive layout, into JavaScript and the DOM, then builds a full responsive website and finishes with practical techniques and advanced exercises. The concepts, sequence, and example intent follow the Web Design for Beginners: Real World Coding in HTML & CSS-style curriculum (HTML, CSS & JavaScript), but every explanation is written from scratch.

How to use this guide

Most examples are live and editable: the code on the left runs in the sandboxed preview on the right. Change any code and it re-runs automatically — or press Ctrl/Cmd+Enter. Reset restores the original; Expand opens a roomy full-screen view (Esc to close).

To work along locally, create a folder and an index.html file, then open it in your browser — no build step or installation is required. For a live-reloading workflow, use an editor extension such as Live Server. Everything here runs offline in any modern browser.

HTML Foundations

The building blocks of every web page: the document skeleton, elements and attributes, text and structure, links, images, lists, tables, and HTML5 semantic markup — ending with a complete page you publish online.

Getting Started with the Course

Objective: Understand what this course covers and how HTML, CSS, and JavaScript split the work of building a web page.

Every modern web page is built from three cooperating technologies, and each has one clear job. HTML supplies the structure and meaning — the headings, paragraphs, links, and images. CSS supplies the presentation — colors, spacing, fonts, and layout. JavaScript supplies the behavior — responding to clicks, updating the page, and talking to servers.

Keeping these three concerns separate is not just tidy, it is the single most important habit in front-end work: you can restyle a whole site by editing CSS without touching the content, and add interactivity in JavaScript without rewriting the markup. This section builds your HTML foundation; later sections layer CSS and JavaScript on top.

  • HTML = structure and meaning; CSS = appearance; JavaScript = behavior.
  • The three are separate on purpose — change one without breaking the others.
  • HTML files end in .html and open directly in any browser.
  • You will finish this section by publishing a real page to a public URL.

Setting Up Your Coding Environment

Objective: Install a code editor and a browser, and understand how a browser turns a plain .html text file into a rendered page.

You only need two free tools. A code editor — Visual Studio Code is the popular default — gives you syntax highlighting, indentation, and tag suggestions that make markup far easier to write and read. A browser — Chrome, Firefox, or Edge — runs the file and ships with built-in developer tools for inspecting and debugging your page.

An HTML file is just text. When you open it, the browser reads it top to bottom, builds an in-memory model of the page (the DOM, covered later), and paints the result. That is why you can double-click an .html file and see a page with no server, no build step, and no installation.

  • Editor (e.g. VS Code) + browser (e.g. Chrome) is the entire toolkit.
  • Right-click a page and choose Inspect to open developer tools.
  • Name your home page index.html — servers serve it by default.
  • MDN (developer.mozilla.org) is the reference to keep open while learning.
Background A tidy starting layout keeps a project manageable as it grows. Create one folder per project and drop your files inside: 
my-site/

├── styles/

Your First HTML Page

Objective: Learn the four mandatory tags — <html>, <head>, <title>, <body> — and the <!DOCTYPE html> declaration that opens every page.

A well-formed page starts with <!DOCTYPE html>. This is not a tag but a declaration that tells the browser to use modern (HTML5) rendering rules. Below it, <html> is the root that wraps everything. Inside it live two children: <head> holds information about the page (its title, character encoding, links to stylesheets) that the user never sees directly, and <body> holds the visible content.

The <title> goes inside the <head> and names the browser tab and bookmark. A browser will guess a structure even for a nearly empty file, but supplying the full skeleton yourself gives it the information it needs to render your page reliably.

  • <!DOCTYPE html> is the first line of every page — no quotes, no closing.
  • <html> wraps the page; it contains exactly <head> then <body>.
  • <head> = metadata for the browser; <body> = what people see.
  • <title> lives in the head and labels the tab.
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <title>My Page</title>
  </head>
  <body>
    Hello
  </body>
</html>

Skeleton adapted from the course's first-page example, with lang and charset added as modern best practice.

Common pitfall In the live examples below you will not retype this skeleton. The editor already wraps your HTML in a full document, so you write only what goes inside <body>. When you build a real file on disk, though, the whole skeleton is required.

What Is an HTML Element

Objective: Learn the anatomy of an element — opening tag, content, closing tag — and how block elements like <div> and <p> stack down the page.

An element is an opening tag, some content, and a matching closing tag. The tags are the names in angle brackets; the closing tag repeats the name with a leading slash. Everything between them is the element's content, which may include other elements — this nesting is how a page's structure is built.

A <div> is a generic block container with no visual styling of its own; it simply occupies a full-width block and pushes the next element onto a new line. A <p> marks a paragraph and adds vertical space above and below. Indent nested elements as you write — the browser ignores the whitespace, but it makes your code readable.

  • Element = opening tag + content + closing tag: <p>…</p>.
  • Close tags in the reverse order you opened them; never overlap them.
  • <div> groups content into sections and can nest freely.
  • Do not nest a <p> inside another <p> — it is invalid.

Notice each block sits on its own line, and paragraphs have extra space around them.

Common pitfall Nested tags must close in the reverse order you opened them — never let them cross. The browser will try to guess what you meant, but the result is unpredictable across browsers. 
<!-- WRONG: the <b> and <i> overlap -->
<b><i>bold italic</b></i>

<!-- RIGHT: last opened is first closed -->
<b><i>bold italic</i></b>
Think of tags like matching brackets: <b><i>…</i></b> nests cleanly, <b><i>…</b></i> does not.

Headings and Common Tags

Objective: Use headings <h1>–<h6>, paragraphs, <div>, <span>, and code comments.

Headings label sections of content and come in six levels: <h1> is the most important (and largest), down to <h6>. Use them by meaning, not size — one <h1> per page for the main title, then <h2> for major sections, and so on. Search engines and screen readers rely on this outline, so do not skip levels just to get a certain look; that job belongs to CSS.

Where <div> is a block container, <span> is its inline cousin: it wraps a piece of text within a line without breaking it, so you can target that fragment later for styling. Comments — <!-- … --> — let you leave notes or temporarily disable markup; they never appear on the page.

  • Six heading levels; choose by outline importance, not appearance.
  • One <h1> per page is the conventional rule for structure and SEO.
  • <span> is inline (stays in the line); <div> is block (its own line).
  • Comments <!-- … --> are invisible and great for notes or debugging.

Headings shrink from h1 to h4; the commented paragraph never renders. Try un-commenting it.

Self-Closing (Void) Tags

Objective: Recognize elements that have no content — such as <br>, <hr>, <img>, and <input>.

Most elements wrap content, but a handful represent something self-contained and therefore have no closing tag. These are called void (or self-closing) elements. <br> forces a line break inside text, <hr> draws a horizontal rule between sections, <img> embeds an image, and <input> creates a form field.

Because there is nothing to put between an opening and closing tag, you write just the single tag. You may see the older self-closing form <br /> with a trailing slash; in HTML5 the slash is optional, and <br> is perfectly valid.

  • Void elements carry no content, so they have no closing tag.
  • Common ones: <br>, <hr>, <img>, <input>, <meta>.
  • Use <br> for a line break within text, not to space out blocks — that is a CSS job.

The <br> splits one line in two; the <hr> draws a divider; the <input> is a live field.

Element Attributes

Objective: Add information to elements with attributes — name/value pairs written in the opening tag — and meet the common global attributes.

Attributes configure an element. They live inside the opening tag as name="value" pairs, and you can list several separated by spaces. Values go in quotes (double or single). Some attributes are specific to one element (like src on an image), while global attributes work on almost any element.

The four global attributes you will use constantly: id gives an element a unique name so CSS or JavaScript can target exactly it; class tags an element with a reusable label so you can style many elements together; style applies CSS inline for a quick one-off; and title shows a small tooltip on hover. Prefer class for styling and reserve inline style for experiments.

  • Attributes are name="value" pairs inside the opening tag.
  • id is unique per page; class can repeat and group elements.
  • style applies CSS inline; keep it minimal and prefer classes.
  • Always quote attribute values — it avoids subtle parsing bugs.

The #main-title id and .note class rules in the CSS tab target elements by their attributes.

Common pitfall An id must be unique on the page. If two elements share one id, document.getElementById, a #id anchor link, and a #id CSS selector all quietly use only the first match — a bug with no error message. When you need the same value on many elements, use class instead. Also always quote attribute values: an unquoted value that contains a space is truncated, so title=Click me keeps only Click and treats me as a stray attribute.

Links and the Anchor Tag

Objective: Use the <a> element to link pages: absolute vs relative URLs, the target attribute, and in-page anchors.

Links are what make the web a web. The anchor element <a> becomes a link when you give it an href (hypertext reference) — the destination. An absolute URL gives the full path including the domain (https://example.com/page) and is used to link to other sites. A relative URL (about.html) is resolved relative to the current file and is how you link between your own pages.

The target attribute controls where the link opens. The default opens in the same tab; target="_blank" opens a new tab and keeps your page open. You can also link within a page: give an element an id, then point an href at #that-id to jump (and smoothly scroll) to it. Prefixing an href with mailto: opens the reader's email client instead.

  • href holds the destination; without it, <a> is not a link.
  • Absolute = full URL with domain; relative = a path near the current file.
  • target="_blank" opens in a new tab; add rel="noopener" for safety.
  • href="#id" jumps to an element on the same page; ids are case-sensitive.

The first link scrolls to #bottom; the MDN link opens a new tab; the mailto link launches an email draft.

Common pitfall Broken relative links are the number-one beginner bug, and it comes down to how a path is resolved. Suppose the current page is /blog/post.html:
  • about.html — a bare name means the same folder → /blog/about.html.
  • ../about.html — ../ climbs up one folder → /about.html.
  • /about.html — a leading slash means the site root, ignoring the current folder → /about.html.
One more trap: on real web servers file paths are case-sensitive, so About.html and about.html are different files — even though Windows and macOS often treat them as the same locally, letting a broken link work on your machine but 404 once deployed. (Modern browsers already imply rel="noopener" for target="_blank", but adding it explicitly keeps you safe on older ones.)

Adding Images

Objective: Embed images with <img>, understand the required src and important alt attributes, and control sizing.

<img> is a void element that pulls a picture into the page. Its one required attribute is src, the path to the image — relative for your own files, absolute for images hosted elsewhere. Supported formats include JPEG, PNG, GIF, SVG, and WebP.

Always add an alt attribute describing the image. Screen readers read it aloud, and if the image fails to load, the browser shows the text instead — so it is both an accessibility feature and a fallback. For sizing, modern best practice is the opposite of the old "set only one dimension" advice: set both the width and height attributes to the image's real pixel dimensions. The browser then reserves the correct box before the image downloads, so surrounding content does not jump when it arrives (this avoids layout shift, the CLS metric). To make the image scale responsively, add CSS width: 100%; height: auto — the height: auto keeps the aspect ratio while the attributes still supply the intrinsic ratio. Just make sure the two attributes match the image's true proportions, or it will look squashed.

  • src is required; it is the image's location.
  • alt describes the image for screen readers and broken-image fallback.
  • Set both width and height attributes to the real pixel size — it reserves the box and prevents layout shift (CLS).
  • Scale responsively with CSS width: 100%; height: auto; keep the attributes' ratio true to the image.
  • SVG and WebP are modern, efficient formats worth knowing.

The first image loads at its natural 140px; the second is the same image scaled to 280px wide with height:auto, so it stays proportional; the last fails on purpose so you can see the alt fallback.

Common pitfall Skipping alt (or writing alt="" for a meaningful image) makes your page inaccessible and hurts SEO. Describe what the image conveys; only use empty alt="" for purely decorative images.

Lists: Unordered, Ordered, and Description

Objective: Group content with <ul>, <ol>, and <dl>.

HTML has three list types. An unordered list <ul> shows bullets and is for items whose order does not matter. An ordered list <ol> numbers its items and is for sequences like steps. Both hold list items marked with <li>. A description list <dl> pairs terms (<dt>) with their descriptions (<dd>) — perfect for glossaries or key/value data.

Lists nest: put another <ul> or <ol> inside an <li> to create sub-items. You can change an ordered list's numbering with the type attribute (1, A, a, i) and its first number with start.

  • <ul> = bullets, <ol> = numbers; both contain <li> items.
  • <dl> pairs <dt> terms with <dd> descriptions.
  • Nest a list inside an <li> for sub-lists.
  • Navigation menus are usually built from a <ul> of links.

type="a" start="3" makes the outer list count with letters starting at c (c, d, e). Change type to "1", "A", or "i", and start to any number, to see the markers change.

Tables

Objective: Present tabular data with <table>, <tr>, <th>, <td>, and group rows with <thead>/<tbody>.

A table is for genuinely tabular data — rows and columns that intersect to give a value. The <table> wraps everything; each row is a <tr>; inside a row, a header cell is a <th> (bold and centered by default) and a data cell is a <td>. Group the header row in <thead> and the data rows in <tbody> so the structure is clear to browsers and assistive tech.

Tables have no built-in borders. Add them with CSS, and use border-collapse: collapse to merge the doubled cell borders into single clean lines. Do not use tables to lay out a page — that is what CSS layout is for; tables are strictly for data.

  • <tr> = row; <th> = header cell; <td> = data cell.
  • Wrap header rows in <thead>, body rows in <tbody>.
  • Borders come from CSS; border-collapse: collapse tidies them.
  • Tables are for data, never for page layout.

Remove border-collapse: collapse in the CSS tab to see the doubled borders it prevents.

HTML5 Semantic Elements

Objective: Give your page meaningful structure with <header>, <nav>, <main>, <section>, <article>, <aside>, and <footer>.

You could build a whole page from <div> elements, but a wall of identical divs tells no one what each region is for. HTML5 added semantic elements that behave like a div visually yet carry meaning. <header> is the top banner, <nav> holds navigation links, <main> is the primary content, <footer> is the bottom bar. Within content, <section> groups related material, <article> is a self-contained piece (a post, a card), and <aside> is tangential content like a sidebar.

The payoff is real: search engines rank the structure better, screen readers let users jump between landmarks, and your own code becomes far easier to read. Reach for a semantic element first, and fall back to <div> only when nothing more specific fits.

  • Semantic tags look like divs but describe their role.
  • Page shell: <header>, <nav>, <main>, <footer>.
  • Content: <section> groups, <article> stands alone, <aside> is secondary.
  • Better accessibility and SEO — for free.

Each region is a named landmark. Visually plain, but far more meaningful than nested divs.

Publish with GitHub Pages

Objective: Turn a folder of HTML into a live, shareable URL using GitHub Pages — free static hosting.

A file on your computer opens under the file:// protocol, which only you can see. To share it, the page must live on a web host. GitHub Pages serves static HTML/CSS/JS straight from a repository for free, making it ideal for a portfolio or a course project. The only requirement is a home page named index.html.

You can do everything in the GitHub web interface — create a repository, add your files, and enable Pages under Settings → Pages — or push from your machine with Git. After you enable Pages, GitHub gives you a public URL; changes take a minute or two to go live.

  • GitHub Pages hosts static sites for free from a repository.
  • Your home page must be named index.html.
  • Enable it under Settings → Pages; you get a public *.github.io URL.
  • Publishing is not instant — allow a minute or two after each push.
Background Publishing from the command line with Git, once you have created an empty repository on GitHub: 
# inside your project folder (with index.html at its root)

git add .
git commit -m "First version of my site"

# connect to the empty GitHub repo you created
git branch -M main

# then, on github.com:
# Settings → Pages → Source: "main" branch → Save
# Your site appears at https://USERNAME.github.io/my-site/

Section quiz

1. What is the difference between a <div> and a <span>, and when would you choose each?

Show answer

A <div> is a block element — it takes the full width and starts on a new line, so it groups larger sections of content. A <span> is inline — it wraps a fragment of text within a line without breaking it. Use <div> to group blocks and <span> to target a few words (for styling or scripting) without disturbing the surrounding text. Prefer semantic elements over <div> when one fits.

2. Why should every <img> have an alt attribute, and what does a good value look like?

Show answer

alt provides a text alternative: screen readers announce it, and if the image fails to load the browser shows it instead. It also helps SEO. A good value briefly describes what the image conveys (e.g. alt="Bar chart of 2025 sales"). Use empty alt="" only for purely decorative images that add no information.

3. You have a link <a href="#contact">Contact</a> but clicking it does nothing. What is the most likely cause?

Show answer

There is no element on the page with id="contact", or the id is spelled/cased differently (ids are case-sensitive). The href="#contact" jumps to the element whose id exactly matches "contact". Add id="contact" to the target section. (If the whole page fits on screen with no scrollbar, the jump also has no visible effect because there is nowhere to scroll.)

4. You need a wrapper for your site's top banner. Why prefer <header> over a plain <div class="header"> when both look identical on screen?

Show answer

<header> is a semantic element: it carries meaning that a generic <div> does not. Screen readers expose it as a landmark users can jump to, search engines understand the page's structure, and other developers (and future you) can read the markup at a glance. A <div> renders the same but tells no tool or person what the region is for. Reach for a semantic element (<header>, <nav>, <article>, …) whenever one fits, and fall back to <div> only for pure styling hooks with no inherent meaning.

5. What is a "void" (self-closing) element, and why does <br> have no </br>?

Show answer

A void element represents something self-contained that has no content between an opening and closing tag, so it is written as a single tag. <br> just marks a line break — there is nothing to wrap — so a closing tag would be meaningless. Other void elements include <hr>, <img>, <input>, and <meta>. In HTML5 the trailing slash (<br />) is optional and <br> is perfectly valid.

6. What does <!DOCTYPE html> do, and what is the difference between the <head> and the <body>?

Show answer

<!DOCTYPE html> is a declaration (not a tag) on the first line that tells the browser to use modern, standards-compliant HTML5 rendering rules; omitting it can trigger legacy "quirks mode". Inside <html>, the <head> holds metadata about the page — the <title>, character encoding, links to stylesheets — which the user does not see directly, while the <body> holds the visible content that renders on screen.

7. When should you use an <ol> versus a <ul>?

Show answer

Choose by whether order carries meaning. Use an ordered list <ol> when the sequence matters — steps in a recipe, a ranked top-10, numbered instructions — because the numbering communicates that order. Use an unordered list <ul> when the items are a set whose order is irrelevant — a list of ingredients, tags, or nav links. Both hold <li> items; the choice is about meaning, not appearance (you can restyle either with CSS).

Introduction to CSS

Objective: Understand what CSS is and the three ways to apply it: inline, internal, and external.

HTML describes what is on the page; CSS (Cascading Style Sheets) describes how it looks — colours, spacing, fonts, and layout. Keeping structure and presentation separate means you can restyle an entire site without touching a single tag of content.

There are three places a browser can read your styles from. Inline styles live in a style attribute on one element. An internal stylesheet sits in a <style> block in the <head>. An external stylesheet is a separate .css file linked with <link> — the professional default, because one file styles every page.

  • CSS separates presentation from HTML structure.
  • Inline = one element; internal = one page; external = whole site.
  • Prefer external stylesheets for anything real — they are cached and reusable.
  • Inline styles are hard to override and should be a last resort.
<!-- 1. Inline: styles one element only -->
<p style="color: teal;">One-off style</p>

<!-- 2. Internal: styles this page, lives in the head -->
<head>
  <style>
    p { color: teal; }
  </style>
</head>

<!-- 3. External: one file for the whole site (recommended) -->
<head>
  <link rel="stylesheet" href="styles.css">
</head>
Background "Cascading" refers to how the browser resolves conflicts when several rules target the same element. You will see the cascade in action in the selectors lecture — for now, just know that the order and origin of a rule matter.

Adding CSS to HTML

Objective: Learn the rule syntax — selector, property, value — and how to connect a stylesheet.

Every CSS rule has the same shape: a selector that chooses elements, then a declaration block in curly braces containing one or more property: value pairs, each ended with a semicolon. Whitespace is ignored, so format for readability.

To attach an external file, add one <link> in the <head>. The href is the path to the file, relative to the HTML document.

  • Syntax: selector { property: value; }.
  • Always end each declaration with a semicolon — it prevents the next line from being swallowed.
  • Comments use /* ... */ (there is no line comment in CSS).
  • Link a file once with <link rel="stylesheet" href="...">.

Selectors and the Cascade

Objective: Use element, class, and id selectors, and understand specificity — how the browser resolves conflicting rules.

You target elements three main ways: by tag name (p), by class (.note, matches every element carrying that class), or by id (#main, matches the one element with that id). Classes are the workhorse — reusable and easy to combine.

When two rules set the same property on the same element, the browser picks a winner by specificity: an id beats a class, a class beats a tag. If specificity ties, the rule written later wins. Inline style attributes outrank all of them, and !important overrides everything (avoid it).

  • tag = every element of that type; .class = reusable label; #id = one unique element.
  • Specificity order (low to high): tag < class < id < inline style.
  • Equal specificity? Source order decides — later wins.
  • Reach for classes; save ids for genuinely unique landmarks.

All three paragraphs get colour rules; the highest-specificity selector wins each time.

Common pitfall When a style "won't apply", it is usually being overridden by a more specific rule, not ignored. Open DevTools, inspect the element, and look for the winning declaration with the crossed-out losers beneath it — do not just pile on !important.

Colors: Background and Font

Objective: Set text and background colour using names, hex, rgb(), and hsl(), including transparency.

Use color for text and background-color for the fill behind an element. A colour can be written four ways: a named keyword (tomato), a hex code (#0b63ce), rgb() channels (0–255), or hsl() (hue 0–360, saturation and lightness as percentages). Add a fourth value — or an 8-digit hex — for alpha transparency.

HSL is the friendliest to reason about: pick a hue, then dial saturation and lightness to make tints and shades of the same colour without guessing hex codes.

  • color styles text; background-color styles the fill.
  • Formats: tomato, #0b63ce, rgb(11 99 206), hsl(212 90% 43%).
  • Alpha 0–1 for transparency: rgb(0 0 0 / 0.5) or hsl(212 90% 43% / 0.5).
  • HSL makes lighter/darker variants of one hue trivial.

Four notations for colour; the last uses / 0.5 alpha so the background shows through.

Common pitfall Two shorthands trip up beginners. A 3-digit hex is just each digit doubled: #0bf is identical to #00bbff. And the legacy comma syntax rgb(11, 99, 206) is exactly as valid as the modern space-separated rgb(11 99 206) — seeing commas elsewhere does not mean the space version is a typo. The old form spells alpha as a separate function, rgba(0, 0, 0, 0.5).

The CSS Box Model

Objective: Understand that every element is a box of content, padding, border, and margin — and how box-sizing changes the maths.

The browser lays out every element as nested rectangles. Content is the text or image, padding is space inside the border, border wraps the padding, and margin is space pushing other elements away. Padding shares the element's background; margin is always transparent.

By default (content-box), width sizes only the content, so padding and border are added on top — a 200px box with 20px padding actually occupies 240px. Setting box-sizing: border-box folds padding and border into the width, which is far easier to plan. Most stylesheets set it globally.

  • Four layers, inside out: content → padding → border → margin.
  • Padding takes the background colour; margin never does.
  • content-box (default) adds padding/border to width; border-box includes them.
  • Start every project with * { box-sizing: border-box; }.

Same declared width, but the first box is visibly wider because padding and border are added on.

Common pitfall Adjacent vertical margins collapse: a 30px bottom margin next to a 20px top margin yields 30px of gap, not 50px. This only happens vertically and only for margins — padding never collapses.

Text Properties

Objective: Shape how text reads with alignment, decoration, transform, spacing, and line-height.

Beyond colour, a handful of properties control the feel of running text. text-align positions lines within their box; text-transform changes letter case at display time (the underlying text is untouched); text-decoration adds or removes underlines; and line-height, letter-spacing, and word-spacing tune the rhythm.

A unitless line-height such as 1.6 is best practice — it scales with the element's font size, keeping paragraphs readable at any size.

  • text-align: left, right, center, justify.
  • text-transform: uppercase, lowercase, capitalize — cosmetic only.
  • text-decoration: underline or none (common on links).
  • Prefer a unitless line-height (e.g. 1.6) for readable paragraphs.

The heading text is lowercase in the HTML — text-transform only changes how it displays.

Objective: Choose typefaces with font-family stacks and control size, weight, and style — plus how web fonts are loaded.

font-family takes a comma-separated stack: the browser uses the first font it has and falls back down the list, so always end with a generic family (serif, sans-serif, monospace). font-size sets the size, font-weight the thickness (400 normal, 700 bold), and font-style handles italics.

  • Always provide a fallback stack ending in a generic family.
  • font-weight: 100–900 (or normal/bold); font-style: italic for slanted text.
  • The font shorthand sets several properties at once.
  • Web fonts add network weight — subset and limit the styles you load.

Link States and Pseudo-Classes

Objective: Style the interactive states of links and elements with :hover, :focus, and friends.

A pseudo-class targets an element in a particular state, written with a single colon. Links have four: :link (unvisited), :visited, :hover (pointer over it), and :active (being clicked). :focus — critical for keyboard users — applies when an element is tabbed to.

For the four link states to all take effect, declare them in the order link, visited, hover, active (mnemonic "LoVe HAte"), because later same-specificity rules win. Pseudo-classes work on any element, not just links — :first-child, :nth-child(), and :checked are everyday tools.

  • Link order matters: :link → :visited → :hover → :active.
  • Never remove :focus visibility — keyboard users rely on it.
  • :hover works on any element, giving cheap interactivity with no JS.
  • Structural pseudo-classes: :first-child, :last-child, :nth-child(2).

Display Properties

Objective: Control how an element flows with block, inline, inline-block, and none.

Every element has a default display. Block elements (like <div> and <p>) start on a new line and stretch to full width. Inline elements (like <span> and <a>) sit in the text flow and ignore width/height and vertical margins. inline-block is the hybrid: it flows inline yet accepts width, height, and full padding — perfect for nav items and buttons.

display: none removes the element from the page entirely, as if it were deleted (unlike visibility: hidden, which leaves an empty gap).

  • block: new line, full width, sizable.
  • inline: in-flow, no width/height, no vertical margin.
  • inline-block: in-flow but sizable — great for menu items.
  • none removes the box; visibility: hidden keeps its space.

Spans are inline by default; inline-block lets them sit in a row yet take a fixed width.

Common pitfall Setting width, height, or top/bottom margin on a plain inline element does nothing — the browser silently ignores them. This is the single most common display surprise: a <span> or <a> just will not take a size. Switch it to inline-block (keeps the in-flow row) or block (new line) and the dimensions apply. Left/right margin and padding do work on inline elements.

Position: static to sticky

Objective: Move elements out of normal flow with relative, absolute, fixed, and sticky, using offsets.

The position property changes how the offset properties top, right, bottom, and left behave. static is the default (offsets ignored). relative nudges an element from where it would normally sit, leaving its space reserved. absolute removes it from flow and positions it against the nearest positioned ancestor. fixed pins it to the viewport so it stays put while scrolling. sticky acts relative until you scroll past a threshold, then sticks like fixed.

A key trick: to position a child absolutely inside a parent, give the parent position: relative. That makes it the reference box the child measures from.

  • Offsets (top/left…) do nothing until position is not static.
  • absolute anchors to the nearest positioned ancestor — set relative on the parent.
  • fixed pins to the viewport; sticky pins after a scroll threshold.
  • absolute and fixed remove the element from normal flow.

The badge sits at the card's corner because the card is position: relative.

Common pitfall An absolutely positioned element jumps to the top-left of the page when no ancestor is positioned. If it lands in the wrong place, add position: relative to the intended parent.

Float and Clear

Objective: Use float to wrap content around an element, and clear to stop the wrap correctly.

float: left or right pulls an element to that side and lets following inline content flow around it — its original purpose is wrapping text around an image. Floated elements are lifted partly out of flow, so a container of only floats can collapse to zero height.

clear forces an element below any earlier floats. The modern fix for a collapsed container is the clearfix: a ::after pseudo-element that clears both sides. Floats were once the main layout tool; today, use them for their real job (text wrap) and reach for Flexbox or Grid for page layout.

  • float: left/right pulls an element aside; text wraps around it.
  • Floats can collapse their parent's height — fix with a clearfix.
  • clear: both pushes an element below preceding floats.
  • For overall layout, prefer Flexbox/Grid; keep float for wrapping.

The symptom: the parent's background and border vanish, and later elements slide up underneath the floats. Objective: Round out your toolkit with z-index, outline, overflow, max-width, opacity, and cursor.

These properties come up constantly. z-index controls stacking order among positioned elements (higher sits on top). overflow decides what happens when content is too big for its box — hidden, scroll, or auto. max-width caps how wide a block can grow, the backbone of centred, readable content columns. opacity fades the whole element (0–1), outline draws a ring that does not affect layout, and cursor changes the mouse pointer.

  • overflow: auto adds scrollbars only when needed.

Combinators

Objective: Combine selectors by relationship: descendant, child, adjacent sibling, and general sibling.

Combinators express how two elements relate in the tree. A space is the descendant combinator (any depth beneath). A > is the child combinator (direct children only). A + selects the one adjacent sibling immediately after. A ~ selects every general sibling that follows. They let you style elements by context instead of adding classes everywhere.

  • a b — any b inside an a (descendant).
  • a > b — only direct-child b.
  • a + b — the single b right after an a.
  • a ~ b — all b siblings after an a.

Common pitfall Keep the two sibling combinators straight: + selects only the single element immediately after, while ~ selects every following sibling. Crucially, neither reaches into descendants — both walk sideways among elements that share the same parent. So in the demo, h3 ~ p italicises the two direct paragraphs but leaves the <p> nested inside the <div> untouched, because that paragraph is a child of the div, not a sibling of the <h3>.

Pseudo-Elements

Objective: Style parts of an element or inject generated content with ::first-letter, ::first-line, ::before, and ::after.

A pseudo-element (double colon) styles a slice of an element that has no tag of its own. ::first-letter and ::first-line target exactly those portions of text — handy for drop caps. ::before and ::after insert a generated box at the start or end of an element's content; they require a content property (even an empty string) to appear.

  • Use two colons (::) to distinguish pseudo-elements from pseudo-classes.
  • ::first-letter / ::first-line style typographic fragments.
  • ::before / ::after need a content value to render.
  • Generated content is decorative — screen readers may skip it, so never put essential text there.

Responsive Layout with Flexbox

Objective: Lay out rows and columns that adapt, using Flexbox — a one-dimensional layout system.

Set display: flex on a container and its direct children become flex items along one axis. flex-direction chooses the main axis (row or column); justify-content aligns items along it, and align-items aligns them on the cross axis. The flex shorthand on each item controls how leftover space is shared, and flex-wrap lets items flow onto new lines.

Flexbox shines for navbars, card rows, and centring. Switching flex-direction inside a media query — row on desktop, column on mobile — is a clean, modern responsive pattern with no floats or clears.

  • display: flex lays children along the main axis.
  • justify-content = main axis; align-items = cross axis.
  • flex: 1 makes an item grow to share free space; gap spaces items.
  • Change flex-direction in a media query to reflow for mobile.

A one-line navbar that stacks into a column under 480px. margin-right: auto on the brand shoves the links to the opposite edge in the row layout.

Common pitfall justify-content always follows the main axis and align-items the cross axis — they do not mean "horizontal" and "vertical". When flex-direction switches to column, the main axis becomes vertical, so the two properties swap roles: now justify-content spaces items top-to-bottom and align-items aligns them left-to-right. Forgetting this flip is why a centring trick that worked in a row suddenly does the opposite in a column.

Responsive Layout with CSS Grid

Objective: Build full-page, two-dimensional layouts with CSS Grid that reflow at a breakpoint.

Where Flexbox handles one axis, Grid handles rows and columns at once. Set display: grid and define tracks with grid-template-columns. The fr unit divides free space into fractions, so 1fr 3fr gives a narrow sidebar and a wide main column. gap adds gutters between tracks without margins.

For responsiveness, redefine the columns inside a media query so a multi-column grid collapses to a single column on phones. Grid is the go-to for page skeletons and card galleries.

  • display: grid + grid-template-columns defines the column tracks.
  • The fr unit shares leftover space: 1fr 3fr = 25% / 75%.
  • gap spaces tracks cleanly — no collapsing margins.
  • Swap the template in a media query to collapse to one column.

grid-column: 1 / -1 makes header and footer span every column; the media query collapses it.

Advanced: Flexbox vs Grid — which to reach for

Use Flexbox when you are arranging items along a single line that should grow, shrink, or wrap — navbars, toolbars, tag rows. Use Grid when you need control over both rows and columns at once — page skeletons, dashboards, image galleries. They compose well: a Grid page whose cells each use Flexbox internally is a common, robust pattern.

Section quiz

1. Two rules set color on the same paragraph: .note { color: orange; } and p { color: gray; }. Which wins, and why?

Show answer

Orange. A class selector (specificity 0-1-0) outranks a tag selector (0-0-1), so .note wins regardless of source order. Source order only breaks ties between equally specific selectors.

2. You give a <div> width: 200px, padding: 20px, and border: 4px. How wide does it actually render with the default box-sizing, and how do you make it stay 200px?

Show answer

With the default content-box it renders 200 + 20 + 20 + 4 + 4 = 248px wide, because padding and border are added on top of the width. Set box-sizing: border-box to fold them into the 200px so it stays 200px.

3. What is the difference between display: none and visibility: hidden?

Show answer

display: none removes the element from the layout entirely — no box, no reserved space, as if it were deleted. visibility: hidden hides the element visually but keeps its space in the layout, leaving a gap where it would have been.

4. You set position: absolute on a badge so it sits in the corner of a card, but it flies to the top-left of the page instead. Why, and what one line fixes it?

Show answer

An absolutely positioned element is measured from its nearest positioned ancestor. If no ancestor has a non-static position, it falls back to the viewport, landing at the page's top-left. Add position: relative to the card so it becomes the reference box; the badge's top/right offsets are then measured from the card's edges. (relative alone doesn't move the card — it just anchors the absolute child.)

5. When would you reach for Flexbox versus Grid, and what does grid-template-columns: 1fr 3fr produce?

Show answer

Flexbox is one-dimensional — use it to lay items along a single row or column (navbars, toolbars, card rows). Grid is two-dimensional — use it when you need rows and columns at once (page skeletons, galleries). 1fr 3fr splits the free space into 1 + 3 = 4 fractions, so the first column takes 1/4 (25%) and the second takes 3/4 (75%) — a narrow sidebar beside a wide main column.

6. What does ul > li match that ul li does not restrict, and how does h3 + p differ from h3 ~ p?

Show answer

ul li is the descendant combinator: it matches every <li> anywhere inside the <ul>, even nested inside another list. ul > li is the child combinator: only <li> that are direct children. For siblings, h3 + p selects only the single <p> immediately after the <h3> (adjacent), while h3 ~ p selects every <p> that follows it among the same siblings (general). Neither sibling combinator reaches into descendants.

JavaScript Fundamentals

The language of interactivity: how to run JavaScript, variables and data types, objects and arrays, operators, functions, conditionals, and loops — the core you need before touching the page.

Introduction to JavaScript

Objective: Understand what JavaScript adds to a web page and where the code actually runs.

HTML gives a page structure and CSS gives it a look, but both are static — once the browser finishes rendering, nothing changes. JavaScript is the third layer: a full programming language that runs inside the browser and can react to clicks, read what a user types, change the page on the fly, and talk to servers. It is what turns a document into an application.

A useful way to think about it: your browser ships with a JavaScript engine (V8 in Chrome, SpiderMonkey in Firefox). When a page loads a script, the engine reads your instructions top to bottom and carries them out one at a time. In this section we learn that instruction set — variables, types, functions, conditions, and loops — before we start manipulating the page itself in the next section.

  • JavaScript is the behavior layer, on top of HTML (structure) and CSS (style).
  • It runs client-side, in the browser, with no separate install for the user.
  • Code executes top to bottom, one statement at a time.
  • The same language now also runs on servers (Node.js), but our focus is the browser.

Getting Started with JavaScript

Objective: Add a <script> to a page, print to the console, and write your first statements.

You add JavaScript with the <script> element. You can write code directly between the tags, but the professional habit is to keep it in a separate .js file and link it with <script src="app.js"></script>. An external file keeps HTML readable, lets several pages share one script, and lets the browser cache it for faster loads.

Place the script just before the closing </body> tag. That way the HTML has already loaded by the time your code runs, so the elements you want to work with actually exist. Your two most-used tools while learning are console.log(...), which prints a value to the browser's developer console, and alert(...), which pops up a blocking dialog. Prefer console.log — it does not interrupt the page and can print many values.

  • Link external scripts with <script src="app.js"></script> — no src when writing inline code.
  • Put the tag at the end of <body> so the page is ready before the script runs.
  • console.log() prints to the dev-tools console (F12 → Console).
  • End statements with a semicolon; comments are // one line or /* many */.

Notice the console messages appear in order, matching the top-to-bottom flow.

Common pitfall If you select an element in JavaScript but the <script> sits in the <head>, the element does not exist yet and you get null. Keep the script at the bottom of <body> (or add the defer attribute) so the HTML is parsed first.

Variables: let and const

Objective: Declare variables, assign and reassign values, and understand why const is the default choice.

A variable is a named box that holds a value so you can reuse it. You declare it once with a keyword, then refer to it by name anywhere in scope. Modern JavaScript uses two keywords: let for a value you intend to change later, and const for one that should never be reassigned. The older var still works but has confusing scoping rules — avoid it.

Reach for const by default and only switch to let when you actually need to reassign. This makes your intent obvious and lets the engine catch mistakes: reassigning a const throws an error immediately.   • const = cannot be reassigned; let = can be reassigned. Prefer const.
  • Declaring the same name twice in one scope is an error.
  • Names: case-sensitive, no spaces, cannot start with a digit, no reserved words.
  • Reassigning a const throws TypeError: Assignment to constant variable.

Data Types

Objective: Recognise the primitive types — string, number, boolean, null, undefined — and inspect any value with typeof.

Every value has a type. The primitives you meet first are string (text in quotes), number (there is no separate integer type), and boolean (true / false). Two special values represent "nothing": undefined means a variable was declared but never assigned, while null is a value you assign to say "intentionally empty". JavaScript is dynamically typed, so the same variable can hold a string now and a number later.

Because types are flexible, the + operator does double duty: with two numbers it adds, but if either side is a string it concatenates, converting the number to text. That is why "100" + 100 gives the string "100100", not 200. Use typeof to check what you actually have — a common surprise is that typeof null is "object", a historic quirk of the language.

  • Primitives: string, number, boolean, undefined, null (plus symbol, bigint).
  • Quotes make a string: '42' is text, 42 is a number.
  • + with any string operand concatenates instead of adding.
  • typeof reveals a value's type; typeof null === 'object' is a known quirk.

Objects and Arrays

Objective: Store many values in one variable — ordered lists with arrays and named fields with objects.

An array holds an ordered list inside square brackets: ['a', 'b', 'c']. You read and write items by a zero-based index, so arr[0] is the first item. An object holds named key: value pairs inside curly braces and has no fixed order; you access members with dot notation (obj.name) or bracket notation (obj['name']). Use bracket notation when the key is stored in a variable or is not a valid identifier. Both can nest freely — arrays inside objects, objects inside arrays.

Here is the idea beginners trip over most: arrays and objects are reference types. A variable does not hold the data itself, it holds a reference to where the data lives in memory. So const b = a copies the reference, not the contents — now a and b point at the same array, and changing one changes the other. (Primitives like numbers and strings are copied by value, so they never share this way.)

  • Arrays: ordered, indexed from 0 — list[0] is the first element.
  • Objects: unordered key: value pairs — access with obj.key or obj['key'].
  • const lets you mutate contents (arr.push(), obj.x = 1) but not reassign the variable.
  • Assigning one array/object to another copies the reference, so both share the same data.

Second, when the key lives in a variable you must use bracket notation: const k = 'last'; user[k] reads the last field, but user.k looks for a literal property named k (and returns undefined).

Operators

Objective: Use arithmetic, assignment, comparison, and logical operators to compute and compare values.

Arithmetic operators are + - * / plus % (remainder) and ** (exponent). Assignment shorthands combine a calculation with a store: val += 3 means val = val + 3, and val++ adds one. Comparison operators (> < >= <= and equality) always return a boolean, which you will feed into conditions and loops.

The single most important rule here: use === (strict equality), not ==. The double-equals compares after silently converting types, so 10 == '10' is true. Triple-equals checks value and type, so 10 === '10' is false — which is almost always what you want. The same holds for !== versus !=. Logical operators && (and), || (or) and ! (not) combine booleans.

  • Arithmetic: + - * / % and **; shorthands += -= *= /=, ++, --.
  • % is the remainder — great for "is it even?" (n % 2 === 0).
  • Comparisons return true/false.
  • Prefer === / !== (checks type too) over == / !=.

Flip === to == on the 10 vs '10' line and re-run to watch coercion turn false into true.

Common pitfall == hides bugs: 0 == '', 0 == false and '' == false are all true because of type coercion. Standardise on === and !== and this whole class of surprises disappears.

Functions

Objective: Package reusable logic into a function, pass it arguments, and return a result.

A function is a named block of code you define once and invoke (call) as many times as you like with the round brackets: adder(5, 10). Values listed in the definition are parameters; the actual values you pass in are arguments. Inside, those arguments behave as local variables that exist only within the function's braces — its own scope. A return statement hands a value back to the caller and immediately stops the function, so any code after return never runs.

There are two common forms. A function declaration (function adder(a, b) { ... }) is hoisted, meaning you may call it above where it is written. A function expression stores a function in a variable (const adder = function (a, b) { ... }) and must be defined before it is used. Arrow functions, a shorter expression syntax, come later in JavaScript in Practice.

  • Define once, call many times — the core payoff of functions is reuse.
  • Parameters are placeholders; arguments are the real values you pass.
  • return sends a value back and ends the function immediately.
  • Variables declared inside a function are local to it (function scope).

One adder definition, called with different arguments, produces different results.

Common pitfall Referring to a function without the round brackets does not run it — adder is the function itself, adder(2, 3) calls it. And any line placed after a return is dead code that never executes.

Conditions

Objective: Run different code depending on whether a condition is true, using if / else if / else and the ternary operator.

An if statement runs its block only when the condition in its brackets is truthy. Add else if to test further conditions in order, and a final else as the catch-all when nothing matched. Only the first matching branch runs, then the chain stops. Conditions usually use the comparison and logical operators from the previous lecture.

JavaScript also treats non-boolean values as "truthy" or "falsy" in a condition. The falsy values are false, 0, '' (empty string), null, undefined, and NaN; everything else is truthy. For a short either/or choice, the ternary operator condition ? whenTrue : whenFalse is a compact one-line alternative to a full if/else.

  • if / else if / else — only the first matching branch runs.
  • Conditions evaluate to a boolean, or to a truthy/falsy value.
  • Falsy: false, 0, '', null, undefined, NaN. Everything else is truthy.
  • Ternary cond ? x : y is a concise inline if/else that returns a value.

Loops

Objective: Repeat work with for, while, and do…while, and iterate over an array's items.

A for loop packs three parts into one line: an initialiser (let i = 0), a condition that keeps it going (i < 10), and an update that eventually ends it (i++). A while loop keeps its parts separate and runs as long as its condition is true — it may run zero times. A do…while loop checks the condition after the body, so it always runs at least once.

To walk an array you can use a counting for loop with arr.length, but the readable modern choices are for…of (gives you each value) and forEach (calls a function with value, index, and the array). Be careful with for…in: it yields keys — property names for objects and index strings for arrays — so use it for objects, not for reading array values.

  • for needs three parts: start, condition, update — miss the update and it loops forever.
  • while may run zero times; do…while always runs at least once.
  • for…of gives array values; for…in gives keys (use for objects).
  • arr.forEach((value, index) => …) is the common way to iterate an array.

Notice the do…while prints once even though its condition is false from the start.

Common pitfall Forgetting to advance the counter (i++ or n++) makes an infinite loop that freezes the tab. Also avoid for…in for arrays — it iterates keys as strings ("0", "1"), not the values you probably want.

Section quiz

1. What does console.log('100' + 100) print, and why?

Show answer

"100100". Because one operand is a string, + performs string concatenation instead of addition, converting the number 100 to text "100" and joining them.

2. After const a = [1, 2]; const b = a; b[0] = 99;, what is a[0]?

Show answer

99. Arrays are reference types, so b = a copies the reference, not the data. Both names point at the same array in memory, so changing b also changes a.

3. Why does a do…while loop differ from a while loop, and when would you choose it?

Show answer

A while loop tests its condition before the body, so it can run zero times. A do…while tests after the body, so it always runs at least once. Choose do…while when the body must execute once before the condition is even checked — for example, prompting for input then validating it.

4. Why is 10 == '10' true but 10 === '10' false?

Show answer

== is loose equality: before comparing it coerces the operands to a common type, converting the string '10' to the number 10, so 10 == 10 is true. === is strict equality: it compares value and type with no conversion, and since a number is not a string it returns false. Prefer === so coercion never surprises you.

5. Of these six values — 0, '', '0', [], null, 'false' — which are falsy?

Show answer

Only 0, '', and null are falsy. The complete falsy list is false, 0, '', null, undefined, and NaN — everything else is truthy. So '0' (a non-empty string), [] (an empty array is still an object), and 'false' (a non-empty string) are all truthy, which trips up many beginners.

6. Given const arr = [1, 2];, does arr.push(3) throw? Does arr = [9] throw?

Show answer

arr.push(3) is fine — it mutates the existing array to [1, 2, 3]. arr = [9] throws TypeError: Assignment to constant variable because it reassigns the binding. const freezes the binding (which value the name points to), not the contents of the object or array it points at.

The DOM: Making Pages Interactive

Connect JavaScript to the page: the DOM tree, selecting and changing elements, reading form values, and responding to events — build up to creating, moving, and animating elements.

Introduction to the DOM

Objective: Understand what the Document Object Model is and how the browser hands your page to JavaScript as a set of objects you can read and change.

When a browser loads your HTML it does not just paint pixels — it builds an in-memory model of every element, called the Document Object Model (DOM). JavaScript reaches that model through one global object: document. Everything on the page — the title, the body, each heading and paragraph — becomes an object with properties you can read and methods you can call.

The crucial idea: changes you make through document update the live page instantly, without editing the HTML file and without reloading. Refresh the page and your changes vanish, because you were manipulating the browser's working copy, not the source file on disk.

  • document is your entry point to the whole page.
  • The DOM is a tree of nodes — mostly element nodes and text nodes.
  • DOM edits are live and temporary; a reload rebuilds the tree from the HTML.
  • This is what makes pages interactive rather than static documents.

The heading text changes from JavaScript alone — the HTML tab still says "Loading…".

The DOM Tree

Objective: See how the DOM is a tree of nodes that mirrors the nesting of your HTML, and learn the vocabulary of parents, children, and siblings.

Because HTML elements nest inside one another, the DOM is naturally a tree. document is the root; <html> hangs off it, <head> and <body> hang off that, and so on down to the text inside each element. Every box in the tree is a node, and relationships use family words: a node's parent contains it, its children are the nodes directly inside it, and siblings share the same parent.

From any element you can walk in every direction: parentNode goes up, children goes down, and nextElementSibling / previousElementSibling move across. The *Element* versions skip whitespace text nodes, which is almost always what you want.

  • The tree shape follows your HTML nesting exactly.
  • parentNode, children, firstElementChild, lastElementChild navigate the tree.
  • nextElementSibling / previousElementSibling step between siblings.
  • Prefer the Element variants — they ignore stray whitespace nodes.

Navigate up, down, and sideways from a single element reference.

Selecting Elements

Objective: Grab the elements you want to work with using getElementById, querySelector, and querySelectorAll.

Before you can change an element you must find it. document.getElementById('id') is the fastest way to fetch a single element by its id. The more flexible pair is querySelector and querySelectorAll, which take any CSS selector string — the exact same syntax you already use for styling. querySelector returns the first match (or null); querySelectorAll returns a NodeList of every match.

A NodeList is array-like: it has a length and supports forEach, so you can loop over every matched element and act on each one.

  • getElementById — one element by id, no # prefix in the string.
  • querySelector('.item') — first element matching a CSS selector.
  • querySelectorAll('.item') — a NodeList of all matches; loop with forEach.
  • If nothing matches, querySelector returns null and querySelectorAll returns an empty list.

Same CSS selectors you style with — .class, #id, tag names — all work here.

Common pitfall Calling a script before the elements exist returns null. Put your <script> at the end of <body>, or wait for DOMContentLoaded (see lecture 9). Reading a property of null throws "Cannot read properties of null".

Manipulating Elements

Objective: Read and change an element's content and attributes with textContent, innerHTML, and the attribute methods.

Once you hold an element, updating it is just property assignment. textContent gets or sets the plain text inside an element — anything that looks like a tag is treated as literal characters. innerHTML gets or sets the element's inner markup, so a string like "<strong>Hi</strong>" is parsed into real bold text.

Attributes have their own methods: getAttribute, setAttribute, and removeAttribute. Many common attributes (id, src, href, value) are also exposed as direct properties, e.g. img.src = '...'.

  • textContent — plain text; safe, fast, no markup parsing.
  • innerHTML — parses an HTML string into elements.
  • setAttribute(name, value) / getAttribute(name) for any attribute.
  • Common attributes double as properties: el.id, el.src, input.value.

The paragraph text is replaced; the empty div is filled with parsed bold markup.

Common pitfall Never put user-typed text into innerHTML — a value like <img src=x onerror=...> would run as code (a cross-site-scripting hole). For anything a person typed, use textContent, which cannot execute.

Setting Styles with JavaScript

Objective: Change appearance from script using the style property and, more cleanly, classList.

Every element has a style object whose properties map to CSS. The names switch to camelCase because a hyphen is not legal in a JavaScript identifier: background-color becomes style.backgroundColor, font-size becomes style.fontSize. Values are always strings, units included ('16px').

Setting many inline styles gets unwieldy fast. The better pattern is to define a class in your CSS and toggle it with classList: add, remove, toggle, and contains. This keeps styling in the stylesheet and lets JavaScript flip a single switch.

  • el.style.color = 'red' sets one inline property; names are camelCase.
  • Style values are strings and need units: '10px', not 10.
  • classList.toggle('active') adds the class if absent, removes it if present.
  • Prefer class toggling over piling on inline styles.

Each click toggles the .highlight class — CSS owns the look, JS owns the switch.

Common pitfall A bare number is silently ignored: el.style.width = 10 does nothing at all — no error, no change. Style values must be strings with units, so write el.style.width = '10px'. The failure is easy to miss precisely because nothing throws.

Forms and Input Values

Objective: Read and set what the user has typed using the value property, and check boxes with checked.

Form controls carry their current contents in a property, not in the HTML. For text inputs, textareas, and selects, that property is value — read it to find out what the user typed, assign to it to pre-fill or clear the field. Checkboxes and radio buttons instead expose a boolean checked property.

Reading value at the right moment matters: fetch it inside an event handler (a button click, a form submit) so you get the latest text, not whatever was there when the page loaded.

  • input.value reads and writes the field's text.
  • checkbox.checked is a boolean, not a string.
  • Text-input value is always a string — convert with Number(...) for maths.
  • Read values inside an event handler to capture the current input.

Event Listeners

Objective: React to user actions with addEventListener, and inspect what happened through the event object.

addEventListener(type, handler) tells an element to call your function whenever an event of that type occurs — 'click', 'mouseover', 'keydown', and dozens more. Its big advantage over the old el.onclick = ... style is that you can attach many listeners for the same event without them overwriting each other.

Your handler receives an event object describing what happened. The most useful property is event.target — the element the event originated on — so one handler can serve many elements. Events also bubble: a click on a child fires listeners on that child, then its parent, then up the tree to document.

A listener can be removed later with el.removeEventListener('click', handler) — but this only works if you pass the same named function you attached, so an inline arrow function can never be removed (there is no reference to hand back). addEventListener also takes an optional third argument: pass an options object like { once: true } to auto-remove the listener after it fires once, or { capture: true } (equivalently a bare true) to run the listener during the capture phase — the downward trip from document to the target, the inverse of bubbling.

  • el.addEventListener('click', fn) — attach; multiple listeners coexist.
  • The handler's first argument is the event object.
  • event.target is where the event started.
  • Events bubble up from the target through its ancestors; capture is the reverse direction.
  • removeEventListener needs a named handler; the { once, capture } options tune when/how often it runs.

The button relabels itself using event.target — the handler never hard-codes the element.

Common pitfall Assigning el.onclick = fn keeps only the last function you set — a second assignment silently replaces the first. addEventListener stacks handlers instead, which is why it is the modern default.

Handling Element Events

Objective: Wire up the everyday element events — click, input, change, and submit — and stop a form from reloading the page.

Different controls fire different events. input fires on every keystroke in a text field, ideal for live previews. change fires once the user commits a value and moves on (leaving a text box, toggling a checkbox, picking a select option). submit fires on the <form> itself when the user submits.

By default a form submission reloads the page. In a JavaScript-driven app you almost always call event.preventDefault() in the submit handler to cancel that reload and handle the data yourself.

  • input — fires on every keystroke; great for live feedback.
  • change — fires when a value is committed (blur, toggle, select).
  • submit — fires on the form; pair with preventDefault().
  • event.preventDefault() cancels the browser's default action.

The paragraph updates as you type; submitting logs the value instead of reloading.

Page Events

Objective: Respond to page-level events: knowing when the DOM is ready, and listening for keyboard input with keydown and keyup.

Some events belong to the page as a whole rather than one element, so you attach them to document or window. The most important is DOMContentLoaded on document: it fires once the HTML has been fully parsed and the DOM tree exists. Running your setup code inside it guarantees your querySelector calls find their targets, even if the script loads in the <head>.

Keyboard events also live at the document level. keydown fires as a key goes down (and repeats if held); keyup fires when it is released. The event's key property is the character or name ('a', 'Enter', 'ArrowLeft').

  • DOMContentLoaded fires when the DOM is parsed and safe to touch.
  • keydown / keyup report keyboard activity on the whole page.
  • event.key gives the pressed key's value or name.
  • Page-level events attach to document or window.

Creating Elements

Objective: Build new DOM nodes from scratch with createElement and attach them with appendChild — including generating a list from an array of data.

You are not limited to elements written in the HTML; you can manufacture new ones. The recipe is three steps: document.createElement('li') makes a detached node, you fill it (set textContent, classes, attributes), then parent.appendChild(node) inserts it into the tree so it appears on screen. Until you append it, the node exists only in memory.

This is the heart of data-driven UI: loop over an array and create one element per item. Change the data, re-render, and the page follows.

  • document.createElement(tag) makes a node that is not yet on the page.
  • Configure it (textContent, classList, attributes) before or after inserting.
  • parent.appendChild(node) / parent.append(node) puts it in the tree.
  • Loop over data to generate a list, table, or gallery.

The empty <ul> fills with one <li> per array item — pure data-to-DOM.

Advanced: append vs appendChild

append() is the newer method: it accepts multiple nodes and plain strings in one call (parent.append(node, ' text', another)) and returns nothing. appendChild() takes a single node and returns it. For a single element either works; reach for append when you want to insert text or several items at once.

Moving Elements

Objective: Reorder nodes in the tree and understand why appending an element that already exists moves it rather than copying it.

A DOM node can occupy exactly one place in the tree at a time. So when you pass an existing element to appendChild or insertBefore, the browser removes it from its old spot and drops it in the new one — no clone is made. That single rule lets you reorder a list, promote an item to the top, or rotate elements with almost no code.

For positional movement (as opposed to reordering) you set position and adjust left/top through the style object — the same technique the animation lecture builds on.

  • Re-appending an existing node moves it; it is never duplicated.
  • parent.insertBefore(node, reference) drops a node ahead of another.
  • parent.prepend(node) moves a node to the front of its parent.
  • For pixel movement, set position then update style.left / style.top.

Each click rotates the list — the same three nodes, just reordered, never cloned.

Animation with requestAnimationFrame

Objective: Produce smooth, frame-based motion by letting the browser call your update function once per repaint with requestAnimationFrame.

To animate, you nudge an element a little, then again, then again — many times a second. You could use setInterval, but it ignores the display's refresh rate and keeps running in background tabs. requestAnimationFrame(callback) asks the browser to run your callback right before the next repaint, typically about 60 times per second, and pauses automatically when the tab is hidden — smoother motion and less wasted work.

The pattern is a self-scheduling loop: your frame function updates the state, applies it to the element's style, then calls requestAnimationFrame again to queue the next frame. The browser passes your callback one argument — a high-resolution timestamp (a DOMHighResTimeStamp, in milliseconds since page load) marking when this repaint began. To stop a loop, capture the id that requestAnimationFrame returns and pass it to cancelAnimationFrame(id).

  • requestAnimationFrame(fn) runs fn once, just before the next paint, and returns an id.
  • Your callback receives a millisecond timestamp; compare timestamps across frames to measure elapsed time.
  • Re-request inside the callback to keep the loop going; cancelAnimationFrame(id) halts it.
  • It syncs to the screen refresh and pauses in hidden tabs.
  • Update state, write to style, request the next frame — repeat.

The ball bounces between the track's edges — one small position change per frame. Press Stop to cancel the loop.

Common pitfall Moving dir = 2 pixels per frame ties speed to the display: on a 120 Hz screen the callback runs twice as often, so the ball travels twice as fast as on a 60 Hz one. For frame-rate-independent motion, scale the movement by elapsed time using the callback's timestamp — remember the previous timestamp, compute delta = timestamp - last, and move speed * delta pixels. (This is the opposite trade-off from setInterval, whose interval is fixed in milliseconds but out of sync with the actual repaint.)

Section quiz

1. What is the difference between querySelector and querySelectorAll, and what does each return when nothing matches?

Show answer

querySelector returns the first element matching the CSS selector, or null if none match. querySelectorAll returns a NodeList of all matches, or an empty NodeList (length 0) if none match. You can loop a NodeList with forEach.

2. Why should you use textContent rather than innerHTML when inserting text a user typed?

Show answer

innerHTML parses its string as HTML, so malicious input like <img src=x onerror=...> could execute — a cross-site-scripting (XSS) vulnerability. textContent treats everything as literal text and never runs code, so it is safe for untrusted input.

3. You call parent.appendChild(existingNode) where existingNode is already somewhere in the page. What happens to it, and why?

Show answer

It is moved, not copied. A DOM node can exist in only one place in the tree, so appending a node that is already attached removes it from its old parent and inserts it at the new location. This is how you reorder lists with a single line.

4. Why prefer el.addEventListener('click', fn) over el.onclick = fn?

Show answer

addEventListener lets you attach many handlers for the same event on the same element, and they all run. el.onclick is a single property, so a second assignment silently overwrites the first — you can only ever have one handler. addEventListener also supports options like { once: true } and capture-phase listening, and pairs with removeEventListener.

5. In a form's submit handler, what does event.preventDefault() do, and why is it usually needed?

Show answer

It cancels the browser's default action for the event — for a form submit, that default is navigating/reloading the page (sending the data to the server and loading the response). In a JavaScript-driven app you want to keep the current page and handle the data yourself (validate, send with fetch, update the DOM), so you call preventDefault() to stop the reload.

6. Why is requestAnimationFrame preferred over setInterval for animation?

Show answer

requestAnimationFrame runs your callback in sync with the display's repaint (typically ~60 times/second) so motion is smooth with no dropped or doubled frames, and it automatically pauses in hidden/background tabs, saving CPU and battery. setInterval fires on a fixed millisecond schedule that is unaware of the refresh rate and keeps running when the tab is not visible, wasting work and producing juddery motion.

A Responsive Navbar with CSS

Objective: Style the list of links into a horizontal bar on wide screens that collapses into a vertical, stacked menu on narrow screens using a media query.

Straight out of the box a <ul> shows bullet points and stacks vertically. To make a bar, remove the list styling (list-style:none, zero margin and padding) and lay the items in a row. There are modern one-liners for this (flexbox), but this project follows the classic technique: set each <li> to display:inline-block so the items sit side by side, and make each link a display:block so its whole rectangle — not just the text — is clickable and paddable.

A media query then supplies a different set of rules below a chosen width (the "breakpoint"). Below 740px we flip the items back to display:block so they stack full-width, one link per row — the familiar mobile menu. Nothing about the HTML changes; only the rules that apply change with the viewport.

  • list-style:none plus margin:0; padding:0 strips the default list look.
  • display:inline-block on <li> lays items in a row; display:block on <a> makes the full box clickable.
  • @media (max-width:740px){ ... } holds the rules that apply only on narrow screens.
  • Same HTML, different CSS per width — that is what "responsive" means.

Notice how each link fills its own row once the preview is narrower than the breakpoint.

Adding Google Fonts

Objective: Load a hosted web font and apply it across the site, understanding the @import / <link> syntax and how a font stack falls back.

By default browsers render text in a plain system font. A web-font service such as Google Fonts hosts thousands of families you can pull in without downloading anything: you add a <link> in the <head> (or an @import at the very top of your CSS), then name the family in a font-family rule. Always end the stack with a generic fallback like sans-serif so the page stays readable while the web font downloads — or if it never arrives.

Because this guide runs fully offline, the live demo below uses a system font stack rather than a network font. The two static blocks show the exact syntax you would paste into a real project.

While the web font downloads the browser has to decide what to show. Two behaviours have names: FOIT (Flash Of Invisible Text) hides the text until the font arrives — bad on slow connections, the reader stares at a blank space; FOUT (Flash Of Unstyled Text) shows the fallback font immediately and swaps when the web font loads. The &display=swap parameter already sitting in the Google Fonts URL requests FOUT — text is readable instantly and simply restyles a moment later, which is almost always what you want. That is the real reason the parameter is there.

  • Load once in the <head> with <link>, or at the top of the stylesheet with @import.
  • Apply with font-family: 'Family Name', fallback; — the last entry is a generic family.
  • Set the body font once on body; every element inherits it unless overridden.
  • Web fonts cost a network request; a good fallback prevents invisible or jumping text.
Background The two ways to load a Google font. The <link> tag is generally preferred because the browser can start fetching it in parallel with the CSS, whereas @import blocks until the stylesheet is parsed.

Main Content Columns

Objective: Style the main content: float an image beside the text, add separators between articles, fix layout with box-sizing and overflow, and dress up the first letter of each block.

Inside each article we float the image to one side so the paragraph text wraps around it. But a floated element is removed from normal flow — the article no longer measures it when computing its own height. If the image is taller than the wrapping text, it sticks out of the bottom of the article and overlaps the next block. The classic fix is overflow:hidden on the article: that establishes a new block formatting context, which forces the box to grow tall enough to contain its float. (This is exactly the same collapse you will clear with a ::after clearfix in the footer lecture.)

Two more touches make the content feel designed. box-sizing:border-box (applied to every element with the universal selector *) makes width include padding and border, so a width you set is the width you get — no surprise overflow. And the ::first-letter pseudo-element selects the first typographic letter of a block-level element (only block containers — it does nothing on an inline element), letting you enlarge and color the opening letter of a paragraph for a magazine "drop cap" effect.

  • float:left on an image wraps following text around it.
  • overflow:hidden on the parent contains a floated child so it stops overlapping the next block.
  • * { box-sizing: border-box } makes padding and border count inside the declared width.
  • p::first-letter targets just the first character — great for a drop cap.

Delete overflow:hidden and re-run — watch the image escape its article and overlap the next one.

Footer Columns

Objective: Build a multi-column footer with floats that collapses to a single stacked column on small screens, and clear the floats correctly so the footer background fills.

Give the columns percentage widths that add up to 100% (here 25% / 50% / 25%). Then inside the mobile media query set each column to width:100% so they stack. This is the same responsive pattern as the navbar: floats and fixed proportions on wide screens, full-width stacking on narrow ones.

  • In the mobile query, set every column to width:100% so they stack.

Navbar JavaScript

Objective: Add a menu button that shows and hides the navigation on small screens by toggling a CSS class with JavaScript.

CSS alone can lay the menu out responsively, but showing and hiding it on a tap needs JavaScript.   • document.querySelector('.menu') grabs the button; a class selector starts with a dot.
  • element.classList.toggle('hide') flips a class on and off in one call.
  • Let CSS own the look (.hide{display:none}); let JS own only the switching.
  • On wide screens you would leave the menu visible; the toggle matters on mobile.

Section quiz

1. Why wrap navigation links in a <ul> inside <nav> instead of using bare <a> tags?

Show answer

A navbar is semantically a list of links. The <ul>/<li> structure communicates that to screen readers and search engines, and the visible bullets are trivially removed with list-style:none. <nav> marks the block as the primary navigation region.

2. A floated image is overlapping the block below it. What single CSS rule on the image's parent is the classic fix, and why does it work?

Show answer

overflow:hidden on the parent. It establishes a new block formatting context, forcing the parent to grow tall enough to contain its floated children instead of collapsing around them. (A ::after clearfix achieves the same for a footer of floated columns.)

3. What does classList.toggle('hide') do, and what does it return?

Show answer

It adds the class hide if the element does not have it, and removes it if it does — one call handles both opening and closing. It returns true when the class ends up present (added) and false when it ends up absent (removed).

4. The exact same HTML shows the nav links in a row on a laptop but stacked in a column on a phone. What CSS makes that happen, and what does @media (max-width:740px){ ... } mean?

Show answer

A media query. The rules inside @media (max-width:740px){ ... } apply only when the viewport is 740px wide or narrower. The base CSS sets li{display:inline-block} (a row); the media query overrides it with li{display:block; width:100%} (a stack). The markup never changes — only which rules are active changes with the viewport width, which is the whole idea of responsive design.

5. You set an element to width:100% and add padding:16px and a 1px border, and it overflows its parent. Why — and what one declaration fixes it?

Show answer

By default (box-sizing:content-box) width is the content box only, so padding and border are added on top of 100%, making the element wider than its parent. * { box-sizing: border-box } makes width include padding and border, so the element stays exactly 100% wide. This is also why floated columns whose percentages plus borders exceed 100% wrap to a new row without it.

6. Your media query works when you drag the desktop browser narrow, but the site still looks like a shrunken desktop page on a real phone. What is almost certainly missing?

Show answer

The <meta name="viewport" content="width=device-width, initial-scale=1"> tag in the <head>. Without it a phone renders at a virtual ~980px width and scales the result down, so a max-width:740px query is never satisfied and the mobile layout never appears.

Arrow Function Expressions

Objective: Write functions with the concise arrow syntax, use implicit return, and understand how this behaves differently from a normal function.

An arrow function is a shorter way to write a function expression. You drop the function keyword and place a fat arrow => between the parameter list and the body. When the body is a single expression you can omit the braces and the return keyword — the value of that expression is returned automatically (an implicit return).

Arrows shine as short callbacks passed to array methods like map, filter, and reduce. Their headline behavioural difference is that an arrow does not get its own this; it inherits this from the surrounding code. That is usually helpful, but it means an arrow is the wrong tool for an object method that needs to refer to its own object. Two related consequences follow: an arrow has no own arguments object (it borrows the enclosing one), and it cannot be called with new — arrows are not constructors. So reach for a normal function for object methods, constructors, and anything relying on arguments.

  • One parameter can skip the parentheses; zero or multiple parameters need them: x => … vs () => … vs (a, b) => ….
  • No braces means implicit return; add braces and you must write return yourself.
  • Arrows inherit this — great for callbacks, wrong for object methods.
  • No own arguments and no new: arrows can't be methods that use this, constructors, or arguments-based functions.
  • Default parameter values work just like in normal functions.

Notice how the arrow callbacks read almost like a sentence: filter the evens, map to squares, reduce to a sum.

Array .every(): Compare Contents

Objective: Use .every() to test whether all elements of an array satisfy a condition, returning a single true/false answer.

array.every(callback) runs your test function on each element. It returns true only if the callback returns a truthy value for every element; the moment one element fails, it stops early and returns false. Its mirror image is .some(), which returns true if at least one element passes.

A handy use is comparing two arrays for equality: check that they are the same length and that every element at index i matches the other array's element at the same index.

  • .every() is a short-circuiting AND across the array — one failure ends it.
  • On an empty array it returns true (there is nothing that fails).
  • The callback receives (element, index, array) — the index is useful for comparisons.
  • Pair it with .some() when you instead need "at least one".

Change a value in one of the arrays and re-run to watch sameContents flip to false.

Common pitfall .every() on an empty array returns true, which can surprise you (e.g. "are all items in stock?" answers true for an empty cart). Guard with a length check first if an empty array should mean "no".

Summing Array Values with a Loop

Objective: Accumulate a running total across an array using an explicit loop, and see the shorter reduce() equivalent.

The classic accumulator pattern is: start a variable at 0, walk the array, and add each element to that variable. A for…of loop reads cleanly because you name the element directly instead of indexing by number. Once you are comfortable with the loop, reduce() expresses the very same idea in one line.

  • Initialise the accumulator before the loop, not inside it.
  • for…of gives you each value; for (let i…) gives you the index if you also need position.
  • reduce((sum, n) => sum + n, 0) is the idiomatic one-liner — the 0 is the starting value.
  • The same pattern works for products, maximums, or building a string.
  • Never sum an array with for…in — it hands you string keys, not values.
Common pitfall Do not use for…in to add up an array. for…in iterates the keys as strings ("0", "1", …) and can even pick up inherited or extra properties, so total += i string-concatenates ("0" + "1" + …) instead of adding. Use for…of (values) or a classic indexed for (let i = 0; i < arr.length; i++). In the indexed form, cache the length — for (let i = 0, n = arr.length; i < n; i++) — so it isn't re-read every iteration.
const prices = [12.5, 4, 8.25, 30];

// Explicit accumulator loop
let total = 0;
for (const price of prices) {
  total += price;
}
console.log("loop total:", total);

// Same result with reduce()
const total2 = prices.reduce(
  (sum, p) => sum + p,
  0
);
console.log("reduce total:", total2);

// Average
console.log("average:", total / prices.length);

Closures: A Simple Example

Objective: See a function that "remembers" a variable from the scope where it was created, even after that outer function has finished.

When a function is defined inside another function, the inner function keeps a live link to the outer function's variables. That combination — an inner function plus the surrounding variables it can still reach — is a closure. The classic demonstration is a counter: an outer function declares count and returns an inner function that increments it. Each call to the returned function sees the same count.

  • The inner function has access to the outer function's variables — that link survives after the outer function returns.
  • The remembered variable is not reset between calls; it persists.
  • Each call to the outer function creates a fresh, independent closure.

Closures Explained

Objective: Understand why closures work and recognise where they show up in everyday JavaScript.

Every function in JavaScript remembers the environment in which it was created — a chain of scopes reaching outward to the global scope. When you return an inner function, that scope chain goes with it, so the variables it referenced stay alive in memory instead of being discarded. That is all a closure is: a function bundled with the variables it closed over.

You have already been using closures without naming them. Every event handler that references a variable from the surrounding function is a closure. So is any callback passed to setTimeout, and every "private counter" or memoised helper.

  • A closure = a function + the scope it was defined in.
  • Captured variables persist because the function still references them.
  • Real uses: private state, event handlers, timers, function factories, memoisation.
  • Closures capture the variable, not a snapshot of its value — mind loop variables.
Common pitfall Closures capture the variable itself, not its value at capture time. A classic bug: for (var i = 0; i < 3; i++) setTimeout(() => console.log(i)) logs 3 3 3, because all three closures share one var i. Use let i instead — each loop iteration gets a fresh binding, and it logs 0 1 2.
Advanced: closures for memoisation

Because a closure can hold private state, it is perfect for caching results. The returned function keeps a cache object that outside code cannot see or corrupt:

function memoize(fn) {
  const cache = {};                 // private to the returned function
  return function (n) {
    if (n in cache) return cache[n];
    return (cache[n] = fn(n));
  };
}
const slowSquare = n => n * n;
const fastSquare = memoize(slowSquare);
fastSquare(9); // computes, stores 81
fastSquare(9); // returns cached 81

JSON and JavaScript

Objective: Convert between JavaScript objects and JSON text with JSON.stringify and JSON.parse.

JSON (JavaScript Object Notation) is a plain-text format for structured data — the lingua franca that servers and browsers use to exchange information. It looks like a JavaScript object literal, but with strict rules: keys must be double-quoted strings, and only strings, numbers, booleans, null, arrays, and objects are allowed (no functions, no undefined).

JSON.stringify(value) turns a live object into a JSON string you can store or send. JSON.parse(text) does the reverse, rebuilding a real object you can work with. A common beginner trick with stringify is passing a third argument for indentation to get readable output.

  • JSON.stringify(obj) → text; JSON.parse(text) → object.
  • Pass a number as the third stringify argument to pretty-print with that many spaces.
  • JSON keys are always double-quoted; single quotes or trailing commas are invalid.
  • Functions and undefined are silently dropped during stringify.

The stringify→parse round trip also gives you a cheap deep copy of plain data.

Common pitfall JSON.parse throws a SyntaxError on malformed text (a stray trailing comma, single quotes, or an empty string). When the data comes from outside your code, wrap the call in try { … } catch { … } so one bad payload does not crash the page.
The deep-copy trick has limits JSON.parse(JSON.stringify(x)) only round-trips plain JSON data. Anything JSON can't represent is lost or mangled: Date objects become strings, undefined, functions, and Symbols are dropped, NaN/Infinity turn into null, and Map/Set collapse to empty {}. For a faithful deep copy of those, use the built-in structuredClone(x), the modern purpose-built cloner (it still can't copy functions or DOM nodes, but it preserves Dates, Maps, Sets, typed arrays, and cyclic references).

Requesting Data: XHR vs Fetch

Objective: Request data over the network and understand why the promise-based fetch() API has replaced the older XMLHttpRequest.

For years the only way to load data without reloading the page was XMLHttpRequest (XHR) — powerful but verbose, built around callback properties like onload and onerror. Modern code uses fetch(), which returns a promise. A promise represents a value that will arrive later; you attach .then() to handle success and .catch() to handle failure, avoiding deeply nested callbacks.

With fetch the response arrives in two stages: first a Response object (headers, status), then the body once you call .json() or .text() — each of which is itself asynchronous, hence the second .then().

The most common fetch misconception is thinking a bad HTTP status rejects the promise. It does not. The promise rejects only on a genuine network failure (no connection, DNS error, or a CORS block) — an HTTP 404 or 500 still fulfils normally, just with response.ok === false. So your .catch() silently misses server errors unless you check response.ok yourself and throw when it is false, exactly as the demo below does.

  • fetch(url) returns a promise resolving to a Response.
  • Call response.json() (or .text()) — also a promise — to read the body.
  • .catch() handles network errors; check response.ok for HTTP errors like 404.
  • fetch is shorter and composable; XHR is legacy but still works.

Section quiz

1. What does n => n * 3 return, and why is there no return keyword?

Show answer

It returns n multiplied by 3. An arrow function with no braces uses an implicit return — the single expression after the arrow is automatically returned, so writing return would be redundant (and adding braces would then require it).

2. What does [].every(x => x > 100) evaluate to, and why can that be surprising?

Show answer

It evaluates to true. .every() returns true when no element fails the test, and an empty array has no elements to fail — so vacuously every element passes. Guard with a length check when "empty" should mean "no".

3. In function makeAdder(x){ return y => x + y; }, how do makeAdder(5) and makeAdder(10) keep their x values separate?

Show answer

Each call to makeAdder creates a new closure with its own captured x. The returned functions reference different, independent variable environments, so add5 remembers x = 5 while add10 remembers x = 10.

5. A fetch example logs "(this logs first)" before the fetched data, and a request to a missing URL that returns 404 never reaches .catch(). Explain both.

Show answer

Logs first: fetch is asynchronous. The synchronous code — including that console.log — runs to completion first; the .then() callbacks are queued and only run later, once the response arrives. 404 skips catch: fetch's promise rejects only on a network-level failure, not on HTTP error statuses. 6. When inserting text a user typed into the page, why prefer element.textContent = value over element.innerHTML = value?

Show answer

textContent treats the value as plain text, so any < or > shows literally and no markup is interpreted. innerHTML parses the string as HTML, so hostile input like <img src=x onerror=…> could inject and run script — a cross-site-scripting (XSS) hole. Custom map() Implementation

Objective: Recreate Array.prototype.map to understand how a higher-order function loops, calls a callback per element, and collects the returned values.

map is not magic. It walks an array from start to finish, hands each element to a function you supply, and pushes whatever that function returns into a brand-new array. The original array is never touched. Writing your own version makes the "callback per item" pattern concrete and shows why map always returns an array of the same length.

The real map passes three things to your callback: the current value, its index, and the whole array. Most callbacks only use the value, but supporting all three keeps our clone faithful.

  • map is a transform: input array in, new array out — same length, one result per element.
  • The callback signature is (value, index, array).
  • It never mutates the source; it builds and returns a fresh array.
  • A plain for loop plus push is all the machinery you need.

Notice the source array is identical after the calls — map transforms, it never edits in place.

Implement a Simple Promise

Objective: Build a minimal promise-like object that supports .then(), .catch(), and .finally() chaining, to demystify how async control flow is wired up.

A promise is just an object with a small state machine. It starts pending, then settles exactly once — to fulfilled (with a value) or rejected (with an error). Handlers you register with .then() are queued; when the promise settles, each queued handler runs. If a handler is registered after the promise has already settled, it runs right away.

The one rule that makes promises predictable is: a promise settles only once. Calling resolve after reject (or twice) does nothing. Our version enforces that with a single guard.

  • Three states: pending to fulfilled or rejected — one-way, one-time.
  • resolve(value) / reject(error) are supplied to your executor function.
  • Handlers run asynchronously — here we defer with setTimeout(…, 0) as a simplification. Real promise handlers run on the microtask queue, which drains before any setTimeout (macrotask) callback, so a native .then() fires before a setTimeout(…, 0) scheduled at the same moment.
  • .catch(fn) is .then(undefined, fn); .finally(fn) runs on either outcome.

Handlers registered while pending fire the moment the promise resolves; finally runs last, whatever the outcome.

Deep Cloning an Object

Objective: Write a function that copies a nested object or array so the copy shares no references with the original.

In JavaScript, objects and arrays are held by reference. A spread copy ({ ...obj }) or Object.assign only duplicates the top level: nested objects are still the same objects, shared between original and copy. Change a nested value on one and it changes on both. A deep clone walks the whole structure and rebuilds every nested object and array from scratch.

The trick is recursion. Primitives (strings, numbers, booleans, null) have nothing nested, so return them as-is. Arrays map each element through the cloner. Objects copy each own key through the cloner. Every branch bottoms out at a primitive.

  • Objects and arrays are copied by reference, not by value.
  • Spread and Object.assign are shallow — nested objects stay shared.
  • Deep clone recurses into every array element and object property.
  • Guard the base case: return anything that is not an object (including null) unchanged.

Editing shallow.address changed the original; editing deep.address did not.

Custom debounce()

Objective: Build a debounce helper that delays a function until a burst of rapid calls has stopped — essential for input, resize, and scroll handlers.

Some events fire dozens of times a second: typing in a search box, resizing the window, scrolling. Running expensive work on every one of those is wasteful. Debouncing waits until the activity pauses: each new call cancels the previous pending timer and starts a fresh one, so the wrapped function runs only once, after the last call plus the delay.

The pattern relies on a closure: the returned function remembers a timeoutId across calls. clearTimeout cancels the still-pending run; a new setTimeout schedules the next one. Using rest args (...args) forwards whatever arguments the event passes.

  • Debounce runs a function only after calls stop for delay milliseconds.
  • Each call cancels the previous timer via a closed-over timeoutId.
  • Great for search-as-you-type, window resize, and autosave.
  • Debounce waits for a pause; throttle (a cousin) runs at most once per interval instead.

Type fast: the keystroke counter climbs every key, but the search count only ticks up 500ms after you stop.

// map: transform each item into a NEW array (same length, no mutation)
function myMap(arr, cb) {
  const out = [];
  for (let i = 0; i < arr.length; i++) out.push(cb(arr[i], i, arr));
  return out;
}

// promise: pending -> fulfilled | rejected, settles ONCE
new Promise((resolve, reject) => { /* call one of them */ });

// deep clone: recurse; return primitives, rebuild arrays & objects
function deepClone(v) {
  if (v === null || typeof v !== "object") return v;   // null check first!
  if (Array.isArray(v)) return v.map(deepClone);
  const o = {};
  for (const k in v) if (Object.hasOwn(v, k)) o[k] = deepClone(v[k]);
  return o;
}
// modern built-in alternative: structuredClone(obj)

// debounce: run only after calls stop for `delay` ms
// regular function + apply preserves `this` for method calls
function debounce(fn, delay) {
  let id;
  return function (...args) {
    clearTimeout(id);
    id = setTimeout(() => fn.apply(this, args), delay);
  };
}

1. Your custom map callback forgets to return anything. What does the resulting array contain, and why?

Show answer

An array of the same length filled with undefined. map pushes whatever the callback returns; a function with no return returns undefined, so every slot gets undefined.

2. Why does a spread copy like { ...user } fail to fully isolate a nested user.address object?

Show answer

Spread is a shallow copy: it duplicates only the top-level properties. The value of address is a reference, and that same reference is copied — so both objects point at one shared nested object. A deep clone recurses and rebuilds nested objects instead.

3. In debounce, what would break if you moved let timeoutId; inside the returned function instead of outside it?

Show answer

Each call would create a brand-new timeoutId local to that invocation, so clearTimeout would never cancel the previous timer. The whole point — remembering the pending timer across calls via the closure — would be lost, and the function would fire on every call.

4. In the MyPromise above, what guarantees a promise settles only once? If resolve("ok") runs and then reject(err) is called a moment later, what happens?

Show answer

The single guard if (this.state !== "pending") return; at the top of settle(). Both resolve and reject are settle(...) in disguise, so the first one to run flips state away from "pending" and records the value; every later call hits the guard and returns immediately. So resolve("ok") wins — the promise is fulfilled with "ok", the later reject(err) is silently ignored, and the value never changes.

Deploy to GitHub Pages from Local

Objective: Push a local project to a repository and serve it as a live site with a public URL — for free.

A page on your hard drive is only visible to you. To share it, the files need to live on a web server. GitHub Pages is a free host that turns any public repository of static files (HTML, CSS, JS, images) into a live website. You do not need a server of your own, a build step, or a credit card — GitHub serves the files directly.

The workflow is a loop you will repeat forever: edit files locally, stage the changes, commit them into your project's history, then push the commit up to GitHub. A minute or two after each push, the live site updates. Git tracks every version, so nothing is ever lost and you can always roll back.

  • GitHub Pages hosts static sites for free from a public repo.
  • The core Git cycle is add → commit → push.
  • Your entry file must be named index.html — Pages serves it as the site root.
  • Enable Pages under Settings → Pages and choose the branch (usually main).
  • Deploys are not instant: allow a minute or two for changes to appear.
Background Git is the version-control tool that runs on your machine; GitHub is a website that hosts Git repositories in the cloud. Install Git first (git --version confirms it is available), then create a free account at github.com. An editor with an integrated terminal, such as VS Code, keeps the code and the commands in one window.

A JavaScript Coding Exercise

Objective: Combine DOM selection, events, and a small piece of logic into one working feature — a random-number generator with a live result on the page.

Here is a compact challenge that touches every core skill from the course. Write a function that returns a random whole number between a min and a max value, inclusive of both ends, then wire it to a button so each click shows a fresh number on the page.

The math is the interesting part. Math.random() returns a decimal in the range [0, 1) — zero is possible, but 1 is never reached. To land on an integer from min to max and include max, scale by the size of the range plus one, floor the result, then shift up by min:

function randomBetween(min, max) {
  // (max - min + 1) makes the top value reachable;
  // Math.floor drops the decimal; + min shifts the range.
  return Math.floor(Math.random() * (max - min + 1)) + min;
}

console.log(randomBetween(1, 6));  // a fair dice roll: 1..6
console.log(randomBetween(0, 2));  // 0, 1, or 2
Console4 2

Why + 1? Suppose min = 1 and max = 6. The range width is 6 - 1 + 1 = 6. Math.random() * 6 spans [0, 6), and flooring gives an integer 0..5; adding min shifts that to 1..6. Drop the + 1 and you could never roll a 6.

  • Math.random() is in [0, 1) — never exactly 1.
  • Inclusive integer formula: Math.floor(Math.random() * (max - min + 1)) + min.
  • Read inputs with input.value, and convert with Number(...) — values are strings.
  • Update the page with textContent inside a click handler.
