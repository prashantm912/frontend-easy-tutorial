JavaScript

JavaScript is the language of the web platform: a dynamically typed, single-threaded, event-driven runtime with first-class functions and prototypal objects. This page moves from the fundamentals — types, operators, control flow — through closures, objects, the DOM, and event handling, and then into the runtime concurrency model: promises, the event loop, workers, and service-worker caching. Every runnable example executes in a sandboxed iframe; JavaScript-only examples print their results through console.log, which the preview captures into a console panel below the code.

Fundamentals

JavaScript has seven primitive types — string, number, boolean, null, undefined, bigint, and symbol — plus the object type for everything else (arrays, functions, plain objects). Primitives are immutable and compared by value; objects are compared by reference. The typeof operator reports a value's type as a string, with the historical quirk that typeof null returns "object".

Declare bindings with const by default and let when you must reassign; both are block-scoped and live in the temporal dead zone until initialized. Avoid var, which is function-scoped and hoisted in ways that surprise. Template literals (backticks) support interpolation and multi-line strings.

Distinguish coercion (implicit type changes the engine performs, e.g. during == or +) from conversion (explicit calls like Number(x) or String(x)). Prefer strict equality ===, which never coerces, over loose ==, whose coercion rules are a frequent source of bugs.

Note: Prefer const and === as defaults. The one broadly accepted use of == is x == null, which is true for exactly null and undefined.

Operators & control flow

Beyond the arithmetic (+ - * / % **) and comparison operators, modern JavaScript offers the nullish coalescing operator ?? (falls back only for null/undefined, unlike || which falls back for any falsy value) and optional chaining ?. (short-circuits to undefined if a reference is nullish instead of throwing).

Logical operators short-circuit: a && b evaluates b only when a is truthy, and a || b evaluates b only when a is falsy — both return the value that decided the result, not a boolean. Control flow uses if/else, the switch statement, the ternary ?: expression, and the loop family for, for…of (iterables/values), for…in (enumerable keys), and while.

Example: switch, ternary, and the loop family

Note: Use for…of for arrays and other iterables; use for…in only for plain-object keys, since it also walks inherited enumerable properties. To iterate an array's index/value pairs cleanly, use for (const [i, v] of arr.entries()).

Functions & closures

Functions come in three flavors: declarations (function f() {}, hoisted), expressions (const f = function () {}, not hoisted), and arrow functions (const f = () => {}). Arrows have no this, arguments, or prototype of their own — they capture this lexically from the enclosing scope, which makes them ideal for callbacks but unsuitable as methods or constructors.

A closure is a function together with the lexical environment it was created in. Because inner functions retain access to their outer variables even after the outer function returns, closures let you build factories with private, encapsulated state. Default parameters supply fallbacks, and rest parameters (...args) collect remaining arguments into a real array.

Note: A classic pitfall is capturing a loop variable declared with var in a closure — every closure shares one binding. Using let gives each iteration a fresh binding, which is almost always what you want.

Objects & arrays

Object and array literals are the workhorses of JavaScript data. Destructuring pulls values out by shape (const {a, b} = obj, const [x, y] = arr), the spread operator copies or merges ({...a, ...b}, [...arr]), and rest gathers the remainder. The array methods map, filter, and reduce express transformations declaratively without mutating the source.

For keyed collections beyond plain objects, use Map (any key type, ordered, size-aware) and Set (unique values). JSON.stringify/JSON.parse serialize and revive plain data. Favor immutable updates — produce new objects/arrays rather than mutating — which makes state changes predictable and easy to reason about.

Example: map/filter/reduce, JSON, immutable update

Note: JSON.stringify drops undefined, functions, and symbols, turns Date into a string, and throws on circular references. For a true deep clone of structured data (including Map, Set, typed arrays, cycles), prefer the built-in structuredClone(value).

DOM manipulation

The DOM API lets scripts read and mutate the page. document.querySelector returns the first match for a CSS selector; querySelectorAll returns a static NodeList of all matches. Build nodes with document.createElement, set their text with textContent, and attach them with append/appendChild.

Prefer textContent over innerHTML when inserting user-supplied data: innerHTML parses its input as markup, so untrusted strings can inject scripts (an XSS vector). Manage classes with the classList API, read/write data-* attributes through the dataset object, and use getAttribute/setAttribute for arbitrary attributes.

Note: When you build interactive DOM, keep it accessible — use semantic elements, set textContent for labels, and add ARIA only where native semantics fall short. Screen readers announce well-structured markup for free.

Events & delegation

Attach handlers with element.addEventListener(type, handler, options). The handler receives an Event object exposing target (where it originated), currentTarget (where the listener is attached), and methods like preventDefault() (cancel default behavior) and stopPropagation() (halt further travel). Events flow in three phases: capturing (root → target), the target itself, then bubbling (target → root).

Event delegation exploits bubbling: instead of binding a listener to every child, bind one listener to a shared ancestor and inspect event.target.closest(selector) to find which child was acted on. This uses less memory, requires no rebinding when children are added or removed, and keeps handler logic in one place.

Example: a direct handler with the event object

Example: event delegation on a list container

Note: Delegation scales far better than per-child listeners. A list of 10,000 rows needs one listener, not 10,000; and dynamically added rows work immediately with no extra bookkeeping. Match with closest() so clicks on nested elements still resolve to the right row.

Async & promises

A Promise represents a value that will be available later. It settles once — either fulfilled (via resolve) or rejected (via reject) — and you react with .then, .catch, and .finally. The combinators Promise.all (wait for all, reject on first failure), Promise.allSettled (wait for all, never rejects), and Promise.race (settle as soon as the first does) coordinate multiple async operations.

The async/await syntax is sugar over promises: an async function returns a promise, and await pauses it until the awaited promise settles, letting you write sequential-looking code with ordinary try/catch error handling. Below, because this site runs offline, a mockFetch resolves fake JSON via setTimeout; the real fetch() call is shown in a comment and works identically.

Note: await in a loop runs requests sequentially. To run independent requests concurrently, start them all first and then await Promise.all([...]) — this often turns seconds of waiting into milliseconds.

The event loop

JavaScript runs on a single thread with an event loop. Synchronous code runs to completion on the call stack. When the stack empties, the loop drains the microtask queue (promise callbacks, queueMicrotask) completely, then takes exactly one macrotask (a setTimeout callback, an I/O event) and repeats. Microtasks therefore always run before the next macrotask, even a setTimeout(fn, 0).

This ordering explains a common interview puzzle: synchronous logs print first, then any queued promise callbacks, and only then timer callbacks. Understanding it prevents subtle bugs where you assume a timer fires before promise resolution completes.

Note: Long synchronous work blocks the single thread — rendering, input, and timers all stall. Break heavy loops into chunks (e.g. via setTimeout, requestIdleCallback, or a Web Worker) so the loop can service other tasks.

Debounce & throttle

Debounce delays invoking a function until a quiet period has elapsed since the last call — ideal for reacting to the end of a burst (search-as-you-type, resize, autosave). Throttle caps invocation to at most once per interval regardless of how often it is called — ideal for steady sampling during a burst (scroll position, pointer move, analytics pings).

Both are built with closures over a timer. Debounce resets its timer on every call; throttle ignores calls until its cooldown expires. The examples below simulate a rapid stream of events and log how many times the raw handler would fire versus the wrapped one.

Note: Debounce and throttle protect expensive handlers (network calls, layout, re-renders) from firing on every high-frequency event. Reach for debounce when only the settled state matters, and throttle when you need periodic updates during the activity.

Modules

ES modules give each file its own scope with explicit export and import. You can have any number of named exports and at most one default export. Imports are statically analyzable, which lets bundlers perform tree-shaking (dropping unused exports) and code-splitting (loading chunks on demand). The import() expression loads a module dynamically at runtime and returns a promise for its namespace.

A sandboxed srcdoc cannot fetch external files, but it can construct a module from a Blob URL and import() it — a faithful, runnable demonstration of module loading. The static source form is shown first as a reference code block.

Example: the static ESM source (reference)

// math.js — a module with named + default exports
export const PI = 3.14159;
export function square(x) { return x * x; }
export default function add(a, b) { return a + b; }

// app.js — importing from it
import add, { PI, square } from "./math.js";
console.log(add(2, 3));   // 5
console.log(square(4));   // 16
console.log(PI);          // 3.14159

// Dynamic import: load on demand, returns a Promise<namespace>
const mod = await import("./math.js");
console.log(mod.default(10, 20)); // 30
Result: logs 5, 16, 3.14159, then 30 once the dynamic import resolves. A bundler would tree-shake square out of the final bundle if nothing imported it.

Note: Native ESM works in every modern browser via <script type="module">. Bundlers (Vite, esbuild, Rollup, webpack) still add value: tree-shaking, code-splitting, minification, and consolidating many small requests into optimized chunks.

Patterns

Design patterns package recurring solutions. The revealing module pattern uses a closure (often an IIFE) to keep state private and return only the public API. The observer pattern lets a subject notify a list of dependents when it changes. Pub/sub (an event emitter) decouples publishers from subscribers through named topics, so neither side references the other directly.

These patterns underpin much of the ecosystem: reactive state libraries are observers, event buses are pub/sub, and encapsulated singletons are revealing modules. The runnable examples below implement a small event emitter and a subject/observer pair from scratch.

Performance & memory

JavaScript is garbage-collected, but memory still leaks when references keep objects reachable longer than intended. Common culprits: intervals and event listeners that are never removed, growing global caches, and closures that retain detached DOM nodes. A leak shows up as memory that climbs and never comes back down.

The fixes are disciplined cleanup: clearInterval/clearTimeout, removeEventListener, bounding or clearing caches, and using WeakMap/WeakSet so entries vanish once their keys are otherwise unreachable. The first example demonstrates a leak; the second applies the fixes and shows the retained state shrinking.

Web Workers

Web Workers run JavaScript on a separate thread, keeping the main thread free for rendering and input. They communicate only by message passing: worker.postMessage(data) sends, and each side listens via onmessage. Because there is no shared mutable state, workers are ideal for CPU-heavy tasks — parsing, image processing, cryptography — that would otherwise freeze the UI.

A worker is normally its own file, but you can create one from an inline Blob URL, which runs correctly inside the sandbox here. The first example offloads a compute-heavy sum; the second is a ping/pong round-trip that echoes a transformed message.

Service Workers

A service worker is a background script that sits between the page and the network, acting as a programmable proxy. It powers offline support and fine-grained caching for Progressive Web Apps. Its lifecycle has three key events: install (pre-cache the app shell), activate (clean up old caches), and fetch (intercept requests and decide cache-vs-network).

Unlike everything else on this page, a service worker cannot run inside a sandboxed file preview — registration requires a secure context (HTTPS or localhost) and a real origin. The code below is therefore presented as reference, with a simulated result describing its behavior.

Example: registration (in the page)

// In your page script — register once the page has loaded.
if ("serviceWorker" in navigator) {
  window.addEventListener("load", async () => {
    try {
      const reg = await navigator.serviceWorker.register("/sw.js");
      console.log("SW registered, scope:", reg.scope);
    } catch (err) {
      console.error("SW registration failed:", err);
    }
  });
}
Result: on a secure origin the browser installs /sw.js and logs its scope. On file:// or inside this sandbox, navigator.serviceWorker is unavailable, so the guard skips registration — this is reference code, not runnable here.

Example: the service worker (sw.js) — cache-first shell

// sw.js — a cache-first app-shell strategy with old-cache pruning.
const CACHE = "app-shell-v2";
const SHELL = ["/", "/index.html", "/styles/site.css", "/scripts/site.js"];

// install: pre-cache the shell, then activate immediately
self.addEventListener("install", (event) => {
  event.waitUntil(
    caches.open(CACHE).then((cache) => cache.addAll(SHELL))
  );
  self.skipWaiting();
});

// activate: delete caches from previous versions
self.addEventListener("activate", (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(keys.filter((k) => k !== CACHE).map((k) => caches.delete(k)))
    )
  );
  self.clients.claim();
});

// fetch: serve from cache, fall back to network (and cache the result)
self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => {
      if (cached) return cached;                 // cache hit
      return fetch(event.request).then((res) => {
        const copy = res.clone();
        caches.open(CACHE).then((c) => c.put(event.request, copy));
        return res;                              // network fallback
      }).catch(() => caches.match("/index.html")); // offline fallback
    })
  );
});
Result: after activation, repeat visits load the shell instantly from the cache and work offline. Bumping CACHE to a new version (e.g. app-shell-v3) triggers a fresh install and prunes the old cache on activate.

Note: Service workers require HTTPS (or localhost), run in their own worker scope with no DOM access, and update in the background — a new worker takes control on the next navigation unless you call skipWaiting() plus clients.claim(). Always version your cache names.

  • IndexedDB — a transactional client-side database for structured, larger-than-localStorage data.

Feature detection

An Employee Dashboard runs across a spread of browsers, so it must ask the runtime what it can do rather than sniff a user-agent string. Feature detection tests for the capability itself: canvas.getContext("2d") returns a context or null, typeof localStorage and a write/read probe confirm working storage, "geolocation" in navigator checks a property, !!window.WebSocket checks a constructor, and CSS.supports("transition", "all") checks a CSS feature. This mirrors the HTML page's feature-detection and progressive-enhancement guidance from the markup side.

Collect these probes once into a reusable supports capability object, then branch on its booleans. Libraries like Modernizr automate exactly this, adding class hooks to the <html> element (e.g. canvas / no-canvas) so CSS can react too — but the underlying tests are the native ones below. We hand-roll the detector so it runs offline; the Modernizr equivalent follows as reference.

Note: Test for the capability, never the browser name. User-agent strings are spoofed, shared across engines, and change constantly, whereas a direct probe ("geolocation" in navigator) is always accurate for the runtime executing it. Detect once at startup and cache the result.

Web Storage API

Web Storage gives each origin a synchronous, string-keyed key/value store. localStorage persists until explicitly cleared; sessionStorage lives only for the tab's session. Both share the API — setItem, getItem, removeItem, clear, key(i), and length — and store only strings, so structured data (an employee record) round-trips through JSON.stringify/JSON.parse. A storage event fires in other tabs of the same origin whenever a value changes, which is how a dashboard keeps multiple open tabs in sync.

The preview iframe here has allow-scripts but not allow-same-origin, so touching localStorage throws a SecurityError. That is exactly the condition to design for: feature-detect, wrap access in try/catch, and fall back to an in-memory shim so the app degrades instead of crashing. On a real same-origin HTTPS page the same code writes to disk-backed storage directly.

// The storage event fires in OTHER tabs of the same origin, never the
// tab that made the change. Use it to keep dashboard tabs in sync.
window.addEventListener("storage", (e) => {
  if (e.key === "dashboard:employee:42") {
    console.log("another tab changed the record");
    console.log("old:", e.oldValue, "new:", e.newValue);
    rerenderEmployeeCard(JSON.parse(e.newValue));
  }
});

// Quota / availability is not guaranteed — always guard writes:
try {
  localStorage.setItem("big", hugeString);
} catch (err) {
  if (err.name === "QuotaExceededError") { /* evict or warn */ }
}
Result: open the same origin in two tabs; writing dashboard:employee:42 in one tab fires a storage event in the other with oldValue/newValue, so both cards re-render. The event does not fire in the tab that wrote it — that tab already knows.

Note: Web Storage is synchronous — every read and write blocks the main thread. Keep values small, batch related fields into a single JSON blob rather than many keys, and move large or structured datasets to IndexedDB, which is asynchronous and far higher-capacity.

Note: Storage is partitioned per origin and typically capped around 5 MB. In private-browsing or sandboxed contexts access can throw or silently reset, so the feature-detect-plus-fallback pattern above is mandatory, not optional.

Canvas 2D API

The <canvas> element — introduced on the HTML side under the canvas element — is a bitmap surface you draw on imperatively through a context from canvas.getContext("2d"). The context exposes state (fillStyle, strokeStyle, font, textAlign, textBaseline) and drawing calls (fillRect, strokeRect, beginPath+arc, fill, stroke, fillText). Paths are built then painted, and clip() constrains subsequent drawing to the current path.

A dashboard's most common canvas job is the generated avatar: when an employee has no photo, draw a colored circle with their initials, deriving a stable background color by hashing the name. Both examples below are fully runnable — an HTML editor supplies the <canvas> and a JS editor draws into it.

Geolocation API

navigator.geolocation.getCurrentPosition(success, error, options) asks the device for a one-shot fix. The success callback receives a GeolocationPosition with coords (latitude, longitude, accuracy, and sometimes altitude/heading/speed) and a timestamp; the error callback receives a GeolocationPositionError whose code is PERMISSION_DENIED (1), POSITION_UNAVAILABLE (2), or TIMEOUT (3). The options object accepts enableHighAccuracy, timeout, and maximumAge. For live tracking, watchPosition returns an id you later pass to clearWatch.

Geolocation requires a secure context and explicit user permission, and the sandboxed preview cannot grant it — the call errors out. So each example feature-detects, supplies an error handler, and falls back to a mock position, guaranteeing visible output. On a real HTTPS page the same code prompts the user and returns their true coordinates.

Note: Geolocation only works in a secure context (HTTPS or localhost) and always requires explicit user permission — the browser prompts on first use and the request can be denied at any time. WebSocket API

A WebSocket opens a persistent, full-duplex channel over a single TCP connection — ideal for a dashboard's real-time feed of events like "New employee added…". You construct one with new WebSocket("wss://host/path") and react to four events: open, message (its data is usually a JSON string you JSON.parse), close, and error. You push data with send(), inspect the connection through readyState (CONNECTING 0, OPEN 1, CLOSING 2, CLOSED 3), and shut down with close().

A sandbox cannot open a real socket, so these examples use a MockSocket that mirrors the interface — addEventListener, send, close, readyState — and emits fake messages via setTimeout. The real new WebSocket("wss://…") usage is shown as reference; swapping the mock for it changes one line.

Example: the real WebSocket (reference)

const ws = new WebSocket("wss://api.example.com/live");

ws.addEventListener("open", () => ws.send(JSON.stringify({ subscribe: "employees" })));
ws.addEventListener("message", (ev) => {
  const msg = JSON.parse(ev.data);
  if (msg.type === "employee_added") renderNewEmployee(msg);
});
ws.addEventListener("close", (ev) => console.log("closed", ev.code, ev.reason));
ws.addEventListener("error", () => console.log("socket error"));

// Heartbeat: ping periodically so proxies don't drop an idle connection.
setInterval(() => { if (ws.readyState === WebSocket.OPEN) ws.send("ping"); }, 30000);
Result: against a live server this streams events as they happen with near-zero latency and no polling. Swapping MockSocket/FlakySocket for new WebSocket(url) is the only change — the event wiring is identical.

Note: Always use the secure wss:// scheme in production (plain ws:// is blocked from HTTPS pages). Servers and proxies drop idle sockets, so send periodic heartbeat pings, and pair every socket with reconnect-and-backoff logic — networks and load balancers close connections routinely.

this & function binding

The value of this is determined by how a function is called, not where it is defined. Four rules resolve it, in ascending precedence: default (a standalone call — undefined in strict mode, the global object in sloppy mode), implicit (obj.method() binds this to obj), explicit (fn.call/apply/bind set it directly), and new (constructing binds this to the fresh instance).

Arrow functions ignore all four rules: they have no this of their own and close over the this of the enclosing lexical scope, so they cannot be rebound by call/apply/bind. The most common bug is the lost this: extracting a method (const f = obj.method) detaches it from its receiver, so calling f() loses obj. Fix it with .bind(obj), an arrow wrapper, or a class-field arrow.

Prototypes & classes

Every object has an internal link to a prototype object; property lookups that miss on the object itself walk up this prototype chain until they hit a match or reach null. Object.getPrototypeOf(obj) reads the link and Object.create(proto) makes a new object with a chosen prototype. This delegation is the whole object model — class syntax is sugar over it.

A class body holds a constructor, prototype methods (shared by all instances), get/set accessors, static members and static blocks, and #private fields/methods that enforce hard privacy the runtime guarantees (unlike closure privacy). extends sets up the prototype chain between subclass and superclass, and super calls the parent constructor or method. instanceof asks whether a constructor's prototype appears anywhere on an object's chain.

Error handling

throw raises any value, but you should throw Error objects so callers get a .name, .message, and a .stack. try runs guarded code, catch handles a throw, and finally always runs — even when try or catch executes a return. The built-in hierarchy (TypeError, RangeError, SyntaxError, …) all extend Error, so instanceof Error catches them all while instanceof TypeError narrows.

Define domain errors as subclasses (class ValidationError extends Error), set this.name, and chain root causes with the { cause } option. Async code propagates failures as rejected promises: try/catch around await and a trailing .catch() are equivalent. Unhandled rejections surface on the global unhandledrejection event; Promise.any aggregates all rejections into an AggregateError.

Iterators & generators

An object is iterable if it has a [Symbol.iterator]() method returning an iterator — an object with a next() that yields { value, done }. Implementing this protocol makes your type work with for…of, spread [...x], array destructuring, Array.from, and the Map/Set constructors. These built-ins never special-case arrays; they simply consume the iterator protocol.

Generator functions (function*) build iterators for you: each yield produces a value and suspends until the next next(), enabling lazy and even infinite sequences. yield* delegates to another iterable and can capture its return value. Async generators pair yield with await and are consumed by for await…of, the natural fit for paginated or streamed data.

Note: Generators are lazy — an infinite sequence costs nothing until you pull values, and stopping a for…of early suspends the generator instead of running it to completion. This makes them ideal for streaming pipelines where materializing the whole dataset would be wasteful or impossible.

Regular expressions

A regex is written as a literal (/pattern/flags) or built dynamically with new RegExp(string, flags) — use the constructor when the pattern comes from data. Flags tune behavior: g (all matches), i (case-insensitive), m (^/$ match per line), s (. matches newlines), u (Unicode), and y (sticky, anchored at lastIndex).

Patterns combine character classes (\d, [a-z]), quantifiers (*, +, {2,4}), anchors (^, $, \b), and alternation (a|b). Groups capture ((…)), name captures ((?<year>…)), and backreference (\1). Lookaround asserts without consuming: lookahead (?=…)/(?!…) and lookbehind (?<=…)/(?<!…). Extract with test, exec, match, and matchAll; transform with replace/replaceAll.

Beware catastrophic backtracking: nested quantifiers like (a+)+ on non-matching input can hang; keep patterns unambiguous.

The object model & immutability

Every property has a descriptor. Data properties carry value, writable, enumerable, and configurable; accessor properties replace value/writable with get/set. Object.defineProperty creates them with full control (defaults are all false), and Object.getOwnPropertyDescriptor(s) reads them back. Enumerability governs whether a key shows up in for…in, spread, Object.keys, and JSON.stringify.

Three levels of lockdown exist: Object.preventExtensions (no new props), Object.seal (also no delete/reconfigure), and Object.freeze (also no writes). All three are shallow — nested objects remain mutable, so deep immutability needs a recursive helper. For value-level transformations, pair Object.entries/fromEntries with array methods to rebuild objects immutably, and use Object.getOwnPropertyDescriptors for a faithful clone that preserves accessors.
