Java Functional Programming — Lambdas & Streams

A hands-on study guide to functional programming in Java. It starts from the shift in mindset — describing what you want computed instead of scripting how to loop — and builds the whole toolkit on top of it: the filter/map/reduce stream pipeline; lambda expressions and method references; the standard functional interfaces (Predicate, Consumer, Function, Supplier, BinaryOperator and their Bi/primitive cousins); behavior parameterization and higher-order functions; sorting with Comparator; Collectors and groupingBy; Optional; building streams from ranges, arrays and BigInteger; lazy evaluation and parallel streams; and functional handling of files and threads. Each lecture pairs a focused Java example with the console output it produces, traced by hand (Java runs on the JVM, not in the browser).

Prerequisites & how to read this You should be comfortable with core Java syntax — classes, methods, generics, collections. Functional programming arrived in Java 8 and was refined in Java 9+; examples target JDK 11+ (a few notes flag features that need a later release). Run any snippet with javac File.java && java File, or paste one-liners straight into jshell. Use the filter box in the sidebar to jump to any lecture. Prose is paraphrased from the course for study; code examples are reimplemented and improved.

1. Getting Started with Functional Programming

This opening section reframes a task you already know how to do — walking a list and printing values — by shifting from the imperative how to the functional what. Along the way you meet the four building blocks used throughout the course: the Stream pipeline, filter, map, and the forEach terminal operation, expressed with lambdas and method references.

Run locally Save a snippet as e.g. FP01Functional.java and run javac FP01Functional.java && java FP01Functional. For quick one-liners, start jshell and paste expressions directly. Requires JDK 11 or newer.

01. Why functional programming? Imperative vs declarative

Objective: Understand how the functional (declarative) style differs from the traditional structured (imperative) style, and see the same task written both ways.

In the imperative or structured style you spell out how the work is done: set up a loop, advance an index or iterator, keep a mutable accumulator, and test each element yourself. The reader has to execute the steps mentally to discover the intent.

In the functional or declarative style you describe what you want: take the elements as a Stream, keep the ones you care about, transform them, and consume the result. The how — iteration order, temporary variables, plumbing — is handled by the library.

  • Imperative = how (explicit loops, mutable state); declarative = what (a pipeline of operations).
  • A Stream is a sequence of elements you process through a chain of operations.
  • The same result comes out; the functional version reads as intent, not mechanics.
  • Java gained streams, lambdas, and method references in Java 8.
// Imperative — we describe HOW: loop, then print each element
static void printAllStructured(List<Integer> numbers) {
    for (int number : numbers) {
        System.out.println(number);
    }
}

// Declarative — we describe WHAT: stream the elements, print each
static void printAllFunctional(List<Integer> numbers) {
    numbers.stream()
           .forEach(System.out::println);
}

// Both called with:
// List<Integer> numbers = List.of(12, 9, 13, 4, 6, 2, 4, 12, 15);
Console output12 9 13 4 6 2 4 12 15

Source: reimplemented from the course examples FP01Structured and FP01Functional.

AdvancedThe two styles are not just cosmetic. Because a declarative pipeline says what rather than how, the library is free to change the how — for example running the pipeline in parallel with .parallel() — without you rewriting the loop. Intermediate operations are also lazy: they describe work that only happens when a terminal operation runs.
Pitfall A stream does nothing on its own. numbers.stream().filter(...) with no terminal operation (like forEach) produces no output at all — the pipeline is only a description until something consumes it.

02. Your first functional program — stream, filter, forEach

Objective: Write your first functional program: turn a list into a Stream and print every element with a terminal forEach.

Every stream pipeline starts from a source. Calling .stream() on a collection gives you a Stream<Integer> — the entry point onto which you chain operations. forEach is a terminal operation: it consumes each element and performs an action, here printing.

The action passed to forEach is System.out::println, a method reference. It is shorthand for the lambda x -> System.out.println(x): a reference to the method, not a call to it.

  • collection.stream() is the pipeline's entry point, producing a Stream<T>.
  • forEach is a terminal operation — it triggers the pipeline and returns void.
  • System.out::println is a method reference passed as behavior, equivalent to x -> System.out.println(x).
List<Integer> numbers = List.of(12, 9, 13, 4, 6, 2, 4, 12, 15);

numbers.stream()                     // Stream<Integer> — the entry point
       .forEach(System.out::println);  // terminal op: consume each element
Console output12 9 13 4 6 2 4 12 15

Source: reimplemented from the course example FP01Functional (printAllNumbersInListFunctional).

AdvancedMethod references come in several shapes: System.out::println is a reference on a specific object; Integer::sum and ClassName::staticMethod reference static methods; String::toUpperCase references an instance method to be applied to each stream element. The course later replaces a custom FP01Functional::print reference with the built-in System.out::println for the same effect with less code.
Pitfall Write System.out::println, not System.out.println(). The first passes the method as a value for forEach to call per element; the second calls println once, right now, and hands its void result to forEach — which does not compile.

03. Refining with filter — structured vs functional

Objective: Print only the even numbers, comparing a structured loop-with-if against a stream with filter.

To keep only some elements, the structured version nests an if inside the loop. The functional version inserts a filter stage into the pipeline. filter takes a condition that returns a boolean; elements for which it is true pass through to the next stage, and the rest are dropped.

filter is an intermediate operation: it returns a new Stream, so you can keep chaining. Nothing is actually filtered until a terminal operation (here forEach) runs the pipeline.

  • filter keeps elements whose condition evaluates to true.
  • The condition is a boolean-returning function — a Predicate.
  • filter is intermediate (returns a Stream); forEach is terminal.
  • number % 2 == 0 is the classic even test.
// Structured: an if inside the loop decides what to keep
static void printEvenStructured(List<Integer> numbers) {
    for (int number : numbers) {
        if (number % 2 == 0) {
            System.out.println(number);
        }
    }
}

// Functional: filter keeps only elements whose test is true
static void printEvenFunctional(List<Integer> numbers) {
    numbers.stream()
           .filter(number -> number % 2 == 0)
           .forEach(System.out::println);
}
Console output12 4 6 2 4 12

Source: reimplemented from the course examples FP01Structured and FP01Functional.

AdvancedThe course first passed a method reference, filter(FP01Functional::isEven), where isEven is a separate boolean method. That works identically to the inline condition — both supply a Predicate<Integer>. Choosing between them is about reuse: extract a named method when the same test is used in several places, inline it when it is used once.
Pitfall The filter body must be a boolean test, not an assignment. number = 2 assigns and is a type error here; the even test is number % 2 == 0 with ==.

04. Lambda expressions

Objective: Learn lambda syntax — the expression form and the block form — and see that a lambda is an inline implementation of a functional interface.

A lambda is an anonymous function you write directly where a behavior is expected. Its simplest form is parameter -> expression: the value of the expression is returned automatically. When you need several statements, use the block form parameter -> { ... return value; }, where return is explicit.

Under the hood a lambda is an inline implementation of a functional interface — an interface with a single abstract method. Because filter expects a Predicate<Integer> (one Integer in, one boolean out), the compiler treats number -> number % 2 == 0 as that predicate.

  • Expression form: number -> number % 2 == 0 (implicit return).
  • Block form: number -> { return number % 2 == 0; } (explicit return).
  • A lambda targets a functional interface — here Predicate<Integer>.
  • A lambda inlines logic, removing the need for a separate named method.
List<Integer> numbers = List.of(12, 9, 13, 4, 6, 2, 4, 12, 15);

// filter expects a Predicate<Integer>; the lambda IS that predicate.
numbers.stream()
       .filter(number -> number % 2 == 0)  // parameter -> boolean expression
       .forEach(System.out::println);
Console output12 4 6 2 4 12

Source: reimplemented from the course example FP01Functional (lambda variant of the even-number filter).

AdvancedThe expression and block forms are interchangeable when they compute the same value; the filter above could equally be written number -> { return number % 2 == 0; }. The block form is required once you need multiple statements. A lambda can also be assigned to a variable of its interface type, e.g. Predicate<Integer> isEven = x -> x % 2 == 0;, and reused across pipelines.
Pitfall The braces of the block form change the rules: x -> x % 2 == 0 returns the value, but x -> { x % 2 == 0; } does not compile — a block body needs an explicit return.

06. map — transforming elements

Objective: Use map to transform each element of a stream, on its own and combined with filter.

map applies a function to every element and replaces it with the result. Where filter decides which elements continue, map decides what each becomes. It can also change the element type — mapping a Stream<String> of course names to their lengths yields a Stream<Integer>.

map is an intermediate operation, so it composes cleanly after filter: keep the evens, then square each; keep the odds, then cube each.

  • map(x -> f(x)) transforms each element into f(x).
  • filter selects elements; map transforms them.
  • map may change the element type, e.g. Stream<String> to Stream<Integer>.
  • Order matters: filter then map transforms only the survivors.

Squares of the even numbers. Six evens (12, 4, 6, 2, 4, 12) survive the filter, and each is squared.

numbers.stream()
       .filter(number -> number % 2 == 0)  // keep evens
       .map(number -> number * number)      // transform: square
       .forEach(System.out::println);
Console output144 16 36 4 16 144

Cubes of the odd numbers.

numbers.stream()
       .filter(number -> number % 2 != 0)          // keep odds
       .map(number -> number * number * number)     // transform: cube
       .forEach(System.out::println);
Console output729 2197 3375

Course name plus its length. Here map builds a new String from each course.

courses.stream()
       .map(course -> course + " " + course.length())
       .forEach(System.out::println);
Console outputSpring 6 Spring Boot 11 API 3 Microservices 13 AWS 3 PCF 3 Azure 5 Docker 6 Kubernetes 10

Source: reimplemented from the course examples FP01Functional and FP01Exercises.

AdvancedThe type change is real: courses.stream().map(course -> course.length()) produces a Stream<Integer> whose elements are 6, 11, 3, 13, 3, 3, 5, 6, 10. When you map to a primitive and want numeric operations, the specialized mapToInt gives an IntStream with helpers such as sum() and average().
Pitfall map does not modify the source list — streams never mutate their source. It produces a new stream of transformed values; the original numbers and courses lists are unchanged (and, being List.of(...), are immutable anyway).

07. Review — the functional building blocks

Objective: Recap the core pieces — Stream, filter, forEach, map, lambdas, and method references — and see them work together in one pipeline.

Stream. The functional entry point. list.stream() turns a collection into a Stream<T>, a sequence of elements you process by declaring what to do rather than looping.

filter. An intermediate operation that keeps only elements whose predicate is true and drops the rest, returning a smaller Stream.

map. An intermediate operation that transforms each element with a function, possibly changing its type, returning a Stream of the results.

forEach. A terminal operation that consumes each element to perform a side effect such as printing; it returns void and triggers the whole pipeline.

Lambda. Inline behavior, parameter -> expression (or a { ... return ...; } block), supplied wherever a functional interface is expected. Method reference. An even shorter form when a method already does the job, e.g. System.out::println or Integer::sum.

  • Pipeline = source (stream) then intermediate ops (filter, map) then a terminal op (forEach).
  • Intermediate operations are lazy; the terminal operation drives execution.
  • Lambdas and method references both provide the behavior each stage needs.
List<Integer> numbers = List.of(12, 9, 13, 4, 6, 2, 4, 12, 15);

numbers.stream()                        // 1. source: List -> Stream
       .filter(number -> number % 2 == 0)  // 2. keep evens   (intermediate)
       .map(number -> number * number)     // 3. square them  (intermediate)
       .forEach(System.out::println);      // 4. print each   (terminal)
Console output144 16 36 4 16 144

Source: reimplemented as an original illustration combining the section's operations.

AdvancedA stream is single-use: once a terminal operation has run, the same stream instance cannot be reused — you build a fresh one from the source. This, plus the laziness of intermediate operations, is what lets the library fuse the stages and, when asked, run them in parallel.
Pitfall Reusing a consumed stream throws IllegalStateException: stream has already been operated upon or closed. Start each pipeline from list.stream() again rather than storing and re-running a stream variable.
Cheat-sheet
list.stream()                     // List<T> -> Stream<T>   (entry point / source)

.filter(x -> booleanTest(x))      // Stream<T> -> Stream<T> : keep matches   (intermediate)
.map(x -> transform(x))           // Stream<T> -> Stream<R> : transform each (intermediate)

.forEach(System.out::println)     // terminal: consume each element, returns void

// Lambda (expression):  parameter -> expression       e.g. n -> n % 2 == 0
// Lambda (block):       parameter -> { return ...; }
// Method reference:     System.out::println   Integer::sum   String::toUpperCase

// filter selects WHICH elements pass; map decides WHAT each becomes.
// Intermediate ops (filter, map) are LAZY — nothing runs until a terminal op.
// Streams never mutate their source; List.of(...) lists are immutable.

2. Playing with Streams

A stream is a pipeline: you start from a source, chain intermediate operations that describe transformations, and finish with a single terminal operation that actually runs the work. This section folds streams down to one value with reduce, reshapes them with distinct, sorted and comparators, gathers them back into a List with collect, and pins down the intermediate-vs-terminal distinction that governs it all.

01. reduce — aggregating a stream to one value

Objective: Replace a manual accumulation loop with reduce, which folds every element of a stream into a single result.

Summing a list the structured way means declaring a mutable sum, looping, and mutating it on each pass. The functional equivalent is reduce(identity, accumulator): it seeds the fold with an identity value and repeatedly applies a BinaryOperator that combines the running total with the next element.

For addition the identity is 0 (adding it changes nothing) and the accumulator is (x, y) -> x + y. The two versions below produce the same answer, but the stream version states what you want instead of how to iterate.

  • reduce is a terminal operation — it consumes the stream and returns a plain value, not a stream.
  • First argument = identity (starting value); second argument = a BinaryOperator<T> that merges two values into one.
  • The accumulator is called once per element; the result of each call becomes the running total for the next.
  • Swap the accumulator to fold differently: product, max, string concatenation — same shape, different operator.
List<Integer> numbers = List.of(12, 9, 13, 4, 6, 2, 4, 12, 15);

// Structured: declare a mutable accumulator and loop
int sum = 0;
for (int n : numbers) {
    sum += n;
}
System.out.println(sum);           // 77

// Functional: fold the stream with reduce
int total = numbers.stream()
                   .reduce(0, (x, y) -> x + y);
System.out.println(total);         // 77
Console output77 77

Source: reimplemented from the course examples FP02Structured and FP02Functional.

AdvancedThere are three reduce overloads. The two-argument reduce(BinaryOperator<T>) has no identity, so it returns an Optional<T> (empty on an empty stream). The three-argument reduce(identity, accumulator, combiner) adds a combiner that merges partial results — needed when a parallel stream splits the work across threads and the accumulator's input and output types differ.
Pitfall The identity must truly be neutral for the operator. 0 is correct for a sum but wrong for a product (everything becomes 0) — use 1 there. A bad identity silently corrupts the result rather than throwing.

02. How reduce works — lambdas & method references

Objective: See the step-by-step accumulation inside reduce, then tighten the accumulator from a lambda to a method reference.

To make the fold visible, temporarily replace the accumulator with a block lambda that prints its two inputs — a is the running aggregate, b is the next element — before returning their sum. Reading the trace, each line's returned total is the a on the following line, ending at 77.

Once the logic is understood, drop the noise. (x, y) -> x + y already does nothing but call an existing method, so it can be written as the method reference Integer::sum. A method reference reads cleaner precisely when the lambda's whole body is a single call whose parameters line up with the target method's parameters.

  • The accumulator receives (aggregate, nextElement) in that order and returns the new aggregate.
  • The identity 0 is the aggregate for the very first call, paired with the first element.
  • (x, y) -> x + y and Integer::sum are interchangeable here — both are BinaryOperator<Integer>.
  • Prefer the method reference when it names the operation directly; prefer the lambda when you need extra logic.
int sum = numbers.stream()
                 .reduce(0, (a, b) -> {
                     System.out.println("a: " + a + ", b: " + b);
                     return a + b;
                 });
System.out.println("Sum: " + sum);

// Same behaviour, no noise — method reference:
// numbers.stream().reduce(0, Integer::sum);
Console outputa: 0, b: 12 a: 12, b: 9 a: 21, b: 13 a: 34, b: 4 a: 38, b: 6 a: 44, b: 2 a: 46, b: 4 a: 50, b: 12 a: 62, b: 15 Sum: 77

Source: reimplemented from the course example FP02DebuggingReduce.

AdvancedFor a sequential stream the fold is strictly left-to-right, so a non-associative accumulator still gives predictable output. For a parallel stream the elements are combined in an unspecified grouping, so the accumulator must be associative and stateless — (x, y) -> x + y is fine, but subtraction or "append to a shared list" would give wrong or nondeterministic results.
Pitfall Don't reach for a method reference when the argument order doesn't match. Integer::sum works because sum(a, b) maps cleanly onto (aggregate, next); an operation like subtraction has no matching two-arg method, so keep the explicit lambda.

03. Stream puzzles in JShell

Objective: Use JShell to probe how the accumulator's return value shapes reduce's output, and derive max/min from it.

JShell (a REPL shipped since Java 9) evaluates one expression at a time and prints each result as $N ==> value, which makes it ideal for poking at streams. Four small puzzles reveal the mechanics: returning only the aggregate x ignores every element and leaves the identity untouched (0); returning only y keeps overwriting until the last element wins (15).

From there, a ternary turns reduce into max or min. x > y ? x : y keeps the larger of the pair each step (max = 15); x > y ? y : x keeps the smaller (min = 2). Seeding max with Integer.MIN_VALUE and min with Integer.MAX_VALUE guarantees the first real element wins the first comparison.

  • The accumulator's return value is the next aggregate — return x and nothing accumulates; return y and only the last element survives.
  • Max: reduce(Integer.MIN_VALUE, (x, y) -> x > y ? x : y).
  • Min: reduce(Integer.MAX_VALUE, (x, y) -> x > y ? y : x).
  • The seed doubles as a safe extreme: nothing is larger than MAX_VALUE, nothing smaller than MIN_VALUE.
jshell> List<Integer> numbers = List.of(12, 9, 13, 4, 6, 2, 4, 12, 15)

jshell> numbers.stream().reduce(0, (x, y) -> x)

jshell> numbers.stream().reduce(0, (x, y) -> y)

jshell> numbers.stream().reduce(Integer.MIN_VALUE, (x, y) -> x > y ? x : y)

jshell> numbers.stream().reduce(Integer.MAX_VALUE, (x, y) -> x > y ? y : x)
JShell outputnumbers ==> [12, 9, 13, 4, 6, 2, 4, 12, 15] $2 ==> 0 $3 ==> 15 $4 ==> 15 $5 ==> 2

Source: reimplemented from the course's JShell reduce session.

AdvancedThese are teaching reductions. In real code, reach for the purpose-built terminals: numbers.stream().max(Comparator.naturalOrder()) and .min(...) return an Optional<Integer> (no magic seed needed), and on a primitive IntStream you get .max(), .min(), .sum() and .average() directly.
Pitfall Seeding max with 0 instead of Integer.MIN_VALUE looks fine here but breaks on an all-negative list: 0 is larger than every element, so it leaks through as the "maximum". Match the seed to the operation's true extreme.

04. Exercises — sum of squares, cubes & odds

Objective: Combine map, filter and reduce to compute sum of squares, sum of cubes, and the sum of odd numbers.

The clean idiom is transform-then-fold: map reshapes every element, then reduce collapses the result. Sum of squares is map(x -> x * x).reduce(0, Integer::sum) = 835; sum of cubes swaps in x * x * x = 10109; sum of odds filters first with filter(x -> x % 2 == 1) then folds = 37 (only 9, 13 and 15 are odd).

It is tempting to do it all in one accumulator: reduce(0, (x, y) -> x * x + y * y). Don't. That squares the running aggregate on every step (0→144→144²+81=20817→…), which is neither a sum of squares nor even stable — the values explode past Integer.MAX_VALUE and overflow. Keep the per-element transform in map and let reduce only add.

  • Sum of squares: .map(x -> x * x).reduce(0, Integer::sum) → 835.
  • Sum of cubes: .map(x -> x * x * x).reduce(0, Integer::sum) → 10109.
  • Sum of odds: .filter(x -> x % 2 == 1).reduce(0, Integer::sum) → 37.
  • One responsibility per stage: map/filter shape elements, reduce only combines.
// WRONG: squares the running aggregate too, then overflows int
// 0,12 -> 144 ; 144,9 -> 20817 ; 20817,13 -> ... blows past MAX_VALUE
int bad = numbers.stream().reduce(0, (x, y) -> x * x + y * y);

// RIGHT: map first, then reduce
int sumOfSquares = numbers.stream()
                          .map(x -> x * x)
                          .reduce(0, Integer::sum);
int sumOfCubes = numbers.stream()
                        .map(x -> x * x * x)
                        .reduce(0, Integer::sum);
int sumOfOdds = numbers.stream()
                       .filter(x -> x % 2 == 1)
                       .reduce(0, Integer::sum);

System.out.println(sumOfSquares);
System.out.println(sumOfCubes);
System.out.println(sumOfOdds);
Console output835 10109 37

Source: reimplemented from the course exercises (map/filter/reduce) and the FP02StreamOperations example.

AdvancedBoxing Integer values through map/reduce costs allocations and risks silent int overflow. For numeric work, drop to a primitive stream: numbers.stream().mapToInt(x -> x * x).sum(). If a genuine total might exceed Integer.MAX_VALUE, use .mapToLong(x -> (long) x * x).sum().
Pitfall x % 2 == 1 misses negative odds — in Java -3 % 2 is -1, not 1. For lists that may hold negatives, test x % 2 != 0 instead.

05. distinct & sorted

Objective: Remove duplicates with distinct() and order elements with sorted(), understanding that both are intermediate and non-mutating.

distinct() keeps only the first occurrence of each element (comparing via equals/hashCode) in encounter order. Our list has 4 and 12 twice, so their second appearances are dropped, leaving 12, 9, 13, 4, 6, 2, 15. sorted() arranges elements in natural (ascending) order and keeps duplicates.

Both are intermediate: each returns a brand-new Stream and leaves the source list untouched — nothing runs until a terminal op like forEach. Chaining distinct().sorted() gives the unique values in ascending order: 2, 4, 6, 9, 12, 13, 15.

  • distinct() dedupes by equals/hashCode, preserving first-seen order.
  • sorted() requires elements to be Comparable; it sorts ascending and is stable.
  • Both return a new stream and never mutate the underlying collection.
  • Order of chaining matters for cost, not result here: distinct().sorted() sorts fewer elements.

System.out.println("---");

06. Sorting streams with Comparators

Objective: Drive sorted with a Comparator to sort in natural order, reverse order, or by a custom key such as string length.

The no-arg sorted() uses each element's natural ordering. To customise it, pass a Comparator: Comparator.naturalOrder() and Comparator.reverseOrder() are the built-ins, and reverseOrder() (or naturalOrder().reversed()) flips the direction. For anything else, Comparator.comparing(keyExtractor) builds a comparator from a key.

Sorting the courses list by title length with Comparator.comparing(str -> str.length()) puts the three-letter titles first and "Microservices" last. Because stream sorted is stable, titles of equal length keep their original relative order — "Spring" stays ahead of "Docker", both length 6.

  • sorted(Comparator.reverseOrder()) sorts descending; naturalOrder().reversed() is equivalent.
  • Comparator.comparing(str -> str.length()) sorts by any derived key; Comparator.comparing(String::length) is the method-reference form.
  • Ties preserve encounter order because sorted is a stable sort.
  • Chain multiple keys with thenComparing(...) to break ties deterministically.

07. collect — gathering elements into a List

Objective: Turn a transformed stream back into a List with collect(Collectors.toList()), and know the modern Stream.toList() shorthand.

Where reduce folds to a single value, collect accumulates elements into a container. The everyday case is collect(Collectors.toList()), a terminal op that materialises the pipeline into a new List. Mapping each course title to its length and collecting produces [6, 11, 3, 13, 3, 3, 5, 6, 10], in the source's original order.

  • collect(Collectors.toList()) is terminal — it runs the pipeline and returns a List.
  • Stream.toList() (Java 16+) is the concise, unmodifiable-result equivalent.
  • Collectors.toList() gives no mutability or type guarantee; don't depend on being able to add to it.
  • Other collectors follow the same shape: toSet(), toUnmodifiableList(), joining(", "), groupingBy(...).

08. Intermediate vs terminal operations

Objective: Classify every stream operation as intermediate or terminal, and understand why a stream can be traversed only once.

Intermediate operations — filter, map, sorted, distinct, limit, peek — return a new Stream and are lazy: they merely describe work. Nothing executes until a terminal operation — forEach, collect, reduce, count, min/max, findFirst — is attached. The terminal op returns a non-stream (a value, a collection, or nothing) and drives the whole pipeline to completion in a single pass.

Because that pass consumes the stream, a stream is single-use. In the example, declaring the filtered stream prints nothing (it is lazy); collect triggers execution and yields the even numbers; then trying to reuse the already-consumed stream throws IllegalStateException. Build a fresh stream from the source each time.

  • Intermediate = returns Stream, lazy: filter, map, sorted, distinct, limit, peek.
  • Terminal = returns non-stream, eager: forEach, collect, reduce, count, min/max, findFirst.
  • No terminal op ⇒ no work happens; the pipeline is just a description.
  • A stream is consumed by its terminal op and cannot be reused — re-derive it from the source.
Stream<Integer> s = numbers.stream()
                          .filter(x -> x % 2 == 0); // intermediate: lazy, nothing runs yet

List<Integer> evens = s.collect(Collectors.toList()); // terminal: pipeline runs
System.out.println(evens);

s.forEach(System.out::println); // reusing an already-consumed stream
Console output[12, 4, 6, 2, 4, 12] Exception in thread "main" java.lang.IllegalStateException: stream has already been operated upon or closed

Source: reimplemented as an original illustration of laziness and single-use streams (per FP02StreamOperations op annotations).

AdvancedLaziness enables short-circuiting: with findFirst or limit, elements are pulled through the pipeline only until the terminal op is satisfied, so later elements may never be processed. peek exists purely to observe elements as they flow (for debugging) — it is intermediate and does nothing without a terminal op to pull elements through it.
Pitfall Storing a stream in a variable and reusing it after a terminal op throws IllegalStateException: stream has already been operated upon or closed. Streams are not collections — treat them as one-shot; if you need the data twice, collect to a List first, then re-stream that.
Cheat-sheet

3. Functional Interfaces & Lambdas

A functional interface is an interface with exactly one abstract method, and a lambda is a concise implementation of that method. This is the machinery behind every stream operation: filter takes a Predicate, map takes a Function, forEach takes a Consumer, and reduce takes a BinaryOperator. This section tours the core interfaces in java.util.function, shows how behavior can be passed as a parameter, and ends with lambda-syntax puzzles and method references.

01. Predicate, Consumer & Function

Objective: Meet the three interfaces you will use most — Predicate, Function, Consumer — and see how each lambda is really just a one-method object.

When you write filter(x -> x % 2 == 0), the compiler needs a target type. That type is Predicate<Integer>: an interface whose single abstract method is test(T) returning boolean. The lambda supplies the body of test. Likewise map expects a Function<T, R> (apply(T) returns R) and forEach expects a Consumer<T> (accept(T) returns void).

Assigning each lambda to a named variable makes the pipeline read like plain English and shows exactly which interface each stage plugs into. The verbose anonymous-inner-class form below is what the lambda replaces — same object, far more boilerplate.

  • Predicate<T> — test(T): boolean, e.g. x -> x % 2 == 0.
  • Function<T, R> — apply(T): R, e.g. x -> x * x.
  • Consumer<T> — accept(T): void, e.g. System.out::println.
  • A lambda is just shorthand for a class implementing the interface's one abstract method.
import java.util.List;
import java.util.function.Predicate;
import java.util.function.Function;
import java.util.function.Consumer;

List<Integer> numbers = List.of(1, 2, 3, 4, 5, 6);

// Each lambda targets a functional interface
Predicate<Integer>          isEven = x -> x % 2 == 0;   // test(T): boolean
Function<Integer, Integer>  square = x -> x * x;        // apply(T): R
Consumer<Integer>           print  = System.out::println; // accept(T): void

// The SAME Predicate written the long way (what the lambda replaces)
Predicate<Integer> isEvenVerbose = new Predicate<Integer>() {
    @Override
    public boolean test(Integer x) {
        return x % 2 == 0;
    }
};

numbers.stream()
       .filter(isEven)   // filter  needs a Predicate
       .map(square)      // map     needs a Function
       .forEach(print);  // forEach needs a Consumer

AdvancedThe lambda is not compiled to an anonymous class file. Java uses the invokedynamic bytecode instruction plus LambdaMetafactory to build the implementing object lazily at first use, which avoids generating a separate .class per lambda and lets the JVM optimize it.
Pitfall An interface qualifies as functional only if it has exactly one abstract method. default and static methods don't count, so Predicate can still declare and, or, negate as defaults and remain a target for lambdas.

02. BinaryOperator & exercises

Objective: Learn BinaryOperator<T> — the two-in, one-out (all same type) interface that reduce uses — and build a few of your own.

A BinaryOperator<T> combines two values of type T into one value of type T: its abstract method is apply(T, T) returning T. It is simply the special case of BiFunction<T, U, R> where all three type parameters are the same T. That is exactly the shape reduce(identity, accumulator) wants.

The method reference Integer::sum is a ready-made BinaryOperator<Integer> equivalent to (a, b) -> a + b. Because the interface is generic over behavior, you can just as easily express max, min, or product.

  • BinaryOperator<T> — apply(T, T): T; it extends BiFunction<T, T, T>.
  • Integer::sum is the same as (a, b) -> a + b.
  • reduce(0, op) folds a stream to a single value using the operator.
  • Swap the body to get other folds: (a, b) -> a > b ? a : b for max, (a, b) -> a * b for product.
import java.util.List;
import java.util.function.BinaryOperator;

List<Integer> numbers = List.of(1, 2, 3, 4, 5, 6);

BinaryOperator<Integer> sum      = Integer::sum;            // (a, b) -> a + b
BinaryOperator<Integer> max      = (a, b) -> a > b ? a : b;
BinaryOperator<Integer> multiply = (a, b) -> a * b;

int total = numbers.stream().reduce(0, sum);   // 1+2+3+4+5+6

System.out.println("Sum: " + total);
System.out.println("Max of 3, 7: " + max.apply(3, 7));
System.out.println("3 * 4 = " + multiply.apply(3, 4));
Console outputSum: 21 Max of 3, 7: 7 3 * 4 = 12

Source: reimplemented from the course example (FP03FunctionalInterfaces, BinaryOperator exercise).

Advancedreduce has a three-argument overload, reduce(identity, accumulator, combiner), used by parallel streams: the combiner merges partial results computed on different threads. For a single-type reduction the accumulator is a BinaryOperator; the identity must be a true identity value (x op identity == x), e.g. 0 for sum, 1 for product.
Pitfall Using Integer.MIN_VALUE as the identity for a max-reduce works, but a plain 0 would be wrong when the stream can contain only negative numbers — the "identity" must not alter the result.

03. Behavior parameterization

Objective: Pass behavior (not just data) into a method, so one method serves many cases. This is the central idea of functional programming.

Printing even numbers and printing multiples of three are the same code differing only in one condition. Instead of copying the method, accept a Predicate parameter and let the caller supply the condition. The method filterAndPrint(numbers, predicate) now works for any rule.

The parameter type Predicate<? super Integer> uses a lower-bounded wildcard so the method also accepts, say, a Predicate<Number> — a predicate written for a supertype can safely test Integer values (PECS: a consumer of input is ? super).

  • Behavior parameterization = passing logic as a method argument.
  • Signature: filterAndPrint(List<Integer>, Predicate<? super Integer>).
  • One method, many behaviors — the caller decides the rule at the call site.
  • Functions become first-class citizens: stored, passed, and returned like any value.
import java.util.List;
import java.util.function.Predicate;

public class BehaviorParam {
    public static void main(String[] args) {
        List<Integer> numbers = List.of(1, 2, 3, 4, 5, 6);

        filterAndPrint(numbers, x -> x % 2 == 0);  // caller supplies "even"
        System.out.println("---");
        filterAndPrint(numbers, x -> x % 3 == 0);  // caller supplies "multiple of 3"
    }

    // The behavior (the Predicate) is decided by whoever calls this method
    static void filterAndPrint(List<Integer> numbers, Predicate<? super Integer> predicate) {
        numbers.stream()
               .filter(predicate)
               .forEach(System.out::println);
    }
}

AdvancedYou can build predicates dynamically and combine them with the default methods: evenPred.and(x -> x > 3) yields a new Predicate matching numbers that are both even and greater than 3. A factory method Predicate<Integer> greaterThan(int n) { return x -> x > n; } returns behavior — functions returning functions.
Pitfall Declaring the parameter as the raw Predicate (no type argument) compiles but loses type safety and triggers unchecked warnings. Keep the generic: Predicate<? super Integer>.

04. Behavior parameterization exercise

Objective: Apply the same idea to transformations: one mapAndCreateNewList method that squares, cubes, or doubles depending on the Function passed in.

Where the previous lecture parameterized a filter with a Predicate, here we parameterize a map with a Function<Integer, Integer>. The method streams the list, applies the caller's function, and collects the results into a new list. Squaring, cubing, and doubling become three one-line call sites over the same reusable method.

  • Signature: mapAndCreateNewList(List<Integer>, Function<Integer, Integer>).
  • Square = x -> x * x, cube = x -> x * x * x, double = x -> x + x.
  • collect(Collectors.toList()) gathers the mapped stream into a new list.
  • Same method, different transform — no duplicated stream plumbing.

05. Supplier & UnaryOperator

Objective: Add two more interfaces: Supplier<T> (no input, produces a value) and UnaryOperator<T> (a Function whose input and output types match).

A Supplier<T> takes nothing and returns a T via get() — perfect for lazy value creation, factory logic, or random generation. A UnaryOperator<T> is just Function<T, T> with a shorter name; it transforms a value into another value of the same type via apply(T), which is exactly what List.replaceAll expects.

  • Supplier<T> — get(): T, no parameters, e.g. () -> "Ranga".
  • UnaryOperator<T> — apply(T): T, e.g. x -> 3 * x; it extends Function<T, T>.
  • A Supplier is called with .get(); a UnaryOperator with .apply(...).
  • Suppliers defer work — the body runs only when get() is invoked.
import java.util.Random;
import java.util.function.Supplier;
import java.util.function.UnaryOperator;

Supplier<String>        greeter = () -> "Ranga";   // no input -> a value
UnaryOperator<Integer>  triple  = x -> 3 * x;      // same type in and out

System.out.println(greeter.get());
System.out.println(triple.apply(10));

// A Supplier that produces a fresh die roll each call (1..6)
Supplier<Integer> dice = () -> new Random().nextInt(6) + 1;
System.out.println("Rolled a valid face? " + (dice.get() >= 1));

06. BiPredicate, BiFunction, BiConsumer & primitive interfaces

Objective: Handle two arguments with the Bi* family, and learn why primitive specializations like IntBinaryOperator exist.

The Bi* interfaces mirror the single-argument ones but take two inputs: BiPredicate<T, U> (test(T, U): boolean), BiFunction<T, U, R> (apply(T, U): R), and BiConsumer<T, U> (accept(T, U): void). There is no built-in "Bi-Supplier" — a supplier by definition takes nothing.

Generic interfaces work on objects, so an int flowing through Function<Integer, Integer> is boxed to Integer and unboxed again. Over large datasets that autoboxing is real overhead. The JDK therefore ships primitive specializations that operate on int/long/double directly.

  • BiPredicate<T, U>, BiFunction<T, U, R>, BiConsumer<T, U> take two arguments.
  • Primitive families avoid boxing: IntPredicate, IntFunction<R>, IntUnaryOperator, IntBinaryOperator, IntConsumer, IntSupplier, ToIntFunction<T> — plus Long* and Double* equivalents.
  • IntBinaryOperator's method is applyAsInt(int, int) — note the ...AsInt naming.
  • Boxing conversions (int ↔ Integer) cost memory and time in hot loops.
import java.util.function.BiPredicate;
import java.util.function.BiFunction;
import java.util.function.BiConsumer;
import java.util.function.IntBinaryOperator;

BiPredicate<Integer, String> check = (n, s) -> n < 10 && s.length() > 5;
BiFunction<Integer, String, String> join = (n, s) -> n + " " + s;
BiConsumer<Integer, String> printBoth = (n, s) -> {
    System.out.println(n);
    System.out.println(s);
};

System.out.println(check.test(5, "in28minutes"));  // 5<10 && len 11>5
System.out.println(check.test(15, "test"));         // 15<10 is false
System.out.println(join.apply(15, "in28minutes"));
printBoth.accept(15, "in28minutes");

// Primitive specialization: no Integer boxing
IntBinaryOperator intSum = (x, y) -> x + y;
System.out.println(intSum.applyAsInt(4, 6));

07. Puzzles with functional interfaces & lambdas

Objective: Nail the lambda-syntax rules through a set of puzzles: parameter forms, inferred vs explicit types, and when braces and return are mandatory.

Lambda syntax bends in a few ways. Parameter types can be inferred from the target type or written explicitly — but all or none, never a mix. A single-expression body needs no braces and no return; a block body { ... } requires an explicit return when the interface's method returns a value. A single parameter may drop its parentheses; zero parameters must be written as an empty ().

  • Inferred: (x, y) -> x + y; explicit: (Integer x, Integer y) -> x + y — don't mix the two styles.
  • Expression body: no braces, no return. Block body: braces required, plus return if a value is expected.
  • One param can omit parentheses (str -> ...); zero params need ().
  • str -> {} is a valid Consumer (one param, empty body, returns void).
import java.util.function.BinaryOperator;
import java.util.function.Supplier;
import java.util.function.Consumer;

BinaryOperator<Integer> inferred = (x, y) -> x + y;               // types inferred

Supplier<String>        block    = () -> { return "Ranga"; };      // block: needs return
Consumer<String>        ignore   = str -> {};                     // one param, empty body

System.out.println(explicit.apply(10, 20));

System.out.println(block.get());

System.out.println("done");

08. Method references

Objective: Replace simple lambdas with method references, and recognize the four kinds and the lambda each one stands in for.

When a lambda does nothing but call one existing method, a method reference (Type::method) says the same thing more directly. There are four kinds: a reference to a static method, to an instance method bound to a specific object, to an instance method of an arbitrary object of a type, and to a constructor. Each is exactly equivalent to a lambda — the compiler just infers the parameters.

  • Static: Integer::sum = (a, b) -> Integer.sum(a, b).
  • Bound instance (on a specific object): System.out::println = x -> System.out.println(x).
  • Unbound instance (on an arbitrary receiver): String::toUpperCase = s -> s.toUpperCase().
  • Constructor: ArrayList::new = () -> new ArrayList<>().
import java.util.List;
import java.util.ArrayList;
import java.util.function.Supplier;

List<String> courses = List.of("Java", "Python", "JavaScript");

// bound instance ref    System.out::println  ==  x -> System.out.println(x)
courses.stream()

       .forEach(System.out::println);

02. allMatch, noneMatch & anyMatch

Objective: Ask yes/no questions about a whole stream with the three matching terminal operations, each taking a Predicate and returning a boolean.

Extracting the conditions into named predicates keeps the pipeline readable and lets you reuse the same rule across several queries.

  • All three take a Predicate<Course> and return boolean.
  • allMatch = "for all", anyMatch = "there exists", noneMatch = "for none".
  • Short-circuiting: they may inspect only part of the stream.

03. Sorting with Comparator — comparing, thenComparing, reversed

Objective: Build comparators with comparingInt, chain a tie-breaker with thenComparingInt, and flip direction with reversed() — noting exactly what reversed() reverses.

sorted(comparator) needs to be told how to order two courses. Comparator.comparingInt(Course::getNoOfStudents) extracts an int key and compares it without boxing. thenComparingInt(Course::getReviewScore) adds a tie-breaker used only when the first key is equal. Java's stream sort is stable, so elements the comparator considers equal keep their original encounter order.

The subtle part is reversed(): it reverses the entire comparator built so far. In the third comparator below, both the students key and the review-score tie-breaker flip to descending. That is why, among the 21000-student courses, byStudents.reversed() keeps AWS before Azure (stable order on the equal key) while the fully-reversed comparator puts Azure (99) ahead of AWS (92).

  • comparingInt / comparingLong / comparingDouble avoid boxing versus comparing.
  • thenComparingInt only breaks ties on the previous key.
  • reversed() inverts the whole chain, not just the last key.

04. skip, limit, takeWhile & dropWhile

Objective: Slice a stream by position with limit/skip, and by a leading condition with the Java 9+ takeWhile/dropWhile — understanding why the latter two are not the same as filter.

limit(n) keeps the first n elements; skip(n) discards the first n. Chained after a sorted, they express "top 5" or "page 2" queries cleanly.

takeWhile(pred) takes elements from the front while the predicate holds and stops at the very first element that fails — it never looks further. dropWhile(pred) is the mirror: it discards the leading run that matches, then keeps everything after the first failure, even elements that would satisfy the predicate. The examples below run takeWhile/dropWhile over the original (unsorted) list to make that leading-run behaviour obvious.

  • limit(n) / skip(n): positional slicing, great for paging.
  • takeWhile/dropWhile act only on the leading run; they stop testing after the first miss.
  • Not filter: dropWhile keeps Azure (99) simply because it appears after the first non-matching course.

        .sorted(byStudentsThenReviews)
        .limit(5)
        .collect(Collectors.toList()));

System.out.println(courses.stream()
        .sorted(byStudentsThenReviews)
        .skip(3)
        .limit(5)
        .collect(Collectors.toList()));

// takeWhile / dropWhile run over the ORIGINAL order

        .takeWhile(c -> c.getReviewScore() >= 95)
        .collect(Collectors.toList()));

        .dropWhile(c -> c.getReviewScore() >= 95)
        .collect(Collectors.toList()));

05. max, min, findFirst, findAny & Optional

Objective: Retrieve a single element with max/min/findFirst/findAny, all of which return an Optional<Course>, and handle absence safely with orElse.

These operations may find nothing — a filtered stream can be empty — so instead of returning null they return an Optional<Course>: a box that either holds a value (Optional[...]) or is Optional.empty. That forces the caller to acknowledge the "maybe absent" case and eliminates a whole class of NullPointerExceptions.

max(cmp) and min(cmp) are defined purely in terms of the comparator you pass. Because byStudentsThenReviews is already a descending comparator, its "maximum" is the course it ranks highest — the one with the fewest students — so max returns FullStack and min returns Microservices. findFirst returns the first element in encounter order; findAny may return any matching element (identical here on a sequential stream). orElse(fallback) unwraps the Optional, substituting a default when it is empty.

  • All four return Optional<Course> — never null.
  • Optional.empty is printed when the (possibly filtered) stream has no element.
  • orElse(default) supplies a fallback; ifPresent, map, orElseThrow are the other common exits.
  • max/min mirror the comparator's direction — a reversed comparator swaps which end you get.
System.out.println(courses.stream().max(byStudentsThenReviews));
System.out.println(courses.stream().min(byStudentsThenReviews));

5. Creating Streams

01. Stream.of and Arrays.stream

Objective: Build a stream directly from values with Stream.of and from a primitive array with Arrays.stream, and understand why an object Stream<Integer> differs from a primitive IntStream.

A stream need not come from a collection. Stream.of(v1, v2, ...) creates one from a comma-separated list of values — ideal for quick experiments or small fixed datasets. The result is an ordinary stream: you can count() it, reduce() it, map, or filter just as before.

Because Stream.of(12, 9, ...) infers a Stream<Integer>, each int is boxed into an Integer object. That is fine for a handful of values but adds boxing/unboxing overhead over large numeric datasets.

Arrays.stream(array) streams an array's elements. For a primitive int[] it returns a specialized IntStream — a stream of raw ints — which exposes numeric terminal operations directly: sum(), average(), min(), max(). No boxing, and the code reads naturally.

Watch the return types: sum() gives a plain int, but average() returns an OptionalDouble and min()/max() return an OptionalInt, because a stream might be empty and there would then be no value to hand back.

  • Stream.of(values...) creates a Stream<T> from listed values — no collection required.
  • Stream.of(ints...) is a boxed Stream<Integer>; Arrays.stream(int[]) is a primitive IntStream.
  • An IntStream offers sum(), average(), min(), max() without extra mapping.
  • count() returns a long; sum() returns an int; average() an OptionalDouble; min()/max() an OptionalInt.

02. IntStream & LongStream — range, rangeClosed, iterate

Objective: Generate numeric streams on demand with range, rangeClosed, and iterate; bound an infinite iterate with limit; and inspect a pipeline with peek.

Often you want a run of numbers that no collection holds yet. IntStream.range(start, end) produces start up to end - 1 (the end is exclusive); IntStream.rangeClosed(start, end) includes end. Both are primitive streams, so sum() and friends are available directly.

IntStream.iterate(seed, next) is more general: start at seed and apply the next function repeatedly, each element derived from the previous one. This stream is infinite, so you must cap it with limit(n) before a terminal operation — otherwise it never finishes.

peek(action) lets you observe elements as they flow past without changing them, which is handy for tracing. Like other intermediate operations it is lazy: the peek action only fires when a terminal operation pulls elements through.

A primitive IntStream cannot be collected into a List<Integer> directly. Call boxed() first to turn it into a Stream<Integer>, then collect.

  • range(1, 10) → 1..9 (end-exclusive); rangeClosed(1, 10) → 1..10 (inclusive).
  • range(1, 10).sum() = 45; rangeClosed(1, 10).sum() = 55.
  • iterate(seed, x -> next) is infinite — always bound it with limit(n).
  • iterate(1, e -> e + 2).limit(10) = the first ten odd numbers; their sum is 100.
  • peek observes without modifying; boxed() converts an IntStream to a Stream<Integer> so it can be collected.

03. Big-number calculations with BigInteger

Objective: Recognize silent integer overflow when values exceed the int/long limits, and use BigInteger with streams to compute arbitrarily large results such as 50!.

Java's primitive integers are fixed-width. An int is 32-bit with a maximum of Integer.MAX_VALUE = 2147483647; a long is 64-bit with a maximum of Long.MAX_VALUE = 9223372036854775807. When a result exceeds the maximum, Java does not throw — it silently wraps around, so you get a wrong answer with no warning at all.

Factorials expose this fast. IntStream.rangeClosed(1, 50).reduce(1, (x, y) -> x * y) multiplies 1 through 50 as ints; the true value is astronomically large, and the wrapped int product lands on 0 (50! is divisible by 247, so the low 32 bits are all zero). Switching to LongStream only buys a little room: the largest factorial that fits in a long is 20!; 50! overflows to a meaningless negative value.

The fix is BigInteger, an arbitrary-precision integer that grows as large as memory allows and never overflows. Map each number to a BigInteger and fold with BigInteger::multiply, seeded from BigInteger.ONE.

LongStream.rangeClosed(1, 50).mapToObj(BigInteger::valueOf) converts the primitive longs into BigInteger objects (mapToObj because the target is an object, not a primitive), then reduce(BigInteger.ONE, BigInteger::multiply) folds them into the exact 65-digit answer.

  • Integer.MAX_VALUE = 2147483647; Long.MAX_VALUE = 9223372036854775807.
  • Overflow is silent — no exception; the value simply wraps (often to 0 or a negative).
  • As int, 50! reduces to 0; as long it overflows to garbage past 20!.
  • BigInteger is arbitrary-precision: use BigInteger.valueOf(n), BigInteger.ONE, and multiply.
  • mapToObj bridges a primitive stream to a Stream<BigInteger>; reduce with an identity of BigInteger.ONE.

6. Advanced Stream Techniques

Four techniques that separate casual stream users from confident ones: joining and flatMap for reshaping data, higher-order functions that build behaviour on the fly, the laziness that makes pipelines efficient, and parallel() for putting every CPU core to work.

01. joining & flatMap

Objective: Fold a stream of strings into one string with Collectors.joining, and flatten a stream of arrays or nested streams into a single stream with flatMap.

Collectors.joining(sep) concatenates every element of a Stream<String> into one string, placing sep between elements. It is the stream equivalent of String.join.

flatMap is the flattening cousin of map. map is strictly 1 → 1: each element becomes exactly one result. So map(course -> course.split("")) turns each string into a String[], leaving you holding a Stream<String[]> — a stream of arrays, not of letters. flatMap is 1 → many, then flatten: its mapper returns a stream (here Arrays::stream turns each array into a Stream<String>), and flatMap concatenates all those sub-streams into one flat Stream<String>. Add distinct() to get the set of letters used.

  • joining(", ") / joining(" ") — many strings become one, with a chosen separator.
  • map = 1→1; flatMap = 1→many then flatten (mapper must return a Stream).
  • "Spring".split("") yields a String[] of single characters (Java 8+ drops the leading empty token).
  • flatMap(Arrays::stream) is the idiom for turning a Stream<T[]> into a Stream<T>.

02. Higher-order functions

Objective: Recognise higher-order functions — functions that take and/or return other functions — and use a method that returns a function to generate behaviour dynamically.

A higher-order function (HOF) either takes one or more functions as parameters, returns a function, or both. This is possible because in functional Java, functions are first-class values: you can store a lambda in a variable, pass it as an argument, and return it from a method — just like an int or a String.

The returning side is the interesting one. A method like multiplier(int factor) returns a Function whose lambda captures factor; calling multiplier(3) and multiplier(10) mints two different functions from one definition. The same trick builds tunable predicates: createPredicateWithCutoffReviewScore(int) returns a Predicate<Course> parameterised by the cutoff, replacing a pile of near-identical hardcoded predicates.

The taking side is behaviour parameterization: a method such as filterAndPrint(numbers, Predicate<? super Integer>) accepts the deciding logic as an argument. In fact the whole Stream API is higher-order — filter, map, and reduce are methods that take functions.

  • HOF = takes a function, returns a function, or both.
  • Returning a function lets you generate custom logic at runtime from parameters (a closure over those parameters).
  • createPredicateWithCutoffReviewScore(95) returns a ready-to-use Predicate<Course>.
  • Higher-order functions are the foundation the entire stream API rests on.
// HOF that RETURNS a function: the lambda captures 'factor'
static Function<Integer, Integer> multiplier(int factor) {
    return x -> x * factor;
}

// RETURNS a Predicate tuned by a runtime cutoff
static Predicate<Course> minReviewScore(int cutoff) {
    return course -> course.getReviewScore() > cutoff;
}

public static void main(String[] args) {
    Function<Integer, Integer> triple  = multiplier(3);
    Function<Integer, Integer> tenfold = multiplier(10);
    System.out.println(triple.apply(10));    // 30
    System.out.println(tenfold.apply(10));   // 100

    // a predicate built on the fly, passed straight to filter
    System.out.println(
        courses.stream()
               .filter(minReviewScore(95))
               .collect(Collectors.toList()));
}

AdvancedReturning a lambda creates a closure: the captured variable (factor, cutoff) must be effectively final, but it lives on as part of the returned function even after the enclosing method returns. 03. Streams are lazy — intermediate operations & performance

Objective: Understand that intermediate operations (filter, map, peek) do nothing until a terminal operation runs, and see how laziness plus short-circuiting means less work.

Intermediate operations are lazy: they only describe the pipeline. Nothing is processed until a terminal operation (findFirst, collect, forEach, count…) pulls elements through. This is not a minor detail — it is the core performance idea. Because a short-circuiting terminal like findFirst stops at the first result, the stream processes elements one at a time, vertically (each element flows all the way down the pipeline before the next is pulled), and quits as soon as it has an answer.

The classic proof uses peek, which lets you observe each element as it passes a stage without changing it. Filtering the 9 courses for names longer than 11 characters, only "Microservices" (13) qualifies ("Spring Boot" is exactly 11, not greater). Watch the interleaving: the source is pulled until Microservices passes the filter, is upper-cased, and satisfies findFirst — then everything stops.

  • Intermediate ops build a plan; a terminal op triggers execution.
  • Processing is element-by-element (vertical), not stage-by-stage.
  • findFirst short-circuits: AWS, PCF, Azure, Docker, Kubernetes are never touched.
  • With no terminal operation, the pipeline runs zero elements — the peeks print nothing.
List<String> courses = List.of(
    "Spring", "Spring Boot", "API", "Microservices",
    "AWS", "PCF", "Azure", "Docker", "Kubernetes");

String result = courses.stream()
    .peek(c -> System.out.println("source : " + c))
    .filter(c -> c.length() > 11)
    .peek(c -> System.out.println("passed : " + c))
    .map(String::toUpperCase)
    .peek(c -> System.out.println("mapped : " + c))
    .findFirst()          // short-circuits at the 1st match
    .orElse("none");
System.out.println("result = " + result);

AdvancedNot every operation short-circuits or streams lazily. sorted and distinct are stateful — they must buffer/consume elements before emitting, so putting them before a findFirst forces more work. Short-circuiting operations include findFirst, findAny, anyMatch/allMatch/noneMatch, and limit. findAny differs from findFirst only in parallel: it may return any matching element, letting the runtime avoid tracking encounter order.
Pitfall Treating peek as a reliable side-effect stage. Because of laziness and short-circuiting, peek may run for only a few elements — or zero, if there is no terminal op. It is a debugging aid, not a substitute for forEach. Also remember a stream is single-use: consuming it twice throws IllegalStateException.

04. Parallelizing streams

Objective: Use parallel() to spread stream work across CPU cores, and know when parallelism helps versus when its overhead makes things slower.

Because functional pipelines describe what to compute rather than looping with mutable state, the runtime is free to split the work. Calling parallel() tells Java to divide the stream into chunks, process each chunk on a separate thread of the common ForkJoinPool (one per core), and merge the partial results. The code barely changes — one method call — yet on a multicore CPU a large, CPU-bound job can finish noticeably faster.

Summing 0…999,999,999 is a good fit: it is huge, purely arithmetic, and + is associative, so the chunks recombine correctly. The course measured roughly 869 ms sequential vs ~601 ms parallel — treat those as illustrative; the exact figures depend on your hardware and core count.

  • parallel() = split across ForkJoinPool threads, compute partials, merge.
  • Same result as the sequential stream when the operation is associative and side-effect-free.
  • Worthwhile only for large, CPU-bound, stateless work — splitting/merging has real overhead.
  • For small streams or IO-bound work, parallel is usually slower.
// sequential
long t0 = System.currentTimeMillis();
long seq = LongStream.range(0, 1_000_000_000L).sum();
System.out.println(seq);
System.out.println((System.currentTimeMillis() - t0) + " ms");

// parallel: split across cores, then merge
long t1 = System.currentTimeMillis();
long par = LongStream.range(0, 1_000_000_000L)
                     .parallel()
                     .sum();
System.out.println(par);
System.out.println((System.currentTimeMillis() - t1) + " ms");

AdvancedAll parallel streams share one JVM-wide common ForkJoinPool (default parallelism = availableProcessors() - 1 worker threads, plus the calling thread). 7. Functional Programming Makes Java Easy

The same lambdas, streams, and functional interfaces you learned for collections carry straight into everyday tasks: mutating lists in place, streaming files, and launching threads. This closing section shows those payoffs and where functional style belongs in a real, still-object-oriented Java application.

01. Modifying lists — replaceAll & removeIf

Objective: Transform or delete list elements in place with List.replaceAll and List.removeIf, and understand why List.of(...) lists reject those calls.

Not everything needs a full stream pipeline. When you actually want to change the list you already hold, List exposes two functional methods. replaceAll(UnaryOperator) runs a function over every element and writes the result back into the same slot, and removeIf(Predicate) deletes every element that matches a condition. Both mutate the list directly — no new collection, no explicit loop, no Iterator juggling.

The catch: they only work on a mutable list. List.of(...) returns an immutable list, so calling replaceAll on it throws UnsupportedOperationException. The fix is to wrap the data in a mutable list first — new ArrayList<>(courses) — then mutate that copy.

  • replaceAll(x -> f(x)) replaces each element with f(x) in place (same type in, same type out — it takes a UnaryOperator<E>).
  • removeIf(x -> test(x)) removes every element for which the predicate is true.
  • List.of(...), List.copyOf(...), and Collectors.toUnmodifiableList() all produce immutable lists — mutating them throws at runtime, not compile time.
  • Contrast: stream().map(...) builds a brand-new list and never touches the source.

02. Playing with files

Objective: Read and process files with the stream-returning java.nio.file API — Files.lines for content and Files.list for directory entries.

The modern file API hands you streams, so file processing becomes just another pipeline. Files.lines(path) returns a Stream<String> that reads the file lazily, one line at a time — it never loads the whole file into memory, which matters for large files. From there the familiar operators apply: split lines into words, flatMap the word arrays into a single stream, then distinct and sorted.

Two things demand care. First, Files.lines throws a checked IOException, so the enclosing method must declare throws IOException or catch it. Second — unlike a stream over an in-memory collection — this stream is backed by an open file handle, so you should open it in a try-with-resources block so the handle is closed when the pipeline finishes.

  • Files.lines(path) → lazy Stream<String>, one line at a time (memory-friendly).
  • .map(l -> l.split(" ")) gives Stream<String[]>; .flatMap(Arrays::stream) flattens it to Stream<String> of words.
  • sorted() uses natural order, so uppercase words (ASCII 65+) sort before lowercase ones (97+).
  • Files.list(dir) → Stream<Path> of directory entries; combine with .filter(Files::isDirectory).
  • Wrap file streams in try-with-resources; the method must handle IOException.
// file.txt:
//   Some Text
//   Some Text that will be saved
//   A lot of Text Present in here

         .flatMap(Arrays::stream)        // Stream<String>: every word
         .distinct()                     // drop duplicate words
         .sorted()                       // natural (ASCII) order
         .forEach(System.out::println);
}   // enclosing method: ... throws IOException

Streams over in-memory collections don't need closing, which is why this one is easy to forget.

03. Playing with threads

Objective: Use a lambda as a thread body — Runnable is a functional interface — and understand that concurrent output order is non-deterministic.

Runnable has exactly one abstract method, run(), which makes it a functional interface. So instead of the verbose anonymous-class form, the thread's work can be written as a lambda. You then pass that Runnable to a Thread and call start(). The lambda is pure boilerplate reduction — same behaviour, far less ceremony.

When several threads run the same task, they execute concurrently, and the scheduler interleaves their output however it likes. The order you see below is just one illustrative run; the interleaving — and the thread ids — will differ every time you run it, and across machines.

  • Runnable is a functional interface (single method run()), so () -> { ... } is a valid thread body.
  • new Thread(task).start() launches a new thread; the same Runnable instance can be reused for many threads.
  • Output interleaving across threads is non-deterministic — never rely on a specific order.
  • Thread.currentThread().getId() returns the id; in JDK 19+ the preferred call is threadId().
// Verbose: anonymous Runnable implementation
Runnable verbose = new Runnable() {
    @Override public void run() {
        for (int i = 0; i < 3; i++)
            System.out.println(Thread.currentThread().getId() + ":" + i);
    }
};

// Runnable is FUNCTIONAL -> collapse the whole thing to a lambda
Runnable task = () -> {
    for (int i = 0; i < 3; i++)
        System.out.println(Thread.currentThread().getId() + ":" + i);
};

new Thread(task).start();   // reuse the same Runnable
new Thread(task).start();
new Thread(task).start();

AdvancedA Runnable takes no arguments and returns nothing. When you need a result or want to throw a checked exception, use Callable<V> with an ExecutorService instead of raw Threads. Both accept lambdas. Java 21's virtual threads (Thread.ofVirtual().start(task)) also take a plain Runnable, so the same lambda scales to millions of lightweight threads.
Pitfall Calling task.run() instead of new Thread(task).start() executes the body on the current thread — no new thread is created and nothing runs in parallel. Always call start() to actually spawn a thread. Also, variables captured by the lambda must be effectively final.

04. Functional programming in real applications

Objective: Recognise where functional style pays off in real Java code, and why it complements — rather than replaces — object-oriented design.

Java stays fundamentally object-oriented: you still model the domain with classes and objects. Functional programming is a complement, and it shines in specific places. The clearest one is processing collections — turning nested loops and mutable accumulators into a declarative filter → map → reduce/collect pipeline that reads like a description of the result rather than a recipe for computing it.

Beyond collections, the same building blocks show up in stream-based file and I/O processing, easily parallelizable CPU work (.parallel()), and cleaner behaviour parameterization — passing a lambda where you used to pass a one-method interface. Frameworks lean on this heavily: Spring's functional routing, reactive stacks like Reactor/RxJava, and callback-style APIs are all built on the standard functional interfaces you now know.

  • Best starting point: any code that filters, transforms, or aggregates a collection.
  • Declarative pipelines say what you want, not how to loop — fewer moving parts, fewer bugs.
  • FP complements OOP; it doesn't replace class design, inheritance, or application architecture.
  • The real shift is mindset — expect a learning curve moving from explicit loops and mutable state to declarative, stateless thinking.
  • Prefer immutable data and side-effect-free functions; it makes code easier to reason about and safe to parallelize.
List<String> courses =
    List.of("Spring", "Java", "Kubernetes", "Microservices");

courses.stream()                       // a declarative data pipeline
       .filter(c -> c.length() > 6)    // keep the longer names
       .map(String::toUpperCase)       // transform each element
       .forEach(System.out::println);  // consume the result

"Spring" (6) and "Java" (4) fail the > 6 filter and are dropped; the two survivors are uppercased. Nothing mutated the original courses list.

AdvancedFunctional style buys the most in parallelism: a side-effect-free pipeline over an immutable source can often become .parallelStream() or .parallel() with no other change, letting the fork/join pool split the work across cores. That safety comes only because the functions don't mutate shared state — the moment a lambda writes to an external variable, parallel execution becomes a race. This is the practical reason "prefer immutability" is more than style advice.
Pitfall Don't force everything into streams. Deeply nested, multi-line lambda chains can be harder to read than a plain loop, and side effects inside forEach/peek — especially in parallel streams — reintroduce exactly the bugs FP was meant to remove. Use functional style where it clarifies; keep a loop where a loop is clearer.

8. Quick Review — Functional Programming Q&A

A fast, interview-style recap of everything from the earlier sections. Each lecture answers a handful of common questions, backed by one worked example. Use it to consolidate — not to learn a topic for the first time.

01. Functional vs imperative & core concepts

Objective: Contrast the imperative and functional styles and name the pillars that functional code stands on.

Q: How is the functional style different from the imperative style?
A: Imperative code spells out how to reach a result — step by step, with loops and a mutable accumulator you keep updating. Functional code is declarative: you describe what you want as a stream pipeline and let the library decide how to run it. The imperative version tends to be verbose and full of mutating variables; the functional version is concise and free of shared state.

Q: What are the core ideas behind functional programming?
A: Functions are first-class (stored in variables, passed as arguments, returned from methods); pure functions depend only on their inputs and produce no side effects; data is immutable; and small operations compose into larger ones. Together these make code easier to reason about and to parallelize.

  • Imperative = how (loops + mutable state); functional/declarative = what (a pipeline).
  • Pillars: first-class functions, purity, immutability, composition, no side effects.
  • The functional form of "sum these numbers" is a reduce, not a running total.
// Imperative — HOW: mutable accumulator, explicit loop
int sum = 0;
for (int n : List.of(1, 2, 3, 4)) {
    sum += n;                    // state mutates each pass
}
System.out.println("imperative: " + sum);

// Functional — WHAT: fold the stream with addition
int total = List.of(1, 2, 3, 4).stream()
        .reduce(0, Integer::sum);   // no mutable state
System.out.println("functional: " + total);
Console outputimperative: 10 functional: 10

Source: reimplemented as an original illustration.

AdvancedA pure function is referentially transparent: a call can be replaced by its result without changing the program's meaning. That property is what makes memoization (caching results) and safe reordering possible.
Pitfall Using a stream only to poke an external counter or list inside forEach re-introduces mutable state and defeats the purpose. Prefer reduce, count, or collect so the result comes out of the pipeline, not a side effect.

02. Creating streams & stream operations

Objective: Recall the many ways to build a stream and the intermediate-vs-terminal split that governs when a pipeline actually runs.

Q: What are the common ways to obtain a stream?
A: From a collection with collection.stream(); from a handful of known values with Stream.of(a, b, c); from an array with Arrays.stream(arr); from a numeric range with IntStream.range / rangeClosed; from a generating rule with Stream.generate or Stream.iterate; and from a file, line by line, with Files.lines(path). For a map you can stream its keySet(), values(), or entrySet().

Q: What separates an intermediate from a terminal operation?
A: Intermediate operations (filter, map, sorted, ...) are lazy and return a new stream; nothing happens until a terminal operation (collect, forEach, reduce, count, anyMatch, ...) consumes the pipeline.

Q: What should you always keep in mind about streams?
A: A stream is single-use — after one terminal op it is consumed; the source is never mutated; and laziness plus short-circuiting (e.g. findFirst, limit) means the pipeline may stop early and never touch every element.

  • Creation: stream(), Stream.of, Arrays.stream, IntStream.range/rangeClosed/iterate, Files.lines.
  • Intermediate ops are lazy and chainable; a terminal op fires the whole chain once.
  • Streams do not store data — they process it on the fly, so they stay memory-efficient even for millions of elements.
// A few ways to build a stream
IntStream.rangeClosed(1, 5)              // 1,2,3,4,5
         .forEach(System.out::print);
System.out.println();

int rangeSum = IntStream.range(1, 5).sum();  // 1+2+3+4 (5 excluded)
System.out.println("range sum: " + rangeSum);

// Lazy intermediates run only when a terminal op arrives
List<Integer> squares = List.of(3, 1, 2).stream()
        .map(n -> n * n)     // intermediate (lazy)
        .sorted()            // intermediate (lazy)
        .toList();           // terminal -> pipeline executes
System.out.println(squares);
Console output12345 range sum: 10 [1, 4, 9]

AdvancedStream.iterate and Stream.generate build infinite streams. You must bound them — with limit(n), a takeWhile condition, or a short-circuiting terminal like findFirst — or the terminal operation never finishes.
Pitfall Reusing a consumed stream throws IllegalStateException: stream has already been operated upon or closed. A stream is not a collection — build a fresh one each time you need to traverse the data again.

03. Lambdas & method references

Objective: Review lambda syntax and the four kinds of method reference, and know when a reference reads better than a lambda.

Q: What forms can a lambda take?
A: () -> expr for no arguments, x -> expr for one (parentheses optional), (x, y) -> expr for several, and a block body x -> { ...; return v; } when the logic needs multiple statements. The arrow separates the parameters from the body; a single-expression body needs no braces or return.

Q: What are the four kinds of method reference?
A: static (Integer::parseInt), bound instance where the receiver is a fixed object (System.out::println), unbound instance where the receiver is each stream element (String::toUpperCase), and constructor (ArrayList::new).

Q: When is a method reference cleaner than a lambda?
A: When the lambda does nothing but forward its argument to one existing method. s -> s.toUpperCase() becomes String::toUpperCase — shorter, and it states intent without the boilerplate.

  • A lambda is an anonymous function: parameters, an arrow, and a body.
  • A method reference is shorthand for a lambda that just calls one method.
  • Four kinds: static, bound instance, unbound instance, constructor.
// Lambda forms
Runnable greet                     = () -> System.out.println("hi");    // no args
Function<Integer, Integer> square  = x -> x * x;                        // one arg
Comparator<String> byLength        = (a, b) -> a.length() - b.length(); // two args
Function<String, String> shout     = s -> {                             // block body
    String trimmed = s.strip();
    return trimmed.toUpperCase();
};

// The four kinds of method reference
Stream.of("1", "2").map(Integer::parseInt);   // 1. static
names.forEach(System.out::println);           // 2. bound instance
names.stream().map(String::toUpperCase);      // 3. unbound instance
Stream.generate(ArrayList::new).limit(2);     // 4. constructor

Source: reimplemented as an original illustration.

AdvancedThe difference between String::toUpperCase (unbound) and greeting::length (bound) is the receiver: an unbound reference uses each incoming element as the target, while a bound reference fixes the target to a captured object. The compiler selects the right shape from the target functional interface.
Pitfall A method reference only compiles if its signature matches the target single abstract method. Mismatched arity or return type (e.g. handing a two-argument method to a one-argument Function) is a compile error, not a runtime one — the fix is usually to fall back to an explicit lambda.

04. Functional interfaces in depth

Objective: Recall the built-in functional interfaces, what makes an interface "functional," and why the primitive variants exist.

Q: What makes an interface a functional interface?
A: It declares exactly one abstract method (a SAM), so a lambda or method reference can supply that method's body. The optional @FunctionalInterface annotation asks the compiler to enforce the single-abstract-method rule.

Q: What are the key functional interfaces?
A: Predicate<T> (T → boolean), Function<T, R> (T → R), Consumer<T> (T → void), Supplier<T> (() → T), UnaryOperator<T> (T → T), and BinaryOperator<T> ((T, T) → T). The Bi- variants — BiFunction, BiPredicate, BiConsumer — take two inputs. The operator interfaces are special cases where all types are the same.

Q: Why do primitive functional interfaces like IntPredicate exist?
A: To avoid autoboxing. Predicate<Integer> boxes each int into an Integer object; IntPredicate works on the raw primitive — faster execution, less heap allocation, cleaner code. The same pattern gives IntFunction, IntConsumer, IntBinaryOperator, and their long/double cousins.

  • Functional interface = one abstract method; lambdas implement it behind the scenes.
  • Operators (UnaryOperator, BinaryOperator) fix input and output to the same type.
  • Primitive interfaces (IntPredicate, ...) skip boxing/unboxing for better performance.
Predicate<Integer>       isEven = n -> n % 2 == 0;      // T -> boolean
Function<String, Integer> length = String::length;      // T -> R
Consumer<String>         print  = System.out::println;  // T -> void
Supplier<Double>         random = Math::random;         // () -> T
UnaryOperator<Integer>   triple = x -> x * 3;            // T -> T
BinaryOperator<Integer>  add    = Integer::sum;          // (T,T) -> T
BiFunction<String, String, String> join = (a, b) -> a + b; // (T,U) -> R

// Primitive specialization — operates on int, no boxing
IntPredicate isOdd = n -> n % 2 != 0;

Source: reimplemented as an original illustration.

AdvancedA functional interface may still declare default and static methods — only the single abstract method counts. That is exactly how Predicate ships composition helpers like and, or, and negate without ceasing to be functional.
Pitfall Function<Integer, Integer> and IntUnaryOperator are not interchangeable. The primitive-specialized interfaces have different method names (applyAsInt vs apply) and a different generic shape, so you cannot pass one where the other is expected.

05. Collectors, Optional & parallel streams

Objective: Review grouping data with Collectors.groupingBy, handling absence with Optional, and when parallel streams pay off.

Q: When do you use Collectors.groupingBy?
A: To bucket stream elements into a Map keyed by some attribute — e.g. courses grouped by category. A downstream collector shapes each bucket: counting() for a tally, mapping(...) to transform elements, maxBy(comparator) for the top element per group.

Q: What problem does Optional solve?
A: It models a possibly-absent value explicitly so you handle "missing" deliberately instead of hitting a NullPointerException. Read it with isPresent, ifPresent(consumer), orElse(default), orElseGet(supplier), orElseThrow(), or transform it with map. Use Optional.ofNullable(v) when v might be null — Optional.of(null) throws.

Q: When do parallel streams actually help?
A: For large, CPU-bound workloads whose operation is associative and stateless, over a source that splits cheaply. Because functional pipelines carry no shared mutable state, Java can fan the work across cores and recombine the results. For small or I/O-bound work the fork/join overhead usually makes them slower.

  • groupingBy(classifier, downstream) → a Map of buckets, optionally aggregated.
  • Optional forces you to deal with absence and avoids scattered null checks.
  • Add .parallel() only for big, CPU-bound, associative, stateless work.
record Course(String name, String category, int rating) {}

List<Course> courses = List.of(
    new Course("AWS",    "Cloud",     95),
    new Course("Azure",  "Cloud",     90),
    new Course("Spring", "Framework", 97),
    new Course("JPA",    "Framework", 92));

// Count courses per category (downstream: counting)
Map<String, Long> perCategory = courses.stream()
    .collect(Collectors.groupingBy(Course::category, Collectors.counting()));
System.out.println(perCategory);

// Highest-rated course per category (maxBy -> Optional)
Map<String, Optional<Course>> best = courses.stream()
    .collect(Collectors.groupingBy(Course::category,
             Collectors.maxBy(Comparator.comparingInt(Course::rating))));
System.out.println(best.get("Cloud").map(Course::name).orElse("none"));
Console output{Cloud=2, Framework=2} AWS

AdvancedgroupingBy returns a HashMap by default, so key iteration order is unspecified. The three-argument overload lets you pass a map supplier for ordering — groupingBy(classifier, TreeMap::new, downstream) gives you sorted keys.
Pitfall Calling Optional.get() on an empty optional throws NoSuchElementException — reach for orElse/orElseGet/ifPresent/map instead. And never parallelize a tiny list: the thread-coordination cost dwarfs the work.

06. Higher-order functions & the benefits of FP

Objective: Tie together higher-order functions, behavior parameterization, and first-class functions, then summarize why functional programming is worth it.

Q: What is a higher-order function?
A: A function that takes another function as a parameter and/or returns one. filterAndPrint(list, predicate) accepts behavior; minRating(cutoff) returns a ready-made Predicate. Both treat functions as ordinary values.

Q: How do higher-order functions, behavior parameterization, and first-class functions relate?
A: They are three views of one idea — functions as values. First-class means a function can be stored, passed, and returned. Behavior parameterization means the varying logic is handed to a method as an argument, so callers decide what it does. A higher-order function is any method that consumes or produces functions. Functional programming makes all three natural.

Q: What are the concrete benefits of the functional style?
A: Concise, declarative code with less boilerplate; fewer bugs thanks to immutability and no shared mutable state; easy parallelism (just .parallel(), no locks or race conditions to manage); and composable, reusable behavior you can pass around.

  • Higher-order function = takes and/or returns a function.
  • Behavior parameterization + first-class functions are the same idea from different angles.
  • Payoff: declarative code, immutability, trivial parallelism, reusable behavior.
// A factory that RETURNS a function -> higher-order function
static Predicate<Course> minRating(int cutoff) {
    return course -> course.rating() >= cutoff;
}

// Behavior parameterization: the caller supplies the logic
Predicate<Course> topTier = minRating(95);   // stored in a variable
courses.stream()
       .filter(topTier)                       // passed as an argument
       .map(Course::name)
       .forEach(System.out::println);

AdvancedA factory like minRating(95) returns a closure — the returned predicate captures the cutoff value. That gives you parameterized, reusable behavior with no subclassing and no duplicated filtering code.
Pitfall A lambda can only capture effectively final local variables. Trying to mutate a captured local won't compile, and mutating shared state from a lambda — especially inside a parallel stream — creates race conditions. Keep lambdas pure.

Wrapping up

That completes the tour of functional programming in Java. The core is small and composes endlessly: turn a collection into a Stream, chain lazy intermediate operations — filter, map, sorted, distinct, limit — and finish with one terminal operation such as forEach, collect, reduce, or a match/find. Lambdas and method references supply the behaviour; the standard functional interfaces give that behaviour a type; and higher-order methods let you pass behaviour as an argument, so the same pipeline can be reshaped without rewriting its plumbing. The recurring lesson: prefer expressions over statements, keep functions free of side effects, let the stream stay lazy until a terminal call pulls the data through, and reach for parallel() only once you have measured a real, CPU-bound bottleneck on a large dataset. Written this way, Java code reads as a description of the result rather than a script of the steps — shorter, clearer, and easier to parallelize.
