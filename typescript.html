TypeScript

TypeScript is a statically typed superset of JavaScript that compiles to plain JavaScript. It adds a structural type system, generics, and a rich set of type-level operators on top of the language you already know, catching whole classes of bugs before the code ever runs. This page moves from setup and basic types through generics, mapped and conditional types, discriminated unions, type guards, declaration files, compiler configuration, and a JavaScript-to-TypeScript migration path. This docsite is fully offline and bundles no compiler, so TypeScript source cannot be type-checked or executed live. For every concept the TypeScript source is shown in a static code block paired with a Result box that states the simulated outcome — either the expected tsc diagnostic (with its error code) for type-error cases, or the runtime output of the emitted JavaScript. Where a concept has a directly runnable JavaScript equivalent after type erasure, an interactive editor is also provided so you can run the erased behavior and read real console output.

Setup & tooling

TypeScript ships as the typescript npm package, whose entry point is the command-line compiler tsc. The compiler reads .ts files, type-checks them, and emits .js (and optionally .d.ts declaration files and source maps). Types are a compile-time-only construct: they are fully erased from the emitted JavaScript, so nothing about your type annotations survives into the runtime bundle.

A project is described by a tsconfig.json at its root. Running npx tsc with no file arguments discovers that config and compiles the whole project according to it. Modern editors (VS Code ships the TypeScript language service in-box) read the same tsconfig.json to power inline diagnostics, autocompletion, and refactoring — the editor and the command-line compiler agree because they share the compiler.

For iteration without a separate build step, ts-node runs .ts files directly; esbuild and swc transpile TypeScript to JavaScript extremely fast by stripping types without checking them; and bundlers such as vite use those transpilers for dev servers while leaving full type-checking to tsc --noEmit in CI. The division of labour is deliberate: fast tools erase types, tsc verifies them.

Example: hello.ts

// hello.ts
function greet(name: string): string {
  return `Hello, ${name}!`;
}

const message = greet("world");
console.log(message);

// A type error is caught before emit:
greet(42); // Argument of type 'number' is not assignable to 'string'
Result: tsc reports error TS2345: Argument of type 'number' is not assignable to parameter of type 'string'. on the greet(42) call. Removing that line, the emitted hello.js runs and prints Hello, world!.

Example: minimal tsconfig.json

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "outDir": "dist",
    "rootDir": "src",
    "declaration": true,
    "sourceMap": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  },
  "include": ["src"]
}
Result: Running npx tsc discovers this config, type-checks every file under src/ in strict mode, and emits .js, .d.ts, and .js.map files into dist/. With no type errors, tsc exits with code 0 and prints nothing.

Note: Install per-project with npm install --save-dev typescript and invoke via npx tsc so the compiler version is pinned in package.json rather than depending on a global install.

Note: For large codebases, run type-checking as its own CI job with tsc --noEmit while your bundler handles the actual transpile. Type-checking is the slow part; emitting via esbuild/swc is near-instant.

Basic types

TypeScript's primitive types mirror JavaScript's runtime values: string, number, boolean, null, undefined, bigint, and symbol. Collections are typed as arrays (number[] or the equivalent Array<number>) and tuples ([string, number]), where a tuple fixes both the length and the type at each position. An enum defines a named set of constants and, unlike most type constructs, emits real runtime code.

Three special types govern the "escape hatches." any disables type-checking entirely and is contagious; unknown is the type-safe counterpart — you can hold any value but must narrow it before use; and never is the empty type, the return type of a function that never returns and the result of an exhausted union. Literal types ("GET" | "POST", 200 | 404) let a type describe a specific value, not just its category.

Wherever the type is obvious from the initializer, TypeScript infers it, so explicit annotations on locals are usually redundant. Annotations earn their place at boundaries: function parameters, public return types, and exported values.

Example: primitives, arrays, tuples, and inference

let title: string = "DevDocs";
let count = 3;                 // inferred as number
const done: boolean = false;
let maybe: number | undefined; // undefined until assigned

const scores: number[] = [88, 91, 79];
const pair: [string, number] = ["age", 42]; // fixed-shape tuple

enum Direction { North, East, South, West }
const heading: Direction = Direction.East; // 1 at runtime

let payload: unknown = JSON.parse('{"ok":true}');
if (typeof payload === "object" && payload !== null) {
  console.log("parsed an object");
}

type Method = "GET" | "POST"; // literal union
const verb: Method = "GET";
console.log(title, count, heading, verb);
Result: Type-checks cleanly. The emitted JavaScript prints DevDocs 3 1 GET (the enum member Direction.East is 1), then earlier logs parsed an object.

Example: any vs unknown vs never, and an intentional error

let loose: any = "text";
loose.toFixed(2); // allowed — 'any' disables checking (may crash at runtime)

let safe: unknown = "text";
safe.toFixed(2);  // ERROR: cannot call methods on 'unknown' without narrowing

function fail(msg: string): never {
  throw new Error(msg); // never returns normally
}
Result: tsc reports error TS2571: Object is of type 'unknown'. on safe.toFixed(2). The loose.toFixed(2) line is accepted at compile time (because any opts out) but would throw TypeError: loose.toFixed is not a function at runtime.

Note: Prefer unknown to any for values whose type you do not yet know (parsed JSON, catch bindings). It preserves type safety by forcing you to narrow before use.

Structural typing

TypeScript is structurally typed: assignability is determined by shape, not by name or declared lineage. If a value has at least the members a target type requires, with compatible types, it is assignable — regardless of which class or interface it was declared against. This is sometimes called "duck typing": if it walks like a duck and quacks like a duck, the type system treats it as a duck.

The one deliberate exception is excess property checking. When you assign an object literal directly to a typed target, TypeScript flags properties that the target does not declare, on the theory that a stray property in a fresh literal is almost always a typo. Assigning through an intermediate variable bypasses the check, because the value is no longer a "fresh" literal.

Example: assignability by shape

interface Point { x: number; y: number; }

function length(p: Point): number {
  return Math.hypot(p.x, p.y);
}

// No 'implements Point' anywhere — shape alone qualifies it.
const vec = { x: 3, y: 4, label: "v1" };
console.log(length(vec)); // 5 — extra 'label' is fine via a variable

class Vector2 {
  constructor(public x: number, public y: number) {}
}
console.log(length(new Vector2(6, 8))); // 10 — class matches structurally
Result: Type-checks cleanly; vec is accepted because it has x and y, and Vector2 matches by shape. The emitted JavaScript prints 5 then 10.

Example: excess property check on a literal

interface Config { url: string; timeout?: number; }

// Fresh object literal: excess 'retries' is flagged.
const bad: Config = { url: "/api", retries: 3 };

// Same object via a variable: the check does not apply.
const raw = { url: "/api", retries: 3 };
const ok: Config = raw; // accepted
Result: tsc reports error TS2353: Object literal may only specify known properties, and 'retries' does not exist in type 'Config'. on bad. The ok assignment through raw is accepted because the value is no longer a fresh literal.

Note: To accept known-plus-arbitrary keys deliberately, add an index signature ([key: string]: unknown) to the target type rather than reaching for any.

Interfaces & type aliases

Both interface and type name a shape. Interfaces are limited to object and function shapes but support declaration merging (two interface declarations with the same name combine) and extends. Type aliases can name any type — unions, tuples, primitives, mapped and conditional types — and compose via intersection (&). Members may be marked optional with ? or immutable with readonly, and an index signature describes objects with dynamic keys.

Use an interface for object shapes you expect others to extend or augment, especially public API surfaces and ambient declarations that benefit from merging. Reach for a type alias when you need unions, tuples, or any of the type-level operators an interface cannot express.

Example: interface with optional, readonly, and index signature

interface User {
  readonly id: number;   // cannot be reassigned after construction
  name: string;
  email?: string;        // optional
  [attr: string]: unknown; // index signature: arbitrary extra keys
}

const u: User = { id: 1, name: "Ada", role: "admin" };
u.name = "Ada L.";       // ok
u.id = 2;                // ERROR: 'id' is readonly
console.log(u.email ?? "no email on file");
Result: tsc reports error TS2540: Cannot assign to 'id' because it is a read-only property. on u.id = 2. With that line removed, the emitted JavaScript prints no email on file.

Example: extends, intersection, and declaration merging

interface HasId { id: number; }
interface Timestamped { createdAt: Date; }

// Interface extension:
interface Entity extends HasId, Timestamped { name: string; }

// Equivalent via type-alias intersection:
type EntityT = HasId & Timestamped & { name: string };

// Declaration merging: a second interface adds to the first.
interface Entity { updatedAt?: Date; }

const e: Entity = { id: 1, name: "row", createdAt: new Date() };
console.log(e.id, e.name, e.updatedAt);
Result: Type-checks cleanly. Entity now requires id, createdAt, and name and permits an optional updatedAt from the merged declaration. The emitted JavaScript prints 1 row undefined.

Note: Declaration merging is unique to interface. A type alias declared twice with the same name is a duplicate-identifier error (TS2300), which is exactly why library authors expose extension points as interfaces.

Typing functions

Function parameters carry annotations; the return type is usually inferred but can be stated explicitly to lock a public contract. Parameters may be optional (?), carry defaults (which makes them optional), or gather the rest into a typed array (...args: number[]). A standalone function type is written as an arrow expression, (a: number, b: number) => number, and is what you annotate callbacks and stored functions with.

Overloads let one implementation present several typed call signatures: you list the overload signatures, then write a single implementation signature (which is not itself callable from outside). The special this parameter — a fake first parameter that is erased at runtime — types the receiver a function expects when it relies on this.

Example: optional, default, rest, and a function-type callback

function format(label: string, value: number, unit = "px"): string {
  return `${label}: ${value}${unit}`;
}

function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}

// Function-type expression as a parameter:
function mapNums(xs: number[], fn: (n: number) => number): number[] {
  return xs.map(fn);
}

console.log(format("width", 12));        // width: 12px
console.log(sum(1, 2, 3, 4));            // 10
console.log(mapNums([1, 2, 3], n => n * n)); // [ 1, 4, 9 ]
Result: Type-checks cleanly. The emitted JavaScript prints width: 12px, then 10, then [ 1, 4, 9 ].

Because that logic survives type erasure unchanged, you can run the equivalent JavaScript directly:

Example: overloads and this typing

// Overload signatures:
function parse(input: string): string[];
function parse(input: number): number[];
// Implementation signature (not visible to callers):
function parse(input: string | number): string[] | number[] {
  return typeof input === "string"
    ? input.split(",")
    : [input];
}

const a = parse("x,y,z"); // typed string[]
const b = parse(7);       // typed number[]

interface Counter { value: number; }
// 'this' typed via the fake first parameter:
function increment(this: Counter, by: number): void {
  this.value += by;
}
const c: Counter = { value: 0 };
increment.call(c, 5);
console.log(a, b, c.value);
Result: Type-checks cleanly; a resolves to string[] and b to number[] via overload resolution. The emitted JavaScript prints [ 'x', 'y', 'z' ] [ 7 ] 5.

Note: Prefer a single signature with a union or generic over overloads when the return type can be expressed without them — overloads are harder to maintain and do not compose.

Generics

Generics parameterize a definition over a type. A generic function captures the relationship between its inputs and outputs — identity<T>(x: T): T returns exactly what it received, at whatever type the caller used. A generic class carries a type parameter across all its members, so a Box<T> stores and returns a T. Constraints (T extends U) restrict which types are allowed, default type parameters supply a fallback, and generic interfaces describe reusable parameterized shapes.

Type parameters are erased at compile time: at runtime identity<number> and identity<string> are the same function. That is why the erased behavior is directly runnable as JavaScript.

Example: generic function with a constraint and default

function identity<T>(value: T): T {
  return value;
}

// Constraint: T must have a 'length'.
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

// Default type parameter:
function wrap<T = string>(value: T): T[] {
  return [value];
}

const n = identity(42);            // T inferred as number
const s = longest("abc", "de");    // "abc"
const w = wrap(true);              // boolean[]; T defaults to string only if uninferable
console.log(n, s, w);
Result: Type-checks cleanly. identity(42) is typed number, longest returns the longer argument, and wrap(true) is boolean[]. The emitted JavaScript prints 42 abc [ true ].

Example: generic class and generic interface

// Generic interface:
interface Container<T> {
  get(): T;
  set(value: T): void;
}

// Generic class implementing it:
class Box<T> implements Container<T> {
  constructor(private value: T) {}
  get(): T { return this.value; }
  set(value: T): void { this.value = value; }
  map<U>(fn: (value: T) => U): Box<U> {
    return new Box(fn(this.value));
  }
}

const box = new Box<number>(10);
box.set(21);
const doubled = box.map(n => n * 2); // Box<number>
const asText = box.map(n => `#${n}`); // Box<string>
console.log(doubled.get(), asText.get());
Result: Type-checks cleanly. box.map transforms the element type, so doubled is Box<number> and asText is Box<string>. The emitted JavaScript prints 42 #21.

Note: Let inference do the work — write identity(42), not identity<number>(42). Only supply explicit type arguments when inference cannot determine them or you need to widen deliberately.

Mapped & utility types

Three operators drive type-level transformation over object types. keyof T yields the union of T's keys. Indexed access T[K] reads the type at a key (or union of keys). A mapped type, { [K in keyof T]: … }, rebuilds an object type key by key, optionally adding or removing readonly and ? modifiers (-readonly, -?) and remapping keys with an as clause.

The standard library ships utility types built from these primitives: Partial<T> makes all members optional, Required<T> the reverse, Readonly<T> freezes them, Pick<T, K> and Omit<T, K> select or drop keys, Record<K, V> builds a dictionary type, and ReturnType<F> extracts a function's return type.

Example: keyof, indexed access, and a mapped type

interface Product {
  id: number;
  name: string;
  price: number;
}

type ProductKey = keyof Product;        // "id" | "name" | "price"
type PriceType = Product["price"];      // number

// Mapped type: make every member nullable.
type Nullable<T> = { [K in keyof T]: T[K] | null };
type NProduct = Nullable<Product>;
// { id: number | null; name: string | null; price: number | null }

const draft: NProduct = { id: 1, name: null, price: 9.99 };
console.log(draft);
Result: Type-checks cleanly. ProductKey resolves to "id" | "name" | "price", PriceType to number, and Nullable<Product> makes each field accept null. The emitted JavaScript prints { id: 1, name: null, price: 9.99 }.

Example: built-in utility types

interface User {
  id: number;
  name: string;
  email: string;
}

type PatchUser = Partial<User>;                 // all optional
type PublicUser = Omit<User, "email">;          // { id; name }
type NameOnly = Pick<User, "name">;             // { name }
type FrozenUser = Readonly<User>;               // all readonly
type UsersById = Record<number, User>;          // dictionary

function makeUser(): User {
  return { id: 1, name: "Ada", email: "ada@x.dev" };
}
type MakeUserResult = ReturnType<typeof makeUser>; // User

const patch: PatchUser = { name: "Grace" };     // email/id omitted OK
const table: UsersById = { 1: makeUser() };
console.log(patch, Object.keys(table));
Result: Type-checks cleanly. patch may specify any subset of User, PublicUser drops email, and UsersById maps numeric ids to users. The emitted JavaScript prints { name: 'Grace' } [ '1' ].

Note: Omit is homomorphic-ish but does not preserve unions the way Exclude-based filtering does; for discriminated unions prefer Extract/Exclude on the union directly rather than Omit on each member.

Conditional types

A conditional type chooses between two types based on an assignability test: T extends U ? X : Y. When the checked type is a naked type parameter and the input is a union, the conditional distributes over each member — ToArray<string | number> becomes string[] | number[], not (string | number)[]. Wrapping the parameter in a tuple ([T] extends [U]) suppresses distribution when you want the union treated as a whole.

The infer keyword introduces a fresh type variable inside the extends clause, capturing part of the matched type. This is how helpers like ReturnType, an element-type extractor, or an Awaited-style unwrapper are written.

Example: distributive conditional + element extraction with infer

// Distributes over unions:
type ToArray<T> = T extends unknown ? T[] : never;
type A = ToArray<string | number>; // string[] | number[]

// 'infer' captures the element type of any array:
type ElementType<T> = T extends (infer E)[] ? E : T;
type B = ElementType<number[]>;    // number
type C = ElementType<string>;      // string (not an array -> itself)
Result: These are type-only declarations, so no JavaScript is emitted. The compiler resolves A to string[] | number[], B to number, and C to string.

Example: an Awaited-like unwrapper

// Recursively unwrap nested Promises.
type Unwrap<T> =
  T extends Promise<infer Inner> ? Unwrap<Inner> : T;

type R1 = Unwrap<Promise<number>>;          // number
type R2 = Unwrap<Promise<Promise<string>>>;  // string
type R3 = Unwrap<boolean>;                  // boolean

// Suppress distribution by wrapping in a tuple:
type IsExactlyString<T> = [T] extends [string] ? true : false;
type R4 = IsExactlyString<string | number>; // false
Result: Type-only; nothing is emitted. The compiler resolves R1 to number, R2 to string, R3 to boolean, and R4 to false (because the wrapped test treats the union as a whole).

Note: Conditional types can recurse (as Unwrap does), but the compiler enforces an instantiation-depth limit; extremely deep recursion triggers error TS2589: Type instantiation is excessively deep and possibly infinite.

Discriminated unions

A discriminated (tagged) union is a union of object types that share a common literal-typed field — the discriminant, conventionally named kind, type, or tag. Because each member pins that field to a distinct literal, checking it in a switch or if narrows the value to exactly one member, and the type system tracks which properties are then available.

Pairing the switch with a default branch that assigns the value to never gives you compile-time exhaustiveness: if a new member is added to the union but not handled, the assignment fails to type-check, turning "I forgot a case" into a build error.

Example: tagged union with exhaustive switch

type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; side: number }
  | { kind: "rect"; w: number; h: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle": return Math.PI * shape.radius ** 2;
    case "square": return shape.side ** 2;
    case "rect":   return shape.w * shape.h;
    default: {
      // If a new variant is added and left unhandled, this errors.
      const _exhaustive: never = shape;
      return _exhaustive;
    }
  }
}
console.log(area({ kind: "rect", w: 3, h: 4 })); // 12
Result: Type-checks cleanly with all three cases handled. If you add a { kind: "triangle" … } member without a case, tsc reports error TS2322: Type '{ kind: "triangle"; … }' is not assignable to type 'never'. at the default. As written, the emitted JavaScript prints 12.

The runtime narrowing is just a switch on a string — run the erased JavaScript and see it dispatch:

Example: a Result union for error handling

type Result<T> =
  | { ok: true; value: T }
  | { ok: false; error: string };

function parseIntSafe(s: string): Result<number> {
  const n = Number(s);
  return Number.isNaN(n)
    ? { ok: false, error: `not a number: ${s}` }
    : { ok: true, value: n };
}

const r = parseIntSafe("42");
if (r.ok) {
  console.log(r.value.toFixed(1)); // narrowed to the success member
} else {
  console.log(r.error);            // narrowed to the failure member
}
Result: Type-checks cleanly; inside if (r.ok) only value is accessible and in the else only error is. The emitted JavaScript prints 42.0.

Note: Prefer a discriminated Result union over throwing for recoverable, expected failures. It forces callers to handle both branches and keeps error information in the type.

Type guards

Narrowing shrinks a value's type within a block. The built-in guards are typeof (for primitives), instanceof (for class instances), the in operator (for property presence), and truthiness/equality checks. For custom logic, a user-defined type guard is a function whose return type is a type predicate arg is T; when it returns true, the caller's binding is narrowed to T.

An assertion function, declared with an asserts return annotation, narrows by throwing instead of returning a boolean: after the call succeeds, the value is known to satisfy the asserted type for the rest of the scope. This is the type-level counterpart to a runtime invariant check.

Example: typeof / instanceof / in and a user-defined guard

function describe(x: string | number | Date): string {
  if (typeof x === "string") return x.toUpperCase(); // narrowed: string
  if (x instanceof Date) return x.toISOString();      // narrowed: Date
  return x.toFixed(2);                                // narrowed: number
}

interface Cat { meow(): void; }
interface Dog { bark(): void; }

// 'in' narrows by property presence:
function speak(pet: Cat | Dog): void {
  if ("meow" in pet) pet.meow(); else pet.bark();
}

// User-defined type guard:
function isString(v: unknown): v is string {
  return typeof v === "string";
}
const vals: unknown[] = ["a", 1, "b"];
console.log(vals.filter(isString)); // typed string[]
Result: Type-checks cleanly. vals.filter(isString) is typed string[] thanks to the predicate. The emitted JavaScript prints [ 'a', 'b' ].

Example: assertion function

// Assertion function: throws unless the condition holds.
function assertIsDefined<T>(
  value: T,
  name: string
): asserts value is NonNullable<T> {
  if (value === null || value === undefined) {
    throw new Error(`${name} is not defined`);
  }
}

function firstChar(s: string | null): string {
  assertIsDefined(s, "s"); // after this line, s is 'string'
  return s.charAt(0);      // no '!' needed; already narrowed
}
console.log(firstChar("TypeScript")); // "T"
Result: Type-checks cleanly. After assertIsDefined, s is narrowed from string | null to string, so s.charAt(0) needs no non-null assertion. The emitted JavaScript prints T.

Note: A guard's runtime check must actually match its declared predicate. A dishonest guard (returning true without really verifying T) silently lies to the type system and reintroduces the very bugs types are meant to prevent.

Declaration files

A declaration file (.d.ts) contains types only — no runtime code. It describes the shape of JavaScript that exists elsewhere: a compiled library, a global script, or an untyped dependency. The declare keyword introduces ambient declarations that assert "this thing exists at runtime, here is its type" without emitting anything.

To type an untyped package you write a module declaration matching its import specifier; to type globals injected by a script or platform you augment the global scope. For popular libraries you rarely write these by hand — the DefinitelyTyped repository publishes community-maintained types under the @types/* scope (e.g. npm i -D @types/node), and TypeScript picks them up automatically.

Example: a module .d.ts for an untyped library

// types/legacy-slug.d.ts
// Provides types for a JS-only package 'legacy-slug'.
declare module "legacy-slug" {
  export interface SlugOptions {
    lower?: boolean;
    separator?: string;
  }
  export function slugify(input: string, opts?: SlugOptions): string;
  const _default: { slugify: typeof slugify };
  export default _default;
}
Result: With this file included, import { slugify } from "legacy-slug" type-checks and slugify("Hello World", { lower: true }) is typed as returning string. No JavaScript is emitted from the .d.ts itself.

Example: ambient global augmentation

// types/globals.d.ts
// Describe a global injected by an inline <script> and extend Window.
declare global {
  interface Window {
    __APP_VERSION__: string;
  }
  // A bare ambient global variable:
  const __BUILD_HASH__: string;
}

// 'export {}' makes this file a module so 'declare global' is valid.
export {};
Result: After this augmentation, window.__APP_VERSION__ and the bare global __BUILD_HASH__ are typed as string everywhere. Accessing an undeclared global instead yields error TS2304: Cannot find name '…'.

Note: declare global is only valid inside a module. If a .d.ts has no top-level import/export, add export {}; to make it a module; otherwise its declarations leak into the global scope and can collide.

Compiler options

The compilerOptions block in tsconfig.json controls both how strictly code is checked and what gets emitted. The single most consequential flag is strict, an umbrella that enables a family of sub-flags — most importantly noImplicitAny (an un-annotated parameter whose type cannot be inferred is an error, not silent any) and strictNullChecks (null and undefined are no longer assignable to every type, so they must be handled explicitly).

Emit and resolution flags decide the output shape: target sets the JavaScript language level, module/moduleResolution select the module system and how imports are resolved, esModuleInterop smooths CommonJS/ES-module default-import differences, skipLibCheck skips type-checking of .d.ts dependencies for speed, paths defines import aliases, and declaration/sourceMap emit .d.ts and source maps.

Example: a commented, recommended tsconfig.json

{
  "compilerOptions": {
    /* Type checking */
    "strict": true,            // enables the strict family (see below)
    "noImplicitAny": true,     // (part of strict) no silent 'any'
    "strictNullChecks": true,  // (part of strict) null/undefined are explicit
    "noUncheckedIndexedAccess": true, // arr[i] is T | undefined

    /* Output */
    "target": "ES2022",        // language level of emitted JS

Migrating JS to TS

Adopting TypeScript is incremental. Turn on allowJs so .js and .ts can coexist in one project, then add checkJs (or a per-file // @ts-check comment) to type-check JavaScript files using their JSDoc annotations. This lets the compiler surface real bugs in existing JavaScript before you rename anything.

Next, rename files from .js to .ts one module at a time, letting inference and any existing JSDoc types carry over, and replace JSDoc type comments with real TypeScript annotations as you go. Start with strict off (or only noImplicitAny) and ratchet strictness up file by file — many teams keep a lenient baseline and tighten it as coverage grows.

Example: BEFORE — JavaScript with // @ts-check and JSDoc

// total.js
// @ts-check

/**
 * @param {{ price: number, qty: number }[]} items
 * @returns {number}
 */
function total(items) {
  return items.reduce((sum, item) => sum + item.price * item.qty, 0);
}

module.exports = { total };
Result: With // @ts-check, the compiler reads the JSDoc and type-checks this JavaScript. Calling total([{ price: "9", qty: 2 }]) reports error TS2322: Type 'string' is not assignable to type 'number'. — a real bug caught without converting the file.

Example: AFTER — the same function as TypeScript

// total.ts
interface LineItem {
  price: number;
  qty: number;
}

export function total(items: LineItem[]): number {
  return items.reduce((sum, item) => sum + item.price * item.qty, 0);
}

// Usage:

  { price: 9.5, qty: 2 },
  { price: 4.0, qty: 1 },
];

Result: Type-checks cleanly. The JSDoc has become a named LineItem interface and a real signature; passing a wrongly typed item is now a compile error. The emitted JavaScript prints 23.

Note: Convert leaf modules (few dependents) first so downstream files inherit accurate types. Track progress with tsc --noEmit and treat the error count as a burn-down list.

Best practices

Good TypeScript is less about clever types and more about a few consistent habits. Prefer unknown to any for untyped inputs and narrow at the boundary where data enters your program — after that point everything downstream is safely typed. Model exclusive states as discriminated unions rather than loose string enums plus optional fields, so illegal combinations are unrepresentable. Avoid the non-null assertion ! unless you can point to the invariant that guarantees non-null; a real check or assertion function is safer and self-documenting.

Keep types close to the data they describe — colocate a type with its function or module rather than hoarding a central "types" dump — and let inference remove annotations that add no information. The patterns below compose into most day-to-day designs:

  • Discriminated unions — a kind/type tag plus an exhaustive switch with a never default makes missing cases a build error.
  • Conditional types with infer — extract and transform types (element types, return types, unwrapped promises) instead of hand-writing parallel type definitions.
  • Type guards & assertion functions — validate at boundaries (parsed JSON, API responses) and let narrowing propagate the proven type.
  • Branded types — attach a phantom tag to a primitive so distinct domain values cannot be interchanged, as below.

Example: branded types for domain safety

// A 'brand' distinguishes structurally identical primitives.
type Brand<T, B> = T & { readonly __brand: B };
type UserId = Brand<string, "UserId">;
type OrderId = Brand<string, "OrderId">;

function asUserId(s: string): UserId { return s as UserId; }

function loadUser(id: UserId): void { /* ... */ }

const uid = asUserId("u_123");
loadUser(uid);            // ok
loadUser("u_123");        // ERROR: plain string is not a UserId
const oid = "o_9" as OrderId;
loadUser(oid);            // ERROR: OrderId is not a UserId
Result: tsc reports error TS2345: Argument of type 'string' is not assignable to parameter of type 'UserId'. on loadUser("u_123") and a similar TS2345 for the OrderId call. Branding is erased at runtime, so the emitted JavaScript treats all three as plain strings.

Example: narrow at the boundary, unknown over any

interface ApiUser { id: number; name: string; }

function isApiUser(v: unknown): v is ApiUser {
  return (
    typeof v === "object" && v !== null &&
    typeof (v as Record<string, unknown>).id === "number" &&
    typeof (v as Record<string, unknown>).name === "string"
  );
}

// Parse returns 'unknown' — narrow before trusting it.
function parseUser(json: string): ApiUser {
  const data: unknown = JSON.parse(json);
  if (!isApiUser(data)) throw new Error("malformed user payload");
  return data; // narrowed to ApiUser here
}
console.log(parseUser('{"id":1,"name":"Ada"}').name); // "Ada"
Result: Type-checks cleanly. JSON.parse is treated as unknown and only becomes ApiUser after the guard, so no any leaks in. The emitted JavaScript prints Ada.

The DOM type library

The types for document, window, every element, and every browser event do not live in your code — they ship with TypeScript as lib.dom.d.ts. Which built-in libraries are loaded is governed by two tsconfig options. target sets the JavaScript language level and picks a default library set; lib overrides that set explicitly. A browser project typically uses "lib": ["ES2022", "DOM", "DOM.Iterable"]; a Node-only project omits "DOM" so that document is deliberately an error. Nothing in lib.dom.d.ts is emitted — it is pure ambient typing over APIs the runtime already provides.

Because these APIs are shared and generic, their return types are deliberately wide. document.querySelector("canvas") is typed Element | null, not HTMLCanvasElement, and canvas.getContext("2d") is CanvasRenderingContext2D | null. The generic querySelector<T> lets you assert the concrete element type, and the nullable results force you to choose between a non-null assertion (!, which you own the correctness of) and a real type guard (which the compiler verifies). Event handlers are typed through overloads on addEventListener: the string event name selects the precise event type passed to your listener.

Example: typing elements, querySelector generics, and a nullable context

// lib.dom.d.ts types these globals; no import needed with "DOM" in lib.

// loose.getContext(...) // ERROR: 'getContext' does not exist on 'Element'

// The generic form narrows to the concrete element type:
const canvas = document.querySelector<HTMLCanvasElement>("#board");
// canvas is HTMLCanvasElement | null

// getContext is itself nullable — the browser may not grant a 2d context:

ctx.fillRect(0, 0, 10, 10);          // and 'ctx' is possibly 'null' too

Example: non-null assertion vs a real type guard, plus a typed event handler

const canvas = document.querySelector<HTMLCanvasElement>("#board");

// Erased at runtime; a wrong promise becomes a real TypeError.

}

Modeling the domain with types

An Employee-Dashboard's data model is where types pay off first: a well-shaped Employee interface makes malformed records unrepresentable. Mark identity fields readonly so they cannot drift after construction, mark genuinely optional fields (like a photoUrl) with ?, and nest related structure (a contact block, an Organization reference) rather than flattening everything into one wide bag of loose strings.

Employment status is a closed set of mutually exclusive states, which is exactly what a union expresses. You can model it as a plain string-literal union or, when each state carries different data, as a discriminated union with a status tag. Compared with an enum — which emits a runtime object and reverse-maps numeric members — a string-literal union is erased entirely, is trivially serialisable, and narrows through a normal switch. Reach for an enum only when you need a named runtime value to iterate or pass around.

A generic, type-safe storage wrapper

localStorage is stringly-typed: getItem returns string | null and setItem takes a string, so every read is an untyped JSON.parse that hands you back any unless you intervene. A generic wrapper fixes this. Parameterising over the stored value type T lets get() return T | null, while modelling the raw parse result as unknown forces a validation step before the value is trusted. Constraining keys with keyof against a schema type turns a typo'd key into a compile error instead of a silent null.

Because the preview iframe runs sandboxed without allow-same-origin, touching localStorage throws a SecurityError there — which is the perfect excuse to build the resilience every real wrapper needs anyway: feature-detect the API, wrap access in try/catch, and fall back to an in-memory object typed against the same interface. On a genuine same-origin HTTPS page the real backend is selected and persists across reloads; the fallback keeps the app working everywhere else.

Example: generic function form over a typed key schema

// A schema maps each storage key to the type stored under it.
interface Schema {
  "dash:theme": "light" | "dark";
  "dash:lastEmployeeId": string;
  "dash:filters": { status: string; org: string };
}

// keyof Schema keeps callers from inventing keys; the value type
// is looked up per key via indexed access Schema[K].
function loadItem<K extends keyof Schema>(key: K): Schema[K] | null {
  const raw = storageGet(key);           // string | null (detected backend)
  if (raw === null) return null;
  const parsed: unknown = JSON.parse(raw); // parse result is 'unknown'
  return parsed as Schema[K];            // trusted only after your own check
}

function saveItem<K extends keyof Schema>(key: K, value: Schema[K]): void {
  storageSet(key, JSON.stringify(value));
}

const theme = loadItem("dash:theme");    // inferred: "light" | "dark" | null
saveItem("dash:theme", "dark");          // ok
saveItem("dash:theme", "blue");          // ERROR: not assignable
loadItem("dash:unknown");                // ERROR: not a key of Schema
Result: theme is inferred as "light" | "dark" | null. tsc reports error TS2345: Argument of type '"blue"' is not assignable to parameter of type '"light" | "dark"'. on the bad saveItem, and error TS2345 again for the "dash:unknown" key that is not in keyof Schema.

Example: generic class form with a feature-detected fallback

// A Store<T> over a single key, JSON-encoded, with a validator.
class Store<T> {
  private mem: string | null = null;     // in-memory fallback slot
  private readonly usable: boolean;

  constructor(
    private readonly key: string,
    private readonly validate: (v: unknown) => v is T
  ) {
    // Feature-detect a WORKING localStorage (may throw when sandboxed):
    let ok = false;
    try {
      ok = typeof localStorage !== "undefined";
      if (ok) { localStorage.setItem("__probe__", "1"); localStorage.removeItem("__probe__"); }
    } catch { ok = false; }
    this.usable = ok;
  }

  get(): T | null {
    const raw = this.usable ? localStorage.getItem(this.key) : this.mem;
    if (raw === null) return null;
    const parsed: unknown = JSON.parse(raw);   // narrow 'unknown' via validate
    return this.validate(parsed) ? parsed : null;
  }

  set(value: T): void {
    const raw = JSON.stringify(value);
    if (this.usable) localStorage.setItem(this.key, raw);
    else this.mem = raw;               // same interface, different backend
  }
}

const isTheme = (v: unknown): v is "light" | "dark" =>
  v === "light" || v === "dark";
const themeStore = new Store("dash:theme", isTheme);
themeStore.set("dark");
const current = themeStore.get();      // inferred: "light" | "dark" | null
Result: Type-checks cleanly. Hovering current shows the inferred type "light" | "dark" | null: the class threads T from the validator's predicate through get(). When localStorage is unavailable, usable is false and the identical interface is served from the in-memory mem slot.

Typing async & browser APIs

Older browser APIs predate Promises and use success/error callbacks — navigator.geolocation.getCurrentPosition(onOk, onErr) being the canonical example. Promisifying it means wrapping the callback pair in a new Promise<GeolocationPosition>, resolving in the success callback and rejecting with the typed GeolocationPositionError in the error callback. The resulting Promise<GeolocationPosition> is await-able and its rejection reason carries the API's own error code constants.

Streaming APIs such as WebSocket deliver messages as opaque strings; typing the protocol is what makes them safe. Model each server message as a member of a discriminated union keyed on a type field, write a user-defined type guard (msg is EmployeeAddedMsg) to narrow a parsed payload, and dispatch with a switch whose default assigns to never for compile-time exhaustiveness. Feature detection itself can be typed as a mapped/Record type so the capability object is complete and each flag is a boolean.

Example: promisifying geolocation with a typed error

// GeolocationPosition and GeolocationPositionError come from lib.dom.d.ts.
function getPosition(
  options?: PositionOptions
): Promise<GeolocationPosition> {
  return new Promise((resolve, reject) => {
    if (!("geolocation" in navigator)) {
      reject(new Error("geolocation unsupported"));
      return;
    }
    navigator.geolocation.getCurrentPosition(
      (pos: GeolocationPosition) => resolve(pos),
      (err: GeolocationPositionError) => reject(err), // typed error
      options
    );
  });
}

async function whereAmI(): Promise<string> {
  try {
    const pos = await getPosition({ timeout: 5000 });
    return `${pos.coords.latitude},${pos.coords.longitude}`;
  } catch (e) {
    // GeolocationPositionError has numeric 'code' constants:
    if (e instanceof GeolocationPositionError && e.code === e.PERMISSION_DENIED) {
      return "permission denied";
    }
    return "unavailable";
  }
}
Result: Type-checks cleanly. getPosition is typed Promise<GeolocationPosition>, so await-ing it yields a GeolocationPosition and pos.coords.latitude is a number. The catch narrows the rejection to GeolocationPositionError before reading its code, matching the PERMISSION_DENIED constant.

Example: a discriminated WebSocket protocol with an exhaustive switch

// The message protocol as a discriminated union keyed on 'type':
interface EmployeeAddedMsg { type: "employee_added"; id: string; name: string; }
interface EmployeeRemovedMsg { type: "employee_removed"; id: string; }
interface HeartbeatMsg { type: "heartbeat"; at: number; }
type ServerMsg = EmployeeAddedMsg | EmployeeRemovedMsg | HeartbeatMsg;

// User-defined type guard for one member:
function isEmployeeAdded(m: ServerMsg): m is EmployeeAddedMsg {
  return m.type === "employee_added";
}

function handle(msg: ServerMsg): string {
  switch (msg.type) {
    case "employee_added":   return `+ ${msg.name} (${msg.id})`;
    case "employee_removed": return `- ${msg.id}`;
    case "heartbeat":        return `beat @ ${msg.at}`;
    default: {
      // Exhaustiveness: a new member not handled here fails to compile.
      const _never: never = msg;
      return _never;
    }
  }
}

// Feature detection typed as a Record/mapped capability object:
type Capability = "webSocket" | "geolocation" | "localStorage";
const caps: Record<Capability, boolean> = {
  webSocket: typeof WebSocket !== "undefined",
  geolocation: "geolocation" in navigator,
  localStorage: (() => { try { return typeof localStorage !== "undefined"; } catch { return false; } })(),
};
Result: Type-checks cleanly with all three cases handled, so msg is never at the default and the assignment succeeds. Add a { type: "employee_updated"; … } member to ServerMsg without a matching case and tsc reports error TS2322: Type 'EmployeeUpdatedMsg' is not assignable to type 'never'. at const _never: never = msg. The Record<Capability, boolean> annotation forces every capability flag to be present.

Generic API response type

Fetching an employee record from a backend crosses a trust boundary: the network can return data, an HTTP error, or nothing at all. Modelling that as ApiResponse<T> — a discriminated generic union whose ok flag tags the success and failure members — makes the two outcomes impossible to confuse. The success member carries a typed data: T; the failure member carries an error: string. Because ok is a literal-typed discriminant, narrowing on res.ok tells the compiler that res.data exists and is a T in the true branch, and that res.error exists in the false branch.

The generic async wrapper api<T> centralises the try/catch, the status check, and the JSON parse so every call site inherits the same discipline. Constraining the type parameter with <T extends object> restricts the payload to object shapes (the employee records this dashboard exchanges), and a default type parameter (<T extends object = Record<string, unknown>>) supplies a fallback when the caller does not specify one. Every path returns a fully-formed ApiResponse<T>, so a call site can never receive a half-populated object.

Example: the discriminated ApiResponse union and the typed wrapper

// A discriminated generic union: 'ok' is the literal-typed discriminant.
type ApiResponse<T> =
  | { ok: true;  status: number; data: T }
  | { ok: false; status: number; error: string };

interface Employee { id: string; name: string; }

// Generic constraint (<T extends object>) + default type parameter.
async function api<T extends object = Record<string, unknown>>(
  url: string
): Promise<ApiResponse<T>> {
  try {
    const res = await fetch(url);
    if (!res.ok) {
      return { ok: false, status: res.status, error: res.statusText };
    }
    const data = (await res.json()) as T; // parse result asserted to T
    return { ok: true, status: res.status, data };
  } catch (e) {
    // Network/parse failure never leaks out as a rejected promise:
    return { ok: false, status: 0, error: (e as Error).message };
  }
}

// The default type parameter applies when none is given:
const anyShape = api("/api/health");          // Promise<ApiResponse<Record<string, unknown>>>
const typed    = api<Employee>("/api/emp/1"); // Promise<ApiResponse<Employee>>
console.log(typeof anyShape, typeof typed);
Result: Type-checks cleanly. api<Employee>(…) resolves to Promise<ApiResponse<Employee>>; the bare api("/api/health") falls back to the default and is Promise<ApiResponse<Record<string, unknown>>>. Passing a non-object type argument, e.g. api<string>(url), fails: error TS2344: Type 'string' does not satisfy the constraint 'object'.

Example: a consumer that narrows on res.ok

async function loadEmployee(id: string): Promise<string> {
  const res = await api<Employee>(`/api/emp/${id}`);
  if (res.ok) {
    // In this branch res is the success member:
    //   res.data is Employee, res.error does NOT exist.
    return `Loaded ${res.data.name} (HTTP ${res.status})`;
  }
  // In the else branch res is the failure member:
  //   res.error is string, res.data does NOT exist.
  return `Failed: ${res.error} (HTTP ${res.status})`;
}

// Reaching for the wrong member is a compile error:

  const res = await api<Employee>(`/api/emp/${id}`);
  return res.data.name; // ERROR: 'data' does not exist on the failure member
}
Result: loadEmployee type-checks: inside if (res.ok) the compiler narrows res to { ok: true; status: number; data: Employee }, so res.data.name is a string. broken fails with error TS2339: Property 'data' does not exist on type 'ApiResponse<Employee>'. because data is absent from the un-narrowed union.

Advanced types for the employee model

Two type-level tools shape the employee model. First, union types pin a field to a closed set of values: a string-literal union Role lists the legal roles, and a discriminated union of employee variants lets each role carry its own extra fields — a manager has directReports, a contractor has an agency and an endDate — so an employee record can never hold a field that its role does not permit. Second, mapped types derive PATCH shapes from the base Employee without restating its fields: the built-in Partial<Employee> makes every property optional for a PATCH DTO, and a hand-written { [K in keyof Employee]?: Employee[K] } shows exactly the mechanism Partial is built from.

Mapped-type modifiers give precise control: +?/-? add or strip optionality, readonly/-readonly add or strip immutability, and an as clause remaps keys (for example prefixing them). A recursive DeepPartial makes optionality reach into nested objects like contact, and the derived DTO utilities Pick, Omit, Required, and Readonly carve purpose-built shapes out of the one canonical Employee definition.

Example: a Role literal union and a discriminated employee union

// (a) A closed string-literal union of roles.
type Role = "admin" | "manager" | "engineer" | "contractor";

interface Contact { email: string; phone?: string; }

interface EmployeeBase {
  readonly id: string;
  name: string;
  contact: Contact;
}

// A discriminated union keyed on 'role': each variant adds its own fields.
type Employee =
  | (EmployeeBase & { role: "admin"; permissions: string[] })
  | (EmployeeBase & { role: "manager"; directReports: string[] })
  | (EmployeeBase & { role: "engineer"; stack: string[] })
  | (EmployeeBase & { role: "contractor"; agency: string; endDate: string });

function describe(e: Employee): string {
  switch (e.role) {
    case "admin":      return `${e.name}: ${e.permissions.length} perms`;
    case "manager":    return `${e.name}: ${e.directReports.length} reports`;
    case "engineer":   return `${e.name}: ${e.stack.join("/")}`;
    case "contractor": return `${e.name}: via ${e.agency} until ${e.endDate}`;
  }
}

const mgr: Employee = {

Example: Partial vs a hand-written patch, and mapped-type modifiers

interface FlatEmployee {
  readonly id: string;
  name: string;
  role: Role;
  salary?: number;
}

// (b) Built-in Partial for a PATCH DTO: every field optional.
type EmployeePatchBuiltin = Partial<FlatEmployee>;

// Hand-written equivalent -- this IS how Partial is defined.
type EmployeePatch = { [K in keyof FlatEmployee]?: FlatEmployee[K] };

// Modifiers: '-?' makes all fields required, '-readonly' removes readonly.
type FullyRequired = { -readonly [K in keyof FlatEmployee]-?: FlatEmployee[K] };
// '+readonly'/'+?' explicitly ADD the modifiers (the '+' is optional syntax).
type FrozenOptional = { +readonly [K in keyof FlatEmployee]+?: FlatEmployee[K] };

// Key remapping via 'as': prefix every key to build a form-field name map.
type FormFields<T> = { [K in keyof T as `field_${string & K}`]: T[K] };

DOM typing for forms and events

An employee form is where unknown DOM values meet a typed model. lib.dom.d.ts types the pieces: a <form> is an HTMLFormElement, and its form.elements is an HTMLFormControlsCollection whose namedItem("field") returns the wide Element | RadioNodeList | null. Because the collection cannot know which concrete control lives under a name, you cast to the specific element type — HTMLInputElement, HTMLSelectElement, HTMLTextAreaElement — before reading .value. The generic querySelector<HTMLInputElement>("#name") does the same at the query, returning HTMLInputElement | null.

Events are typed by their precise class. A submit handler receives a SubmitEvent, calls e.preventDefault(), and reads e.currentTarget — which the addEventListener overload for "submit" types as HTMLFormElement when the listener is attached to a form. An input handler narrows e.target (typed EventTarget | null) with an instanceof HTMLInputElement check before touching .value. Parsing a submission goes through FormData, whose get() returns FormDataEntryValue (that is, string | File); a typed parse function coerces those entries into an Employee-shaped object.

Runtime validation with types

Types are erased at compile time, so nothing about an Employee annotation survives to guard a value that arrives at runtime — a fetch body, a JSON.parse result, a form submission. Validation is the code that re-establishes, at runtime, the guarantee the type describes. Two type-level building blocks give it a typed vocabulary: a Result<T, E> discriminated union that carries either a validated value: T or an errors: E[] array, and a FieldErrors<T> mapped type — Partial<Record<keyof T, string>> — that maps each field name to an optional message.

A user-defined type guard isEmployee(x: unknown): x is Employee answers "is this shape valid?" and narrows in the true branch; an assertion function assertEmployee(x): asserts x is Employee answers the same question by throwing on failure, narrowing everything after the call. Both take unknown so they can sit exactly at the boundary — right after JSON.parse — before any code trusts the payload. Small per-field validators (each returning a message or null) then compose into a validate(form): Result<Employee, string> that collects every error rather than failing on the first.

Example: Result, FieldErrors, a type guard, and an assertion function

// A discriminated Result: success carries a value, failure an error list.
type Result<T, E = string> =
  | { ok: true; value: T }
  | { ok: false; errors: E[] };

// FieldErrors<T>: each key of T maps to an optional message.
type FieldErrors<T> = Partial<Record<keyof T, string>>;

interface Employee {
  name: string;
  role: "engineer" | "manager";
  age: number;
}

// A user-defined type guard validating an UNKNOWN shape.
function isEmployee(x: unknown): x is Employee {
  if (typeof x !== "object" || x === null) return false;
  const o = x as Record<string, unknown>;
  return (
    typeof o.name === "string" &&
    (o.role === "engineer" || o.role === "manager") &&
    typeof o.age === "number"
  );
}

// An assertion function: narrows by throwing instead of returning boolean.
function assertEmployee(x: unknown): asserts x is Employee {
  if (!isEmployee(x)) throw new Error("payload is not an Employee");
}

// At the boundary: JSON.parse yields 'unknown', validate before trusting.
function fromJson(json: string): Employee {
  const data: unknown = JSON.parse(json);
  assertEmployee(data);   // after this line, data is Employee
  return data;            // no cast needed; already narrowed
}

const errs: FieldErrors<Employee> = { age: "must be positive" }; // name/role omitted OK
console.log(fromJson('{"name":"Ada","role":"engineer","age":30}').name, errs);
Result: Type-checks cleanly. Inside fromJson, data starts as unknown and is narrowed to Employee by assertEmployee, so return data needs no cast. FieldErrors<Employee> resolves to { name?: string; role?: string; age?: string }, so the partial { age: "…" } is accepted. The emitted JavaScript prints Ada { age: 'must be positive' }.

Example: per-field validators composed into validate()

interface EmployeeInput { name: string; role: string; age: number; }

// Each validator returns an error message or null (valid).
const validators: { [K in keyof Employee]: (v: EmployeeInput[K]) => string | null } = {
  name: (v) => (v.trim().length >= 2 ? null : "name must be at least 2 chars"),
  role: (v) => (v === "engineer" || v === "manager" ? null : "role is invalid"),
  age:  (v) => (v >= 18 && v <= 100 ? null : "age must be 18-100"),
};

// Compose them; collect ALL errors, not just the first.
function validate(input: EmployeeInput): Result<Employee, string> {
  const errors: string[] = [];
  for (const key of Object.keys(validators) as (keyof Employee)[]) {
    const message = validators[key](input[key] as never);
    if (message) errors.push(message);
  }
  return errors.length
    ? { ok: false, errors }
    : { ok: true, value: input as Employee };
}

const good = validate({ name: "Ada", role: "engineer", age: 30 });
const bad  = validate({ name: "A", role: "wizard", age: 12 });
console.log(good.ok, bad.ok);
Result: Type-checks cleanly. validate returns Result<Employee, string>; narrowing on .ok exposes value in the success branch and errors in the failure branch. At runtime good.ok is true and bad.ok is false with three collected messages, so the emitted JavaScript prints true false.

Decorators

A decorator is a function evaluated at declaration time — when the class is defined, not when it is instantiated — that observes or replaces the declaration it is attached to. TypeScript has two incompatible decorator systems, and which one is active depends on a compiler flag, so knowing which you are writing against is essential. (A) Legacy "experimental" decorators are enabled by "experimentalDecorators": true and predate the language standard; this is the system Angular, NestJS, and TypeORM use. (B) TC39 standard decorators are the Stage-3 proposal shipped in TypeScript 5.0+, and are the default whenever experimentalDecorators is off. The two have different function signatures and are not interchangeable — code written for one will not run under the other.

In the legacy system the decorator's parameters depend on what it decorates: a method decorator receives (target, propertyKey, descriptor) and may return a replacement PropertyDescriptor; a property decorator receives (target, propertyKey) with no descriptor; a class decorator receives the constructor; a parameter decorator receives (target, propertyKey, parameterIndex). With "emitDecoratorMetadata": true and import "reflect-metadata", the compiler also emits design:type / design:paramtypes metadata that frameworks read via Reflect.getMetadata — the mechanism behind Angular's and NestJS's dependency injection.

In the standard system every decorator has the uniform signature (value, context), where context is an object { kind, name, addInitializer, metadata, … }. A method decorator's value is the method function (return a replacement to wrap it); a field decorator's value is undefined and it may return an initializer (initialValue) => newValue; context.addInitializer(fn) registers per-instance setup; context.metadata is a shared object for cross-decorator communication. The new accessor keyword creates an auto-accessor a decorator can wrap with a custom getter/setter.

Example: tsconfig for each system

  "compilerOptions": {
    "experimentalDecorators": true,   // enable the legacy system
    "emitDecoratorMetadata": true,    // emit design:type metadata

  }
}

// (B) TC39 STANDARD decorators -- the default when the flag is OFF:
{
  "compilerOptions": {
    "experimentalDecorators": false,  // (or simply omit it)
    "target": "ES2022"
    // No reflect-metadata; context.metadata is built in.
  }

Result: These are configuration files, so no JavaScript is emitted from them. With experimentalDecorators: true the compiler accepts the three-argument legacy signatures and (given emitDecoratorMetadata) emits Reflect.metadata(...) calls; with the flag off it accepts only the (value, context) standard signatures and rejects a legacy-shaped decorator with error TS1206: Decorators are not valid here. or a signature-mismatch error.

Classes in depth

TypeScript layers a full nominal-feeling class model on top of JavaScript's runtime classes. The access modifiers public (the default), private, and protected are compile-time visibility rules — they are erased at emit, so a TS-private field is still reachable at runtime via bracket access. The ECMAScript #private field is different: it is enforced by the runtime and truly inaccessible from outside the class. readonly forbids reassignment after construction. Parameter properties — a modifier on a constructor parameter — declare and initialise a field in one stroke, a TypeScript-only shorthand with no JavaScript equivalent.

An abstract class cannot be instantiated and may declare abstract members that subclasses must implement; unlike an interface it can also carry concrete state and behaviour. implements checks a class against one or more interface shapes without inheriting anything; extends inherits implementation from exactly one base. The override keyword (plus noImplicitOverride) makes an intent to replace a base method explicit, so a typo in the method name becomes an error instead of an accidental new method. A method that returns this yields a polymorphic this type, which keeps fluent chains correctly typed through subclasses.

Example: abstract base, concrete subclasses, and TS2511 on instantiation

  }
}

class Circle extends Shape {
  constructor(private readonly r: number) { super(); }
  area(): number { return Math.PI * this.r ** 2; }
}
class Square extends Shape {
  constructor(private readonly side: number) { super(); }
  area(): number { return this.side ** 2; }
}

const shapes: Shape[] = [new Circle(2), new Square(3)];
console.log(shapes.map(s => s.describe()));

const bad = new Shape(); // ERROR: cannot create an instance of an abstract class

  // parameter declare AND assign the field in one line.
  constructor(
    public readonly id: string,         // public + readonly
    protected balance: number,          // visible to subclasses only
    pin: string
  ) {
    this.#pin = pin;
  }
  check(pin: string): boolean { return this.#pin === pin; } // in-class access OK
}

class Savings extends Account {

    this.balance *= 1 + rate;           // 'protected' is reachable in subclass
  }

Result: tsc reports three errors: TS2540 on a.id = "ac_2", TS2445: Property 'balance' is protected and only accessible within class 'Account' and its subclasses., and TS18013: Property '#pin' is not accessible outside class 'Account' because it has a private identifier. With those lines removed the emitted JavaScript prints ac_2 true. Note that a TS-private field would be silently readable at runtime via a["balance"], whereas #pin genuinely throws.

Example: implements multiple interfaces, accessors, and override + noImplicitOverride

interface Named { readonly name: string; }
interface Serializable { toJSON(): string; }

// A class can implement SEVERAL interfaces (structural checks) while
// extending at most one base.
class Person implements Named, Serializable {
  #first: string;
  constructor(first: string, private last: string) { this.#first = first; }
  // Accessor pair: 'name' looks like a field but runs code.
  get name(): string { return `${this.#first} ${this.last}`; }
  set name(full: string) { [this.#first, this.last] = full.split(" "); }
  toJSON(): string { return JSON.stringify({ name: this.name }); }
}

class Employee extends Person {
  constructor(name: string, public title: string) {
    const [f, l] = name.split(" "); super(f, l ?? "");
  }
  // 'override' is REQUIRED here under noImplicitOverride:
  override toJSON(): string {
    return JSON.stringify({ name: this.name, title: this.title });
  }
  // A typo'd name is caught because 'override' finds no base member:
  override tojson(): string { return ""; } // ERROR TS4113
}
Result: With "noImplicitOverride": true, the correctly-spelled override toJSON type-checks, but override tojson reports error TS4113: This member cannot have an 'override' modifier because it is not declared in the base class 'Person'. Omitting override on the real toJSON would instead report TS4114: This member must have an 'override' modifier because it overrides a member in the base class. The get/set name accessors present as an ordinary property while running code on read and write.

Example: polymorphic this for a fluent builder, static members, and generic-static rules

// Returning 'this' (not the concrete class) keeps subclass chains typed.
class QueryBuilder {
  protected parts: string[] = [];
  where(clause: string): this { this.parts.push(`WHERE ${clause}`); return this; }
  build(): string { return this.parts.join(" "); }

  static create(): QueryBuilder { return new QueryBuilder(); } // static factory
  static readonly dialect = "ansi";                            // static field
  static { console.log("QueryBuilder loaded"); }               // static block
}

class PagedQuery extends QueryBuilder {
  limit(n: number): this { this.parts.push(`LIMIT ${n}`); return this; }
}

// Because 'where' returns 'this', the chain stays a PagedQuery and 'limit'
// is still available afterwards:
const q = new PagedQuery().where("active = 1").limit(10).build();
console.log(q);

class Box<T> {
  constructor(public value: T) {}
  // ERROR: a class type parameter cannot be used in a STATIC member.
  static empty(): T { return undefined as any; } // TS2302
}

Because where returns this, the chain stays typed as PagedQuery, so .limit(10) type-checks and the run prints WHERE active = 1 LIMIT 10. Type assertions & casting

A type assertion tells the compiler to treat a value as a specific type. The two syntaxes — value as T and the older <T>value — are equivalent, but the angle-bracket form is banned in .tsx files because it collides with JSX, so as is the portable choice. An assertion is not a runtime cast: types are erased, so the emitted JavaScript is unchanged and a wrong assertion becomes a latent bug that surfaces as a crash later. The compiler only permits an assertion between types that overlap on the assignability lattice; to bridge two unrelated types you must launder through unknown with a double assertion value as unknown as T — a deliberate smell.

Two related operators fill specific gaps. A const assertion (as const) narrows literals to their exact values, makes object/array members readonly, and turns arrays into readonly tuples. The non-null assertion ! drops null/undefined from a value's type, and the definite-assignment assertion x!: T promises a field is initialised even though the compiler cannot see where. All of these trust you; unlike a type guard or runtime validation, none of them verify anything at runtime. When you want a check that keeps a value's narrow inferred type instead of overriding it, reach for satisfies (covered in its own section) rather than as.

Example: as const literal narrowing and deep readonly vs the plain object

// WITHOUT as const: values widen to their base types, object is mutable.
const loose = { method: "GET", retries: 3, tags: ["a", "b"] };
// typeof loose = { method: string; retries: number; tags: string[] }
loose.method = "POST"; // allowed

// WITH as const: literals stay literal, everything becomes readonly.
const strict = { method: "GET", retries: 3, tags: ["a", "b"] } as const;
// typeof strict = {
//   readonly method: "GET"; readonly retries: 3;
//   readonly tags: readonly ["a", "b"];
// }
strict.method = "POST"; // ERROR TS2540: readonly
strict.tags.push("c");  // ERROR TS2339: push does not exist on readonly tuple

type Method = typeof strict.method; // exactly "GET", not string
console.log(loose.method, strict.method);
Result: tsc reports error TS2540: Cannot assign to 'method' because it is a read-only property. and error TS2339: Property 'push' does not exist on type 'readonly ["a", "b"]'. The type Method resolves to the literal "GET". With the error lines removed the emitted JavaScript (the as const is erased) prints GET GET.

Example: a wrong assertion type-checks but crashes at runtime

// The compiler believes you; the runtime does not.
const value: unknown = "42";           // it is really a string
const n = value as number;             // no error: 'as' overrides the checker
console.log(n.toFixed(2));             // typed OK, but 'n' is a string at runtime

// Contrast: a validated conversion cannot lie, because it checks.
function toNumber(v: unknown): number {
  if (typeof v === "number") return v;
  const parsed = Number(v);
  if (Number.isNaN(parsed)) throw new Error(`not a number: ${String(v)}`);
  return parsed;
}
console.log(toNumber("42").toFixed(2)); // "42.00" — actually a number
Result: The assertion form type-checks (the compiler trusts as number) but throws at runtime: TypeError: n.toFixed is not a function, because n is still the string "42". The validated toNumber("42") path prints 42.00 because it actually converts. An assertion moves a bug from compile time to run time; a guard or validator prevents it.

Example: the double assertion as unknown as T for unrelated types

interface Cat { meow(): void; }
interface Dog { bark(): void; }

const cat: Cat = { meow() { console.log("meow"); } };

// Direct assertion between unrelated shapes is rejected:
const wrong = cat as Dog; // ERROR TS2352: neither type sufficiently overlaps

// The escape hatch: launder through 'unknown'. Compiles, but you now OWN
// the correctness — nothing verifies 'cat' actually has 'bark'.
const forced = cat as unknown as Dog;
forced.bark(); // typed OK; throws at runtime because there is no 'bark'
Result: The direct cat as Dog reports error TS2352: Conversion of type 'Cat' to type 'Dog' may be a mistake because neither type sufficiently overlaps with the other. If this was intentional, convert the expression to 'unknown' first. The double assertion cat as unknown as Dog compiles, but calling forced.bark() throws TypeError: forced.bark is not a function. Treat as unknown as T as a red flag reserved for genuinely dynamic bridges (e.g. test doubles), never routine code.

Example: non-null assertion vs a verified null check, and definite assignment

// Both compile; only one is verified by the compiler.
const el = document.querySelector("#save");

// (A) Non-null assertion: YOU promise it exists. Erased; a wrong promise
//     becomes 'Cannot read properties of null'.
el!.addEventListener("click", () => {});

// (B) A real check the compiler proves; no '!' needed afterwards.
if (el) el.addEventListener("click", () => {});

// Definite-assignment assertion: 'ready!' promises a later assignment
// the compiler cannot see (e.g. set by an init() called in the ctor).
class Widget {
  private ready!: boolean;      // no initializer, no error under strict
  constructor() { this.init(); }
  private init(): void { this.ready = true; }
}
console.log(new Widget());
Result: Both DOM forms type-check, but only (B) is guaranteed safe by the compiler — (A)'s el! throws TypeError at runtime if no #save element exists. The definite-assignment ready!: boolean suppresses error TS2564: Property 'ready' has no initializer and is not definitely assigned in the constructor. that strict mode would otherwise raise — a promise the compiler again cannot verify.

Note: Every assertion is a place where you have overruled the type checker, so it is where bugs hide. Grep for as and ! in review: each one should be justified by an invariant you can name. Prefer a guard, a validator, or satisfies whenever one of them can express the same intent while keeping verification on the compiler's side.

The utility type catalog

Beyond the object utilities shown earlier (Partial, Required, Readonly, Pick, Omit, Record), the standard library ships a full set of utilities for functions, unions, and strings — every one built from the primitives already taught: keyof, indexed access, mapped types, and conditional types with infer. Knowing they exist prevents you from hand-rolling parallel types that drift out of sync with their source.

The function utilities extract pieces of a callable or class: Parameters<F> yields a function's argument tuple, ReturnType<F> its return type, ConstructorParameters<C> a class's constructor arguments, and InstanceType<C> the type it constructs. Awaited<T> unwraps Promise nesting (the vetted, thenable-aware version of the hand-written Unwrap from the conditional-types section). The union utilities filter union members: Extract<T, U> keeps members assignable to U, Exclude<T, U> drops them, and NonNullable<T> removes null and undefined. The string utilities Uppercase, Lowercase, Capitalize, and Uncapitalize transform string-literal types and pair naturally with template literal types.

Example: Parameters, ReturnType, and reusing an existing function's argument types

function createEmployee(name: string, role: "eng" | "mgr", age: number) {
  return { name, role, age, id: crypto.randomUUID() };
}

// Derive the argument tuple and the return shape from the function itself.
type CreateArgs = Parameters<typeof createEmployee>;   // [string, "eng"|"mgr", number]
type Employee   = ReturnType<typeof createEmployee>;   // { name; role; age; id }

// A wrapper that forwards the SAME arguments without restating them:
function loggedCreate(...args: CreateArgs): Employee {
  console.log("creating", args[0]);
  return createEmployee(...args);
}

const e = loggedCreate("Ada", "eng", 30);
console.log(e.name, e.role);
// A wrong argument is still checked against the derived tuple:
loggedCreate("Ada", "wizard", 30); // ERROR TS2345
Result: CreateArgs resolves to [name: string, role: "eng" | "mgr", age: number] and Employee to the object return shape, so loggedCreate stays in lockstep with createEmployee automatically. The last call fails: error TS2345: Argument of type '"wizard"' is not assignable to parameter of type '"eng" | "mgr"'.

Example: Awaited over an async function, plus ConstructorParameters/InstanceType

async function fetchUser(id: string): Promise<{ id: string; name: string }> {
  return { id, name: "Ada" };
}

// Awaited unwraps the Promise; combine with ReturnType for the resolved value.
type Resolved = Awaited<ReturnType<typeof fetchUser>>; // { id: string; name: string }
type Deep = Awaited<Promise<Promise<number>>>;           // number (recursive)

class Repository {
  constructor(public url: string, public retries: number) {}
  find(id: string) { return id; }
}
type RepoArgs = ConstructorParameters<typeof Repository>; // [string, number]
type Repo     = InstanceType<typeof Repository>;          // Repository

function makeRepo(...args: RepoArgs): Repo { return new Repository(...args); }
console.log(makeRepo("/api", 3).url);
Result: Type-checks cleanly. Resolved is { id: string; name: string } (the value inside the promise), Deep flattens to number, RepoArgs is [url: string, retries: number], and Repo is the Repository instance type. The emitted JavaScript prints /api. Unlike a hand-written unwrapper, Awaited also understands custom then-ables.

Example: Extract, Exclude, and NonNullable filtering unions

type Role = "admin" | "manager" | "engineer" | "contractor";
type Action =
  | { type: "save"; payload: string }
  | { type: "delete"; id: string }
  | { type: "noop" };

// Exclude removes members assignable to the second type:
type NonAdmin = Exclude<Role, "admin">;        // "manager" | "engineer" | "contractor"
// Extract keeps only members assignable to the second type:
type SaveAction = Extract<Action, { type: "save" }>; // { type: "save"; payload: string }
// NonNullable strips null and undefined:
type MaybeName = string | null | undefined;
type Name = NonNullable<MaybeName>;             // string

const roles: NonAdmin[] = ["manager", "engineer"];
function handleSave(a: SaveAction): string { return a.payload; }
console.log(roles, handleSave({ type: "save", payload: "row" }));
Result: Type-only resolution: NonAdmin drops "admin", SaveAction narrows the action union to its "save" member (so a.payload is available), and Name collapses to string. Assigning "admin" into a NonAdmin[] would fail with TS2322. The emitted JavaScript prints [ 'manager', 'engineer' ] row.

Example: string utilities and how a utility is built from primitives

type Event = "click" | "hover";
type Handler = `on${Capitalize<Event>}`; // "onClick" | "onHover"
type Loud = Uppercase<"get" | "post">;   // "GET" | "POST"
type Soft = Uncapitalize<"UserId">;      // "userId"

// Every "new" utility is expressible from the primitives already taught.
// Here is how Exclude and NonNullable are actually defined in lib.es5.d.ts:
type MyExclude<T, U> = T extends U ? never : T;        // distributive conditional
type MyNonNullable<T> = T & {};                        // intersection trick
type MyReturnType<F> = F extends (...a: any) => infer R ? R : never; // infer

type A = MyExclude<"a" | "b" | "c", "b">;              // "a" | "c"
type B = MyNonNullable<string | null>;                 // string
const handlers: Handler[] = ["onClick", "onHover"];
console.log(handlers);
Result: Type-only: Handler is "onClick" | "onHover", Loud is "GET" | "POST", and Soft is "userId". The hand-written MyExclude, MyNonNullable, and MyReturnType resolve identically to the built-ins, demonstrating that the whole catalog is sugar over conditional types, infer, and intersections. The emitted JavaScript prints [ 'onClick', 'onHover' ].

Note: Derive types from their source with these utilities instead of duplicating them. Parameters<typeof fn>, ReturnType<typeof fn>, and Awaited<…> keep a wrapper, a mock, or a cache entry automatically correct as the source function changes — a copy-pasted signature does not.

Enums in depth (and the union alternative)

An enum is one of the few TypeScript constructs that emits real runtime code. A numeric enum auto-increments from 0 (or from an explicit value) and builds a runtime object with a reverse mapping: both Direction.East → 1 and Direction[1] → "East" exist. A string enum has no reverse map and serialises to its own string, which makes it far safer across JSON, URL, and storage boundaries. A const enum is fully inlined at every use site and emits no object at all — but it is incompatible with isolatedModules/single-file transpilers (esbuild, swc, Babel) because inlining needs whole-program knowledge, and erasableSyntaxOnly rejects it outright.

Heterogeneous enums (mixing string and numeric members) are legal but confusing and should be avoided. In modern code the common alternative to an enum is a frozen object plus a derived union: const Status = { … } as const paired with type Status = typeof Status[keyof typeof Status]. This gives you named runtime values and a plain string-literal union, with no reverse-map object and no const enum tooling caveats. Reach for a real enum only when you specifically want its nominal-ish behaviour (an enum member is not freely interchangeable with its bare literal) or need to iterate the runtime object.

Example: numeric enum reverse mapping vs a const enum that inlines to nothing

// Numeric enum: emits a runtime object WITH a reverse map.
enum Direction { North, East, South, West } // 0,1,2,3
console.log(Direction.East);   // 1
console.log(Direction[1]);     // "East"  <- reverse mapping

// Emitted JS (roughly):
//   var Direction;
//   (function (D) { D[D["North"]=0]="North"; D[D["East"]=1]="East"; ... })
//   (Direction || (Direction = {}));

// const enum: NO object is emitted; each use is replaced by the literal.
const enum Level { Low, Medium, High }
const chosen = Level.High;     // emitted JS is literally: const chosen = 2;
console.log(chosen);           // 2
Result: The numeric enum prints 1 then "East" and leaves a real reverse-mapped object in the bundle. The const enum emits no Level object — Level.High is compiled to the bare literal 2, so console.log(chosen) prints 2 with zero runtime footprint for the enum itself.

Example: string enum at a serialization boundary vs a numeric enum

// String enum: each member IS its string. No reverse map.
enum Status { Active = "active", OnLeave = "on_leave", Terminated = "terminated" }

const payload = JSON.stringify({ status: Status.OnLeave });
console.log(payload);          // {"status":"on_leave"} — self-describing

// Numeric enum across the same boundary serialises to an opaque number:
enum NStatus { Active, OnLeave, Terminated }
console.log(JSON.stringify({ status: NStatus.OnLeave })); // {"status":1}
// A consumer receiving 1 must know the enum's order to decode it — brittle.

console.log(Status.OnLeave, (Status as Record<string, string>)["OnLeave"]);
Result: The string enum serialises to {"status":"on_leave"} — legible and stable even if members are reordered. The numeric enum serialises to {"status":1}, which is meaningless to any consumer that does not share the exact member ordering, and reordering members silently changes the wire format. Prefer string enums (or a string-literal union) at every boundary that leaves the program.

Example: the recommended as-const object + derived union

// A frozen object gives named runtime values; the derived type gives a union.
const Status = {
  Active: "active",
  OnLeave: "on_leave",
  Terminated: "terminated",
} as const;

// The value type is the union of the object's value types:
type Status = typeof Status[keyof typeof Status]; // "active"|"on_leave"|"terminated"

function label(s: Status): string {
  return s.replace("_", " ");
}

console.log(Status.OnLeave);        // "on_leave" — a real runtime value
console.log(label("active"));       // plain literals are assignable, unlike enums
label("retired");                   // ERROR TS2345: not a Status
Result: Status (the type) resolves to "active" | "on_leave" | "terminated". Unlike an enum, a bare literal like "active" is directly assignable to the union, so label("active") type-checks; label("retired") fails with error TS2345: Argument of type '"retired"' is not assignable to parameter of type 'Status'. The emitted JavaScript prints on_leave then active. You get names, values, and a serialisable union with no reverse-map object.

Example: the const enum + isolatedModules caveat

// With "isolatedModules": true (required by esbuild / swc / Babel / Vite):
export const enum Color { Red, Green, Blue }
//           ^^^^^ ERROR under isolatedModules

// Single-file transpilers compile each file in isolation and cannot inline
// a const enum whose declaration lives in another module, so they forbid it.
// Fix: drop 'const' (accept the runtime object) or use the as-const pattern.
const Color2 = { Red: 0, Green: 1, Blue: 2 } as const;
type Color2 = typeof Color2[keyof typeof Color2]; // 0 | 1 | 2
Result: Under "isolatedModules": true, tsc reports error TS1208: 'Color' cannot be compiled under '--isolatedModules' because it is a const enum. Consider removing 'const'. (and erasableSyntaxOnly rejects all enums). The as const object replacement carries no such restriction and works under every transpiler, which is why it is the portable default in bundler-based toolchains.

Note: In any project transpiled by esbuild, swc, Babel, or Vite (i.e. with isolatedModules on), avoid const enum entirely — the inlining it relies on is unavailable single-file. Use the as const object + derived union: it is erasable, tree-shakeable, serialisable, and free of reverse-map indirection.

Tuples & variadic tuples

A tuple type fixes both the length and the per-position type of an array: [string, number] is exactly two elements. Elements can be optional ([a: string, b?: number]), rest ([string, ...number[]] — a head of known types followed by any number more), and labeled ([x: number, y: number]) purely for readable editor hints. A readonly tuple forbids mutation and is what as const produces from an array literal. Because a tuple's length is a literal type, destructuring stays precisely typed.

Variadic tuple types let you spread one tuple type inside another — [...T, U] — so a function can compose, concatenate, or partially apply argument lists while preserving each element's type. This is the machinery behind typed concat, curry, and the way Parameters<F> feeds back into wrappers. Spreading a tuple type into a rest parameter ((...args: T)) is how you type functions over an arbitrary but statically-known argument shape.

Example: optional, rest, labeled elements, and readonly tuples

// Fixed + optional + rest, all in one signature:
type Row = [id: number, name: string, ...tags: string[]];
const r1: Row = [1, "Ada"];                 // rest may be empty
const r2: Row = [2, "Grace", "eng", "lead"];

// Labeled tuple: labels are hints only, they do not change the type.
function dist([x, y]: [x: number, y: number]): number {
  return Math.hypot(x, y);
}

// readonly tuple: mutation methods are removed from the type.
const point: readonly [number, number] = [3, 4];
point[0] = 9;      // ERROR TS2540: readonly
const frozen = [10, 20] as const; // readonly [10, 20]

console.log(r2.length, dist([3, 4]), point[0], frozen[1]);
Result: tsc reports error TS2540: Cannot assign to '0' because it is a read-only property. on point[0] = 9. With that removed, the emitted JavaScript prints 4 5 3 20 — r2.length is 4, dist returns 5, and both tuples read normally. The labels x/y and id/name/tags show up in editor tooltips but are erased.

Example: variadic concat preserving element types through a spread

// [...T] captures each element's type; the result splices both tuples.
function concat<T extends readonly unknown[], U extends readonly unknown[]>(
  a: [...T], b: [...U]
): [...T, ...U] {
  return [...a, ...b];
}

const merged = concat([1, "two"] as const, [true, 3] as const);
// typed as readonly [1, "two", true, 3] — element types are PRESERVED,
// not widened to (number | string | boolean)[].
const first: 1 = merged[0];
const third: true = merged[2];
console.log(merged);
Result: Type-checks cleanly. merged is typed [1, "two", true, 3], so merged[0] is the literal 1 and merged[2] is true — a plain (string|number|boolean)[] would have lost both positions. The emitted JavaScript prints [ 1, 'two', true, 3 ].

Example: partial application typed with a variadic tuple

// Bind the FIRST argument; the remaining-args type is computed from
// Parameters<F> with a variadic 'infer Rest'.
function partial<A, R extends unknown[], Out>(
  fn: (a: A, ...rest: R) => Out,
  a: A
): (...rest: R) => Out {
  return (...rest: R) => fn(a, ...rest);
}

function makeLabel(prefix: string, id: number, active: boolean): string {
  return `${prefix}#${id}${active ? "*" : ""}`;
}

const withUser = partial(makeLabel, "user"); // (id: number, active: boolean) => string
console.log(withUser(7, true));   // "user#7*"
withUser("7", true);              // ERROR TS2345: 'id' must be a number
Result: Type-checks cleanly. withUser is typed (id: number, active: boolean) => string — the bound prefix is gone and the remaining tuple [number, boolean] is captured by the variadic R. The bad call fails: error TS2345: Argument of type 'string' is not assignable to parameter of type 'number'. The valid call prints user#7*.

Example: a useState-like readonly tuple return via as const

// Returning 'as const' makes the tuple a fixed-shape readonly pair,
// so callers can destructure with precise per-slot types.
function useState<T>(initial: T): readonly [T, (next: T) => void] {
  let value = initial;
  const set = (next: T) => { value = next; };
  return [value, set] as const; // readonly [T, (next: T) => void]
}

const [count, setCount] = useState(0); // count: number, setCount: (n: number)=>void
setCount(5);
console.log(count);
// The pair is a fixed 2-tuple, so a third element is a type error:
const [a, b, c] = useState("x"); // ERROR TS2493: tuple has no element at index 2
Result: useState(0) is typed readonly [number, (next: number) => void], so count is number and setCount takes a number. Destructuring a third element fails: error TS2493: Tuple type 'readonly [string, (next: string) => void]' of length '2' has no element at index '2'. The emitted JavaScript prints 0 (the closure captures the value; this mirrors React's hook shape, not its re-render semantics).

Note: Return as const tuples from functions whose callers destructure positionally (hook-style pairs, [error, value] results). The fixed length and per-slot types make const [a, b] = … precise and catch an extra or missing binding at compile time, which a plain array return cannot.

The satisfies operator & const contexts

There are three ways to relate a value to a type, and they differ in what the compiler keeps. A type annotation (const x: T = v) checks v against T and then widens x to T, discarding the narrower inferred type. An assertion (v as T) overrides the checker and can lie. The satisfies operator (v satisfies T) does both good things at once: it validates that v is assignable to T — catching missing or excess keys a plain as would ignore — while keeping v's precise inferred type for downstream use.

This makes satisfies the right tool for config objects, palettes, and Record-shaped maps that must conform to a contract yet keep their literal element types (so config.home is "/", not string). It pairs with two const-context features: as const for deep readonly, literal-narrowed values, and const type parameters (<const T>) that preserve literal narrowness through a generic call without an as const at every call site. Use as const when you also want immutability; satisfies alone keeps values mutable but literally typed.

Example: satisfies keeps literal types while validating a Record shape

interface Route { path: string; auth: boolean; }

// satisfies validates each value AND keeps the narrow inferred type.
const routes = {
  home:    { path: "/",       auth: false },
  profile: { path: "/me",     auth: true  },
} satisfies Record<string, Route>;

// Because the inferred type is kept, key access is literal and precise:
const p: "/" = routes.home.path;        // OK: path is "/", not string
const keys = Object.keys(routes);       // "home" | "profile" keys are known

// A malformed value is REJECTED (unlike 'as'):
const bad = {
  home: { path: "/", auth: "nope" },    // ERROR TS2322: string not assignable to boolean
} satisfies Record<string, Route>;
console.log(routes.home.path, keys);
Result: The routes object validates against Record<string, Route> yet routes.home.path keeps the literal type "/", so const p: "/" assigns cleanly. The bad block fails: error TS2322: Type 'string' is not assignable to type 'boolean'. on auth: "nope". The emitted JavaScript prints / [ 'home', 'profile' ].

Example: annotation vs as vs satisfies, side by side

type Palette = Record<string, string>;

// (1) Annotation: validates, but WIDENS — primary is now 'string'.
const a: Palette = { primary: "#0b6" };
const t1: "#0b6" = a.primary; // ERROR TS2322: a.primary is 'string', not the literal

// (2) Assertion: keeps the shape you claim, but can LIE.
const b = { primary: 123 } as unknown as Palette; // compiles; value is wrong

// (3) satisfies: validates AND keeps the literal type.
const c = { primary: "#0b6" } satisfies Palette;
const t3: "#0b6" = c.primary; // OK: literal preserved

console.log(a.primary, c.primary);
Result: The annotation path fails at t1: error TS2322: Type 'string' is not assignable to type '"#0b6"'. because a.primary was widened to string. The assertion b compiles despite holding a number — a latent bug. Only satisfies gives both guarantees: c.primary stays "#0b6" and t3 assigns. The emitted JavaScript prints #0b6 #0b6.

Example: const type parameters preserve literal tuples without as const

// Without 'const T', inference widens the argument tuple:
function loose<T extends readonly unknown[]>(...xs: T): T { return xs; }
const w = loose("a", 1, true); // (string | number | boolean)[] — widened

// With a 'const' type parameter, literal narrowness is preserved at the
// call site WITHOUT the caller writing 'as const':
function tuple<const T extends readonly unknown[]>(...xs: T): T { return xs; }
const t = tuple("a", 1, true); // readonly ["a", 1, true] — narrow

const head: "a" = t[0]; // OK: literal preserved by 'const T'
console.log(w, t);
Result: Type-checks cleanly. w is inferred as the widened (string | number | boolean)[], but t is the narrow readonly ["a", 1, true] because the <const T> parameter applies an implicit as const to the argument — so t[0] is the literal "a". The emitted JavaScript (both const modifiers erased) prints [ 'a', 1, true ] [ 'a', 1, true ].

Example: as const for deep readonly + literal narrowing, combined with satisfies

// as const: deep readonly + literal narrowing on a nested structure.
const theme = {
  colors: { bg: "#111", fg: "#eee" },
  spacing: [4, 8, 16],
} as const;
// theme.colors.bg is "#111"; theme.spacing is readonly [4, 8, 16].
theme.colors.bg = "#000";   // ERROR TS2540: deeply readonly
theme.spacing.push(32);     // ERROR TS2339: no push on readonly tuple

// Combine when you want BOTH validation and deep-readonly literals:
const config = {
  mode: "dark",
  retries: 3,
} as const satisfies { mode: "light" | "dark"; retries: number };
console.log(theme.colors.fg, config.mode);
Result: tsc reports error TS2540 on theme.colors.bg = "#000" (the readonly reaches nested objects) and error TS2339 on theme.spacing.push(32). The as const satisfies … combination validates mode against "light" | "dark" while keeping it the literal "dark" and freezing the object. With the error lines removed the emitted JavaScript prints #eee dark.

Note: Reach for satisfies whenever a value must both conform to a contract and keep its precise literals — config maps, discriminated-union factories, exhaustive Record lookups. Reserve as for the rare case where you genuinely know more than the compiler and cannot express it; satisfies covers most uses as was historically abused for, without the risk of a silent lie.

Modules, namespaces & import type

ES modules (import/export) are the default unit of organisation. TypeScript's older namespace construct (formerly "internal modules") predates them, emits an IIFE-wrapped object, and is largely legacy — appropriate today mainly for ambient .d.ts grouping, not for application code, where files-as-modules is the norm. A type-only import (import type { X }, or the inline import { type X }) guarantees the binding is erased from the emitted JavaScript, so it can never accidentally pull a value (and its side effects) into the bundle.

Two compiler options make type-only imports mandatory rather than optional. isolatedModules and the stricter verbatimModuleSyntax require that anything used only as a type be imported with import type, because a single-file transpiler cannot see across modules to know whether an imported name is a type or a value. Beyond imports, module augmentation (declare module "pkg" { … }) adds members to an existing module's types — the module-scoped counterpart to the global augmentation shown earlier — and export type / export type * re-export types without risking a value re-export.

Example: import type is erased; a value import survives

// user.ts
export interface User { id: string; name: string; }
export function makeUser(id: string): User { return { id, name: "?" }; }

// consumer.ts
import type { User } from "./user";     // TYPE-only: erased from emit
import { makeUser } from "./user";      // VALUE: survives into the JS

const u: User = makeUser("u_1");
console.log(u.id);

// Emitted consumer.js (roughly):
//   import { makeUser } from "./user";   <- the 'import type' line is GONE
//   const u = makeUser("u_1");
//   console.log(u.id);
Result: Type-checks cleanly. The import type { User } line produces no runtime import — it is fully erased — so the module graph does not load ./user for the type alone. The import { makeUser } line is a real value import and remains in consumer.js. Erasing type-only imports avoids dragging modules (and their side effects) into the bundle just to reference a shape.

Example: verbatimModuleSyntax forces splitting a mixed import (TS1484)

// With "verbatimModuleSyntax": true, a mixed import where SOME names are
// types must separate them, because the emitter copies import syntax verbatim
// and cannot strip a value-position type on its own.

// BEFORE — rejected:
import { makeUser, User } from "./user"; // ERROR TS1484 on 'User' (a type)

// AFTER — split the type out (inline form):
import { makeUser, type User } from "./user"; // OK: 'type' marks the erasable one
// ...or two statements:
import { makeUser } from "./user";
import type { User } from "./user";
Result: Under "verbatimModuleSyntax": true, the mixed import reports error TS1484: 'User' is a type and must be imported using a type-only import when 'verbatimModuleSyntax' is enabled. Marking it type User inline (or moving it to a separate import type) resolves it and makes the emit unambiguous: value bindings stay, type bindings vanish.

Example: module augmentation adds a property to an existing module

// express-augment.d.ts (or any module file)
// Add a 'userId' to Express's Request WITHOUT editing the library.
import "express";

declare module "express" {
  interface Request {
    userId?: string;   // merged into the library's own Request interface
  }
}

// Now, anywhere in the app:
import type { Request } from "express";
function handler(req: Request): void {
  console.log(req.userId?.toUpperCase()); // 'userId' is typed 'string | undefined'
}
Result: After this augmentation, req.userId is typed string | undefined on every Express Request across the codebase — the declaration merges into the library's own Request interface via the same declaration-merging mechanism interfaces use. Without the augmentation, req.userId would report error TS2339: Property 'userId' does not exist on type 'Request'.

Example: a legacy namespace vs the equivalent ES-module refactor

// LEGACY: a namespace groups members behind one object (emits an IIFE).
namespace Geometry {
  export interface Point { x: number; y: number; }
  export function dist(a: Point, b: Point): number {
    return Math.hypot(a.x - b.x, a.y - b.y);
  }
}
const d1 = Geometry.dist({ x: 0, y: 0 }, { x: 3, y: 4 });

// MODERN: the same code as an ES module (geometry.ts):
//   export interface Point { x: number; y: number; }
//   export function dist(a: Point, b: Point): number { ... }
// consumer.ts:
//   import { dist, type Point } from "./geometry";
//   const d2 = dist({ x: 0, y: 0 }, { x: 3, y: 4 });
console.log(d1);
Result: Both express the same grouping. The namespace version emits a runtime Geometry object (an IIFE that attaches dist), and Geometry.dist(...) prints 5. The ES-module version emits a standard module with a tree-shakeable dist export and no wrapper object. Prefer the module form in application code; keep namespace for ambient declaration grouping in .d.ts files where modules are not available.

Note: Turn on verbatimModuleSyntax in new projects. It forces an explicit import type/export type for every type-only binding, which makes emit predictable, prevents accidental value imports (and their side effects), and lets fast single-file transpilers erase types without cross-module analysis. The one-time cost is annotating your type imports.

Template literal types

A template literal type builds string-literal types by interpolation, using the same backtick syntax as runtime template strings but at the type level: `on${Capitalize<Event>}` or `/users/${number}`. When an interpolated slot is a union, the type distributes, producing the cartesian product of all combinations. The intrinsic string types Uppercase, Lowercase, Capitalize, and Uncapitalize transform each literal as it is spliced in.

Template patterns also work in the matching direction: T extends `${infer Head}-${infer Tail}` parses a string literal at the type level, capturing pieces with infer. Combined with key remapping in a mapped type (as `get${Capitalize<K>}`), templates generate whole derived shapes — getter-name maps, typed event channels, route-parameter extractors — from a source type, with no hand-written parallel definitions.

Example: interpolation, distribution, and the intrinsic string types

type Domain = "user" | "order";
type Lifecycle = "Created" | "Deleted";

// Distributive: every Domain x every Lifecycle -> 4 event names.
type EventName = `${Capitalize<Domain>}${Lifecycle}`;
// "UserCreated" | "UserDeleted" | "OrderCreated" | "OrderDeleted"

type Handler = `on${EventName}`;   // "onUserCreated" | ... (4 members)
type HttpUrl = `https://${string}`; // any string starting with https://

const ok: HttpUrl = "https://x.dev/api"; // OK
const bad: HttpUrl = "http://x.dev";     // ERROR TS2322: missing 'https://'
const evt: Handler = "onOrderDeleted";
console.log(ok, evt);
Result: EventName expands to the 4-member cartesian product and Handler prefixes each with on. tsc reports error TS2322: Type '"http://x.dev"' is not assignable to type '`https://${string}`'. on bad. With that line removed, the emitted JavaScript prints https://x.dev/api onOrderDeleted.

Example: parsing a route with infer in a template pattern

// Extract the dynamic segments of a route into a params object type.
type Params<Path extends string> =
  Path extends `${string}:${infer Param}/${infer Rest}`
    ? { [K in Param | keyof Params<`/${Rest}`>]: string }
    : Path extends `${string}:${infer Last}`
      ? { [K in Last]: string }
      : {};

type R = Params<"/users/:userId/posts/:postId">;
// { userId: string; postId: string }

function getParams<P extends string>(_pattern: P, values: Params<P>): Params<P> {
  return values;
}
const p = getParams("/users/:userId/posts/:postId", { userId: "u1", postId: "p9" });
console.log(p.userId, p.postId);
getParams("/users/:userId", { wrong: "x" }); // ERROR TS2353: 'userId' missing
Result: Params<"/users/:userId/posts/:postId"> resolves to { userId: string; postId: string } by recursively matching each :segment with infer. The valid getParams call type-checks and prints u1 p9; the last call fails because the object is missing userId and has an excess wrong key (error TS2353 / TS2345).

Example: key remapping to build a getter-name map

interface Employee { name: string; age: number; active: boolean; }

// Remap each key K to `get${Capitalize<K>}` returning that field's type.
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};
type EmployeeGetters = Getters<Employee>;
// { getName(): string; getAge(): number; getActive(): boolean }

const e: Employee = { name: "Ada", age: 30, active: true };
const getters: EmployeeGetters = {
  getName: () => e.name,
  getAge: () => e.age,
  getActive: () => e.active,
};
console.log(getters.getName(), getters.getAge());
Result: Getters<Employee> resolves to { getName(): string; getAge(): number; getActive(): boolean } — the as clause renames each key and Capitalize upper-cases the first letter, while the value type () => T[K] preserves each field's type. Omitting getActive from the object would fail with error TS2741: Property 'getActive' is missing. The emitted JavaScript prints Ada 30.

Example: recursive template inference — kebab-case to camelCase

// Recursively fold each '-x' segment into 'X' at the type level.
type CamelCase<S extends string> =
  S extends `${infer Head}-${infer Tail}`
    ? `${Head}${CamelCase<Capitalize<Tail>>}`
    : S;

type A = CamelCase<"background-color">;  // "backgroundColor"
type B = CamelCase<"data-user-id">;      // "dataUserId"

// Apply it to remap object keys from kebab to camel:
type Camelize<T> = { [K in keyof T as CamelCase<string & K>]: T[K] };
type Styles = Camelize<{ "font-size": number; "line-height": number }>;
// { fontSize: number; lineHeight: number }
const s: Styles = { fontSize: 14, lineHeight: 1.5 };
console.log(s.fontSize, s.lineHeight);
Result: Type-only resolution: CamelCase<"background-color"> is "backgroundColor" and Camelize<…> rewrites every kebab key to camelCase, so Styles is { fontSize: number; lineHeight: number }. Writing "font-size" as a key of s would now be an error. The emitted JavaScript prints 14 1.5.

Note: Template-literal parsing is powerful but each recursive match instantiates types; deeply nested or unbounded patterns can slow the checker and, past the depth limit, trigger TS2589. Use them for genuinely string-shaped contracts (routes, event names, CSS keys) where the safety pays for the compile cost — not to reformat arbitrary strings the runtime could handle.

Type-level programming

The type system is Turing-complete: conditional types with infer, recursion, and template literals compose into a small functional language that computes types from types. The core toolkit is exactly those three features plus mapped types. The practical payoff is types that stay correct automatically — dotted key paths of a nested object, the value type at a path, a genuinely deep Readonly — instead of hand-maintained parallel definitions that drift.

The cost is real. Recursive conditional types hit an instantiation-depth limit and fail with TS2589; since TS 4.5, tail-recursive conditionals get an elimination optimisation that lets an accumulator-style type recurse far deeper before hitting it. Even within limits, elaborate type-level programs slow every compile and every editor keystroke, so treat them as a last resort reserved for eliminating a real class of bugs or duplication — the same caution the best-practices section raised.

Example: Paths<T> — every dotted key path of a nested object

type Paths<T> = T extends object
  ? {
      [K in keyof T & string]:
        T[K] extends object ? K | `${K}.${Paths<T[K]>}` : K;
    }[keyof T & string]
  : never;

interface Employee {
  id: string;
  contact: { email: string; phone: string };
  org: { costCenter: string };
}
type EmployeePath = Paths<Employee>;
// "id" | "contact" | "contact.email" | "contact.phone" | "org" | "org.costCenter"

const p: EmployeePath = "contact.email"; // OK
const bad: EmployeePath = "contact.fax"; // ERROR TS2322: not a valid path
console.log(p);
Result: Paths<Employee> resolves to the union of every dotted path, including nested ones like "contact.email" and "org.costCenter". Assigning "contact.fax" fails: error TS2322: Type '"contact.fax"' is not assignable to type 'Paths<Employee>'. The emitted JavaScript prints contact.email.

Example: a typed get(obj, path) returning the value type at that path

// PathValue<T, P> walks the dotted path to find the value type.
type PathValue<T, P extends string> =
  P extends `${infer Head}.${infer Rest}`
    ? Head extends keyof T ? PathValue<T[Head], Rest> : never
    : P extends keyof T ? T[P] : never;

function get<T, P extends Paths<T>>(obj: T, path: P): PathValue<T, P & string> {
  return (path as string).split(".").reduce<any>((o, k) => o?.[k], obj);
}

const employee: Employee = {
  id: "e1", contact: { email: "a@x.dev", phone: "555" }, org: { costCenter: "CC-1" },
};
const cc = get(employee, "org.costCenter"); // typed string
const em = get(employee, "contact.email");  // typed string
get(employee, "org.budget");                // ERROR TS2345: invalid path
console.log(cc, em);
Result: Type-checks. get(employee, "org.costCenter") is typed string because PathValue walks org → costCenter; an invalid path like "org.budget" is rejected by the P extends Paths<T> constraint (error TS2345). The emitted JavaScript reduces over the split path and prints CC-1 a@x.dev.

Example: tail-recursive Join vs a naive version that risks TS2589

// Tail-recursive: an accumulator (Acc) carries the result, so TS 4.5+
// eliminates the recursion frame and it scales to long tuples.
type Join<T extends readonly string[], Sep extends string, Acc extends string = ""> =
  T extends readonly [infer Head extends string, ...infer Tail extends string[]]
    ? Join<Tail, Sep, Acc extends "" ? Head : `${Acc}${Sep}${Head}`>
    : Acc;

type Route = Join<["users", "u1", "posts"], "/">; // "users/u1/posts"

// Naive NON-tail version (nests the recursive call inside the template):
// each frame is retained, so a very long tuple can hit the depth limit.
type JoinNaive<T extends readonly string[], Sep extends string> =
  T extends readonly [infer H extends string, ...infer R extends string[]]
    ? R extends [] ? H : `${H}${Sep}${JoinNaive<R, Sep>}`
    : "";
type Route2 = JoinNaive<["a", "b", "c"], "-">; // "a-b-c" (fine while short)
console.log("" as Route, "" as Route2);
Result: Type-only: both resolve Route to "users/u1/posts" and Route2 to "a-b-c". The accumulator form is tail-recursive, so TS 4.5+ optimises it and it handles long tuples; the naive form retains a frame per element and a sufficiently long input triggers error TS2589: Type instantiation is excessively deep and possibly infinite. The emitted JavaScript prints two empty lines (the values are placeholder empty strings).

Example: a rigorous DeepReadonly tested against a mutation

// Recurse through objects AND arrays; leave primitives and functions alone.
type DeepReadonly<T> =
  T extends (infer E)[] ? ReadonlyArray<DeepReadonly<E>>
  : T extends (...args: any[]) => any ? T
  : T extends object ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
  : T;

interface Team { name: string; members: { id: string }[]; meta: { size: number } }
const team: DeepReadonly<Team> = {
  name: "eng", members: [{ id: "e1" }], meta: { size: 1 },
};
team.name = "ops";          // ERROR TS2540: readonly
team.meta.size = 2;         // ERROR TS2540: nested readonly
team.members.push({ id: "e2" }); // ERROR TS2339: no push on ReadonlyArray
team.members[0].id = "x";   // ERROR TS2540: element is readonly too
console.log(team.name);
Result: DeepReadonly<Team> freezes every level: tsc reports TS2540 on team.name, team.meta.size, and team.members[0].id, and TS2339 on team.members.push(...) because arrays become ReadonlyArray. A shallow Readonly<Team> would catch only the top-level assignment. With the mutation lines removed the emitted JavaScript prints eng.

Note: Type-level programs run inside the compiler on every keystroke. A Paths<T> or DeepReadonly<T> over a large, deeply-nested type can noticeably slow autocompletion and tsc. Prefer tail-recursive formulations, cap depth deliberately, and reach for this machinery only when the alternative is an error-prone hand-maintained type — clarity for the next reader still outranks type-golf.

Variance & advanced generics

Variance describes how subtyping of a type parameter relates to subtyping of the container. Given Dog is a subtype of Animal: a position is covariant if Box<Dog> is a subtype of Box<Animal> (outputs flow this way — a readonly array of dogs is a readonly array of animals), and contravariant if the relationship flips (function parameters work this way — a handler that accepts any Animal can stand in where a Dog handler is expected). A mutable array is invariant: allowing Dog[] where Animal[] is expected is unsound, because the callee could push a Cat.

The strictFunctionTypes flag makes standalone function-type parameters checked contravariantly (sound); it deliberately exempts method parameters, which stay bivariant for ergonomic reasons (arrays, the DOM). TypeScript infers variance automatically, but you can annotate it explicitly with out/in on a type parameter (interface Box<out T>) both to document intent and to have the compiler enforce it. Higher-order generic functions preserve inference through composition when their type parameters are threaded correctly.

Example: mutable-array unsoundness vs a covariant readonly array

class Animal { move() {} }
class Dog extends Animal { bark() {} }
class Cat extends Animal { meow() {} }

// Mutable arrays are INVARIANT — this assignment is REJECTED, and rightly so:
const dogs: Dog[] = [new Dog()];
const animals: Animal[] = dogs; // ERROR TS2322 under strict variance checks
// If it were allowed, the next line would smuggle a Cat into 'dogs':
// animals.push(new Cat());  // 'dogs' now contains a non-Dog — unsound.

// readonly arrays ARE covariant — safe, because you cannot push:
const roDogs: readonly Dog[] = [new Dog()];
const roAnimals: readonly Animal[] = roDogs; // OK: no mutation possible
console.log(roAnimals.length);
Result: Assigning Dog[] to Animal[] reports error TS2322: Type 'Dog[]' is not assignable to type 'Animal[]'. because a mutable array is invariant — permitting it would let a Cat be pushed into an array typed Dog[]. The readonly version is accepted because a ReadonlyArray exposes no mutators, making covariance sound. The emitted JavaScript prints 1.

Example: contravariant callback parameters and the strictFunctionTypes flip

class Animal { move() {} }
class Dog extends Animal { bark() {} }

type DogHandler = (d: Dog) => void;

// A handler that accepts ANY Animal is MORE general, so it can stand in
// where a Dog handler is expected — parameters are contravariant.
const animalHandler: (a: Animal) => void = (a) => a.move();
const asDogHandler: DogHandler = animalHandler; // OK: contravariance

// The reverse is UNSAFE: a Dog-only handler cannot accept every Animal.
const dogOnly: DogHandler = (d) => d.bark();
const asAnimalHandler: (a: Animal) => void = dogOnly; // ERROR under strictFunctionTypes
console.log(typeof asDogHandler);
Result: animalHandler is assignable to DogHandler (a more-general parameter is fine). The reverse fails under "strictFunctionTypes": true: error TS2322: Type '(d: Dog) => void' is not assignable to type '(a: Animal) => void'. Types of parameters 'd' and 'a' are incompatible. because a Dog handler could call .bark() on a Cat. Note that if these were declared as methods the check would be bivariant and the unsafe direction would be allowed. The emitted JavaScript prints function.

Example: explicit in/out variance annotations and TS2636

// 'in' marks a CONTRAVARIANT parameter: Event is used only in input position.
interface Emitter<in Event> {
  emit(event: Event): void;
}
// 'out' marks a COVARIANT parameter: T is used only in output position.
interface Producer<out T> {
  get(): T;
}

// The compiler ENFORCES the annotation. If a member uses 'Event' in an
// output position, it violates the declared contravariance:
interface Broken<in Event> {
  emit(event: Event): void;
  last(): Event; // ERROR TS2636: 'Event' is declared 'in' but used in an out position
}
declare const p: Producer<Dog>;
const pa: Producer<Animal> = p; // OK: 'out' makes Producer covariant
console.log(typeof pa);
Result: The out T annotation on Producer makes Producer<Dog> assignable to Producer<Animal>. The Broken interface fails: error TS2636: Type 'Event' is declared with a variance annotation 'in' but is used in a covariant position within its type. because last(): Event returns the parameter, contradicting the declared in. Variance annotations both document and enforce the intended relationship.

Example: a compose helper threading inference through two generics

// Each type parameter is inferred from one function and reused in the next,
// so the composed function's types are computed end-to-end.
function compose<A, B, C>(
  f: (a: A) => B,
  g: (b: B) => C
): (a: A) => C {
  return (a) => g(f(a));
}

const parse = (s: string): number => s.length;   // A=string, B=number
const isBig = (n: number): boolean => n > 3;      // B=number, C=boolean
const check = compose(parse, isBig);              // (a: string) => boolean

console.log(check("hello"), check("hi"));
// A mismatch in the middle is caught: B of f must feed g's input.
compose(parse, (b: string) => b); // ERROR TS2345: number not assignable to string
Result: Type-checks cleanly. compose(parse, isBig) is inferred as (a: string) => boolean — A comes from parse's input, B is threaded from parse's output into isBig's input, and C is isBig's output. The mismatched call fails: error TS2345: Argument of type '(b: string) => string' is not assignable… because parse returns number, not string. The valid calls print true false.

Note: Keep strictFunctionTypes on for sound callback checking, but remember the intentional method-bivariance exemption: obj.method(cb) where method is declared with method syntax does not get the contravariant check that a property-style method: (cb) => void field would. When you want the strict behaviour, declare the field as a function-type property rather than a method.

Control-flow analysis & narrowing

The compiler tracks each variable's type as control flows through your code, refining ("narrowing") it at every branch. Truthiness checks remove null/undefined/0/""; equality checks (===/!==) narrow against literals and discriminants; the in operator narrows a union by property presence; and typeof/instanceof narrow by runtime tag. Since TS 4.4, a const alias of a discriminant (const k = shape.kind) preserves narrowing when you later switch on k, which earlier versions could not do.

Narrowing is flow-sensitive: it holds across statements until something invalidates it — a reassignment, or an intervening function call the compiler cannot reason about (which may have mutated a captured variable). It also does not carry into a nested closure, because the closure might run later, after the variable changed. When a union is exhausted, the residual type is never, which powers exhaustiveness checks; and whether a variable was declared let or const affects how aggressively the compiler is willing to narrow it.

Example: truthiness/equality/in narrowing, and reassignment widening it back

function render(x: string | null): number {
  if (x) {
    // Truthiness narrowed x to 'string' here:
    const len = x.length;      // OK, no '!' needed
    x = null;                  // reassignment WIDENS x back to 'string | null'
    return x.length;           // ERROR TS18047: x is possibly 'null' again
    return len;
  }
  return 0;
}

// 'in' narrows a union by property presence:
type Fish = { swim(): void }; type Bird = { fly(): void };
function move(pet: Fish | Bird): void {
  if ("swim" in pet) pet.swim(); // narrowed to Fish
  else pet.fly();                // narrowed to Bird
}
Result: Inside if (x), x is narrowed to string, so x.length is initially valid. After x = null the compiler widens x back to string | null, so the second x.length reports error TS18047: 'x' is possibly 'null'. The in operator narrows pet to Fish in the if and Bird in the else. Narrowing is a property of the flow position, not the variable forever.

Example: aliasing a discriminant still narrows (TS 4.4+)

type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; side: number };

function area(shape: Shape): number {
  // Alias the discriminant to a const. Since TS 4.4 this STILL narrows 'shape'.
  const kind = shape.kind;
  switch (kind) {
    case "circle": return Math.PI * shape.radius ** 2; // shape narrowed to circle
    case "square": return shape.side ** 2;             // shape narrowed to square
  }
}
console.log(area({ kind: "circle", radius: 2 }).toFixed(2));
// Pre-4.4, 'shape' would NOT narrow through the alias and 'shape.radius'
// would error with TS2339 in the 'circle' case.
Result: Type-checks cleanly on TS 4.4+. The const kind = shape.kind alias carries the discriminant relationship, so switching on kind narrows shape itself — shape.radius is available in the circle case and shape.side in the square case. The emitted JavaScript prints 12.57. On pre-4.4 compilers the same code failed with error TS2339 because narrowing did not follow the alias.

Example: narrowing does not survive into a nested closure

function scheduleGreeting(name: string | undefined): void {
  if (name === undefined) return;
  // Here 'name' is narrowed to 'string'.
  console.log(name.toUpperCase()); // OK — synchronous, still narrowed

  setTimeout(function () {
    // Inside the callback the narrowing is LOST: the compiler assumes
    // 'name' could have been reassigned before this runs later.
    console.log(name.toUpperCase()); // ERROR TS18048: 'name' is possibly 'undefined'
  }, 100);
}
Result: The synchronous name.toUpperCase() is fine because name is narrowed to string after the guard. Inside the setTimeout callback the same expression reports error TS18048: 'name' is possibly 'undefined'. — a closure may execute after the enclosing scope changed name, so the compiler conservatively drops the narrowing. Capturing the narrowed value in a const before the closure (const n = name;) restores it, because a const cannot be reassigned.

Example: exhaustive narrowing to never, broken by a new union member

type Status = "active" | "paused" | "closed";

function label(s: Status): string {
  switch (s) {
    case "active": return "Active";
    case "paused": return "Paused";
    case "closed": return "Closed";
    default: {
      // All members handled, so 's' is 'never' here — the assignment is legal.
      const _exhaustive: never = s;
      return _exhaustive;
    }
  }
}
console.log(label("paused"));
// Add "archived" to Status WITHOUT a case, and the default breaks:
//   const _exhaustive: never = s;
//   -> ERROR TS2322: Type '"archived"' is not assignable to type 'never'.
Result: With all three members handled, s is narrowed to never at the default, so const _exhaustive: never = s assigns cleanly and the run prints Paused. Adding "archived" to Status without a matching case leaves s as "archived" at the default, and the assignment fails: error TS2322: Type '"archived"' is not assignable to type 'never'. — turning a forgotten case into a compile error.
