Java Collections — from Basics to Advanced

A hands-on study guide to storing, sorting, traversing and processing data with core Java. It moves from raw arrays through the entire Collection framework — List, Set, Queue and Map with every major implementation — then on to concurrent collections, lambdas and functional interfaces, the Streams API, and generics. Each lecture pairs a focused Java example with the console output it produces, traced by hand (Java runs on the JVM, not in the browser).

Prerequisites & how to read this You should be comfortable with core Java syntax (classes, methods, primitive types, loops). Every example is plain Java that runs on JDK 17+; compile and run any snippet with javac File.java && java File, or paste a fragment into jshell. Use the filter box in the sidebar to jump to any lecture. Prose is paraphrased from the course for study; code examples are reimplemented and improved.

1. Introduction

This opening section sets the stage for the whole course: what you will learn, why data handling dominates real Java applications, and how the sections are ordered so you can study them as independent, self-contained units.

Run locally Every example in this guide is plain, backend Java that runs on JDK 17 or newer. Save a snippet to a file named after its public class, for example Welcome.java, then compile and run it with javac Welcome.java followed by java Welcome. For quick one-liners you can skip the file entirely and paste them into jshell, the interactive Java shell shipped with the JDK.

01. Introduction

Objective: Understand the scope, motivation, and structure of the course so you can navigate the material efficiently and know how each topic connects to real-world data handling.

The course is a broad tour of the tools core Java gives you for storing, organizing, and processing data: arrays, strings, the Collections framework (lists, sets, maps, trees), sorting and traversal, concurrent collections for multi-threaded programs, generics, and finally the Java 8 features of lambdas and streams. The framing observation is a practical one drawn from years of building Java systems: a large share of application code — on the order of 70% — is really about moving data from a data store into the program and then shaping, presenting, and making decisions on it. Mastering the data-handling toolkit therefore pays off across almost every project you will ever write.

The learning path is deliberately layered. It begins with arrays, the most basic mechanism for holding a fixed number of homogeneous elements. From there it moves into the Collections API, where you meet the List, Set, and Map families along with tree-based structures, then covers how to sort data and how to traverse it in both the classic style (loops and iterators) and newer idioms. Next come concurrent collections, which matter as soon as several threads touch the same data. After that, generics explains the type machinery that the entire Collections API is built on and that you will need if you ever design your own data structure such as a linked list or a sorted set. The course closes with lambdas and streams, the Java 8 constructs that make data processing concise and fast.

A key organizing principle is modularity: most sections are self-contained, so you can study or skip them in almost any order. There are only two prerequisite chains to respect — study collections before concurrent collections, and study lambdas before streams and parallel streams. One headline fact worth remembering early: a well-formed hash map offers average O(1) cost for inserting and looking up an element, which is exactly why maps are such a central data structure.

  • Most application code is data handling — arrays, collections, generics, lambdas, and streams are the core toolkit for it.
  • Sections are self-contained; two ordering rules apply: collections before concurrent collections, and lambdas before streams.
  • The roadmap: arrays → lists/sets/maps/trees → sorting & traversal → concurrent collections → generics → lambdas → streams.
  • Hash-based maps give average O(1) insertion and search, which is why they are heavily emphasized.
  • Generics underpin the whole Collections API and are essential for building your own data structures.
import java.util.ArrayList;
import java.util.HashMap;
import java.util.List;
import java.util.Map;

// arrays, generics, a List, a Map, and a Java 8 stream.
public class Welcome {
    public static void main(String[] args) {
        // Array: fixed-size, homogeneous storage.
        String[] topics = { "arrays", "collections", "generics", "lambdas", "streams" };

        // Generic List: a growable, type-safe sequence.
        List<String> roadmap = new ArrayList<>();
        for (String t : topics) {
            roadmap.add(t);
        }

        // Generic Map: average O(1) lookup by key.
        Map<String, Integer> order = new HashMap<>();
        for (int i = 0; i < roadmap.size(); i++) {
            order.put(roadmap.get(i), i + 1);
        }

        System.out.println("Topics to master: " + roadmap.size());
        System.out.println("Streams come at step " + order.get("streams"));

        // Stream + lambda: concise data processing (Java 8+).
        // "arrays" (6), "lambdas" (7) and "streams" (7) pass; the rest are longer.
        long shortNames = roadmap.stream()
                                 .filter(name -> name.length() <= 7)
                                 .count();
        System.out.println("Topics named in 7 chars or fewer: " + shortNames);
    }
}

AdvancedThe single word "collections" is ambiguous in Java and worth disambiguating up front. Collection (capital C, singular) is the root interface of the framework for groups of elements, extended by List, Set, and Queue. Collections (plural) is a separate utility class full of static helpers such as Collections.sort(list) and Collections.unmodifiableList(list). Notably, Map does not extend Collection — it models key-value associations rather than a plain group of elements, which is why it sits on its own branch of the hierarchy. One more subtlety behind the "average O(1)" claim: it holds only when keys spread evenly across buckets. Since Java 8, a bucket that collects too many colliding keys is converted from a linked list into a balanced tree, so a degenerate, all-collisions bucket degrades to O(log n) rather than O(n).
Pitfall Treating an array as a resizable collection. A Java array has a fixed length chosen at creation time — topics.length can never change. When you need to grow or shrink a sequence, reach for ArrayList (or another List), not an array. And prefer generic declarations like List<String> over raw List, so the compiler catches type mistakes instead of letting them surface as a ClassCastException at runtime.

2. Java Arrays in Depth

Arrays are Java's most fundamental storage mechanism, and many of the collections you will meet later are built on top of them — ArrayList, ArrayDeque, Vector, and the bucket table of HashMap all store their data in an array (others, such as LinkedList and TreeMap/TreeSet, use linked nodes or trees instead). This section works through single- and multi-dimensional arrays: how to declare, create and initialize them, how to traverse them, and how array references and elements behave when you assign and reassign them.

Run locally Each example is a class with a main method. Compile and run with javac ArrayDemo.java then java ArrayDemo (JDK 17+). For quick one-liners, launch jshell and paste snippets directly. No build tool or external dependency is needed.

01. Arrays and single-dimensional arrays

Objective: Understand what an array is, how to create a 1-D array, and the rules that govern its size and element type.

Storing a handful of values in separate variables is fine, but storing hundreds or thousands that way is impractical. An array lets you keep many values of the same type under a single variable. It is a fixed-size, indexed collection of homogeneous elements: you must state the size when you create it with new, and that size can never grow or shrink afterwards. Elements are addressed by a zero-based index, so an array of length ten uses indices 0 through 9 (length minus one).

Although int is a primitive, new int[1000] really does create an object. Java defines a hidden class for every array type; calling getClass().getName() on an int[] returns [I — one bracket for a single dimension, and I for the int element type. These array classes belong to the language itself and are not directly accessible to your code.

  • Size is mandatory at creation and permanently fixed; omit it and you get a compile error.
  • All elements share one type; a numeric array pre-fills with 0.
  • The runtime class name encodes shape and type: [I = 1-D int, [[I = 2-D int, [D = 1-D double, [Z = 1-D boolean.
  • A size may be any value promotable to int (byte, short, char, int) — even 'a', which becomes 97.
public class ArrayIntroduction {
    public static void main(String[] args) {
        int[] x = new int[1000];
        System.out.println(x.getClass().getName()); // hidden array class
        System.out.println(x.length);
        System.out.println(x[0]);                    // default value

        int[] empty = new int[0];   // size zero is legal
        System.out.println(empty.length);

        int[] z = new int['a'];     // 'a' promotes to 97
        System.out.println(z.length);
    }
}

AdvancedThe size is an int, so the largest literal you can write is Integer.MAX_VALUE = 2147483647; a bigger literal such as 2147483648 is rejected at compile time with "integer number too large". In practice the JVM caps the usable maximum a few elements below that (it reserves room for the array header), and even a legal size throws OutOfMemoryError at runtime if the heap cannot hold it — an int[] needs roughly four bytes per element, so new int[214748364] alone wants about 800 MB.
Pitfall new int[-1] compiles cleanly — the compiler only checks that the size type is legal, not its value. At runtime it throws NegativeArraySizeException. Guard any computed size before allocating.

02. Multi-dimensional arrays

Objective: See that a multi-dimensional array is really an array of arrays, and learn to build 2-D and 3-D arrays level by level.

A multi-dimensional array is nothing more than an array whose elements are themselves arrays. new int[3][] creates a 2-D array with base size three: three slots, each of which can hold a reference to a 1-D int array. You then attach a 1-D array to each slot, and those sub-arrays may have different lengths — this is a "jagged" array. Trying to drop a plain int into a base slot fails to compile, because that slot expects an array, not a scalar.

The same idea stacks upward. A 3-D array (new int[2][][]) is a collection of 2-D arrays, each of which is a collection of 1-D arrays. Only the deepest level holds the actual int values; every level above stores references to the next level down. You must always supply the base (leftmost) size; the deeper sizes may be filled in later.

  • Only the base dimension's size is required at creation; deeper dimensions can be sized afterward.
  • Sub-arrays can differ in length, giving jagged shapes.
  • Upper levels hold references; only the last (1-D) level stores primitives.
  • Java permits many dimensions (the language spec caps it at 255), though real code rarely exceeds three.
public class MultidimensionalArrays {
    public static void main(String[] args) {
        int[][] x = new int[3][];   // base size 3, jagged
        x[0] = new int[3];
        x[1] = new int[1];
        x[2] = new int[2];
        System.out.println(x.length);
        System.out.println(x[0].length);
        System.out.println(x[2].length);

        int[][][] z = new int[2][][];   // 3-D, base size 2
        z[0] = new int[3][];
        z[0][0] = new int[3];
        z[0][0][0] = 1;
        System.out.println(z[0][0][0]);
        System.out.println(z.length);
    }
}

03. Declaration, initialization and creation

Objective: Distinguish the separate acts of declaring, creating and initializing an array, and understand default values for both primitive and reference slots.

Declaring a variable (int[] x;) only names a reference; no array object exists yet. You can combine all three steps in one line with a brace literal (int[] y = {1,2,3,4,5};), but that curly-brace form is only valid at the point of declaration — you cannot write x = {1,2,3}; on a later line. Alternatively, new int[3] creates the object and fills every slot with the default value, which is 0 for an int array; you can then overwrite those defaults individually.

Printing the reference itself (not its elements) invokes the default toString, which yields the class name plus an @ and a hexadecimal hash, e.g. [I@1b6d3586. For multi-dimensional arrays, the default of an uninitialized inner slot is null, not zero, because that slot holds an object reference. Touching b[0][0] before assigning b[0] therefore throws a NullPointerException.

  • The { ... } literal only works on the same line as the declaration.
  • new int[n] zero-fills a numeric array; you can then override any element.
  • Printing the array variable shows class@hash, not the contents.
  • Unfilled inner slots of a 2-D array default to null; only leaf 1-D slots default to 0.
public class ArraysDemo {
    public static void main(String[] args) {
        int[] y = {1, 2, 3, 4, 5};   // declare + create + init
        int[] z = new int[3];        // zero-filled
        System.out.println(z[0]);
        z[0] = 10; z[1] = 20; z[2] = 30;
        for (int ele : z) System.out.println(ele);
        System.out.println(z);       // prints the reference

        int[][] b = new int[2][];    // base size 2
        System.out.println(b[0]);    // inner slot is null
        b[0] = new int[2];
        b[1] = new int[3];
        System.out.println(b[0][0]); // now a real 0

        int[][] c = {{1, 2, 3}, {6, 7}};
        for (int[] row : c)
            for (int ele : row)
                System.out.println(ele);
    }
}

AdvancedDefault values follow the element type: numeric arrays start at 0/0.0, boolean[] at false, char[] at '\u0000' (the null character, code point 0 — an invisible control character, not a space), and any array of references (including inner arrays of a multi-dimensional array) at null. Pitfall Reading through an unassigned inner slot (b[0][0] while b[0] is still null) throws NullPointerException. Attach the inner array before indexing into it.

04. Playing with array syntax

Objective: Recognize which of Java's many array declaration and creation syntaxes are legal and which are not.

Java is unusually flexible about where the brackets sit. The clearest form keeps the brackets with the type (int[] x1;), which reads as "x1 is of type int-array", but int x3[]; is equally valid. Whitespace between the type, brackets and name is ignored, so odd spacing still compiles. What you cannot do is put a size in a declaration (int[3] x4; is illegal) — sizes belong to creation, not declaration.

When several variables share one declaration, the brackets placed before the first name apply to all of them, while brackets after a name apply only to that one. So in int[] x18[], x19;, x18 is 2-D (type brackets plus its own trailing pair) but x19 is 1-D. For creation, you must fill dimension sizes from the base outward: new int[3][4][] is fine (trailing sizes optional), but new int[3][][5] or new int[][4][5] are errors because a deeper size is given before the level above it exists.

  • Brackets may precede or follow the variable name; both compile.
  • A size in a declaration (int[3] x) is always illegal.
  • Type-level brackets before a name apply to every variable in the statement; trailing brackets apply to just one.
  • On creation, sizes must be provided left-to-right; deeper sizes may be omitted, never front-loaded.
public class PlayingWithSyntax {
    public static void main(String[] args) {
        int[] x1;         // clearest form
        int x3[];         // also legal
        // int[3] x4;     // ERROR: no size in a declaration

        int[] x18[], x19; // x18 is 2-D, x19 is 1-D
        int[][] x22, x23; // leading [][] binds to the type -> BOTH are 2-D

        int[][][] x25 = new int[3][4][5]; // ok
        int[][][] x26 = new int[3][4][];  // ok, trailing size omitted
        // int[][][] bad = new int[3][][5]; // ERROR: gap before size
    }
}

05. Traversing arrays and the length variable

Objective: Loop over 1-D and 2-D arrays using both the enhanced for-each loop and a classic index-based for loop.

Because the array variable is only a reference, printing it never prints its contents — you must visit each element. The enhanced for-each loop is the simplest way to read every value. When you need the index (for example to write values back), use a counted for loop bounded by length. Note that length is a public final field, not a method: it is a.length, never a.length(), and it cannot be reassigned.

A 2-D array needs nested loops. The outer loop walks the base array; the inner loop walks each 1-D sub-array. With index loops, the inner bound is b[i].length — the length of the specific row i — which correctly handles jagged rows of differing sizes. A 3-D array simply adds a third nested loop.

  • Use for-each to read; use an indexed for when you also need to write or know the position.
  • length is a final field: a.length, not a method call.
  • The loop bound is i < length (strictly less), since valid indices are 0 .. length-1.
  • Nest one loop per dimension; the inner bound b[i].length respects jagged rows.
public class TraversingArrays {
    public static void main(String[] args) {
        int[] a = new int[5];
        for (int e : a) System.out.println(e);   // five zeros

        int len = a.length;
        for (int i = 0; i < len; i++) {
            a[i] = i + 1;
            System.out.println("index " + i + " -> " + a[i]);
        }

        int[][] b = new int[2][3];
        for (int i = 0; i < b.length; i++)
            for (int j = 0; j < b[i].length; j++)
                System.out.println(b[i][j]);      // six zeros
    }
}

06. Types of array by the elements they hold

Objective: Learn what each kind of array — primitive, class, abstract-class, and interface — is allowed to store.

A primitive array accepts any element that can be implicitly promoted to its type: an int[] happily stores a char ('a' becomes 97) or a byte. A reference array can store objects that are instances of its declared type or any subtype, which is the substitution principle at work: an Object[] can hold a String because String is-a Object.

You can even declare an array whose type is abstract or an interface — what you cannot do is store an instance of the abstract type or interface itself, only concrete subtypes/implementers. A Number[] can hold an Integer or Byte but never a bare Number (it is abstract). A Runnable[] can hold any object whose class implements Runnable, such as a Thread; slots you never assign stay null.

  • Primitive arrays accept implicitly promotable values (char/byte into int).
  • Reference arrays accept the declared type or any subtype.
  • Abstract-class and interface arrays are legal, but hold only concrete subtype/implementer instances.
  • Unassigned reference slots remain null.
public class ElementAssignmentsDemo {
    public static void main(String[] args) {
        int[] x = new int[3];
        x[0] = 'a';                 // char promoted to int -> 97
        byte b = 2;
        x[1] = b;                   // byte promoted to int
        System.out.println(x[0]);
        System.out.println(x[1]);

        Object[] objArray = new Object[5];
        objArray[0] = new Object();
        objArray[1] = "hello";      // String is-a Object

        Number[] numArray = new Number[3];      // abstract element type
        numArray[0] = Integer.valueOf(1);       // concrete subtype
        numArray[1] = Byte.valueOf((byte) 2);
        System.out.println(numArray[0]);

        Runnable[] r = new Runnable[5];         // interface element type
        r[0] = new Thread();                    // Thread implements Runnable
        System.out.println(r[1]);               // never assigned -> null
    }
}

AdvancedReference-array assignments are checked partly at runtime. Because an Object[] can point at a String[], storing a non-String into it compiles but throws ArrayStoreException at runtime — this is Java's array covariance, and it is precisely the hole that generics were designed to close for collections.
Pitfall Declaring new Number[3] or new Runnable[5] is fine, but numArray[0] = new Number() will not compile — you cannot instantiate an abstract class or interface, so only concrete subtype objects can occupy the slots.

07. Assigning and reassigning array references

Objective: Understand that assigning one array variable to another shares the same object, and learn which array-reference types are assignment-compatible.

Writing int[] a = x; does not copy the array — it makes a point at the very same object x points at. A change made through either reference is visible through the other, since there is only one underlying array. When you later reassign b = c;, b abandons its original array (which, if nothing else references it, becomes eligible for garbage collection) and now points where c points. Assigning them back and forth changes nothing once they already share an object.

Reference compatibility for arrays mirrors class inheritance, but only for reference element types. A String[] can be assigned to an Object[] variable because String is a subclass of Object. Primitive array types, however, have no such relationship: [S (short array) and [I (int array) are unrelated classes, so a short[] cannot be assigned to an int[] variable even though a single short value promotes to int without issue.

  • Array assignment shares the object; no copy is made.

01. Collections Overview

Objective: Understand what a Java collection is, why the Collections API exists, and how collections compare to plain arrays so you can pick the right container for a task.

A collection (sometimes called a container) is simply a group of individual objects treated as a single unit. Collections were designed to solve the two headaches you meet with arrays. First, an array's length is fixed the moment you declare it, whereas a collection is growable: you never state a size up front, and it expands or shrinks on demand as you add or remove elements. Second, an array can only store one homogeneous type, while a collection can hold homogeneous data (when you use generics) or heterogeneous data (when you do not).

The trade-offs go beyond flexibility. From a memory standpoint, an array of size 100 that only ever holds 10 elements wastes 90 slots, so arrays are a poor fit when you cannot predict the count; collections allocate roughly what they need and resize as they fill. From a performance standpoint, arrays win — they are the leaner choice when the size is known ahead of time and you do not need any richer underlying data structure, because an array is itself a fundamental data structure. Each collection type, by contrast, is a class built on top of a standard data structure (an ArrayList, for example, is backed by an array internally) and ships with many predefined utility methods such as sorting, searching, and inserting. Arrays offer none of those built-in operations, so collections meaningfully cut development time. One last distinction: arrays can store both primitives and object references, but collections store objects only — primitives are autoboxed into their wrapper types.

  • A collection is a resizable group of objects; no fixed size is declared at creation.
  • Generics make a collection homogeneous (List<String>); raw usage lets it stay heterogeneous.
  • Arrays waste memory when over-sized; collections grow and shrink to fit.
  • Arrays are faster and simpler when the size is known in advance and no extra structure is needed.
  • Collections are classes over standard data structures with ready-made methods (sort, search, add, remove).
  • Arrays hold primitives or objects; collections hold objects only (primitives get autoboxed).

01. Collection Framework

Objective: Understand what the Collections Framework is, the three parts it is built from, and how its core interfaces and classes are organised so you can pick the right type for any job.

A collection is a group of individual objects treated as one single unit. In everyday life a bag of coins or a bundle of banknotes is a collection; in programming it is a group of discrete objects held together. The Java Collections Framework is a coherent library that bundles the interfaces, implementation classes, and algorithms for this task into one place, so a programmer does not reinvent data structures and can process data with ready-made features. It rests on three pillars: interfaces (the abstract contracts such as List, Set, Map), implementations (concrete classes like ArrayList, HashSet, HashMap that wrap an underlying data structure), and algorithms (reusable operations such as sorting and searching provided through utility methods).

The framework was introduced in Java 1.2 and is anchored by a set of root interfaces. The top interface is Collection, which declares the most common operations that apply to any group of objects. Three sub-interfaces refine it — List, Set, and Queue:

  • List — an ordered group where insertion order is preserved and duplicates are allowed. Implementations: ArrayList, LinkedList, and the legacy Vector (whose subclass is Stack).
  • Set — a group where order is not guaranteed and duplicates are not allowed. Implementation: HashSet, with subclass LinkedHashSet (added in 1.4, which does preserve insertion order).
    • SortedSet → NavigableSet — a refinement of Set (not of Collection directly) kept in a sorting order, no duplicates; NavigableSet extends SortedSet to add navigation methods (lower, higher, floor, ceiling). Both are implemented by TreeSet.
  • Queue — holds objects prior to processing, typically in first-in-first-out order. Implementations include PriorityQueue and the BlockingQueue family (LinkedBlockingQueue, PriorityBlockingQueue).

These Collection-based interfaces all model a group of single objects. When you instead need key/value pairs, you use Map. Crucially, Map is not a child of Collection, yet it is still part of the Collections Framework. Its main implementation is HashMap, with children/relatives LinkedHashMap, WeakHashMap, and IdentityHashMap. The legacy Hashtable also implements Map but additionally extends the abstract Dictionary class, and its subclass is Properties. The sorted view of a map is SortedMap → NavigableMap, implemented by TreeMap, which keeps entries ordered by their keys.

Vector, Stack, Hashtable, and Dictionary are legacy types: they existed before Java 1.2 and were retrofitted into the framework, which is why they carry older, synchronised designs.

Two classic "gotcha" facts fall straight out of the hierarchy. Objective: Understand what the root Collection interface represents and learn the common methods that every list, set, and queue inherits from it.

Collection is the first and most general interface in the collections hierarchy. It models a group of individual (discrete) objects. Java deliberately ships no concrete class that implements Collection directly; instead it provides more specialised sub-interfaces such as List, Set, and Queue, and it is those that concrete classes implement. (The abstract skeletal class AbstractCollection does implement Collection, but you never instantiate it directly.) Because these methods are declared on Collection itself, every class throughout the hierarchy makes them available, so the same vocabulary works for any collection type you pick.

The core methods fall into a few groups. Adding: add(o) inserts a single element and returns true if the collection changed; for a Set (which forbids duplicates) adding an element already present returns false, whereas a List permits duplicates and returns true. addAll(c) copies every element of another collection into the invoking one. Removing: remove(o) drops a single element, removeAll(c) drops everything the collection shares with c (a set difference), retainAll(c) keeps only the elements also present in c (an intersection), and clear() empties the collection entirely. Querying: contains(o) and containsAll(c) test membership, isEmpty() reports whether there are zero elements, and size() returns the element count. Traversal & conversion: iterator() hands back an Iterator for looping, and toArray() converts the collection into a plain array.

  • Collection is the root interface; it has no direct concrete implementation.
  • Real classes implement the sub-interfaces List, Set, and Queue.
  • add returns false when a Set rejects a duplicate; a List accepts duplicates and returns true.
  • Bulk operations: addAll, removeAll (difference), retainAll (intersection), clear.
  • Inspection: contains, containsAll, isEmpty, size.
  • iterator() gives an Iterator; toArray() exports to an array.

The names differ by a single letter but the concepts are unrelated in kind. Collection (singular) is the root-level interface of the framework — the base type from which List, Set, and Queue descend and from which most collection classes ultimately inherit. Collections (plural) is a utility class: a final class full of static helper methods that operate on collection objects. Both live in the java.util package, which is why they show up together in imports and cause confusion.

Where Collection defines the shape of a collection, Collections provides ready-made algorithms you invoke on one: sort, reverse, shuffle, binarySearch, max, min, disjoint, frequency, and many more. In short: Collection is the base interface for lists, sets, and queues; Collections is the toolbox of searching, sorting, and rearranging utilities that work across all of them.

  • Collection is an interface; Collections is a utility class — one letter, two different things.
  • Both reside in java.util.
  • Collections exposes only static methods — you call them as Collections.sort(list), never by instantiating it.
  • Typical helpers: sort, reverse, shuffle, binarySearch, max, min, disjoint, frequency.

01. Generics Overview

Objective: Understand why generics exist by contrasting a fixed-length, type-safe array with a pre-generics ArrayList, and see how raw collections postpone type errors from compile time to run time.

Java is a statically typed language: the type of every variable and expression is fixed and known at compile time. Arrays lean on that guarantee. An array is a homogeneous, fixed-length container — when you declare String[] names = new String[20], the compiler permits only String values in it. Trying to drop an Employee into a String[] is rejected during compilation with "incompatible types". That built-in restriction is exactly what we mean by type safety: the container is bound to a single element type, so an entire class of mistakes is caught before the program ever runs.

So why not use arrays everywhere? Because an array's length is fixed at creation and cannot grow afterwards — you must commit to a size before filling it (even if that size is computed at run time). Real programs rarely know in advance how many elements they will hold, so the Collections framework adds growable structures such as ArrayList, where you never declare a capacity. The catch is that a raw ArrayList (no type parameter) stores plain Object references. Since every class in Java ultimately extends Object, a raw list will happily accept a String and an Employee side by side — it is a heterogeneous bag. (Primitives such as int cannot be stored directly; the list holds object references, so a primitive is autoboxed into its wrapper class Integer — since Java 5 the compiler does this for you.)

Storing mixed types is only half the trouble. When you pull an element back out with get, a raw list returns Object, so the compiler forces you to cast to the type you expect. Casting means treating a value of one declared type as another. If you cast correctly, all is well. But if you cast an element to the wrong type — say you know a slot really holds a String yet you cast it to Employee — the compiler stays silent and the mistake only surfaces at run time as a ClassCastException. Arrays never put you in this position: because the element type is known, retrieval needs no cast at all. This late failure with raw collections is the pain point that motivated generics in Java 5; the next section shows how a type parameter like ArrayList<String> restores compile-time safety and removes the casts.

  • Arrays are homogeneous and fixed-length: type-safe by construction, but you must know the size in advance.
  • Java is statically typed, so an array rejects a wrong-typed element at compile time.
  • A raw ArrayList holds Object references, so it accepts any type and is not type-safe.
  • Retrieving from a raw list returns Object and requires a manual cast.
  • A wrong cast compiles cleanly but throws ClassCastException at run time — errors caught at compile time are always preferable.
  • Collections were not type-safe before Java 5; generics were introduced to fix this.

01. The List Interface

Objective: Understand what the List interface guarantees and why every list class shares the same core behaviour.

Introduced in Java 1.2, List is a child interface of Collection. Because it extends Collection it inherits methods such as size(), isEmpty(), contains(), iterator(), and toArray(), and adds its own positional operations: get(index), set(index, element), add(index, element), remove(index), indexOf(), lastIndexOf(), and listIterator(). The defining characteristic of a list is that it is an ordered collection that allows duplicates: "ordered" means insertion order is preserved and every element is reachable by its index. Since List is an interface you cannot instantiate it directly — you reference it and assign an object of one of its implementation classes (ArrayList, LinkedList, Vector, Stack). Consequently all four classes are ordered, index-based, and duplicate-friendly, and all expose the full set of List methods.

  • List extends Collection and was added in Java 1.2.
  • Ordered = insertion order preserved and accessible by index (positional access).
  • Duplicates are allowed (the same value may appear multiple times).
  • Coding to the interface (List<T> l = new ArrayList<>();) lets you swap implementations freely.

02. ArrayList

Objective: Create and use an ArrayList, understand its resizing behaviour, and see why generics are preferred over raw types.

ArrayList (in java.util) is a resizable array — its backing data structure is a plain array. It preserves insertion order and accepts duplicate and heterogeneous objects. A no-arg new ArrayList<>() conceptually starts with a default capacity of 10 (in JDK 17 the backing array is actually length 0 until the first add, which then allocates the 10 slots — a lazy-init optimisation). When the array fills up, it grows by about 1.5×, the old elements are copied into the new larger array, and the old array is discarded for garbage collection. You can pass an initial capacity (new ArrayList<>(30)) to avoid repeated growth if you know the size in advance, or pass an existing collection to build an equivalent list. Core methods: add(e) and add(index, e), two overloads of remove — remove(int index) and remove(Object o) — plus get(index). The original demo uses a raw ArrayList, which the compiler flags with "ArrayList is a raw type…". Raw types defer type errors to runtime and force manual casts, so once you know the element type, prefer the generic form ArrayList<String>.

  • Backing structure is an array; default capacity 10 (allocated lazily on first add), grows by ~1.5× when full.
  • remove(int) deletes by index; remove(Object) deletes by value — watch the overload.
  • Raw types compile with a warning and risk runtime ClassCastException; generics make it type-safe and cast-free.
  • Constructors: no-arg, initial-capacity, and copy-from-collection.

03. Important Concepts of ArrayList

Objective: Learn the marker interfaces ArrayList implements and understand its performance profile for random access versus insert/delete.

Collections are used to hold and transfer groups of objects, sometimes across a network — which requires objects to be Serializable. That is why the collection classes, including ArrayList, implement Serializable. They also implement Cloneable, allowing duplicate copies for safe experimentation and recovery. Additionally, ArrayList implements RandomAccess, an empty marker interface in java.util that signals the class supports fast, constant-time positional access. Because its elements sit in contiguous memory, ArrayList (and Vector) is the best choice when the frequent operation is index-based / random retrieval: get(i) is O(1). (Searching for a value with contains/indexOf is still O(n), but the contiguous layout is cache-friendly, so it beats a linked list in practice.) It is the worst choice for frequent insertion or deletion in the middle: inserting or removing an element forces every following element to shift by one position, and if the array is full this shifting coincides with allocating and copying into a brand-new larger array — an expensive combination.

  • ArrayList implements three interfaces: Serializable, Cloneable, and RandomAccess.
  • RandomAccess is a marker interface (no methods) advertising fast index access.
  • Best for index retrieval (get(i) is O(1)); worst for mid-list insert/delete because of element shifting.
  • Growth (copy to a bigger array) plus shifting is why mid-list writes are slow.

Pitfall Choosing ArrayList for a workload dominated by inserts/removes at the front or middle silently makes it O(n) per operation. Match the structure to the dominant operation.

04. LinkedList

Objective: Understand the node-based structure of LinkedList and use its list and deque-style methods.

LinkedList is a doubly-linked structure where each element is a separate node holding a data part plus address/pointer(s) to the neighbouring node(s). It allows duplicates, heterogeneous objects, and null values, and implements List, so it shares the ordered/duplicate semantics. It provides the usual list methods — add(e), add(index, e) (inserts and shifts, does not overwrite), set(index, e) (replaces in place), and remove — plus position-specific helpers addFirst, addLast, getFirst, getLast, removeFirst, and removeLast. Those first/last methods exist because LinkedList is well suited to building stacks (LIFO) and queues (FIFO). Note the difference between set and add(index, …): set overwrites the element already at that index, whereas add(index, …) inserts a new element and pushes existing ones one place to the right.

  • Each node stores data + link(s); no requirement for contiguous memory.
  • Allows duplicates, heterogeneous elements, and null.
  • set(i, e) replaces; add(i, e) inserts and shifts.
  • Extra methods (addFirst/Last, getFirst/Last, removeFirst/Last) support stack and queue use.
import java.util.LinkedList;

        System.out.println(l);

        System.out.println(l);

        System.out.println(l);

05. LinkedList vs ArrayList

Objective: Decide between ArrayList and LinkedList based on the dominant operation.

ArrayList is poor for insertion and deletion because a single change can shift many elements. LinkedList excels at insertion and deletion in the middle: you simply re-point a node's link to splice a node in or out, with no bulk shifting. The trade-off reverses for search. ArrayList stores elements in contiguous memory and supports fast random access, so retrieval is quick; LinkedList can only reach a node by walking from its predecessor, so finding an element means visiting nodes one by one from the head — O(n) for a target near the tail. In short: choose LinkedList when insertions/deletions dominate, and ArrayList when searching/random access dominates. The right choice is dictated by your program's most frequent operation.

  • LinkedList wins on mid-list insert/delete — just relink nodes, no shifting.
  • ArrayList wins on search/random access — contiguous memory, O(1) indexing.
  • LinkedList search is O(n): the JVM walks node-by-node from the head.
  • Pick by the dominant operation, not by habit.

06. Vector

Objective: Understand Vector as a synchronized, legacy resizable array and how its growth and capacity differ from ArrayList.

Vector is another resizable-array implementation of List — like ArrayList, its backing structure is an array, it preserves insertion order, allows duplicates and heterogeneous objects, and implements Serializable, Cloneable, and RandomAccess. The key distinction: all of Vector's methods are synchronized, so only one thread can operate on it at a time, making it thread-safe. ArrayList is unsynchronized and therefore faster but not thread-safe. Vector is a legacy class present since Java 1.0 (retrofitted into the List hierarchy in 1.2). Growth also differs: a full ArrayList grows by ~1.5×, but a full Vector doubles its capacity by default. Constructors let you tune this: new Vector(initialCapacity) sets the starting size, and new Vector(initialCapacity, capacityIncrement) grows by a fixed increment instead of doubling — useful to avoid wasting memory on very large vectors. A copy constructor from any Collection is also available. Because it is legacy, Vector carries duplicate methods: add/addElement, remove/removeElement, get/elementAt, plus firstElement, lastElement, size(), and capacity().

  • Resizable array like ArrayList, but all methods are synchronized (thread-safe, slower).
  • Legacy class from Java 1.0; a full vector doubles its capacity (vs ~1.5× for ArrayList) unless a capacityIncrement is set.
  • size() = elements currently stored; capacity() = total slots allocated.
  • Legacy synonyms exist: addElement/add, elementAt/get, removeElement/remove.

07. Stack

Objective: Use the Stack class and its LIFO operations push, pop, peek, empty, and search.

Stack is a child class of Vector that models the last-in-first-out (LIFO) data structure: the most recently pushed element is the first to come out. It inherits all of Vector's traits (ordered, duplicates allowed, heterogeneous objects, synchronized) and adds five stack-specific methods. push(e) places an element on top; pop() removes and returns the top element; peek() returns the top element without removing it; empty() returns true when the stack has no elements; and search(o) returns the 1-based offset of an element measured from the top, or -1 if it is absent. Stack has a single no-arg constructor.

  • Stack extends Vector and is LIFO.
  • pop() removes and returns the top; peek() only reads it.
  • empty() tests for emptiness; search(o) gives a 1-based position from the top (or -1).
  • Only one constructor: new Stack().
import java.util.Stack;

08. Cursors: Enumeration, Iterator, and ListIterator

Objective: Traverse collections with the three cursor types and know their relative capabilities.

A cursor lets you walk through a collection's elements one at a time. Java offers three. Enumeration is the oldest, works only with legacy classes (like Vector), is obtained with elements(), and is read-only with two methods: hasMoreElements() and nextElement(). Iterator is the universal cursor for any Collection, obtained with iterator(); it adds safe removal during traversal via hasNext(), next(), and remove(), but only moves forward. ListIterator is specific to List objects, obtained with listIterator(); it is bidirectional and the most powerful — beyond hasNext()/next() it offers hasPrevious()/previous(), plus add(), remove(), and set() to modify the list during iteration. The three demos below show each cursor in action.

  • Enumeration: legacy-only, read-only, forward — hasMoreElements()/nextElement().
  • Iterator: universal for any collection, forward, supports remove().
  • ListIterator: lists only, bidirectional, supports add(), remove(), set().
  • Obtain them via elements(), iterator(), and listIterator() respectively.

09. Section Summary

Objective: Consolidate where each list class fits in the Collection API and when to use it.

This section covered the List interface and its implementation classes together with their underlying data structures. Use List whenever you need to represent a group of objects as one entity with insertion order preserved and duplicates allowed. Vector and Stack are legacy classes from Java 1.0 that were folded into the List hierarchy in 1.2.   • List = ordered, indexable, duplicate-allowing group of objects.
  • ArrayList/Vector are array-backed (fast index access); LinkedList is node-backed (fast insert/delete at a held position).
  • Vector and Stack are synchronized legacy classes; Stack is LIFO.
  • Cursors: Enumeration (legacy), Iterator (universal), ListIterator (list-only, bidirectional).

8. Set, its implementation classes and more

The Set interface models a group of distinct objects held as a single unit: duplicates are rejected and there is no positional index. This section walks the whole Set family — HashSet, LinkedHashSet, the SortedSet and NavigableSet sub-interfaces, their TreeSet implementation, and the Comparable/Comparator interfaces that make ordered sets possible.

Run locally Each example is an ordinary class with a main method. Compile with javac HashSetDemo.java then run with java HashSetDemo (JDK 17+). 01. Introduction to Set

Objective: Understand what a Set represents, where it sits in the Collection hierarchy, and how it differs from List.

Set is the third major sub-interface of Collection (after we have already covered List). Unlike a List, a Set preserves neither insertion order nor an index, and it forbids duplicate elements. You reach for a Set whenever you want to represent a bunch of unique, unordered objects as one entity. Importantly, Set adds no new methods of its own — it simply inherits the general-purpose methods (add, remove, contains, iterator, size, and so on) declared in Collection.

The family tree: HashSet implements Set directly, and LinkedHashSet is a subclass of HashSet. Separately, SortedSet extends Set, NavigableSet extends SortedSet, and TreeSet is the concrete class implementing NavigableSet. A set can hold heterogeneous objects (different types), except when ordering is required.

  • No duplicates, no insertion order, no index-based access.
  • Set declares no methods beyond what Collection already provides.
  • Direct implementation: HashSet (and its child LinkedHashSet).
  • Sub-interfaces for ordering: SortedSet → NavigableSet → implemented by TreeSet.
  • A plain set can store heterogeneous elements; an ordered set cannot.
Pitfall Do not expect a Set to remember the order in which you inserted elements. Iterating a HashSet may print elements in a seemingly random order — if order matters, pick LinkedHashSet (insertion order) or TreeSet (sorted order).

02. HashSet

Objective: Learn how HashSet stores elements via hashing, the meaning of initial capacity and load factor, and the four constructors.

HashSet is the first implementation of Set; its underlying data structure is a hash table. Because elements are placed by hash code, membership tests (contains) are extremely fast — hashing is the go-to technique when searching is the dominant operation. When you add an object, the set computes the object's hash code and derives a bucket index (conceptually hashCode % tableSize); that index dictates where the element lands, which is exactly why insertion order is not preserved. HashSet forbids duplicates, allows heterogeneous elements, permits a single null, and implements Serializable and Cloneable.

Two knobs control resizing. Initial capacity defaults to 16. Load factor defaults to 0.75, meaning that once the table is ~75% full it rehashes into a new table of roughly double the capacity and the old table becomes eligible for garbage collection. The four constructors are: no-arg (capacity 16), capacity only, capacity + load factor, and a constructor that copies from any existing Collection.

  • Backed by a hash table; iteration order is not guaranteed.
  • Best when frequent lookups/searches dominate.
  • Default initial capacity 16, default load factor 0.75.
  • Growing past the load-factor threshold reallocates a larger table (~2×) and rehashes.
  • Allows one null and heterogeneous elements; no duplicates.
import java.util.ArrayList;

AdvancedChoosing an initial capacity large enough to hold your expected element count avoids repeated rehashing. 03. LinkedHashSet

Objective: See how LinkedHashSet keeps the speed of hashing while additionally preserving insertion order.

LinkedHashSet is a subclass of HashSet, introduced in Java 1.4, so it inherits every HashSet property — the one added guarantee is that insertion order is preserved. It achieves this because its underlying data structure is a combination of a hash table (for fast lookup and uniqueness) and a doubly linked list (which threads the elements in the order they were added). Use it when you need both uniqueness and predictable, insertion-ordered iteration.

The course demo shows three constructors: a no-arg one (capacity 16), one taking an initial capacity, and one taking capacity plus load factor. For completeness, LinkedHashSet in fact has a fourth constructor — a Collection copy constructor (like HashSet's) that builds an insertion-ordered set from any existing collection. As with HashSet, all usable methods come from Collection (add, addAll, iterator, toArray, and so on).

  • Child class of HashSet; all HashSet behavior still applies.
  • Extra guarantee: iteration follows insertion order.
  • Backed by hash table + doubly linked list.
  • Four constructors: default, capacity, capacity + load factor, and Collection (copy).

04. SortedSet

Objective: Understand the SortedSet sub-interface and its six navigation/range methods, backed by TreeSet.

SortedSet (Java 1.2) extends Set, so it inherits "no duplicates." Its distinguishing property is that elements are stored in a defined sort order rather than being unordered — with numbers this means ascending numeric order, with strings alphabetical order. Because it keeps a definite ordering, you can meaningfully ask for the first element, the last element, or a range. The implementation class is TreeSet; you typically program to the interface by assigning a TreeSet object to a SortedSet reference.

SortedSet adds six methods on top of Collection: first(), last(), headSet(o) (everything strictly before o), tailSet(o) (everything from o onward, inclusive), subSet(from, to) (from-inclusive, to-exclusive), and comparator() (returns null for natural ordering). Because sorting requires comparison, a SortedSet accepts only homogeneous elements — mixing, say, an Employee and a Student would leave the set unable to compare them.

  • Sub-interface of Set; keeps elements in sorted order.
  • Natural order: ascending for numbers, alphabetical for strings.
  • Six methods: first, last, headSet, tailSet, subSet, comparator.
  • subSet/tailSet include the lower bound; headSet/subSet exclude the upper bound.
  • Only homogeneous elements — heterogeneous objects cannot be ordered.

05. NavigableSet

Objective: Use NavigableSet's closest-match and retrieve-and-remove methods for navigating a sorted set.

NavigableSet is the child interface of SortedSet (added in Java 1.6), so all sorted-set properties still hold; it layers on navigation methods that report the closest match to a target value. The key methods are: floor(x) (greatest element ≤ x, else null), lower(x) (greatest element strictly < x), ceiling(x) (smallest element ≥ x), higher(x) (smallest element strictly > x), pollFirst() (retrieve and remove the smallest), pollLast() (retrieve and remove the largest), and descendingSet() (a view in reverse order). As always, the concrete class is TreeSet.

  • Extends SortedSet (Java 1.6); adds nearest-neighbor navigation.
  • floor/ceiling are inclusive of an exact match; lower/higher are strict.
  • pollFirst/pollLast both return and remove the boundary element.
  • descendingSet() returns the elements in reverse order.
  • Any of these return null when no qualifying element exists.

06. TreeSet

Objective: Understand TreeSet's balanced-tree storage, its four constructors, and why stored elements must be comparable and homogeneous.

TreeSet implements NavigableSet (hence also SortedSet and Set). Its backing structure is a balanced tree (a red-black tree): smaller values sort to the left and larger values sort to the right, and after each insert the tree rotates and recolors itself (red-black balancing) to keep its height O(log n). That self-balancing is why even already-sorted input keeps add/contains at O(log n), and why the first-inserted element is not necessarily the permanent root. Because insertion follows a sort order, elements must be homogeneous — inserting incompatible types triggers a ClassCastException at runtime. Older tutorials say a null may be inserted as the very first element of an empty TreeSet; on modern JDKs (7+) that is no longer true for natural ordering — even the first add(null) throws NullPointerException, because the tree type-checks the key by comparing it against itself. Treat TreeSet as null-hostile.

Four constructors exist: no-arg (natural ordering), one taking a Comparator (custom ordering), one taking a SortedSet, and one taking any Collection (for interchangeability, e.g. building a sorted set from a HashSet). Prefer the generic form TreeSet<Integer> over a raw type. Crucially, whatever you store with natural ordering must implement Comparable — this is why Integer, String, etc. work out of the box, and why a custom class like Employee must implement Comparable and override compareTo.

  • Backed by a self-balancing (red-black) tree; keeps elements sorted, no duplicates.
  • Elements must be homogeneous and mutually comparable, or you get ClassCastException.
  • Four constructors: default, Comparator, SortedSet, and Collection.
  • For natural ordering, the element type must implement Comparable.
  • Null-hostile — do not rely on storing null.

07. Comparable(I) and Comparator(I)

Objective: Distinguish Comparable (natural ordering) from Comparator (custom ordering) and implement a descending TreeSet.

Only Comparable objects can be placed in a natural-ordered TreeSet. That is why String works but StringBuffer does not — StringBuffer is not Comparable. Comparable lives in java.lang and declares a single method, int compareTo(Object), used for the default natural sort order. When you add to a TreeSet, the JVM calls compareTo to decide placement: a positive result means "greater" (go right of the compared node), negative means "smaller" (go left), and 0 means "equal" (a duplicate — dropped). For example adding M, then R ("R".compareTo("M") is positive so R goes to one side), then K (negative, the other side) builds the balanced tree.

When the natural order isn't what you want — say you want descending order or to sort Employee by salary — you supply a Comparator instead. Comparator lives in java.util and declares two methods, compare(o1, o2) and equals(Object). You only need to implement compare: every class already inherits an equals from Object, so the compiler doesn't force you to write it. Pass your comparator to the TreeSet constructor and the JVM uses your compare instead of compareTo. The demo below reverses the usual signs (+1 when the first value is smaller, -1 when larger) to obtain descending order.

  • Comparable (java.lang, one method compareTo) = natural ordering, baked into the class.
  • Comparator (java.util, compare + inherited equals) = external, custom ordering.
  • Return value convention: negative → less, 0 → equal (duplicate), positive → greater.
  • No comparator passed → JVM uses compareTo; comparator passed → JVM uses compare.
  • Flipping the signs in compare flips the sort direction.
import java.util.Comparator;
import java.util.TreeSet;

// Custom order: descending, by flipping the usual signs
class MySorting implements Comparator<Integer> {
    @Override public int compare(Integer d1, Integer d2) {
        if (d1 < d2) return +1;   // smaller -> treat as "after"
        if (d1 > d2) return -1;   // larger  -> treat as "before"
        return 0;                 // equal   -> duplicate, dropped
    }
}

public class ComparableDemo {
    public static void main(String[] args) {
        // No comparator -> natural ascending order:
        // TreeSet<Integer> data = new TreeSet<>();

        // Comparator supplied -> custom descending order:
        TreeSet<Integer> data = new TreeSet<>(new MySorting());
        data.add(152); data.add(185); data.add(254);
        data.add(10);  data.add(132); data.add(10); // duplicate 10 ignored
        System.out.println(data);
    }
}

AdvancedModern code rarely writes a whole comparator class. The same descending order is a one-liner: new TreeSet<>(Comparator.reverseOrder()), and multi-key ordering composes fluently, e.g. Comparator.comparingInt(Employee::getSalary).thenComparing(Employee::getName). A subtle rule: for a TreeSet, element equality is decided by compareTo/compare returning 0 — not by equals — so a comparator that returns 0 for two "different" objects will silently discard one. Also avoid the old return d1 - d2 trick for int comparators: it overflows for large-magnitude values; use Integer.compare(d1, d2).
Pitfall A comparator that is inconsistent (returns 0 for values you consider distinct, like comparing only one field that ties) causes the TreeSet to drop elements you expected to keep. Make sure compare returns non-zero for every pair that should coexist.

08. Summary

Objective: Consolidate the entire Set family — implementations, sub-interfaces, ordering, and when to choose each.

We began with Set, the child interface of Collection that forbids duplicates and does not preserve insertion order. Being an interface, it needs concrete classes: HashSet (hash-table backed, fastest search, unordered) and its child LinkedHashSet (hash table + linked list, insertion order preserved). We then covered the two ordering sub-interfaces: SortedSet (sorts on insertion, holds only homogeneous Comparable elements) and its child NavigableSet (added in Java 1.6, with closest-match navigation), implemented by TreeSet. Finally we compared Comparable (natural ordering via compareTo) with Comparator (custom ordering via compare). All of these represent a group of unique objects as one entity.

  • Fastest lookups, order irrelevant → HashSet.
  • Uniqueness + insertion order → LinkedHashSet.
  • Uniqueness + sorted order / range queries → TreeSet (via SortedSet/NavigableSet).
  • Natural order → implement Comparable; custom order → supply a Comparator.
  • Ordered sets require homogeneous, comparable elements.

The Queue interface models a group of elements that are waiting to be processed. It is open at both ends — one for inserting and one for removing — and by default honours first-in, first-out (FIFO) order.

01. Queue Interface

Objective: Understand what the Queue interface represents, where it sits in the collection hierarchy, and the six queue-specific methods it adds on top of Collection.

Queue is the fourth core interface of the Collections Framework (after List and Set), introduced in Java 1.5. It extends Collection, so every Collection method is available on a queue as well. While a List or Set simply holds objects as a single group, a Queue specifically represents objects that are lined up and about to be processed. A queue is open at both ends: elements enter at the tail and leave from the head. By default it follows the FIFO rule — the element inserted first is the first one to be taken out — which is the opposite of the legacy Stack class that works last-in, first-out (LIFO).

On top of the inherited Collection methods, Queue defines six methods that come in two flavours: one that throws an exception on failure and one that returns a special value (null or false) instead.

  • Queue extends Collection and was added in Java 1.5.
  • Insertion happens at the tail, removal at the head; default behaviour is FIFO.
  • add(e) / offer(e) — append at the tail (or by priority in a PriorityQueue). add throws on capacity failure; offer returns false.
  • remove() / poll() — remove and return the head. remove throws NoSuchElementException when empty; poll returns null.
  • element() / peek() — return the head without removing it. element throws NoSuchElementException when empty; peek returns null.
  • Implementations include LinkedList (attached to the queue family in 1.5), PriorityQueue, and the BlockingQueue hierarchy (PriorityBlockingQueue, LinkedBlockingQueue).
import java.util.LinkedList;

        System.out.println("Remaining:   " + line);

02. PriorityQueue

Objective: Learn how PriorityQueue orders elements by priority rather than by arrival, its constructors, and its rules about ordering, duplicates and nulls.

PriorityQueue is a concrete implementation of Queue that hands elements back in priority order instead of insertion order. Priority comes either from the elements' natural ordering (their Comparable implementation) or from a Comparator you supply. Because elements are arranged by priority, the original insertion order is not preserved. When you rely on the default natural ordering, all elements must be mutually comparable, so heterogeneous (mixed-type) objects are rejected with a ClassCastException; supplying your own Comparator lifts that restriction because you define how any two elements compare. Inserting null is never allowed and throws NullPointerException.

The class offers several constructors: a no-arg one (default capacity 11, natural ordering), one taking an initial capacity, one taking capacity plus a Comparator, one taking a SortedSet, and one taking any Collection — the last two build a priority queue equivalent to the supplied structure. (Since Java 8 there is also a Comparator-only constructor.) All the Queue and Collection methods (offer, poll, peek, size, remove, isEmpty, and so on) are available.

  • Elements leave in priority order, not insertion order; insertion order is not preserved.
  • Priority = natural ordering (Comparable) by default, or a custom Comparator.
  • Duplicates are permitted (a PriorityQueue is not a Set); only null is rejected — it throws NullPointerException.
  • With natural ordering, elements must be mutually comparable (no heterogeneous types); a Comparator removes that constraint.
  • Default initial capacity is 11; it grows automatically as needed.
  • Backed by a binary heap — peek() is the minimum (highest-priority) element, but iterating does not visit elements in sorted order.
Course correction The lecture states that duplicates are not permitted in a PriorityQueue. That is not accurate for the JDK class: a PriorityQueue is not a Set, so equal elements are freely stored side by side in the heap (the second example below inserts 30 twice and both survive). The only value it truly rejects is null.
import java.util.PriorityQueue;

02. Hashing

Objective: Learn what a hash function is, how hashCode()/equals() cooperate, and how collisions are resolved.

Searching an unsorted array is linear — you may inspect every element. Hashing replaces that scan with (ideally) a single probe: a hash function maps an object to an index in a backing array (the hash table). In Java that function is Object.hashCode(), which yields an integer for any object (for example, Integer.valueOf(12).hashCode() is 12 and 'A' hashes toward 65). The index is derived by taking the hash modulo the table size. When two distinct objects land on the same index, that is a collision; Java resolves it by chaining the colliding entries in a linked list (which converts to a balanced tree once a bucket grows large).

  • Equal objects (per equals) must return the same hashCode; the reverse need not hold.
  • Always override hashCode and equals together, using fields that uniquely identify the object.
  • Bucket index = hash % tableSize (conceptually); collisions chain into a list/tree at that bucket.
  • A good hash spreads objects evenly, keeping lookups close to O(1).
import java.util.Objects;

public class Student {
    final int id;
    final String name;

    Student(int id, String name) { this.id = id; this.name = name; }

    @Override public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Student s)) return false;
        return id == s.id && Objects.equals(name, s.name);
    }
    @Override public int hashCode() {
        return Objects.hash(id, name);   // consistent with equals
    }

    public static void main(String[] args) {
        Student a = new Student(1, "Ann");
        Student b = new Student(1, "Ann");
        System.out.println(a.equals(b));                 // logically equal
        System.out.println(a.hashCode() == b.hashCode()); // so hashes match
    }
}

03. HashMap - Introduction

Objective: Meet HashMap, its backing hash table, and its defining characteristics.

HashMap is the first and most common implementation of Map. Its underlying data structure is a hash table — internally an array of Node objects, where each node holds the key's hash, the key, the value, and a reference to the next node (a linked list for collision chaining). Because placement is driven by the key's hash, insertion order is not preserved. It permits one null key and any number of null values, and accepts heterogeneous keys and values. HashMap implements Serializable and Cloneable and extends AbstractMap; it is the best choice for fast lookups.

  • Backing structure is a hash table (array of Node = hash, key, value, next).
  • No guaranteed order; unique keys, duplicate values allowed.
  • At most one null key, many null values.
  • Extends AbstractMap; implements Serializable, Cloneable.

04. HashMap - Internal Working

Objective: Trace how put places entries into buckets, how collisions chain, and how the common methods behave.

When you call put, the map hashes the key, reduces that hash to a bucket index, and stores a Node there. A fresh map starts with 16 buckets and a load factor of 0.75, giving a resize threshold of 12 (0.75 × 16); crossing it doubles the table. If two keys hash to the same index (the classic "FB"/"Ea" pair, whose string hashes both equal 2236), the second node is linked after the first in that bucket. A null key always maps to index 0. Re-putting an existing key replaces its value rather than adding a duplicate. The example below exercises the everyday methods; because it is a HashMap, the printed order is unspecified.

  • Default capacity 16, load factor 0.75, threshold 12 → then resize (double).
  • put on an existing key overwrites the value and returns the old one.
  • Colliding keys chain in one bucket; a null key sits at index 0.
  • keySet, values, entrySet, containsKey, remove(k,v), computeIfAbsent cover most needs.

05. Difference Between HashMap and Hashtable

Objective: Contrast the modern HashMap with the legacy Hashtable across synchronization, nulls, and lineage.

Both store key–value pairs using hashing, but they differ in important ways. HashMap is unsynchronized and therefore not thread-safe — concurrent access can corrupt data — but that makes it faster because threads never block. Hashtable synchronizes every method, so it is thread-safe but slower, since only one thread operates at a time behind a lock. HashMap allows one null key and many null values; Hashtable forbids null entirely. HashMap arrived in Java 1.2 with the Collections Framework and extends AbstractMap, whereas Hashtable is a legacy class from Java 1.0 that extends Dictionary. You can synchronize a HashMap when needed via Collections.synchronizedMap(map).

  • HashMap: unsynchronized, fast, allows one null key + null values.
  • Hashtable: fully synchronized, slower, no nulls at all.
  • Wrap with Collections.synchronizedMap (or use ConcurrentHashMap) for thread safety.
  • Lineage: HashMap → 1.2, extends AbstractMap; Hashtable → 1.0 legacy, extends Dictionary.
import java.util.*;

        try {
            ht.put(null, "boom");      // NOT allowed
        } catch (NullPointerException e) {

  • Iteration follows insertion order (unlike HashMap).
  • Introduced in Java 1.4; ideal for caches.

07. IdentityHashMap

Objective: Understand how IdentityHashMap compares keys by reference (==) rather than by equals.

IdentityHashMap looks and behaves like HashMap — same methods, same key–value storage — with one crucial twist in how it decides whether two keys are “the same.” A normal HashMap uses equals() to detect duplicate keys, so two Integer(10) objects count as one key. IdentityHashMap instead uses reference equality (==): two keys are duplicates only if they are literally the same object. So two distinct new Integer(10) objects, equal by value but different references, are stored as two separate entries. Use it when identity, not logical equality, is what matters.

  • HashMap dedupes keys with equals; IdentityHashMap uses ==.
  • Equal-by-value but different-reference keys create two entries here.
  • Use when object identity (not value) defines a key.
import java.util.IdentityHashMap;

08. WeakHashMap

Objective: Understand how WeakHashMap lets the garbage collector reclaim keys that are otherwise unreferenced.

In a regular HashMap, a key object stays alive as long as it is in the map, even if nothing else references it — the map holds a strong reference, so it “wins” over the garbage collector. WeakHashMap reverses this: it holds its keys through weak references, so once a key has no other strong references it becomes eligible for collection, and its entry is automatically dropped from the map. Here the GC dominates. Unlike HashMap, WeakHashMap implements only Map — not Serializable or Cloneable — so it has no clone() method and no serialized state.

  • HashMap keeps keys via strong references; WeakHashMap via weak references.
  • A key with no external strong reference is GC'd and its entry vanishes.
  • WeakHashMap does not implement Serializable/Cloneable.
import java.util.WeakHashMap;

public class WeakHashMapDemo {
    static class Temp {
        @Override public String toString() { return "temp"; }
    }

    public static void main(String[] args) throws InterruptedException {
        WeakHashMap<Temp, String> m = new WeakHashMap<>();
        Temp t = new Temp();
        m.put(t, "element");
        System.out.println(m);       // {temp=element}

        t = null;                    // key now unreferenced elsewhere
        System.gc();                 // request collection
        Thread.sleep(3000);          // give the GC time to run
        System.out.println(m);       // {} - entry removed
    }
}

09. SortedMap

Objective: Learn the SortedMap interface, which keeps entries in ascending key order, and its range/endpoint methods.

SortedMap is a sub-interface of Map that maintains its entries sorted by key in ascending order — note the ordering is on keys, never on values. Ordering follows either the keys' natural ordering (Comparable) or a supplied Comparator. It disallows null keys. On top of the inherited map methods it adds ordering-aware operations: comparator(), firstKey()/lastKey(), and the range views headMap(toKey) (keys strictly less than toKey), tailMap(fromKey) (keys ≥ fromKey), and subMap(fromKey, toKey).

  • Entries sorted ascending by key (natural order or a Comparator).
  • No null keys.
  • Adds firstKey, lastKey, headMap, tailMap, subMap, comparator.
  • Its implementation class is TreeMap (via the NavigableMap sub-interface).
import java.util.SortedMap;

10. NavigableMap

Objective: Master NavigableMap's navigation methods for finding neighbouring keys and building fine-grained range views.

NavigableMap is a sub-interface of SortedMap that adds navigation methods — the map analogue of NavigableSet. Its implementation class is again TreeMap. The methods come in families: closest-key lookups (ceilingKey/ceilingEntry for ≥, floorKey for ≤, higherKey for strictly >, lowerKey for strictly <), reverse views (descendingKeySet, descendingMap), and inclusive-bound range views (headMap/tailMap/subMap with a boolean to include or exclude the endpoint). By convention a range's start bound is included and its end bound excluded unless you say otherwise.

  • ceiling = smallest key ≥ arg; floor = largest key ≤ arg.
  • higher = smallest key strictly > arg; lower = largest key strictly < arg.
  • ...Key returns the key; ...Entry returns the whole pair.
  • Boolean overloads of headMap/tailMap/subMap control endpoint inclusion.

11. TreeMap

Objective: Understand TreeMap's red–black-tree backing, its four constructors, and natural vs custom key ordering.

TreeMap is the concrete implementation of NavigableMap, SortedMap, and Map, extending AbstractMap. Its underlying data structure is a red–black tree — a self-balancing binary search tree where every node is coloured red or black, keeping insert/delete/search at O(log n) even under frequent updates. Insertion order is not preserved; entries are always ordered by key. Keys must be unique; values may repeat. With natural ordering the keys must be mutually comparable and homogeneous, or you get a ClassCastException; with a supplied Comparator that restriction relaxes. Since Java 1.7, a TreeMap rejects a null key outright (older versions allowed it only into an empty map), though values may still be null. It has four constructors: no-arg (natural order), Comparator-based, copy-from-SortedMap, and copy-from-Map.

  • Backed by a self-balancing red–black tree → O(log n) operations.
  • Sorted by key; natural order needs comparable, homogeneous keys.
  • Custom Comparator allows heterogeneous keys and any order.
  • No null keys (since 1.7); null values permitted.
import java.util.Comparator;
import java.util.TreeMap;

public class TreeMapDemo {
    public static void main(String[] args) {
        // 1) Natural ordering of Integer keys
        TreeMap<Integer, String> m = new TreeMap<>();
        m.put(23, "");
        m.put(10, "");
        m.put(10, "Basics");   // duplicate key -> value overwritten
        m.put(11, "Strong");
        System.out.println(m);

        // 2) Custom comparator: reverse alphabetical by String key
        Comparator<String> reverse = (a, b) -> b.compareTo(a);
        TreeMap<String, Integer> m1 = new TreeMap<>(reverse);
        m1.put("John", 10);
        m1.put("Shiva", 10);
        m1.put("Rohn", 10);
        System.out.println(m1);
    }
}

12. Summary

Objective: Consolidate the whole Map family into a single mental model for choosing the right implementation.

We began with Map itself — not a child of Collection but part of the same framework, representing data as key–value pairs. We covered the hashing technique that underpins the hash-based implementations, then the concrete classes: HashMap (with its child LinkedHashMap), IdentityHashMap, and WeakHashMap. On the ordered side we saw the SortedMap interface, its NavigableMap child, and their shared implementation TreeMap. The takeaway is a decision rule: reach for HashMap for speed, LinkedHashMap for insertion order, TreeMap for sorted keys and range queries, and the specialised IdentityHashMap/WeakHashMap only when their exact semantics are required.

  • Fast general-purpose lookups → HashMap.
  • Predictable insertion order / caches → LinkedHashMap.
  • Sorted keys, ranges, nearest-key queries → TreeMap (NavigableMap).
  • Reference-identity keys → IdentityHashMap; auto-evicting keys → WeakHashMap.

Source: section recap; no code example.

Pitfall Choosing a Map for the wrong reason costs later: picking TreeMap when you never query ranges pays an unnecessary O(log n) tax, while picking HashMap when you need ordered iteration forces awkward sorting on every read.

11. Concurrent Collections in Depth

Ordinary Java collections such as ArrayList and HashMap are not thread-safe, and the legacy synchronized ones (Vector, Hashtable) lock the whole object and cripple throughput. This section explains the fail-fast problem and ConcurrentModificationException, then shows how the java.util.concurrent collections (ConcurrentHashMap, CopyOnWriteArrayList, CopyOnWriteArraySet) deliver both safety and performance.

01. Section introduction

Objective: Preview the roadmap of the section so you know what problems the concurrent collections solve and in what order they are covered.

This section builds up in a deliberate order. First it revisits why the older Java collections struggle in a multi-threaded world — poor performance and the fail-fast behaviour that surfaces as ConcurrentModificationException. It then explains how the concurrent collections were designed to fix the shortcomings of the old synchronization approach. From there it dives into the internal design of ConcurrentHashMap, followed by CopyOnWriteArrayList and finally CopyOnWriteArraySet.

  • Old collections have performance and fail-fast limitations in concurrency.
  • ConcurrentModificationException is a central pain point to understand.
  • Concurrent collections replace whole-object locking with smarter strategies.
  • The three headline types studied are ConcurrentHashMap, CopyOnWriteArrayList and CopyOnWriteArraySet.

02. Need of concurrent collections

Objective: Understand why concurrency causes data-inconsistency risks and why the legacy synchronization approach, though safe, is a performance bottleneck.

Concurrency means running several parts of a program at the same time to raise performance and throughput. Java achieves this with multithreading. The trouble starts when multiple threads touch the same shared resource — in our case, the same collection object. If one thread is halfway through modifying the data while another reads or changes it, the collection can end up in an inconsistent state. To prevent this, classic Java code wraps access in synchronized blocks or methods so that only one thread enters at a time. But synchronization must be used carefully: while a thread holds the lock, every other thread waits, which turns parallel work back into sequential work and defeats the point of multithreading.

The legacy classes Vector and Hashtable are synchronized by default — safer but slower than ArrayList or HashMap. You can also wrap non-synchronized collections using the Collections utility class: Collections.synchronizedList, synchronizedMap, synchronizedSet. So why introduce concurrent collections at all? Because both Vector/Hashtable and the synchronized wrappers achieve safety by locking the entire object — even for read operations threads must queue one behind another. That is not optimized for performance. Concurrent collections were introduced precisely to remove this bottleneck.

  • Concurrency boosts throughput but risks data inconsistency on shared objects.
  • synchronized guarantees safety but serializes access, undermining parallelism.
  • Vector/Hashtable and Collections.synchronizedXxx lock the whole object for reads and writes alike.
  • Concurrent collections were created to keep thread safety while restoring performance.
import java.util.*;

        safe.add("B");

        // Manual synchronization is still required when iterating a synced wrapper.
        synchronized (safe) {

03. ConcurrentModificationException

Objective: See how an ordinary ArrayList throws ConcurrentModificationException when one thread iterates while another modifies it.

In a multithreaded program many threads may operate on a single collection to gain performance and scalability. With the old collection classes this creates a hazard: if one thread iterates the collection while another thread structurally modifies it (adds or removes an element), the iterating thread can throw a ConcurrentModificationException. The demo uses a shared static ArrayList. The main thread seeds it with names and starts iterating with an Iterator, sleeping briefly between elements. A child thread wakes up and adds one more element to the same list. When the main thread's iterator next checks the list and detects the structural change, it fails fast and throws.

  • Old collections are fail-fast: they detect concurrent structural change and throw immediately.
  • The exception protects you from silently reading inconsistent data.
  • It is triggered by a modification-count (modCount) mismatch inside the iterator.
  • The exact element at which it fails depends on thread timing, so it is non-deterministic.
import java.util.ArrayList;
import java.util.Iterator;

public class ConcurrentModiDemo extends Thread {

04. How concurrent collections solved the problems

Objective: Understand the improved locking strategy that lets concurrent collections be both safe and fast.

Concurrent collections were designed with a smarter locking mechanism. Where the legacy classes locked the entire object, the new collections lock only the specific segment being written to, leaving the rest of the structure free for other threads. This dramatically improves throughput and, at the same time, allows the collection to be modified while it is being iterated without throwing ConcurrentModificationException. For any multithreaded scenario, concurrent collections are therefore the better choice over the traditional synchronized collections.

  • Whole-object locking is replaced by fine-grained, segment-level locking.
  • Reads generally proceed without locking; only the written region is guarded.
  • Iteration during modification no longer throws ConcurrentModificationException.
  • Prefer concurrent collections over legacy synchronized ones in multithreaded code.
Pitfall Reaching for Vector or Hashtable "because they're thread-safe". They are, but their whole-object locking makes them slow. For new concurrent code prefer ConcurrentHashMap and friends.

05. ConcurrentHashMap hierarchy and methods

Objective: Place ConcurrentHashMap in the type hierarchy and master its four atomic ConcurrentMap methods.

All concurrent collections live in the java.util.concurrent package. In the map hierarchy, the Map interface is extended by the ConcurrentMap interface, which adds some concurrency-focused methods, and ConcurrentHashMap implements ConcurrentMap. So it offers every method of Map plus these extras. Four of them are especially important because they perform a check-and-act as a single atomic step:

  • putIfAbsent(K, V): inserts only if the key is not already present; returns the existing value if present (leaving it untouched), or null if it inserted.
  • remove(Object key, Object value): deletes the entry only if the key is currently mapped to that exact value.
  • replace(K, V oldValue, V newValue): swaps the value only if the key currently maps to oldValue; returns a boolean.
  • replace(K, V value) (Map default method since Java 8): if the key exists, replaces its value and returns the previous value; if absent, does nothing and returns null.
import java.util.concurrent.ConcurrentHashMap;

public class ConcurrentHashMapDemo {
    public static void main(String[] args) {
        ConcurrentHashMap<Integer, String> map = new ConcurrentHashMap<>();
        map.put(0, "Basics");
        map.put(1, "Strong");
        map.put(0, "Tech");                 // key 0 already present -> value replaced
        System.out.println(map);

        // putIfAbsent: key 0 exists, so nothing changes
        map.putIfAbsent(0, "Hello");
        System.out.println(map);

        // remove(key, value): removes only if 1 is mapped to "Strong"
        map.remove(1, "Strong");
        System.out.println(map);

        // replace(key, oldValue, newValue): 0 maps to "Tech", not "Wrong" -> no change
        map.replace(0, "Wrong", "Know");
        System.out.println(map);

        // replace(key, value): key 0 exists -> value replaced with "BasicsStrong"
        map.replace(0, "BasicsStrong");
        System.out.println(map);
    }
}

06. ConcurrentHashMap internal implementation

Objective: Explain how bucket/segment-level locking and the concurrency level let ConcurrentHashMap be both safe and fast.

Comparing the three map implementations makes the design clear. HashMap is not thread-safe, so in a concurrent setting it risks data inconsistency and ConcurrentModificationException. Hashtable is thread-safe but slow, because it locks the whole object so only one thread — reading or writing — can operate at a time. ConcurrentHashMap is thread-safe and performant: reads are not locked at all, and on a write only the particular segment being updated is locked, never the whole object. In the classic (Java 7) design the map was split into 16 independently-locked segments — this count was called the concurrency level — so up to 16 threads could write at once. Since Java 8 that fixed segmentation is gone: each write locks only the individual bin being updated (via CAS + synchronized), so write parallelism scales with the table size rather than a fixed 16. The 16 you will still see is just the default initial table capacity, and the constructor's concurrencyLevel is now only a sizing hint — best left at the default.

  • ConcurrentHashMap locks only the segment being updated, not the entire object.
  • It outperforms Hashtable and Collections.synchronizedMap.
  • It is fail-safe: iteration never throws ConcurrentModificationException.
  • null is not permitted as a key or a value.
  • The initial table holds 16 bins; since Java 8 there is no fixed 16-segment cap — concurrencyLevel is only a constructor sizing hint, best left at the default.
AdvancedThe 16-segment model describes the classic (Java 7) implementation. Since Java 8, ConcurrentHashMap dropped the fixed Segment array and instead locks the head node of an individual bin using CAS and synchronized, giving even finer granularity. The constructor still accepts a concurrencyLevel argument, but it is now only a sizing hint. The conceptual takeaway is unchanged: lock a tiny region, not the whole map.
Pitfall Putting a null key or value into a ConcurrentHashMap. Unlike HashMap, it rejects them with a NullPointerException — a null value would make it impossible to distinguish "absent" from "mapped to null" in a concurrent setting.

A ConcurrentHashMap is fail-safe: its internal design supports iteration and modification at the same time. The program seeds the map with four entries, the main thread iterates the entry set while sleeping briefly per element, and a child thread adds a fifth entry during that window. Run it with a plain HashMap and it throws; change only the declared type to ConcurrentHashMap and it completes cleanly.

  • Fail-safe (ConcurrentHashMap): iterates over a live view without throwing.
  • The only code change needed is the collection type — the logic stays identical.
  • A fail-safe iterator may or may not reflect concurrent additions; it will not error.

08. CopyOnWriteArrayList

Objective: Understand the copy-on-write strategy behind the thread-safe List and its iterator's restrictions.

CopyOnWriteArrayList is the thread-safe version of ArrayList. In the hierarchy it implements List, which implements Collection; while Collection and List come from java.util, this class lives in java.util.concurrent. Its safety strategy is different from ConcurrentHashMap: reads never lock and any number of threads can read at once, but every write operation creates a fresh cloned copy of the backing array. The update is applied to that clone, and afterwards the reference is swapped so readers see the new array. Because a full copy is made per mutation, 100 updates means 100 clones — a real performance overhead. Use it only when writes are rare and reads dominate.

Two consequences for its iterator: first, it is fail-safe, so no ConcurrentModificationException; second — and unlike an ArrayList iterator — the copy-on-write iterator's remove() is not supported. Because the iterator walks a fixed snapshot, removing through it could not be reflected consistently, so calling itr.remove() throws UnsupportedOperationException.

  • Thread-safe drop-in for ArrayList, in java.util.concurrent.
  • Reads are lock-free; each write clones the whole backing array.
  • Best for read-heavy, write-rare workloads because of the cloning cost.
  • Its iterator is fail-safe but remove() throws UnsupportedOperationException.
import java.util.Iterator;

09. CopyOnWriteArraySet

Objective: Learn the thread-safe Set that is backed by CopyOnWriteArrayList and keeps only unique elements.

CopyOnWriteArraySet is the concurrent version of Set, and internally it is implemented on top of CopyOnWriteArrayList. Its copy-on-write mechanism is therefore identical: writes go to a cloned copy, reads go to the main object, and the JVM keeps them in sync after the update. It is fail-safe like its list cousin. The one meaningful difference is that, being a set, it stores only unique elements. In the hierarchy it implements Set (which implements Collection) and lives in java.util.concurrent. It exposes no methods of its own beyond those of Collection and Set — add, addAll, clear, and so on. It has two constructors: a no-arg one, and one that accepts an existing collection and returns the equivalent set with duplicates discarded.

  • Concurrent, fail-safe Set built internally on CopyOnWriteArrayList.
  • Same copy-on-write behaviour: clone on write, lock-free reads.
  • Only difference from the list: it stores unique elements.
  • Constructing one from a collection drops duplicates automatically.

12. Lambda and Collections in Depth

Lambda expressions bring functional-style programming to Java: they let you pass a piece of behaviour as an argument instead of wrapping it in a full class. This section builds lambdas from first principles, connects them to functional interfaces, then puts them to work sorting collections and driving the built-in java.util.function interfaces, finishing with the :: method- and constructor-reference operator.

01. Lambda introduction

Objective: Understand what a lambda expression is and why Java added it.

Java is an object-oriented language, so historically the only way to hand a chunk of behaviour to a method was to wrap it in an object. Lambda expressions, added in Java 8, let Java borrow ideas from functional programming: you can pass a function or behaviour as an argument on the fly. Languages such as JavaScript, Python and C# already had this, and Java was a late arrival. The payoff is cleaner, shorter code, and — combined with streams — the door to easy parallel processing.

  • A lambda lets you treat behaviour as data: pass it around like any other value.
  • It enables a functional style on top of Java's object model.
  • The main wins are readability, less boilerplate, and stream-based parallelism.
  • Lambdas only make sense in the context of a functional interface (covered shortly).
Pitfall A lambda is not a free-floating function. It always has a target type — some functional interface it implements. Writing a lambda where the compiler cannot infer that target type is a compile error.

02. Write Lambda Expression

Objective: Learn the mechanical steps that turn an ordinary method into an equivalent lambda.

A lambda is essentially an anonymous function: a function with no name, no access modifier, and no explicit return type. To convert a normal method into a lambda you strip those three things away and insert the arrow token -> between the parameter list and the body. Two further simplifications apply: if the body is a single statement you may drop the braces, and if the compiler can infer parameter types you may omit them. When the body is a single expression that produces a value, you can also drop the return keyword and the braces together.

  • Remove the method name, the access modifier, and the return type.
  • Insert -> between parameters and body.
  • Single-statement body → braces optional; single expression → return optional too.
  • Parameter types are optional because of type inference; multiple statements still need braces and an explicit return.
// Ordinary methods (left) and their lambda equivalents (right, in comments)

// Type 1 - no arguments, no return value
public void printName() {          // ()  -> System.out.println("John")
    System.out.println("John");
}

// Type 2 - two arguments, no return value
public void add(int a, int b) {    // (a, b) -> System.out.println(a + b)
    System.out.println(a + b);
}

// Type 3 - one argument, returns int
public int length(String s) {      // s -> s.length()
    return s.length();
}

// Type 4 - one argument, multiple statements (braces + return required)

    return x;

03. Functional Interfaces

Objective: Define what makes an interface "functional" and recognise valid versus invalid cases.

An interface is a functional interface when it declares exactly one abstract method. Familiar JDK examples include Runnable (its single abstract method is run), Comparable (compareTo) and Callable (call). Since Java 8 an interface may also contain any number of default and static methods — the one-method rule applies only to abstract methods. So an interface with one abstract method plus several default/static helpers is still functional; an interface with two abstract methods is not; and an interface with zero abstract methods is not functional either.

  • Exactly one abstract method → functional interface.
  • default and static methods do not count against the limit and may appear freely.
  • Two abstract methods → a valid interface, but not a functional one.
  • Zero abstract methods → also not functional.
// VALID: one abstract method
public interface ValidFunInterface1 {
    void method1();
}

// VALID: one abstract method + a default + a static method
public interface ValidFunInterface2 {
    void method1();              // the single abstract method
    default void method2() { }   // allowed
    static  void method3() { }   // allowed
}

// INVALID: two abstract methods
public interface InvalidFunInterface1 {
    void method1();
    void method2();             // second abstract method breaks it
}

04. Invoke Lambda

Objective: Wire a lambda to a functional interface, and use @FunctionalInterface to guard the contract.

The annotation @FunctionalInterface from java.lang asks the compiler to verify that the interface really has exactly one abstract method; add a second abstract method and compilation fails with a clear error. To invoke a lambda you assign it to a reference of the functional-interface type — the lambda is the implementation of that single abstract method — and then call the method. Compared with the old approach (a separate class implementing the interface), the lambda supplies the behaviour inline, so there is no extra class to write.

  • @FunctionalInterface makes the compiler enforce the one-abstract-method rule.
  • Assign the lambda to a variable of the interface type; that variable's abstract method now runs the lambda body.
  • No separate implementing class is needed — behaviour is provided on the fly.
@FunctionalInterface
interface MyFunctionalInterface {
    void funMethod();            // single abstract method
}

public class LambdaIntroduction {
    public static void main(String[] args) {
        // The lambda IS the implementation of funMethod()
        MyFunctionalInterface fn = () -> System.out.println("FunctionalMethod");
        fn.funMethod();          // invokes the lambda
    }
}

11. Lambda with Collections

Objective: Recall the collection types and see where lambdas will simplify collection code.

A collection is a group of discrete objects. The framework offers several shapes: a List preserves insertion order and allows duplicates; a Set stores only unique elements; a Map supports very fast lookups because it is backed by a hashing algorithm. Lambdas streamline common operations over these structures — most immediately, custom sorting via Comparator.

  • List: ordered, allows duplicates.
  • Set: unique elements only.
  • Map: fast key-based lookup via hashing.
  • Lambdas cut boilerplate when supplying behaviour such as sort order.

12. Comparator using Lambda

Objective: Recognise Comparator as a functional interface that a lambda can implement.

Comparator<T> declares a single abstract method, compare(T o1, T o2), so it is a functional interface — a perfect target for a lambda. It defines sorting order for collections by comparing two objects, returning a negative number, zero, or a positive number depending on whether the first argument should come before, at the same rank as, or after the second. Because it is functional, you can supply that comparison as a lambda instead of a whole class.

  • Comparator<T> has one abstract method: int compare(T o1, T o2).
  • Return negative / zero / positive to mean "before" / "equal" / "after".
  • Being functional, it accepts a lambda as its implementation.
Pitfall Never implement compare with o1 - o2 on values that can be large or negative — the subtraction can overflow int and flip the sign. Prefer explicit comparisons or Integer.compare(o1, o2).

13. Sorting a List without Lambda and with Lambda

Objective: Sort a list with a custom comparator, first the verbose way then with a one-line lambda.

A freshly built list keeps its insertion order. Collections.sort(list) alone sorts in natural ascending order. For a custom order you need a Comparator. The pre-lambda approach is to write a class implementing Comparator and override compare; the lambda approach passes the comparison inline to Collections.sort(list, comparator). The comparator below returns -1 when o1 > o2, so larger values are ranked first — a descending sort.

  • Insertion order is preserved until you sort.
  • Collections.sort(list, comparator) takes any Comparator, including a lambda.
  • Returning -1 for o1 > o2 produces descending order.
  • The lambda removes the need for a separate comparator class.
import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

public class SortingListWithoutLambda {
    public static void main(String[] args) {
        List<Integer> list = new ArrayList<>();
        list.add(10);
        list.add(5);
        list.add(0);
        list.add(3);
        list.add(11);
        System.out.println(list);          // insertion order

        // Comparator supplied inline as a lambda -> descending order
        Collections.sort(list, (o1, o2) -> {
            if (o1 > o2) {

                return 0;
            }

        System.out.println(list);          // sorted, largest first

14. Predefined Functional Interfaces — Predicate

Objective: Meet the java.util.function package and the Predicate interface.

Rather than hand-writing a functional interface for every need, Java ships a library of them in java.util.function — Predicate, Function, Consumer, Supplier and more. A Predicate<T> represents a boolean-valued test: its single abstract method test(T t) checks a condition and returns true or false. Typical uses are validation checks, for example "is this salary at least 4000?" or "does this email contain an @".

  • java.util.function provides ready-made functional interfaces — no need to roll your own.
  • Predicate<T>'s abstract method is boolean test(T t).
  • A predicate answers a yes/no question about its input.
Bigger picture The package has more than these four. For two inputs there are BiFunction<T,U,R>, BiPredicate<T,U> and BiConsumer<T,U>. When input and output share a type, UnaryOperator<T> (a Function<T,T>) and BinaryOperator<T> save keystrokes. And to avoid boxing every int/long/double into a wrapper, primitive specialisations exist: IntPredicate, ToIntFunction<T>, IntUnaryOperator, Supplier's cousin IntSupplier, and so on. Reaching for these in hot loops removes autoboxing overhead.

19. The :: operator — method reference

Objective: Reuse an existing method as a functional-interface implementation with the :: operator.

When the behaviour a lambda needs already exists as a method, you can point the functional interface straight at it with the double-colon :: operator instead of re-writing the body. The rules are that the argument types must be compatible with the abstract method, and the referenced method's return type must be assignable to the interface method's return type: a value-returning method can satisfy a void interface method (the value is discarded), but a void method cannot satisfy a value-returning one. Only the method's own name is free to differ. For a static method the syntax is ClassName::methodName. For an instance method you first create an object and write objectRef::methodName. A method reference is therefore an alternative to a lambda that promotes code reuse.

  • Static method: ClassName::method (no parentheses).
  • Instance method: create an object, then obj::method.
  • Only the argument types must match the interface's abstract method.
  • A method reference is a drop-in alternative to an equivalent lambda.
interface MyFunInterfaceNew {
    void method1();          // takes no arguments
}

public class DoubleColonMethodReference {
    public static void main(String[] args) {
        // Lambda supplies the behaviour inline
        MyFunInterfaceNew funInter = () -> System.out.println("I am Lambda");
        funInter.method1();

        // Static method reference: argument lists match (both take none)
        MyFunInterfaceNew i = DoubleColonMethodReference::method2;
        i.method1();

        // Instance method reference: need an object first
        DoubleColonMethodReference obj = new DoubleColonMethodReference();
        MyFunInterfaceNew i1 = obj::method3;
        i1.method1();
    }

    public static void method2() {          // static
        System.out.println("I am not a Lambda");
    }

    public void method3() {                 // instance
        System.out.println("i am here, refer to me");
    }
}

13. Streams in Depth and Collections

The Stream API, introduced in Java 8, lets you process collections declaratively — filtering, transforming, sorting, and aggregating data in a few readable lines instead of hand-written loops, and optionally in parallel across CPU cores. This section builds streams up from first syntax to a measured sequential-vs-parallel benchmark.

Run locally Save each example as ClassName.java, compile with javac ClassName.java, then run with java ClassName (JDK 17+). For quick one-liners, start jshell and paste a stream pipeline directly. The parallel demo needs both ParallelStreamsDemo.java and Employee.java in the same folder.

01. Section Overview

Objective: Understand what streams are, why they were added to Java, and the kind of collection-processing problem they solve.

A stream is a pipeline for processing the objects held inside a collection. Collections such as List and Map already exist to store groups of objects; streams, added in Java 8, exist to process them. The classic motivation: given a list of Employee objects, produce a new collection containing only those earning at least $5000. With streams you express that in a single concise expression, and because streams can spread work across multiple CPU cores, the processing can be faster than an equivalent hand-written loop.

  • Streams process the elements of a collection; they do not store data themselves.
  • Introduced in Java 8 as a declarative alternative to explicit loops.
  • They shorten code and can exploit multi-core processors for speed.
  • A stream does not modify its source collection — it produces a result.
Pitfall A stream is not a data structure — you cannot index into it or reuse it. Once a stream has been consumed by a terminal operation, attempting to operate on it again throws IllegalStateException. Create a fresh stream from the source each time.

02. Streams basics — getting familiar with the syntax

Objective: Contrast the classic loop-and-if approach with the equivalent stream pipeline using filter and map.

Start with a list of integers 10, 20, 30, 40, 50. Suppose you want a new list containing only the values greater than or equal to 30. The traditional way is to loop over the list, test each element with an if, and add the matches to a new list. With streams the same result comes from one expression: list.stream().filter(i -> i >= 30).collect(Collectors.toList()). The filter method takes a predicate (a boolean lambda) and keeps only the elements that satisfy it. The map method instead transforms every element — for example doubling each value — and returns a list of the same length. In both cases collect(Collectors.toList()) gathers the processed elements back into a List.

  • collection.stream() opens a pipeline over the collection.
  • filter(predicate) keeps elements where the boolean lambda is true.
  • map(function) transforms each element; output length equals input length.
  • collect(Collectors.toList()) is a terminal step that builds a List.
  • A whole loop-and-if block collapses into one readable line.
import java.util.ArrayList;
import java.util.List;
import java.util.stream.Collectors;

public class StreamsIntroduction {
    public static void main(String[] args) {
        ArrayList<Integer> list = new ArrayList<>();
        list.add(10);
        list.add(20);
        list.add(30);
        list.add(40);
        list.add(50);

        // filter: keep values >= 30
        List<Integer> l1 = list.stream()
                               .filter(i -> i >= 30)
                               .collect(Collectors.toList());
        System.out.println(l1);

        // map: double every value
        List<Integer> l2 = list.stream()
                               .map(i -> i * 2)
                               .collect(Collectors.toList());
        System.out.println(l2);
    }
}

03. Streams in detail — filter() and map()

Objective: Learn where stream() comes from and precisely how the two configuration operations, filter and map, differ.

The stream() method was added to the Collection interface in Java 8 as a default method, so every collection object inherits it. Stream work divides into two phases: configuration (setting up what should happen) and processing (actually producing a result). Configuration is done with two building blocks. filter applies a boolean condition and can shrink the data — its signature is Stream<T> filter(Predicate<? super T>), and a predicate is exactly a boolean-valued lambda. map applies a calculation to each element and returns the transformed data — its signature is <R> Stream<R> map(Function<? super T, ? extends R>), where Function is a predefined functional interface. The key distinction: filter may reduce the element count, while map always preserves it.

  • stream() is a default method on Collection, available to every collection.
  • filter takes a Predicate — a boolean lambda — and may reduce length.
  • map takes a Function — a transforming lambda — and preserves length.
  • These are the two core "configuration" operations of a pipeline.

Source: reimplemented from course example StreamsIntroduction.java.

AdvancedBecause map can change the element type, its generic parameter R differs from the input type T. For example List<String> names; names.stream().map(String::length) produces a Stream<Integer>. When the transform itself yields a stream (e.g. mapping each order to a stream of items), use flatMap to flatten the nested streams into one.

04. More methods in Streams — collect() and count()

Objective: Move from configuration to processing (terminal) operations, using collect to gather results and count to tally them.

Once the pipeline is configured with filter and/or map, a terminal operation actually processes the data. collect gathers the elements into a new collection — for example Collectors.toList() builds a List. Another common terminal is count(), which returns a long giving the number of elements that survived the pipeline. Filtering the sample list for values >= 30 and then calling count() reports 3, matching the three elements (30, 40, 50) that passed the filter.

  • filter/map are intermediate (configuration) steps; they return a stream.
  • collect and count are terminal (processing) steps; they return a result.
  • count() returns a long — the number of surviving elements.
  • A pipeline needs exactly one terminal operation to run.

05. Some more on Streams — sorting (natural and custom)

Objective: Sort stream elements both in natural order and with a custom comparator supplied as a lambda, and meet the other common terminals forEach, min, and max.

Streams offer two overloads of sorted. The no-argument sorted() arranges elements in their natural order — for integers, ascending. The argument form sorted(Comparator) takes a comparator, typically written as a lambda, for custom ordering. To sort the integer list descending, pass (i1, i2) -> i2.compareTo(i1), which reverses the usual comparison. Streams also expose other useful terminals such as forEach (run a side effect per element), and min/max (which return an Optional because the stream might be empty).

  • sorted() with no argument sorts in natural (ascending) order.
  • sorted(comparator) sorts using a custom rule.
  • A descending comparator is (a, b) -> b.compareTo(a).
  • min/max return Optional<T>; forEach runs a side effect per element.
import java.util.List;
import java.util.stream.Collectors;

public class StreamSorting {
    public static void main(String[] args) {
        List<Integer> list = List.of(30, 10, 50, 20, 40);

        // natural (ascending) order
        List<Integer> l3 = list.stream()
                               .sorted()
                               .collect(Collectors.toList());
        System.out.println("Ascending  : " + l3);

        // custom (descending) order
        List<Integer> l4 = list.stream()
                               .sorted((i1, i2) -> i2.compareTo(i1))
                               .collect(Collectors.toList());
        System.out.println("Descending : " + l4);
    }
}

06. Parallel Streams — the concept

Objective: Grasp what parallel streams are and how parallel processing differs from sequential processing.

Parallelism is about speed. Modern machines almost always have multi-core processors, and a program should be structured to exploit them: more cores means more work done per unit of time. A stream can be processed in one of two modes. Sequential processing is like a single-lane road — elements are handled one after another. Parallel processing is like a multi-lane road — the workload is split so several elements are processed simultaneously, so the "traffic" clears faster. The next lecture turns this idea into measurable code.

  • Parallel streams trade a single lane for many, using multiple CPU cores.
  • Sequential = one element at a time; parallel = many at once.
  • Streams are sequential by default; parallelism is opt-in.
  • Parallelism pays off most on large datasets and CPU-heavy work.
Context Going parallel is not free: the framework has to split the source, hand pieces to worker threads, and merge the partial results. That coordination costs time, so a parallel stream only wins when the per-element work (or the element count) is large enough to outweigh the overhead. Splitting also works best on sources with cheap, even partitioning — an ArrayList or an array splits well; a LinkedList does not.

07. Sequential vs Parallel streams — performance comparison

Objective: Build and benchmark a parallel stream against a sequential one, and know the three conditions parallel operations must satisfy.

Streams are sequential by default. There are two ways to go parallel: call parallelStream() on a collection, or call parallel() on an existing stream. Parallelism comes with obligations — the result must match what sequential processing would produce. For that to hold, the operation should be stateless (one element's processing must not depend on another's), non-interfering (the source data must not be modified during the operation), and associative (the outcome must not depend on the order in which elements are combined). The demo builds a list of 600 Employee objects and counts those earning more than 1000, first sequentially then in parallel, timing each with System.currentTimeMillis(). Both report the same count (500). The parallel run may finish faster on a multi-core machine, but for a workload this small the split/merge overhead can easily cancel any gain — and the numbers below are dominated by one-time JVM warm-up that the first (sequential) run pays for, so treat them as illustrative, not a real speedup measurement.

  • Create parallel streams with collection.parallelStream() or stream().parallel().
  • Correctness needs stateless, non-interfering, associative operations.
  • A correct parallel pipeline yields the same result as the sequential one.
  • Timing with System.currentTimeMillis() shows the (warm-up-sensitive) elapsed time.

08. Summary — what we learned about Streams

Objective: Consolidate the section into a mental map of stream configuration, processing, and parallelism.

The section moved from basic stream syntax, to the configuration operations filter and map, to processing operations such as collect and count, then to further methods including sorted, min, and max. It closed with parallel streams — arguably the most powerful idea here — and a side-by-side benchmark showing that a correctly written parallel pipeline produces the identical result while, on the right (large enough) workload, running faster on multi-core hardware.

  • Pipeline shape: source → intermediate ops (filter/map/sorted) → terminal op (collect/count/forEach).
  • Intermediate ops are lazy; a terminal op triggers execution.
  • filter can shrink the data; map preserves its length.
  • Parallelism boosts speed when operations are stateless, non-interfering, and associative — and the workload is big enough to earn back the overhead.

14. Generics In Depth

Generics add compile-time type checking to Java. Introduced in Java 5, they let you write one flexible class, interface, or method that works safely with many types — catching type mistakes while you compile instead of blowing up at runtime with a ClassCastException.

01. Understanding the Concept — Generic Classes

Objective: Understand why generics exist and how to turn an ordinary class into a reusable, type-safe generic class using a type parameter.

Generics are a compile-time type-checking mechanism. The problem they solve: compile errors are easy to fix because the compiler points at them, but runtime errors like ClassCastException are invisible to the compiler and force you to debug to find the offending line. By declaring the type you intend to use, you let the compiler reject mistakes up front. Generics also promote code reuse.

Consider a plain Data class holding one field. If you make the field a String, it only works for text. Switching the field to Object to accept anything looks tempting, but then the compiler cannot stop you casting a stored Integer back to String — that mistake surfaces only at runtime as a ClassCastException. Writing a separate class per type (one for String, one for Integer, one for Double...) does not scale. The answer is a type parameter: put <T> after the class name as a placeholder for the real type supplied later. A parameterized reference such as Data<String> tells the compiler exactly what is inside, so no manual casting is needed and wrong assignments are refused at compile time.

  • Generics were added in Java 5; they perform type checking at compile time, not runtime.
  • Two main benefits: stronger type safety and code reuse.
  • <T> is a formal type parameter — a placeholder for a real type. T is a common name, but any identifier works.
  • Using Object instead of a type parameter defeats safety: mistakes become runtime ClassCastExceptions.
  • A type parameter applies to instance members only — you cannot use T in a static field or static method, because statics are shared across all instances while each instance may bind T to a different type.
// Data.java  -- a generic class with one type parameter T
public class Data<T> {
    private T data;                       // T is a placeholder for the real type

    public Data(T data) { this.data = data; }

    public T getData() { return data; }
    public void setData(T data) { this.data = data; }

    @Override
    public String toString() { return "Data is " + data; }
}
// DemoGenerics.java
public class DemoGenerics {
    public static void main(String[] args) {
        Data<String>  d1 = new Data<>("Hello");   // T = String
        Data<Integer> d2 = new Data<>(42);        // T = Integer, same class reused

        System.out.println(d1);
        System.out.println(d2);

        String s = d1.getData();  // no cast needed; compiler knows it is String
        int    n = d2.getData();  // auto-unboxed Integer
        System.out.println(s + " / " + n);
        // String bad = d2.getData(); // COMPILE error: caught before you run
    }
}

02. Generic Type 2 — Generic Interfaces

Objective: Declare a generic interface and implement it with a specific type argument.

Just as a class can carry a type parameter, an interface can too. You write <T> after the interface name and use T in its method signatures. An implementing class then either supplies a concrete type (becoming type-specific) or stays generic itself and forwards its own type parameter to the interface. This is the interface counterpart of the generic class from the previous lecture and underpins much of the JDK, where interfaces like List<E>, Comparable<T>, and Iterator<E> are all generic.

  • An interface may declare one or more type parameters after its name: interface Container<T>.
  • An implementing class picks a concrete type (implements Container<String>) or forwards its own (class Box<T> implements Container<T>).
  • Core JDK abstractions (List, Map, Comparable, Iterable) are generic interfaces.
// A generic interface
public interface Container<T> {
    void put(T item);
    T get();
}

// Option A: implement with a concrete type
class StringBox implements Container<String> {
    private String value;
    public void put(String item) { this.value = item; }
    public String get() { return value; }
}

// Option B: stay generic and forward the type parameter
class GenericBox<T> implements Container<T> {
    private T value;
    public void put(T item) { this.value = item; }
    public T get() { return value; }
}

03. Exercising — Subclassing

Objective: Learn the rules for extending and implementing generic types, and when the compiler forbids a combination.

A generic type is either a generic class or a generic interface, and both can participate in inheritance. A generic interface can extend another generic interface using the same type parameter, and a generic class can extend another generic class. When you extend or implement while keeping the same type parameter, an assignment from subtype to supertype is allowed only when both are parameterized with the same type — Data<String> is a BaseData<String>, but not a BaseData<Integer>. This dependency guarantees that instantiating the subclass fixes the same actual type for the superclass and interface.

Several finer cases are worth knowing. A generic interface can be implemented by a non-generic class only if that class supplies a concrete type (a wrapper like Integer, or any class/interface name). A generic class can extend a non-generic type freely, and a non-generic class can extend a generic class by naming the type. Finally, two generic classes may implement the same generic interface, but a single class cannot implement the same interface twice with different arguments — if one path fixes Integer and another leaves T, the mismatch is rejected.

  • Generic interface may extends a generic interface with the same parameter; generic class may extends a generic class the same way.
  • Subtype-to-supertype assignment needs matching type arguments on both sides.
  • A non-generic class implementing/extending a generic type must specify the concrete type argument.
  • A generic class may extend a non-generic type; a class cannot implement one interface twice with conflicting type arguments.
class BaseData<T> { }
class Data<T> extends BaseData<T> { }   // same parameter forwarded up

public class SubclassDemo {
    public static void main(String[] args) {
        Data<String> d4 = new Data<>();
        BaseData<String> b1 = d4;           // OK: both are <String>
        System.out.println("Assigned Data<String> to BaseData<String>");

        // BaseData<Integer> bad = d4;     // COMPILE error: <String> vs <Integer>
    }
}

04. Generics with Collections

Objective: See why the entire Collections Framework was made generic and how parameterizing a collection restores type safety.

Beyond stronger compile-time checking and eliminating manual casts, generics let library authors write algorithms that operate over collections of different element types while remaining type-safe and readable. That is why, in Java 5, the whole Collections Framework was rewritten to be generic. Without generics, a raw List accepts any object, so adding a stray Integer to a list you intended for Strings compiles cleanly but throws ClassCastException when you iterate and cast each element. Parameterizing the list as List<String> makes the compiler reject the wrong add call, and iteration needs no cast because the element type is already known.

  • Generics give stronger compile-time checks, type safety, no manual casts, and reusable algorithms.
  • The Collections Framework was rewritten for generics in Java 5.
  • A raw List triggers an "unchecked / raw type" warning and allows any element — a runtime hazard.
  • E is the conventional letter for a collection's element type.

05. Raw Type

Objective: Understand what a raw type is, why it still exists, and why you should avoid it.

A raw type is the name of a generic class or interface used without any type argument — for example List instead of List<String>, or a custom Box instead of Box<Integer>. A raw generic behaves like the pre-generics version: it accepts elements of any type and only issues a warning, never an error. Raw types were kept when generics arrived in Java 5 purely for backward compatibility with older code. You can still use them, but for a container meant to hold a single type they are unsafe: they defer errors to runtime and force you to cast on retrieval. If you genuinely want a container open to any type, prefer Box<Object> over the raw Box so you keep some checking and silence the raw-type warning.

  • Raw type = generic type used with no type argument (Box rather than Box<Integer>).
  • Raw types accept any element and only warn, never error — unsafe.
  • They survive only for backward compatibility with pre-Java-5 code.
  • To allow any type on purpose, use <Object> instead of a raw type.
// Box.java -- a generic class
import java.util.List;

public class Box<T> {
    private List<T> list;
    public Box(List<T> list) { this.list = list; }
    public List<T> getList() { return list; }
    public void setList(List<T> list) { this.list = list; }
}
// RawTypeDemo.java
import java.util.List;

public class RawTypeDemo {
    public static void main(String[] args) {

        Box<Integer> genBox = new Box<>(List.of(1, 2, 3));

            System.out.println(n);
        }

        // Raw use -- allowed for compatibility, but only warns
        Box rawBox = new Box(List.of(1, 2, "strong basics")); // mixed types!

06. Multiple Type Parameters — HashMap

Objective: Build and use a generic type that declares more than one type parameter.

A generic class is not limited to a single placeholder. HashMap<K, V> is the classic example: K stands for the key type and V for the value type, and the class uses each wherever needed. You can define as many type parameters as you like, separated by commas. To illustrate, a Bin<D, W> class can model a rubbish bin that separates dry trash of type D from wet trash of type W, each stored in its own field with its own getter and setter.

  • Declare several type parameters in the angle brackets, comma-separated: class Bin<D, W>.
  • HashMap<K, V> uses K for keys and V for values.
  • Each parameter can be bound to a different concrete type at instantiation.

07. Sub-Typing

Objective: Understand that parameterized types do not inherit along their type arguments (invariance).

Ordinary inheritance says an Integer is a Number, so an Integer value can go where a Number is expected. Parameterized types do not follow this rule: List<Integer> is not a subtype of List<Number>, even though Integer is a subtype of Number. This property is called invariance. Subtyping still works along the raw type — ArrayList<String> is a List<String> — but never along the type argument. If it did, you could add a Double through a List<Number> reference that actually points at a List<Integer>, corrupting it. Wildcards (covered later) are the tool for relaxing this safely.

  • Parameterized types are invariant: List<Integer> is not assignable to List<Number>.
  • Subtyping holds along the raw/base type (ArrayList<E> is a List<E>), not the argument.
  • Invariance prevents a supertype reference from inserting an incompatible element.
import java.util.ArrayList;
import java.util.List;

public class SubTypingDemo {
    public static void main(String[] args) {
        List<Integer> ints = new ArrayList<>();  // OK: ArrayList<Integer> is a List<Integer>

        // List<Number> nums = ints;             // COMPILE error: invariance
        // nums.add(3.14);                       // would have corrupted an Integer list

        List<? extends Number> readOnly = ints;  // wildcard: the safe way (see later)
        System.out.println("List<Integer> assigned to List<? extends Number>");
    }
}

08. Generic Methods

Objective: Add a type parameter to a single method (or constructor) instead of parameterizing the whole class.

Sometimes only one method needs a type parameter and there is no reason to make the enclosing class generic. In that case you declare the type parameter on the method itself, placing <T> immediately before the return type. Without that leading <T> the compiler treats T as an undefined type; with it, the method becomes a genuine generic method that can be invoked with any type. Because a constructor is a special kind of method, it too can be made generic the same way — a non-generic class can still have a generic constructor. The JDK is full of generic methods; look inside java.util.Collections and you will find sort, binarySearch, and many others declared this way.

  • Declare the parameter before the return type: public static <T> T concat(T a, T b).
  • The class need not be generic for a method inside it to be generic.
  • Constructors can be generic too, even in a non-generic class.
  • Prefer a generic method over overloading the same method for many types.
// GenericMethod.java -- a NON-generic class with a generic method
public class GenericMethod {

    // <T> before the return type makes this a generic method
    public static <T> String concat(T data) {
        return "Data is " + data;
    }

    public static void main(String[] args) {
        System.out.println(concat("Hello"));  // T = String
        System.out.println(concat(100));       // T = Integer
        System.out.println(concat(3.5));       // T = Double
    }
}

09. Bounded Type Parameter

Objective: Restrict a type parameter to a type and its subtypes using extends, with a class, an interface, or multiple interfaces.

An unbounded type parameter accepts any reference type, but often you want to constrain it. If a generic class only makes sense for numbers, bound T with <T extends Number> — now T may be Number or any subtype (Integer, Double, Float...), and passing a List<String> is rejected at compile time. The keyword is always extends, whether the bound is a class or an interface. The same bounding applies to a generic method: public static <T extends Number> void sort(List<T> list). You can also bound with an interface (<T extends IBound>), and even with several interfaces joined by & (<T extends IBound1 & IBound2>). When you bound with multiple interfaces, the argument's class must implement all of them. Java has no multiple class inheritance, so you can bound with at most one class but any number of interfaces.

  • <T extends Number> limits T to Number or its subtypes; extends is used for both classes and interfaces.
  • Bounding works on generic classes and generic methods alike.
  • Multiple bounds are joined with &: <T extends A & I1 & I2> — a class bound (if any) must come first.
  • At most one class bound; unlimited interface bounds (Java has no multiple class inheritance).
import java.util.List;

public class BoundedDemo {

    // T must be Number or a subtype -> sorting numeric data only
    public static <T extends Number> void printSorted(List<T> list) {
        List<T> copy = new java.util.ArrayList<>(list);
        copy.sort(null);                 // natural order
        for (T element : copy) {
            System.out.println(element);
        }
    }

    public static void main(String[] args) {
        printSorted(List.of(5, 2, 9, 1));      // OK: Integer extends Number
        printSorted(List.of(8.5, 9.8, 2.1));   // OK: Double  extends Number
        // printSorted(List.of("b", "a"));     // COMPILE error: String is not a Number
    }
}

10. Wildcards — Upper-Bounded Wildcards

Objective: Use the ? wildcard for an unknown type, and ? extends Type to accept a type or any of its subtypes.

The question mark ? is the wildcard, representing an unknown type. It can appear as the type of a parameter, a field, a local variable, or occasionally a return type. A method like display(List<?> list) accepts a list of any element type — iterate its elements as Object. Wildcards differ from type parameters: a type parameter (<T>) is only for generic classes, methods, and constructors, whereas a wildcard is used at the point where you reference a generic type to constrain which types a method may legally be called with. You cannot use ? as a bare stand-in for the element type inside the method body, nor declare a variable's own type as just ? without a surrounding generic type.

Because parameterized types are invariant (Lecture 07), a DataNode<Number> cannot hold a DataNode<Integer> as its "next" reference, even though Integer is a subtype of Number. The fix is a wildcard. An unbounded wildcard ? lets the next node be any type; an upper-bounded wildcard ? extends Number restricts it to Number or its subtypes, so you can chain integer, double, and other numeric nodes but not a string node.

  • ? means "unknown type"; usable for parameters, fields, locals, sometimes return types.
  • Wildcards go at the reference site; type parameters go on the generic class/method/constructor declaration.
  • ? extends Number is an upper-bounded wildcard: the type is Number or a subtype.
  • Wildcards solve the invariance problem so a node can reference a related-but-different parameterized type.
// DataNode.java -- a linked-list node; "next" uses an upper-bounded wildcard
public class DataNode<E> {
    private E data;
    private DataNode<? extends Number> next;  // any node holding Number or a subtype

    public DataNode(E data, DataNode<? extends Number> next) {
        this.data = data;
        this.next = next;
    }
    public E getData() { return data; }
    public DataNode<? extends Number> getNext() { return next; }

    @Override
    public String toString() {
        return "DataNode{data=" + data + ", next=" + next + "}";
    }
}
// LinkedListDemo.java
public class LinkedListDemo {
    public static void main(String[] args) {
        DataNode<Integer> node2 = new DataNode<>(20, null);   // last node
        DataNode<Integer> node1 = new DataNode<>(35, node2);  // points to node2
        System.out.println(node1);

        // Mixed numeric chain is now legal thanks to ? extends Number:
        DataNode<Double> d = new DataNode<>(9.5, node2);
        System.out.println(d);
    }
}

Pitfall You cannot write void m(? list) or use a bare ? as a method return type without a surrounding generic type — that is illegal. The wildcard must qualify a generic type, e.g. List<?>.

11. Lower-Bounded Wildcards

Objective: Use ? super Type to accept a type or any of its supertypes.

The counterpart of an upper bound is the lower-bounded wildcard, written ? super Type, where the bound may be any class or interface. It matches the named type and its supertypes. If a method takes List<? super Integer>, you can pass a List<Integer> or a List<Number> (because Number is a supertype of Integer), but not a List<Double> or a List<String>, because neither Double nor String is a supertype of Integer. Where extends walks down the hierarchy toward subtypes, super walks up toward supertypes. A wildcard can carry an upper bound or a lower bound, but never both at once.

  • ? super Integer is a lower-bounded wildcard: Integer or any supertype.
  • extends = "this type or below"; super = "this type or above".
  • A single wildcard can be upper-bounded or lower-bounded, not both simultaneously.
import java.util.ArrayList;
import java.util.List;

    // Accepts a list of Integer or any supertype of Integer (Number, Object...)
    static void display(List<? super Integer> list) {
        for (Object o : list) {

12. Restrictions on Wildcards

Objective: Learn what wildcards cannot do — where they are illegal and which operations they block.

Wildcards solve invariance problems but come with restrictions. First, you cannot use a wildcard as a type parameter in a class-level declaration header — you cannot replace T with ? in class Box<?>. Second, unlike type parameters, wildcards do not support multiple bounds: ? extends Number & Comparable is a syntax error. As for operations: reading from a wildcard collection is always fine, but writing is restricted. With an unbounded (?) or upper-bounded (? extends Number) wildcard you can read but cannot add, because the exact element type is unknown — the compiler cannot verify that what you add is compatible. With a lower-bounded wildcard (? super Integer) you can safely add Integer values, since whatever the list's real type is, it is a supertype of Integer and can hold one. The same rules apply when you instantiate objects with wildcards: a wildcard is illegal on the right-hand side of new, and you cannot write into an unknown-type list.

  • Wildcards cannot appear as a type parameter in a class/interface declaration header.
  • Wildcards do not allow multiple bounds (no &).
  • Reading is always allowed; writing is blocked for unbounded and upper-bounded wildcards.
  • A lower-bounded wildcard (? super Integer) permits adding Integer (and its subtypes).
  • You cannot use a wildcard on the right side of new.
import java.util.List;

public class WildcardRestrictions {

    static void readAny(List<?> list) {
        for (Object o : list) System.out.println(o);  // read: OK
        // list.add(10);        // COMPILE error: cannot add to List<?>
    }

    static void readUpper(List<? extends Number> list) {
        Number first = list.get(0);   // read: OK
        // list.add(1);         // COMPILE error: cannot add to List<? extends Number>
        System.out.println(first);
    }

    static void writeLower(List<? super Integer> list) {
        list.add(1);            // OK: Integer fits any supertype list
        list.add(2);
        System.out.println(list);
    }

    public static void main(String[] args) {
        List<Integer> nums = new java.util.ArrayList<>(List.of(7, 8));
        readAny(nums);
        readUpper(nums);
        writeLower(nums);
    }
}

13. Behaviour in Generic and Non-Generic Areas

Objective: See how the same object's type checking depends on whether it is used in a generic or a raw (non-generic) context.

An object's compile-time behaviour depends on where it is used, not only on how it was declared. Suppose you create a List<String> and pass it to a method whose parameter is a raw List. Inside that raw-typed method the compiler has lost the element type, so it lets you add any object — and those elements really do land in your originally-string-only list, producing only an unchecked warning. So a generic object behaves like a non-generic one when it flows into a non-generic area. The reverse also holds: a raw list passed to a method whose parameter is List<String> is checked as string-only inside that method. The rule: in a generic area only the parameterized type is accepted; in a non-generic (raw) area any type may be added to the same object.

  • Type checking is determined by the reference's context, not just the object's origin.
  • Passing a List<String> into a raw List parameter lets any element be added (unchecked warning only).
  • Passing a raw list into a List<String> parameter enforces string-only inside that method.
import java.util.ArrayList;
import java.util.List;

public class Behaviour {

    // Non-generic (raw) area: any element may be added
    @SuppressWarnings({"unchecked", "rawtypes"})
    static void addAnything(List l) {
        l.add(1);        // int into a list that was declared List<String>!
        l.add(true);     // boolean too -- only an unchecked warning
    }

    public static void main(String[] args) {
        List<String> l = new ArrayList<>();
        l.add("strong");         // generic area: only String allowed here
        addAnything(l);          // raw area: pollutes the same object
        System.out.println(l);
    }
}

Pitfall Handing a parameterized collection to legacy raw-type APIs silently defeats your type safety. If you must, isolate the raw call and revalidate elements, and never suppress the unchecked warning without being certain it is safe.

14. Type Erasure

Objective: Understand how the compiler implements generics by erasing type information, and the consequences at runtime.

Generics are a compile-time device implemented by the Java compiler through a front-end transformation called type erasure. During erasure the compiler removes every type parameter and replaces it with its first bound — or with Object when the parameter is unbounded (a bounded <T extends Number> becomes Number). It also inserts casts where needed. At runtime the generic type information is completely gone, so type safety and casting exist only at compile time.

This explains several behaviours. If you defeat compile-time checks (for example by aliasing a List<String> through a raw List and adding an Integer), the bad value goes in with only an unchecked warning; the ClassCastException appears later, at the compiler-inserted cast when something reads it as String. Assigning a generic object to a raw reference behaves like a raw list — List l1 = new ArrayList<String>() accepts any element and prints without a runtime exception, because the JVM never saw the <String>. And you cannot overload two methods that differ only by type argument — m(List<String>) and m(List<Integer>) both erase to m(List), so they clash.

  • Erasure replaces each type parameter with its first bound, or Object if unbounded.
  • The compiler inserts casts; at runtime no generic type info remains.
  • Circumventing checks via raw types defers the ClassCastException to the inserted cast.
  • You cannot overload methods that differ only in type argument — they erase to the same signature.
import java.util.LinkedList;
import java.util.List;

public class TypeErasureDemo {

    // aliasing through a raw reference sneaks an Integer into a String list
    @SuppressWarnings({"unchecked", "rawtypes"})

        List<String> a = new LinkedList<>();
        List b = a;              // raw alias to the same object
        b.add(1);                // unchecked warning only; Integer goes in
        return a.get(0);         // compiler inserts (String) cast here
    }

    public static void main(String[] args) {
        System.out.println(broken());
    }

15. Generics — Summary

Objective: Consolidate the section's guidance into a short set of best practices.

Summarising helps the material stick. The overriding rule is to avoid raw types — always aim for type-safe code so you get neither compile-time warnings nor runtime exceptions. If you are genuinely certain a piece of code is type-safe, you may silence an unchecked warning with the @SuppressWarnings("unchecked") annotation, but use it sparingly. Prefer generic classes and interfaces over non-generic ones whenever a type would otherwise be fixed or duplicated, and prefer a single generic method over overloading the same method for many types. Use bounded wildcards to increase flexibility of use.

The topics covered, in order: generic types (generic classes and interfaces), subclassing them, collections with generics, generic methods, bounded type parameters, wildcards (unbounded, upper-bounded, and lower-bounded), restrictions on wildcards, how objects behave in generic versus non-generic areas, and finally type erasure. Applying these makes real code safer and less redundant.

  • Avoid raw types; keep code type-safe to prevent warnings and runtime failures.
  • Use @SuppressWarnings("unchecked") only when you have verified safety.
  • Prefer generic classes/interfaces and generic methods over rigid or overloaded alternatives.
  • Reach for bounded wildcards to make APIs flexible yet safe.

// Generic class / interface
class Box<T> { T v; }                      // one type parameter
class Pair<K, V> { K k; V v; }             // multiple parameters
interface Container<T> { T get(); }        // generic interface

// Generic method (parameter before return type)
static <T> T identity(T x) { return x; }
static <T extends Number> double sum(List<T> xs) { ... }  // bounded

// Bounded type parameter
<T extends Number>                         // T is Number or subtype
<T extends Comparable<T>>                  // self-comparable
<T extends A & I1 & I2>                    // one class + many interfaces

// Wildcards (reference site only)
List<?>               unbounded  // read as Object; cannot add (except null)
List<? extends Number> upper     // read Number; cannot add   (Producer)
List<? super Integer>  lower     // add Integer; read Object  (Consumer)
// PECS: Producer Extends, Consumer Super

// Raw type (avoid; backward-compat only)
List raw = new ArrayList();      // unchecked warning, no type safety

// Type erasure facts (compile time only)
//   <T>             -> Object
//   <T extends X>   -> X
//   no new T[], no obj instanceof List<String>, no overload by type arg
@SuppressWarnings("unchecked")   // only when proven safe

Section quiz

1. Why can a type parameter T not be used in a static field of a generic class?

  1. Static fields cannot hold objects.
  2. A type parameter is bound per instance, but a static field is shared across all instances, so T has no single meaning.
  3. The compiler forbids static fields in generic classes entirely.
  4. Static fields are erased before compilation.
Answerb — each instance may bind T to a different type, so a single shared static member could not pick one; type parameters apply to instance members only.

2. Which wildcard lets you safely add Integer values to the list inside a method?

  1. List<?>
  2. List<? extends Integer>
  3. List<? extends Number>
  4. List<? super Integer>
Answerd — a lower-bounded wildcard (? super Integer) guarantees the list's real type is a supertype of Integer, so an Integer always fits. Unbounded and upper-bounded wildcards are read-only (Consumer Super).

3. Why do void m(List<String>) and void m(List<Integer>) fail to compile as an overload pair?

  1. List cannot be a method parameter.
  2. After type erasure both signatures become m(List), causing a name clash.
  3. Java forbids more than one method with the same name.
  4. String and Integer are unrelated types.
Answerb — type erasure removes the type arguments, so both erase to the identical raw signature m(List) and the compiler reports a name clash.
