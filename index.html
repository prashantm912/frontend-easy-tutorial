DevDocs

DevDocs is a hands-on learning reference for front-end web development, covering HTML, CSS, JavaScript, and TypeScript from first principles through advanced patterns. Almost every concept ships with a live, editable example you can run and modify in place, so you learn by doing rather than only by reading. The entire site is static and self-contained: no build step, no network requests, and no external dependencies. Download it once and it works completely offline — just open the files in any modern browser.

How to use

Each guide is a single page split into short, focused sections. The sidebar on the left lists every section on the current page; the search box above it filters that list live as you type, so you can jump to a topic without scrolling. The breadcrumb trail at the top of the content shows where you are and links back to Home.

Runnable examples put an editor on the left and a live preview on the right. Edit any code, press Run to re-render the preview, or press Reset to restore the original snippet. JavaScript examples render their output through an in-preview console panel, so remember to console.log(...) anything you want to inspect. Try the mini playground below — it is a full HTML + CSS + JS sandbox seeded with a working example.

Example: the mini playground

Note: every interactive control is keyboard reachable. Use the "Skip to content" link at the very top to bypass the header, and Tab through editors, tabs, and buttons in a logical order.

Angular

Angular is a batteries-included, TypeScript-first framework for single-page applications: components and templates, dependency injection, a router, forms, and an HTTP client all ship first-party. The Angular guide starts from first principles — what a component is, the model–view relationship, and the three kinds of data binding — then works through inputs and outputs, the Angular 17+ block control flow, directives, pipes, dependency injection, change detection, and the full lifecycle.

From there it covers the modern, signals-first Angular of versions 17–22: standalone components, deferrable views with @defer, Signals and the signal-based input()/output()/model() APIs, the Resource API, zoneless change detection, SSR with hydration, and testing. Every concept pairs a static code block with a side-by-side Output box.

Java Multithreading

A standalone, backend study guide to writing correct, fast concurrent programs in core Java — the second course under Java Programming essentials. It rewrites a full "Java Multithreading, Concurrency & Performance Optimization" course into an original, paraphrased reference: thread creation and coordination, optimizing for latency and throughput, the stack/heap memory model, race conditions, data races and deadlocks, the full synchronization toolkit (synchronized, volatile, ReentrantLock, read-write locks, semaphores, condition variables), lock-free CAS algorithms, blocking vs non-blocking IO threading models, and virtual threads.

As with the Collections guide, each lecture pairs a focused, reimplemented Java example with its simulated console output — and because concurrency output is inherently non-deterministic, traces that depend on thread interleaving are clearly labelled illustrative. Case study design: Restaurant Chain Management

A worked High-Level Design (HLD) and Low-Level Design (LLD) for an Angular case study — a restaurant-chain management SPA with Admin (restaurant/menu CRUD + menu assignment), Restaurant Owner (menu view), and User (browse → order → confirmation) roles. It deliberately exercises as many concepts as possible from the Angular Core Deep Dive and Web Design courses, and it stays frontend-only: the backend is simulated with json-server + a dev-server proxy (with a hand-written Express variant documented as an alternative).

Every design decision is stated as what (the concept/tech) and why (the requirement it satisfies and which course teaches it), and Section 16 of the LLD is a full concept-to-file traceability matrix. The page is standalone and offline, with architecture, routing, ER and sequence diagrams drawn as inline SVG, a sticky filtered contents, theme toggle, and print styles.

Use case: Employee Dashboard

The newly added sections across the four guides are not isolated tricks — taken together they teach everything needed to build an offline-capable, capability-aware Employee Profile Dashboard: a page that renders each employee as a responsive card, draws an avatar on a <canvas> with an initials fallback, keeps working without a network by persisting to localStorage, detects and gracefully degrades optional browser features, streams live status over a WebSocket, and exposes machine-readable profile data through microdata.

The map below links each dashboard requirement to the exact section that teaches it. Follow the links in roughly this order and you will have covered every capability the dashboard depends on, from structural markup to typed data models.

Requirement → topic map

  • Feature detection / Modernizr-style guards — detect optional APIs before use: HTML: Feature detection and JavaScript: Feature detection.
  • Canvas avatar with initials fallback — draw the avatar and fall back to rendered initials: HTML: Canvas element and JavaScript: Canvas API.
  • LocalStorage offline UI — persist and rehydrate profile state with no network: JavaScript: Web storage.
  • Geolocation — show an employee's approximate location on demand: JavaScript: Geolocation.
  • WebSockets real-time updates — push live presence and status changes: JavaScript: WebSockets.
  • Microdata / structured data — mark up each profile as a machine-readable entity: HTML: Structured data.
  • mailto / sms contact links — wire up native contact actions: HTML: Contact links.
  • Responsive card grid — lay out profile cards that reflow across viewports: CSS: Dashboard patterns.
  • Hover animations — add compositor-friendly hover feedback to cards: CSS: Card interactions.
  • Collapsible sections — progressively disclose detail within a card: CSS: Collapsible sections.
  • Typed data model (TypeScript) — model the employee, storage, and network layers with types: DOM lib types, Modeling an employee, Typed storage generics, and Typed async APIs.

Note: this is a learning map, not an implementation. It points at the sections that teach each capability so you can assemble the dashboard yourself — the site ships no Employee Dashboard build.

Use case: Typed Employee Model

The advanced TypeScript sections are not standalone syntax lessons — read together they produce a bug-resistant, strongly-typed employee data flow, the same shape you reuse in an Angular application: a modeled Employee entity, a generic ApiResponse<T> envelope, role unions and partial-update mapped types, typed DOM events and form elements, typed async/await guarded by type predicates, and decorators that log methods and validate properties. Because the type system enforces the contract at compile time, whole classes of shape-mismatch and null-handling bugs never reach the browser.

The map below links each artifact and requirement of a Strongly-Typed Employee Model + Form Handling use case to the exact page anchor that teaches it. Follow the links in roughly this order and you will have covered every type-level technique the flow depends on, from the entity model through decorators.

Artifact → topic map

  • employee.model.ts — the entity, its role field, and how updates are typed: Role unions & updates and Modeling an employee.
  • form-handler.ts — typed form elements, typed events, and method/property decorators: Typed forms and Decorators.
  • validation.types.ts — the types that describe and enforce field validation: Validation types.
  • Generics / ApiResponse<T> — a reusable, type-parameterized response envelope: Generic ApiResponse<T> and Generics.
  • Advanced types (role unions, partial-update mapped type) — model exact role sets and safe partial updates: Role unions & updates and Mapped & utility types.
  • DOM typing (typed events + form elements) — reach form fields and events without any: Typed forms and DOM lib types.
  • Typed async/await + type guards — await typed responses and narrow them safely: Typed async APIs, Generic ApiResponse<T>, and Type guards.
  • Decorators (log method + validate property) — cross-cutting logging and validation via decorators: Decorators.

Note: this is a learning map, not an implementation. It points at the sections that teach each type-level technique so you can assemble the typed employee flow yourself — the site ships no Employee Model build.
